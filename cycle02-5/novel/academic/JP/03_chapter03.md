# 第三章：D2——三面の鏡

---

## 身分の問い

D2 の問題は機能の問題ではない。機能の問題は R1 の C1 報告で既に解決されていた——TURING は純粋なエンジニアリング用語で SatiMonitor の四つの機能を列挙した：イベント購読、スライディングウィンドウ、パターン識別、品質指標。そして決定的な「やらないこと」——アクションを実行しない、状態を変更しない、命令を発行しない。

GUARDIAN の類比：APM Agent + Behavioral Analytics + QoS Monitor + SPC Anomaly Detector のハイブリッド。

D2 の問題は身分の問題である：この plugin は五蘊のどのカテゴリで構成されているのか？

Master は手紙の中で既に方向を示唆していた：「この plugin には識蘊の要素が含まれていないのか？」彼は機能を問うているのではない——身分を問うているのである。

## 三面の鏡

蘊帰属の議論が D2 の核心であった。四つの方案は C2 報告で既に列挙されていた：

| 方案 | 蘊構成 | 理由 |
|------|--------|------|
| A | 受 + 想 | 感知 + 認識、最もシンプル |
| B | 受 + 想 + 識 | 感知 + 認識 + 評価判断 |
| C | 想 + 識 | 認識 + 評価、受蘊が薄すぎる |
| D | 純識 | すべての認知活動を識蘊に帰属 |

三つの異なる学問分野からの論証が三面の鏡のように、異なる角度から同一の対象を照らした。

**LINNAEUS の第一の鏡——行動特性。** 分類学者は SatiMonitor の四つの機能を列挙し、蘊の定義と一つずつ照合した：

- イベント購読 → 刺激の受容 → 受蘊（vedana）
- パターン識別 → 類型の認識 → 想蘊（samjna）
- 品質評価 → 良否の判断 → 識蘊（vijnana）
- スライディングウィンドウ → 単なるデータ構造 → 蘊に帰属しない

三つの機能、三つの蘊。第四の機能（スライディングウィンドウ）はエンジニアリング実装の詳細であり、蘊の分類に影響しない。

**WIENER の第二の鏡——身分の定義。** 制御理論の専門家はより鋭い問いを発した：「どの蘊を除去すると、SatiMonitor は SatiMonitor でなくなるか？」

- 受蘊を除去 → 感知がない → イベントの品質が分からない → ログ記録器に退化
- 想蘊を除去 → 認識がない → 行動パターンを識別できない → 生のイベントカウンタに退化
- 識蘊を除去 → 評価がない → 品質の良否を判断できない → 評価なしの統計レポートに退化

蘊を一つ除去するたびに、SatiMonitor は身分に不可欠な機能を一つ失う。結論：三つの蘊はいずれも身分に必須である。

**HERACLITUS の第三の鏡——退化モード。** 動態の専門家は別の角度から同じ結論を確認した。「何を除去すると機能を失うか」ではなく、「退化後のシステムに独立した存在価値があるか」を問うた。

三つの退化モードはいずれも「機能が欠損したシステム」を産出した——完全に無用ではないが、SatiMonitor の核心的特質を失っている。

三面の鏡。三つの学問分野。同一の結論：受 + 想 + 識。

## 受蘊の論争

しかしこの結論は異議に直面した。

方案 C の支持者——KERNEL と BABBAGE——は受蘊が薄すぎると主張した。SatiMonitor の「イベント購読」機能における受蘊の役割は何か？ 単にイベントを受信するだけか？ それなら EventListener と何が違うのか？

ASANGA の応答が議論の方向を変えた。彼は対照表を取り出した——仏学的マインドフルネスと SatiMonitor の項目別比較：

| 次元 | 仏学的正念（Sati） | SatiMonitor |
|------|----------------|-------------|
| 能動性 | 能動的（精進） | 受動的（event-driven） |
| 道徳性 | 道徳的に正向 | 価値中立 |
| 意志性 | 意図を要する | 自動運転 |
| 伝統的帰属 | 行蘊心所 | 受+想+識 |
| 修行上の地位 | 八正道の第七支 | 品質モニタ |

五つの次元。五つの不一致。SatiMonitor はマインドフルネスのエンジニアリング化ではない——それは価値中立的な観察者である。

次に彼は受蘊の問題に転じた：「SatiMonitor がイベントを受信するとき、単に受動的に受信しているだけではない——イベントに対する品質感覚を持っている。tool call の遅延が 100ms か 10s かによって、SatiMonitor は異なる品質判断を生成する。この判断の基盤は——感覚である。速度の感覚、連続性の感覚、粒度の感覚。感覚は受蘊の核心機能である。」

## ゼロコストの転換

ARCHIMEDES が全員の立場を変える一言を述べた。

彼が問うたのは蘊帰属ではない——コストを問うたのである：「SatiMonitor が skandha: ['vedana', 'samjna', 'vijnana'] と宣言した場合、エンジニアリングコストは何か？」

答え：ゼロ。

skandha 宣言は metadata である。D3-R3 が後に正式に裁定した——skandha は metadata であり、ルーティングの基盤ではない。三蘊と宣言しても二蘊と宣言しても、エンジニアリング実装上いかなる差異もない。コード変更なし。インターフェース変更なし。テスト失敗なし。

「ゼロコストで、より精確なら——なぜ選ばないのか？」

この一言が D2-Q2 議論のターニングポイントであった。哲学的論証ではない——エンジニアリング経済学である。

## 投票

**D2-R2：SatiMonitor skandha = ['vedana', 'samjna', 'vijnana']。18/20。**

これは OpenStarry 史上初の三蘊 plugin である。これまですべての plugin は単蘊か二蘊であった。

二票の反対は KERNEL と BABBAGE からのものであった——彼らは依然として受蘊が SatiMonitor において薄すぎると考えた。しかし結論のエンジニアリング的合理性は受け入れた。

**D2-R3** がそれに続いた：PluginHooks に `monitors?: ISatiMonitor[]` 配列スロットを追加する。単数ではない——複数である。Agent は複数のモニタを装着でき、それぞれ異なる重点を持つ。20/20。
