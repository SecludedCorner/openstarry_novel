# 第二章：D1——仏学対応づけの境界

---

## 十度の全員一致

D1 は本プロジェクトの歴史的記録を打ち立てた：十項目の投票、十度の 20/20、少数意見ゼロ。

20 名のメンバー、6 つの学問分野にまたがる研究チームにおいて、この程度の合意は極めて稀である。Cycle 02-4 の 55 項目の決議のうち、全員一致で可決されたものは三分の一に満たなかった。なぜ D1 はそれを成し遂げたのか？

意見の相違がなかったからではない。相違がフレームワークによって解消されたからである。

四層フレームワークと三項テストは R0 の時点で既に存在し、R1 で三つの独立した学問分野によって論証され、R2 で異議なく承認された。D1 の投票時には、フレームワーク自体はもはや議論の対象ではなかった——議論の道具であった。全員が同じ物差しで測定するとき、測定結果は自然と一致する。

## D1-Q2：五十件の対応づけの運命

### Batch A：Master が批判した 5 項目（全て REMOVE）

śīla = Hard Rules、upāya = Soft Rules、smṛti = Monitoring、戒定慧三学の対応づけ、Doc 41 の装飾的ラベル。

NAGARJUNA はこの場面で、六回の研究を通じて一度もしなかったことをした——一人称で過ちを認めた。

「A-1 から A-3——戒、善巧、マインドフルネス——これらは Cycle 02-3 の D5 議論終了後に付け加えたものだ。議論の中で導かれた結論ではない。事後に貼り付けたラベルだ。**後貼りラベル。**」

彼はこの精確な、逃げのない言葉を用いた。外科医が自分の手術痕を指して「この一刀は逸れた」と言うように。

「Hard Rules を śīla と呼ぶ必要はない。その機能はルール執行であって、戒律の修行ではない。śīla でラベルを貼ることは、機能を説明しているのではなく、宗教的装飾を施しているのだ。」

PASCAL はノートに数式を記した：$P(\text{後貼りラベルが装飾} \mid \text{作成者自身が認める}) \to 1.0$

事後確率は 1 に収束する。ラベルの作成者自身がそれを装飾だと認めたとき、それ以上の証拠は不要である。

投票：**20/20。**

### Batch B：8 項目の学術的内容（全て RELOCATE）

このバッチには精細な分離が必要であった——白黒の KEEP や REMOVE ではなく、灰色の RELOCATE である。

ARCHIMEDES は RELOCATE の模範例を確立した：Doc 37 §9 における PASCAL の数学的形式化 $Var(\epsilon) = f(\theta_{moha})$。数学公式は本文に保持し（独立したエンジニアリング価値がある）、仏学引用は付録に移動する（設計理由であり、エンジニアリング仕様ではない）。

「数学を保持し、仏学を移動する」——エンジニアリング内容と仏学内容の境界面がメスで切り開かれ、それぞれが然るべき場所に配された。

MN 18 の経典引用、Doc 41 の DD-13 仏学背景、Doc 44 §10 の仏学的設計理由——各項目に具体的な配置方案がある。廃棄ではない、移転である。

投票：**20/20。**

### Batch C：7 項目のコード識別子（全て KEEP）

TURING は意見を述べなかった。端末で一行のコマンドを実行しただけであった：

```bash
rg "Moha|Drishti|Mana|Sneha" --type ts -l
```

十七のファイル。Moha は import 文に、class 定義に、describe ブロックに、expect アサーションに現れた。ラベルではない——コードの一部である。

LINNAEUS は分類学の観点から確認した：「これらの名前は既にシステムの分類学的ラベルとなっている。付加的なものではない——身分的なものである。Moha は何かに貼り付けられたラベルではない。Moha はその対象の名前そのものなのだ。」

投票：**20/20。**

## D1-Q7/Q8：二つの文書の運命と種子

Doc 16（装飾比率 〜80%）と Doc 31（装飾比率 〜70%）。

ARCHIMEDES は Scheme B 分割を提案した：エンジニアリング価値のある段落を抽出し、残りを削除するか付録に格下げする。BABBAGE は Doc 31 の IPC Cocycle 条件に修正を加えた——数学的厳密性があり、独立して保存する価値がある。

D1-Q7/Q8：分割。両票とも **20/20。**

しかしこの二つの決議は一つの種子を埋めた。これらは後に Master 自身の手で覆される——チームの判断が誤っていたからではなく、判断のフレームワークにもう一つの次元が欠けていたからである。フレームワークは「エンジニアリング内容」と「仏学的装飾」を区別したが、「エンジニアリング文書に埋め込まれた仏学」と「仏学対応づけを目的とした独立文書」を区別していなかった。

Doc 16 は仏学が埋め込まれたエンジニアリング文書ではない。Doc 16 自体が仏学対応づけ文書なのだ——「装飾比率 80%」でそれを評価するのは、「散文比率 80%」で詩を評価するようなものである。基準が誤っていた。

しかしこの洞察は、第五章で Master が審査を行うまで現れない。
