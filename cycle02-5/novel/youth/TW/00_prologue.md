# 序章：一封讓人不安的信

---

想像你花了好幾個月的時間蓋一棟房子。你量了每塊磚，畫了每張設計圖，和一群朋友一起討論了無數次「這面牆該放這裡還是那裡」。

然後有一天，房子的主人走進來，看了看四周，說：

「我越來越看不懂你們在做什麼了。」

這就是 Cycle 02-5 的開場。

---

## 師父的信

師父——我們叫他 Master——是 OpenStarry 的設計者。整個系統是他的構想。研究團隊的工作是幫他把構想變成精確的設計圖。

信只有 45 行。但研究總監 SUNYATA 讀了五遍。

師父指出了幾件讓他不滿意的事：

**第一，團隊把一個應該是外掛的東西塞進了核心。** IGearArbiter 是一個判斷用哪種方式處理問題的「仲裁器」。按照系統的基本原則——核心不應該有任何具體功能——它應該是外掛（plugin），不是核心的一部分。

師父說這是「非常嚴重的錯誤」。在六輪研究中，他只用過兩次這個措辭。

**第二，太多佛學裝飾。** 系統用了很多佛學術語來命名——五蘊（五個核心分類）、煩惱（四個情緒模組）等等。有些命名是好的：Moha（迷惑）模組真的在模擬自我迷惑的行為。

但有些命名太牽強了。師父舉了一個例子：「event-driven 不等於正念。只是正念是用 event-driven 做的。」

event-driven 是一種程式設計方式。正念是一種心靈修行。說它們相等，就像說螺絲刀等於修理——你可以用螺絲刀來修理東西，但螺絲刀不是修理本身。

**第三，正念監控器的身份問題。** 系統中有一個叫「SatiMonitor」的元件，Sati 是巴利文的「正念」。師父問：「這個 plugin 難道沒有識蘊的成分在嗎？」

這個問題後來改變了整個研究的方向。

---

## 凌晨的對話

SUNYATA 讀完信後，在凌晨三點把劇場的燈光從暖色調到冷白色。他沒有告訴任何人為什麼。

凌晨四點，哲學家 NAGARJUNA 自己來了。他承認自己在之前的工作中犯了錯——在工程文件上貼了不必要的佛學標籤。「那不是映射。那是裝飾。」

凌晨四點半，數學家 PASCAL 來了。他寫了一個簡單的公式：

```
保留裝飾的成本 = 每位讀者 × 每次閱讀 × 困惑時間 = 不斷累積
移除有用東西的成本 = 搜尋一次就好 = 可控

保留裝飾的損害 >> 移除的損害
```

凌晨五點，佛學家 ASANGA 來了。他提出了一個簡單的判斷標準：如果一個佛學概念驅動了工程決策——保留。如果只是裝飾——移除。

三個人。三個學科。三條不同的推理路徑——範疇分析、量化損害、因果效力——但同一個結論：裝飾性的佛學映射應該被清理掉。

這就像三位醫生用不同的檢查方法——X 光、驗血、超音波——最後得出了相同的診斷。不同方法得出相同結論，表示這個結論很可能是對的。

---

## 四層框架

SUNYATA 根據這些討論，制定了一套判斷標準：

| 怎麼做 | 什麼時候 |
|--------|---------|
| **保留** | 已經是程式碼一部分的佛學名字，或師父確認過的設計概念 |
| **搬家** | 有學術價值但不屬於工程正文的佛學內容 → 移到附錄 |
| **移除** | 純粹裝飾性的佛學標籤 |
| **整份文件評估** | 佛學裝飾比例太高的文件 |

加上三項測試：
1. 移除它會改變設計嗎？不會 → 移除
2. 這個名字在程式碼中被使用了嗎？是 → 保留
3. 這個佛學概念驅動了設計決策嗎？是 → 保留

然後，研究分成三條路線同時進行：
- **路線 A**：搞清楚五蘊架構到底怎麼運作
- **路線 B**：審計所有佛學映射
- **路線 C**：重新定位「正念」元件的身份

二十位研究者。九份獨立報告。然後是辯論。
