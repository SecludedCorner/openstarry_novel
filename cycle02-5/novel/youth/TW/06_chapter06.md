# 第六章：D5——沒有佛學的辯論

---

## 十個人，零個佛學家

D5 是一場特殊的辯論。

只有十位參與者——而且沒有佛學家。NAGARJUNA 和 ASANGA 自願退出了。NAGARJUNA 的臨別語很有意思：「D4 證明了名字需要對定義負責。D5 會證明工程設計不需要佛學名字也能完成。這兩件事同樣重要。」

D5 的議題是純工程問題：把 D4 改完的介面設計成可以實際執行的完整規格。師父的要求是「規格達到 100%，然後交給工程團隊」。

---

## 程式碼考古學

TURING 為 D5 準備了一份特別的報告——他分析了 14 個原始碼文件中的每一個設計模式。每一個 timeout 是多少？每一個 Registry 怎麼運作？每一種失敗時怎麼處理？

他把這份報告叫做「程式碼考古學」。

這份報告改變了辯論的方式。之前的辯論建立在「原則」上——原則可以有不同的解讀。但 TURING 的報告建立在「事實」上——事實只有一個。

---

## 九個決定

D5 用七十五分鐘做了九項投票。速度很快——因為每一項都有事實基礎。

最重要的幾個決定：

**IConfidenceAuditor 用獨立的插槽。** 不和監控器共用。原因：監控器只觀察（沒有副作用），審計器會呼叫 AI 模型（有副作用）。觀察者和有副作用的元件不應該混在一起。全票通過。

**回傳類型用雙模式。** 可以同步也可以非同步。這是 D5 中最接近的投票——5 票對 3 票。爭議不是「雙模式好不好」，而是「應不應該打破現有的設計模式」。最終「一致性」贏了——因為在工程中，和現有程式碼保持一致通常比追求完美更重要。

**timeout 200 毫秒。** 如果審計在 200 毫秒內沒回應，就跳過（不修正信心度）。和現有的齒輪仲裁器保持一致。全票通過。

**只允許一個審計器。** 不搞複雜的多審計器聚合策略。遵循 YAGNI 原則——You Ain't Gonna Need It（你用不到的東西別先做）。6 對 2 通過。

**均等權重。** 品質向量的四個維度——連續性、粒度、速度、均衡性——每個都是 0.25。控制理論專家 WIENER 說：「沒有運行數據的時候，等權重是最保守的選擇。」全票通過。

---

## 最終規格

九項投票結束後，TURING 在白板上寫下了最終的介面：

```typescript
export interface IConfidenceAuditor extends IVijnana {
  readonly id: string;
  audit(confidence: number, context: GearContext):
    ConfidenceAuditResult | Promise<ConfidenceAuditResult>;
}

export interface ConfidenceAuditResult {
  delta: number;       // 被限制在 [-0.05, +0.05]
  reasoning: string;   // 審計紀錄
}
```

「100%，」他說。

這不是概念設計。這是可以直接打開程式編輯器開始寫的完整規格。

---

## 零佛學

記錄員 SCRIBE 統計了 D5 全場辯論中佛學術語被使用的次數。

零。

D5 是本專案歷史上第一場完全沒有佛學內容的辯論。不是刻意避開——是自然的結果。當你討論的是 timeout 數值和失敗處理策略時，你不需要引用任何佛學經典。

NAGARJUNA 在辯論結束後走過來說：「D4 證明了名字需要對定義負責。D5 證明了——有些工程問題根本不需要定義。只需要規格。」

這不矛盾——這是同一架天平的兩面。一面問「你的名字配得上定義嗎？」另一面問「你的規格足夠精確嗎？」D4 校準了第一面。D5 完成了第二面。

兩面都需要。缺一不可。
