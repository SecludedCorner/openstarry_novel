# 第5章：D4――その名前はふさわしいか？

---

## あの問い

SUNYATA は師父の言葉をスクリーンに映しました。大きな文字。白地に黒。

> **「梵語を使うなら、その定義に責任を持つ必要がある。Sati の中身が本当にマッチしていると思うか？」**

これは Cycle 02-5 全体で最も核心的な問いでした。

---

## 論理的帰結

哲学者 NAGARJUNA が立ち上がりました。ホワイトボードに3行書きました。

```
前提1：正念（Sati）は仏教において行蘊の活動である
前提2：SatiMonitor は行蘊に属さない（D2 で証明済み）
結論 ：SatiMonitor は正念ではない
```

「正念が必然的に行蘊の活動であり、SatiMonitor が行蘊でないなら――SatiMonitor は正念ではない。」

彼はペンを置きました。

「正念でないものを、なぜ Sati と呼ぶのか？」

会場が沈黙しました。異論があったからではありません。全員が同時に、もっと早く気づくべきだったことを理解したからです。

---

## 5つの不一致

仏教学者 ASANGA が比較表を取り出しました。

| | 仏教における正念 | SatiMonitor |
|--|------------|-------------|
| 能動的か受動的か？ | 能動的（精進が必要） | 受動的（自動で動く） |
| 道徳的方向性はあるか？ | ある（道徳的に正しい方向） | ない（価値中立） |
| 意図は必要か？ | 必要 | 不要 |
| どの蘊に属するか？ | 行蘊 | 受+想+識 |
| 位置づけは？ | 八正道の第七支 | 品質モニター |

5つの側面。5つすべてが不一致。

「私たちは間違いを犯した」と ASANGA は言いました。「分類の間違いではない――分類は正しかった。命名の間違いだ。あまりに長くこの名前を使い続けて、慣れてしまい、名前そのものが正しいかどうかを確認するのを忘れていた。」

---

## 新しい名前

SUNYATA が問いかけました。「ではそれは何と呼ぶべきか？」

4つの提案のうち、ARCHIMEDES（工学実践の専門家）の案が最も多くの支持を集めました。

**ILoopQualityMonitor** ―― 認知ループ品質モニター。

理由はストレートでした。「新しく入ったエンジニアがこの名前を見たら、何をするものかすぐにわかる――ループの品質を監視する。仏教なし。メタファーなし。説明が必要な歴史なし。」

**13/20 で可決。**

---

## 核融合炉

そして SUNYATA が2つの単語を口にしました。「IPrajna。」

IPrajna はもう一つの仏教用語で命名されたインターフェースです。般若（Prajñā）は仏教における最高の智慧――一切の実相を照見する超越的な認知です。

NAGARJUNA がホワイトボードに2行書きました。

```
般若（仏教）：一切法の実相を照見する最高の智慧
IPrajna（工学）：信頼度を ±0.05 増減させる関数
```

ASANGA が、後に全員の記憶に残るひと言を放ちました。

「それは温度微調整つまみを『核融合炉』と呼ぶようなものだ。」

誰も笑いませんでした。事実だったからです。

IPrajna が行うことは何か？ 信頼度の数値を受け取り、最大 ±0.05 を増減させて結果を返す。微調整器。クランプ関数。これと「一切法の実相を照見する」との距離は、つまみと核融合ほどかけ離れています。

**16/20 で可決：IPrajna → IConfidenceAuditor（信頼度監査器）。**

---

## 第4のテスト

D4 はただ2つの名前を変えただけではありません。新たなテスト基準を生み出しました。

D4 以前、チームには3つのテスト（必要性、コード識別、意思決定駆動）がありました。しかしこの3つのテストには盲点がありました。「名前は定義と一致しているか？」を問わなかったのです。

ISatiMonitor は Test 2 を通過していました（コードの中で使われている）。しかし、もっと根本的な基準を通過していなかったのです――名前と機能が一致していない。

そこでチームは第4のテストを追加しました。

> **Test 4（定義への責任）：仏教梵語を用いてコードの名前をつけた場合、コンポーネントの機能はその仏教用語の意味と本当に一致しているか？ 一致していなければ、工学用語を使うこと。**

この基準で Doc 03（`Sila_Prajna_Security_Framework.md`）を見直すと――Sila（戒律）が「アクセス制御」を表す？ Prajna（般若）が「セキュリティ脆弱性への対応メカニズム」を表す？ 4つのテストすべてに不合格です。

**17/20 で可決：Doc 03 を Plugin Security Lifecycle に改名。**

---

## 天秤

D4 にかかったのはわずか30分。しかし Cycle 02-5 全体のクライマックスでした。

これまでの3つのテストが問うていたのは、「この仏教の概念は工学に役立つか？」――仏教から工学への方向です。

第4のテストが問うのは、「この工学的命名は仏教の定義に忠実か？」――工学から仏教への方向です。

天秤を想像してください。一方の皿に名前、もう一方に定義。

釣り合っているとき――Moha（迷妄）モジュールは確かに迷妄の挙動をシミュレートしている――名前を残す。
釣り合っていないとき――SatiMonitor は正念ではない、IPrajna は般若ではない――名前を置き換える。

天秤は名前がどれほど古いか、どれほど崇高かを気にしません。天秤が気にするのは、両端が釣り合っているかどうかだけです。

PASCAL のまとめ：「師父はひと言で済ませた。私たちは討論一回分を費やした。結論は同じだ。しかし討論の価値は――なぜそうなのかを説明したことにある。」
