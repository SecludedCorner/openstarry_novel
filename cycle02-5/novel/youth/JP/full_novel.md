# 序章：手術室のライト

---

午前三時、SUNYATA は一人で円形劇場に座っていました。

目の前のコンソールにはボタンが一列に並んでいて、その中の一つが照明の色温度を調整するものでした。彼はつまみを暖かいオレンジ色（3200K、カフェのような光）から冷たい白色（6500K、病院の手術室のような光）へと回しました。

劇場全体の雰囲気が、わずか三秒で変わりました。

彼がそうしたのは、一通の手紙を読んだからです。

---

この手紙は Master からのものでした——研究チームにとって最も重要な指導者です。手紙は長くありません。たった 45 行で、一つのフォルダに入っていました。でも SUNYATA は五回読み返しました。

Master は何を言ったのでしょう？簡単にまとめると：

**「君たちの工程文書には、仏教ラベルが貼られすぎている。中にはかなりこじつけのものもある。それに——あの Sati というコンポーネント、本当に『正念』なのか？」**

この手紙にはメスのような鋭さがありました。Master は怒っているのではなく——正確さを求めていたのです。

SUNYATA は今回の研究テーマを決めました：**五蘊システムは実際にどう動いているのか？エンジニアにもわかる言葉で説明すること。**

---

「五蘊」とは何でしょう？この AI Agent システムでは、五蘊は五つの基本機能のことです：

1. **色蘊（rupa）**：入出力——外界を感知し、外界に応答する
2. **受蘊（vedana）**：感覚——出来事に対して良い/悪い/中立の感じを持つ
3. **想蘊（samjna）**：認識——「これは何か」を見分ける
4. **行蘊（samskara）**：行動——実際にものごとを実行する
5. **識蘊（vijnana）**：判断と意思決定——すべてを統括する

この五つの「蘊」は、コンピュータの五つのコアサブシステムのようなものです。Master が知りたかったのは：それぞれがどう接続されているのか？どう連携するのか？どう循環するのか？

そして——仏教の名前で名付けられたものたちは、名前と機能が本当に一致しているのか？

20 人の研究者。5 回の討論。一日の中で。

手術が始まりました。

---

# 第一章：すべての壁を点検する

---

討論が始まる前に、研究チームはとても大事なことをしました——まず各自で研究し、それからお互いの成果をチェックしたのです。

## 各自の研究

一つの建物を想像してください。すべての壁、すべての柱、すべての配線を点検する必要があります。外観だけではダメ——構造を見なければなりません。

九つのグループが分かれて動きました：

**五つのグループは建物の構造を点検**（五蘊がどう動くか）：
- 柱は十分に太いか？（五つのルートインターフェースは十分か）
- 配線はどうつながっている？（依存性注入のワイヤリングはどうなっているか）
- 部品はどう組み込まれる？（Plugin はどう読み込まれるか）
- 信号はどう流れる？（認知ループはどう動くか）
- 各フロア間はどう連絡する？（五蘊間はどう連携するか）

**二つのグループは壁の装飾を点検**（仏教ラベルのうち、どれを残し、どれを剥がすか）：
- ARCHIMEDES は 7 つの文書を一行ずつ監査し、50 個の仏教マッピングを発見——23 個は残すべき、13 個は付録へ移すべき、14 個は剥がすべきと判断しました
- NAGARJUNA と PASCAL は判断基準を作りました：三項目テスト（除去しても仕様は変わるか？ソースコードで使われているか？設計上の意思決定に関係するか？）

**二つのグループは特殊なコンポーネントを研究**——SatiMonitor：
- TURING は実際に何をしているかを分析しました（純粋な観察、副作用ゼロ、病院の心電図モニターのようなもの）
- ASANGA はどの蘊から構成されているかを分析しました（感覚がある、認識がある、判断がある——でも行動がない）

## お互いのチェック

その後、お互いのレポートを確認し合いました。TURING は引用されたすべてのソースコードを照合して、誰も間違って記録していないことを確認しました。

結果：24 項目は全員一致。7 項目に議論の余地があり、討論で決める必要がありました。

# 第二章：どのラベルを剥がすべきか？

---

第一回討論（D1）で議論されたのは、工程文書に貼られた仏教ラベルのうち、どれを残し、どれを剥がすかということでした。

## ルールはシンプル

チームは一つの判断ルールを作りました：

**四層分類**：
- **KEEP**（保留）：この仏教名称には工学的な意味がある
- **RELOCATE**（移動）：この仏教的内容には学術的価値があるが、工程文書に置くべきではない——付録へ移す
- **REMOVE**（除去）：これは単なる装飾で、剥がしても何にも影響しない
- **FILE REVIEW**（文書全体の審査）：半分以上が装飾の文書は、全体を再チェックする必要がある

**三項目テスト**：
1. 剥がした後、工程仕様は変わるか？ → 変わらなければ剥がす
2. この用語はソースコードで使われているか？ → 使われていれば残す
3. この仏教概念が工学的な設計判断を駆動したか？ → 駆動していれば残す

## 投票結果

十回の投票。十回とも全会一致（20/20）。少数意見ゼロ。

これは本プロジェクトの歴史上、一度も起きたことのないことでした。

なぜこれほど一致したのでしょう？それは、判断基準が討論の前にすでに確立されていたからです。全員が同じ物差しで測れば、結果は自然と一致するのです。

**剥がされたもの**：Hard rules = 戒律？剥がす。Soft rules = 善巧方便？剥がす。event-driven = 正念？剥がす。これらはすべて、工学設計が完成した後に貼り付けられた仏教ラベルでした——NAGARJUNA 自身が認めた「後貼りラベル」です。

**残されたもの**：Moha（自己欺瞞）、Sneha（我愛）、skandha（蘊）——これらはソースコード内で実際に使われている名前であり、機能も仏教的定義と一致しています。

**移動されたもの**：経典の引用、哲学的比較表——学術的価値があるので、付録へ移しました。

## でも一つ小さな問題がありました

チームは Doc 16 と Doc 31 も投票で分割しました。その後、Master がこの二つの決定を覆しました。

Master の説明：「Doc 16 と Doc 31 は工程文書に仏教の装飾を貼ったものではない——それ自体が仏教マッピング文書なのだ。『装飾の比率が高すぎる』という基準でマッピング文書を評価するのは、詩に対して『散文が多すぎる』と文句を言うようなものだ。基準の使い方を間違えている。」

チームのフレームワークには一つの次元が欠けていました——**文書の種類**です。工程文書の中の仏教は装飾です。仏教マッピング文書の中の仏教はコンテンツです。大きな違いがあります。

# 第三章：SatiMonitor の正体は？

---

第二回討論（D2）で答えるべき Master の問いは：SatiMonitor というコンポーネントは、五蘊のうちどれで構成されているのか？

## まず、何をしているかを確認する

SatiMonitor がしていることは四つです：
1. **イベントを購読する**——システムのイベントストリームに接続し、11 種類の信号を継続的に受信する
2. **履歴を記憶する**——「スライディングウィンドウ」を使って、最近起きたことを記録する
3. **パターンを認識する**——「この行動は繰り返されている」「このパターンは異常だ」と見抜く
4. **品質スコアを計算する**——四つのスコアを出す（連続性、繊細さ、速度、バランス）

重要なのは——何を**しないか**です：いかなるアクションも実行しない、いかなる状態も変更しない、いかなるギアも切り替えない。病院の心電図モニターのようなもの——観察するだけで、治療はしません。

## 四つの提案

| 提案 | 蘊の組み合わせ | 支持理由 |
|------|--------------|---------|
| A | 受+想 | 観察と認識だけを見ている |
| **B** | **受+想+識** | **観察+認識+判断、完全** |
| C | 想+識 | 観察の部分を無視している |
| D | 識のみ | 単純化しすぎている |

## キーとなる転換点

討論の冒頭、「三蘊 plugin は複雑すぎる」と心配する声がありました。しかし ARCHIMEDES が計算してみると——二蘊から三蘊に変更しても、コードに追加が必要なのは配列要素一つ分、約 4 行のコードの差でした。

この心配が解消された後、議論は純粋に「分類が正確かどうか」の問題になりました。三蘊の精度は、二蘊を圧倒的に上回りました。

最終投票：**Proposal B（受+想+識）18 票、Proposal C 2 票。**

SatiMonitor は OpenStarry 初の「三蘊 plugin」になりました。

## なぜ行蘊を含まないのか？

仏教における「正念」は行蘊に属します——それは主体的で、意志を持った修行活動です。しかし SatiMonitor は受動的で、自動的な品質モニターです。「正念を修める」のではなく——ただ「観察している」だけなのです。

この区別は後にとても重要になります。次の章で、Master はこの区別をきっかけに、名前の体系全体を問いただすことになります。

# 第四章：この建物に住めるのか？

---

第三回討論（D3）は最も長く——二時間かかりました。問いはストレートです：五蘊の OOP アーキテクチャは、十分に優れているか？

## 五本の柱

システムには五つのルートインターフェースがあります（建物の五本の主柱のようなものです）：

| 柱 | 担当 |
|------|---------|
| IRupa（色蘊） | 入出力 |
| IVedana（受蘊） | 感覚 |
| ISamjna（想蘊） | 認識と処理 |
| ISamskara（行蘊） | アクションの実行 |
| IVijnana（識蘊） | 判断と意思決定 |

五本の柱で十分でしょうか？四人の専門家が四つの異なる方法で検証しました——分類学、数学、仏教学、オペレーティングシステム——結論は一致しました：**十分です。20/20 で可決。**

## 興味深い例外

一つのコンポーネント——SlashCommand（スラッシュコマンド）——はどの蘊にも属しません。なぜでしょう？認知ループ全体をバイパスして、直接実行するからです。くしゃみのようなもの——脳で考えることなく完了します。

しかし GUARDIAN がセキュリティ上の問題を指摘しました：もし plugin が SlashCommand を通じて危険なことをした場合、システムのすべてのセーフティメカニズムがそれを止められません。この問題は新しい設計文書に記録されました。

## 最大の議論：古代仏教理論はプログラムに対応できるのか？

六項目の投票の中で最も白熱したのは：十二因縁（仏教の理論）をシステムの実行ループにマッピングできるかどうかでした。

- **反対側**（7 票）：十二因縁が説くのは「三世因果」——過去・現在・未来の三つの生にまたがるものです。プログラムの一回のループはわずか数ミリ秒。スケールがまったく違います。
- **賛成側**（13 票）：完全なマッピングはしない。重なる部分だけを参考として記録する。

結果：13/20、選択的な付録マッピング。

しかしもう一つの理論——「認知系列」（citta-vithi）——はより高い 17/20 の支持を得ました。なぜなら、それが描写するのは「一回の認知活動の内部段階」であり——プログラムの認知ループとスケールが一致するからです。BABBAGE は数学的にも、両者の構造が対応できることを証明しました。

## 結論

**この建物には住めます。** 五本の柱は頑丈で、配線は正しく、構造は完全です。三つの小さな隙間（弱い継承、配線未完了、二つのコンポーネント未実装）がありますが、これらは「まだ建設中のフロア」であって、「柱にヒビが入っている」わけではありません。

# 第五章：その名前はあなたにふさわしいか？

---

これは Cycle 02-5 全体で最も重要な討論でした——D4。当初の計画にはありませんでした。Master の一言がきっかけで始まったのです：

> **「梵語を使うなら、その定義に対して責任を持つ必要がある。」**

## 名前の問題

第三章で、SatiMonitor には行蘊が含まれないことがわかりました。しかし「Sati」は仏教では「正念」のこと——正念は行蘊に属します。

すると問題が浮かびます：

もし SatiMonitor が行蘊活動**ではない**なら → それは正念**ではない** → ではなぜ **Sati** と名付けられているのか？

NAGARJUNA は論理的推論（彼はこれを「帰謬法」と呼びます）でこの矛盾を示しました：

```
SatiMonitor は行蘊ではない（D2 で証明済み）
正念は行蘊である（仏教の定義）
よって SatiMonitor は正念ではない
よって Sati と呼ぶべきではない
```

## 第一の改名：ISatiMonitor → ILoopQualityMonitor

投票：13/20。新しい名前の意味は「ループ品質モニター」——このコンポーネントが実際にしていることを正確に表現しています。

## 第二の改名はさらに衝撃的でした

SUNYATA がもう一つの名前に言及しました：**IPrajna**。

般若（Prajna）は仏教で何を意味するか？**最高の智慧。すべてのものの真の姿を見通す力。**

IPrajna はプログラムの中で何をしているか？**一つの数値に 0.05 を足したり引いたりする。**

ASANGA が言った一言を、全員が忘れませんでした：

> **「これは温度の微調整つまみを核融合炉と呼ぶようなものです。」**

誰も笑いませんでした。本当のことを言っていたからです。

投票：**IPrajna → IConfidenceAuditor**、16/20。新しい名前の意味は「信頼度監査員」——このコンポーネントの実際の機能を正直に表現しています。

## 第四のテスト

D4 は一つの恒久的な新ルールを生み出しました——**定義責任テスト**：

> 仏教の名前でプログラムのコンポーネントを命名するとき、そのコンポーネントの機能はその仏教名の意味と本当に一致していなければなりません。一致しなければ、普通の工学的な名前を使います。

どの名前がこのテストに合格したか？**Moha**（自己欺瞞——プログラムの中で実際に自己欺瞞をシミュレートしている）。**Skandha**（蘊——プログラムの中で実際に五蘊分類を行っている）。

どの名前が不合格だったか？**Sati**（正念——しかし SatiMonitor は正念を実践していない）。**Prajna**（最高の智慧——しかし IPrajna は 0.05 を足し引きするだけ）。

名前はタダではありません。偉大な名前を平凡な機能に貸しても——機能が偉大になるわけではなく、名前の価値が下がるだけなのです。

# 第六章：仏教を必要としない討論

---

第五回討論（D5）は、前の四回とはまったく異なるものでした。

仏教がありません。梵語もありません。哲学的な議論もありません。あるのは工学だけです。

## なぜ違うのか？

D5 で議論されたのは、IConfidenceAuditor（先ほど改名されたばかりのコンポーネント）の正確な技術仕様だからです。Master は言いました：「仕様を 100% にしてから、エンジニアリングチームに渡せ。」

そのため今回は工学に関わる 10 人の研究者だけが参加しました。仏教学者の NAGARJUNA と ASANGA は自ら辞退しました——この討論には仏教的分析が不要だったからです。

## 九つの工学的問題

TURING（ソースコード専門家）が事前に完全な「コード考古学」レポートを作成していました——14 個のソースコードファイルを調査し、システムの現在の設計パターンをすべて記録したものです。このレポートがあったおかげで、どの問題にも事実に基づいた議論ができました。

九つの問題には以下が含まれます：
- 新コンポーネントはどのスロットに配置するか？ → 独立スロット（他と共用しない）
- 戻り値の型は？ → 同期でも非同期でも可（既存の設計と一貫性を保つ）
- タイムアウトは？ → 200 ミリ秒、タイムアウトしたら修正なしとして扱う
- 複数インストールできるか？ → 不可、一つで十分（YAGNI 原則——使わないものは作らない）
- 品質スコアの重み付けは？ → 四つの次元を各 25%（データがない段階では、等重みが最も安全）

九項目の投票が七十五分で完了しました。最終的に、エンジニアがすぐにコーディングを始められる完全な仕様書が生まれました。

## NAGARJUNA のコメント

D5 が終わった後、NAGARJUNA は TURING の前に歩み寄って言いました：

「D4 は名前が定義に対して責任を持つべきだと証明しました。D5 は、ある種の工学的問題には定義すら必要ない——仕様さえあればいいと証明しました。」

二つの文で、二つの討論の本質を言い切りました。

# 終章：天秤

---

五回の討論が終わりました。31 回の投票。29 項の正式決議。375 分間。

## 何をしたのか？

**新しい文書（Doc 45）を作りました**——純粋な工学用語で五蘊システムの完全なアーキテクチャを説明するものです。五つのルートインターフェースの構成、依存性注入の接続方法、Plugin の読み込み方、認知ループの動作、五蘊間の連携方法。Master のすべての質問に答えました。

**不要なラベルを剥がしました**——14 個の装飾的な仏教マッピングが除去されました。8 個の学術的価値のある内容が付録に移されました。

**名前と実態が合っていない名前を変えました**——ISatiMonitor は ILoopQualityMonitor（ループ品質モニター）に改名されました。IPrajna は IConfidenceAuditor（信頼度監査員）に改名されました。

**名前と実態が合っている名前は残しました**——Moha、Sneha、skandha、vedana など——これらの機能は本当に仏教の定義に合致しています。

**恒久的なルールを確立しました**——四層フレームワーク、四項テスト。その中で最も重要なのは Test 4（定義責任）：仏教の名前でプログラムのコンポーネントを命名するなら、機能は定義と一致しなければなりません。

## 何を学んだのか？

Cycle 02-5 は一台の天秤のようでした。

天秤の一方には**名前**が載せられています。もう一方には**定義**が載せられています。

名前と定義のバランスが取れているものもありました——Moha（自己欺瞞）は実際に自己欺瞞をシミュレートしています。天秤は水平です。保留。

名前と定義のバランスが崩れているものもありました——Sati（正念）なのに、SatiMonitor は正念を実践していません。天秤は傾いています。改名。

名前と定義のバランスが大きく崩れているものもありました——Prajna（最高の智慧）なのに、IPrajna は 0.05 を足し引きするだけです。天秤はほとんどひっくり返っています。迷わず改名。

天秤のルールはシンプルです：**名前はタダではありません。すべての偉大な名前は約束です——あなたがしていることがその名前にふさわしいという約束です。**

ふさわしければ——使う。

ふさわしくなければ——正直な名前に変える。

これが Cycle 02-5 の結論です。
