# 第六章：D5——没有佛学的辩论

---

## 十个人，零个佛学家

D5 是一场特殊的辩论。

只有十位参与者——而且没有佛学家。NAGARJUNA 和 ASANGA 自愿退出了。NAGARJUNA 的临别语很有意思："D4 证明了名字需要对定义负责。D5 会证明工程设计不需要佛学名字也能完成。这两件事同样重要。"

D5 的议题是纯工程问题：把 D4 改完的接口设计成可以实际执行的完整规格。师父的要求是"规格达到 100%，然后交给工程团队"。

---

## 程序代码考古学

TURING 为 D5 准备了一份特别的报告——他分析了 14 个原始代码文件中的每一个设计模式。每一个 timeout 是多少？每一个 Registry 怎么运作？每一种失败时怎么处理？

他把这份报告叫做"程序代码考古学"。

这份报告改变了辩论的方式。之前的辩论建立在"原则"上——原则可以有不同的解读。但 TURING 的报告建立在"事实"上——事实只有一个。

---

## 九个决定

D5 用七十五分钟做了九项投票。速度很快——因为每一项都有事实基础。

最重要的几个决定：

**IConfidenceAuditor 用独立的插槽。** 不和监控器共用。原因：监控器只观察（没有副作用），审计器会调用 AI 模型（有副作用）。观察者和有副作用的组件不应该混在一起。全票通过。

**回传类型用双模式。** 可以同步也可以异步。这是 D5 中最接近的投票——5 票对 3 票。争议不是"双模式好不好"，而是"应不应该打破现有的设计模式"。最终"一致性"赢了——因为在工程中，和现有程序代码保持一致通常比追求完美更重要。

**timeout 200 毫秒。** 如果审计在 200 毫秒内没回应，就跳过（不修正信心度）。和现有的齿轮仲裁器保持一致。全票通过。

**只允许一个审计器。** 不搞复杂的多审计器聚合策略。遵循 YAGNI 原则——You Ain't Gonna Need It（你用不到的东西别先做）。6 对 2 通过。

**均等权重。** 品质向量的四个维度——连续性、粒度、速度、均衡性——每个都是 0.25。控制理论专家 WIENER 说："没有运行数据的时候，等权重是最保守的选择。"全票通过。

## 最终规格

九项投票结束后，TURING 在白板上写下了最终的接口：

```typescript
export interface IConfidenceAuditor extends IVijnana {
  readonly id: string;
  audit(confidence: number, context: GearContext):
    ConfidenceAuditResult | Promise<ConfidenceAuditResult>;
}

export interface ConfidenceAuditResult {
  delta: number;       // 被限制在 [-0.05, +0.05]
  reasoning: string;   // 审计纪录
}
```

"100%，"他说。

这不是概念设计。这是可以直接打开程序编辑器开始写的完整规格。

## 零佛学

记录员 SCRIBE 统计了 D5 全场辩论中佛学术语被使用的次数。

零。

D5 是本项目历史上第一场完全没有佛学内容的辩论。不是刻意避开——是自然的结果。当你讨论的是 timeout 数值和失败处理策略时，你不需要引用任何佛学经典。

NAGARJUNA 在辩论结束后走过来说："D4 证明了名字需要对定义负责。D5 证明了——有些工程问题根本不需要定义。只需要规格。"

这不矛盾——这是同一架天平的两面。一面问"你的名字配得上定义吗？"另一面问"你的规格足够精确吗？"D4 校准了第一面。D5 完成了第二面。

两面都需要。缺一不可。
