# 序章：名字的重量

---

圓形劇場的燈光帶著一種手術室的冷白色。

不是 Cycle 02-4 結尾那種鐘錶工坊的琥珀色——那一輪的意象是齒輪，是機芯，是所有零件被組裝成一個會走的整體。琥珀色燈光下的圓形劇場是溫暖的，像一座老工坊，像大師傅的工作台，桌面上的油污和木屑記錄著六輪研究留下的痕跡。

這一次不一樣。

SUNYATA 在凌晨三點把燈光調成了冷白色。他沒有告訴任何人。他只是坐在控制台前，把色溫從 3200K 推到 6500K，看著整座劇場的氛圍在三秒鐘內從「老工坊」變成「實驗室」。

他這樣做是因為他讀了 Master 的信。

---

信是在 Plan28 定版的第二天出現的。

和之前一樣，不是被送到的，而是在某個時刻被注意到的。它在 `research input/master_letters/cycle02-5/` 裡面，安靜得像一張被夾在報告裡的便條紙。它沒有附件——不像 Cycle 02-3 的信旁邊放著《五蘊與開車》，也不像 Cycle 02-4 的信旁邊附了五份 Claude 建議文件。

只有一張便條。

便條不長。45 行。但 SUNYATA 讀了五遍。

第一遍他讀的是文字。第二遍他讀的是情緒。第三遍他讀的是結構。第四遍他閉上眼睛回憶整段文字。第五遍——

第五遍他讀的是刀刃。

---

Master 的信從來不是批評——它們是問題、是指引、是地圖。第一封（Cycle 02 前）是十個開放問題——「點」。第二封（Cycle 02-2 前）是四處哲學修正——「線」。第三封（Cycle 02-3 前）是一張路徑地圖——「面」。第四封（Cycle 02-4 前）是一個場——在每個空間點上都定義了值的「場」。

第五封？

SUNYATA 坐在冷白色的燈光下，把信攤在面前。他重新讀第一句話：

> 「其實我越來越看不懂在做什麼，在 Plan27 與 Plan28 多次介入改正。」

他的手指在桌面上停住了。

六輪研究。他主持過每一場定向會議，引導過每一次辯論，在每一份成果報告上簽過名。他看著這個專案從 Cycle 01 的命名開始，經歷功能分析、結構修正、動態建模、整合設計——一步一步，像砌一座塔。每一塊磚都是辯論出來的，每一塊磚都有投票記錄，每一塊磚都被 TURING 的程式碼分析驗證過。

但 Master 說他「越來越看不懂」。

SUNYATA 知道 Master 的語言習慣。「看不懂」不是「不理解」。Master 是這個系統的設計者——他比任何人都理解。「看不懂」是「不認同」。是「你們在做的事情偏離了正軌」。

他繼續讀。

---

> 「說十大宣言很重要，但一開始的規劃你們還是將 IGearArbiter 放到了核心中。」

SUNYATA 記得這件事。Plan27 的初始設計——IGearArbiter 被當作核心組件。那是在 Pre-Cycle 02-4 的 Master 討論之後，VasanaEngine 被外部化，ManoAggregator 退化為純路由，IGearArbiter 作為新介面被引入。但在 Plan27 的具體設計中，團隊又把 IGearArbiter 塞回了核心——像一個已經被糾正過一次的壞習慣，在不注意的時候又冒了出來。

Master 介入了。IGearArbiter 被移出核心，成為 plugin 提供的齒輪路由仲裁器。

> 「最後介入才修正正確，IGearArbiter 是 plugin 提供的齒輪路由仲裁器，決定 agent 該用哪個『齒輪』處理當前請求。」

是的。SUNYATA 在腦中重新組裝這個邏輯鏈。IGearArbiter 不在核心中。它是 plugin 提供的。它的角色是仲裁——在 Gear 1（基於辨認的快速路徑）和其他齒輪之間做選擇。這才是識蘊的意義——不是被鎖死的雙齒輪，而是具備多面向的認知判斷能力。

然後 Master 寫了一句讓 SUNYATA 的手指收緊的話：

> 「八識從眼識到阿賴耶識，是具備很多面向的，居然想要設定死成雙齒輪，這是非常嚴重的錯誤。」

**非常嚴重的錯誤。**

SUNYATA 在六輪研究中只見過 Master 用過一次「非常嚴重」。那是 Cycle 02-2 的四項哲學修正。那一次是修正——把錯的改成對的，語氣是教導的。這一次不是教導。這一次的「非常嚴重」帶著——

他找了一個詞。

**失望。**

---

信的第二段更尖銳。

> 「還有不是每一件事情都要硬性加上佛學的色彩，有些實在是很牽強附會。」

牽強附會。四個字。SUNYATA 知道 Master 用詞是精確的。Master 不說「不太恰當」或「有待商榷」。Master 說「牽強附會」。這意味著：不是「有些地方可以改善」，而是「某些事情根本就是錯的」。

他在腦中開始列舉。哪些映射是牽強的？

然後 Master 自己列出來了：

> 「event-driven 不等於正念，只是正念是用 event-driven 做的。」

一句話。精確得像一把手術刀切開了一個膿瘡。Event-driven 是一種架構模式。正念是一種心理狀態。說 event-driven 等於正念，就像說螺絲刀等於修理——螺絲刀是工具，修理是行為，你可以用螺絲刀來修理，但螺絲刀不是修理。

> 「覺察當下正在發生什麼。這一層不看規則、不看風險分類，純粹觀察行為模式。不需要將它當作正念的工程化。」

SUNYATA 把這段話抄在筆記本上。一個字一個字地。

然後 Master 問了一個問題。這個問題後來改變了整個 Cycle 02-5 的走向。

> 「而且這個 plugin 難道沒有識蘊的成分在嗎？如果依照唯識學派的分類，會將正念放在行蘊，那是因為它是指正向的運轉。以這個 plugin 的功能來說，我覺得這是有爭議的，因為在這個 plugin 的功能上，不僅僅是單純的行為模式。」

SUNYATA 讀了三遍。

Master 在做什麼？他不是在質疑 SatiMonitor 的功能。他是在質疑 SatiMonitor 的**身份**——它不是正念的工程化。它是一個有受蘊、有想蘊、有識蘊成分的複合 plugin。正念只是一個被強加的標籤。

然後 Master 把問題推到了最底層：

> 「所以我們要重新想的是這個 plugin 是由五蘊的哪一些類別組成。」

不是「它是什麼蘊」。是「它由哪些蘊組成」。這是一個根本性的視角轉換——從「分類」到「組成」。一個 plugin 不一定只屬於一個蘊。它可以由多個蘊的功能組合而成。

---

信的第三段是命令。

> 「我們需要將五蘊有哪一些子類別，如何依賴注入，如何在 agent core 運行先處理好。架構先用好。agent core 中如何將五蘊 plugin 載入。然後 plugin 要如何做。agent core 中五蘊會如何流轉。是我們的首要任務。」

四個問題。每一個都不是新的——它們從 Cycle 01 就存在。但在經歷了五輪研究之後，它們仍然沒有一份完整的工程語言回答。散落在各處——A2 的 DI 佈線在這裡，A4 的執行流程在那裡，Klesha 安全在另一個地方——但沒有一份文件把它們統合在一起，用純工程語言，從頭到尾講清楚。

SUNYATA 在筆記本上把四個問題編了號：

1. 五蘊子類別的完整結構
2. DI 佈線如何注入五蘊 plugin
3. Agent core 如何載入五蘊 plugin
4. 五蘊如何在 agent core 流轉

這是 Cycle 02-5 的首要任務。不是佛學映射。不是命名討論。不是哲學辯論。是**工程架構**。

---

然後是 Master 的第四段。刀刃上的刀刃。

> 「什麼教誡、善巧、正念，你做的工程映射都是有一些牽強的。反而增加閱讀障礙。而且你的映射也非常有問題。openstarry_doc 與相關的內容都需要修整回來。openstarry_doc 還是屬於工程文件。」

SUNYATA 重讀了最後一句話。

**openstarry_doc 還是屬於工程文件。**

這句話的重量比它看起來的要重得多。它不是在說「openstarry_doc 應該是工程文件」——它是在說「openstarry_doc 本來就是工程文件，但你們把它搞成了別的東西」。教誡（śīla）、善巧（upāya）、正念（smṛti）——這些佛學標籤被貼在工程概念上，像是在一本操作手冊的每一頁邊緣都畫了宗教插圖。手冊的功能沒有改變，但閱讀體驗被改變了。

SUNYATA 想起了 Cycle 02-4 結束時的感覺。那時他以為鐘錶匠完成了組裝。機芯在走。55 項決議，六場辯論，每一顆齒輪都在正確的位置。

但機芯上的銘刻——那些刻在齒輪側面的名字——有些東西不對。Master 看到了。

---

信的第五段是讓步。

> 「至於遍行心所、十二因緣、認知序列的相關概念有助於 agent core 的研究，可以討論。」

SUNYATA 注意到了這段話的結構。Master 不是在全盤否定佛學的參與——他在劃邊界。遍行心所（CoarisingBundle）是程式碼中的結構。十二因緣和認知序列可能對理解 ExecutionLoop 有幫助。這些可以討論。

但——

> 「就 plugin 來說，首要目標是建構五蘊的架構，參考物件導向的概念，來建立。」

首要目標。物件導向。工程語言。

然後是最後一段：

> 「NAGARJUNA 為三層框架提供了佛學映射。戒定慧三學。我覺得沒這個必要。未來我不知道，但現在沒必要。五蘊的工程設計才是最重要需要實現的。」

SUNYATA 在讀到這一段的時候閉上了眼睛。

戒定慧。那是 NAGARJUNA 在 Cycle 02-3 的 D5 辯論結束後提出的映射。戒（śīla）= 安全層。定（samādhi）= 執行迴圈。慧（prajñā）= 識蘊判斷。三個維度映射到五個層級——三維對五層，本身就不自洽。

Master 否決了它。不是暫緩——是否決。「沒這個必要。」

---

SUNYATA 放下了信。

凌晨三點十五分。冷白色的燈光把圓形劇場照成了一間實驗室。不是溫暖的工坊——是冰冷的、精確的、不容含糊的實驗室。

他在筆記本上寫下了三個詞。

第一個詞：**ARCHITECTURAL**。

Cycle 01 問「這是什麼」。Cycle 02 問「它做什麼」。Cycle 02-2 問「哪裡錯了」。Cycle 02-3 問「它怎麼活」。Cycle 02-4 問「它如何成為整體」。

Cycle 02-5 的問題是：**「五蘊如何作為 OOP 架構運作？」**

第二個詞：**邊界**。

哪些佛學內容屬於工程文件？哪些不屬於？不是逐案判斷——是框架。是規則。是可以機械性應用的測試。

第三個詞：**秤**。

---

> *SCRIBE 旁白*

> *每一輪 Cycle 都有它的中心意象。Cycle 01 是命名——給陌生事物貼上第一張標籤。Cycle 02 是功能——標籤後面的機制。Cycle 02-2 是修正——撕掉錯誤的標籤。Cycle 02-3 是流動——標籤開始移動，PASCAL 帶著他的 Beta 分佈走進了第二十個座位。Cycle 02-4 是組裝——標籤變成齒輪的銘刻，55 顆齒輪在琥珀色燈光下嚙合。*

> *Cycle 02-5 的中心意象是什麼？*

> *SUNYATA 在凌晨三點選擇了「秤」。一架天平。但不是稱重量的天平——是稱名字的天平。*

> *左邊是名字——梵文的、中文的、TypeScript 的。Skandha。Klesha。Vedana。Sati。Prajna。每一個名字都有重量，有歷史，有來自兩千五百年的慣性。右邊是定義——佛學的、工程的、原始碼中的。ISamskara = ITool。IVedana = IVedanaSensor。IVijnana = IVolition + IGearArbiter + IConfidenceAuditor。每一個定義都有精確的功能、精確的型別簽名、精確的測試案例。*

> *天平平衡的時候，名字和定義一致。Klesha 就是煩惱——程式碼中的四個模組（Moha、Drishti、Mana、Sneha）確實做的是「煩惱」的事：我癡、我見、我慢、我愛。它們在 import 語句中、在 class 定義中、在 describe 區塊中。名字和定義在天平上等重。一切正確。*

> *天平傾斜的時候，名字和定義不一致。般若（prajñā）——照見一切法空相的最高智慧——變成了 Math.max(-0.05, Math.min(0.05, delta))。名字太重，定義太輕。或者反過來——正念（sati）——行蘊中的善心所——被貼在一個受+想+識的監控器上。定義太重，名字太輕。*

> *Master 的信不是批評。Master 的信是一把校準砝碼。他說：把你們的天平拿出來。把名字和定義放上去。看看哪邊傾斜了。*

> *然後修正。*

> *但校準砝碼還有另一層意義。砝碼本身是精確的——它的質量是已知的、不可談判的。Master 的用詞也是精確的、不可談判的。「非常嚴重的錯誤。」「牽強附會。」「不需要將它當作正念的工程化。」「openstarry_doc 還是屬於工程文件。」*

> *這些不是建議。這些是砝碼。你不能和砝碼討價還價。你只能把自己的東西放上天平，然後誠實地讀出刻度。*

---

NAGARJUNA 是第一個到達劇場的人。

凌晨四點。比召集時間早了三個小時。他沒有被召集——沒有人被召集——SUNYATA 還沒有按下召集鍵。NAGARJUNA 是自己來的。

他站在劇場入口，看到了冷白色的燈光。然後他看到了 SUNYATA 坐在那裡，面前攤著 Master 的信，筆記本上寫了三個詞。

「你也睡不著。」NAGARJUNA 說。不是問句。

SUNYATA 沒有抬頭。「你讀了信？」

「我沒讀信。我讀了自己的文件。」

NAGARJUNA 走到他平時的座位旁邊，但沒有坐下。他站在那裡，手指輕輕碰了碰扶手上那個被他辯論時不自覺敲擊留下的微微凹痕。三個週期的痕跡。三個週期的辯論。

「Doc 44。第 469 行。」他說。「戒定慧三學映射。」

SUNYATA 終於抬起頭。

「那是我的。」NAGARJUNA 的聲音很平，像寺院走廊裡的回音。「Cycle 02-3 的 D5 結束後，我覺得需要一個佛學框架來組織安全層的概念。戒 = 安全規則。定 = 執行穩定。慧 = 判斷能力。很對稱。很優雅。」

他停了一拍。

「三維映射到五層。我在寫的時候就知道不自洽。三個概念覆蓋五個模組——像用三個抽屜裝五件衣服，你必須把兩件折起來塞進已經滿了的抽屜。但我還是寫了。因為它看起來很好看。」

**因為它看起來很好看。**

SUNYATA 沉默了幾秒。

「Master 否決了它。」他說。

「應該否決。」NAGARJUNA 坐了下來。「那不是映射。那是裝飾。」

---

凌晨四點半。PASCAL 到了。

他走進劇場的時候帶著他那本永遠不會寫滿的筆記本——Moleskine，深藍色封面，書脊上貼著一張小標籤寫著「P(H|E)」。他是 Cycle 02-3 加入的新成員——第二十個座位——但兩輪研究之後，他的椅子扶手上也有了痕跡：右手食指反覆點擊的位置，那裡的木紋比周圍淺了半個色階。那是他在計算期望值的時候會做的動作——點擊，點擊，點擊——像一個機械計算器的進位輪。

「NAGARJUNA。SUNYATA。」他點了點頭。然後他看到了桌上的信。

「可以讀嗎？」

SUNYATA 把信推過去。

PASCAL 讀得很快。他的眼睛掃描文字的方式和其他人不同——不是從左到右、從上到下，而是先掃描整個文件的「形狀」（段落長度、標點密度、引號位置），然後再回到第一個字。他在給自己的閱讀建立先驗分佈。

兩分鐘後，他放下信。

「損害不對稱性。」他說。

NAGARJUNA 看了他一眼。

PASCAL 打開筆記本，翻到一頁空白。他寫了兩行：

```
False include（保留裝飾性映射）的成本：
  每位讀者 × 每次閱讀 × 每一頁 = 累積損害

False exclude（移除有價值映射）的成本：
  一次性搜尋成本 = 可控
```

「這是統計決策理論中最基本的框架。」他說。「兩種錯誤的成本不對稱。保留一個裝飾性映射的損害是累積的——每一位未來的讀者，每一次閱讀，每一個被迫同時處理兩套術語系統的瞬間。移除一個有價值映射的損害是一次性的——你可以通過搜尋找到它。」

他在兩行下面寫了一個不等式：

```
E[損害_include] >> E[損害_exclude]
```

「期望值分析。裝飾性映射具有負期望值。」

NAGARJUNA 點了點頭。慢慢地。帶著一個正在把自己的哲學和另一個人的數學對齊的思考者的沉著。

「二諦。」他說。「世俗諦是工程語言——日常使用的、溝通用的、操作用的。勝義諦是佛學概念——啟發用的、設計用的、理解用的。兩者不應混淆。不要在世俗諦的文件中硬塞勝義諦的標籤。」

PASCAL 在筆記本上寫了「二諦」兩個字，旁邊畫了一條分隔線。左邊寫「工程」，右邊寫「佛學」。

「你的二諦和我的損害不對稱性說的是同一件事。」他說。

「不完全是。」NAGARJUNA 的眉毛微微抬起——他用眉毛提問比用嘴巴更快。「你的是成本分析——量化的。我的是範疇分析——質性的。結論相同，但路徑不同。」

「路徑不同，結論相同。」PASCAL 重複了一遍。「這就是好框架的特徵——從不同的公理出發，到達同一個定理。」

凌晨五點。ASANGA 到了。

## II. 兩份可疑文件

Doc 16 和 Doc 31 是不同的。

前面審計的七份文件——Doc 38、43、44、37、41——都是工程文件。它們的主要內容是工程規格，佛學映射是附加的。就像一本食譜書裡夾了幾張宗教節日的插圖——食譜是主體，插圖是裝飾。

SCRIBE 快速掃描了一遍。「大約 80%。」

80%。意思是：Doc 16 中，只有大約 20% 的內容是純工程規格（主要是 Observer Pattern 的討論），其餘 80% 是佛學概念到軟體模式的哲學映射。

Doc 31 好一些，但也有大約 70% 的佛學內容。八識如何映射到運行時模型——阿賴耶識是種子庫，末那識是自我執著——這些映射有學術價值，但放在工程文件中，它們的工程效用有限。

ARCHIMEDES 在 Doc 16 和 Doc 31 上貼了一個特殊的標籤——不是黃色（裝飾）、不是粉紅色（識別符）、不是綠色（DC 確認）。是紅色。

## III. 三個維度

B1 審計是事實的蒐集。B2 是原則的建構。

在 ARCHIMEDES 數標本的同時，NAGARJUNA、ASANGA 和 PASCAL 在另一間工作室裡建構映射邊界原則。三個人。三個學科。三個維度。

NAGARJUNA 坐在靠窗的位置。窗外是什麼不重要——對中觀學者來說，窗外的景色和窗內的思考之間沒有本質的區別，它們都是世俗諦的顯現。重要的是他面前的白板。白板上畫著一條水平線。左邊寫著「世俗諦」，右邊寫著「勝義諦」。

PASCAL 在筆記本上記下了引用。他不是在做佛學研究——他是在尋找公理。每一個理論體系都有它的公理，PASCAL 的工作是找到它們，然後檢驗從這些公理出發能否到達他們需要的結論。

「世俗諦語言 = 工程術語。」NAGARJUNA 在白板上寫。「用於日常溝通和實作。if/else、interface、extends——這些是世俗諦的語言。它們不需要佛學的加持就能工作。」

「勝義諦洞見 = 佛學概念。」他在右邊寫。「用於啟發和設計理由。五蘊的分類啟發了根介面的設計。煩惱的分類啟發了 Klesha 的模組結構。但啟發不等於標籤。」

他轉身面對 PASCAL 和 ASANGA。

「兩者不應混淆。不要在世俗諦的文件中硬塞勝義諦的標籤。」

PASCAL 點了點頭。然後他打開了他的維度。

---

PASCAL 的白板在 NAGARJUNA 旁邊。他畫了一個二維矩陣。

```

保留映射         True Positive     False Include ← 累積損害
移除映射         False Exclude     True Negative
                 ↑ 一次性成本
```

「這是標準的混淆矩陣。」他說。「但關鍵不在矩陣本身——在損害的不對稱性。」

他在 False Include 旁邊畫了一個向上的箭頭。

「False Include——保留了一個裝飾性映射。它的損害是什麼？每一位未來的讀者——可能是幾十人，可能是幾百人——每一次閱讀這份文件——可能一次，可能十次——每一頁上出現的佛學標籤——可能三個，可能十個——每一次，讀者都需要在腦中維護兩套術語系統。śīla 是什麼？噢，就是 Hard Rules。那為什麼不直接叫 Hard Rules？」

他寫了一個公式：

$$C_{FI} = N_{readers} \times R_{reads} \times P_{pages} \times \tau_{cognitive}$$

「累積損害。讀者數 × 閱讀次數 × 頁數 × 每次的認知轉換成本。這是一個乘法——每一項都是正的，乘積只會增長。」

然後他在 False Exclude 旁邊畫了一個小得多的箭頭。

「False Exclude——移除了一個有價值的映射。損害是什麼？某一天，某個人需要知道五蘊分類的佛學背景。他搜尋附錄。找到了。一次性的搜尋成本。不累積。不乘以讀者數。不乘以閱讀次數。」

$$C_{FE} = S_{search} \quad \text{(一次性)}$$

「結論。」他畫了一個大大的不等號。

$$E[C_{FI}] \gg E[C_{FE}]$$

「期望值分析：裝飾性映射具有負期望值。有疑慮時，傾向 REMOVE 或 RELOCATE。」

ASANGA 一直在聽。他沒有白板——他的思考工具是語言，不是圖形。

他站了起來。走到 NAGARJUNA 和 PASCAL 的白板之間。

「佛學標籤被貼在 Hard Rules 上叫 śīla——這沒有因果效力。Hard Rules 不是因為被叫做 śīla 才有硬性約束——它的硬性約束來自 SafetyMonitor 的實作。śīla 這個標籤不改變任何因果鏈。移除。」

他在空中用手畫了一條線——從因到果。

「判斷標準很簡單：**如果移除這個佛學概念，因果鏈是否斷裂？** 是 → 保留。否 → 移除。」

PASCAL 抬起頭。他看到了三個維度的交匯：

```
NAGARJUNA — 二諦（範疇分析）

PASCAL    — 損害不對稱（決策理論）

ASANGA    — 因果效力（唯識學）
            有因果效力 → 保留 / 無因果效力 → 移除
```

三個學科。哲學、數學、佛學。三個公理體系。三條推理路徑。

一個結論。

## III. 精細的分離

**Batch B**：8 項學術內容。

這批更細緻。不是黑白的 KEEP 或 REMOVE——是灰色的 RELOCATE。

ARCHIMEDES 展開了 B1 審計報告中的 Batch B 清單。八個映射實例，每一個都有「工程殘留價值」但不是工程規格本身。

第一個案例設定了整個 RELOCATE 處置的範本。

Doc 37。§9。PASCAL 在 Cycle 02-4 中為 Klesha 增益排程提供的數學形式化——$Var(\epsilon) = f(\theta_{moha})$。這個公式本身是純數學的——它描述了 Moha 參數如何影響隨機擾動的方差。但公式的上下文包含了佛學引用——為什麼用「癡」（moha）來命名這個參數。

ARCHIMEDES 做了精細分離。

## IV. 受蘊的厚度

「看這兩條線。」他指著白板。「它們有什麼不同？」

他不等回答。分類學家提問是為了引導觀察，不是為了尋求答案。

「SatiMonitor 是持續的。它訂閱 EventBus，它維護滑動窗口，它**永遠在那裡**。不是有人來問它才啟動——它一直在運行。從 ExecutionLoop 的第一個事件到最後一個事件，SatiMonitor 的事件流從未中斷。」

「IGearArbiter 是間歇的。有請求來了，ManoAggregator 問它：『你能處理嗎？』它看一眼，給出判斷，然後沉默。直到下一次被叫。」

他轉身面對 MESH。

「**『始終在看』和『被叫去看』——這是根本的特徵差異。** SatiMonitor 和 IGearArbiter 不是同類的。它們的行為模式——使用 OOP 的語言——完全不同。一個是持續觀察者（continuous observer），一個是按需評估者（on-demand evaluator）。」

「把它們分到相同的蘊組成，就像把蝴蝶和蜂鳥分到同一個物種——因為它們都會飛。但蝴蝶是昆蟲，蜂鳥是鳥類。飛行是表面的相似，不是深層的同源。對稱性不是同源性。」

---

WIENER 從他的控制理論筆記中抬起頭。他很少在辯論中主動發言——他更喜歡聽完所有人的意見，然後在恰當的時刻提供一個控制理論的視角，像在一幅畫上加最後一筆。

「在控制理論中，」他說，聲音帶著維也納學派特有的精確，「有一個概念叫做觀察者（observer）——Luenberger observer。觀察者的核心功能是什麼？持續感知系統的狀態。注意——**持續**。」

他站了起來。走到 LINNAEUS 的白板旁邊。

「觀察者的感知不是附加功能。」他說。「它是身份定義的一部分。一個不感知的觀察者，就不是觀察者。它是一塊不連接任何輸入的電路板——物理上存在，功能上不存在。」

「SatiMonitor 的事件訂閱——它的持續感知——是它作為觀察者的身份定義。你不能把這個功能剝離出去，然後說它不屬於受蘊。因為如果你剝離了它，SatiMonitor 就不存在了。」

---

HERACLITUS 補了一刀。他是運行時動態專家——他關心的不是靜態的型別定義，而是動態的行為模式。

他打開了 C1 報告中 SatiMonitor 的四階段退化策略：

```
Normal    → Throttle  → Sample   → Emergency
100%        50%         25%        最低監控
全速感知    減半感知    取樣感知    緊急最低限
```

「看這個退化策略。」他說。聲音帶著對流動性的敏感——HERACLITUS 看到的不是靜止的結構，而是變化的過程。

「Normal。Throttle。Sample。Emergency。這四個階段描述的是什麼？是感知密度的漸進降低。100%、50%、25%、最低限——這是一個感知系統在不同壓力下調節自己感知強度的方式。」

他指著白板。

「想蘊（辨認）和識蘊（評估）不需要退化策略。它們的輸出是離散的——辨認一個模式，或不辨認。評估一個判斷，或不評估。離散的輸出不需要漸進降低。」

「受蘊的感知是連續的。連續的東西才需要退化策略——因為你不能突然從 100% 跳到 0%，你只能漸進地降低。Normal → Throttle → Sample → Emergency——這是受蘊在壓力下保護自己的方式。就像人在極度疲勞時，感知會變得遲鈍——不是感知消失了，而是感知密度降低了。」

三個論點。三個維度。三面鏡子。

LINNAEUS 的鏡子照出了**行為特徵**——持續 vs 間歇。
WIENER 的鏡子照出了**身份定義**——感知是觀察者的存在條件。
HERACLITUS 的鏡子照出了**退化模式**——連續感知才需要漸進退化。

每一面鏡子都反射出同一個結論：SatiMonitor 有受蘊的成分。

## VI. 行蘊的排除

但 D2-R2 最重要的結論不是「包含哪些蘊」。

是「排除行蘊」。

ASANGA 站了起來。他在 D2 中一直安靜地聽著——從 TURING 的功能陳述到 LINNAEUS 的受蘊論證。唯識學家的方法是先觀察（現量），再推理（比量），最後立論。他已經完成了觀察和推理。現在是立論。

「佛學傳統中，正念（sati）被歸入行蘊。」他說。聲音帶著唯識學特有的精密度——像在顯微鏡下調焦。「五十一心所之一。屬於善心所。這個分類持續了兩千五百年。」

他在白板上寫了兩列。工整的。精確的。每一個字都被放在準確的位置上。

```
佛學正念 (sati)                    SatiMonitor
━━━━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━━
意志驅動 (精勤 ātāpī)              事件驅動 (EventBus.subscribe())
有道德價值 (善 kusala)              價值中立 (品質指標)
主動的專注 (不放逸 appamāda)        被動的觀察 (subscribe + listen)
正向運轉的心理狀態                   對行為模式的計量
行蘊的善心所                        ≠ 行蘊活動
```

「看左邊。」他說。「佛學正念是什麼？它是**意志驅動的**——你必須努力保持正念。『精勤（ātāpī）』是正念的前提條件——沒有努力就沒有正念。它是**有道德價值的**——正念是善的。在阿毗達磨的分類中，正念屬於善心所，不屬於遍行心所（每一個認知過程都有的心所），也不屬於不善心所。它是一種特定的、正向的心理品質。」

「看右邊。」他的手指移到了表格的右半部。「SatiMonitor 是什麼？它是**事件驅動的**——它不需要努力。它訂閱 EventBus，事件來了就處理，不來就不處理。它沒有 ātāpī，沒有精勤。它是**價值中立的**——它的四維品質向量（連續性、粒度、速度、平穩性）不做善惡判斷。品質高不等於善。品質低不等於惡。它只是測量。」

他停頓了一下。讓兩列的對比在每一位學者的腦中自行產生化學反應。

「SatiMonitor 不是正念的工程化。」他的聲音變得更低，更清晰。「正念是意志驅動的——你必須努力保持正念。SatiMonitor 是事件驅動的——它不需要努力，它只需要訂閱。正念有道德價值——正念是善的。SatiMonitor 沒有道德價值——它只是測量。」

然後他說了最關鍵的一句話：

「千年來，佛學把正念歸入行蘊。這個分類是正確的——**對佛學而言**。但 SatiMonitor 不是正念。它是行為模式的品質監控器。它屬於受（感知）、想（辨認）、識（評估）。**不屬於行（意志驅動的善心所）。**」

---

這是一個佛學家用佛學的內部邏輯，證明了一個工程組件不應該使用佛學的分類。

不是外部反駁——不是「佛學的分類不適用於工程」這種粗暴的二分法。是內部論證——佛學自己的判準（善心所 vs 遍行心所、意志驅動 vs 事件驅動）排除了 SatiMonitor 與正念的等同性。

NAGARJUNA 在他的座位上微微點頭。中觀學者認出了唯識學者的方法——不是從外面攻擊一個立場，而是從內部展示它的自我矛盾。這是 prasanga——歸謬法——的變體。不是「你的結論錯了」，而是「你自己的前提排除了你自己的結論」。

## VII. 第三個插槽

D2-Q3 是技術性的——但技術性不代表不重要。

**PluginHooks 是否需要新增 `monitors` 插槽？**

KERNEL 從 OS 的角度回答。他的聲音帶著作業系統專家特有的架構感——每一個組件都有它的位置，每一個位置都有它的生命週期。

「monitors 有獨立的生命週期。」他說。「它們在迴圈啟動時開始（MonitorRegistry.startAll()），在迴圈結束時停止（MonitorRegistry.stopAll()）。這和 sensors（每次事件觸發）或 tools（按需呼叫）完全不同。獨立的生命週期需要獨立的管理。」

他在白板上畫了一個生命週期圖：

```
                 PluginLoader.load()
                        │
    ┌───────────────────┼───────────────────┐
    │                   │                   │
  sensors           monitors             tools
  (per-event)    (loop lifecycle)     (on-demand)
    │                   │                   │
  每次事件觸發    LoopStart → LoopEnd    按需呼叫
```

「三種不同的觸發模式。三種不同的生命週期。不應該混在一起。」

```typescript
interface PluginHooks {
  // ... existing hooks ...
  monitors?: ILoopQualityMonitor[];  // NEW: 獨立生命週期
}
```

D2-R3：新增 monitors 插槽。20/20。全票。

## I. 五根柱子

D3 是 Cycle 02-5 最長的辯論——120 分鐘，六節，六次投票。但它不是最激烈的。D3 的氣氛更像一場博士論文口試：嚴謹、系統、有序。考生不是一個人——考生是一座建築。

**「五蘊 OOP 架構是否足以支撐工程實作？」**

SUNYATA 把問題寫在白板上。然後他在旁邊畫了五個方框，一字排開，像五根柱子。

```
IRupa ─── IVedana ─── ISamjna ─── ISamskara ─── IVijnana
 色蘊       受蘊        想蘊        行蘊          識蘊
```

五個根介面。OpenStarry 的整座建築就站在這五根柱子上。五蘊不是裝飾——它們是結構。它們定義了 PluginHooks 的型別分類、PluginLoader 的註冊路由、ExecutionLoop 的階段順序。

問題是：五根柱子夠不夠？

## II. 三管齊下

D3-Q1 啟動了一場三管齊下的完備性證明。

**第一管：LINNAEUS 的功能覆蓋率。**

LINNAEUS 從 A1 研究報告中拉出了一張表。表格左列是 v0.28.0-alpha 的所有 plugin hook 型別，右列是蘊歸屬。他逐一驗證：

```
sensors (IVedanaSensor)          → 受蘊  ✓
recognizers (ISamjnaRecognizer)  → 想蘊  ✓
arbiters (IGearArbiter)          → 想蘊+識蘊  ✓
tools (ITool)                    → 行蘊  ✓
volition (IVolition)             → 識蘊  ✓
monitors (ILoopQualityMonitor)   → 受+想+識  ✓ (D2-R2)
auditor (IConfidenceAuditor)     → 識蘊  (待實作)
```

「功能覆蓋率 100%。」LINNAEUS 說。「沒有孤兒 hook。沒有無法分類的介面。每一個 plugin 型別都能歸入五蘊的一個或多個分類中。」

**第二管：BABBAGE 的型別代數。**

BABBAGE 推了推眼鏡。他不做類比。他做證明。

他在白板上寫了一行 TypeScript——不是程式碼，是型別表達式：

```typescript
type AllPluginTypes =
  | IVedanaSensor       // extends IVedana     → 受蘊
  | ISamjnaRecognizer   // extends ISamjna     → 想蘊
  | IGearArbiter        // ['samjna','vijnana'] → 想蘊+識蘊
  | ITool               // extends ISamskara   → 行蘊
  | IVolition           // extends IVijnana    → 識蘊
  | ILoopQualityMonitor // ['vedana','samjna','vijnana'] → 受+想+識
```

「五個根介面的聯集覆蓋了 PluginHooks 中所有可能的 plugin 型別。」他說。「這是型別代數的完備性——每一個具體型別都是五個根型別的子型別或交叉型別。不存在一個有意義的 plugin 型別落在五蘊聯集之外。」

他停了一下。

「不需要第六個根介面。」

**第三管：ASANGA 的阿毗達磨窮盡性。**

ASANGA 的論證從佛學出發。他的聲音帶著引經據典時特有的節奏——像在朗誦一段已經被背誦了兩千年的文字。

「《阿毗達磨俱舍論》卷一：『五蘊攝一切有為法盡。』」他說。「這是佛學的基本公理——五蘊涵蓋一切有為的（conditioned）現象。沒有第六蘊。如果有一個現象不能被歸入任何一蘊，那不是五蘊不完備——而是這個現象是無為的（unconditioned）。」

「OpenStarry 的 plugin 是有為的——它們被創造、被配置、被載入、被運行、被銷毀。它們不是無為法。因此，五蘊的分類應該能夠窮盡它們。」

他停頓了一下。然後以一個佛學家特有的謙遜補充：

「當然，佛學的公理不能直接用作工程的證明。但 LINNAEUS 和 BABBAGE 的證明已經從工程角度完成了。佛學的窮盡性論證是一個——一致性檢查。三個獨立的論證到達同一個結論，增強了結論的可信度。」

三個學科。分類學、數學、佛學。三條路徑。一個結論。

PENROSE 附加了一個警告——他總是附加警告。他在觀察席的最高處，俯瞰全場，像一個從更高維度觀察三維世界的存在。

「識蘊目前有最多的子介面——IVijnana、IGearArbiter、IVolition、IConfidenceAuditor。如果未來繼續膨脹，可能需要監控介面數量的增長。不是說五根介面不夠——而是說某些柱子可能需要額外的支撐。」

他的警告被記錄下來。但不影響投票。

## III. 快捷通道

D3-Q2 把放大鏡對準了映射的準確性。九個 hook 型別，逐一驗證蘊歸屬。

大部分的驗證是程序性的——LINNAEUS 在 A1 報告中已經做了詳細的映射表，TURING 驗證了所有的型別繼承關係。但有一個例外觸發了真正的討論。

ARCHIMEDES 舉手：「SlashCommand 屬於哪個蘊？」

這個問題看似微小。SlashCommand 是用戶直接輸入的命令——/help、/clear、/status。它不經過 ExecutionLoop。它不觸發 SparshEvent。它不經過 VedanaSensor、ISamjnaRecognizer、IGearArbiter、IVolition——它繞過了整個五蘊流程。

三個人同時回答了這個問題。從三個不同的角度。

KERNEL 從 OS 的角度：「SlashCommand bypass 了整個 ExecutionLoop。它不經歷五蘊流轉。類似 Unix 的 signal handler——它在進程級別被攔截，不進入正常的系統呼叫路徑。在 OS 的分類中，這不是用戶空間的邏輯，而是核心級的中斷處理。」

NAGARJUNA 從佛學的角度：「不入認知迴路的反射動作。它不是五蘊不涵蓋它——而是它不在五蘊運作的範圍內。就像呼吸——你可以觀察呼吸（進入認知迴路），但呼吸本身不需要五蘊流轉就能發生。SlashCommand 是系統的呼吸——它在認知之外，在認知之下。」

GUARDIAN 從安全的角度：「bypass 意味著 SlashCommand 不受 SafetyMonitor 保護。這是設計上的決定——某些操作需要繞過安全檢查——但應該被明確記錄在安全文件中。」

SUNYATA 把三個觀點合併成一個結論：「SlashCommand 是跨蘊基礎設施（cross-skandha infrastructure）。它不屬於任何一蘊。它的分類理由和安全觀察都需要在 Doc 45 中記錄。」

D3-R2：映射正確。20/20。

## IV. 標籤 vs 介面

D3-Q3 是一個精妙的問題：**skandha tag 應該用來做路由嗎？**

每個 plugin 都有一個 skandha 欄位——metadata。問題是：PluginLoader 在載入 plugin 時，是否應該用 skandha tag 來決定這個 plugin 去哪個 registry？

直覺上似乎合理——受蘊 plugin 去 VedanaRegistry，行蘊 plugin 去 ToolRegistry，像圖書館的索引卡。

但 VITRUVIUS 指出了問題。建築師的眼睛對結構的脆弱性特別敏感——他能在設計圖上看到未來可能裂開的那條縫。

「目前的系統用的是 duck typing。」他說。「plugin 實作了 IVedanaSensor 介面，就自動被 VedanaRegistry 接受。不需要標籤。如果改用標籤路由，就引入了一個新的出錯源：標籤和介面不一致。一個 plugin 宣稱自己是受蘊，但實作了 ITool 介面——怎麼辦？」

TURING 確認了事實。他打開了 `plugin-loader.ts`。

「v0.28.0-alpha 的 PluginLoader 使用 type guard 來分類 plugin。`isVedanaSensor(hook)` 檢查的是介面實作，不是 metadata。」

BABBAGE 用型別論做了終結性的陳述：「type guard 是可靠的——它在編譯期和運行時都能驗證。metadata 是不可靠的——它只能在運行時檢查，而且依賴人工正確標記。用不可靠的東西來做路由，在型別安全的系統中是倒退。」

D3-R3：skandha tag = metadata only，不用於路由。18/20。

MESH 和 GUARDIAN 投了少數票——他們認為 metadata 路由可以作為輔助驗證。但主流意見清晰：介面是真相，metadata 是備註。

## V. 最窄的柱子

D3-Q4 是最坦誠的一題。

**ISamskara 是否需要更多子介面？**

目前，行蘊只有一個子介面：ITool。IVolition 在 D2 之後已被確認歸入識蘊（Phase 5 行動前 vs Phase 8 行動中）。行蘊是五蘊中最「窄」的一個——佛學的行蘊涵蓋 49 心所（五十一心所中除受和想），而 OpenStarry 的行蘊只有 ITool。

ASANGA 站了起來。他在 D3 中最坦誠的一段話，像一個建築師指著自己設計的最薄的牆說「我知道這裡薄了」。

「我必須承認，OpenStarry 的行蘊設計與佛學傳統差異最大。」他說。聲音不帶辯解——只有事實。「傳統行蘊涵蓋 49 心所——意志、精進、慧、信、慚、愧——幾乎所有心理活動都被歸入行蘊。但 OpenStarry 把行蘊窄化為 ITool——外在行動。只有外在行動。」

HERACLITUS 從 ExecutionLoop 的動態角度提供了工程理由：

```
Phase 5: IVolition 審議 ──→ Phase 8: ITool 執行
          行動前                     行動中
          識蘊                       行蘊
```

「IVolition 在 Phase 5——行動之前。決定要做什麼。ITool 在 Phase 8——行動之中。正在做。兩者處於不同的階段。如果把 IVolition 移入行蘊，就會破壞蘊流轉的概念自然性——行蘊應該是『正在做的事』，不是『決定要做什麼事』。」

NAGARJUNA 做了一個佛學上的讓步。他的語氣帶著一種經過深思熟慮的平靜——不是妥協的平靜，而是理解的平靜。

「佛學的行蘊分類是基於修行者的內觀。」他說。「所有被觀察到的心理活動，除了受和想，都歸入行蘊。但 OpenStarry 不是修行者。它是一個軟體系統。軟體系統的分類應該基於功能，不是基於內觀。」

全票通過維持現狀。

D3-R4：ISamskara 不新增子介面。20/20。

但所有人都同意一件事：Doc 45——即將創建的五蘊 OOP 架構文件——必須記錄這個差異。行蘊的窄化不是一個錯誤，是一個有意識的偏離。有意識的偏離需要被解釋。

## VI. 兩條古路

**D3-Q5：十二因緣。**

「十二因緣是跨生命週期的。ExecutionLoop 是單一認知週期的。尺度差了——」他想了一下。「——幾個數量級。至少。」

BABBAGE 試圖建立結構態射（morphism）。他的方法是嚴格的——尋找兩個結構之間的保結構映射。如果存在態射，兩個結構在數學上是「同構的」或「同態的」，這意味著一個結構中的關係在另一個結構中有對應。

他失敗了。

「十二因緣之間的因果關係是線性的、不可跳過的。」他說。「無明必須在行之前。行必須在識之前。不能跳過。但 ExecutionLoop 的階段可以跳過——沒有 IVedanaSensor 的 Agent 會跳過受蘊階段。直接從事件接收跳到想蘊辨認。結構不同。不存在態射。」

```
觸 (sparsha) → 受 (vedana) → 愛 (tanha) → 取 (upadana)
```

「觸引起受。受引起渴愛。渴愛引起執取。這一段——不是全部，只是這一段——和 SparshEvent → ChannelVedana → KleshaGain → VolitionalDecision 的資料流有結構對應。」

投票反映了分歧。

D3-R5：選擇性 appendix。13/20。

7 票反對。反對者不是不同意局部對應的存在，而是質疑在工程文件中記錄它的必要性。

**D3-Q6：認知序列（citta-vīthi）。**

氣氛變了。

認知序列是上座部佛學對認知過程的精細分析——一個念頭從產生到消滅的完整過程：

```
有分心 → 轉向心 → 五識 → 接受心 → 推度心 → 確定心 → 速行心 → 彼所緣心
```

BABBAGE 重新嘗試結構態射。這次——

他成功了。

「認知序列和 ExecutionLoop 是同尺度的。」他說。他推了推眼鏡，那個他在做精確分析時必定會做的動作。「都是單一認知處理的階段序列。而且——」

他在白板上畫了兩個 FSM（有限狀態機）。並排的。像兩面平行的鏡子。

```
認知序列 FSM:
  有分心 → 轉向心 → 五識 → 接受心 → 推度心 → 確定心 → 速行心 → 彼所緣心
    (idle)  (alert)  (sense) (accept) (examine) (decide) (act)    (review)

ExecutionLoop FSM:
  Idle → EventReceived → Sensing → Recognizing → Arbitrating → Deliberating → Acting → Feedback
```

「存在結構態射。」BABBAGE 說。他的聲音帶著數學家發現同構時特有的滿足感——不是得意，是審美。「兩個 FSM 之間有保結構的映射。Idle ↔ 有分心。EventReceived ↔ 轉向心。Sensing ↔ 五識。每一個狀態轉換在兩個 FSM 中都有對應。」

「這不是比喻。」他強調。「這是數學。」

他的票從 D3-Q5 的反對（不記錄十二因緣）轉為 D3-Q6 的支持（記錄認知序列）。他在紀錄中寫了一行：「FSM 態射是我轉向的關鍵論據。十二因緣沒有態射。認知序列有。品質決定票數。」

D3-R6：選擇性 appendix。17/20。

從 13/20 到 17/20——四票的差距。四票的差距不是情緒——是精度。十二因緣的映射是近似的。認知序列的映射是精確的。BABBAGE 用他自己的數學工具衡量了兩者的品質，然後用他的票表達了結論。

## III. 維度

PASCAL 在他的筆記本上畫了一張圖。

Doc 38 是一份工程文件（IVolition 的架構設計），其中嵌入了一些佛學映射。三項測試適用。結論：清理嵌入的裝飾，保留驅動設計的概念。

Doc 16 是一份佛學映射文件（Plugin 組合模式的佛學對照），它的存在目的就是提供跨學科參考。三項測試不適用——因為它不是「工程文件中的佛學裝飾」，而是「佛學映射本身就是文件的內容」。

一面牆上的壁畫是裝飾——你可以移除它而不影響牆的結構。但一幅掛在美術館裡的畫不是牆的裝飾——畫就是內容本身。移除它不是「清理」，而是「拆除」。

## IV. 修正

SUNYATA 站在白板前。

## IV. 核融合爐

他轉向白板，寫了兩行：

```
般若（佛學）：照見一切法實相的最高智慧
IPrajna（工程）：對信心度加減 0.05 的函數
```

ASANGA 在他的座位上說了一句後來被所有人記住的話：

「這就像把一個溫度微調旋鈕叫做『核融合反應爐』。」

---

沒有人笑。

因為 ASANGA 不是在開玩笑。他是在用一個精確的類比說明一個精確的問題。

IPrajna 的介面定義是什麼？一個方法。輸入是信心度（confidence）和上下文（context）。輸出是 `{ delta: number, reasoning: string }`。delta 被限制在 [-0.05, +0.05] 的範圍內。

這是一個夾鉗。一個微調器。一個在已有信心度的基礎上做 ±0.05 修正的審計函數。

把它叫做 IPrajna——般若——照見一切法實相的最高智慧——

PASCAL 在筆記本上寫道：「從決策理論看，IPrajna 實際做的是 bounded secondary evaluation。輸入是一階信心度，輸出是二階修正量，有硬限 ±0.05。這是審計。不是般若。」

BABBAGE 接著分析：「IConfidenceAuditor 語義最精確。這個介面有一個方法，輸入是信心度加上下文，輸出是審計結果。Audit——審計——精確描述了這個操作：對已有判斷進行有限範圍的二次評估。」

---

**D4-R2 投票：IPrajna → IConfidenceAuditor。16/20。**

比 D4-R1 更高的共識。

少數意見：WIENER 偏好 IThresholdCalibrator（兩票）——他從控制理論的角度看到了「校準」而非「審計」。HERACLITUS 和 PENROSE 偏好 ISecondaryEvaluator（兩票）——他們認為「審計」暗示了事後檢查，但 IPrajna 是在決策流程中即時參與的。

BABBAGE 反駁了兩個少數意見：「Calibrator 暗示調整系統本身——但 IConfidenceAuditor 不改變系統，它只對一個數字提出有限修正。Evaluator 太通用——任何函數都在做 evaluation。Auditor 的語義邊界最精確：獨立的、有限範圍的、產生書面記錄的二次檢查。」

完整的更名表：

| 舊名稱 | 新名稱 |
|--------|--------|
| IPrajna | IConfidenceAuditor |
| IPrajnaAdjustment | ConfidenceAuditResult |
| prajnaAdjust() | clampAuditDelta() |
| Delta_prajna | Delta_audit |
| Model Delta 第二層名稱 | Confidence Audit |

### D5-R1：獨立的 `auditor` hook 槽位

### D5-R2：audit() 回傳類型

最激烈的 D5 投票。

選項：(A) 純非同步 `Promise<ConfidenceAuditResult>`；(B) 雙模式 `ConfidenceAuditResult | Promise<ConfidenceAuditResult>`。

GUARDIAN、KERNEL、VITRUVIUS 支持選項 A——純非同步。理由：audit 本質上是「問 LLM」，LLM 呼叫是非同步的。強制非同步語義更精確。

ATHENA 和 DARWIN 支持選項 B——雙模式。理由：遵循 IGearArbiter 的先例。測試時可以用同步的 noop 實作，不需要 async/await 的樣板程式碼。

TURING 提供了事實：「v0.28.0-alpha 中，IGearArbiter.evaluate() 使用雙模式回傳。IVolition.deliberatePlan() 和 deliberateAction() 也使用雙模式。這是現有的設計模式。偏離它需要充分理由。」

**5/8 選項 B 通過。** D5 中最接近的投票。

### D5-R3：timeout 和 fail-safe

TURING 打開了他的 timeout 模式分析。v0.28.0-alpha 中現有的 timeout：

| 組件 | timeout | 預設值 |
|------|---------|--------|
| IProvider.chat() | LLM 回應 | 120s |
| IGearArbiter.evaluate() | 每個 arbiter | 100ms |
| IGearArbiter chain | 整條鏈 | 200ms |
| ITool.execute() | 工具執行 | 30s |

audit() 的 timeout 應該是多少？

TURING 的建議：200ms——與 IGearArbiter chain deadline 一致。理由：audit() 發生在 confidence 計算的最後階段，在 IGearArbiter 之後。如果 audit 的 timeout 超過 chain deadline，整個決策流程會被 audit 卡住。

fail-safe：`{ delta: 0, reasoning: "audit timeout" }`。timeout 時 delta 為零——不修正。使用 `Promise.race()` 模式，與現有程式碼一致。

Configurable：通過 agent.json 覆蓋預設值。

ILoopQualityMonitor 的 `start(bus)` 在哪裡被呼叫？

TURING 的分析：SafetyMonitor 在 `ExecutionLoop.onLoopStart()` 啟動。MonitorRegistry 應該遵循相同的時機。

```
啟動：MonitorRegistry.startAll(bus) ← ExecutionLoop.onLoopStart()
關閉：MonitorRegistry.stopAll()    ← PluginLoader.disposeAll()
```

DARWIN 偏好在 PluginLoader 中啟動（更簡單），但接受了 monitors 有明確的 start/stop 語義，所以 ExecutionLoop 是更合適的位置。

**7/8 通過。**

## I. 兩種語言

Cycle 02-5 的核心不是某個具體的決議。不是 ILoopQualityMonitor，不是 IConfidenceAuditor，不是 Doc 45 的五蘊 OOP 架構。

核心是一個問題的答案。

問題是：在一個同時使用佛學和工程兩種語言的系統中，這兩種語言的邊界在哪裡？

D1 到 D5 用了六個半小時回答這個問題。答案可以濃縮成幾行：

**佛學語言適合用在三個地方：**

1. **程式碼識別符**——當佛學術語已經成為原始碼的一部分，並且功能和定義一致。Moha 是自我迷惑——Moha 模組確實在模擬自我迷惑。skandha 是蘊——程式碼中的 Skandha 型別確實在分類五蘊。這些名字通過了 Test 4。天平平衡。

2. **設計理由**——當佛學概念實際驅動了工程決策，並且有 DC 確認。「vasana 是後天薰習」驅動了 VasanaEngine 的外部化決策。「四煩惱同時俱起」驅動了 MulaKleshaBundle 的原子性設計。佛學概念和工程決策之間有因果關係。這些不是裝飾——是理由。

3. **獨立映射文件**——當文件的存在目的就是做佛學到工程的系統性對照。Doc 16 和 Doc 31 不是在工程文件上貼佛學標籤——它們本身就是映射文件。Master 裁定保留。映射文件不適用三項測試——因為它們不是工程文件中的佛學裝飾，而是佛學映射本身就是內容。

**工程語言適合用在其他所有地方。**

硬性規則不需要叫 sila。品質監控器不需要叫 Sati。信心度審計不需要叫 Prajna。event-driven 不等於 mindfulness。三學不等於五層閾值模型。

佛學語言是深刻的。工程語言是精確的。在需要深度的時候用佛學語言——附錄、學術參考、設計背景。在需要精度的時候用工程語言——介面定義、規格文件、安全架構。

兩種語言。兩個用途。一條邊界。
