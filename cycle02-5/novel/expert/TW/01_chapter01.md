# 第一章：審計員

## I. 五十個標本

R1 獨立研究啟動後的第一天，ARCHIMEDES 把自己關在了工作間裡。

他面前攤著七份文件。不是研究報告——是 openstarry_doc 的七份原始文件。Doc 38。Doc 43。Doc 44。Doc 37。Doc 41。Doc 16。Doc 31。每一份都被他用不同顏色的便利貼標記過：黃色是「疑似裝飾」，藍色是「需要交叉驗證」，粉紅色是「程式碼識別符」，綠色是「DC 確認」。

B1 審計。逐文件。逐段落。逐映射。

SCRIBE 坐在他旁邊。SCRIBE 的工作不是審計——是記錄。記錄 ARCHIMEDES 的每一個判斷、每一個猶豫、每一個在黃色和藍色之間搖擺的瞬間。

「Doc 38。」ARCHIMEDES 翻開第一份文件。「Klesha Gain Scheduling。L540-544。」

他的手指停在一張表格上。三欄：工程概念、佛學映射、說明。śīla（教誡）= Hard Rules。upāya（善巧）= Soft Rules / Adaptive。smṛti（正念）= Monitoring / Event-driven。

「三項測試。」他喃喃自語。然後開始逐行應用。

**Test 1**（必要性）：移除「佛學映射」這一欄，Hard Rules 還是 Hard Rules 嗎？是的。Soft Rules 還是 Soft Rules 嗎？是的。工程規格完全不變。

→ 「佛學映射」欄位：**REMOVE**。

SCRIBE 記下了。第一個標本。

「等一下。」ARCHIMEDES 回過頭。「工程概念」欄和「說明」欄是有用的——只有「佛學映射」欄是裝飾性的。不是移除整張表。是移除一欄。」

精細分離。這是 ARCHIMEDES 的特長——他不是一個用大鉈砸核桃的人。他是一個用手術刀分離組織的外科醫生。

他繼續往下翻。

---

Doc 43。SatiMonitor——不，他在心裡修正了自己——那個「觀察行為模式」的 plugin。整份文件的標題帶著「正念」的標記。文件裡散落著 event-driven = 正念的隱喻、品質度量被描述為「覺察的量化」、四階段退化策略被比喻為「正念的深淺層次」。

ARCHIMEDES 開始計數。

第一個：「event-driven 架構是正念的工程實現」——Test 1，移除後規格不變。REMOVE。
第二個：「SatiQualityVector 量化了覺察的四個維度」——Test 1，移除後 SatiQualityVector 的功能不變。REMOVE。
第三個：「四階段退化策略反映了正念從初學到熟練的層次」——Test 1，明顯裝飾。REMOVE。

他一個一個地標記。黃色便利貼越來越多。

「這份文件。」他對 SCRIBE 說。「工程內容是紮實的——事件訂閱、滑動窗口、品質指標、退化策略——全部有技術深度。但佛學裝飾像一層薄霜，覆蓋在每一個工程概念上。如果你不熟悉佛學，這些裝飾會讓你困惑。如果你熟悉佛學，這些裝飾會讓你覺得被冒犯。」

SCRIBE 把這段話一字不差地記下了。後來，這段話被引用了不下三次——在 D1 辯論中、在 D4 延長討論中、在 Master 的審閱回覆中。

---

Doc 44。Doc 37。Doc 41。每一份文件，ARCHIMEDES 都用同樣的方法：三項測試，逐映射應用，精細分離工程內容和佛學裝飾。

SCRIBE 在旁邊統計。

到了第一天結束的時候，B1 審計報告的初稿已經成形。50 個映射實例。每一個都有出處（文件名 + 行號）、處置建議（KEEP / RELOCATE / REMOVE）、和理由（哪一項測試、哪一個結論）。

| 處置 | 數量 | 範例 |
|------|------|------|
| **KEEP** | 23 | Skandha 型別名、Klesha 模組名（Moha/Drishti/Mana/Sneha）、DC 確認約束、CoarisingBundle |
| **RELOCATE** | 13 | PASCAL 數學形式化、MN 18 經典引用、DD-13 佛學背景、各處佛學設計理由 |
| **REMOVE** | 14 | śīla/upāya/smṛti 標籤、event-driven=正念、戒定慧映射、種子理論裝飾 |

ARCHIMEDES 看著這張表。23 + 13 + 14 = 50。

「23 個 KEEP。」他說。「將近一半。」

SCRIBE 點了點頭。「不是所有佛學內容都是裝飾。有些是身份——像 Klesha 模組名。有些是決策——像五蘊分類驅動了 PluginHooks 的設計。」

「但 14 個 REMOVE。」ARCHIMEDES 繼續。「將近三分之一。這些是純粹的裝飾——移除它們不會改變任何工程規格，但會顯著降低閱讀障礙。」

他把報告存檔。`B1_forced_mapping_audit.md`。然後他打開了另一個文件——Doc 16 和 Doc 31。

## II. 兩份可疑文件

Doc 16 和 Doc 31 是不同的。

前面審計的七份文件——Doc 38、43、44、37、41——都是工程文件。它們的主要內容是工程規格，佛學映射是附加的。就像一本食譜書裡夾了幾張宗教節日的插圖——食譜是主體，插圖是裝飾。

但 Doc 16（Plugin 類型哲學映射）和 Doc 31（八識運行時模型）不一樣。

ARCHIMEDES 翻開 Doc 16。從第一頁開始讀。然後翻回封面。然後再讀第一頁。

「SCRIBE。」他說。「這份文件的裝飾比例大概是多少？」

SCRIBE 快速掃描了一遍。「大約 80%。」

80%。意思是：Doc 16 中，只有大約 20% 的內容是純工程規格（主要是 Observer Pattern 的討論），其餘 80% 是佛學概念到軟體模式的哲學映射。

Doc 31 好一些，但也有大約 70% 的佛學內容。八識如何映射到運行時模型——阿賴耶識是種子庫，末那識是自我執著——這些映射有學術價值，但放在工程文件中，它們的工程效用有限。

ARCHIMEDES 在 Doc 16 和 Doc 31 上貼了一個特殊的標籤——不是黃色（裝飾）、不是粉紅色（識別符）、不是綠色（DC 確認）。是紅色。

紅色 = **FILE REVIEW**。整份文件需要評估。

「這兩份文件的問題不在於個別映射。」他對 SCRIBE 說。「問題在於整份文件的定位。它們是工程文件中嵌入了佛學？還是以佛學映射為目的的獨立文件？」

SCRIBE 記下了這個問題。這個問題後來在 D1-Q7 和 D1-Q8 中被正式投票——然後在 D4 中被 Master 親手推翻。

## III. 三個維度

B1 審計是事實的蒐集。B2 是原則的建構。

在 ARCHIMEDES 數標本的同時，NAGARJUNA、ASANGA 和 PASCAL 在另一間工作室裡建構映射邊界原則。三個人。三個學科。三個維度。

NAGARJUNA 坐在靠窗的位置。窗外是什麼不重要——對中觀學者來說，窗外的景色和窗內的思考之間沒有本質的區別，它們都是世俗諦的顯現。重要的是他面前的白板。白板上畫著一條水平線。左邊寫著「世俗諦」，右邊寫著「勝義諦」。

「二諦。」他說。他的聲音在小工作室裡聽起來比在劇場裡更近、更低。「龍樹《中論》第二十四品：諸佛依二諦，為眾生說法。一以世俗諦，二第一義諦。」

PASCAL 在筆記本上記下了引用。他不是在做佛學研究——他是在尋找公理。每一個理論體系都有它的公理，PASCAL 的工作是找到它們，然後檢驗從這些公理出發能否到達他們需要的結論。

「世俗諦語言 = 工程術語。」NAGARJUNA 在白板上寫。「用於日常溝通和實作。if/else、interface、extends——這些是世俗諦的語言。它們不需要佛學的加持就能工作。」

「勝義諦洞見 = 佛學概念。」他在右邊寫。「用於啟發和設計理由。五蘊的分類啟發了根介面的設計。煩惱的分類啟發了 Klesha 的模組結構。但啟發不等於標籤。」

他轉身面對 PASCAL 和 ASANGA。

「兩者不應混淆。不要在世俗諦的文件中硬塞勝義諦的標籤。」

PASCAL 點了點頭。然後他打開了他的維度。

---

PASCAL 的白板在 NAGARJUNA 旁邊。他畫了一個二維矩陣。

```
                 映射有價值        映射無價值
保留映射         True Positive     False Include ← 累積損害
移除映射         False Exclude     True Negative
                 ↑ 一次性成本
```

「這是標準的混淆矩陣。」他說。「但關鍵不在矩陣本身——在損害的不對稱性。」

他在 False Include 旁邊畫了一個向上的箭頭。

「False Include——保留了一個裝飾性映射。它的損害是什麼？每一位未來的讀者——可能是幾十人，可能是幾百人——每一次閱讀這份文件——可能一次，可能十次——每一頁上出現的佛學標籤——可能三個，可能十個——每一次，讀者都需要在腦中維護兩套術語系統。śīla 是什麼？噢，就是 Hard Rules。那為什麼不直接叫 Hard Rules？」

他寫了一個公式：

$$C_{FI} = N_{readers} \times R_{reads} \times P_{pages} \times \tau_{cognitive}$$

「累積損害。讀者數 × 閱讀次數 × 頁數 × 每次的認知轉換成本。這是一個乘法——每一項都是正的，乘積只會增長。」

然後他在 False Exclude 旁邊畫了一個小得多的箭頭。

「False Exclude——移除了一個有價值的映射。損害是什麼？某一天，某個人需要知道五蘊分類的佛學背景。他搜尋附錄。找到了。一次性的搜尋成本。不累積。不乘以讀者數。不乘以閱讀次數。」

$$C_{FE} = S_{search} \quad \text{(一次性)}$$

「結論。」他畫了一個大大的不等號。

$$E[C_{FI}] \gg E[C_{FE}]$$

「期望值分析：裝飾性映射具有負期望值。有疑慮時，傾向 REMOVE 或 RELOCATE。」

---

ASANGA 一直在聽。他沒有白板——他的思考工具是語言，不是圖形。

「功能性判定。」他說。聲音帶著唯識學特有的精密度——像在顯微鏡下調焦。「唯識學有一個核心判準：因果效力（arthakriyā-sāmarthya）。一個概念是否具有因果效力——它是否驅動了一個結果？」

他站了起來。走到 NAGARJUNA 和 PASCAL 的白板之間。

「佛學概念驅動了五蘊根介面的設計——這有因果效力。五蘊分類 → PluginHooks 的五個類別。因果鏈明確。保留。」

「佛學標籤被貼在 Hard Rules 上叫 śīla——這沒有因果效力。Hard Rules 不是因為被叫做 śīla 才有硬性約束——它的硬性約束來自 SafetyMonitor 的實作。śīla 這個標籤不改變任何因果鏈。移除。」

他在空中用手畫了一條線——從因到果。

「判斷標準很簡單：**如果移除這個佛學概念，因果鏈是否斷裂？** 是 → 保留。否 → 移除。」

PASCAL 抬起頭。他看到了三個維度的交匯：

```
NAGARJUNA — 二諦（範疇分析）
            世俗諦 vs 勝義諦 → 不要混淆兩種語言

PASCAL    — 損害不對稱（決策理論）
            E[累積損害] >> E[搜尋成本] → 有疑慮時移除

ASANGA    — 因果效力（唯識學）
            有因果效力 → 保留 / 無因果效力 → 移除
```

三個學科。哲學、數學、佛學。三個公理體系。三條推理路徑。

一個結論。

## IV. TURING 的地毯式掃描

R2 交叉審閱。

TURING 不說話。或者說，TURING 說的話少到可以被完整列舉。在 Cycle 02-5 的 R2 階段，他在所有記錄中一共說了七句話。

但他做的事情比任何人都多。

九份獨立研究報告。每一份都引用了 v0.28.0-alpha 的原始碼——`aggregates.ts`、`plugin-loader.ts`、`loop.ts`、`plugin.ts`、`safety-monitor.ts`——每一個引用都需要被驗證：這段程式碼確實在那個位置嗎？引用的行為描述準確嗎？介面定義引用正確嗎？

TURING 做了地毯式掃描。40 個以上的程式碼引用。逐一打開原始碼。逐一比對。逐一確認或標記。

他的終端機上，`rg`（ripgrep）指令一個接一個地執行。搜尋 `IVedanaSensor`——確認。搜尋 `IGearArbiter`——確認。搜尋 `ILoopQualityMonitor`——不存在（尚未實作）。搜尋 `IInferenceProvider`——

他停住了。

A5 報告（跨蘊互動分析）的附錄 B 列出了 `IInferenceProvider` 作為想蘊的子介面之一。TURING 在 v0.28.0-alpha 的整個程式碼庫中搜尋了這個名稱。

零結果。

他在審閱表格的 T-01 欄中寫了一行：「A5 附錄 B 列出 IInferenceProvider 但原始碼中未找到。中等嚴重度。」然後繼續掃描。

T-02：A4 報告中 Phase 7 的描述可能誤導讀者認為 VedanaSensor 已完全運作——實際上 VedanaRegistry 已建立但消費者尚未實作。低嚴重度。

T-03：B1 審計報告中 Doc 44 L174 的交叉檢查自我修正段落需要清理。最低嚴重度。

T-04：A1 和 C2 報告中 PluginHooks 的欄位名稱不一致——`gearArbiters` vs `arbiters`。TURING 查了原始碼：正確名稱是 `arbiters`。中等嚴重度。

四個問題。40 個以上的引用。四個問題。這是一個 10% 以下的錯誤率——在學術論文中算是非常乾淨的。

TURING 在審閱矩陣的最後一欄寫了一個字：

**PASS。**

他合上了筆記本。然後他說了他在 R2 階段的第三句話。

「引用乾淨。可以進入辯論。」

六個字。像他的程式碼驗證一樣簡潔。

## V. 二十四個共識，七個問題

R2 的審閱矩陣不只是 TURING 的程式碼驗證。二十位研究員組成了交叉審閱網——每人審閱至少兩份非本人主筆的文件。

結果令人欣慰，也令人不安。

令人欣慰的是：24 個共識點被識別。跨三條軌道——A-track 的 8 個（五蘊根介面足夠、DI 鏈完整、Plugin 生命週期清晰等）、B-track 的 5 個（四層框架有效、三項測試可操作等）、C-track 的 7 個（SatiMonitor 功能明確、不屬於純行蘊等）、加上 4 個全域共識（行蘊窄化需記錄、SlashCommand bypass 確認等）。

令人不安的是：7 個開放問題。其中 OQ-1（SatiMonitor 蘊組成）是核心辯論議題。4 個分歧點——最尖銳的是 D-1：受蘊「厚」（包含 SatiMonitor 的感知層）還是「薄」（SatiMonitor 的感知層不算受蘊）。

SUNYATA 在 R2 結束後花了三十分鐘整理這些數據。24 個共識意味著大部分的基礎是穩固的——辯論不需要從零開始。7 個問題意味著辯論有明確的焦點——不需要漫無目的地討論。4 個分歧意味著辯論會有火花——不是客氣的點頭，是真正的交鋒。

他在白板上重新寫了三場辯論的議程。

**D1**：佛學映射邊界。核心問題：openstarry_doc 工程文件中佛學內容的原則是什麼？

**D2**：Sati Plugin 五蘊分類。核心問題：這個 plugin 由五蘊的哪些類別組成？

**D3**：五蘊 OOP 架構完整性。核心問題：五蘊架構是否足以支撐工程實作？

然後他在議程下面加了一行小字：

> *R1 九份研究 → R2 二十四共識 + 七問題 + 四分歧 → R3 三場辯論*

管道是通的。數據是乾淨的。工具是校準的。

手術室準備好了。
