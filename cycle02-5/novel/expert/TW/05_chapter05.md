# 第五章：裁定

---

D3 結束四十分鐘後。

圓形劇場的冷白燈光下，二十位研究者在做一件奇怪的事——他們什麼都沒做。

D1 到 D3 之間的每個間隔都被塞滿了：交叉審閱、數據比對、投影片準備、咖啡。但 D3 之後的空白是不同的。不是休息。是等待。因為每個人都知道接下來會發生什麼——Master 的審閱。

在前幾輪 Cycle 中，Master 的介入通常發生在特定的時間節點：研究開始前（信件）、某個決議引發爭議後（DC-1 到 DC-12）、或者整輪結束時（路線圖建議）。但 Cycle 02-5 不一樣。Master 在信中明確說了他要看什麼——清理結果。他要看的不是抽象原則——他要看具體執行。

---

## I. 審閱

Master 的審閱不是一份書面報告。

它以一種更直接的方式出現。SUNYATA 在他的終端上看到了一段話，然後他把投影幕打開，讓所有人都看到。

審閱的篇幅不長。但它精確得像手術刀。

Master 首先確認了大部分 D1 決議——四層框架、三項測試、Batch A 的五項 REMOVE、Batch B 的八項 RELOCATE、Batch C 的七項 KEEP、Doc 38 表格清理、Doc 44 三學映射移除、Klesha 模組名保留。

「這些都正確。」

然後他提到了兩個名字：Doc 16 和 Doc 31。

---

## II. 推翻

「D1-Q7 和 D1-Q8，」SUNYATA 唸出 Master 的原話，「這兩項決議我不同意。」

劇場裡的空氣凝固了。

D1-Q7：Doc 16 的拆分——將 Section 5 提取為獨立文件，其餘刪除。20/20 通過。

D1-Q8：Doc 31 的拆分降級——Section 3.1 移入 Doc 35，Section 6.3 獨立為短文件，其餘降級為 Appendix_B。20/20 通過。

兩個 20/20。兩個全票。在本輪歷史上最乾淨的辯論中以完美的共識通過。

Master 推翻了它們。

---

SUNYATA 繼續唸：

「Doc 16 不是一份嵌入了佛學的工程文件。Doc 16 本身就是一份佛學映射文件——它的存在目的就是做佛學到工程的系統性對照。你們的三項測試是正確的，四層框架也是正確的，但你們在 Doc 16 上用錯了標準。」

ARCHIMEDES 低頭看他的審計表。他在 B1 中給 Doc 16 標記了「~80% 裝飾性」。80%——他數過。一行一行數的。但數字正確不代表結論正確。

「Doc 31 也一樣，」Master 繼續，「八識運作模型是一份以佛學概念為主體的分析性文件。它不是工程文件裡的裝飾——它就是一份佛學參考文件。你們需要區分兩種情況：第一種是工程文件中嵌入的佛學裝飾——應該清理。第二種是以佛學映射為目的的獨立文件——它有工程參考價值，應該保留。」

---

NAGARJUNA 是第一個站起來的。

「Master 指出了我們框架的盲點，」他說。他的聲音沒有顫抖。中觀學者被告知自己的分析框架有缺口——這在他的傳統中不是恥辱，而是日常。龍樹本人的全部哲學建立在「一切見皆可遣除」之上——包括自己的見。

「三項測試問的是：映射是否改變工程規格？是否是程式碼識別符？是否驅動設計決策？但這三個問題假設了一個前提——被審查的文件是工程文件。如果文件本身就是佛學映射文件，那麼用工程文件的標準來衡量它——」

「就像用散文比例來評判詩，」SCRIBE 在記錄中寫道。

---

## III. 維度

PASCAL 在他的筆記本上畫了一張圖。

三項測試是三個軸——Necessity、Code Identity、Decision Driver。每一份文件在這三個軸上有一個坐標。D1 的判斷基於這個三維空間。

但 Master 指出的是第四個軸：**文件的存在目的**。

Doc 38 是一份工程文件（IVolition 的架構設計），其中嵌入了一些佛學映射。三項測試適用。結論：清理嵌入的裝飾，保留驅動設計的概念。

Doc 16 是一份佛學映射文件（Plugin 組合模式的佛學對照），它的存在目的就是提供跨學科參考。三項測試不適用——因為它不是「工程文件中的佛學裝飾」，而是「佛學映射本身就是文件的內容」。

差別在於：裝飾和內容。

一面牆上的壁畫是裝飾——你可以移除它而不影響牆的結構。但一幅掛在美術館裡的畫不是牆的裝飾——畫就是內容本身。移除它不是「清理」，而是「拆除」。

「Doc 16 和 Doc 31 是掛在美術館裡的畫，」PASCAL 說，「不是貼在工程圖紙上的壁畫。」

---

ARCHIMEDES 重新打開他的審計表。

「B1 審計的前提假設是：所有被審計的文件都是工程文件，其中可能嵌入了佛學裝飾。我按照這個前提逐行審計，標記裝飾比例。Doc 16 的 80% 和 Doc 31 的 70%——數字是對的。但前提是錯的。」

他在審計表上加了一個新的欄位：**文件類型**。

| 文件 | 類型 | 三項測試適用？ |
|------|------|----------------|
| Doc 38 | 工程文件 | 是 |
| Doc 43 | 工程文件 | 是 |
| Doc 44 | 工程文件 | 是 |
| Doc 37 | 工程文件 | 是 |
| Doc 41 | 設計決策文件 | 是 |
| **Doc 16** | **佛學映射文件** | **否** |
| **Doc 31** | **佛學映射文件** | **否** |

七份文件。五份工程文件。一份設計決策文件。兩份佛學映射文件。

前五份適用三項測試。第六份因為記錄設計理由而需要特殊處理。最後兩份——不是清理的對象。

---

## IV. 修正

SUNYATA 站在白板前。

「團隊的回應，」他說。

不需要投票。Master 的裁定不是議案——它是裁定。但 SUNYATA 想確認每個人都理解了原因，而不只是結果。

NAGARJUNA 先說：「中觀的兩諦框架——世俗諦和勝義諦——我用它建議將工程文件中的佛學內容分離到附錄。但 Doc 16 和 Doc 31 不是世俗諦文件中的勝義諦注釋——它們本身就是在做兩諦之間的系統性映射。映射文件不屬於任何一個層面。它們是連接兩個層面的橋樑。刪除橋樑不是清理——是破壞。」

ASANGA 接著說：「我用唯識學的功能定位法——如果佛學概念沒有驅動工程決策，它就是裝飾。但 Doc 16 整份文件的功能就是提供佛學到工程的映射。用『驅動工程決策』來評判一份映射文件，就像用『是否能打釘子』來評判一面鏡子——標準選錯了。」

TURING 說：「從原始碼驗證的角度，Doc 16 的 Section 5（IObserver 組合模式）確實有直接的工程價值。但 Master 的裁定不是說 Section 5 應該被提取——而是說整份文件都應該保留。這意味著其餘部分的學術參考價值也被認定了。」

GUARDIAN 說：「安全角度——Doc 31 的八識運作模型包含 ConcurrentSenseProcessing 的分析，這對多通道安全審計有參考意義。我在 D1 時沒有反對拆分，因為我同意 Cocycle condition 可以獨立。但 Master 的裁定讓我重新考慮：拆分之後，上下文消失了。Cocycle condition 在八識框架中的意義和在獨立文件中的意義不一樣。」

---

SUNYATA 在白板上寫下了修正：

```
D1-Q7 (Doc 16): 20/20 拆分 → Master 裁定恢復原狀
D1-Q8 (Doc 31): 20/20 拆分 → Master 裁定恢復原狀
理由：佛學映射文件 ≠ 工程文件，不適用三項測試
```

然後他在下面加了一行：

```
四層框架更新：
KEEP / RELOCATE / REMOVE / FILE REVIEW → 加入前提檢查
前提：文件類型識別（工程文件 / 設計決策 / 佛學映射）
```

三項測試沒有被否定。它們只是獲得了一個適用條件。就像一把精密的卡尺——它能測量任何零件的尺寸，但你首先得確認你測量的是零件，而不是設計圖紙本身。

---

## V. 種子

SCRIBE 在記錄中寫下了這一段。

D1 的 SCRIBE 旁白中有一段話：「D1 有一個隱藏的種子。D1-Q7 和 D1-Q8 的拆分決策——後來，Master 親手推翻了這兩項決議。理由不是團隊的判斷錯了，而是團隊的判斷框架還少了一個維度。」

種子現在發芽了。

但 SCRIBE 知道——Master 的審閱不只是推翻兩項決議。Master 審閱的真正重量在於他沒有展開的那句話——

「用梵文就需要對他的定義負責。」

這句話不在 Doc 16/31 的討論中。它出現在 Master 審閱的另一部分。它指向一個不同的方向。它指向 D4。

---

D3 結束時，SUNYATA 在白板上列出了三個「已知缺口」。其中兩個名字是 IPrajna 和 ISatiMonitor。白板上的文字不帶情緒。它們只是「尚未實作」的介面名稱。

但 Master 看到了白板上的名字——或者更準確地說，他看到了名字背後的問題。

「ISatiMonitor，」他寫道，「你覺得 Sati 的內容完全 match 嗎？它歸在哪一蘊？」

這不是一個新問題。D2 已經回答了蘊歸屬——[vedana, samjna, vijnana]。三蘊。不包含行蘊。

但 Master 問的不是蘊歸屬。他問的是名字。

Sati 是巴利文的「正念」。正念在佛學中是行蘊的心所——它是主動的、有意志的、道德正向的修行活動。但 D2 剛剛證明了 SatiMonitor 不是行蘊——它是被動的、自動的、價值中立的品質監控器。

如果 SatiMonitor 不是行蘊活動——那它就不是正念——那它為什麼叫 Sati？

D2 的 SCRIBE 旁白預言了這一刻：「名字的問題，要等 Master 親自來問。」

Master 問了。

---

## VI. 過渡

SUNYATA 環顧劇場。

二十位研究者。D1 到 D3 已經消耗了大約四個半小時。十九項投票。整個佛學映射體系被審計、分類、部分拆除。五蘊 OOP 架構通過了完備性驗證。Sati Plugin 被重新歸類為三蘊組合。

這些都是預定議程上的工作。R0 定向時設計的三條研究軌道——Track A（五蘊架構）、Track B（映射審計）、Track C（Sati 重分類）——全部完成。

但 Master 的審閱打開了一扇計畫外的門。

不是一個新問題——而是一個老問題的新角度。SatiMonitor 的蘊歸屬解決了。映射邊界原則建立了。架構驗證通過了。但名字——名字的問題一直在那裡，像一條地下水脈，在所有辯論的表面之下流淌，只是沒有人挖一口井把它引出來。

直到 Master 用一句話挖了那口井：

「用梵文就需要對他的定義負責。」

SUNYATA 轉向團隊。

「我們需要一場計畫外的辯論，」他說。「D4。議題：命名修正。」

他看了一眼 NAGARJUNA。哲學家的表情是平靜的——但那是一種中觀學者在自己的論證被推向邏輯極端時特有的平靜。他知道接下來會發生什麼。因為他自己在 D2 中種下了那顆種子——

如果 SatiMonitor 不是行蘊活動，那它就不是 Sati。

那個推論在 D2 結束時被記錄了。但沒有人把它拉到終點。

現在，Master 替他們拉了。

---

> *SCRIBE 旁白*

> *Master 的裁定有兩個層次。*

> *表面的層次是兩項決議的推翻。D1-Q7 和 D1-Q8——Doc 16 和 Doc 31 恢復原狀。這是具體的、可執行的、明確的。團隊錯了，Master 修正了，修正被接受了。在形式上，這和 Cycle 02-3 的 DC-10 或 02-4 的 DC-6 沒有區別——Master 有最終裁定權，他行使了這個權力。*

> *但深層的層次是一個新的維度。三項測試和四層框架是 Cycle 02-5 的基礎設施——它們在 R0 被提出，在 R1 被三個學科論證，在 R2 以零異議通過，在 D1 以十個 20/20 被使用。它們是有效的。但它們是不完整的。*

> *不完整不是錯誤。不完整是——尚未完成。*

> *Master 加入了一個前提檢查：在應用三項測試之前，先識別文件類型。工程文件用三項測試。映射文件用不同的標準。設計決策文件需要特殊處理。這不是推翻三項測試——這是為三項測試增加一個入口條件。*

> *就像一把卡尺——它的精度沒有問題，但你得先確認你量的是零件，不是設計圖紙。*

> *但裁定的真正重量不在 Doc 16 和 Doc 31。*

> *裁定的真正重量在那句話：「用梵文就需要對他的定義負責。」*

> *這句話像一顆石頭丟進了池塘。D1 到 D3 的所有討論是池塘的水面。石頭沉下去了。漣漪還沒有擴散到邊緣。*

> *但它會擴散。在下一場辯論中——D4——漣漪會觸及每一個用梵文命名的介面。ISatiMonitor。IPrajna。戒慧安全框架。*

> *三項測試問「這個映射是否有工程價值？」*

> *Master 問的是「這個名字是否對得起它的定義？」*

> *不同的問題。更深的問題。更危險的問題。*

> *因為工程價值可以被衡量——通過測試、程式碼驗證、設計影響追溯。但定義的忠實度是一個哲學問題。它問的不是「這個名字有用嗎？」——它問的是「這個名字誠實嗎？」*

> *這架天平——序章中出現的天平——現在被校準了。不是用工程價值校準。是用定義的忠實度校準。*

> *接下來會發生什麼，取決於天平兩端放上去的東西。*

> *一端是名字。另一端是定義。*

> *如果兩端平衡——名字保留。*

> *如果兩端不平衡——*

> *D4 會告訴我們答案。*
