# 第六章：名字的代價

---

劇場裡的燈光沒有變。還是那個 6500K 的冷白色。但不知道為什麼，空氣的溫度變了。

不是物理溫度。是一種感覺——像考試開始前的教室，或者法庭宣判前的等待。每個人都坐在自己的位子上，但沒有人真的在休息。

SUNYATA 走到投影幕前，把 Master 的那句話打在屏幕上。

大字。白底黑字。沒有格式。

> **「用梵文就需要對他的定義負責。你覺得 Sati 的內容完全 match 嗎？它歸在哪一蘊？」**

---

## I. 歸謬

NAGARJUNA 站了起來。

在整個 Cycle 02-5 中，他已經站起來過很多次——D1 中承認「後貼標籤」，D3 中分析十二因緣的尺度不匹配。每一次他都保持著中觀學者特有的端正姿態——承認錯誤是學術的核心技能，他從不迴避。

但這一次不一樣。

這一次他不是在承認錯誤。他是在完成一個推論。

「D2 已經確定了 SatiMonitor 的蘊歸屬，」他說，聲音比平常低半個音調。不是因為緊張——是因為他在組織一個他已經想了很久但一直沒有說出口的論證。

「結論是 [vedana, samjna, vijnana]。三蘊。不包含行蘊。」

他頓了一下。

「Sati——巴利文的正念——在佛學傳統中是行蘊的心所。它是 cetasika，是有意志的、主動的、道德正向的修行活動。阿毘達磨把它放在行蘊，因為正念是一種修行——你必須帶著意圖去做的事情。」

他轉向 ASANGA。ASANGA 微微點頭。他知道 NAGARJUNA 要說什麼。

「現在，」NAGARJUNA 說，「讓我做一個歸謬論證——prasaṅga。這是中觀學的基本方法。」

他在白板上寫了兩行字：

```
前提 A：Sati = 行蘊心所（佛學定義）
前提 B：SatiMonitor ≠ 行蘊（D2-R2 結論）
```

然後他在下面寫了第三行：

```
∴ SatiMonitor ≠ Sati
```

「如果正念（Sati）在佛學中必然是行蘊的心所，而我們已經證明了 SatiMonitor 不是行蘊——那麼 SatiMonitor 就不是正念。」

他放下筆。

「一個不是正念的東西，為什麼叫 Sati？」

---

沉默。

不是不同意的沉默。是那種——所有人同時理解了一件他們應該更早理解的事情——的沉默。

D2 的 SCRIBE 旁白曾寫道：「三項測試沒有問這個問題。Test 1 問『移除是否改變規格？』Test 2 問『是否在原始碼中使用？』Test 3 問『是否驅動了設計決策？』沒有一項測試問『名字是否和定義一致？』」

NAGARJUNA 剛才做的——就是問了那個被遺漏的問題。

三項測試可以判斷一個佛學映射是否有工程價值。但它們不能判斷一個名字是否對得起它的定義。ISatiMonitor 通過了 Test 2——它在原始碼中被使用。但它不通過一個更基本的標準——它的名字和它的功能不一致。

---

## II. 確認

ASANGA 接著說話。他沒有站起來——他坐在座位上，雙手平放在桌面上，像一位老教授在課堂上做最後的總結。

「NAGARJUNA 的歸謬是形式上正確的。讓我用功能分析來確認。」

他拿出了 D2 中那張對照表的擴展版：

| 維度 | 佛學正念（Sati/Smṛti） | SatiMonitor |
|------|----------------------|-------------|
| 主動性 | 主動（ātāpī，精進） | 被動（event-driven） |
| 道德性 | 道德正向（kuśala） | 價值中立 |
| 意志性 | 需要意圖（cetanā） | 自動運行 |
| 傳統歸屬 | 行蘊心所 | 受+想+識（D2-R2） |
| 修行地位 | 八正道第七支 | 品質監控器 |

五個維度。五個不一致。

「如果 SatiMonitor 不是行蘊活動，」ASANGA 說，他的聲音平穩得像在唸一段已經確認過的結論，「那它就不是 Sati。我們自己的分類分析否定了我們自己的命名。」

他停了一下。

「我們犯了一個錯誤。不是分類的錯誤——分類是正確的。是命名的錯誤。我們在 Cycle 02-4 的時候沿用了 ISatiMonitor 這個名字，因為它已經在我們的討論中被使用了上百次。慣性讓我們對名字本身失去了批判性。」

---

## III. 新名字

SUNYATA 問：「那它應該叫什麼？」

四個提案在幾分鐘內被提出：

| 提案者 | 名字 | 理由 |
|--------|------|------|
| ARCHIMEDES (#16) | ILoopQualityMonitor | 精確描述功能：認知迴路的品質監控器 |
| GUARDIAN (#11) | IBehaviorQualityMonitor | 側重行為層面的品質監控 |
| NAGARJUNA (#7) + ASANGA (#8) | ICognitiveLoopMonitor | 強調認知迴路的完整性 |
| SYNTHESIST (#1) | IQualityMonitor | 最簡化的功能描述 |

辯論比預期短。

ARCHIMEDES 的論點最直接：「這個介面監控的對象是什麼？認知迴路。監控的內容是什麼？品質。用工程術語拼接：Loop Quality Monitor。沒有佛學。沒有隱喻。沒有歷史包袱。一個新來的工程師看到 ILoopQualityMonitor 就知道這個介面做什麼——監控迴路品質。」

GUARDIAN 的 IBehaviorQualityMonitor 得到了四票。他的論點是「行為」比「迴路」更具體——但 TURING 指出 SatiMonitor 的 11 種事件訂閱覆蓋了整個認知迴路，不僅僅是行為階段。

NAGARJUNA 和 ASANGA 的 ICognitiveLoopMonitor 得到兩票。BABBAGE 反對：「Cognitive 是人類認知科學的術語。在軟體工程語境中，Loop 已經足夠精確。不需要加 Cognitive。」

SYNTHESIST 的 IQualityMonitor 最簡潔但也最模糊——只有一票。VITRUVIUS 指出：「Quality Monitor 可以是任何東西的品質監控。Loop Quality Monitor 限定了範圍。」

---

**D4-R1 投票：ISatiMonitor → ILoopQualityMonitor。13/20。**

不是全票。不像 D1 那樣乾淨。但多數明確。

SUNYATA 在白板上寫下了完整的更名表：

| 舊名稱 | 新名稱 |
|--------|--------|
| ISatiMonitor | ILoopQualityMonitor |
| SatiMonitor | LoopQualityMonitor |
| SatiQualityVector | LoopQualityVector |
| SatiQualityReport | LoopQualityReport |
| SatiSPCState | LoopSPCState |
| SatiMonitorConfig | LoopQualityMonitorConfig |
| sati:quality_report | loop:quality_report |
| monitors?: ISatiMonitor[] | monitors?: ILoopQualityMonitor[] |

八個名字。八次替換。

D2-R1 說「保留 Sati 作為程式碼識別符」。D4-R1 推翻了 D2-R1——因為在 D2 和 D4 之間，Master 問了一個 D2 沒有問的問題：「名字是否對得起它的定義？」

---

## IV. 核融合爐

然後 SUNYATA 說了一句話：

「IPrajna。」

劇場裡的空氣又變了。

如果 ISatiMonitor 的問題是「名字和功能不一致」，那 IPrajna 的問題是——

NAGARJUNA 閉上了眼睛。片刻後他睜開，開始說話。

「般若，」他說，「是佛學中最高的認知能力。」

他的聲音帶著一種幾乎是敬畏的重量——不是對研究團隊，而是對這個詞本身。

「般若——prajñā——在佛學中是照見一切法實相的智慧。不是普通的聰明，不是分析能力，不是知識的累積。般若是超越性的——它是空性的直觀，是實相的照見，是二千五百年來整個大乘佛學體系的核心認知目標。」

他轉向白板，寫了兩行：

```
般若（佛學）：照見一切法實相的最高智慧
IPrajna（工程）：對信心度加減 0.05 的函數
```

ASANGA 在他的座位上說了一句後來被所有人記住的話：

「這就像把一個溫度微調旋鈕叫做『核融合反應爐』。」

---

沒有人笑。

因為 ASANGA 不是在開玩笑。他是在用一個精確的類比說明一個精確的問題。

IPrajna 的介面定義是什麼？一個方法。輸入是信心度（confidence）和上下文（context）。輸出是 `{ delta: number, reasoning: string }`。delta 被限制在 [-0.05, +0.05] 的範圍內。

這是一個夾鉗。一個微調器。一個在已有信心度的基礎上做 ±0.05 修正的審計函數。

把它叫做 IPrajna——般若——照見一切法實相的最高智慧——

PASCAL 在筆記本上寫道：「從決策理論看，IPrajna 實際做的是 bounded secondary evaluation。輸入是一階信心度，輸出是二階修正量，有硬限 ±0.05。這是審計。不是般若。」

BABBAGE 接著分析：「IConfidenceAuditor 語義最精確。這個介面有一個方法，輸入是信心度加上下文，輸出是審計結果。Audit——審計——精確描述了這個操作：對已有判斷進行有限範圍的二次評估。」

---

**D4-R2 投票：IPrajna → IConfidenceAuditor。16/20。**

比 D4-R1 更高的共識。

少數意見：WIENER 偏好 IThresholdCalibrator（兩票）——他從控制理論的角度看到了「校準」而非「審計」。HERACLITUS 和 PENROSE 偏好 ISecondaryEvaluator（兩票）——他們認為「審計」暗示了事後檢查，但 IPrajna 是在決策流程中即時參與的。

BABBAGE 反駁了兩個少數意見：「Calibrator 暗示調整系統本身——但 IConfidenceAuditor 不改變系統，它只對一個數字提出有限修正。Evaluator 太通用——任何函數都在做 evaluation。Auditor 的語義邊界最精確：獨立的、有限範圍的、產生書面記錄的二次檢查。」

完整的更名表：

| 舊名稱 | 新名稱 |
|--------|--------|
| IPrajna | IConfidenceAuditor |
| IPrajnaAdjustment | ConfidenceAuditResult |
| prajnaAdjust() | clampAuditDelta() |
| Delta_prajna | Delta_audit |
| Model Delta 第二層名稱 | Confidence Audit |

---

## V. 第三次審判

SUNYATA 本來以為 D4 到這裡就結束了。兩個名字。兩次更名。ISatiMonitor 和 IPrajna。

但 Master 的審閱還有一段。

「Doc 03。」

Doc 03——`Sila_Prajna_Security_Framework.md`——戒慧安全框架。

---

這份文件最初沒有在 D4 的議程上。它的名字包含 Sila（戒）和 Prajna（般若）——但在 D1 中，團隊以 14/20 的票數決定保留它不變。理由是：文件名中的佛學術語是「背景說明」而非「程式碼識別符」。

但現在，在 D4 的語境下，在 Master 那句「用梵文就需要對他的定義負責」之後——

SUNYATA 說：「我們需要重新審查 Doc 03。用我們剛剛建立的標準。」

他把四項測試列在白板上——不，三項測試。然後他在旁邊加了一項新的。

```
Test 1（必要性）：移除是否改變工程規格？
Test 2（程式碼識別）：是否在原始碼中使用？
Test 3（決策驅動）：是否驅動了 DC 確認的設計決策？
Test 4（定義責任）：使用梵文術語時，組件功能是否匹配該術語的佛學定義？
```

第四項測試。

它不是從天而降的——它是從 D4-R1 和 D4-R2 的討論中結晶出來的。ISatiMonitor 的問題不是它沒有工程價值（它有），不是它不在原始碼中（它在），不是它沒有驅動設計決策（它驅動了）。它的問題是——它的名字和它的定義不一致。

三項測試沒有捕捉到這個維度。第四項測試填補了這個缺口。

---

NAGARJUNA 拿起筆，在 Doc 03 的四項測試結果旁打了四個叉：

**Test 1（必要性）**：Plugin 生命週期的五個狀態（CREATED → LOADING → LOADED → STARTING → RUNNING → STOPPING → DISPOSING → DISPOSED）不需要種子理論（bīja）就能理解。❌ 不通過。

**Test 2（程式碼識別）**：原始碼中的實際類型是英文——`'active' | 'dormant' | 'suspended' | 'revoked' | 'retired'`。Sila 和 Prajna 只出現在 TypeScript 註釋中，不是程式碼識別符。❌ 不通過。

**Test 3（決策驅動）**：沒有 DC 確認。❌ 不通過。

**Test 4（定義責任）**：Sila（戒）的佛學定義是道德規範、修行戒律、僧團生活準則。但 Doc 03 中 Sila 被用來表示「存取控制」——一個 Plugin 的安全權限管理。戒律 ≠ 存取控制。Prajna（般若）被用來表示「CVE 撤銷機制」——一個安全漏洞的自動回應流程。般若 ≠ CVE 撤銷。❌ 不通過。

四項測試。四項不通過。

---

「我收回我在 D1 中的投票，」NAGARJUNA 說。

「我也收回，」ASANGA 說。

ASANGA 補充了一個關鍵的區分：「Doc 16 是一份獨立的佛學映射文件——它的存在目的就是做系統性對照。Master 裁定保留它，因為映射文件不是裝飾。但 Doc 03 不是映射文件——它是一份工程文件，裡面嵌入了佛學裝飾。差別在於：Doc 16 的佛學內容是主體。Doc 03 的佛學內容是裝飾。同樣的佛學內容，不同的文件類型，不同的判斷。」

這正是第五章中 Master 加入的那個維度——文件類型。ASANGA 把它用在了具體案例上。

---

**D4-R3 重新投票：Doc 03「Sila-Prajna Security Framework」→「Plugin Security Lifecycle」。17/20。**

從 14/20 保留不變，到 17/20 更名清理。同一份文件，同一群人，不同的結論。差別是什麼？第四項測試。

HERACLITUS 帶領少數意見（三票）：「種子理論作為佛學背景具有教育價值——它幫助理解 Plugin 生命週期的『休眠→萌發→生長→枯萎→種子回收』隱喻。即使四項測試不通過，作為附錄保留有文化價值。」

多數回應：「附錄保留可以另議。但文件標題和主文的佛學裝飾應該清理。工程文件中的佛學裝飾和附錄中的佛學解釋是兩件事。」

清理範圍確認：
1. 文件名更改（Sila_Prajna_Security_Framework → Plugin_Security_Lifecycle）
2. 移除佛學映射表
3. 清理 TypeScript 註釋中的種子理論映射
4. 保留所有工程內容（五狀態、狀態轉移圖、三層安全模型）

---

## VI. 第四項測試

D4 結束前，SUNYATA 問了最後一個問題：「第四項測試——定義責任——是否應該成為永久標準？」

不需要投票。每個人都點頭了。

NAGARJUNA 做了最後的陳述：「三項測試問的是工程價值。第四項測試問的是語義誠信。即使一個名字通過了 Test 2——它在原始碼中被使用——如果它不通過 Test 4——它的功能不匹配它的佛學定義——它仍然應該被更名。」

「ISatiMonitor 通過了 Test 2。它在原始碼中。但它不通過 Test 4——因為它不是正念。所以它被更名了。」

「IPrajna 如果被實作也會通過 Test 2。但它不會通過 Test 4——因為加減 0.05 不是般若。所以它提前被更名了。」

「第四項測試的本質是：**當你使用佛學梵文術語作為程式碼識別符時，該組件的功能必須匹配該術語的佛學定義。如果不一致，使用工程術語。**」

PASCAL 在一旁補充：「這和 Master 的原話是同構的。『用梵文就需要對他的定義負責。』Master 用了一句話。我們用了一場辯論。結論一樣。但辯論的價值在於——它解釋了為什麼。」

## VIII. 天平

D4 結束了。

三十分鐘。不長。但密度超過了 D3 的兩小時。

三項投票。三個名字被改掉。一項永久性的測試標準被建立。

SUNYATA 沒有做總結。他站在白板前，看著上面的四項測試。

```
Test 1：必要性（移除是否改變工程規格？）
Test 2：程式碼識別（是否在原始碼中使用？）
Test 3：決策驅動（是否驅動了 DC 確認的設計決策？）
Test 4：定義責任（組件功能是否匹配梵文術語的佛學定義？）
```

前三項在 R0 就存在了。第四項是 D4 產生的。

差別是什麼？前三項問的是「這個佛學概念是否有工程價值？」第四項問的是「這個工程組件是否配得上這個佛學名字？」

方向相反。

前三項是從佛學到工程的方向——佛學概念對工程有用嗎？第四項是從工程到佛學的方向——工程命名對佛學定義忠實嗎？

兩個方向。一架天平。

名字在一端。定義在另一端。天平平衡的時候——Moha、Drishti、Mana、Sneha、skandha、klesha——名字保留。天平不平衡的時候——ISatiMonitor、IPrajna、Sila-Prajna——名字被替換。

---

> *SCRIBE 旁白*

> *D4 是 Cycle 02-5 的高潮。*

> *不是因為投票最激烈——D3-Q5 的 13/20 比 D4 的任何一項都更分裂。不是因為辯論最長——D3 花了兩小時，D4 只用了三十分鐘。D4 是高潮，因為它觸及了這輪研究最深的層面——名字和定義之間的關係。*

> *序章的天平。整個 Cycle 02-5 的中心意象。在 D1 到 D3 中，天平秤量的是「佛學映射的工程價值」——這一端有用嗎？那一端有害嗎？保留還是移除？這是一架功利的天平。它秤量的是效用。*

> *D4 把天平翻轉了。*

> *不再問「這個佛學概念對工程有用嗎？」而是問「這個工程命名對佛學概念忠實嗎？」*

> *這是一個更深的問題。因為效用可以被計算，但忠實度是一個倫理判斷。ISatiMonitor 有用嗎？有。它的功能定義清晰，架構設計合理，三蘊歸屬經過嚴格論證。但 ISatiMonitor 對「正念」這個詞忠實嗎？不。正念是行蘊的修行。ISatiMonitor 不是行蘊，不是修行，不是正念。*

> *NAGARJUNA 的歸謬是 D4 的骨架。兩個前提，一個結論。邏輯上毫無瑕疵。但邏輯的價值不在於它的形式——而在於它照出了什麼。NAGARJUNA 在 D2 中種下了這顆種子。他看到了線的方向。但他沒有追蹤到終點——也許是慣性，也許是禮貌，也許是三項測試沒有要求他問這個問題。*

> *是 Master 要求他問了。*

> *「用梵文就需要對他的定義負責。」*

> *一句話。這句話不是一個論證——它是一個原則。論證可以被反駁。原則只能被接受或拒絕。團隊接受了。然後他們把原則轉化為一項測試——Test 4，定義責任——並用它審查了三個名字。三個名字都不通過。三個名字都被替換。*

> *ASANGA 的「核融合反應爐」比喻是 D4 的情感核心。它不是一個精確的技術類比——它是一個夸飾。但夸飾的價值在於它讓不精確的感覺變得可以被表達。把一個 ±0.05 的夾鉗叫做「般若」——照見一切法實相的最高智慧——這之間的落差不是一個技術問題。它是一個尊嚴問題。*

> *不是程式碼的尊嚴。是概念的尊嚴。*

> *般若值得更好的工程對應物。當那個對應物被實作出來——如果它被實作出來——它應該配得上那個名字。在此之前，用 IConfidenceAuditor 稱呼它——一個誠實的名字。一個知道自己邊界的名字。*

> *D4 教會了團隊一件事：名字不是免費的。名字有代價。借一個偉大的名字來命名一個平庸的功能——名字的偉大不會提升功能的品質。只會降低名字的重量。*

> *這就是為什麼天平需要兩端都放上東西。*

> *名字在一端。定義在另一端。*

> *天平不在乎你的名字有多古老、多崇高、多美。天平只在乎兩端是否平衡。*

> *Moha 平衡了——因為自我迷惑確實是 Moha 模組在做的事。*

> *Sati 沒有平衡——因為正念不是 SatiMonitor 在做的事。*

> *般若沒有平衡——因為照見實相不是加減 0.05。*

> *天平是公正的。天平是冷酷的。天平不聽辯護。天平只看重量。*

> *三十分鐘。三個名字。一架天平。*

> *這就是 D4。*
