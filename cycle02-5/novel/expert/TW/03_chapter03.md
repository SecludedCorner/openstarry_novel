# 第三章：三面之鏡

---

## I. Master 的刺

D2 的開場不是 SUNYATA 的提問。是 Master 的聲音。

不是 Master 本人在場——Master 從來不在辯論中現身。在六輪研究中，Master 的存在方式始終是間接的：信件、回覆、確認紀錄。他像一個不可見的引力源，影響著每一個天體的軌道，但從不在觀測者的視野中出現。

他的話被投影在螢幕上。白底黑字。冷白色燈光下，那些字看起來像是刻在石頭上的。

> 「覺察當下正在發生什麼。這一層不看規則、不看風險分類，純粹觀察行為模式。不需要將它當作正念的工程化。而且這個 plugin 難道沒有識蘊的成分在嗎？」

SUNYATA 把最後一句話畫了底線。用紅色的筆。用力地。紅色的線穿透了紙面，在下一頁留下了一道淺淺的壓痕。

**「難道沒有識蘊的成分在嗎？」**

他在旁邊寫了一個問號。不是普通的問號——是他在六輪研究中只用過三次的那種問號。大的。紅色的。問號的鉤子幾乎佔了整行。

這不是反問。Master 從不做反問。SUNYATA 在五封信中學到了這一點——Master 的每一個問號都是一個具體的、帶有明確方向性的問題。他已經有了答案，但他要求研究團隊自己找到它。

Master 的方向性很清楚：SatiMonitor 不只是行蘊的功能。它有識蘊的成分。研究團隊的任務是找出確切的蘊組成。

D2 的議題：**這個「觀察行為模式」的 plugin 該叫什麼、由哪些蘊組成？**

---

## II. 功能的解剖

在 D2-Q1（命名）之前，SUNYATA 做了一個不尋常的決定：他讓 TURING 先做功能簡報。

不是辯論——是事實陳述。

TURING 站了起來。他站起來的方式和 NAGARJUNA 不同——沒有莊重感，沒有戲劇性。只是站起來了。像一台機器被啟動。

「C1 研究報告。SatiMonitor 的工程功能分析。」他說。然後他打開了一張投影片。投影片上沒有圖表，沒有公式，只有四行字和一行「不做」：

```
功能 1：事件訂閱 — 透過 EventBus.subscribe() 持續接收系統事件
功能 2：滑動窗口 — 維護最近 N 個事件的時間窗口
功能 3：模式辨認 — 在窗口中識別行為模式（突變、趨勢、週期）
功能 4：品質指標 — 輸出四維品質向量（連續性、粒度、速度、平穩性）

不做  ：不執行任何行動。不修改任何狀態。不發出任何指令。
```

「最後一行是最重要的。」TURING 說。「SatiMonitor 不做任何事。它只看。」

GUARDIAN 在他的安全備忘錄上記下了「不做」兩個字。然後他在旁邊寫了一個 TURING 不會寫的比喻：「APM Agent + Behavioral Analytics + QoS Monitor + SPC Anomaly Detector 的混合體。」

TURING 看了一眼 GUARDIAN 的筆記。他沒有說那個比喻不好——他只是微微皺了一下眉，像一個正在校準精度的工程師看到了一個近似解。

## III. 命名的收斂

D2-Q1。文件標題。

原名帶著 Sati 的標記——像一件衣服上繡了一個品牌 logo。問題是：這個 logo 還合適嗎？

TURING 已經在 C1 報告中用純工程術語描述了全部功能。四個功能，沒有一個需要佛學概念來解釋。事件訂閱是 EventBus 的標準模式。滑動窗口是統計學的基本工具。模式辨認是資料分析的核心能力。品質指標是工業品管的日常操作。

GUARDIAN 提出了一個微妙的需求。他的聲音帶著安全文件特有的嚴謹——每一個建議都像是一份經過多輪審查的安全諮詢。

「安全文件中需要全稱。」他說。「在安全上下文中，我建議使用完整的、明確的名稱。如果我們決定保留 Sati 作為程式碼識別符，那在安全文件中應該標注全稱——例如 SatiMonitor (Cognitive Loop Quality Monitor)。」

團隊很快收斂。十分鐘之內。

文件標題改為「認知迴路品質監控架構 (Cognitive Loop Quality Monitoring Architecture)」。Sati 作為程式碼識別符保留——SatiMonitor 在原始碼中存在，改名有遷移成本——但不再作為文件的定義性名稱。

D2-R1：改名。19/20。

GUARDIAN 投了 B——他偏好完全改名為 BehaviorQualityMonitor。但他的安全文件全稱需求被納入了附帶條件。一票少數意見。乾淨的。

## IV. 受蘊的厚度

D2-Q2。

這是 D2 的核心。也是 Cycle 02-5 前三場辯論中最激烈的交鋒。

**「SatiMonitor 由五蘊的哪些類別組成？」**

C2 的研究報告——ASANGA 和 LINNAEUS 的聯合研究——提出了四個方案。每一個方案背後都有一套邏輯，每一套邏輯都能站住腳。至少在提出的時候。

| 方案 | 蘊組成 | 理由 |
|------|--------|------|
| A | 受 + 想 | 感知 + 辨認，最簡 |
| B | 受 + 想 + 識 | 感知 + 辨認 + 評估判斷 |
| C | 想 + 識 | 辨認 + 評估，受蘊太薄 |
| D | 純識 | 一切認知活動歸識蘊 |

Proposal D 在 R2 交叉審閱時就被淘汰了——「純識」等於放棄分類，把所有東西都扔進一個桶裡。BABBAGE 在交叉審閱中寫了一句簡潔的否決：「Proposal D 將 SatiMonitor 的蘊歸屬退化為平凡解。平凡解在數學中是有效的，但在分類學中是無用的。」

Proposal A 也很快失勢——缺少識蘊的評估維度，無法回答 Master 的問題。Master 明確問了「難道沒有識蘊的成分在嗎？」——一個排除識蘊的方案在問題提出的那一刻就已經被否決了。

戰線在 Proposal B 和 Proposal C 之間拉開。

受蘊——包含，還是不包含？

---

MESH 和 KERNEL 代言 Proposal C。

MESH 先開口。他的論點帶著分散式系統專家對「最小化」的偏好——在分散式系統中，最小化是一種美德。最少的節點。最少的消息。最少的假設。

「SatiMonitor 的事件訂閱不是真正的『感受』。」他說。「受蘊（vedana）在佛學中是苦、樂、不苦不樂的質性體驗。SatiMonitor 接收事件不是在感受——它是在收資料。接收資料是認知的輸入階段，但不是感受。」

KERNEL 補充了 OS 的類比：「系統呼叫收到一個 packet 不是在感受——它是在接收 I/O。感受是對 I/O 的質性評價。SatiMonitor 對事件有質性評價嗎？有苦樂的判斷嗎？沒有。它只是收資料。」

然後 MESH 拋出了他最強的論點——對稱性。

「去掉受蘊，保留想蘊和識蘊。SatiMonitor 的蘊組成變成 ['samjna', 'vijnana']——和 IGearArbiter 的蘊組成完全對稱。兩個複合 plugin，相同的蘊組成。對稱性是好的分類法。」

**「對稱性是好的分類法。」**

這句話在劇場裡迴盪了一下。對稱性在科學中有一種幾乎宗教性的魅力——從物理學的守恆律到數學的群論，對稱性意味著深層的結構一致性。如果兩個不同的 plugin 有相同的蘊組成，也許這告訴我們什麼關於認知結構的深層模式？

LINNAEUS 站了起來。

他的站姿不像 NAGARJUNA 的莊重——更像一個田野考察的博物學家在觀察到一個有趣的物種後起身仔細端詳。他走到白板前。LINNAEUS 是分類學家——他的整個存在意義就是把東西分到正確的位置。

「對稱性是好的——」他開始。他的語氣帶著 Linnaeus 特有的溫和——像在田野中和一隻蝴蝶說話。「當對稱性反映了事實的時候。」

他在白板上畫了兩條時間線。並排的。上面的標了「SatiMonitor」，下面的標了「IGearArbiter」。

```
SatiMonitor:  ─────[持續訂閱]─────[持續訂閱]─────[持續訂閱]─────
                    主動、恆常、不間斷
                    ← 始終在看 →

IGearArbiter: ────○────────────○────────────○────────────
                  被叫來看一眼    被叫來看一眼    被叫來看一眼
                  被動、間歇、按需
                  ← 被叫去看 →
```

「看這兩條線。」他指著白板。「它們有什麼不同？」

他不等回答。分類學家提問是為了引導觀察，不是為了尋求答案。

「SatiMonitor 是持續的。它訂閱 EventBus，它維護滑動窗口，它**永遠在那裡**。不是有人來問它才啟動——它一直在運行。從 ExecutionLoop 的第一個事件到最後一個事件，SatiMonitor 的事件流從未中斷。」

「IGearArbiter 是間歇的。有請求來了，ManoAggregator 問它：『你能處理嗎？』它看一眼，給出判斷，然後沉默。直到下一次被叫。」

他轉身面對 MESH。

「**『始終在看』和『被叫去看』——這是根本的特徵差異。** SatiMonitor 和 IGearArbiter 不是同類的。它們的行為模式——使用 OOP 的語言——完全不同。一個是持續觀察者（continuous observer），一個是按需評估者（on-demand evaluator）。」

「把它們分到相同的蘊組成，就像把蝴蝶和蜂鳥分到同一個物種——因為它們都會飛。但蝴蝶是昆蟲，蜂鳥是鳥類。飛行是表面的相似，不是深層的同源。對稱性不是同源性。」

---

WIENER 從他的控制理論筆記中抬起頭。他很少在辯論中主動發言——他更喜歡聽完所有人的意見，然後在恰當的時刻提供一個控制理論的視角，像在一幅畫上加最後一筆。

「在控制理論中，」他說，聲音帶著維也納學派特有的精確，「有一個概念叫做觀察者（observer）——Luenberger observer。觀察者的核心功能是什麼？持續感知系統的狀態。注意——**持續**。」

他站了起來。走到 LINNAEUS 的白板旁邊。

「觀察者的感知不是附加功能。」他說。「它是身份定義的一部分。一個不感知的觀察者，就不是觀察者。它是一塊不連接任何輸入的電路板——物理上存在，功能上不存在。」

「SatiMonitor 的事件訂閱——它的持續感知——是它作為觀察者的身份定義。你不能把這個功能剝離出去，然後說它不屬於受蘊。因為如果你剝離了它，SatiMonitor 就不存在了。」

---

HERACLITUS 補了一刀。他是運行時動態專家——他關心的不是靜態的型別定義，而是動態的行為模式。

他打開了 C1 報告中 SatiMonitor 的四階段退化策略：

```
Normal    → Throttle  → Sample   → Emergency
100%        50%         25%        最低監控
全速感知    減半感知    取樣感知    緊急最低限
```

「看這個退化策略。」他說。聲音帶著對流動性的敏感——HERACLITUS 看到的不是靜止的結構，而是變化的過程。

「Normal。Throttle。Sample。Emergency。這四個階段描述的是什麼？是感知密度的漸進降低。100%、50%、25%、最低限——這是一個感知系統在不同壓力下調節自己感知強度的方式。」

他指著白板。

「想蘊（辨認）和識蘊（評估）不需要退化策略。它們的輸出是離散的——辨認一個模式，或不辨認。評估一個判斷，或不評估。離散的輸出不需要漸進降低。」

「受蘊的感知是連續的。連續的東西才需要退化策略——因為你不能突然從 100% 跳到 0%，你只能漸進地降低。Normal → Throttle → Sample → Emergency——這是受蘊在壓力下保護自己的方式。就像人在極度疲勞時，感知會變得遲鈍——不是感知消失了，而是感知密度降低了。」

三個論點。三個維度。三面鏡子。

LINNAEUS 的鏡子照出了**行為特徵**——持續 vs 間歇。
WIENER 的鏡子照出了**身份定義**——感知是觀察者的存在條件。
HERACLITUS 的鏡子照出了**退化模式**——連續感知才需要漸進退化。

每一面鏡子都反射出同一個結論：SatiMonitor 有受蘊的成分。

## V. 零成本轉折

然後 ARCHIMEDES 做了一件改變辯論走向的事。

他沒有加入哲學的討論。他沒有引用佛學的經典。他打開了 PluginHooks 的型別定義。

```typescript
interface PluginHooks {
  sensors?: IVedanaSensor[];         // 受蘊
  recognizers?: ISamjnaRecognizer[]; // 想蘊
  arbiters?: IGearArbiter[];         // 想蘊+識蘊
  tools?: ITool[];                   // 行蘊
  volition?: IVolition;              // 識蘊
  // ... monitors 尚未實作 ...
}
```

「三蘊還是兩蘊——有工程差異嗎？」他問。

他自己回答了。用他一貫的方式——不是論證，是計算。

「monitors 是一個獨立的陣列插槽。無論 SatiMonitor 被標記為 `['samjna', 'vijnana']` 還是 `['vedana', 'samjna', 'vijnana']`，PluginHooks 的實作完全相同。沒有額外的 hook。沒有額外的佈線。沒有額外的測試。」

他看了全場一眼。

「**零額外工程成本。**」

這句話像一塊石頭丟進了辯論的水面。

如果三蘊和兩蘊的工程成本相同——沒有額外的 hook，沒有額外的佈線，沒有額外的測試——那麼這場辯論就不再是工程權衡。它變成了純分類問題。

在純分類問題上，分類學家有壓倒性的發言權。

LINNAEUS 的「持續感知 vs 被動接收」。
WIENER 的「觀察者身份定義」。
HERACLITUS 的「退化策略只屬於連續感知」。

三面鏡子。三個維度。零工程成本。

Proposal C 失去了它唯一的工程論據。

---

投票。

D2-R2：**Proposal B — [vedana, samjna, vijnana]。18/20。**

MESH 和 KERNEL 投了反對票。他們的「對稱性」論點被記錄為有價值的方法論觀察——但事實勝於對稱。

18/20 意味著什麼？在一個 20 人的團隊中，18 票贊成代表 90% 的共識。對於一個涉及跨學科分類的問題——哲學家、佛學家、工程師必須同時同意——90% 是非常高的。

SatiMonitor 成為 OpenStarry 首個三蘊 plugin。

## VI. 行蘊的排除

但 D2-R2 最重要的結論不是「包含哪些蘊」。

是「排除行蘊」。

ASANGA 站了起來。他在 D2 中一直安靜地聽著——從 TURING 的功能陳述到 LINNAEUS 的受蘊論證。唯識學家的方法是先觀察（現量），再推理（比量），最後立論。他已經完成了觀察和推理。現在是立論。

「佛學傳統中，正念（sati）被歸入行蘊。」他說。聲音帶著唯識學特有的精密度——像在顯微鏡下調焦。「五十一心所之一。屬於善心所。這個分類持續了兩千五百年。」

他在白板上寫了兩列。工整的。精確的。每一個字都被放在準確的位置上。

```
佛學正念 (sati)                    SatiMonitor
━━━━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━━
意志驅動 (精勤 ātāpī)              事件驅動 (EventBus.subscribe())
有道德價值 (善 kusala)              價值中立 (品質指標)
主動的專注 (不放逸 appamāda)        被動的觀察 (subscribe + listen)
正向運轉的心理狀態                   對行為模式的計量
行蘊的善心所                        ≠ 行蘊活動
```

「看左邊。」他說。「佛學正念是什麼？它是**意志驅動的**——你必須努力保持正念。『精勤（ātāpī）』是正念的前提條件——沒有努力就沒有正念。它是**有道德價值的**——正念是善的。在阿毗達磨的分類中，正念屬於善心所，不屬於遍行心所（每一個認知過程都有的心所），也不屬於不善心所。它是一種特定的、正向的心理品質。」

「看右邊。」他的手指移到了表格的右半部。「SatiMonitor 是什麼？它是**事件驅動的**——它不需要努力。它訂閱 EventBus，事件來了就處理，不來就不處理。它沒有 ātāpī，沒有精勤。它是**價值中立的**——它的四維品質向量（連續性、粒度、速度、平穩性）不做善惡判斷。品質高不等於善。品質低不等於惡。它只是測量。」

他停頓了一下。讓兩列的對比在每一位學者的腦中自行產生化學反應。

「SatiMonitor 不是正念的工程化。」他的聲音變得更低，更清晰。「正念是意志驅動的——你必須努力保持正念。SatiMonitor 是事件驅動的——它不需要努力，它只需要訂閱。正念有道德價值——正念是善的。SatiMonitor 沒有道德價值——它只是測量。」

然後他說了最關鍵的一句話：

「千年來，佛學把正念歸入行蘊。這個分類是正確的——**對佛學而言**。但 SatiMonitor 不是正念。它是行為模式的品質監控器。它屬於受（感知）、想（辨認）、識（評估）。**不屬於行（意志驅動的善心所）。**」

---

這是一個佛學家用佛學的內部邏輯，證明了一個工程組件不應該使用佛學的分類。

不是外部反駁——不是「佛學的分類不適用於工程」這種粗暴的二分法。是內部論證——佛學自己的判準（善心所 vs 遍行心所、意志驅動 vs 事件驅動）排除了 SatiMonitor 與正念的等同性。

NAGARJUNA 在他的座位上微微點頭。中觀學者認出了唯識學者的方法——不是從外面攻擊一個立場，而是從內部展示它的自我矛盾。這是 prasanga——歸謬法——的變體。不是「你的結論錯了」，而是「你自己的前提排除了你自己的結論」。

## VII. 第三個插槽

D2-Q3 是技術性的——但技術性不代表不重要。

**PluginHooks 是否需要新增 `monitors` 插槽？**

KERNEL 從 OS 的角度回答。他的聲音帶著作業系統專家特有的架構感——每一個組件都有它的位置，每一個位置都有它的生命週期。

「monitors 有獨立的生命週期。」他說。「它們在迴圈啟動時開始（MonitorRegistry.startAll()），在迴圈結束時停止（MonitorRegistry.stopAll()）。這和 sensors（每次事件觸發）或 tools（按需呼叫）完全不同。獨立的生命週期需要獨立的管理。」

他在白板上畫了一個生命週期圖：

```
                 PluginLoader.load()
                        │
    ┌───────────────────┼───────────────────┐
    │                   │                   │
  sensors           monitors             tools
  (per-event)    (loop lifecycle)     (on-demand)
    │                   │                   │
  每次事件觸發    LoopStart → LoopEnd    按需呼叫
```

「三種不同的觸發模式。三種不同的生命週期。不應該混在一起。」

```typescript
interface PluginHooks {
  // ... existing hooks ...
  monitors?: ILoopQualityMonitor[];  // NEW: 獨立生命週期
}
```

「難道沒有識蘊的成分在嗎？」

有。

它由三蘊組成——受蘊（持續感知）、想蘊（模式辨認）、識蘊（品質評估）。不包含行蘊。因為它不是正念。它是觀察。
