# 第一章：审计员

---

## I. 五十个标本

R1 独立研究启动后的第一天，ARCHIMEDES 把自己关在了工作间里。

他面前摊着七份文件。不是研究报告——是 openstarry_doc 的七份原始文件。Doc 38。Doc 43。Doc 44。Doc 37。Doc 41。Doc 16。Doc 31。每一份都被他用不同颜色的便利贴标记过：黄色是「疑似装饰」，蓝色是「需要交叉验证」，粉红色是「代码标识符」，绿色是「DC 确认」。

B1 审计。逐文件。逐段落。逐映射。

SCRIBE 坐在他旁边。SCRIBE 的工作不是审计——是记录。记录 ARCHIMEDES 的每一个判断、每一个犹豫、每一个在黄色和蓝色之间摇摆的瞬间。

「Doc 38。」ARCHIMEDES 翻开第一份文件。「Klesha Gain Scheduling。L540-544。」

他的手指停在一张表格上。三栏：工程概念、佛学映射、说明。śīla（教诫）= Hard Rules。upāya（善巧）= Soft Rules / Adaptive。smṛti（正念）= Monitoring / Event-driven。

「三项测试。」他喃喃自语。然后开始逐行应用。

**Test 1**（必要性）：移除「佛学映射」这一栏，Hard Rules 还是 Hard Rules 吗？是的。Soft Rules 还是 Soft Rules 吗？是的。工程规格完全不变。

→ 「佛学映射」栏位：**REMOVE**。

SCRIBE 记下了。第一个标本。

「等一下。」ARCHIMEDES 回过头。「工程概念」栏和「说明」栏是有用的——只有「佛学映射」栏是装饰性的。不是移除整张表。是移除一栏。」

精细分离。这是 ARCHIMEDES 的特长——他不是一个用大锤砸核桃的人。他是一个用手术刀分离组织的外科医生。

他继续往下翻。

---

Doc 43。SatiMonitor——不，他在心里修正了自己——那个「观察行为模式」的 plugin。整份文件的标题带着「正念」的标记。文件里散落着 event-driven = 正念的隐喻、品质度量被描述为「觉察的量化」、四阶段退化策略被比喻为「正念的深浅层次」。

ARCHIMEDES 开始计数。

第一个：「event-driven 架构是正念的工程实现」——Test 1，移除后规格不变。REMOVE。
第二个：「SatiQualityVector 量化了觉察的四个维度」——Test 1，移除后 SatiQualityVector 的功能不变。REMOVE。
第三个：「四阶段退化策略反映了正念从初学到熟练的层次」——Test 1，明显装饰。REMOVE。

他一个一个地标记。黄色便利贴越来越多。

「这份文件。」他对 SCRIBE 说。「工程内容是扎实的——事件订阅、滑动窗口、品质指标、退化策略——全部有技术深度。但佛学装饰像一层薄霜，覆盖在每一个工程概念上。如果你不熟悉佛学，这些装饰会让你困惑。如果你熟悉佛学，这些装饰会让你觉得被冒犯。」

SCRIBE 把这段话一字不差地记下了。后来，这段话被引用了不下三次——在 D1 辩论中、在 D4 延长讨论中、在 Master 的审阅回复中。

---

Doc 44。Doc 37。Doc 41。每一份文件，ARCHIMEDES 都用同样的方法：三项测试，逐映射应用，精细分离工程内容和佛学装饰。

SCRIBE 在旁边统计。

到了第一天结束的时候，B1 审计报告的初稿已经成形。50 个映射实例。每一个都有出处（文件名 + 行号）、处置建议（KEEP / RELOCATE / REMOVE）、和理由（哪一项测试、哪一个结论）。

| 处置 | 数量 | 范例 |
|------|------|------|
| **KEEP** | 23 | Skandha 类型名、Klesha 模块名（Moha/Drishti/Mana/Sneha）、DC 确认约束、CoarisingBundle |
| **RELOCATE** | 13 | PASCAL 数学形式化、MN 18 经典引用、DD-13 佛学背景、各处佛学设计理由 |
| **REMOVE** | 14 | śīla/upāya/smṛti 标签、event-driven=正念、戒定慧映射、种子理论装饰 |

ARCHIMEDES 看着这张表。23 + 13 + 14 = 50。

「23 个 KEEP。」他说。「将近一半。」

SCRIBE 点了点头。「不是所有佛学内容都是装饰。有些是身份——像 Klesha 模块名。有些是决策——像五蕴分类驱动了 PluginHooks 的设计。」

「但 14 个 REMOVE。」ARCHIMEDES 继续。「将近三分之一。这些是纯粹的装饰——移除它们不会改变任何工程规格，但会显著降低阅读障碍。」

他把报告存档。`B1_forced_mapping_audit.md`。然后他打开了另一个文件——Doc 16 和 Doc 31。

## II. 两份可疑文件

Doc 16 和 Doc 31 是不同的。

前面审计的七份文件——Doc 38、43、44、37、41——都是工程文件。它们的主要内容是工程规格，佛学映射是附加的。就像一本食谱书里夹了几张宗教节日的插图——食谱是主体，插图是装饰。

但 Doc 16（Plugin 类型哲学映射）和 Doc 31（八识运行时模型）不一样。

ARCHIMEDES 翻开 Doc 16。从第一页开始读。然后翻回封面。然后再读第一页。

「SCRIBE。」他说。「这份文件的装饰比例大概是多少？」

SCRIBE 快速扫描了一遍。「大约 80%。」

80%。意思是：Doc 16 中，只有大约 20% 的内容是纯工程规格（主要是 Observer Pattern 的讨论），其余 80% 是佛学概念到软件模式的哲学映射。

Doc 31 好一些，但也有大约 70% 的佛学内容。八识如何映射到运行时模型——阿赖耶识是种子库，末那识是自我执着——这些映射有学术价值，但放在工程文件中，它们的工程效用有限。

ARCHIMEDES 在 Doc 16 和 Doc 31 上贴了一个特殊的标签——不是黄色（装饰）、不是粉红色（标识符）、不是绿色（DC 确认）。是红色。

红色 = **FILE REVIEW**。整份文件需要评估。

「这两份文件的问题不在于个别映射。」他对 SCRIBE 说。「问题在于整份文件的定位。它们是工程文件中嵌入了佛学？还是以佛学映射为目的的独立文件？」

SCRIBE 记下了这个问题。这个问题后来在 D1-Q7 和 D1-Q8 中被正式投票——然后在 D4 中被 Master 亲手推翻。

## III. 三个维度

B1 审计是事实的搜集。B2 是原则的建构。

在 ARCHIMEDES 数标本的同时，NAGARJUNA、ASANGA 和 PASCAL 在另一间工作室里建构映射边界原则。三个人。三个学科。三个维度。

NAGARJUNA 坐在靠窗的位置。窗外是什么不重要——对中观学者来说，窗外的景色和窗内的思考之间没有本质的区别，它们都是世俗谛的显现。重要的是他面前的白板。白板上画着一条水平线。左边写着「世俗谛」，右边写着「胜义谛」。

「二谛。」他说。他的声音在小工作室里听起来比在剧场里更近、更低。「龙树《中论》第二十四品：诸佛依二谛，为众生说法。一以世俗谛，二第一义谛。」

PASCAL 在笔记本上记下了引用。他不是在做佛学研究——他是在寻找公理。每一个理论体系都有它的公理，PASCAL 的工作是找到它们，然后检验从这些公理出发能否到达他们需要的结论。

「世俗谛语言 = 工程术语。」NAGARJUNA 在白板上写。「用于日常沟通和实作。if/else、interface、extends——这些是世俗谛的语言。它们不需要佛学的加持就能工作。」

「胜义谛洞见 = 佛学概念。」他在右边写。「用于启发和设计理由。五蕴的分类启发了根接口的设计。烦恼的分类启发了 Klesha 的模块结构。但启发不等于标签。」

他转身面对 PASCAL 和 ASANGA。

「两者不应混淆。不要在世俗谛的文件中硬塞胜义谛的标签。」

PASCAL 点了点头。然后他打开了他的维度。

---

PASCAL 的白板在 NAGARJUNA 旁边。他画了一个二维矩阵。

```
                 映射有价值        映射无价值
保留映射         True Positive     False Include ← 累积损害
移除映射         False Exclude     True Negative
                 ↑ 一次性成本
```

「这是标准的混淆矩阵。」他说。「但关键不在矩阵本身——在损害的不对称性。」

他在 False Include 旁边画了一个向上的箭头。

「False Include——保留了一个装饰性映射。它的损害是什么？每一位未来的读者——可能是几十人，可能是几百人——每一次阅读这份文件——可能一次，可能十次——每一页上出现的佛学标签——可能三个，可能十个——每一次，读者都需要在脑中维护两套术语系统。śīla 是什么？噢，就是 Hard Rules。那为什么不直接叫 Hard Rules？」

他写了一个公式：

$$C_{FI} = N_{readers} \times R_{reads} \times P_{pages} \times \tau_{cognitive}$$

「累积损害。读者数 × 阅读次数 × 页数 × 每次的认知转换成本。这是一个乘法——每一项都是正的，乘积只会增长。」

然后他在 False Exclude 旁边画了一个小得多的箭头。

「False Exclude——移除了一个有价值的映射。损害是什么？某一天，某个人需要知道五蕴分类的佛学背景。他搜寻附录。找到了。一次性的搜寻成本。不累积。不乘以读者数。不乘以阅读次数。」

$$C_{FE} = S_{search} \quad \text{(一次性)}$$

「结论。」他画了一个大大的不等号。

$$E[C_{FI}] \gg E[C_{FE}]$$

「期望值分析：装饰性映射具有负期望值。有疑虑时，倾向 REMOVE 或 RELOCATE。」

---

ASANGA 一直在听。他没有白板——他的思考工具是语言，不是图形。

「功能性判定。」他说。声音带着唯识学特有的精密度——像在显微镜下调焦。「唯识学有一个核心判准：因果效力（arthakriyā-sāmarthya）。一个概念是否具有因果效力——它是否驱动了一个结果？」

他站了起来。走到 NAGARJUNA 和 PASCAL 的白板之间。

「佛学概念驱动了五蕴根接口的设计——这有因果效力。五蕴分类 → PluginHooks 的五个类别。因果链明确。保留。」

「佛学标签被贴在 Hard Rules 上叫 śīla——这没有因果效力。Hard Rules 不是因为被叫做 śīla 才有硬性约束——它的硬性约束来自 SafetyMonitor 的实作。śīla 这个标签不改变任何因果链。移除。」

他在空中用手画了一条线——从因到果。

「判断标准很简单：**如果移除这个佛学概念，因果链是否断裂？** 是 → 保留。否 → 移除。」

PASCAL 抬起头。他看到了三个维度的交汇：

```
NAGARJUNA — 二谛（范畴分析）
            世俗谛 vs 胜义谛 → 不要混淆两种语言

PASCAL    — 损害不对称（决策理论）
            E[累积损害] >> E[搜寻成本] → 有疑虑时移除

ASANGA    — 因果效力（唯识学）
            有因果效力 → 保留 / 无因果效力 → 移除
```

三个学科。哲学、数学、佛学。三个公理体系。三条推理路径。

一个结论。

## IV. TURING 的地毯式扫描

R2 交叉审阅。

TURING 不说话。或者说，TURING 说的话少到可以被完整列举。在 Cycle 02-5 的 R2 阶段，他在所有记录中一共说了七句话。

但他做的事情比任何人都多。

九份独立研究报告。每一份都引用了 v0.28.0-alpha 的源代码——`aggregates.ts`、`plugin-loader.ts`、`loop.ts`、`plugin.ts`、`safety-monitor.ts`——每一个引用都需要被验证：这段代码确实在那个位置吗？引用的行为描述准确吗？接口定义引用正确吗？

TURING 做了地毯式扫描。40 个以上的代码引用。逐一打开源代码。逐一比对。逐一确认或标记。

他的终端机上，`rg`（ripgrep）指令一个接一个地执行。搜寻 `IVedanaSensor`——确认。搜寻 `IGearArbiter`——确认。搜寻 `ILoopQualityMonitor`——不存在（尚未实作）。搜寻 `IInferenceProvider`——

他停住了。

A5 报告（跨蕴互动分析）的附录 B 列出了 `IInferenceProvider` 作为想蕴的子接口之一。TURING 在 v0.28.0-alpha 的整个代码库中搜寻了这个名称。

零结果。

他在审阅表格的 T-01 栏中写了一行：「A5 附录 B 列出 IInferenceProvider 但源代码中未找到。中等严重度。」然后继续扫描。

T-02：A4 报告中 Phase 7 的描述可能误导读者认为 VedanaSensor 已完全运作——实际上 VedanaRegistry 已建立但消费者尚未实作。低严重度。

T-03：B1 审计报告中 Doc 44 L174 的交叉检查自我修正段落需要清理。最低严重度。

T-04：A1 和 C2 报告中 PluginHooks 的栏位名称不一致——`gearArbiters` vs `arbiters`。TURING 查了源代码：正确名称是 `arbiters`。中等严重度。

四个问题。40 个以上的引用。四个问题。这是一个 10% 以下的错误率——在学术论文中算是非常干净的。

TURING 在审阅矩阵的最后一栏写了一个字：

**PASS。**

他合上了笔记本。然后他说了他在 R2 阶段的第三句话。

「引用干净。可以进入辩论。」

六个字。像他的代码验证一样简洁。

## V. 二十四个共识，七个问题

R2 的审阅矩阵不只是 TURING 的代码验证。二十位研究员组成了交叉审阅网——每人审阅至少两份非本人主笔的文件。

结果令人欣慰，也令人不安。

令人欣慰的是：24 个共识点被识别。跨三条轨道——A-track 的 8 个（五蕴根接口足够、DI 链完整、Plugin 生命周期清晰等）、B-track 的 5 个（四层框架有效、三项测试可操作等）、C-track 的 7 个（SatiMonitor 功能明确、不属于纯行蕴等）、加上 4 个全域共识（行蕴窄化需记录、SlashCommand bypass 确认等）。

令人不安的是：7 个开放问题。其中 OQ-1（SatiMonitor 蕴组成）是核心辩论议题。4 个分歧点——最尖锐的是 D-1：受蕴「厚」（包含 SatiMonitor 的感知层）还是「薄」（SatiMonitor 的感知层不算受蕴）。

SUNYATA 在 R2 结束后花了三十分钟整理这些数据。24 个共识意味着大部分的基础是稳固的——辩论不需要从零开始。7 个问题意味着辩论有明确的焦点——不需要漫无目的地讨论。4 个分歧意味着辩论会有火花——不是客气的点头，是真正的交锋。

他在白板上重新写了三场辩论的议程。

**D1**：佛学映射边界。核心问题：openstarry_doc 工程文件中佛学内容的原则是什么？

**D2**：Sati Plugin 五蕴分类。核心问题：这个 plugin 由五蕴的哪些类别组成？

**D3**：五蕴 OOP 架构完整性。核心问题：五蕴架构是否足以支撑工程实作？

然后他在议程下面加了一行小字：

> *R1 九份研究 → R2 二十四共识 + 七问题 + 四分歧 → R3 三场辩论*

管道是通的。数据是干净的。工具是校准的。

手术室准备好了。
