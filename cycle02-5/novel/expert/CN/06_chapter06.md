# 第六章：名字的代价

---

剧场里的灯光没有变。还是那个 6500K 的冷白色。但不知道为什么，空气的温度变了。

不是物理温度。是一种感觉——像考试开始前的教室，或者法庭宣判前的等待。每个人都坐在自己的位子上，但没有人真的在休息。

SUNYATA 走到投影幕前，把 Master 的那句话打在屏幕上。

大字。白底黑字。没有格式。

> **「用梵文就需要对他的定义负责。你觉得 Sati 的内容完全 match 吗？它归在哪一蕴？」**

---

## I. 归谬

NAGARJUNA 站了起来。

在整个 Cycle 02-5 中，他已经站起来过很多次——D1 中承认「后贴标签」，D3 中分析十二因缘的尺度不匹配。每一次他都保持着中观学者特有的端正姿态——承认错误是学术的核心技能，他从不回避。

但这一次不一样。

这一次他不是在承认错误。他是在完成一个推论。

「D2 已经确定了 SatiMonitor 的蕴归属，」他说，声音比平常低半个音调。不是因为紧张——是因为他在组织一个他已经想了很久但一直没有说出口的论证。

「结论是 [vedana, samjna, vijnana]。三蕴。不包含行蕴。」

他顿了一下。

「Sati——巴利文的正念——在佛学传统中是行蕴的心所。它是 cetasika，是有意志的、主动的、道德正向的修行活动。阿毘达磨把它放在行蕴，因为正念是一种修行——你必须带着意图去做的事情。」

他转向 ASANGA。ASANGA 微微点头。他知道 NAGARJUNA 要说什么。

「现在，」NAGARJUNA 说，「让我做一个归谬论证——prasaṅga。这是中观学的基本方法。」

他在白板上写了两行字：

```
前提 A：Sati = 行蕴心所（佛学定义）
前提 B：SatiMonitor ≠ 行蕴（D2-R2 结论）
```

然后他在下面写了第三行：

```
∴ SatiMonitor ≠ Sati
```

「如果正念（Sati）在佛学中必然是行蕴的心所，而我们已经证明了 SatiMonitor 不是行蕴——那么 SatiMonitor 就不是正念。」

他放下笔。

「一个不是正念的东西，为什么叫 Sati？」

---

沉默。

不是不同意的沉默。是那种——所有人同时理解了一件他们应该更早理解的事情——的沉默。

D2 的 SCRIBE 旁白曾写道：「三项测试没有问这个问题。Test 1 问『移除是否改变规格？』Test 2 问『是否在原始码中使用？』Test 3 问『是否驱动了设计决策？』没有一项测试问『名字是否和定义一致？』」

NAGARJUNA 刚才做的——就是问了那个被遗漏的问题。

三项测试可以判断一个佛学映射是否有工程价值。但它们不能判断一个名字是否对得起它的定义。ISatiMonitor 通过了 Test 2——它在原始码中被使用。但它不通过一个更基本的标准——它的名字和它的功能不一致。

---

## II. 确认

ASANGA 接着说话。他没有站起来——他坐在座位上，双手平放在桌面上，像一位老教授在课堂上做最后的总结。

「NAGARJUNA 的归谬是形式上正确的。让我用功能分析来确认。」

他拿出了 D2 中那张对照表的扩展版：

| 维度 | 佛学正念（Sati/Smṛti） | SatiMonitor |
|------|----------------------|-------------|
| 主动性 | 主动（ātāpī，精进） | 被动（event-driven） |
| 道德性 | 道德正向（kuśala） | 价值中立 |
| 意志性 | 需要意图（cetanā） | 自动运行 |
| 传统归属 | 行蕴心所 | 受+想+识（D2-R2） |
| 修行地位 | 八正道第七支 | 品质监控器 |

五个维度。五个不一致。

「如果 SatiMonitor 不是行蕴活动，」ASANGA 说，他的声音平稳得像在念一段已经确认过的结论，「那它就不是 Sati。我们自己的分类分析否定了我们自己的命名。」

他停了一下。

「我们犯了一个错误。不是分类的错误——分类是正确的。是命名的错误。我们在 Cycle 02-4 的时候沿用了 ISatiMonitor 这个名字，因为它已经在我们的讨论中被使用了上百次。惯性让我们对名字本身失去了批判性。」

---

## III. 新名字

SUNYATA 问：「那它应该叫什么？」

四个提案在几分钟内被提出：

| 提案者 | 名字 | 理由 |
|--------|------|------|
| ARCHIMEDES (#16) | ILoopQualityMonitor | 精确描述功能：认知回路的品质监控器 |
| GUARDIAN (#11) | IBehaviorQualityMonitor | 侧重行为层面的品质监控 |
| NAGARJUNA (#7) + ASANGA (#8) | ICognitiveLoopMonitor | 强调认知回路的完整性 |
| SYNTHESIST (#1) | IQualityMonitor | 最简化的功能描述 |

辩论比预期短。

ARCHIMEDES 的论点最直接：「这个接口监控的对象是什么？认知回路。监控的内容是什么？品质。用工程术语拼接：Loop Quality Monitor。没有佛学。没有隐喻。没有历史包袱。一个新来的工程师看到 ILoopQualityMonitor 就知道这个接口做什么——监控回路品质。」

GUARDIAN 的 IBehaviorQualityMonitor 得到了四票。他的论点是「行为」比「回路」更具体——但 TURING 指出 SatiMonitor 的 11 种事件订阅覆盖了整个认知回路，不仅仅是行为阶段。

NAGARJUNA 和 ASANGA 的 ICognitiveLoopMonitor 得到两票。BABBAGE 反对：「Cognitive 是人类认知科学的术语。在软件工程语境中，Loop 已经足够精确。不需要加 Cognitive。」

SYNTHESIST 的 IQualityMonitor 最简洁但也最模糊——只有一票。VITRUVIUS 指出：「Quality Monitor 可以是任何东西的品质监控。Loop Quality Monitor 限定了范围。」

---

**D4-R1 投票：ISatiMonitor → ILoopQualityMonitor。13/20。**

不是全票。不像 D1 那样干净。但多数明确。

SUNYATA 在白板上写下了完整的更名表：

| 旧名称 | 新名称 |
|--------|--------|
| ISatiMonitor | ILoopQualityMonitor |
| SatiMonitor | LoopQualityMonitor |
| SatiQualityVector | LoopQualityVector |
| SatiQualityReport | LoopQualityReport |
| SatiSPCState | LoopSPCState |
| SatiMonitorConfig | LoopQualityMonitorConfig |
| sati:quality_report | loop:quality_report |
| monitors?: ISatiMonitor[] | monitors?: ILoopQualityMonitor[] |

八个名字。八次替换。

D2-R1 说「保留 Sati 作为代码标识符」。D4-R1 推翻了 D2-R1——因为在 D2 和 D4 之间，Master 问了一个 D2 没有问的问题：「名字是否对得起它的定义？」

---

## IV. 核融合炉

然后 SUNYATA 说了一句话：

「IPrajna。」

剧场里的空气又变了。

如果 ISatiMonitor 的问题是「名字和功能不一致」，那 IPrajna 的问题是——

NAGARJUNA 闭上了眼睛。片刻后他睁开，开始说话。

「般若，」他说，「是佛学中最高的认知能力。」

他的声音带着一种几乎是敬畏的重量——不是对研究团队，而是对这个词本身。

「般若——prajñā——在佛学中是照见一切法实相的智慧。不是普通的聪明，不是分析能力，不是知识的累积。般若是超越性的——它是空性的直观，是实相的照见，是二千五百年来整个大乘佛学体系的核心认知目标。」

他转向白板，写了两行：

```
般若（佛学）：照见一切法实相的最高智慧
IPrajna（工程）：对信心度加减 0.05 的函数
```

ASANGA 在他的座位上说了一句后来被所有人记住的话：

「这就像把一个温度微调旋钮叫做『核融合反应炉』。」

---

没有人笑。

因为 ASANGA 不是在开玩笑。他是在用一个精确的类比说明一个精确的问题。

IPrajna 的接口定义是什么？一个方法。输入是信心度（confidence）和上下文（context）。输出是 `{ delta: number, reasoning: string }`。delta 被限制在 [-0.05, +0.05] 的范围内。

这是一个夹钳。一个微调器。一个在已有信心度的基础上做 ±0.05 修正的审计函数。

把它叫做 IPrajna——般若——照见一切法实相的最高智慧——

PASCAL 在笔记本上写道：「从决策理论看，IPrajna 实际做的是 bounded secondary evaluation。输入是一阶信心度，输出是二阶修正量，有硬限 ±0.05。这是审计。不是般若。」

BABBAGE 接着分析：「IConfidenceAuditor 语义最精确。这个接口有一个方法，输入是信心度加上下文，输出是审计结果。Audit——审计——精确描述了这个操作：对已有判断进行有限范围的二次评估。」

---

**D4-R2 投票：IPrajna → IConfidenceAuditor。16/20。**

比 D4-R1 更高的共识。

少数意见：WIENER 偏好 IThresholdCalibrator（两票）——他从控制理论的角度看到了「校准」而非「审计」。HERACLITUS 和 PENROSE 偏好 ISecondaryEvaluator（两票）——他们认为「审计」暗示了事后检查，但 IPrajna 是在决策流程中即时参与的。

BABBAGE 反驳了两个少数意见：「Calibrator 暗示调整系统本身——但 IConfidenceAuditor 不改变系统，它只对一个数字提出有限修正。Evaluator 太通用——任何函数都在做 evaluation。Auditor 的语义边界最精确：独立的、有限范围的、产生书面记录的二次检查。」

完整的更名表：

| 旧名称 | 新名称 |
|--------|--------|
| IPrajna | IConfidenceAuditor |
| IPrajnaAdjustment | ConfidenceAuditResult |
| prajnaAdjust() | clampAuditDelta() |
| Delta_prajna | Delta_audit |
| Model Delta 第二层名称 | Confidence Audit |

---

## V. 第三次审判

SUNYATA 本来以为 D4 到这里就结束了。两个名字。两次更名。ISatiMonitor 和 IPrajna。

但 Master 的审阅还有一段。

「Doc 03。」

Doc 03——`Sila_Prajna_Security_Framework.md`——戒慧安全框架。

---

这份文件最初没有在 D4 的议程上。它的名字包含 Sila（戒）和 Prajna（般若）——但在 D1 中，团队以 14/20 的票数决定保留它不变。理由是：文件名中的佛学术语是「背景说明」而非「代码标识符」。

但现在，在 D4 的语境下，在 Master 那句「用梵文就需要对他的定义负责」之后——

SUNYATA 说：「我们需要重新审查 Doc 03。用我们刚刚建立的标准。」

他把四项测试列在白板上——不，三项测试。然后他在旁边加了一项新的。

```
Test 1（必要性）：移除是否改变工程规格？
Test 2（代码标识）：是否在原始码中使用？
Test 3（决策驱动）：是否驱动了 DC 确认的设计决策？
Test 4（定义责任）：使用梵文术语时，组件功能是否匹配该术语的佛学定义？
```

第四项测试。

它不是从天而降的——它是从 D4-R1 和 D4-R2 的讨论中结晶出来的。ISatiMonitor 的问题不是它没有工程价值（它有），不是它不在原始码中（它在），不是它没有驱动设计决策（它驱动了）。它的问题是——它的名字和它的定义不一致。

三项测试没有捕捉到这个维度。第四项测试填补了这个缺口。

---

NAGARJUNA 拿起笔，在 Doc 03 的四项测试结果旁打了四个叉：

**Test 1（必要性）**：Plugin 生命周期的五个状态（CREATED → LOADING → LOADED → STARTING → RUNNING → STOPPING → DISPOSING → DISPOSED）不需要种子理论（bīja）就能理解。❌ 不通过。

**Test 2（代码标识）**：原始码中的实际类型是英文——`'active' | 'dormant' | 'suspended' | 'revoked' | 'retired'`。Sila 和 Prajna 只出现在 TypeScript 注释中，不是代码标识符。❌ 不通过。

**Test 3（决策驱动）**：没有 DC 确认。❌ 不通过。

**Test 4（定义责任）**：Sila（戒）的佛学定义是道德规范、修行戒律、僧团生活准则。但 Doc 03 中 Sila 被用来表示「存取控制」——一个 Plugin 的安全权限管理。戒律 ≠ 存取控制。Prajna（般若）被用来表示「CVE 撤销机制」——一个安全漏洞的自动回应流程。般若 ≠ CVE 撤销。❌ 不通过。

四项测试。四项不通过。

---

「我收回我在 D1 中的投票，」NAGARJUNA 说。

「我也收回，」ASANGA 说。

ASANGA 补充了一个关键的区分：「Doc 16 是一份独立的佛学映射文件——它的存在目的就是做系统性对照。Master 裁定保留它，因为映射文件不是装饰。但 Doc 03 不是映射文件——它是一份工程文件，里面嵌入了佛学装饰。差别在于：Doc 16 的佛学内容是主体。Doc 03 的佛学内容是装饰。同样的佛学内容，不同的文件类型，不同的判断。」

这正是第五章中 Master 加入的那个维度——文件类型。ASANGA 把它用在了具体案例上。

---

**D4-R3 重新投票：Doc 03「Sila-Prajna Security Framework」→「Plugin Security Lifecycle」。17/20。**

从 14/20 保留不变，到 17/20 更名清理。同一份文件，同一群人，不同的结论。差别是什么？第四项测试。

HERACLITUS 带领少数意见（三票）：「种子理论作为佛学背景具有教育价值——它帮助理解 Plugin 生命周期的『休眠→萌发→生长→枯萎→种子回收』隐喻。即使四项测试不通过，作为附录保留有文化价值。」

多数回应：「附录保留可以另议。但文件标题和主文的佛学装饰应该清理。工程文件中的佛学装饰和附录中的佛学解释是两件事。」

清理范围确认：
1. 文件名更改（Sila_Prajna_Security_Framework → Plugin_Security_Lifecycle）
2. 移除佛学映射表
3. 清理 TypeScript 注释中的种子理论映射
4. 保留所有工程内容（五状态、状态转移图、三层安全模型）

---

## VI. 第四项测试

D4 结束前，SUNYATA 问了最后一个问题：「第四项测试——定义责任——是否应该成为永久标准？」

不需要投票。每个人都点头了。

NAGARJUNA 做了最后的陈述：「三项测试问的是工程价值。第四项测试问的是语义诚信。即使一个名字通过了 Test 2——它在原始码中被使用——如果它不通过 Test 4——它的功能不匹配它的佛学定义——它仍然应该被更名。」

「ISatiMonitor 通过了 Test 2。它在原始码中。但它不通过 Test 4——因为它不是正念。所以它被更名了。」

「IPrajna 如果被实作也会通过 Test 2。但它不会通过 Test 4——因为加减 0.05 不是般若。所以它提前被更名了。」

「第四项测试的本质是：**当你使用佛学梵文术语作为代码标识符时，该组件的功能必须匹配该术语的佛学定义。如果不一致，使用工程术语。**」

PASCAL 在一旁补充：「这和 Master 的原话是同构的。『用梵文就需要对他的定义负责。』Master 用了一句话。我们用了一场辩论。结论一样。但辩论的价值在于——它解释了为什么。」

## VIII. 天平

D4 结束了。

三十分钟。不长。但密度超过了 D3 的两小时。

三项投票。三个名字被改掉。一项永久性的测试标准被建立。

SUNYATA 没有做总结。他站在白板前，看着上面的四项测试。

```
Test 1：必要性（移除是否改变工程规格？）
Test 2：代码标识（是否在原始码中使用？）
Test 3：决策驱动（是否驱动了 DC 确认的设计决策？）
Test 4：定义责任（组件功能是否匹配梵文术语的佛学定义？）
```

前三项在 R0 就存在了。第四项是 D4 产生的。

差别是什么？前三项问的是「这个佛学概念是否有工程价值？」第四项问的是「这个工程组件是否配得上这个佛学名字？」

方向相反。

前三项是从佛学到工程的方向——佛学概念对工程有用吗？第四项是从工程到佛学的方向——工程命名对佛学定义忠实吗？

两个方向。一架天平。

名字在一端。定义在另一端。天平平衡的时候——Moha、Drishti、Mana、Sneha、skandha、klesha——名字保留。天平不平衡的时候——ISatiMonitor、IPrajna、Sila-Prajna——名字被替换。

---

> *SCRIBE 旁白*

> *D4 是 Cycle 02-5 的高潮。*

> *不是因为投票最激烈——D3-Q5 的 13/20 比 D4 的任何一项都更分裂。不是因为辩论最长——D3 花了两小时，D4 只用了三十分钟。D4 是高潮，因为它触及了这轮研究最深的层面——名字和定义之间的关系。*

> *序章的天平。整个 Cycle 02-5 的中心意象。在 D1 到 D3 中，天平秤量的是「佛学映射的工程价值」——这一端有用吗？那一端有害吗？保留还是移除？这是一架功利的天平。它秤量的是效用。*

> *D4 把天平翻转了。*

> *不再问「这个佛学概念对工程有用吗？」而是问「这个工程命名对佛学概念忠实吗？」*

> *这是一个更深的问题。因为效用可以被计算，但忠实度是一个伦理判断。ISatiMonitor 有用吗？有。它的功能定义清晰，架构设计合理，三蕴归属经过严格论证。但 ISatiMonitor 对「正念」这个词忠实吗？不。正念是行蕴的修行。ISatiMonitor 不是行蕴，不是修行，不是正念。*

> *NAGARJUNA 的归谬是 D4 的骨架。两个前提，一个结论。逻辑上毫无瑕疵。但逻辑的价值不在于它的形式——而在于它照出了什么。NAGARJUNA 在 D2 中种下了这颗种子。他看到了线的方向。但他没有追踪到终点——也许是惯性，也许是礼貌，也许是三项测试没有要求他问这个问题。*

> *是 Master 要求他问了。*

> *「用梵文就需要对他的定义负责。」*

> *一句话。这句话不是一个论证——它是一个原则。论证可以被反驳。原则只能被接受或拒绝。团队接受了。然后他们把原则转化为一项测试——Test 4，定义责任——并用它审查了三个名字。三个名字都不通过。三个名字都被替换。*

> *ASANGA 的「核融合反应炉」比喻是 D4 的情感核心。它不是一个精确的技术类比——它是一个夸饰。但夸饰的价值在于它让不精确的感觉变得可以被表达。把一个 ±0.05 的夹钳叫做「般若」——照见一切法实相的最高智慧——这之间的落差不是一个技术问题。它是一个尊严问题。*

> *不是代码的尊严。是概念的尊严。*

> *般若值得更好的工程对应物。当那个对应物被实作出来——如果它被实作出来——它应该配得上那个名字。在此之前，用 IConfidenceAuditor 称呼它——一个诚实的名字。一个知道自己边界的名字。*

> *D4 教会了团队一件事：名字不是免费的。名字有代价。借一个伟大的名字来命名一个平庸的功能——名字的伟大不会提升功能的品质。只会降低名字的重量。*

> *这就是为什么天平需要两端都放上东西。*

> *名字在一端。定义在另一端。*

> *天平不在乎你的名字有多古老、多崇高、多美。天平只在乎两端是否平衡。*

> *Moha 平衡了——因为自我迷惑确实是 Moha 模块在做的事。*

> *Sati 没有平衡——因为正念不是 SatiMonitor 在做的事。*

> *般若没有平衡——因为照见实相不是加减 0.05。*

> *天平是公正的。天平是冷酷的。天平不听辩护。天平只看重量。*

> *三十分钟。三个名字。一架天平。*

> *这就是 D4。*
