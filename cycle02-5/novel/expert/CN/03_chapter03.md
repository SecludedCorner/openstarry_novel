# 第三章：三面之镜

---

## I. Master 的刺

D2 的开场不是 SUNYATA 的提问。是 Master 的声音。

不是 Master 本人在场——Master 从来不在辩论中现身。在六轮研究中，Master 的存在方式始终是间接的：信件、回复、确认记录。他像一个不可见的引力源，影响着每一个天体的轨道，但从不在观测者的视野中出现。

他的话被投影在屏幕上。白底黑字。冷白色灯光下，那些字看起来像是刻在石头上的。

> 「觉察当下正在发生什么。这一层不看规则、不看风险分类，纯粹观察行为模式。不需要将它当作正念的工程化。而且这个 plugin 难道没有识蕴的成分在吗？」

SUNYATA 把最后一句话画了底线。用红色的笔。用力地。红色的线穿透了纸面，在下一页留下了一道浅浅的压痕。

**「难道没有识蕴的成分在吗？」**

他在旁边写了一个问号。不是普通的问号——是他在六轮研究中只用过三次的那种问号。大的。红色的。问号的钩子几乎占了整行。

这不是反问。Master 从不做反问。SUNYATA 在五封信中学到了这一点——Master 的每一个问号都是一个具体的、带有明确方向性的问题。他已经有了答案，但他要求研究团队自己找到它。

Master 的方向性很清楚：SatiMonitor 不只是行蕴的功能。它有识蕴的成分。研究团队的任务是找出确切的蕴组成。

D2 的议题：**这个「观察行为模式」的 plugin 该叫什么、由哪些蕴组成？**

---

## II. 功能的解剖

在 D2-Q1（命名）之前，SUNYATA 做了一个不寻常的决定：他让 TURING 先做功能简报。

不是辩论——是事实陈述。

TURING 站了起来。他站起来的方式和 NAGARJUNA 不同——没有庄重感，没有戏剧性。只是站起来了。像一台机器被启动。

「C1 研究报告。SatiMonitor 的工程功能分析。」他说。然后他打开了一张投影片。投影片上没有图表，没有公式，只有四行字和一行「不做」：

```
功能 1：事件订阅 — 透过 EventBus.subscribe() 持续接收系统事件
功能 2：滑动窗口 — 维护最近 N 个事件的时间窗口
功能 3：模式辨认 — 在窗口中识别行为模式（突变、趋势、周期）
功能 4：品质指标 — 输出四维品质向量（连续性、粒度、速度、平稳性）

不做  ：不执行任何行动。不修改任何状态。不发出任何指令。
```

「最后一行是最重要的。」TURING 说。「SatiMonitor 不做任何事。它只看。」

GUARDIAN 在他的安全备忘录上记下了「不做」两个字。然后他在旁边写了一个 TURING 不会写的比喻：「APM Agent + Behavioral Analytics + QoS Monitor + SPC Anomaly Detector 的混合体。」

TURING 看了一眼 GUARDIAN 的笔记。他没有说那个比喻不好——他只是微微皱了一下眉，像一个正在校准精度的工程师看到了一个近似解。

---

## III. 命名的收敛

D2-Q1。文件标题。

原名带着 Sati 的标记——像一件衣服上绣了一个品牌 logo。问题是：这个 logo 还合适吗？

TURING 已经在 C1 报告中用纯工程术语描述了全部功能。四个功能，没有一个需要佛学概念来解释。事件订阅是 EventBus 的标准模式。滑动窗口是统计学的基本工具。模式辨认是数据分析的核心能力。品质指标是工业品管的日常操作。

GUARDIAN 提出了一个微妙的需求。他的声音带着安全文件特有的严谨——每一个建议都像是一份经过多轮审查的安全咨询。

「安全文件中需要全称。」他说。「在安全上下文中，我建议使用完整的、明确的名称。如果我们决定保留 Sati 作为代码标识符，那在安全文件中应该标注全称——例如 SatiMonitor (Cognitive Loop Quality Monitor)。」

团队很快收敛。十分钟之内。

文件标题改为「认知回路品质监控架构 (Cognitive Loop Quality Monitoring Architecture)」。Sati 作为代码标识符保留——SatiMonitor 在源代码中存在，改名有迁移成本——但不再作为文件的定义性名称。

D2-R1：改名。19/20。

GUARDIAN 投了 B——他偏好完全改名为 BehaviorQualityMonitor。但他的安全文件全称需求被纳入了附带条件。一票少数意见。干净的。

## IV. 受蕴的厚度

D2-Q2。

这是 D2 的核心。也是 Cycle 02-5 前三场辩论中最激烈的交锋。

**「SatiMonitor 由五蕴的哪些类别组成？」**

C2 的研究报告——ASANGA 和 LINNAEUS 的联合研究——提出了四个方案。每一个方案背后都有一套逻辑，每一套逻辑都能站住脚。至少在提出的时候。

| 方案 | 蕴组成 | 理由 |
|------|--------|------|
| A | 受 + 想 | 感知 + 辨认，最简 |
| B | 受 + 想 + 识 | 感知 + 辨认 + 评估判断 |
| C | 想 + 识 | 辨认 + 评估，受蕴太薄 |
| D | 纯识 | 一切认知活动归识蕴 |

Proposal D 在 R2 交叉审阅时就被淘汰了——「纯识」等于放弃分类，把所有东西都扔进一个桶里。BABBAGE 在交叉审阅中写了一句简洁的否决：「Proposal D 将 SatiMonitor 的蕴归属退化为平凡解。平凡解在数学中是有效的，但在分类学中是无用的。」

Proposal A 也很快失势——缺少识蕴的评估维度，无法回答 Master 的问题。Master 明确问了「难道没有识蕴的成分在吗？」——一个排除识蕴的方案在问题提出的那一刻就已经被否决了。

战线在 Proposal B 和 Proposal C 之间拉开。

受蕴——包含，还是不包含？

---

MESH 和 KERNEL 代言 Proposal C。

MESH 先开口。他的论点带着分散式系统专家对「最小化」的偏好——在分散式系统中，最小化是一种美德。最少的节点。最少的消息。最少的假设。

「SatiMonitor 的事件订阅不是真正的『感受』。」他说。「受蕴（vedana）在佛学中是苦、乐、不苦不乐的质性体验。SatiMonitor 接收事件不是在感受——它是在收数据。接收数据是认知的输入阶段，但不是感受。」

KERNEL 补充了 OS 的类比：「系统调用收到一个 packet 不是在感受——它是在接收 I/O。感受是对 I/O 的质性评价。SatiMonitor 对事件有质性评价吗？有苦乐的判断吗？没有。它只是收数据。」

然后 MESH 抛出了他最强的论点——对称性。

「去掉受蕴，保留想蕴和识蕴。SatiMonitor 的蕴组成变成 ['samjna', 'vijnana']——和 IGearArbiter 的蕴组成完全对称。两个复合 plugin，相同的蕴组成。对称性是好的分类法。」

**「对称性是好的分类法。」**

这句话在剧场里回荡了一下。对称性在科学中有一种几乎宗教性的魅力——从物理学的守恒律到数学的群论，对称性意味着深层的结构一致性。如果两个不同的 plugin 有相同的蕴组成，也许这告诉我们什么关于认知结构的深层模式？

LINNAEUS 站了起来。

他的站姿不像 NAGARJUNA 的庄重——更像一个田野考察的博物学家在观察到一个有趣的物种后起身仔细端详。他走到白板前。LINNAEUS 是分类学家——他的整个存在意义就是把东西分到正确的位置。

「对称性是好的——」他开始。他的语气带着 Linnaeus 特有的温和——像在田野中和一只蝴蝶说话。「当对称性反映了事实的时候。」

他在白板上画了两条时间线。并排的。上面的标了「SatiMonitor」，下面的标了「IGearArbiter」。

```
SatiMonitor:  ─────[持续订阅]─────[持续订阅]─────[持续订阅]─────
                    主动、恒常、不间断
                    ← 始终在看 →

IGearArbiter: ────○────────────○────────────○────────────
                  被叫来看一眼    被叫来看一眼    被叫来看一眼
                  被动、间歇、按需
                  ← 被叫去看 →
```

「看这两条线。」他指着白板。「它们有什么不同？」

他不等回答。分类学家提问是为了引导观察，不是为了寻求答案。

「SatiMonitor 是持续的。它订阅 EventBus，它维护滑动窗口，它**永远在那里**。不是有人来问它才启动——它一直在运行。从 ExecutionLoop 的第一个事件到最后一个事件，SatiMonitor 的事件流从未中断。」

「IGearArbiter 是间歇的。有请求来了，ManoAggregator 问它：『你能处理吗？』它看一眼，给出判断，然后沉默。直到下一次被叫。」

他转身面对 MESH。

「**『始终在看』和『被叫去看』——这是根本的特征差异。** SatiMonitor 和 IGearArbiter 不是同类的。它们的行为模式——使用 OOP 的语言——完全不同。一个是持续观察者（continuous observer），一个是按需评估者（on-demand evaluator）。」

「把它们分到相同的蕴组成，就像把蝴蝶和蜂鸟分到同一个物种——因为它们都会飞。但蝴蝶是昆虫，蜂鸟是鸟类。飞行是表面的相似，不是深层的同源。对称性不是同源性。」

---

WIENER 从他的控制理论笔记中抬起头。他很少在辩论中主动发言——他更喜欢听完所有人的意见，然后在恰当的时刻提供一个控制理论的视角，像在一幅画上加最后一笔。

「在控制理论中，」他说，声音带着维也纳学派特有的精确，「有一个概念叫做观察者（observer）——Luenberger observer。观察者的核心功能是什么？持续感知系统的状态。注意——**持续**。」

他站了起来。走到 LINNAEUS 的白板旁边。

「观察者的感知不是附加功能。」他说。「它是身份定义的一部分。一个不感知的观察者，就不是观察者。它是一块不连接任何输入的电路板——物理上存在，功能上不存在。」

「SatiMonitor 的事件订阅——它的持续感知——是它作为观察者的身份定义。你不能把这个功能剥离出去，然后说它不属于受蕴。因为如果你剥离了它，SatiMonitor 就不存在了。」

---

HERACLITUS 补了一刀。他是运行时动态专家——他关心的不是静态的类型定义，而是动态的行为模式。

他打开了 C1 报告中 SatiMonitor 的四阶段退化策略：

```
Normal    → Throttle  → Sample   → Emergency
100%        50%         25%        最低监控
全速感知    减半感知    取样感知    紧急最低限
```

「看这个退化策略。」他说。声音带着对流动性的敏感——HERACLITUS 看到的不是静止的结构，而是变化的过程。

「Normal。Throttle。Sample。Emergency。这四个阶段描述的是什么？是感知密度的渐进降低。100%、50%、25%、最低限——这是一个感知系统在不同压力下调节自己感知强度的方式。」

他指着白板。

「想蕴（辨认）和识蕴（评估）不需要退化策略。它们的输出是离散的——辨认一个模式，或不辨认。评估一个判断，或不评估。离散的输出不需要渐进降低。」

「受蕴的感知是连续的。连续的东西才需要退化策略——因为你不能突然从 100% 跳到 0%，你只能渐进地降低。Normal → Throttle → Sample → Emergency——这是受蕴在压力下保护自己的方式。就像人在极度疲劳时，感知会变得迟钝——不是感知消失了，而是感知密度降低了。」

三个论点。三个维度。三面镜子。

LINNAEUS 的镜子照出了**行为特征**——持续 vs 间歇。
WIENER 的镜子照出了**身份定义**——感知是观察者的存在条件。
HERACLITUS 的镜子照出了**退化模式**——连续感知才需要渐进退化。

每一面镜子都反射出同一个结论：SatiMonitor 有受蕴的成分。

## V. 零成本转折

然后 ARCHIMEDES 做了一件改变辩论走向的事。

他没有加入哲学的讨论。他没有引用佛学的经典。他打开了 PluginHooks 的类型定义。

```typescript
interface PluginHooks {
  sensors?: IVedanaSensor[];         // 受蕴
  recognizers?: ISamjnaRecognizer[]; // 想蕴
  arbiters?: IGearArbiter[];         // 想蕴+识蕴
  tools?: ITool[];                   // 行蕴
  volition?: IVolition;              // 识蕴
  // ... monitors 尚未实作 ...
}
```

「三蕴还是两蕴——有工程差异吗？」他问。

他自己回答了。用他一贯的方式——不是论证，是计算。

「monitors 是一个独立的数组插槽。无论 SatiMonitor 被标记为 `['samjna', 'vijnana']` 还是 `['vedana', 'samjna', 'vijnana']`，PluginHooks 的实作完全相同。没有额外的 hook。没有额外的布线。没有额外的测试。」

他看了全场一眼。

「**零额外工程成本。**」

这句话像一块石头丢进了辩论的水面。

如果三蕴和两蕴的工程成本相同——没有额外的 hook，没有额外的布线，没有额外的测试——那么这场辩论就不再是工程权衡。它变成了纯分类问题。

在纯分类问题上，分类学家有压倒性的发言权。

LINNAEUS 的「持续感知 vs 被动接收」。
WIENER 的「观察者身份定义」。
HERACLITUS 的「退化策略只属于连续感知」。

三面镜子。三个维度。零工程成本。

Proposal C 失去了它唯一的工程论据。

---

投票。

D2-R2：**Proposal B — [vedana, samjna, vijnana]。18/20。**

MESH 和 KERNEL 投了反对票。他们的「对称性」论点被记录为有价值的方法论观察——但事实胜于对称。

18/20 意味着什么？在一个 20 人的团队中，18 票赞成代表 90% 的共识。对于一个涉及跨学科分类的问题——哲学家、佛学家、工程师必须同时同意——90% 是非常高的。

SatiMonitor 成为 OpenStarry 首个三蕴 plugin。

## VI. 行蕴的排除

但 D2-R2 最重要的结论不是「包含哪些蕴」。

是「排除行蕴」。

ASANGA 站了起来。他在 D2 中一直安静地听着——从 TURING 的功能陈述到 LINNAEUS 的受蕴论证。唯识学家的方法是先观察（现量），再推理（比量），最后立论。他已经完成了观察和推理。现在是立论。

「佛学传统中，正念（sati）被归入行蕴。」他说。声音带着唯识学特有的精密度——像在显微镜下调焦。「五十一心所之一。属于善心所。这个分类持续了两千五百年。」

他在白板上写了两列。工整的。精确的。每一个字都被放在准确的位置上。

```
佛学正念 (sati)                    SatiMonitor
━━━━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━━
意志驱动 (精勤 ātāpī)              事件驱动 (EventBus.subscribe())
有道德价值 (善 kusala)              价值中立 (品质指标)
主动的专注 (不放逸 appamāda)        被动的观察 (subscribe + listen)
正向运转的心理状态                   对行为模式的计量
行蕴的善心所                        ≠ 行蕴活动
```

「看左边。」他说。「佛学正念是什么？它是**意志驱动的**——你必须努力保持正念。『精勤（ātāpī）』是正念的前提条件——没有努力就没有正念。它是**有道德价值的**——正念是善的。在阿毗达磨的分类中，正念属于善心所，不属于遍行心所（每一个认知过程都有的心所），也不属于不善心所。它是一种特定的、正向的心理品质。」

「看右边。」他的手指移到了表格的右半部。「SatiMonitor 是什么？它是**事件驱动的**——它不需要努力。它订阅 EventBus，事件来了就处理，不来就不处理。它没有 ātāpī，没有精勤。它是**价值中立的**——它的四维品质向量（连续性、粒度、速度、平稳性）不做善恶判断。品质高不等于善。品质低不等于恶。它只是测量。」

他停顿了一下。让两列的对比在每一位学者的脑中自行产生化学反应。

「SatiMonitor 不是正念的工程化。」他的声音变得更低，更清晰。「正念是意志驱动的——你必须努力保持正念。SatiMonitor 是事件驱动的——它不需要努力，它只需要订阅。正念有道德价值——正念是善的。SatiMonitor 没有道德价值——它只是测量。」

然后他说了最关键的一句话：

「千年来，佛学把正念归入行蕴。这个分类是正确的——**对佛学而言**。但 SatiMonitor 不是正念。它是行为模式的品质监控器。它属于受（感知）、想（辨认）、识（评估）。**不属于行（意志驱动的善心所）。**」

---

这是一个佛学家用佛学的内部逻辑，证明了一个工程组件不应该使用佛学的分类。

不是外部反驳——不是「佛学的分类不适用于工程」这种粗暴的二分法。是内部论证——佛学自己的判准（善心所 vs 遍行心所、意志驱动 vs 事件驱动）排除了 SatiMonitor 与正念的等同性。

NAGARJUNA 在他的座位上微微点头。中观学者认出了唯识学者的方法——不是从外面攻击一个立场，而是从内部展示它的自我矛盾。这是 prasanga——归谬法——的变体。不是「你的结论错了」，而是「你自己的前提排除了你自己的结论」。

## VII. 第三个插槽

D2-Q3 是技术性的——但技术性不代表不重要。

**PluginHooks 是否需要新增 `monitors` 插槽？**

KERNEL 从 OS 的角度回答。他的声音带着作业系统专家特有的架构感——每一个组件都有它的位置，每一个位置都有它的生命周期。

「monitors 有独立的生命周期。」他说。「它们在回圈启动时开始（MonitorRegistry.startAll()），在回圈结束时停止（MonitorRegistry.stopAll()）。这和 sensors（每次事件触发）或 tools（按需调用）完全不同。独立的生命周期需要独立的管理。」

他在白板上画了一个生命周期图：

```
                 PluginLoader.load()
                        │
    ┌───────────────────┼───────────────────┐
    │                   │                   │
  sensors           monitors             tools
  (per-event)    (loop lifecycle)     (on-demand)
    │                   │                   │
  每次事件触发    LoopStart → LoopEnd    按需调用
```

「三种不同的触发模式。三种不同的生命周期。不应该混在一起。」

```typescript
interface PluginHooks {
  // ... existing hooks ...
  monitors?: ILoopQualityMonitor[];  // NEW: 独立生命周期
}
```

D2-R3：新增 monitors 插槽。20/20。全票。

---

三项投票。三个方向。

19/20。18/20。20/20。

SatiMonitor 的文件标题被改了。SatiMonitor 的蕴归属被决了。SatiMonitor 的 PluginHooks 插槽被建了。

每一个决议都回答了 Master 的一部分问题。

「难道没有识蕴的成分在吗？」

有。

它由三蕴组成——受蕴（持续感知）、想蕴（模式辨认）、识蕴（品质评估）。不包含行蕴。因为它不是正念。它是观察。

> *D2 最重要的不是蕴归属的结论。*

> *D2 最重要的是 ASANGA 那张对照表——「佛学正念」vs「SatiMonitor」。五行。十个对比。左边是两千五百年的传统，右边是 v0.28.0-alpha 的源代码。*

> *这张表是整个 Cycle 02-5 的分水岭时刻。在这张表之前，研究团队一直在问「SatiMonitor 是哪些蕴的组合？」在这张表之后，问题变成了「SatiMonitor 和佛学正念是同一件事吗？」*

> *答案是不。*

> *这个「不」看起来简单。但它的意涵深远得像地下水脉——在表面之下流过整片地形。如果 SatiMonitor 不是佛学正念，那么 ISatiMonitor 这个名字就有问题。如果 ISatiMonitor 不是正念（行蕴心所），而是一个受+想+识的品质监控器，那么用 Sati 来命名它——*

> *但这个推论要到 D4 才会被完成。此刻，在 D2 结束的安静中，没有人把这根线拉到底。NAGARJUNA 没有。ASANGA 没有。TURING 没有。他们看到了线的方向，但没有追踪到终点。*

> *为什么？*

> *也许是因为惯性。ISatiMonitor 这个名字已经在九份研究报告中使用了上百次。它已经成为了一个「自然的」称呼——当你用一个名字称呼一个东西足够多次，名字就会和东西融为一体，你不再注意到名字本身是否合适。*

> *也许是因为礼貌。SatiMonitor 是某些研究者的心血——提出更名是一件社交上微妙的事。*

> *也许是因为——更深刻地——三项测试没有问这个问题。Test 1 问「移除是否改变规格？」Test 2 问「是否在源代码中使用？」Test 3 问「是否驱动了设计决策？」没有一项测试问「名字是否和定义一致？」*

> *三面镜子——受、想、识——照出了 SatiMonitor 的真实面貌。但镜子只照出形状，不照出名字。*

> *名字的问题，要等 Master 亲自来问。*

> *三面镜子足以看清形状。但名字的重量，需要另一架天平来衡量。*
