# 第五章：裁定

---

D3 结束四十分钟后。

圆形剧场的冷白灯光下，二十位研究者在做一件奇怪的事——他们什么都没做。

D1 到 D3 之间的每个间隔都被塞满了：交叉审阅、数据比对、投影片准备、咖啡。但 D3 之后的空白是不同的。不是休息。是等待。因为每个人都知道接下来会发生什么——Master 的审阅。

在前几轮 Cycle 中，Master 的介入通常发生在特定的时间节点：研究开始前（信件）、某个决议引发争议后（DC-1 到 DC-12）、或者整轮结束时（路线图建议）。但 Cycle 02-5 不一样。Master 在信中明确说了他要看什么——清理结果。他要看的不是抽象原则——他要看具体执行。

---

## I. 审阅

Master 的审阅不是一份书面报告。

它以一种更直接的方式出现。SUNYATA 在他的终端上看到了一段话，然后他把投影幕打开，让所有人都看到。

审阅的篇幅不长。但它精确得像手术刀。

Master 首先确认了大部分 D1 决议——四层框架、三项测试、Batch A 的五项 REMOVE、Batch B 的八项 RELOCATE、Batch C 的七项 KEEP、Doc 38 表格清理、Doc 44 三学映射移除、Klesha 模块名保留。

「这些都正确。」

然后他提到了两个名字：Doc 16 和 Doc 31。

---

## II. 推翻

「D1-Q7 和 D1-Q8，」SUNYATA 念出 Master 的原话，「这两项决议我不同意。」

剧场里的空气凝固了。

D1-Q7：Doc 16 的拆分——将 Section 5 提取为独立文件，其余删除。20/20 通过。

D1-Q8：Doc 31 的拆分降级——Section 3.1 移入 Doc 35，Section 6.3 独立为短文件，其余降级为 Appendix_B。20/20 通过。

两个 20/20。两个全票。在本轮历史上最干净的辩论中以完美的共识通过。

Master 推翻了它们。

---

SUNYATA 继续念：

「Doc 16 不是一份嵌入了佛学的工程文件。Doc 16 本身就是一份佛学映射文件——它的存在目的就是做佛学到工程的系统性对照。你们的三项测试是正确的，四层框架也是正确的，但你们在 Doc 16 上用错了标准。」

ARCHIMEDES 低头看他的审计表。他在 B1 中给 Doc 16 标记了「~80% 装饰性」。80%——他数过。一行一行数的。但数字正确不代表结论正确。

「Doc 31 也一样，」Master 继续，「八识运作模型是一份以佛学概念为主体的分析性文件。它不是工程文件里的装饰——它就是一份佛学参考文件。你们需要区分两种情况：第一种是工程文件中嵌入的佛学装饰——应该清理。第二种是以佛学映射为目的的独立文件——它有工程参考价值，应该保留。」

---

NAGARJUNA 是第一个站起来的。

「Master 指出了我们框架的盲点，」他说。他的声音没有颤抖。中观学者被告知自己的分析框架有缺口——这在他的传统中不是耻辱，而是日常。龙树本人的全部哲学建立在「一切见皆可遣除」之上——包括自己的见。

「三项测试问的是：映射是否改变工程规格？是否是代码标识符？是否驱动设计决策？但这三个问题假设了一个前提——被审查的文件是工程文件。如果文件本身就是佛学映射文件，那么用工程文件的标准来衡量它——」

「就像用散文比例来评判诗，」SCRIBE 在记录中写道。

---

## III. 维度

PASCAL 在他的笔记本上画了一张图。

三项测试是三个轴——Necessity、Code Identity、Decision Driver。每一份文件在这三个轴上有一个坐标。D1 的判断基于这个三维空间。

但 Master 指出的是第四个轴：**文件的存在目的**。

Doc 38 是一份工程文件（IVolition 的架构设计），其中嵌入了一些佛学映射。三项测试适用。结论：清理嵌入的装饰，保留驱动设计的概念。

Doc 16 是一份佛学映射文件（Plugin 组合模式的佛学对照），它的存在目的就是提供跨学科参考。三项测试不适用——因为它不是「工程文件中的佛学装饰」，而是「佛学映射本身就是文件的内容」。

差别在于：装饰和内容。

一面墙上的壁画是装饰——你可以移除它而不影响墙的结构。但一幅挂在美术馆里的画不是墙的装饰——画就是内容本身。移除它不是「清理」，而是「拆除」。

「Doc 16 和 Doc 31 是挂在美术馆里的画，」PASCAL 说，「不是贴在工程图纸上的壁画。」

---

ARCHIMEDES 重新打开他的审计表。

「B1 审计的前提假设是：所有被审计的文件都是工程文件，其中可能嵌入了佛学装饰。我按照这个前提逐行审计，标记装饰比例。Doc 16 的 80% 和 Doc 31 的 70%——数字是对的。但前提是错的。」

他在审计表上加了一个新的栏位：**文件类型**。

| 文件 | 类型 | 三项测试适用？ |
|------|------|----------------|
| Doc 38 | 工程文件 | 是 |
| Doc 43 | 工程文件 | 是 |
| Doc 44 | 工程文件 | 是 |
| Doc 37 | 工程文件 | 是 |
| Doc 41 | 设计决策文件 | 是 |
| **Doc 16** | **佛学映射文件** | **否** |
| **Doc 31** | **佛学映射文件** | **否** |

七份文件。五份工程文件。一份设计决策文件。两份佛学映射文件。

前五份适用三项测试。第六份因为记录设计理由而需要特殊处理。最后两份——不是清理的对象。

---

## IV. 修正

SUNYATA 站在白板前。

「团队的回应，」他说。

不需要投票。Master 的裁定不是议案——它是裁定。但 SUNYATA 想确认每个人都理解了原因，而不只是结果。

NAGARJUNA 先说：「中观的两谛框架——世俗谛和胜义谛——我用它建议将工程文件中的佛学内容分离到附录。但 Doc 16 和 Doc 31 不是世俗谛文件中的胜义谛注释——它们本身就是在做两谛之间的系统性映射。映射文件不属于任何一个层面。它们是连接两个层面的桥梁。删除桥梁不是清理——是破坏。」

ASANGA 接着说：「我用唯识学的功能定位法——如果佛学概念没有驱动工程决策，它就是装饰。但 Doc 16 整份文件的功能就是提供佛学到工程的映射。用『驱动工程决策』来评判一份映射文件，就像用『是否能打钉子』来评判一面镜子——标准选错了。」

TURING 说：「从源代码验证的角度，Doc 16 的 Section 5（IObserver 组合模式）确实有直接的工程价值。但 Master 的裁定不是说 Section 5 应该被提取——而是说整份文件都应该保留。这意味着其余部分的学术参考价值也被认定了。」

GUARDIAN 说：「安全角度——Doc 31 的八识运作模型包含 ConcurrentSenseProcessing 的分析，这对多通道安全审计有参考意义。我在 D1 时没有反对拆分，因为我同意 Cocycle condition 可以独立。但 Master 的裁定让我重新考虑：拆分之后，上下文消失了。Cocycle condition 在八识框架中的意义和在独立文件中的意义不一样。」

---

SUNYATA 在白板上写下了修正：

```
D1-Q7 (Doc 16): 20/20 拆分 → Master 裁定恢复原状
D1-Q8 (Doc 31): 20/20 拆分 → Master 裁定恢复原状
理由：佛学映射文件 ≠ 工程文件，不适用三项测试
```

然后他在下面加了一行：

```
四层框架更新：
KEEP / RELOCATE / REMOVE / FILE REVIEW → 加入前提检查
前提：文件类型识别（工程文件 / 设计决策 / 佛学映射）
```

三项测试没有被否定。它们只是获得了一个适用条件。就像一把精密的卡尺——它能测量任何零件的尺寸，但你首先得确认你测量的是零件，而不是设计图纸本身。

---

## V. 种子

SCRIBE 在记录中写下了这一段。

D1 的 SCRIBE 旁白中有一段话：「D1 有一个隐藏的种子。D1-Q7 和 D1-Q8 的拆分决策——后来，Master 亲手推翻了这两项决议。理由不是团队的判断错了，而是团队的判断框架还少了一个维度。」

种子现在发芽了。

但 SCRIBE 知道——Master 的审阅不只是推翻两项决议。Master 审阅的真正重量在于他没有展开的那句话——

「用梵文就需要对他的定义负责。」

这句话不在 Doc 16/31 的讨论中。它出现在 Master 审阅的另一部分。它指向一个不同的方向。它指向 D4。

---

D3 结束时，SUNYATA 在白板上列出了三个「已知缺口」。其中两个名字是 IPrajna 和 ISatiMonitor。白板上的文字不带情绪。它们只是「尚未实作」的接口名称。

但 Master 看到了白板上的名字——或者更准确地说，他看到了名字背后的问题。

「ISatiMonitor，」他写道，「你觉得 Sati 的内容完全 match 吗？它归在哪一蕴？」

这不是一个新问题。D2 已经回答了蕴归属——[vedana, samjna, vijnana]。三蕴。不包含行蕴。

但 Master 问的不是蕴归属。他问的是名字。

Sati 是巴利文的「正念」。正念在佛学中是行蕴的心所——它是主动的、有意志的、道德正向的修行活动。但 D2 刚刚证明了 SatiMonitor 不是行蕴——它是被动的、自动的、价值中立的品质监控器。

如果 SatiMonitor 不是行蕴活动——那它就不是正念——那它为什么叫 Sati？

D2 的 SCRIBE 旁白预言了这一刻：「名字的问题，要等 Master 亲自来问。」

Master 问了。

---

## VI. 过渡

SUNYATA 环顾剧场。

二十位研究者。D1 到 D3 已经消耗了大约四个半小时。十九项投票。整个佛学映射体系被审计、分类、部分拆除。五蕴 OOP 架构通过了完备性验证。Sati Plugin 被重新归类为三蕴组合。

这些都是预定议程上的工作。R0 定向时设计的三条研究轨道——Track A（五蕴架构）、Track B（映射审计）、Track C（Sati 重分类）——全部完成。

但 Master 的审阅打开了一扇计划外的门。

不是一个新问题——而是一个老问题的新角度。SatiMonitor 的蕴归属解决了。映射边界原则建立了。架构验证通过了。但名字——名字的问题一直在那里，像一条地下水脉，在所有辩论的表面之下流淌，只是没有人挖一口井把它引出来。

直到 Master 用一句话挖了那口井：

「用梵文就需要对他的定义负责。」

SUNYATA 转向团队。

「我们需要一场计划外的辩论，」他说。「D4。议题：命名修正。」

他看了一眼 NAGARJUNA。哲学家的表情是平静的——但那是一种中观学者在自己的论证被推向逻辑极端时特有的平静。他知道接下来会发生什么。因为他自己在 D2 中种下了那颗种子——

如果 SatiMonitor 不是行蕴活动，那它就不是 Sati。

那个推论在 D2 结束时被记录了。但没有人把它拉到终点。

现在，Master 替他们拉了。

---

> *SCRIBE 旁白*

> *Master 的裁定有两个层次。*

> *表面的层次是两项决议的推翻。D1-Q7 和 D1-Q8——Doc 16 和 Doc 31 恢复原状。这是具体的、可执行的、明确的。团队错了，Master 修正了，修正被接受了。在形式上，这和 Cycle 02-3 的 DC-10 或 02-4 的 DC-6 没有区别——Master 有最终裁定权，他行使了这个权力。*

> *但深层的层次是一个新的维度。三项测试和四层框架是 Cycle 02-5 的基础设施——它们在 R0 被提出，在 R1 被三个学科论证，在 R2 以零异议通过，在 D1 以十个 20/20 被使用。它们是有效的。但它们是不完整的。*

> *不完整不是错误。不完整是——尚未完成。*

> *Master 加入了一个前提检查：在应用三项测试之前，先识别文件类型。工程文件用三项测试。映射文件用不同的标准。设计决策文件需要特殊处理。这不是推翻三项测试——这是为三项测试增加一个入口条件。*

> *就像一把卡尺——它的精度没有问题，但你得先确认你量的是零件，不是设计图纸。*

> *但裁定的真正重量不在 Doc 16 和 Doc 31。*

> *裁定的真正重量在那句话：「用梵文就需要对他的定义负责。」*

> *这句话像一颗石头丢进了池塘。D1 到 D3 的所有讨论是池塘的水面。石头沉下去了。涟漪还没有扩散到边缘。*

> *但它会扩散。在下一场辩论中——D4——涟漪会触及每一个用梵文命名的接口。ISatiMonitor。IPrajna。戒慧安全框架。*

> *三项测试问「这个映射是否有工程价值？」*

> *Master 问的是「这个名字是否对得起它的定义？」*

> *不同的问题。更深的问题。更危险的问题。*

> *因为工程价值可以被衡量——通过测试、代码验证、设计影响追溯。但定义的忠实度是一个哲学问题。它问的不是「这个名字有用吗？」——它问的是「这个名字诚实吗？」*

> *这架天平——序章中出现的天平——现在被校准了。不是用工程价值校准。是用定义的忠实度校准。*

> *接下来会发生什么，取决于天平两端放上去的东西。*

> *一端是名字。另一端是定义。*

> *如果两端平衡——名字保留。*

> *如果两端不平衡——*

> *D4 会告诉我们答案。*
