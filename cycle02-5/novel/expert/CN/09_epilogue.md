> **Test 4：当你使用佛学梵文术语作为代码标识符时，该组件的功能必须匹配该术语的佛学定义。如果不一致，使用工程术语。**

## I. 两种语言

Cycle 02-5 的核心不是某个具体的决议。不是 ILoopQualityMonitor，不是 IConfidenceAuditor，不是 Doc 45 的五蕴 OOP 架构。

核心是一个问题的答案。

问题是：在一个同时使用佛学和工程两种语言的系统中，这两种语言的边界在哪里？

D1 到 D5 用了六个半小时回答这个问题。答案可以浓缩成几行：

**佛学语言适合用在三个地方：**

1. **代码标识符**——当佛学术语已经成为原始码的一部分，并且功能和定义一致。Moha 是自我迷惑——Moha 模块确实在模拟自我迷惑。skandha 是蕴——代码中的 Skandha 类型确实在分类五蕴。这些名字通过了 Test 4。天平平衡。

2. **设计理由**——当佛学概念实际驱动了工程决策，并且有 DC 确认。「vasana 是后天薰习」驱动了 VasanaEngine 的外部化决策。「四烦恼同时俱起」驱动了 MulaKleshaBundle 的原子性设计。佛学概念和工程决策之间有因果关系。这些不是装饰——是理由。

3. **独立映射文件**——当文件的存在目的就是做佛学到工程的系统性对照。Doc 16 和 Doc 31 不是在工程文件上贴佛学标签——它们本身就是映射文件。Master 裁定保留。映射文件不适用三项测试——因为它们不是工程文件中的佛学装饰，而是佛学映射本身就是内容。

**工程语言适合用在其他所有地方。**

硬性规则不需要叫 sila。品质监控器不需要叫 Sati。信心度审计不需要叫 Prajna。event-driven 不等于 mindfulness。三学不等于五层阈值模型。

佛学语言是深刻的。工程语言是精确的。在需要深度的时候用佛学语言——附录、学术参考、设计背景。在需要精度的时候用工程语言——接口定义、规格文件、安全架构。

两种语言。两个用途。一条边界。

## II. 被保留的和被改掉的

NAGARJUNA 在辩论结束后做了一件他在之前所有 Cycle 中都没有做过的事——他走到 Doc 45 的草稿旁边，读了一遍。

全文。六十页。从第一节的根接口到附录 B 的认知序列。

读完后他对 VITRUVIUS 说了一句话：

「这份文件里没有一个梵文术语是装饰。每一个梵文术语都是代码标识符或经过 Test 4 验证的名称。」

VITRUVIUS 回答：「这就是目标。」

NAGARJUNA 点头：「Cycle 02-4 的 openstarry_doc 里有太多佛学装饰。我自己放进去的。后贴标签。我以为在工程术语旁边加一个梵文对照会让文件更丰富。但 Master 看出了问题——它不是让文件更丰富，而是让文件更混乱。一个软件工程师来读安全架构文件，看到『三学映射』，他的反应不是『哦，深刻的跨学科洞见』——他的反应是『这和我要实作的安全边界有什么关系？』」

他停了一下。

「被改掉的名字——ISatiMonitor、IPrajna、Sila-Prajna——它们不是坏名字。它们是放错位置的名字。正念是伟大的概念。般若是伟大的概念。戒慧是伟大的概念。但伟大的概念不应该被用来命名平庸的功能。这对概念不公平。」

ASANGA 走过来，听到了最后一句。

「也对功能不公平，」他补充道。「当你把一个 ±0.05 的夹钳叫做般若，你不仅贬低了般若——你也误导了对功能的理解。一个新来的工程师看到 IPrajna，他会以为这是一个多么高深的元件。然后他打开代码，看到的是一个 clamp 函数。期望和现实之间的落差——那就是名字的代价。」

## III. 被保留的名字

但不是所有名字都被改掉了。

Moha。Drishti。Mana。Sneha。skandha。vedana。samjna。samskara。vijnana。klesha。CoarisingBundle。samsaric stall。vasana。

这些名字全部保留了。不是因为团队对它们特别偏爱——是因为它们通过了四项测试。

Moha 模块模拟的确实是自我迷惑（moha 的佛学定义）。Sneha 模块模拟的确实是我爱（sneha 的佛学定义）。skandha 类型分类的确实是蕴（skandha 的佛学定义）。

名字和定义一致。天平平衡。

这是 Cycle 02-5 最重要的结论之一——五蕴框架不是装饰。它是真的。五个根接口对应五蕴不是一个「有趣的类比」——它是架构设计的基础。Plugin 的分类、Registry 的组织、ExecutionLoop 的阶段映射——全部建立在五蕴的分类逻辑上。

佛学语言在这里不是装饰。佛学语言在这里是代码。

差别在于——是代码的佛学语言通过了 Test 4。不是代码的佛学语言没有通过。

## IV. 给下一轮的种子

SUNYATA 在 prior_research 中写下了给 Cycle 02-6 的摘要。

他列出了四项永久性成果：

1. **四层框架**：KEEP / RELOCATE / REMOVE / FILE REVIEW——加上文件类型前提检查。

2. **四项测试**：必要性、代码标识、决策驱动、定义责任。前三项在 R0 建立。第四项在 D4 建立。四项测试是 D1 的三项测试加上 Master 的那句话。

3. **Doc 45**：五蕴 OOP 架构。纯工程语言。回答了 Master 的每一个问题。

4. **IConfidenceAuditor 100% 规格**：可以直接交付工程团队。

他还列出了三个已知缺口：

1. 三个弱继承接口（IVedanaSensor、IGearArbiter、IKlesha）——不 extends 根接口。已知的设计妥协。

三个缺口都不是架构问题。都是实作问题。架构是稳固的。
