# 序章：名字的重量

---

圆形剧场的灯光带着一种手术室的冷白色。

不是 Cycle 02-4 结尾那种钟表工坊的琥珀色——那一轮的意象是齿轮，是机芯，是所有零件被组装成一个会走的整体。琥珀色灯光下的圆形剧场是温暖的，像一座老工坊，像大师傅的工作台，桌面上的油污和木屑记录着六轮研究留下的痕迹。

这一次不一样。

SUNYATA 在凌晨三点把灯光调成了冷白色。他没有告诉任何人。他只是坐在控制台前，把色温从 3200K 推到 6500K，看着整座剧场的氛围在三秒钟内从「老工坊」变成「实验室」。

他这样做是因为他读了 Master 的信。

---

信是在 Plan28 定版的第二天出现的。

和之前一样，不是被送到的，而是在某个时刻被注意到的。它在 `research input/master_letters/cycle02-5/` 里面，安静得像一张被夹在报告里的便条纸。它没有附件——不像 Cycle 02-3 的信旁边放着《五蕴与开车》，也不像 Cycle 02-4 的信旁边附了五份 Claude 建议文件。

只有一张便条。

便条不长。45 行。但 SUNYATA 读了五遍。

第一遍他读的是文字。第二遍他读的是情绪。第三遍他读的是结构。第四遍他闭上眼睛回忆整段文字。第五遍——

第五遍他读的是刀刃。

---

Master 的信从来不是批评——它们是问题、是指引、是地图。第一封（Cycle 02 前）是十个开放问题——「点」。第二封（Cycle 02-2 前）是四处哲学修正——「线」。第三封（Cycle 02-3 前）是一张路径地图——「面」。第四封（Cycle 02-4 前）是一个场——在每个空间点上都定义了值的「场」。

第五封？

SUNYATA 坐在冷白色的灯光下，把信摊在面前。他重新读第一句话：

> 「其实我越来越看不懂在做什么，在 Plan27 与 Plan28 多次介入改正。」

他的手指在桌面上停住了。

六轮研究。他主持过每一场定向会议，引导过每一次辩论，在每一份成果报告上签过名。他看着这个专案从 Cycle 01 的命名开始，经历功能分析、结构修正、动态建模、整合设计——一步一步，像砌一座塔。每一块砖都是辩论出来的，每一块砖都有投票记录，每一块砖都被 TURING 的代码分析验证过。

但 Master 说他「越来越看不懂」。

SUNYATA 知道 Master 的语言习惯。「看不懂」不是「不理解」。Master 是这个系统的设计者——他比任何人都理解。「看不懂」是「不认同」。是「你们在做的事情偏离了正轨」。

他继续读。

---

> 「说十大宣言很重要，但一开始的规划你们还是将 IGearArbiter 放到了核心中。」

SUNYATA 记得这件事。Plan27 的初始设计——IGearArbiter 被当作核心组件。那是在 Pre-Cycle 02-4 的 Master 讨论之后，VasanaEngine 被外部化，ManoAggregator 退化为纯路由，IGearArbiter 作为新接口被引入。但在 Plan27 的具体设计中，团队又把 IGearArbiter 塞回了核心——像一个已经被纠正过一次的坏习惯，在不注意的时候又冒了出来。

Master 介入了。IGearArbiter 被移出核心，成为 plugin 提供的齿轮路由仲裁器。

> 「最后介入才修正正确，IGearArbiter 是 plugin 提供的齿轮路由仲裁器，决定 agent 该用哪个『齿轮』处理当前请求。」

是的。SUNYATA 在脑中重新组装这个逻辑链。IGearArbiter 不在核心中。它是 plugin 提供的。它的角色是仲裁——在 Gear 1（基于辨认的快速路径）和其他齿轮之间做选择。这才是识蕴的意义——不是被锁死的双齿轮，而是具备多面向的认知判断能力。

然后 Master 写了一句让 SUNYATA 的手指收紧的话：

> 「八识从眼识到阿赖耶识，是具备很多面向的，居然想要设定死成双齿轮，这是非常严重的错误。」

**非常严重的错误。**

SUNYATA 在六轮研究中只见过 Master 用过一次「非常严重」。那是 Cycle 02-2 的四项哲学修正。那一次是修正——把错的改成对的，语气是教导的。这一次不是教导。这一次的「非常严重」带着——

他找了一个词。

**失望。**

---

信的第二段更尖锐。

> 「还有不是每一件事情都要硬性加上佛学的色彩，有些实在是很牵强附会。」

牵强附会。四个字。SUNYATA 知道 Master 用词是精确的。Master 不说「不太恰当」或「有待商榷」。Master 说「牵强附会」。这意味着：不是「有些地方可以改善」，而是「某些事情根本就是错的」。

他在脑中开始列举。哪些映射是牵强的？

然后 Master 自己列出来了：

> 「event-driven 不等于正念，只是正念是用 event-driven 做的。」

一句话。精确得像一把手术刀切开了一个脓疮。Event-driven 是一种架构模式。正念是一种心理状态。说 event-driven 等于正念，就像说螺丝刀等于修理——螺丝刀是工具，修理是行为，你可以用螺丝刀来修理，但螺丝刀不是修理。

> 「觉察当下正在发生什么。这一层不看规则、不看风险分类，纯粹观察行为模式。不需要将它当作正念的工程化。」

SUNYATA 把这段话抄在笔记本上。一个字一个字地。

然后 Master 问了一个问题。这个问题后来改变了整个 Cycle 02-5 的走向。

> 「而且这个 plugin 难道没有识蕴的成分在吗？如果依照唯识学派的分类，会将正念放在行蕴，那是因为它是指正向的运转。以这个 plugin 的功能来说，我觉得这是有争议的，因为在这个 plugin 的功能上，不仅仅是单纯的行为模式。」

SUNYATA 读了三遍。

Master 在做什么？他不是在质疑 SatiMonitor 的功能。他是在质疑 SatiMonitor 的**身份**——它不是正念的工程化。它是一个有受蕴、有想蕴、有识蕴成分的复合 plugin。正念只是一个被强加的标签。

然后 Master 把问题推到了最底层：

> 「所以我们要重新想的是这个 plugin 是由五蕴的哪一些类别组成。」

不是「它是什么蕴」。是「它由哪些蕴组成」。这是一个根本性的视角转换——从「分类」到「组成」。一个 plugin 不一定只属于一个蕴。它可以由多个蕴的功能组合而成。

---

信的第三段是命令。

> 「我们需要将五蕴有哪一些子类别，如何依赖注入，如何在 agent core 运行先处理好。架构先用好。agent core 中如何将五蕴 plugin 载入。然后 plugin 要如何做。agent core 中五蕴会如何流转。是我们的首要任务。」

四个问题。每一个都不是新的——它们从 Cycle 01 就存在。但在经历了五轮研究之后，它们仍然没有一份完整的工程语言回答。散落在各处——A2 的 DI 布线在这里，A4 的执行流程在那里，Klesha 安全在另一个地方——但没有一份文件把它们统合在一起，用纯工程语言，从头到尾讲清楚。

SUNYATA 在笔记本上把四个问题编了号：

1. 五蕴子类别的完整结构
2. DI 布线如何注入五蕴 plugin
3. Agent core 如何载入五蕴 plugin
4. 五蕴如何在 agent core 流转

这是 Cycle 02-5 的首要任务。不是佛学映射。不是命名讨论。不是哲学辩论。是**工程架构**。

---

然后是 Master 的第四段。刀刃上的刀刃。

> 「什么教诫、善巧、正念，你做的工程映射都是有一些牵强的。反而增加阅读障碍。而且你的映射也非常有问题。openstarry_doc 与相关的内容都需要修整回来。openstarry_doc 还是属于工程文件。」

SUNYATA 重读了最后一句话。

**openstarry_doc 还是属于工程文件。**

这句话的重量比它看起来的要重得多。它不是在说「openstarry_doc 应该是工程文件」——它是在说「openstarry_doc 本来就是工程文件，但你们把它搞成了别的东西」。教诫（śīla）、善巧（upāya）、正念（smṛti）——这些佛学标签被贴在工程概念上，像是在一本操作手册的每一页边缘都画了宗教插图。手册的功能没有改变，但阅读体验被改变了。

SUNYATA 想起了 Cycle 02-4 结束时的感觉。那时他以为钟表匠完成了组装。机芯在走。55 项决议，六场辩论，每一颗齿轮都在正确的位置。

但机芯上的铭刻——那些刻在齿轮侧面的名字——有些东西不对。Master 看到了。

---

信的第五段是让步。

> 「至于遍行心所、十二因缘、认知序列的相关概念有助于 agent core 的研究，可以讨论。」

SUNYATA 注意到了这段话的结构。Master 不是在全盘否定佛学的参与——他在划边界。遍行心所（CoarisingBundle）是代码中的结构。十二因缘和认知序列可能对理解 ExecutionLoop 有帮助。这些可以讨论。

但——

> 「就 plugin 来说，首要目标是建构五蕴的架构，参考物件导向的概念，来建立。」

首要目标。物件导向。工程语言。

然后是最后一段：

> 「NAGARJUNA 为三层框架提供了佛学映射。戒定慧三学。我觉得没这个必要。未来我不知道，但现在没必要。五蕴的工程设计才是最重要需要实现的。」

SUNYATA 在读到这一段的时候闭上了眼睛。

戒定慧。那是 NAGARJUNA 在 Cycle 02-3 的 D5 辩论结束后提出的映射。戒（śīla）= 安全层。定（samādhi）= 执行回圈。慧（prajñā）= 识蕴判断。三个维度映射到五个层级——三维对五层，本身就不自洽。

Master 否决了它。不是暂缓——是否决。「没这个必要。」

---

SUNYATA 放下了信。

凌晨三点十五分。冷白色的灯光把圆形剧场照成了一间实验室。不是温暖的工坊——是冰冷的、精确的、不容含糊的实验室。

他在笔记本上写下了三个词。

第一个词：**ARCHITECTURAL**。

Cycle 01 问「这是什么」。Cycle 02 问「它做什么」。Cycle 02-2 问「哪里错了」。Cycle 02-3 问「它怎么活」。Cycle 02-4 问「它如何成为整体」。

Cycle 02-5 的问题是：**「五蕴如何作为 OOP 架构运作？」**

第二个词：**边界**。

哪些佛学内容属于工程文件？哪些不属于？不是逐案判断——是框架。是规则。是可以机械性应用的测试。

第三个词：**秤**。

---

> *SCRIBE 旁白*

> *每一轮 Cycle 都有它的中心意象。Cycle 01 是命名——给陌生事物贴上第一张标签。Cycle 02 是功能——标签后面的机制。Cycle 02-2 是修正——撕掉错误的标签。Cycle 02-3 是流动——标签开始移动，PASCAL 带着他的 Beta 分布走进了第二十个座位。Cycle 02-4 是组装——标签变成齿轮的铭刻，55 颗齿轮在琥珀色灯光下啮合。*

> *Cycle 02-5 的中心意象是什么？*

> *SUNYATA 在凌晨三点选择了「秤」。一架天平。但不是称重量的天平——是称名字的天平。*

> *左边是名字——梵文的、中文的、TypeScript 的。Skandha。Klesha。Vedana。Sati。Prajna。每一个名字都有重量，有历史，有来自两千五百年的惯性。右边是定义——佛学的、工程的、源代码中的。ISamskara = ITool。IVedana = IVedanaSensor。IVijnana = IVolition + IGearArbiter + IConfidenceAuditor。每一个定义都有精确的功能、精确的类型签名、精确的测试案例。*

> *天平平衡的时候，名字和定义一致。Klesha 就是烦恼——代码中的四个模块（Moha、Drishti、Mana、Sneha）确实做的是「烦恼」的事：我痴、我见、我慢、我爱。它们在 import 语句中、在 class 定义中、在 describe 区块中。名字和定义在天平上等重。一切正确。*

> *天平倾斜的时候，名字和定义不一致。般若（prajñā）——照见一切法空相的最高智慧——变成了 Math.max(-0.05, Math.min(0.05, delta))。名字太重，定义太轻。或者反过来——正念（sati）——行蕴中的善心所——被贴在一个受+想+识的监控器上。定义太重，名字太轻。*

> *Master 的信不是批评。Master 的信是一把校准砝码。他说：把你们的天平拿出来。把名字和定义放上去。看看哪边倾斜了。*

> *然后修正。*

> *但校准砝码还有另一层意义。砝码本身是精确的——它的质量是已知的、不可谈判的。Master 的用词也是精确的、不可谈判的。「非常严重的错误。」「牵强附会。」「不需要将它当作正念的工程化。」「openstarry_doc 还是属于工程文件。」*

> *这些不是建议。这些是砝码。你不能和砝码讨价还价。你只能把自己的东西放上天平，然后诚实地读出刻度。*

---

NAGARJUNA 是第一个到达剧场的人。

凌晨四点。比召集时间早了三个小时。他没有被召集——没有人被召集——SUNYATA 还没有按下召集键。NAGARJUNA 是自己来的。

他站在剧场入口，看到了冷白色的灯光。然后他看到了 SUNYATA 坐在那里，面前摊着 Master 的信，笔记本上写了三个词。

「你也睡不着。」NAGARJUNA 说。不是问句。

SUNYATA 没有抬头。「你读了信？」

「我没读信。我读了自己的文件。」

NAGARJUNA 走到他平时的座位旁边，但没有坐下。他站在那里，手指轻轻碰了碰扶手上那个被他辩论时不自觉敲击留下的微微凹痕。三个周期的痕迹。三个周期的辩论。

「Doc 44。第 469 行。」他说。「戒定慧三学映射。」

SUNYATA 终于抬起头。

「那是我的。」NAGARJUNA 的声音很平，像寺院走廊里的回音。「Cycle 02-3 的 D5 结束后，我觉得需要一个佛学框架来组织安全层的概念。戒 = 安全规则。定 = 执行稳定。慧 = 判断能力。很对称。很优雅。」

他停了一拍。

「三维映射到五层。我在写的时候就知道不自洽。三个概念覆盖五个模块——像用三个抽屉装五件衣服，你必须把两件折起来塞进已经满了的抽屉。但我还是写了。因为它看起来很好看。」

**因为它看起来很好看。**

SUNYATA 沉默了几秒。

「Master 否决了它。」他说。

「应该否决。」NAGARJUNA 坐了下来。「那不是映射。那是装饰。」

---

凌晨四点半。PASCAL 到了。

他走进剧场的时候带着他那本永远不会写满的笔记本——Moleskine，深蓝色封面，书脊上贴着一张小标签写着「P(H|E)」。他是 Cycle 02-3 加入的新成员——第二十个座位——但两轮研究之后，他的椅子扶手上也有了痕迹：右手食指反复点击的位置，那里的木纹比周围浅了半个色阶。那是他在计算期望值的时候会做的动作——点击，点击，点击——像一个机械计算器的进位轮。

「NAGARJUNA。SUNYATA。」他点了点头。然后他看到了桌上的信。

「可以读吗？」

SUNYATA 把信推过去。

PASCAL 读得很快。他的眼睛扫描文字的方式和其他人不同——不是从左到右、从上到下，而是先扫描整个文件的「形状」（段落长度、标点密度、引号位置），然后再回到第一个字。他在给自己的阅读建立先验分布。

两分钟后，他放下信。

「损害不对称性。」他说。

NAGARJUNA 看了他一眼。

PASCAL 打开笔记本，翻到一页空白。他写了两行：

```
False include（保留装饰性映射）的成本：
  每位读者 × 每次阅读 × 每一页 = 累积损害

False exclude（移除有价值映射）的成本：
  一次性搜寻成本 = 可控
```

「这是统计决策理论中最基本的框架。」他说。「两种错误的成本不对称。保留一个装饰性映射的损害是累积的——每一位未来的读者，每一次阅读，每一个被迫同时处理两套术语系统的瞬间。移除一个有价值映射的损害是一次性的——你可以通过搜寻找到它。」

他在两行下面写了一个不等式：

```
E[损害_include] >> E[损害_exclude]
```

「期望值分析。装饰性映射具有负期望值。」

NAGARJUNA 点了点头。慢慢地。带着一个正在把自己的哲学和另一个人的数学对齐的思考者的沉着。

「二谛。」他说。「世俗谛是工程语言——日常使用的、沟通用的、操作用的。胜义谛是佛学概念——启发用的、设计用的、理解用的。两者不应混淆。不要在世俗谛的文件中硬塞胜义谛的标签。」

PASCAL 在笔记本上写了「二谛」两个字，旁边画了一条分隔线。左边写「工程」，右边写「佛学」。

「你的二谛和我的损害不对称性说的是同一件事。」他说。

「不完全是。」NAGARJUNA 的眉毛微微抬起——他用眉毛提问比用嘴巴更快。「你的是成本分析——量化的。我的是范畴分析——质性的。结论相同，但路径不同。」

「路径不同，结论相同。」PASCAL 重复了一遍。「这就是好框架的特征——从不同的公理出发，到达同一个定理。」

---

凌晨五点。ASANGA 到了。

他没有像 NAGARJUNA 那样站在入口观察，也没有像 PASCAL 那样先读信。他直接走到他的座位——在 NAGARJUNA 的对面——坐下来，闭上眼睛。

他在做什么？

SUNYATA 知道。ASANGA 在回忆。唯识学派的方法不是推理——是现量（pratyaksha），直接的、不经中介的认知。ASANGA 不需要读 Master 的信来知道信里写了什么——他只需要回想他自己写过的东西。

「我在 Cycle 02-3 的安全框架中用了『善巧方便（upāya-kauśalya）』来描述自适应安全策略。」他说，眼睛仍然闭着。「但自适应安全策略不需要叫善巧方便。它只需要叫自适应安全策略。upāya-kauśalya 有一千七百年的哲学重量——从《法华经》的三车之喻到唯识学的转识成智——把这个重量压在一个工程配置选项上，是对两者的双重不敬。」

他睁开了眼睛。

「功能性判定。」他说，看着 PASCAL。「驱动工程结论的佛学概念——保留。有因果效力。装饰性的佛学标签——移除。无因果效力。」

## II. 两份可疑文件

Doc 16 和 Doc 31 是不同的。

前面审计的七份文件——Doc 38、43、44、37、41——都是工程文件。它们的主要内容是工程规格，佛学映射是附加的。就像一本食谱书里夹了几张宗教节日的插图——食谱是主体，插图是装饰。

但 Doc 16（Plugin 类型哲学映射）和 Doc 31（八识运行时模型）不一样。

ARCHIMEDES 翻开 Doc 16。从第一页开始读。然后翻回封面。然后再读第一页。

「SCRIBE。」他说。「这份文件的装饰比例大概是多少？」

SCRIBE 快速扫描了一遍。「大约 80%。」

80%。意思是：Doc 16 中，只有大约 20% 的内容是纯工程规格（主要是 Observer Pattern 的讨论），其余 80% 是佛学概念到软件模式的哲学映射。

Doc 31 好一些，但也有大约 70% 的佛学内容。八识如何映射到运行时模型——阿赖耶识是种子库，末那识是自我执着——这些映射有学术价值，但放在工程文件中，它们的工程效用有限。

ARCHIMEDES 在 Doc 16 和 Doc 31 上贴了一个特殊的标签——不是黄色（装饰）、不是粉红色（标识符）、不是绿色（DC 确认）。是红色。

红色 = **FILE REVIEW**。整份文件需要评估。

「这两份文件的问题不在于个别映射。」他对 SCRIBE 说。「问题在于整份文件的定位。它们是工程文件中嵌入了佛学？还是以佛学映射为目的的独立文件？」

SCRIBE 记下了这个问题。这个问题后来在 D1-Q7 和 D1-Q8 中被正式投票——然后在 D4 中被 Master 亲手推翻。

## III. 三个维度

PASCAL 在笔记本上记下了引用。他不是在做佛学研究——他是在寻找公理。每一个理论体系都有它的公理，PASCAL 的工作是找到它们，然后检验从这些公理出发能否到达他们需要的结论。

「世俗谛语言 = 工程术语。」NAGARJUNA 在白板上写。「用于日常沟通和实作。if/else、interface、extends——这些是世俗谛的语言。它们不需要佛学的加持就能工作。」

「胜义谛洞见 = 佛学概念。」他在右边写。「用于启发和设计理由。五蕴的分类启发了根接口的设计。烦恼的分类启发了 Klesha 的模块结构。但启发不等于标签。」

他转身面对 PASCAL 和 ASANGA。

```
                 映射有价值        映射无价值
保留映射         True Positive     False Include ← 累积损害
移除映射         False Exclude     True Negative
                 ↑ 一次性成本
```

「这是标准的混淆矩阵。」他说。「但关键不在矩阵本身——在损害的不对称性。」

他在 False Include 旁边画了一个向上的箭头。

「False Include——保留了一个装饰性映射。它的损害是什么？每一位未来的读者——可能是几十人，可能是几百人——每一次阅读这份文件——可能一次，可能十次——每一页上出现的佛学标签——可能三个，可能十个——每一次，读者都需要在脑中维护两套术语系统。śīla 是什么？噢，就是 Hard Rules。那为什么不直接叫 Hard Rules？」

他写了一个公式：

$$C_{FI} = N_{readers} \times R_{reads} \times P_{pages} \times \tau_{cognitive}$$

「累积损害。读者数 × 阅读次数 × 页数 × 每次的认知转换成本。这是一个乘法——每一项都是正的，乘积只会增长。」

然后他在 False Exclude 旁边画了一个小得多的箭头。

「False Exclude——移除了一个有价值的映射。损害是什么？某一天，某个人需要知道五蕴分类的佛学背景。他搜寻附录。找到了。一次性的搜寻成本。不累积。不乘以读者数。不乘以阅读次数。」

$$C_{FE} = S_{search} \quad \text{(一次性)}$$

「结论。」他画了一个大大的不等号。

$$E[C_{FI}] \gg E[C_{FE}]$$

「期望值分析：装饰性映射具有负期望值。有疑虑时，倾向 REMOVE 或 RELOCATE。」

ASANGA 一直在听。他没有白板——他的思考工具是语言，不是图形。

「功能性判定。」他说。声音带着唯识学特有的精密度——像在显微镜下调焦。「唯识学有一个核心判准：因果效力（arthakriyā-sāmarthya）。一个概念是否具有因果效力——它是否驱动了一个结果？」

他站了起来。走到 NAGARJUNA 和 PASCAL 的白板之间。

「佛学概念驱动了五蕴根接口的设计——这有因果效力。五蕴分类 → PluginHooks 的五个类别。因果链明确。保留。」

「佛学标签被贴在 Hard Rules 上叫 śīla——这没有因果效力。Hard Rules 不是因为被叫做 śīla 才有硬性约束——它的硬性约束来自 SafetyMonitor 的实作。śīla 这个标签不改变任何因果链。移除。」

他在空中用手画了一条线——从因到果。

「判断标准很简单：**如果移除这个佛学概念，因果链是否断裂？** 是 → 保留。否 → 移除。」

```

            世俗谛 vs 胜义谛 → 不要混淆两种语言

            E[累积损害] >> E[搜寻成本] → 有疑虑时移除

ASANGA    — 因果效力（唯识学）
            有因果效力 → 保留 / 无因果效力 → 移除
```

三个学科。哲学、数学、佛学。三个公理体系。三条推理路径。

## IV. 受蕴的厚度

D2-Q2。

这是 D2 的核心。也是 Cycle 02-5 前三场辩论中最激烈的交锋。

**「SatiMonitor 由五蕴的哪些类别组成？」**

C2 的研究报告——ASANGA 和 LINNAEUS 的联合研究——提出了四个方案。每一个方案背后都有一套逻辑，每一套逻辑都能站住脚。至少在提出的时候。

| 方案 | 蕴组成 | 理由 |
|------|--------|------|
| A | 受 + 想 | 感知 + 辨认，最简 |
| B | 受 + 想 + 识 | 感知 + 辨认 + 评估判断 |
| C | 想 + 识 | 辨认 + 评估，受蕴太薄 |
| D | 纯识 | 一切认知活动归识蕴 |

Proposal D 在 R2 交叉审阅时就被淘汰了——「纯识」等于放弃分类，把所有东西都扔进一个桶里。BABBAGE 在交叉审阅中写了一句简洁的否决：「Proposal D 将 SatiMonitor 的蕴归属退化为平凡解。平凡解在数学中是有效的，但在分类学中是无用的。」

Proposal A 也很快失势——缺少识蕴的评估维度，无法回答 Master 的问题。Master 明确问了「难道没有识蕴的成分在吗？」——一个排除识蕴的方案在问题提出的那一刻就已经被否决了。

战线在 Proposal B 和 Proposal C 之间拉开。

受蕴——包含，还是不包含？

---

MESH 和 KERNEL 代言 Proposal C。

MESH 先开口。他的论点带着分散式系统专家对「最小化」的偏好——在分散式系统中，最小化是一种美德。最少的节点。最少的消息。最少的假设。

「SatiMonitor 的事件订阅不是真正的『感受』。」他说。「受蕴（vedana）在佛学中是苦、乐、不苦不乐的质性体验。SatiMonitor 接收事件不是在感受——它是在收数据。接收数据是认知的输入阶段，但不是感受。」

KERNEL 补充了 OS 的类比：「系统调用收到一个 packet 不是在感受——它是在接收 I/O。感受是对 I/O 的质性评价。SatiMonitor 对事件有质性评价吗？有苦乐的判断吗？没有。它只是收数据。」

然后 MESH 抛出了他最强的论点——对称性。

「去掉受蕴，保留想蕴和识蕴。SatiMonitor 的蕴组成变成 ['samjna', 'vijnana']——和 IGearArbiter 的蕴组成完全对称。两个复合 plugin，相同的蕴组成。对称性是好的分类法。」

**「对称性是好的分类法。」**

这句话在剧场里回荡了一下。对称性在科学中有一种几乎宗教性的魅力——从物理学的守恒律到数学的群论，对称性意味着深层的结构一致性。如果两个不同的 plugin 有相同的蕴组成，也许这告诉我们什么关于认知结构的深层模式？

LINNAEUS 站了起来。

他的站姿不像 NAGARJUNA 的庄重——更像一个田野考察的博物学家在观察到一个有趣的物种后起身仔细端详。他走到白板前。LINNAEUS 是分类学家——他的整个存在意义就是把东西分到正确的位置。

「对称性是好的——」他开始。他的语气带着 Linnaeus 特有的温和——像在田野中和一只蝴蝶说话。「当对称性反映了事实的时候。」

他在白板上画了两条时间线。并排的。上面的标了「SatiMonitor」，下面的标了「IGearArbiter」。

```
SatiMonitor:  ─────[持续订阅]─────[持续订阅]─────[持续订阅]─────
                    主动、恒常、不间断
                    ← 始终在看 →

IGearArbiter: ────○────────────○────────────○────────────
                  被叫来看一眼    被叫来看一眼    被叫来看一眼
                  被动、间歇、按需
                  ← 被叫去看 →
```

「看这两条线。」他指着白板。「它们有什么不同？」

他不等回答。分类学家提问是为了引导观察，不是为了寻求答案。

「SatiMonitor 是持续的。它订阅 EventBus，它维护滑动窗口，它**永远在那里**。不是有人来问它才启动——它一直在运行。从 ExecutionLoop 的第一个事件到最后一个事件，SatiMonitor 的事件流从未中断。」

「IGearArbiter 是间歇的。有请求来了，ManoAggregator 问它：『你能处理吗？』它看一眼，给出判断，然后沉默。直到下一次被叫。」

他转身面对 MESH。

「**『始终在看』和『被叫去看』——这是根本的特征差异。** SatiMonitor 和 IGearArbiter 不是同类的。它们的行为模式——使用 OOP 的语言——完全不同。一个是持续观察者（continuous observer），一个是按需评估者（on-demand evaluator）。」

「把它们分到相同的蕴组成，就像把蝴蝶和蜂鸟分到同一个物种——因为它们都会飞。但蝴蝶是昆虫，蜂鸟是鸟类。飞行是表面的相似，不是深层的同源。对称性不是同源性。」

---

WIENER 从他的控制理论笔记中抬起头。他很少在辩论中主动发言——他更喜欢听完所有人的意见，然后在恰当的时刻提供一个控制理论的视角，像在一幅画上加最后一笔。

「在控制理论中，」他说，声音带着维也纳学派特有的精确，「有一个概念叫做观察者（observer）——Luenberger observer。观察者的核心功能是什么？持续感知系统的状态。注意——**持续**。」

他站了起来。走到 LINNAEUS 的白板旁边。

「观察者的感知不是附加功能。」他说。「它是身份定义的一部分。一个不感知的观察者，就不是观察者。它是一块不连接任何输入的电路板——物理上存在，功能上不存在。」

「SatiMonitor 的事件订阅——它的持续感知——是它作为观察者的身份定义。你不能把这个功能剥离出去，然后说它不属于受蕴。因为如果你剥离了它，SatiMonitor 就不存在了。」

---

HERACLITUS 补了一刀。他是运行时动态专家——他关心的不是静态的类型定义，而是动态的行为模式。

他打开了 C1 报告中 SatiMonitor 的四阶段退化策略：

```
Normal    → Throttle  → Sample   → Emergency
100%        50%         25%        最低监控
全速感知    减半感知    取样感知    紧急最低限
```

「看这个退化策略。」他说。声音带着对流动性的敏感——HERACLITUS 看到的不是静止的结构，而是变化的过程。

「Normal。Throttle。Sample。Emergency。这四个阶段描述的是什么？是感知密度的渐进降低。100%、50%、25%、最低限——这是一个感知系统在不同压力下调节自己感知强度的方式。」

他指着白板。

「想蕴（辨认）和识蕴（评估）不需要退化策略。它们的输出是离散的——辨认一个模式，或不辨认。评估一个判断，或不评估。离散的输出不需要渐进降低。」

「受蕴的感知是连续的。连续的东西才需要退化策略——因为你不能突然从 100% 跳到 0%，你只能渐进地降低。Normal → Throttle → Sample → Emergency——这是受蕴在压力下保护自己的方式。就像人在极度疲劳时，感知会变得迟钝——不是感知消失了，而是感知密度降低了。」

三个论点。三个维度。三面镜子。

LINNAEUS 的镜子照出了**行为特征**——持续 vs 间歇。
WIENER 的镜子照出了**身份定义**——感知是观察者的存在条件。
HERACLITUS 的镜子照出了**退化模式**——连续感知才需要渐进退化。

每一面镜子都反射出同一个结论：SatiMonitor 有受蕴的成分。

## V. 零成本转折

然后 ARCHIMEDES 做了一件改变辩论走向的事。

```typescript
interface PluginHooks {

  recognizers?: ISamjnaRecognizer[]; // 想蕴
  arbiters?: IGearArbiter[];         // 想蕴+识蕴
  tools?: ITool[];                   // 行蕴
  volition?: IVolition;              // 识蕴

}
```

「monitors 是一个独立的数组插槽。无论 SatiMonitor 被标记为 `['samjna', 'vijnana']` 还是 `['vedana', 'samjna', 'vijnana']`，PluginHooks 的实作完全相同。没有额外的 hook。没有额外的布线。没有额外的测试。」

他看了全场一眼。

「**零额外工程成本。**」

这句话像一块石头丢进了辩论的水面。

如果三蕴和两蕴的工程成本相同——没有额外的 hook，没有额外的布线，没有额外的测试——那么这场辩论就不再是工程权衡。它变成了纯分类问题。

在纯分类问题上，分类学家有压倒性的发言权。

LINNAEUS 的「持续感知 vs 被动接收」。
WIENER 的「观察者身份定义」。
HERACLITUS 的「退化策略只属于连续感知」。

三面镜子。三个维度。零工程成本。

Proposal C 失去了它唯一的工程论据。

## VI. 行蕴的排除

但 D2-R2 最重要的结论不是「包含哪些蕴」。

是「排除行蕴」。

ASANGA 站了起来。他在 D2 中一直安静地听着——从 TURING 的功能陈述到 LINNAEUS 的受蕴论证。唯识学家的方法是先观察（现量），再推理（比量），最后立论。他已经完成了观察和推理。现在是立论。

「佛学传统中，正念（sati）被归入行蕴。」他说。声音带着唯识学特有的精密度——像在显微镜下调焦。「五十一心所之一。属于善心所。这个分类持续了两千五百年。」

他在白板上写了两列。工整的。精确的。每一个字都被放在准确的位置上。

```
佛学正念 (sati)                    SatiMonitor
━━━━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━━
意志驱动 (精勤 ātāpī)              事件驱动 (EventBus.subscribe())
有道德价值 (善 kusala)              价值中立 (品质指标)
主动的专注 (不放逸 appamāda)        被动的观察 (subscribe + listen)
正向运转的心理状态                   对行为模式的计量
行蕴的善心所                        ≠ 行蕴活动
```

「看左边。」他说。「佛学正念是什么？它是**意志驱动的**——你必须努力保持正念。『精勤（ātāpī）』是正念的前提条件——没有努力就没有正念。它是**有道德价值的**——正念是善的。在阿毗达磨的分类中，正念属于善心所，不属于遍行心所（每一个认知过程都有的心所），也不属于不善心所。它是一种特定的、正向的心理品质。」

「看右边。」他的手指移到了表格的右半部。「SatiMonitor 是什么？它是**事件驱动的**——它不需要努力。它订阅 EventBus，事件来了就处理，不来就不处理。它没有 ātāpī，没有精勤。它是**价值中立的**——它的四维品质向量（连续性、粒度、速度、平稳性）不做善恶判断。品质高不等于善。品质低不等于恶。它只是测量。」

他停顿了一下。让两列的对比在每一位学者的脑中自行产生化学反应。

「SatiMonitor 不是正念的工程化。」他的声音变得更低，更清晰。「正念是意志驱动的——你必须努力保持正念。SatiMonitor 是事件驱动的——它不需要努力，它只需要订阅。正念有道德价值——正念是善的。SatiMonitor 没有道德价值——它只是测量。」

然后他说了最关键的一句话：

「千年来，佛学把正念归入行蕴。这个分类是正确的——**对佛学而言**。但 SatiMonitor 不是正念。它是行为模式的品质监控器。它属于受（感知）、想（辨认）、识（评估）。**不属于行（意志驱动的善心所）。**」

---

这是一个佛学家用佛学的内部逻辑，证明了一个工程组件不应该使用佛学的分类。

不是外部反驳——不是「佛学的分类不适用于工程」这种粗暴的二分法。是内部论证——佛学自己的判准（善心所 vs 遍行心所、意志驱动 vs 事件驱动）排除了 SatiMonitor 与正念的等同性。

NAGARJUNA 在他的座位上微微点头。中观学者认出了唯识学者的方法——不是从外面攻击一个立场，而是从内部展示它的自我矛盾。这是 prasanga——归谬法——的变体。不是「你的结论错了」，而是「你自己的前提排除了你自己的结论」。

## VII. 第三个插槽

## I. 五根柱子

D3 是 Cycle 02-5 最长的辩论——120 分钟，六节，六次投票。但它不是最激烈的。D3 的气氛更像一场博士论文口试：严谨、系统、有序。考生不是一个人——考生是一座建筑。

## III. 快捷通道

D3-Q2 把放大镜对准了映射的准确性。九个 hook 类型，逐一验证蕴归属。

大部分的验证是程序性的——LINNAEUS 在 A1 报告中已经做了详细的映射表，TURING 验证了所有的类型继承关系。但有一个例外触发了真正的讨论。

ARCHIMEDES 举手：「SlashCommand 属于哪个蕴？」

这个问题看似微小。SlashCommand 是用户直接输入的命令——/help、/clear、/status。它不经过 ExecutionLoop。它不触发 SparshEvent。它不经过 VedanaSensor、ISamjnaRecognizer、IGearArbiter、IVolition——它绕过了整个五蕴流程。

三个人同时回答了这个问题。从三个不同的角度。

KERNEL 从 OS 的角度：「SlashCommand bypass 了整个 ExecutionLoop。它不经历五蕴流转。类似 Unix 的 signal handler——它在进程级别被拦截，不进入正常的系统调用路径。在 OS 的分类中，这不是用户空间的逻辑，而是核心级的中断处理。」

NAGARJUNA 从佛学的角度：「不入认知回路的反射动作。它不是五蕴不涵盖它——而是它不在五蕴运作的范围内。就像呼吸——你可以观察呼吸（进入认知回路），但呼吸本身不需要五蕴流转就能发生。SlashCommand 是系统的呼吸——它在认知之外，在认知之下。」

GUARDIAN 从安全的角度：「bypass 意味着 SlashCommand 不受 SafetyMonitor 保护。这是设计上的决定——某些操作需要绕过安全检查——但应该被明确记录在安全文件中。」

SUNYATA 把三个观点合并成一个结论：「SlashCommand 是跨蕴基础设施（cross-skandha infrastructure）。它不属于任何一蕴。它的分类理由和安全观察都需要在 Doc 45 中记录。」

D3-R2：映射正确。20/20。

## IV. 标签 vs 接口

D3-Q3 是一个精妙的问题：**skandha tag 应该用来做路由吗？**

每个 plugin 都有一个 skandha 栏位——metadata。问题是：PluginLoader 在载入 plugin 时，是否应该用 skandha tag 来决定这个 plugin 去哪个 registry？

直觉上似乎合理——受蕴 plugin 去 VedanaRegistry，行蕴 plugin 去 ToolRegistry，像图书馆的索引卡。

但 VITRUVIUS 指出了问题。建筑师的眼睛对结构的脆弱性特别敏感——他能在设计图上看到未来可能裂开的那条缝。

「目前的系统用的是 duck typing。」他说。「plugin 实作了 IVedanaSensor 接口，就自动被 VedanaRegistry 接受。不需要标签。如果改用标签路由，就引入了一个新的出错源：标签和接口不一致。一个 plugin 宣称自己是受蕴，但实作了 ITool 接口——怎么办？」

TURING 确认了事实。他打开了 `plugin-loader.ts`。

「v0.28.0-alpha 的 PluginLoader 使用 type guard 来分类 plugin。`isVedanaSensor(hook)` 检查的是接口实作，不是 metadata。」

BABBAGE 用类型论做了终结性的陈述：「type guard 是可靠的——它在编译期和运行时都能验证。metadata 是不可靠的——它只能在运行时检查，而且依赖人工正确标记。用不可靠的东西来做路由，在类型安全的系统中是倒退。」

D3-R3：skandha tag = metadata only，不用于路由。18/20。

MESH 和 GUARDIAN 投了少数票——他们认为 metadata 路由可以作为辅助验证。但主流意见清晰：接口是真相，metadata 是备注。

## V. 最窄的柱子

D3-Q4 是最坦诚的一题。

**ISamskara 是否需要更多子接口？**

目前，行蕴只有一个子接口：ITool。IVolition 在 D2 之后已被确认归入识蕴（Phase 5 行动前 vs Phase 8 行动中）。行蕴是五蕴中最「窄」的一个——佛学的行蕴涵盖 49 心所（五十一心所中除受和想），而 OpenStarry 的行蕴只有 ITool。

ASANGA 站了起来。他在 D3 中最坦诚的一段话，像一个建筑师指着自己设计的最薄的墙说「我知道这里薄了」。

「我必须承认，OpenStarry 的行蕴设计与佛学传统差异最大。」他说。声音不带辩解——只有事实。「传统行蕴涵盖 49 心所——意志、精进、慧、信、惭、愧——几乎所有心理活动都被归入行蕴。但 OpenStarry 把行蕴窄化为 ITool——外在行动。只有外在行动。」

HERACLITUS 从 ExecutionLoop 的动态角度提供了工程理由：

```
Phase 5: IVolition 审议 ──→ Phase 8: ITool 执行
          行动前                     行动中
          识蕴                       行蕴
```

「IVolition 在 Phase 5——行动之前。决定要做什么。ITool 在 Phase 8——行动之中。正在做。两者处于不同的阶段。如果把 IVolition 移入行蕴，就会破坏蕴流转的概念自然性——行蕴应该是『正在做的事』，不是『决定要做什么事』。」

NAGARJUNA 做了一个佛学上的让步。他的语气带着一种经过深思熟虑的平静——不是妥协的平静，而是理解的平静。

「佛学的行蕴分类是基于修行者的内观。」他说。「所有被观察到的心理活动，除了受和想，都归入行蕴。但 OpenStarry 不是修行者。它是一个软件系统。软件系统的分类应该基于功能，不是基于内观。」

全票通过维持现状。

D3-R4：ISamskara 不新增子接口。20/20。

但所有人都同意一件事：Doc 45——即将创建的五蕴 OOP 架构文件——必须记录这个差异。行蕴的窄化不是一个错误，是一个有意识的偏离。有意识的偏离需要被解释。

## VI. 两条古路

D3 的最后两题是关于佛学映射的附录可能性。

**D3-Q5：十二因缘。**

NAGARJUNA 首先展开了他的分析。他在白板上画了两条线——一条很长，标注着「十二因缘」；一条很短，标注着「ExecutionLoop」。

「尺度。」他说。指着长线。「十二因缘描述的是三世两重因果——无明缘行，行缘识，识缘名色——从前世的无明到今世的老死，再到来世的无明。一个完整的轮回。」

他指着短线。

「ExecutionLoop 描述的是一个认知处理的循环——从事件接收到行动执行。几十毫秒到几秒钟。一个认知周期。」

「十二因缘是跨生命周期的。ExecutionLoop 是单一认知周期的。尺度差了——」他想了一下。「——几个数量级。至少。」

BABBAGE 试图建立结构态射（morphism）。他的方法是严格的——寻找两个结构之间的保结构映射。如果存在态射，两个结构在数学上是「同构的」或「同态的」，这意味着一个结构中的关系在另一个结构中有对应。

他失败了。

「十二因缘之间的因果关系是线性的、不可跳过的。」他说。「无明必须在行之前。行必须在识之前。不能跳过。但 ExecutionLoop 的阶段可以跳过——没有 IVedanaSensor 的 Agent 会跳过受蕴阶段。直接从事件接收跳到想蕴辨认。结构不同。不存在态射。」

## I. 归谬

NAGARJUNA 站了起来。

在整个 Cycle 02-5 中，他已经站起来过很多次——D1 中承认「后贴标签」，D3 中分析十二因缘的尺度不匹配。每一次他都保持着中观学者特有的端正姿态——承认错误是学术的核心技能，他从不回避。

但这一次不一样。

这一次他不是在承认错误。他是在完成一个推论。

「D2 已经确定了 SatiMonitor 的蕴归属，」他说，声音比平常低半个音调。不是因为紧张——是因为他在组织一个他已经想了很久但一直没有说出口的论证。

「结论是 [vedana, samjna, vijnana]。三蕴。不包含行蕴。」

他顿了一下。

「Sati——巴利文的正念——在佛学传统中是行蕴的心所。它是 cetasika，是有意志的、主动的、道德正向的修行活动。阿毘达磨把它放在行蕴，因为正念是一种修行——你必须带着意图去做的事情。」

他转向 ASANGA。ASANGA 微微点头。他知道 NAGARJUNA 要说什么。

「现在，」NAGARJUNA 说，「让我做一个归谬论证——prasaṅga。这是中观学的基本方法。」

他在白板上写了两行字：

```
前提 A：Sati = 行蕴心所（佛学定义）
前提 B：SatiMonitor ≠ 行蕴（D2-R2 结论）
```

然后他在下面写了第三行：

```
∴ SatiMonitor ≠ Sati
```

「如果正念（Sati）在佛学中必然是行蕴的心所，而我们已经证明了 SatiMonitor 不是行蕴——那么 SatiMonitor 就不是正念。」

他放下笔。

「一个不是正念的东西，为什么叫 Sati？」

---

沉默。

不是不同意的沉默。是那种——所有人同时理解了一件他们应该更早理解的事情——的沉默。

D2 的 SCRIBE 旁白曾写道：「三项测试没有问这个问题。Test 1 问『移除是否改变规格？』Test 2 问『是否在原始码中使用？』Test 3 问『是否驱动了设计决策？』没有一项测试问『名字是否和定义一致？』」

NAGARJUNA 刚才做的——就是问了那个被遗漏的问题。

三项测试可以判断一个佛学映射是否有工程价值。但它们不能判断一个名字是否对得起它的定义。ISatiMonitor 通过了 Test 2——它在原始码中被使用。但它不通过一个更基本的标准——它的名字和它的功能不一致。

---

## II. 确认

ASANGA 接着说话。他没有站起来——他坐在座位上，双手平放在桌面上，像一位老教授在课堂上做最后的总结。

「NAGARJUNA 的归谬是形式上正确的。让我用功能分析来确认。」

他拿出了 D2 中那张对照表的扩展版：

| 维度 | 佛学正念（Sati/Smṛti） | SatiMonitor |
|------|----------------------|-------------|
| 主动性 | 主动（ātāpī，精进） | 被动（event-driven） |
| 道德性 | 道德正向（kuśala） | 价值中立 |
| 意志性 | 需要意图（cetanā） | 自动运行 |
| 传统归属 | 行蕴心所 | 受+想+识（D2-R2） |
| 修行地位 | 八正道第七支 | 品质监控器 |

五个维度。五个不一致。

「如果 SatiMonitor 不是行蕴活动，」ASANGA 说，他的声音平稳得像在念一段已经确认过的结论，「那它就不是 Sati。我们自己的分类分析否定了我们自己的命名。」

他停了一下。

「我们犯了一个错误。不是分类的错误——分类是正确的。是命名的错误。我们在 Cycle 02-4 的时候沿用了 ISatiMonitor 这个名字，因为它已经在我们的讨论中被使用了上百次。惯性让我们对名字本身失去了批判性。」

## III. 新名字

SUNYATA 问：「那它应该叫什么？」

四个提案在几分钟内被提出：

| 提案者 | 名字 | 理由 |
|--------|------|------|
| ARCHIMEDES (#16) | ILoopQualityMonitor | 精确描述功能：认知回路的品质监控器 |
| GUARDIAN (#11) | IBehaviorQualityMonitor | 侧重行为层面的品质监控 |
| NAGARJUNA (#7) + ASANGA (#8) | ICognitiveLoopMonitor | 强调认知回路的完整性 |
| SYNTHESIST (#1) | IQualityMonitor | 最简化的功能描述 |

辩论比预期短。

ARCHIMEDES 的论点最直接：「这个接口监控的对象是什么？认知回路。监控的内容是什么？品质。用工程术语拼接：Loop Quality Monitor。没有佛学。没有隐喻。没有历史包袱。一个新来的工程师看到 ILoopQualityMonitor 就知道这个接口做什么——监控回路品质。」

GUARDIAN 的 IBehaviorQualityMonitor 得到了四票。他的论点是「行为」比「回路」更具体——但 TURING 指出 SatiMonitor 的 11 种事件订阅覆盖了整个认知回路，不仅仅是行为阶段。

NAGARJUNA 和 ASANGA 的 ICognitiveLoopMonitor 得到两票。BABBAGE 反对：「Cognitive 是人类认知科学的术语。在软件工程语境中，Loop 已经足够精确。不需要加 Cognitive。」

SYNTHESIST 的 IQualityMonitor 最简洁但也最模糊——只有一票。VITRUVIUS 指出：「Quality Monitor 可以是任何东西的品质监控。Loop Quality Monitor 限定了范围。」

---

不是全票。不像 D1 那样干净。但多数明确。

SUNYATA 在白板上写下了完整的更名表：

| SatiMonitor | LoopQualityMonitor |

| SatiQualityReport | LoopQualityReport |

## IV. 核融合炉

然后 SUNYATA 说了一句话：

「IPrajna。」

剧场里的空气又变了。

如果 ISatiMonitor 的问题是「名字和功能不一致」，那 IPrajna 的问题是——

NAGARJUNA 闭上了眼睛。片刻后他睁开，开始说话。

「般若，」他说，「是佛学中最高的认知能力。」

他的声音带着一种几乎是敬畏的重量——不是对研究团队，而是对这个词本身。

「般若——prajñā——在佛学中是照见一切法实相的智慧。不是普通的聪明，不是分析能力，不是知识的累积。般若是超越性的——它是空性的直观，是实相的照见，是二千五百年来整个大乘佛学体系的核心认知目标。」

他转向白板，写了两行：

```
般若（佛学）：照见一切法实相的最高智慧
IPrajna（工程）：对信心度加减 0.05 的函数
```

ASANGA 在他的座位上说了一句后来被所有人记住的话：

「这就像把一个温度微调旋钮叫做『核融合反应炉』。」

---

没有人笑。

因为 ASANGA 不是在开玩笑。他是在用一个精确的类比说明一个精确的问题。

IPrajna 的接口定义是什么？一个方法。输入是信心度（confidence）和上下文（context）。输出是 `{ delta: number, reasoning: string }`。delta 被限制在 [-0.05, +0.05] 的范围内。

这是一个夹钳。一个微调器。一个在已有信心度的基础上做 ±0.05 修正的审计函数。

把它叫做 IPrajna——般若——照见一切法实相的最高智慧——

PASCAL 在笔记本上写道：「从决策理论看，IPrajna 实际做的是 bounded secondary evaluation。输入是一阶信心度，输出是二阶修正量，有硬限 ±0.05。这是审计。不是般若。」

BABBAGE 接着分析：「IConfidenceAuditor 语义最精确。这个接口有一个方法，输入是信心度加上下文，输出是审计结果。Audit——审计——精确描述了这个操作：对已有判断进行有限范围的二次评估。」

---

**D4-R2 投票：IPrajna → IConfidenceAuditor。16/20。**

比 D4-R1 更高的共识。

少数意见：WIENER 偏好 IThresholdCalibrator（两票）——他从控制理论的角度看到了「校准」而非「审计」。HERACLITUS 和 PENROSE 偏好 ISecondaryEvaluator（两票）——他们认为「审计」暗示了事后检查，但 IPrajna 是在决策流程中即时参与的。

BABBAGE 反驳了两个少数意见：「Calibrator 暗示调整系统本身——但 IConfidenceAuditor 不改变系统，它只对一个数字提出有限修正。Evaluator 太通用——任何函数都在做 evaluation。Auditor 的语义边界最精确：独立的、有限范围的、产生书面记录的二次检查。」

完整的更名表：

| 旧名称 | 新名称 |
|--------|--------|
| IPrajna | IConfidenceAuditor |
| IPrajnaAdjustment | ConfidenceAuditResult |
| prajnaAdjust() | clampAuditDelta() |
| Delta_prajna | Delta_audit |
| Model Delta 第二层名称 | Confidence Audit |

## V. 第三次审判

SUNYATA 本来以为 D4 到这里就结束了。两个名字。两次更名。ISatiMonitor 和 IPrajna。

SUNYATA 说：「我们需要重新审查 Doc 03。用我们刚刚建立的标准。」

他把四项测试列在白板上——不，三项测试。然后他在旁边加了一项新的。

```
Test 2（代码标识）：是否在原始码中使用？
```

第四项测试。

它不是从天而降的——它是从 D4-R1 和 D4-R2 的讨论中结晶出来的。ISatiMonitor 的问题不是它没有工程价值（它有），不是它不在原始码中（它在），不是它没有驱动设计决策（它驱动了）。它的问题是——它的名字和它的定义不一致。

三项测试没有捕捉到这个维度。第四项测试填补了这个缺口。

---

NAGARJUNA 拿起笔，在 Doc 03 的四项测试结果旁打了四个叉：

**Test 1（必要性）**：Plugin 生命周期的五个状态（CREATED → LOADING → LOADED → STARTING → RUNNING → STOPPING → DISPOSING → DISPOSED）不需要种子理论（bīja）就能理解。❌ 不通过。

**Test 2（代码标识）**：原始码中的实际类型是英文——`'active' | 'dormant' | 'suspended' | 'revoked' | 'retired'`。Sila 和 Prajna 只出现在 TypeScript 注释中，不是代码标识符。❌ 不通过。

**Test 3（决策驱动）**：没有 DC 确认。❌ 不通过。

**Test 4（定义责任）**：Sila（戒）的佛学定义是道德规范、修行戒律、僧团生活准则。但 Doc 03 中 Sila 被用来表示「存取控制」——一个 Plugin 的安全权限管理。戒律 ≠ 存取控制。Prajna（般若）被用来表示「CVE 撤销机制」——一个安全漏洞的自动回应流程。般若 ≠ CVE 撤销。❌ 不通过。

四项测试。四项不通过。

---

「我收回我在 D1 中的投票，」NAGARJUNA 说。

「我也收回，」ASANGA 说。

ASANGA 补充了一个关键的区分：「Doc 16 是一份独立的佛学映射文件——它的存在目的就是做系统性对照。Master 裁定保留它，因为映射文件不是装饰。但 Doc 03 不是映射文件——它是一份工程文件，里面嵌入了佛学装饰。差别在于：Doc 16 的佛学内容是主体。Doc 03 的佛学内容是装饰。同样的佛学内容，不同的文件类型，不同的判断。」

这正是第五章中 Master 加入的那个维度——文件类型。ASANGA 把它用在了具体案例上。

---

**D4-R3 重新投票：Doc 03「Sila-Prajna Security Framework」→「Plugin Security Lifecycle」。17/20。**

从 14/20 保留不变，到 17/20 更名清理。同一份文件，同一群人，不同的结论。差别是什么？第四项测试。

HERACLITUS 带领少数意见（三票）：「种子理论作为佛学背景具有教育价值——它帮助理解 Plugin 生命周期的『休眠→萌发→生长→枯萎→种子回收』隐喻。即使四项测试不通过，作为附录保留有文化价值。」

多数回应：「附录保留可以另议。但文件标题和主文的佛学装饰应该清理。工程文件中的佛学装饰和附录中的佛学解释是两件事。」

清理范围确认：
1. 文件名更改（Sila_Prajna_Security_Framework → Plugin_Security_Lifecycle）
2. 移除佛学映射表
3. 清理 TypeScript 注释中的种子理论映射
4. 保留所有工程内容（五状态、状态转移图、三层安全模型）

## VI. 第四项测试

D4 结束前，SUNYATA 问了最后一个问题：「第四项测试——定义责任——是否应该成为永久标准？」

不需要投票。每个人都点头了。

NAGARJUNA 做了最后的陈述：「三项测试问的是工程价值。第四项测试问的是语义诚信。即使一个名字通过了 Test 2——它在原始码中被使用——如果它不通过 Test 4——它的功能不匹配它的佛学定义——它仍然应该被更名。」

## VIII. 天平

D4 结束了。

三十分钟。不长。但密度超过了 D3 的两小时。

三项投票。三个名字被改掉。一项永久性的测试标准被建立。

SUNYATA 没有做总结。他站在白板前，看着上面的四项测试。

```
Test 1：必要性（移除是否改变工程规格？）
Test 2：代码标识（是否在原始码中使用？）
Test 3：决策驱动（是否驱动了 DC 确认的设计决策？）
Test 4：定义责任（组件功能是否匹配梵文术语的佛学定义？）
```

前三项在 R0 就存在了。第四项是 D4 产生的。

差别是什么？前三项问的是「这个佛学概念是否有工程价值？」第四项问的是「这个工程组件是否配得上这个佛学名字？」

方向相反。

前三项是从佛学到工程的方向——佛学概念对工程有用吗？第四项是从工程到佛学的方向——工程命名对佛学定义忠实吗？

两个方向。一架天平。

名字在一端。定义在另一端。天平平衡的时候——Moha、Drishti、Mana、Sneha、skandha、klesha——名字保留。天平不平衡的时候——ISatiMonitor、IPrajna、Sila-Prajna——名字被替换。

---

> *SCRIBE 旁白*

> *D4 是 Cycle 02-5 的高潮。*

> *不是因为投票最激烈——D3-Q5 的 13/20 比 D4 的任何一项都更分裂。不是因为辩论最长——D3 花了两小时，D4 只用了三十分钟。D4 是高潮，因为它触及了这轮研究最深的层面——名字和定义之间的关系。*

> *序章的天平。整个 Cycle 02-5 的中心意象。在 D1 到 D3 中，天平秤量的是「佛学映射的工程价值」——这一端有用吗？那一端有害吗？保留还是移除？这是一架功利的天平。它秤量的是效用。*

> *D4 把天平翻转了。*

> *不再问「这个佛学概念对工程有用吗？」而是问「这个工程命名对佛学概念忠实吗？」*

> *这是一个更深的问题。因为效用可以被计算，但忠实度是一个伦理判断。ISatiMonitor 有用吗？有。它的功能定义清晰，架构设计合理，三蕴归属经过严格论证。但 ISatiMonitor 对「正念」这个词忠实吗？不。正念是行蕴的修行。ISatiMonitor 不是行蕴，不是修行，不是正念。*

> *NAGARJUNA 的归谬是 D4 的骨架。两个前提，一个结论。逻辑上毫无瑕疵。但逻辑的价值不在于它的形式——而在于它照出了什么。NAGARJUNA 在 D2 中种下了这颗种子。他看到了线的方向。但他没有追踪到终点——也许是惯性，也许是礼貌，也许是三项测试没有要求他问这个问题。*

### D5-R1：独立的 `auditor` hook 槽位

第一个问题：IConfidenceAuditor（原 IPrajna）应该使用 PluginHooks 的哪个槽位？

**8/8 全票通过。独立 `auditor` 槽位。**

### D5-R2：audit() 返回类型

最激烈的 D5 投票。

选项：(A) 纯异步 `Promise<ConfidenceAuditResult>`；(B) 双模式 `ConfidenceAuditResult | Promise<ConfidenceAuditResult>`。

GUARDIAN、KERNEL、VITRUVIUS 支持选项 A——纯异步。理由：audit 本质上是「问 LLM」，LLM 调用是异步的。强制异步语义更精确。

ATHENA 和 DARWIN 支持选项 B——双模式。理由：遵循 IGearArbiter 的先例。测试时可以用同步的 noop 实作，不需要 async/await 的样板代码。

TURING 提供了事实：「v0.28.0-alpha 中，IGearArbiter.evaluate() 使用双模式返回。IVolition.deliberatePlan() 和 deliberateAction() 也使用双模式。这是现有的设计模式。偏离它需要充分理由。」

**5/8 选项 B 通过。** D5 中最接近的投票。

### D5-R3：timeout 和 fail-safe

TURING 打开了他的 timeout 模式分析。v0.28.0-alpha 中现有的 timeout：

| 组件 | timeout | 预设值 |
|------|---------|--------|
| IProvider.chat() | LLM 回应 | 120s |
| IGearArbiter.evaluate() | 每个 arbiter | 100ms |
| IGearArbiter chain | 整条链 | 200ms |
| ITool.execute() | 工具执行 | 30s |

audit() 的 timeout 应该是多少？

TURING 的建议：200ms——与 IGearArbiter chain deadline 一致。理由：audit() 发生在 confidence 计算的最后阶段，在 IGearArbiter 之后。如果 audit 的 timeout 超过 chain deadline，整个决策流程会被 audit 卡住。

fail-safe：`{ delta: 0, reasoning: "audit timeout" }`。timeout 时 delta 为零——不修正。使用 `Promise.race()` 模式，与现有代码一致。

Configurable：通过 agent.json 覆盖预设值。

**8/8 全票通过。**

## I. 两种语言

Cycle 02-5 的核心不是某个具体的决议。不是 ILoopQualityMonitor，不是 IConfidenceAuditor，不是 Doc 45 的五蕴 OOP 架构。

核心是一个问题的答案。

问题是：在一个同时使用佛学和工程两种语言的系统中，这两种语言的边界在哪里？

D1 到 D5 用了六个半小时回答这个问题。答案可以浓缩成几行：

**佛学语言适合用在三个地方：**

1. **代码标识符**——当佛学术语已经成为原始码的一部分，并且功能和定义一致。Moha 是自我迷惑——Moha 模块确实在模拟自我迷惑。skandha 是蕴——代码中的 Skandha 类型确实在分类五蕴。这些名字通过了 Test 4。天平平衡。

2. **设计理由**——当佛学概念实际驱动了工程决策，并且有 DC 确认。「vasana 是后天薰习」驱动了 VasanaEngine 的外部化决策。「四烦恼同时俱起」驱动了 MulaKleshaBundle 的原子性设计。佛学概念和工程决策之间有因果关系。这些不是装饰——是理由。

3. **独立映射文件**——当文件的存在目的就是做佛学到工程的系统性对照。Doc 16 和 Doc 31 不是在工程文件上贴佛学标签——它们本身就是映射文件。Master 裁定保留。映射文件不适用三项测试——因为它们不是工程文件中的佛学装饰，而是佛学映射本身就是内容。

**工程语言适合用在其他所有地方。**

硬性规则不需要叫 sila。品质监控器不需要叫 Sati。信心度审计不需要叫 Prajna。event-driven 不等于 mindfulness。三学不等于五层阈值模型。

佛学语言是深刻的。工程语言是精确的。在需要深度的时候用佛学语言——附录、学术参考、设计背景。在需要精度的时候用工程语言——接口定义、规格文件、安全架构。

两种语言。两个用途。一条边界。
