# 序章：名字的重量

---

圆形剧场的灯光带着一种手术室的冷白色。

不是 Cycle 02-4 结尾那种钟表工坊的琥珀色——那一轮的意象是齿轮，是机芯，是所有零件被组装成一个会走的整体。琥珀色灯光下的圆形剧场是温暖的，像一座老工坊，像大师傅的工作台，桌面上的油污和木屑记录着六轮研究留下的痕迹。

这一次不一样。

SUNYATA 在凌晨三点把灯光调成了冷白色。他没有告诉任何人。他只是坐在控制台前，把色温从 3200K 推到 6500K，看着整座剧场的氛围在三秒钟内从「老工坊」变成「实验室」。

他这样做是因为他读了 Master 的信。

---

信是在 Plan28 定版的第二天出现的。

和之前一样，不是被送到的，而是在某个时刻被注意到的。它在 `research input/master_letters/cycle02-5/` 里面，安静得像一张被夹在报告里的便条纸。它没有附件——不像 Cycle 02-3 的信旁边放着《五蕴与开车》，也不像 Cycle 02-4 的信旁边附了五份 Claude 建议文件。

只有一张便条。

便条不长。45 行。但 SUNYATA 读了五遍。

第一遍他读的是文字。第二遍他读的是情绪。第三遍他读的是结构。第四遍他闭上眼睛回忆整段文字。第五遍——

第五遍他读的是刀刃。

---

Master 的信从来不是批评——它们是问题、是指引、是地图。第一封（Cycle 02 前）是十个开放问题——「点」。第二封（Cycle 02-2 前）是四处哲学修正——「线」。第三封（Cycle 02-3 前）是一张路径地图——「面」。第四封（Cycle 02-4 前）是一个场——在每个空间点上都定义了值的「场」。

第五封？

SUNYATA 坐在冷白色的灯光下，把信摊在面前。他重新读第一句话：

> 「其实我越来越看不懂在做什么，在 Plan27 与 Plan28 多次介入改正。」

他的手指在桌面上停住了。

六轮研究。他主持过每一场定向会议，引导过每一次辩论，在每一份成果报告上签过名。他看着这个专案从 Cycle 01 的命名开始，经历功能分析、结构修正、动态建模、整合设计——一步一步，像砌一座塔。每一块砖都是辩论出来的，每一块砖都有投票记录，每一块砖都被 TURING 的代码分析验证过。

但 Master 说他「越来越看不懂」。

SUNYATA 知道 Master 的语言习惯。「看不懂」不是「不理解」。Master 是这个系统的设计者——他比任何人都理解。「看不懂」是「不认同」。是「你们在做的事情偏离了正轨」。

他继续读。

---

> 「说十大宣言很重要，但一开始的规划你们还是将 IGearArbiter 放到了核心中。」

SUNYATA 记得这件事。Plan27 的初始设计——IGearArbiter 被当作核心组件。那是在 Pre-Cycle 02-4 的 Master 讨论之后，VasanaEngine 被外部化，ManoAggregator 退化为纯路由，IGearArbiter 作为新接口被引入。但在 Plan27 的具体设计中，团队又把 IGearArbiter 塞回了核心——像一个已经被纠正过一次的坏习惯，在不注意的时候又冒了出来。

Master 介入了。IGearArbiter 被移出核心，成为 plugin 提供的齿轮路由仲裁器。

> 「最后介入才修正正确，IGearArbiter 是 plugin 提供的齿轮路由仲裁器，决定 agent 该用哪个『齿轮』处理当前请求。」

是的。SUNYATA 在脑中重新组装这个逻辑链。IGearArbiter 不在核心中。它是 plugin 提供的。它的角色是仲裁——在 Gear 1（基于辨认的快速路径）和其他齿轮之间做选择。这才是识蕴的意义——不是被锁死的双齿轮，而是具备多面向的认知判断能力。

然后 Master 写了一句让 SUNYATA 的手指收紧的话：

> 「八识从眼识到阿赖耶识，是具备很多面向的，居然想要设定死成双齿轮，这是非常严重的错误。」

**非常严重的错误。**

SUNYATA 在六轮研究中只见过 Master 用过一次「非常严重」。那是 Cycle 02-2 的四项哲学修正。那一次是修正——把错的改成对的，语气是教导的。这一次不是教导。这一次的「非常严重」带着——

他找了一个词。

**失望。**

---

信的第二段更尖锐。

> 「还有不是每一件事情都要硬性加上佛学的色彩，有些实在是很牵强附会。」

牵强附会。四个字。SUNYATA 知道 Master 用词是精确的。Master 不说「不太恰当」或「有待商榷」。Master 说「牵强附会」。这意味着：不是「有些地方可以改善」，而是「某些事情根本就是错的」。

他在脑中开始列举。哪些映射是牵强的？

然后 Master 自己列出来了：

> 「event-driven 不等于正念，只是正念是用 event-driven 做的。」

一句话。精确得像一把手术刀切开了一个脓疮。Event-driven 是一种架构模式。正念是一种心理状态。说 event-driven 等于正念，就像说螺丝刀等于修理——螺丝刀是工具，修理是行为，你可以用螺丝刀来修理，但螺丝刀不是修理。

> 「觉察当下正在发生什么。这一层不看规则、不看风险分类，纯粹观察行为模式。不需要将它当作正念的工程化。」

SUNYATA 把这段话抄在笔记本上。一个字一个字地。

然后 Master 问了一个问题。这个问题后来改变了整个 Cycle 02-5 的走向。

> 「而且这个 plugin 难道没有识蕴的成分在吗？如果依照唯识学派的分类，会将正念放在行蕴，那是因为它是指正向的运转。以这个 plugin 的功能来说，我觉得这是有争议的，因为在这个 plugin 的功能上，不仅仅是单纯的行为模式。」

SUNYATA 读了三遍。

Master 在做什么？他不是在质疑 SatiMonitor 的功能。他是在质疑 SatiMonitor 的**身份**——它不是正念的工程化。它是一个有受蕴、有想蕴、有识蕴成分的复合 plugin。正念只是一个被强加的标签。

然后 Master 把问题推到了最底层：

> 「所以我们要重新想的是这个 plugin 是由五蕴的哪一些类别组成。」

不是「它是什么蕴」。是「它由哪些蕴组成」。这是一个根本性的视角转换——从「分类」到「组成」。一个 plugin 不一定只属于一个蕴。它可以由多个蕴的功能组合而成。

---

信的第三段是命令。

> 「我们需要将五蕴有哪一些子类别，如何依赖注入，如何在 agent core 运行先处理好。架构先用好。agent core 中如何将五蕴 plugin 载入。然后 plugin 要如何做。agent core 中五蕴会如何流转。是我们的首要任务。」

四个问题。每一个都不是新的——它们从 Cycle 01 就存在。但在经历了五轮研究之后，它们仍然没有一份完整的工程语言回答。散落在各处——A2 的 DI 布线在这里，A4 的执行流程在那里，Klesha 安全在另一个地方——但没有一份文件把它们统合在一起，用纯工程语言，从头到尾讲清楚。

SUNYATA 在笔记本上把四个问题编了号：

1. 五蕴子类别的完整结构
2. DI 布线如何注入五蕴 plugin
3. Agent core 如何载入五蕴 plugin
4. 五蕴如何在 agent core 流转

这是 Cycle 02-5 的首要任务。不是佛学映射。不是命名讨论。不是哲学辩论。是**工程架构**。

---

然后是 Master 的第四段。刀刃上的刀刃。

> 「什么教诫、善巧、正念，你做的工程映射都是有一些牵强的。反而增加阅读障碍。而且你的映射也非常有问题。openstarry_doc 与相关的内容都需要修整回来。openstarry_doc 还是属于工程文件。」

SUNYATA 重读了最后一句话。

**openstarry_doc 还是属于工程文件。**

这句话的重量比它看起来的要重得多。它不是在说「openstarry_doc 应该是工程文件」——它是在说「openstarry_doc 本来就是工程文件，但你们把它搞成了别的东西」。教诫（śīla）、善巧（upāya）、正念（smṛti）——这些佛学标签被贴在工程概念上，像是在一本操作手册的每一页边缘都画了宗教插图。手册的功能没有改变，但阅读体验被改变了。

SUNYATA 想起了 Cycle 02-4 结束时的感觉。那时他以为钟表匠完成了组装。机芯在走。55 项决议，六场辩论，每一颗齿轮都在正确的位置。

但机芯上的铭刻——那些刻在齿轮侧面的名字——有些东西不对。Master 看到了。

---

信的第五段是让步。

> 「至于遍行心所、十二因缘、认知序列的相关概念有助于 agent core 的研究，可以讨论。」

SUNYATA 注意到了这段话的结构。Master 不是在全盘否定佛学的参与——他在划边界。遍行心所（CoarisingBundle）是代码中的结构。十二因缘和认知序列可能对理解 ExecutionLoop 有帮助。这些可以讨论。

但——

> 「就 plugin 来说，首要目标是建构五蕴的架构，参考物件导向的概念，来建立。」

首要目标。物件导向。工程语言。

然后是最后一段：

> 「NAGARJUNA 为三层框架提供了佛学映射。戒定慧三学。我觉得没这个必要。未来我不知道，但现在没必要。五蕴的工程设计才是最重要需要实现的。」

SUNYATA 在读到这一段的时候闭上了眼睛。

戒定慧。那是 NAGARJUNA 在 Cycle 02-3 的 D5 辩论结束后提出的映射。戒（śīla）= 安全层。定（samādhi）= 执行回圈。慧（prajñā）= 识蕴判断。三个维度映射到五个层级——三维对五层，本身就不自洽。

Master 否决了它。不是暂缓——是否决。「没这个必要。」

---

SUNYATA 放下了信。

凌晨三点十五分。冷白色的灯光把圆形剧场照成了一间实验室。不是温暖的工坊——是冰冷的、精确的、不容含糊的实验室。

他在笔记本上写下了三个词。

第一个词：**ARCHITECTURAL**。

Cycle 01 问「这是什么」。Cycle 02 问「它做什么」。Cycle 02-2 问「哪里错了」。Cycle 02-3 问「它怎么活」。Cycle 02-4 问「它如何成为整体」。

Cycle 02-5 的问题是：**「五蕴如何作为 OOP 架构运作？」**

第二个词：**边界**。

哪些佛学内容属于工程文件？哪些不属于？不是逐案判断——是框架。是规则。是可以机械性应用的测试。

第三个词：**秤**。

---

> *SCRIBE 旁白*

> *每一轮 Cycle 都有它的中心意象。Cycle 01 是命名——给陌生事物贴上第一张标签。Cycle 02 是功能——标签后面的机制。Cycle 02-2 是修正——撕掉错误的标签。Cycle 02-3 是流动——标签开始移动，PASCAL 带着他的 Beta 分布走进了第二十个座位。Cycle 02-4 是组装——标签变成齿轮的铭刻，55 颗齿轮在琥珀色灯光下啮合。*

> *Cycle 02-5 的中心意象是什么？*

> *SUNYATA 在凌晨三点选择了「秤」。一架天平。但不是称重量的天平——是称名字的天平。*

> *左边是名字——梵文的、中文的、TypeScript 的。Skandha。Klesha。Vedana。Sati。Prajna。每一个名字都有重量，有历史，有来自两千五百年的惯性。右边是定义——佛学的、工程的、源代码中的。ISamskara = ITool。IVedana = IVedanaSensor。IVijnana = IVolition + IGearArbiter + IConfidenceAuditor。每一个定义都有精确的功能、精确的类型签名、精确的测试案例。*

> *天平平衡的时候，名字和定义一致。Klesha 就是烦恼——代码中的四个模块（Moha、Drishti、Mana、Sneha）确实做的是「烦恼」的事：我痴、我见、我慢、我爱。它们在 import 语句中、在 class 定义中、在 describe 区块中。名字和定义在天平上等重。一切正确。*

> *天平倾斜的时候，名字和定义不一致。般若（prajñā）——照见一切法空相的最高智慧——变成了 Math.max(-0.05, Math.min(0.05, delta))。名字太重，定义太轻。或者反过来——正念（sati）——行蕴中的善心所——被贴在一个受+想+识的监控器上。定义太重，名字太轻。*

> *Master 的信不是批评。Master 的信是一把校准砝码。他说：把你们的天平拿出来。把名字和定义放上去。看看哪边倾斜了。*

> *然后修正。*

> *但校准砝码还有另一层意义。砝码本身是精确的——它的质量是已知的、不可谈判的。Master 的用词也是精确的、不可谈判的。「非常严重的错误。」「牵强附会。」「不需要将它当作正念的工程化。」「openstarry_doc 还是属于工程文件。」*

> *这些不是建议。这些是砝码。你不能和砝码讨价还价。你只能把自己的东西放上天平，然后诚实地读出刻度。*

---

NAGARJUNA 是第一个到达剧场的人。

凌晨四点。比召集时间早了三个小时。他没有被召集——没有人被召集——SUNYATA 还没有按下召集键。NAGARJUNA 是自己来的。

他站在剧场入口，看到了冷白色的灯光。然后他看到了 SUNYATA 坐在那里，面前摊着 Master 的信，笔记本上写了三个词。

「你也睡不着。」NAGARJUNA 说。不是问句。

SUNYATA 没有抬头。「你读了信？」

「我没读信。我读了自己的文件。」

NAGARJUNA 走到他平时的座位旁边，但没有坐下。他站在那里，手指轻轻碰了碰扶手上那个被他辩论时不自觉敲击留下的微微凹痕。三个周期的痕迹。三个周期的辩论。

「Doc 44。第 469 行。」他说。「戒定慧三学映射。」

SUNYATA 终于抬起头。

「那是我的。」NAGARJUNA 的声音很平，像寺院走廊里的回音。「Cycle 02-3 的 D5 结束后，我觉得需要一个佛学框架来组织安全层的概念。戒 = 安全规则。定 = 执行稳定。慧 = 判断能力。很对称。很优雅。」

他停了一拍。

「三维映射到五层。我在写的时候就知道不自洽。三个概念覆盖五个模块——像用三个抽屉装五件衣服，你必须把两件折起来塞进已经满了的抽屉。但我还是写了。因为它看起来很好看。」

**因为它看起来很好看。**

SUNYATA 沉默了几秒。

「Master 否决了它。」他说。

「应该否决。」NAGARJUNA 坐了下来。「那不是映射。那是装饰。」

---

凌晨四点半。PASCAL 到了。

他走进剧场的时候带着他那本永远不会写满的笔记本——Moleskine，深蓝色封面，书脊上贴着一张小标签写着「P(H|E)」。他是 Cycle 02-3 加入的新成员——第二十个座位——但两轮研究之后，他的椅子扶手上也有了痕迹：右手食指反复点击的位置，那里的木纹比周围浅了半个色阶。那是他在计算期望值的时候会做的动作——点击，点击，点击——像一个机械计算器的进位轮。

「NAGARJUNA。SUNYATA。」他点了点头。然后他看到了桌上的信。

「可以读吗？」

SUNYATA 把信推过去。

PASCAL 读得很快。他的眼睛扫描文字的方式和其他人不同——不是从左到右、从上到下，而是先扫描整个文件的「形状」（段落长度、标点密度、引号位置），然后再回到第一个字。他在给自己的阅读建立先验分布。

两分钟后，他放下信。

「损害不对称性。」他说。

NAGARJUNA 看了他一眼。

PASCAL 打开笔记本，翻到一页空白。他写了两行：

```
False include（保留装饰性映射）的成本：
  每位读者 × 每次阅读 × 每一页 = 累积损害

False exclude（移除有价值映射）的成本：
  一次性搜寻成本 = 可控
```

「这是统计决策理论中最基本的框架。」他说。「两种错误的成本不对称。保留一个装饰性映射的损害是累积的——每一位未来的读者，每一次阅读，每一个被迫同时处理两套术语系统的瞬间。移除一个有价值映射的损害是一次性的——你可以通过搜寻找到它。」

他在两行下面写了一个不等式：

```
E[损害_include] >> E[损害_exclude]
```

「期望值分析。装饰性映射具有负期望值。」

NAGARJUNA 点了点头。慢慢地。带着一个正在把自己的哲学和另一个人的数学对齐的思考者的沉着。

「二谛。」他说。「世俗谛是工程语言——日常使用的、沟通用的、操作用的。胜义谛是佛学概念——启发用的、设计用的、理解用的。两者不应混淆。不要在世俗谛的文件中硬塞胜义谛的标签。」

PASCAL 在笔记本上写了「二谛」两个字，旁边画了一条分隔线。左边写「工程」，右边写「佛学」。

「你的二谛和我的损害不对称性说的是同一件事。」他说。

「不完全是。」NAGARJUNA 的眉毛微微抬起——他用眉毛提问比用嘴巴更快。「你的是成本分析——量化的。我的是范畴分析——质性的。结论相同，但路径不同。」

「路径不同，结论相同。」PASCAL 重复了一遍。「这就是好框架的特征——从不同的公理出发，到达同一个定理。」

---

凌晨五点。ASANGA 到了。

他没有像 NAGARJUNA 那样站在入口观察，也没有像 PASCAL 那样先读信。他直接走到他的座位——在 NAGARJUNA 的对面——坐下来，闭上眼睛。

他在做什么？

SUNYATA 知道。ASANGA 在回忆。唯识学派的方法不是推理——是现量（pratyaksha），直接的、不经中介的认知。ASANGA 不需要读 Master 的信来知道信里写了什么——他只需要回想他自己写过的东西。

「我在 Cycle 02-3 的安全框架中用了『善巧方便（upāya-kauśalya）』来描述自适应安全策略。」他说，眼睛仍然闭着。「但自适应安全策略不需要叫善巧方便。它只需要叫自适应安全策略。upāya-kauśalya 有一千七百年的哲学重量——从《法华经》的三车之喻到唯识学的转识成智——把这个重量压在一个工程配置选项上，是对两者的双重不敬。」

他睁开了眼睛。

「功能性判定。」他说，看着 PASCAL。「驱动工程结论的佛学概念——保留。有因果效力。装饰性的佛学标签——移除。无因果效力。」

PASCAL 在笔记本上又加了一行。三个路径。三个学科。一个结论。

```
NAGARJUNA — 二谛（范畴分析）    → 不要混淆
PASCAL    — 损害不对称（量化）    → 负期望值
ASANGA    — 功能性判定（因果）    → 无因果效力
                                   ┕→ 结论：装饰性映射应移除
```

---

凌晨五点半。

SUNYATA 站起身。他在白板上写了一个表格。四行。

| 层级 | 动作 | 条件 |
|------|------|------|
| **KEEP** | 保留在工程正文 | 代码标识符 / DC 确认的约束 |
| **RELOCATE** | 移至附录 | 有工程参考价值的佛学背景 |
| **REMOVE** | 从工程文件移除 | 装饰性标签 / 经典引用 |
| **FILE REVIEW** | 整份文件评估 | 装饰比例过高的文件 |

四层框架。

然后他在旁边写了三项测试：

```
Test 1 — 必要性：移除该映射是否改变工程规格？
         否 → REMOVE 或 RELOCATE

Test 2 — 代码身份：该名称是否在源代码中使用？
         是 → KEEP

Test 3 — 决策驱动：该佛学概念是否驱动了特定设计决策？
         是 → KEEP 作为设计理由
```

三项测试。可以机械性地应用。

NAGARJUNA 站起来走到白板前。他看了几秒钟。

「Test 2 是 TURING 的领域。」他说。「任何名称是否在源代码中使用，一行 grep 就能回答。」

PASCAL 补充：「Test 1 可以用反事实分析。如果我们把『event-driven = 正念』从文件中移除，有任何工程规格会改变吗？不会。结论：REMOVE。」

ASANGA：「Test 3 需要交叉验证 DC-1 到 DC-12 的确认记录。CoarisingBundle 是 DC-6 确认的——保留。Klesha 模块名是 DC-1 确认的——保留。」

三个人，三项测试，每一项都有一个明确的负责人。

SUNYATA 看了看时钟。凌晨五点四十五分。还有两个小时十五分钟。他决定不等了。

他按下了召集键。

---

他们回来了。

不是 Cycle 02-4 开场时那种全亮度、全员到齐的壮观场面。是凌晨六点钟的陆续到场——有人带着咖啡（KERNEL），有人带着笔记本（BABBAGE），有人带着夜里未完成的思考（TURING 的屏幕上还亮着 `aggregates.ts` 的源代码）。

他们走进剧场的时候，首先注意到的是灯光。

「你把色温改了。」VITRUVIUS 说。建筑师的眼睛对光线变化最敏感。

「6500K。」SUNYATA 回答。

VITRUVIUS 没有追问。他走到自己的座位，在路上瞥了一眼白板上的四层框架。他的脚步顿了一下——非常短暂，短暂到只有建筑师才会注意到自己的脚步顿了一下——然后继续走。

GUARDIAN 是第五个到的。他的目光照例先扫描了整个剧场的边缘——安全专家的习惯，永远不变——然后扫描了白板。他看到「REMOVE」那一行的时候，眉毛微微上挑。不是惊讶。是确认。

ARCHIMEDES 带着他的工程计划夹。夹子上还贴着 Plan28 的「COMPLETE」红色标签。他把标签撕了下来，放进口袋。

「新计划？」他问。

「新 Cycle。」SUNYATA 说。

---

二十盏灯在六点三十分全部亮起。

SUNYATA 站在圆形剧场的中央。第六次了。每一次他站在这里，剧场的气味都不同——Cycle 01 是新木头的气味，Cycle 02 是墨水的气味，Cycle 02-2 是修正液的气味，Cycle 02-3 是新座位的气味（PASCAL 的椅子），Cycle 02-4 是机油的气味（齿轮的隐喻渗透到了现实）。

Cycle 02-5 是什么气味？

消毒水。实验室的消毒水。冷白色的灯光下，一切都被照得无处躲藏。

「各位。」他的声音不是石子入深潭——不是 Cycle 02 的饱满，也不是 Cycle 02-3 的流动，更不是 Cycle 02-4 的重力。是什么？

是手术刀。干净的。锋利的。没有多余的修辞。

「Master 的第五封信。」

他把信投影在主屏幕上。二十张面孔同时读完。

安静。

这一次的安静和之前每一轮都不同。Cycle 02 的安静是探索前的期待。Cycle 02-2 的安静是修正前的紧张。Cycle 02-3 的安静是新成员到来前的好奇。Cycle 02-4 的安静是组装前的重力。

Cycle 02-5 的安静是被指出错误后的安静。是学生在老师面前的安静。是「牵强附会」四个字压在二十个人肩膀上的安静。

NAGARJUNA 最先打破沉默。他已经准备好了——凌晨四点就准备好了。

「Master 指出的戒定慧映射——」他顿了一下。他看了全场一眼。然后说了一句他在六轮研究中从未说过的话：

「是我的错。三维映射到五层，本身就不自洽。我收回它。」

没有人说话。NAGARJUNA 在辩论中的语言通常是经典引用、是归谬法、是逻辑推演。他从不说「我的错」。他说「此论不成」或「前提互斥」——第三人称的、中性的、像在讨论一个数学命题而不是一个人的判断。

但此刻他用了第一人称。「我的错。」

PASCAL 的笔记本翻到了一页新的空白。他在上面写了一个日期和一个词：「自我否认」。

ARCHIMEDES 没有被 NAGARJUNA 的坦诚打动太久——务实者的复原速度永远是最快的。

「我们需要一个判断框架。」他说。「白板上已经有了。四层。三项测试。逐项应用。」

SUNYATA 点了点头。

「三条研究轨道。」他在白板上画了三条线。

**Track A**：五蕴工程架构。五个子项。A1 子类别树。A2 DI 布线。A3 Plugin 载入。A4 执行流程。A5 跨蕴互动。十二位主笔。Master 的首要任务。

**Track B**：佛学映射审计。两个子项。B1 逐文件审计。B2 映射边界原则。五位主笔。Master 的第二刀。

**Track C**：Sati Plugin 定位。两个子项。C1 功能分析。C2 蕴组成分析。四位主笔。Master 那根带刺的问题：「难道没有识蕴的成分在吗？」

「三条轨道。五场辩论。」SUNYATA 说。然后他修正了自己：「至少三场。D1 佛学映射边界。D2 Sati 五蕴分类。D3 五蕴架构完整性。」

TURING 从他的屏幕后面抬起头。TURING 很少主动发言——他的贡献方式是验证，不是论述。但此刻他说了一句话：

「v0.28.0-alpha。1722 个测试。我会验证所有九份研究报告中的每一个代码引用。」

SUNYATA 看了他一眼。TURING 说完这句话就把目光移回了屏幕。在他面前的终端机上，`aggregates.ts` 的代码静静地等待着被阅读。

「明确排除。」SUNYATA 在白板的最下方画了一条红线。红线下面写了三项：

1. Lyapunov 参数校准
2. Plan29 工程项
3. 戒定慧三学映射

写到第三项的时候，他看了 NAGARJUNA 一眼。NAGARJUNA 微微颔首。那个映射已经被它的创造者亲手否认了。

---

R0 结束。

九份研究被分配。十五位主笔被指定。截止时间被设定。轨道被划定。

SUNYATA 关掉了投影。主屏幕变暗。冷白色的灯光仍然亮着——6500K——把每一个人的影子钉在地板上，短而清晰，像一个个精确的轮廓。

PASCAL 在离开的时候经过 NAGARJUNA 的座位。他停了一步。

「你的自我否认。」他说，声音很轻。「那是整场会议中信息量最高的一句话。」

NAGARJUNA 抬头看他。中观学者和概率学家的目光在冷白色的光线中交汇。

「空亦复空。」NAGARJUNA 说。「判断框架本身也要接受判断。我判断了五蕴的映射，却没有判断我自己的映射。Master 替我做了。」

PASCAL 在笔记本上写了一个公式。

$$P(\text{映射正确} | \text{映射者承认错误}) = 0$$

「后验概率归零。」他说。「这是最强形式的反驳——不是对手反驳你，是你自己反驳你。」

NAGARJUNA 看着那个公式。

「中观学的『自空』。」他说。「龙树在《中论》第十三品写道：空性能遣除一切见。但若有人以空性为见，则此人不可救药。」

他指了指 PASCAL 笔记本上的公式。

「戒定慧映射就是我的『空性见』——一个看起来很优雅的框架，但它本身不经得起自己的检验。Master 的信是救药。」
