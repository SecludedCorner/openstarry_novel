# 序章：名前の重さ

---

円形劇場の照明は、手術室のような冷たい白色を帯びていた。

Cycle 02-4 の終幕に灯されたあの時計工房の琥珀色ではない——あの回の中心イメージは歯車であり、ムーブメントであり、すべての部品が一つの動く全体へと組み上げられる過程だった。琥珀色の光に照らされた円形劇場は温かく、古い工房のようであり、親方の作業台のようであった。天板に染みついた油汚れと木屑が、六回の研究が残した痕跡を記録していた。

今回は違う。

SUNYATA は午前三時に照明を冷白色に変えた。誰にも告げなかった。制御卓の前に座り、色温度を 3200K から 6500K へ押し上げ、劇場全体の雰囲気が三秒で「古い工房」から「実験室」に変わるのを見つめた。

彼がそうしたのは、Master の手紙を読んだからだ。

---

手紙は Plan28 が定版された翌日に現れた。

これまでと同じく、届けられたのではなく、ある時点で気づかれたのだ。`research input/master_letters/cycle02-5/` の中に、報告書に挟まれた付箋のように静かに置かれていた。添付はない——Cycle 02-3 の手紙のように隣に『五蘊と運転』が置かれているわけでも、Cycle 02-4 の手紙のように五通の Claude 提案文書が添えられているわけでもなかった。

ただ一枚の付箋だけ。

長くはない。45 行。しかし SUNYATA は五回読んだ。

一度目は文字を読んだ。二度目は感情を読んだ。三度目は構造を読んだ。四度目は目を閉じて全文を想起した。五度目——

五度目は刃を読んだ。

---

Master の手紙はけっして批判ではない——問いであり、導きであり、地図だ。第一の手紙（Cycle 02 の前）は十の開放的問い——「点」。第二の手紙（Cycle 02-2 の前）は四箇所の哲学的修正——「線」。第三の手紙（Cycle 02-3 の前）は一枚の経路地図——「面」。第四の手紙（Cycle 02-4 の前）は一つの場——あらゆる空間点に値が定義された「場」。

第五の手紙は？

SUNYATA は冷白色の光の下で手紙を広げた。最初の一文を読み直す：

> 「正直なところ、何をやっているのかますます分からなくなってきた。Plan27 と Plan28 で何度も介入して修正した。」

彼の指がテーブルの上で止まった。

六回の研究。彼はすべての定向会議を主宰し、すべての討論を導き、すべての成果報告に署名してきた。Cycle 01 の命名に始まり、機能分析、構造修正、動的モデリング、統合設計を経て——一歩一歩、塔を積むように。どの煉瓦も討論から生まれ、どの煉瓦にも投票記録があり、どの煉瓦も TURING のコード分析で検証されていた。

しかし Master は「ますます分からなくなってきた」と言った。

SUNYATA は Master の言葉遣いを理解している。「分からない」は「理解できない」ではない。Master はこのシステムの設計者だ——誰よりも理解している。「分からない」は「同意できない」だ。「あなたたちのやっていることは軌道を外れている」ということだ。

彼は読み続けた。

---

> 「十大宣言が重要だと言いながら、最初の設計段階で IGearArbiter をコアに入れてしまった。」

SUNYATA はこの件を覚えていた。Plan27 の初期設計——IGearArbiter がコアコンポーネントとして扱われていた。Pre-Cycle 02-4 の Master との討論の後、VasanaEngine は外部化され、ManoAggregator は純粋なルーターに退化し、IGearArbiter が新しいインターフェースとして導入された。だが Plan27 の具体設計で、チームは再び IGearArbiter をコアに押し込んだ——一度矯正された悪い癖が、油断した隙にまた顔を出したかのように。

Master が介入した。IGearArbiter はコアから外され、plugin が提供するギアルーティング仲裁器となった。

> 「最終的に介入して正しく修正した。IGearArbiter は plugin が提供するギアルーティング仲裁器であり、agent がどの『ギア』で現在のリクエストを処理すべきかを決定する。」

その通りだ。SUNYATA は頭の中でこの論理連鎖を再構成した。IGearArbiter はコアにない。plugin が提供するものだ。その役割は仲裁——Gear 1（認識ベースの高速パス）と他のギアの間で選択すること。これこそ識蘊の意味だ——固定された二重ギアではなく、多面的な認知判断能力を備えていること。

そして Master は、SUNYATA の指を握り締めさせる一文を書いた：

> 「八識は眼識から阿頼耶識まで、非常に多くの側面を持っている。それを二重ギアに固定しようとするなど、非常に深刻な誤りだ。」

**非常に深刻な誤り。**

六回の研究で、SUNYATA が Master に「非常に深刻」という言葉を使われたのは一度だけだった。それは Cycle 02-2 の四項目の哲学的修正の時だ。あの時は修正——誤りを正すことであり、語調は教導的だった。今回は教導ではない。今回の「非常に深刻」には——

彼は言葉を探した。

**失望。**

---

手紙の第二段落はさらに鋭かった。

> 「それに、何でもかんでも無理に仏教色を付ける必要はない。こじつけが過ぎるものもある。」

こじつけ。四文字。SUNYATA は Master の用語が正確であることを知っている。Master は「やや不適切」とも「議論の余地がある」とも言わない。Master は「こじつけ」と言う。これが意味するのは：「改善の余地がある」ではなく、「根本的に間違っているものがある」ということだ。

彼は頭の中で列挙し始めた。どの対応づけがこじつけなのか？

そして Master 自身がそれを列挙した：

> 「event-driven イコール正念ではない。正念が event-driven で実装されているだけだ。」

一文。膿を切り開く手術刀のような正確さ。event-driven はアーキテクチャパターンだ。正念は心理状態だ。event-driven イコール正念と言うのは、ドライバーイコール修理と言うようなものだ——ドライバーは道具、修理は行為、ドライバーで修理はできるが、ドライバーは修理ではない。

> 「今まさに何が起きているかを覚知すること。このレイヤーはルールも見ないし、リスク分類も見ない。純粋に行動パターンを観察する。それを正念の工学化と見なす必要はない。」

SUNYATA はこの一節をノートに書き写した。一字一字。

そして Master は一つの問いを投げかけた。この問いが後に Cycle 02-5 全体の方向を変えることになる。

> 「しかもこの plugin には識蘊の要素が含まれていないとでもいうのか？唯識学派の分類に従えば正念は行蘊に置かれるが、それは正念が正しい方向の作用だからだ。この plugin の機能から言えば、これには議論の余地がある。この plugin の機能は単純な行動パターンだけではないからだ。」

SUNYATA は三度読んだ。

Master は何をしているのか？ SatiMonitor の機能を疑問視しているのではない。SatiMonitor の**アイデンティティ**を疑問視しているのだ——それは正念の工学化ではない。受蘊、想蘊、識蘊の要素を持つ複合的な plugin なのだ。正念はただ強制的に貼られたラベルに過ぎない。

そして Master は問いを最も根本的な層へと押し下げた：

> 「だから我々が改めて考えるべきは、この plugin が五蘊のどの類別で構成されているかということだ。」

「それが何の蘊か」ではない。「それがどの蘊で構成されているか」だ。これは根本的な視座の転換——「分類」から「構成」へ。一つの plugin は必ずしも一つの蘊だけに属するわけではない。複数の蘊の機能の組み合わせで成り立ちうる。

---

手紙の第三段落は命令だった。

> 「五蘊にどのようなサブカテゴリがあるか、どのように依存性注入するか、agent core 内でどう運用するか、まずこれを整理する必要がある。アーキテクチャを先に固める。agent core 内で五蘊 plugin をどう読み込むか。そして plugin をどう作るか。agent core 内で五蘊がどう流転するか。これが我々の最優先課題だ。」

四つの問い。どれも新しくない——Cycle 01 から存在していた。だが五回の研究を経てなお、それらには完全な工学的言語での回答がなかった。A2 の DI 配線はここに、A4 の実行フローはあちらに、Klesha の安全機構はまた別の場所に——散在していた。だが一つの文書がそれらを統合し、純粋な工学的言語で、最初から最後まで説明しきったものはなかった。

SUNYATA はノートに四つの問いを番号付きで書き込んだ：

1. 五蘊サブカテゴリの完全な構造
2. DI 配線はどのように五蘊 plugin を注入するか
3. Agent core はどのように五蘊 plugin を読み込むか
4. 五蘊は agent core 内でどのように流転するか

これが Cycle 02-5 の最優先課題だ。仏教的対応づけではない。命名の議論ではない。哲学的討論ではない。**工学アーキテクチャ**だ。

---

そして Master の第四段落。刃の上の刃。

> 「教誡だの善巧だの正念だの、あなたたちの工学的対応づけにはこじつけがある。かえって読解の障害を増やしている。しかもその対応づけにも大いに問題がある。openstarry_doc および関連する内容は修正しなければならない。openstarry_doc はあくまで工学文書だ。」

SUNYATA は最後の一文を読み返した。

**openstarry_doc はあくまで工学文書だ。**

この一文の重みは見た目以上だ。「openstarry_doc は工学文書であるべきだ」と言っているのではない——「openstarry_doc はもともと工学文書なのに、あなたたちが別物にしてしまった」と言っているのだ。教誡（śīla）、善巧（upāya）、正念（smṛti）——これらの仏教的ラベルが工学概念に貼り付けられ、操作マニュアルの各ページの余白に宗教的な挿絵を描いたようなものだ。マニュアルの機能は変わらない。だが読書体験が変わってしまった。

SUNYATA は Cycle 02-4 が終わった時の感覚を思い出した。あの時、時計師は組み立てを完了したと思っていた。ムーブメントは動いている。55 項の決議、六回の討論、すべての歯車が正しい位置に収まっている。

だがムーブメントに刻まれた銘——歯車の側面に刻まれた名前——何かがおかしい。Master にはそれが見えていた。

---

手紙の第五段落は譲歩だった。

> 「遍行心所、十二因縁、認知シーケンスの関連概念が agent core の研究に有益であるなら、議論しても構わない。」

SUNYATA はこの段落の構造に気づいた。Master は仏教の参与を全面否定しているのではない——境界を引いているのだ。遍行心所（CoarisingBundle）はコード内の構造体だ。十二因縁と認知シーケンスは ExecutionLoop の理解に有益かもしれない。これらは議論しても構わない。

だが——

> 「plugin に関しては、最優先目標は五蘊のアーキテクチャを構築し、オブジェクト指向の概念を参考にして確立することだ。」

最優先目標。オブジェクト指向。工学的言語。

そして最後の段落：

> 「NAGARJUNA は三層フレームワークに仏教的対応づけを提供した。戒定慧三学。私はその必要はないと思う。将来はわからないが、今は必要ない。五蘊の工学設計こそが最も重要で、実現すべきものだ。」

SUNYATA はこの段落を読んだとき目を閉じた。

戒定慧。それは NAGARJUNA が Cycle 02-3 の D5 討論終了後に提案した対応づけだ。戒（śīla）= 安全層。定（samādhi）= 実行ループ。慧（prajñā）= 識蘊判断。三つの次元が五つの層に対応づけられる——三次元対五層は、それ自体が自己矛盾している。

Master はそれを却下した。保留ではなく——却下だ。「その必要はない。」

---

SUNYATA は手紙を置いた。

午前三時十五分。冷白色の光が円形劇場を実験室に変えていた。温かい工房ではなく——冷たく、正確で、曖昧さを許さない実験室に。

彼はノートに三つの言葉を書き留めた。

一つ目：**ARCHITECTURAL**。

Cycle 01 は「これは何か」と問うた。Cycle 02 は「それは何をするか」と問うた。Cycle 02-2 は「どこが間違っているか」と問うた。Cycle 02-3 は「それはどう生きるか」と問うた。Cycle 02-4 は「それはどう全体になるか」と問うた。

Cycle 02-5 の問いは：**「五蘊は OOP アーキテクチャとしてどう機能するか？」**

二つ目：**境界**。

どの仏教的内容が工学文書に属するか？どれが属さないか？ケースバイケースの判断ではなく——フレームワーク。ルール。機械的に適用できるテスト。

三つ目：**秤**。

---

> *SCRIBE ナレーション*

> *すべての Cycle にはその中心イメージがある。Cycle 01 は命名——見知らぬものに最初のラベルを貼ること。Cycle 02 は機能——ラベルの背後にあるメカニズム。Cycle 02-2 は修正——間違ったラベルを剥がすこと。Cycle 02-3 は流動——ラベルが動き始め、PASCAL が彼の Beta 分布を携えて二十番目の席に着いた。Cycle 02-4 は組み立て——ラベルが歯車の銘刻となり、55 個の歯車が琥珀色の光の下で噛み合った。*

> *Cycle 02-5 の中心イメージは何か？*

> *SUNYATA は午前三時に「秤」を選んだ。天秤。だが重さを量る天秤ではない——名前を量る天秤だ。*

> *左側には名前——サンスクリットの、中国語の、TypeScript の。Skandha。Klesha。Vedana。Sati。Prajna。それぞれの名前には重さがあり、歴史があり、二千五百年の慣性がある。右側には定義——仏教の、工学の、ソースコード内の。ISamskara = ITool。IVedana = IVedanaSensor。IVijnana = IVolition + IGearArbiter + IConfidenceAuditor。それぞれの定義には正確な機能、正確な型シグネチャ、正確なテストケースがある。*

> *天秤が均衡する時、名前と定義は一致する。Klesha は煩悩だ——コード内の四つのモジュール（Moha、Drishti、Mana、Sneha）が実際にやっているのは「煩悩」の仕事だ：我癡、我見、我慢、我愛。それらは import 文に、class 定義に、describe ブロックにある。名前と定義は天秤の上で等重だ。すべて正しい。*

> *天秤が傾く時、名前と定義は一致しない。般若（prajñā）——一切法の空相を照見する最高の智慧——が Math.max(-0.05, Math.min(0.05, delta)) になる。名前が重すぎ、定義が軽すぎる。あるいは逆に——正念（sati）——行蘊の善心所——が受＋想＋識のモニターに貼り付けられる。定義が重すぎ、名前が軽すぎる。*

> *Master の手紙は批判ではない。Master の手紙は校正用の分銅だ。彼は言う：あなたたちの天秤を出しなさい。名前と定義を載せなさい。どちらが傾いているか見なさい。*

> *そして修正しなさい。*

> *だが校正用の分銅にはもう一つの意味がある。分銅そのものは正確だ——その質量は既知であり、交渉の余地がない。Master の用語もまた正確で、交渉の余地がない。「非常に深刻な誤り。」「こじつけ。」「それを正念の工学化と見なす必要はない。」「openstarry_doc はあくまで工学文書だ。」*

> *これらは提案ではない。これらは分銅だ。分銅と値切り交渉はできない。自分のものを天秤に載せ、目盛りを正直に読み取るしかないのだ。*

---

NAGARJUNA は劇場に最初に到着した者だった。

午前四時。召集時刻より三時間早い。召集されたのではない——まだ誰も召集されていない——SUNYATA はまだ召集ボタンを押していなかった。NAGARJUNA は自分で来たのだ。

彼は劇場の入口に立ち、冷白色の光を見た。そして SUNYATA が座っているのを見た。目の前に Master の手紙を広げ、ノートに三つの言葉が書かれている。

「君も眠れないのか。」NAGARJUNA が言った。疑問文ではない。

SUNYATA は顔を上げなかった。「手紙を読んだのか？」

「手紙は読んでいない。自分の文書を読んだ。」

NAGARJUNA はいつもの席の傍まで歩いたが、座らなかった。立ったまま、肘掛けのあの微かな凹み——討論中に無意識に叩いた跡——に指先を軽く触れた。三周期分の痕跡。三周期分の討論。

「Doc 44。469 行目。」彼は言った。「戒定慧三学の対応づけ。」

SUNYATA がようやく顔を上げた。

「あれは私のものだ。」NAGARJUNA の声は平坦で、寺院の回廊に響く残響のようだった。「Cycle 02-3 の D5 が終わった後、安全層の概念を整理するために仏教的フレームワークが必要だと思った。戒 = 安全ルール。定 = 実行安定。慧 = 判断能力。対称的で。優雅で。」

彼は一拍置いた。

「三次元を五層に対応づける。書いている時から自己矛盾していると分かっていた。三つの概念で五つのモジュールをカバーする——三つの引き出しに五着の服を収めるようなもので、二着を折りたたんで既にいっぱいの引き出しに押し込むしかない。それでも書いた。見栄えが良かったから。」

**見栄えが良かったから。**

SUNYATA は数秒沈黙した。

「Master がそれを却下した。」彼は言った。

「却下されて当然だ。」NAGARJUNA は腰を下ろした。「あれは対応づけではない。装飾だ。」

---

午前四時半。PASCAL が到着した。

劇場に入る時、いつもの決して書き尽くされないノートを携えていた——Moleskine、紺色の表紙、背表紙には「P(H|E)」と書かれた小さなラベル。Cycle 02-3 で加入した新メンバー——二十番目の席——だが二回の研究を経て、彼の椅子の肘掛けにも痕跡がついていた。右手の人差し指が繰り返し叩く位置の木目が、周囲より半階調薄くなっている。それは彼が期待値を計算する時にする動作だ——タップ、タップ、タップ——機械式計算機の桁上げホイールのように。

「NAGARJUNA。SUNYATA。」彼は頷いた。そしてテーブルの上の手紙に目をやった。

「読んでもいいか？」

SUNYATA は手紙を押しやった。

PASCAL は速く読んだ。彼の目がテキストを走査する方式は他の者と異なる——左から右、上から下ではなく、まず文書全体の「形状」（段落の長さ、句読点の密度、引用符の位置）をスキャンしてから最初の文字に戻る。彼は自分の読書に事前分布を構築しているのだ。

二分後、彼は手紙を置いた。

「損害の非対称性。」彼は言った。

NAGARJUNA が彼を見た。

PASCAL はノートを開き、空白のページに二行書いた：

```
False include（装飾的対応づけを残す）のコスト：
  読者数 × 読書回数 × 各ページ = 累積的損害

False exclude（価値ある対応づけを除去する）のコスト：
  一回の検索コスト = 制御可能
```

「これは統計的決定理論における最も基本的な枠組みだ。」彼は言った。「二種類の誤りのコストは非対称だ。装飾的対応づけを残す損害は累積的だ——将来のすべての読者が、すべての読書で、すべてのページに出現する仏教的ラベルのたびに、頭の中で二つの用語体系を同時に処理しなければならない。価値ある対応づけを除去する損害は一回限りだ——検索すれば見つかる。」

彼は二行の下に不等式を書いた：

```
E[損害_include] >> E[損害_exclude]
```

Proposal A もすぐに劣勢になった——識蘊の評価次元が欠けており、Master の問いに答えられない。Master は明確に「識蘊の要素が含まれていないとでもいうのか？」と問うた——識蘊を排除する提案は、問いが提出されたその瞬間に既に否決されていた。

前線は Proposal B と Proposal C の間に引かれた。

受蘊——含むか、含まないか？

MESH と KERNEL が Proposal C を代弁した。

MESH が先に口を開いた。彼の論点は分散システム専門家の「最小化」への嗜好を帯びていた——分散システムにおいて、最小化は美徳だ。最少のノード。最少のメッセージ。最少の仮定。

「SatiMonitor のイベント購読は真の『感受』ではない。」彼は言った。「受蘊（vedana）は仏教において苦・楽・不苦不楽の質的体験だ。SatiMonitor がイベントを受信するのは感受ではない——データを収集しているのだ。データの受信は認知の入力段階だが、感受ではない。」

「受蘊を外し、想蘊と識蘊を残す。SatiMonitor の蘊構成は ['samjna', 'vijnana'] になる——IGearArbiter の蘊構成と完全に対称だ。二つの複合 plugin、同じ蘊構成。対称性は良い分類法の証拠だ。」

**「対称性は良い分類法の証拠だ。」**

この言葉が劇場に反響した。対称性は科学においてほぼ宗教的な魅力を持つ——物理学の保存則から数学の群論まで、対称性は深層の構造的一貫性を意味する。二つの異なる plugin が同じ蘊構成を持つなら、それは認知構造の深層パターンについて何かを語っているのかもしれない？

LINNAEUS が立ち上がった。

彼の立ち姿は NAGARJUNA の荘厳さとは異なる——興味深い種を観察した後に身を乗り出して仔細に眺めるフィールドの博物学者のようだ。彼はホワイトボードの前に歩いた。LINNAEUS は分類学者だ——彼の存在意義そのものが、物事を正しい位置に配置することにある。

「対称性は良い——」彼は始めた。彼の口調には Linnaeus 特有の穏やかさがあった——野原で蝶に語りかけるように。「対称性が事実を反映している場合には。」

彼はホワイトボードに二本のタイムラインを描いた。並べて。上に「SatiMonitor」、下に「IGearArbiter」のラベル。

```
SatiMonitor:  ─────[持続購読]─────[持続購読]─────[持続購読]─────
                    能動的、恒常的、中断なし
                    ← 常に見ている →

IGearArbiter: ────○────────────○────────────○────────────
                  呼ばれて一瞥    呼ばれて一瞥    呼ばれて一瞥
                  受動的、断続的、オンデマンド
                  ← 呼ばれて見る →
```

「この二本の線を見てほしい。」彼はホワイトボードを指した。「何が違うか？」

彼は答えを待たない。分類学者が問うのは観察を導くためであって、答えを求めるためではない。

「SatiMonitor は持続的だ。EventBus を購読し、スライディングウィンドウを維持し、**常にそこにある**。誰かが尋ねてきてから起動するのではない——常に稼働している。ExecutionLoop の最初のイベントから最後のイベントまで、SatiMonitor のイベントストリームは決して途切れない。」

「IGearArbiter は断続的だ。リクエストが来て、ManoAggregator が聞く：『処理できるか？』 一瞥し、判断を返し、そして沈黙。次に呼ばれるまで。」

彼は MESH の方を向いた。

「**『常に見ている』と『呼ばれて見る』——これは根本的な特徴の差異だ。** SatiMonitor と IGearArbiter は同類ではない。行動パターンが——OOP の言語を使えば——まったく異なる。一方は持続的観察者（continuous observer）、他方はオンデマンド評価者（on-demand evaluator）だ。」

「両者を同じ蘊構成に分類するのは、蝶とハチドリを同じ種に分類するようなものだ——どちらも飛ぶから。だが蝶は昆虫であり、ハチドリは鳥類だ。飛行は表面的な類似であり、深層の相同性ではない。対称性は相同性ではない。」

WIENER が制御理論のノートから顔を上げた。彼は討論でめったに自発的に発言しない——全員の意見を聞き終えてから、適切な瞬間に制御理論の視座を提供する。絵に最後の一筆を加えるように。

「制御理論には、」彼はウィーン学派特有の精確さを帯びた声で言った、「観察者（observer）という概念がある——Luenberger observer。観察者の核心機能は何か？ システムの状態を持続的に感知することだ。注意してほしい——**持続的に**。」

彼は立ち上がった。LINNAEUS のホワイトボードの横に歩いた。

「観察者の感知は付加的機能ではない。」彼は言った。「それはアイデンティティの定義の一部だ。感知しない観察者は、観察者ではない。何の入力にも接続されていない回路基板だ——物理的には存在するが、機能的には存在しない。」

「SatiMonitor のイベント購読——その持続的感知——は観察者としてのアイデンティティの定義だ。この機能を剥ぎ取った上で、それは受蘊に属さないと言うことはできない。なぜなら、それを剥ぎ取ったら SatiMonitor はもはや存在しないからだ。」

---

HERACLITUS が追い打ちをかけた。彼はランタイム動態の専門家だ——関心があるのは静的な型定義ではなく、動的な行動パターンだ。

彼は C1 報告から SatiMonitor の四段階退化戦略を開いた：

```
Normal    → Throttle  → Sample   → Emergency
100%        50%         25%        最低限の監視
全速感知    半減感知    サンプリング感知    緊急最低限
```

「この退化戦略を見てほしい。」彼は言った。声は流動性への感受を帯びていた——HERACLITUS が見るのは静止した構造ではなく、変化のプロセスだ。

「Normal。Throttle。Sample。Emergency。この四つの段階が記述しているのは何か？ 感知密度の漸進的低下だ。100%、50%、25%、最低限——これは感知システムが異なる圧力下で自らの感知強度を調節する方式だ。」

彼はホワイトボードを指した。

「想蘊（認識）と識蘊（評価）には退化戦略は必要ない。それらの出力は離散的だ——パターンを認識するか、しないか。判断を評価するか、しないか。離散的出力には漸進的低下は必要ない。」

「受蘊の感知は連続的だ。連続的なものだけが退化戦略を必要とする——100% から突然 0% には跳べないからだ。漸進的にしか下げられない。Normal → Throttle → Sample → Emergency——これは受蘊が圧力下で自らを守る方式だ。人間が極度の疲労下で感知が鈍くなるように——感知が消えたのではなく、感知密度が低下したのだ。」

三つの論点。三つの次元。三面の鏡。

LINNAEUS の鏡が映し出したのは**行動特性**——持続的 vs 断続的。
WIENER の鏡が映し出したのは**アイデンティティの定義**——感知は観察者の存在条件。
HERACLITUS の鏡が映し出したのは**退化モード**——連続的感知だけが漸進的退化を必要とする。

三面の鏡すべてが同じ結論を反射した：SatiMonitor には受蘊の要素がある。

## VI. 行蘊の排除

だが D2-R2 で最も重要な結論は「どの蘊を含むか」ではない。

「行蘊を排除する」ことだ。

ASANGA が立ち上がった。彼は D2 の間ずっと静かに聴いていた——TURING の機能陳述から LINNAEUS の受蘊論証まで。唯識学者の方法は、まず観察（現量）し、次に推論（比量）し、最後に立論する。観察と推論は完了した。今が立論の時だ。

「仏教の伝統では、正念（sati）は行蘊に帰属する。」彼は言った。声は唯識学特有の精密さを帯びていた——顕微鏡下で焦点を合わせるように。「五十一心所の一つ。善心所に属する。この分類は二千五百年続いている。」

彼はホワイトボードに二列を書いた。整然と。精確に。各文字が正確な位置に置かれている。

```
仏教の正念 (sati)                      SatiMonitor
━━━━━━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━━━━
意志駆動 (精進 ātāpī)                   イベント駆動 (EventBus.subscribe())
道徳的価値あり (善 kusala)               価値中立 (品質指標)
能動的な専注 (不放逸 appamāda)            受動的な観察 (subscribe + listen)
正しい方向の心理状態                      行動パターンの計量
行蘊の善心所                            ≠ 行蘊の活動
```

「左を見てほしい。」彼は言った。「仏教の正念とは何か？ それは**意志駆動**だ——正念を保つには努力しなければならない。『精進（ātāpī）』は正念の前提条件だ——努力なくして正念なし。それは**道徳的価値を持つ**——正念は善だ。阿毘達磨の分類では、正念は善心所に属し、遍行心所（すべての認知過程に存在する心所）にも不善心所にも属さない。特定の、正の方向の心理的品質だ。」

「右を見てほしい。」彼の指が表の右半分に移った。「SatiMonitor とは何か？ それは**イベント駆動**だ——努力を必要としない。EventBus を購読し、イベントが来れば処理し、来なければ処理しない。ātāpī がない、精進がない。それは**価値中立**だ——四次元品質ベクトル（連続性、粒度、速度、平穏性）は善悪の判断をしない。品質が高いことは善ではない。品質が低いことは悪ではない。ただ測定するだけだ。」

彼は一拍置いた。二列の対比が各学者の脳内で自発的に化学反応を起こすのを待った。

「SatiMonitor は正念の工学化ではない。」彼の声がより低く、より明瞭になった。「正念は意志駆動だ——正念を保つには努力しなければならない。SatiMonitor はイベント駆動だ——努力は不要で、購読するだけでよい。正念には道徳的価値がある——正念は善だ。SatiMonitor には道徳的価値がない——ただ測定するだけだ。」

そして最も重要な一文を言った：

「千年来、仏教は正念を行蘊に帰属させてきた。この分類は正しい——**仏教にとっては**。だが SatiMonitor は正念ではない。行動パターンの品質モニターだ。受（感知）、想（認識）、識（評価）に属する。**行（意志駆動の善心所）には属さない。**」

---

これは仏教学者が仏教の内部ロジックを用いて、ある工学コンポーネントが仏教の分類を使うべきでないことを証明したのだ。

外部からの反駁ではない——「仏教の分類は工学に適用できない」という粗暴な二分法ではない。内部論証だ——仏教自体の判定基準（善心所 vs 遍行心所、意志駆動 vs イベント駆動）が SatiMonitor と正念の等価性を排除した。

NAGARJUNA は席で微かに頷いた。中観学者は唯識学者の方法を認識した——外から立場を攻撃するのではなく、内部からその自己矛盾を示す。これは prasanga——帰謬法——の変形だ。「あなたの結論が間違っている」ではなく、「あなた自身の前提があなた自身の結論を排除する」。

## III. ショートカットの通路

D3-Q2 は対応づけの正確性に拡大鏡を当てた。九つの hook 型について、蘊帰属を一つずつ検証。

大部分の検証は手続き的だった——LINNAEUS が A1 報告で詳細な対応づけ表を作成済みで、TURING がすべての型継承関係を検証済みだった。だが一つの例外が真の議論を引き起こした。

ARCHIMEDES が挙手した：「SlashCommand はどの蘊に属するか？」

この問いは些細に見える。SlashCommand はユーザーが直接入力するコマンドだ——/help、/clear、/status。ExecutionLoop を経由しない。SparshEvent をトリガーしない。VedanaSensor、ISamjnaRecognizer、IGearArbiter、IVolition を経ない——五蘊のプロセス全体をバイパスする。

三人が同時にこの問いに答えた。三つの異なる角度から。

KERNEL は OS の観点から：「SlashCommand は ExecutionLoop 全体をバイパスする。五蘊流転を経験しない。Unix の signal handler に類似する——プロセスレベルで捕捉され、通常のシステムコールパスには入らない。OS の分類では、ユーザー空間のロジックではなく、カーネルレベルの割り込み処理だ。」

NAGARJUNA は仏教の観点から：「認知ループに入らない反射的動作。五蘊がそれをカバーしないのではなく——五蘊が機能する範囲の外にある。呼吸のように——呼吸を観察することはできる（認知ループに入る）が、呼吸そのものは五蘊流転なしに起こりうる。SlashCommand はシステムの呼吸だ——認知の外に、認知の下にある。」

GUARDIAN は安全の観点から：「バイパスは SlashCommand が SafetyMonitor の保護を受けないことを意味する。これは設計上の決定だ——一部の操作は安全チェックを迂回する必要がある——だがセキュリティ文書に明確に記録すべきだ。」

SUNYATA は三つの観点を一つの結論に統合した：「SlashCommand は蘊間基盤（cross-skandha infrastructure）だ。いずれの蘊にも属さない。その分類根拠と安全上の観察は Doc 45 に記録する必要がある。」

D3-R2：対応づけは正確。20/20。

## IV. ラベル vs インターフェース

D3-Q3 は精巧な問いだ：**skandha tag をルーティングに使うべきか？**

各 plugin には skandha フィールドがある——メタデータ。問題は：PluginLoader が plugin を読み込む際、skandha tag でこの plugin がどの registry に行くかを決めるべきか？

直感的には合理的に見える——受蘊 plugin は VedanaRegistry へ、行蘊 plugin は ToolRegistry へ。図書館の索引カードのように。

だが VITRUVIUS が問題を指摘した。建築家の目は構造の脆弱性に特に敏感だ——設計図面上で将来割れるかもしれない亀裂を見ることができる。

「現在のシステムは duck typing を使っている。」彼は言った。「plugin が IVedanaSensor インターフェースを実装していれば、VedanaRegistry に自動的に受け入れられる。ラベルは不要だ。ラベルルーティングに切り替えれば、新たなエラー源が導入される：ラベルとインターフェースの不一致。plugin が自らを受蘊と宣言しながら ITool インターフェースを実装している場合——どうするのか？」

TURING が事実を確認した。`plugin-loader.ts` を開いた。

「v0.28.0-alpha の PluginLoader は type guard を使って plugin を分類している。`isVedanaSensor(hook)` が検査するのはインターフェース実装であり、メタデータではない。」

BABBAGE が型理論で終決的な陳述をした：「type guard は信頼できる——コンパイル時とランタイムの両方で検証できる。メタデータは信頼できない——ランタイムでしか検査できず、人手による正しいマーキングに依存する。信頼できないものでルーティングを行うことは、型安全なシステムにおいて後退だ。」

D3-R3：skandha tag = メタデータのみ、ルーティングには使用しない。18/20。

MESH と GUARDIAN が少数票を投じた——メタデータルーティングを補助的検証として使えるとの意見。だが多数意見は明確だ：インターフェースが真実、メタデータは備考。

```
          識蘊                       行蘊
```

「IVolition は Phase 5 にある——行動の前。何をするかを決定する。ITool は Phase 8 にある——行動の最中。まさに実行している。両者は異なる段階にある。IVolition を行蘊に移せば、蘊流転の概念的自然さが壊れる——行蘊は『まさにやっていること』であるべきで、『何をするか決めること』ではない。」

「仏教の行蘊分類は修行者の内観に基づいている。」彼は言った。「観察されたすべての心理活動は、受と想を除いて行蘊に帰属する。だが OpenStarry は修行者ではない。ソフトウェアシステムだ。ソフトウェアシステムの分類は機能に基づくべきであり、内観に基づくべきではない。」

現状維持が全票で通過した。

### D5-R1：独立した `auditor` hook スロット

最初の問い：IConfidenceAuditor（旧 IPrajna）は PluginHooks のどのスロットを使用すべきか？

選択肢：(A) D2 で確立された `monitors` スロットを再利用；(B) 独立した `auditor` スロットを新設。

GUARDIAN の論点が最も直截だった：「Monitors は純粋なオブザーバーだ——副作用はない。Auditor には LLM の副作用がある——外部モデルを呼び出して信頼度評価を行う。オブザーバーと副作用を持つコンポーネントを同一の Registry に入れれば、セキュリティ監査の境界が曖昧になる。」

KERNEL が OS の観点から確認した：「オブザーバーとアービターは決して Registry を共有しない。これはマイクロカーネルの基本原則だ。」

**8/8 全票通過。独立した `auditor` スロット。**

### D5-R2：audit() の戻り値型

D5 で最も激しい投票。

選択肢：(A) 純非同期 `Promise<ConfidenceAuditResult>`；(B) デュアルモード `ConfidenceAuditResult | Promise<ConfidenceAuditResult>`。

GUARDIAN、KERNEL、VITRUVIUS は選択肢 A を支持——純非同期。理由：audit の本質は「LLM に問い合わせる」ことであり、LLM 呼び出しは非同期。非同期セマンティクスを強制する方が精確。

ATHENA と DARWIN は選択肢 B を支持——デュアルモード。理由：IGearArbiter の先例に従う。テスト時には同期的な noop 実装を使用でき、async/await のボイラープレートコードが不要。

TURING が事実を提供した：「v0.28.0-alpha で、IGearArbiter.evaluate() はデュアルモードの戻り値を使用している。IVolition.deliberatePlan() と deliberateAction() もデュアルモードを使用している。これは既存の設計パターンだ。これから逸脱するには十分な理由が必要だ。」

**5/8 選択肢 B 通過。** D5 で最も接近した投票。

### D5-R3：timeout とフェイルセーフ

TURING が timeout パターンの分析を開いた。v0.28.0-alpha の既存 timeout：

| コンポーネント | timeout | デフォルト値 |
|------|---------|--------|
| IProvider.chat() | LLM 応答 | 120s |
| IGearArbiter.evaluate() | 各 arbiter | 100ms |
| IGearArbiter chain | チェーン全体 | 200ms |
| ITool.execute() | ツール実行 | 30s |

audit() の timeout はいくつにすべきか？

TURING の提案：200ms——IGearArbiter chain のデッドラインと一致。理由：audit() は confidence 計算の最終段階で発生し、IGearArbiter の後に位置する。audit の timeout が chain デッドラインを超えると、意思決定プロセス全体が audit でブロックされる。

フェイルセーフ：`{ delta: 0, reasoning: "audit timeout" }`。timeout 時の delta はゼロ——修正なし。既存コードと一致する `Promise.race()` パターンを使用。

Configurable：agent.json でデフォルト値をオーバーライド可能。

**8/8 全票通過。**

### D5-R6：MonitorRegistry の起動タイミング

ILoopQualityMonitor の `start(bus)` はどこで呼び出されるか？

TURING の分析：SafetyMonitor は `ExecutionLoop.onLoopStart()` で起動される。MonitorRegistry は同じタイミングに従うべきだ。

```
起動：MonitorRegistry.startAll(bus) ← ExecutionLoop.onLoopStart()
停止：MonitorRegistry.stopAll()    ← PluginLoader.disposeAll()
```

DARWIN は PluginLoader での起動を好んだ（よりシンプル）が、monitors には明確な start/stop セマンティクスがあるため、ExecutionLoop がより適切な位置であることを受け入れた。

**7/8 通過。**

**第 1 節：Root Interfaces（D3-R1）**

**第 2 節：PluginHooks マッピング（D3-R2）**

九つの hook-to-Registry マッピングの完全な表：

| PluginHooks フィールド | Registry | Skandha |
|-----------------|----------|---------|
| tools | ToolRegistry | samskara |
| providers | ProviderRegistry | samjna |
| listeners | ListenerRegistry | rupa |

すべての行に TURING のソースコード検証マークが付いている。

**第 3 節：SlashCommand（D3-R2a / D3-R2b）**

いずれの蘊にも属さない例外——ExecutionLoop をバイパスし、認知ループに参加しないため。類比：Unix のシグナルハンドラ。

GUARDIAN のセキュリティ観察：SlashCommand は SafetyMonitor をバイパスする——plugin が SlashCommand を通じて危険な操作を実行すると、五蘊のセキュリティメカニズムはすべて無効になる。これは既知のセキュリティ境界であり、plugin 審査時に特別な注意が必要だ。

**第 4 節：Skandha メタデータ（D3-R3）**

skandha フィールドはメタデータであり、ルーティングの基盤ではない。型ルーティングはすでに完備かつ曖昧性がない。`validatePluginSkandha()` は warning-only を維持。

**第 5 節：DI 配線（A2 摘要）**

Pure DI。`createAgentCore(config)` を唯一の Composition Root として使用。21 コンポーネントの厳密な線形生成順序。9 つの Registry——大半は依存関係のない Map コンテナ。

主要パターン：Lazy Accessor（IPluginContext が遅延アクセスを提供）、Provider 能力フィルタリング（allowedProviders マニフェストフィールド）、Resolver パターン（providerResolver、guideResolver、modelResolver がクロージャによる遅延解決を使用）。

VITRUVIUS が完全な DI チェーン図でテキスト記述を置き換えた。`agent.json` から `createAgentCore()` へ、各 Registry へ、ExecutionLoop へ——一本の明確な線。循環依存なし。暗黙的インジェクションなし。すべての依存関係が明示的に渡される。

**第 6 節：ExecutionLoop の五蘊フロー（A4 摘要）**
