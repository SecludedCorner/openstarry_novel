## I. Master の棘

D2 の開幕は SUNYATA の問いかけではない。Master の声だ。

Master 本人が会場にいるわけではない——Master は決して討論に姿を現さない。六回の研究を通じて、Master の存在様式は常に間接的だった：手紙、返答、確認記録。彼は見えない重力源のように、あらゆる天体の軌道に影響を与えながら、決して観測者の視野には現れない。

彼の言葉がスクリーンに投影された。白地に黒字。冷白色の照明の下で、その文字は石に刻まれたように見えた。

> 「今まさに何が起きているかを覚知すること。このレイヤーはルールも見ないし、リスク分類も見ない。純粋に行動パターンを観察する。それを正念の工学化と見なす必要はない。しかもこの plugin には識蘊の要素が含まれていないとでもいうのか？」

SUNYATA は最後の一文に下線を引いた。赤いペンで。強く。赤い線が紙面を貫通し、次のページに薄い圧痕を残した。

**「識蘊の要素が含まれていないとでもいうのか？」**

彼は横に疑問符を書いた。普通の疑問符ではない——六回の研究で三度しか使ったことのないあの種の疑問符だ。大きな。赤い。疑問符のフックが行全体を占めるほどの。

これは反語ではない。Master は決して反語を使わない。SUNYATA は五通の手紙でこれを学んでいた——Master のすべての疑問符は具体的な、明確な方向性を持つ問いだ。彼は既に答えを持っているが、研究チームに自ら見つけることを求めている。

Master の方向性は明確だ：SatiMonitor は行蘊だけの機能ではない。識蘊の要素がある。研究チームの任務は正確な蘊構成を見出すことだ。

D2 の議題：**この「行動パターンを観察する」plugin は何と呼ぶべきか、どの蘊で構成されるか？**

## II. 機能の解剖

D2-Q1（命名）の前に、SUNYATA は異例の決定をした：TURING に先に機能ブリーフィングをさせた。

討論ではなく——事実陳述だ。

TURING が立ち上がった。NAGARJUNA とは異なる立ち方で——荘厳さも劇的効果もない。ただ立ち上がっただけだ。機械が起動するように。

「C1 研究報告。SatiMonitor の工学的機能分析。」彼は言った。そしてスライドを一枚開いた。スライドにはグラフも数式もなく、四行の文字と一行の「やらないこと」だけがあった：

```
機能 1：イベント購読 — EventBus.subscribe() を通じてシステムイベントを持続的に受信
機能 2：スライディングウィンドウ — 直近 N 件のイベントの時間窓を維持
機能 3：パターン認識 — 窓内で行動パターン（突変、傾向、周期）を識別
機能 4：品質指標 — 四次元品質ベクトル（連続性、粒度、速度、平穏性）を出力

やらないこと：いかなるアクションも実行しない。いかなる状態も変更しない。いかなる指令も発しない。
```

「最後の行が最も重要だ。」TURING は言った。「SatiMonitor は何もしない。ただ観るだけだ。」

GUARDIAN は安全メモに「やらないこと」の二文字を記録した。そして横に TURING なら書かないであろう比喩を書いた：「APM Agent + Behavioral Analytics + QoS Monitor + SPC Anomaly Detector の混合体。」

TURING は GUARDIAN のメモをちらりと見た。その比喻が悪いとは言わなかった——ただ微かに眉をひそめただけだ。精度を校正中の工学者が近似解を見た時のように。

## III. 命名の収束

D2-Q1。文書タイトル。

元の名前には Sati のマークが付いている——服にブランドロゴが刺繍されているように。問題は：このロゴはまだ適切なのか？

TURING は既に C1 報告で純粋な工学用語ですべての機能を記述していた。四つの機能、どれも仏教概念で説明する必要がない。イベント購読は EventBus の標準パターンだ。スライディングウィンドウは統計学の基本ツールだ。パターン認識はデータ分析の核心的能力だ。品質指標は産業品質管理の日常業務だ。

GUARDIAN が微妙な要求を提起した。彼の声は安全文書特有の厳密さを帯びていた——すべての提案が複数の審査を経たセキュリティアドバイザリのように。

「安全文書ではフルネームが必要だ。」彼は言った。「安全コンテキストでは完全で明確な名称を使うことを提案する。Sati をコード識別子として残すのであれば、安全文書ではフルネームを注記すべきだ——例えば SatiMonitor (Cognitive Loop Quality Monitor) のように。」

チームはすぐに収束した。十分以内に。

文書タイトルは「認知ループ品質監視アーキテクチャ (Cognitive Loop Quality Monitoring Architecture)」に変更。Sati はコード識別子として保持——SatiMonitor はソースコードに存在し、改名にはマイグレーションコストがかかる——だが文書の定義的名称としては使用しない。

D2-R1：改名。19/20。

GUARDIAN は B に投票した——彼は完全に BehaviorQualityMonitor への改名を好んだ。だが安全文書でのフルネーム要件は付帯条件として盛り込まれた。一票の少数意見。クリーンだ。

## IV. 受蘊の厚み

D2-Q2。

これが D2 の核心だ。そして Cycle 02-5 の最初の三回の討論で最も激しい衝突でもある。

**「SatiMonitor は五蘊のどのカテゴリで構成されるか？」**

C2 の研究報告——ASANGA と LINNAEUS の共同研究——は四つの提案を出していた。各提案の背後にはロジックの体系があり、各体系はそれ自体で成立しうる。少なくとも提出された時点では。

| 提案 | 蘊構成 | 根拠 |
|------|--------|------|
| A | 受 + 想 | 感知 + 認識、最小構成 |
| B | 受 + 想 + 識 | 感知 + 認識 + 評価判断 |
| C | 想 + 識 | 認識 + 評価、受蘊は薄すぎる |
| D | 純識 | すべての認知活動を識蘊に帰属 |

Proposal D は R2 の交差査読で既に淘汰されていた——「純識」は分類を放棄し、すべてを一つの桶に投げ込むことに等しい。BABBAGE は交差査読で簡潔な否決を書いた：「Proposal D は SatiMonitor の蘊帰属を自明解に退化させる。自明解は数学では有効だが、分類学では無用だ。」

Proposal A もすぐに劣勢になった——識蘊の評価次元が欠けており、Master の問いに答えられない。Master は明確に「識蘊の要素が含まれていないとでもいうのか？」と問うた——識蘊を排除する提案は、問いが提出されたその瞬間に既に否決されていた。

前線は Proposal B と Proposal C の間に引かれた。

受蘊——含むか、含まないか？

---

MESH と KERNEL が Proposal C を代弁した。

MESH が先に口を開いた。彼の論点は分散システム専門家の「最小化」への嗜好を帯びていた——分散システムにおいて、最小化は美徳だ。最少のノード。最少のメッセージ。最少の仮定。

「SatiMonitor のイベント購読は真の『感受』ではない。」彼は言った。「受蘊（vedana）は仏教において苦・楽・不苦不楽の質的体験だ。SatiMonitor がイベントを受信するのは感受ではない——データを収集しているのだ。データの受信は認知の入力段階だが、感受ではない。」

KERNEL が OS のアナロジーで補足した：「システムコールが packet を受信するのは感受ではない——I/O の受信だ。感受とは I/O に対する質的評価だ。SatiMonitor はイベントに質的評価をしているか？ 苦楽の判断をしているか？ していない。ただデータを収集しているだけだ。」

そして MESH は最も強い論点を投じた——対称性。

「受蘊を外し、想蘊と識蘊を残す。SatiMonitor の蘊構成は ['samjna', 'vijnana'] になる——IGearArbiter の蘊構成と完全に対称だ。二つの複合 plugin、同じ蘊構成。対称性は良い分類法の証拠だ。」

**「対称性は良い分類法の証拠だ。」**

この言葉が劇場に反響した。対称性は科学においてほぼ宗教的な魅力を持つ——物理学の保存則から数学の群論まで、対称性は深層の構造的一貫性を意味する。二つの異なる plugin が同じ蘊構成を持つなら、それは認知構造の深層パターンについて何かを語っているのかもしれない？

LINNAEUS が立ち上がった。

彼の立ち姿は NAGARJUNA の荘厳さとは異なる——興味深い種を観察した後に身を乗り出して仔細に眺めるフィールドの博物学者のようだ。彼はホワイトボードの前に歩いた。LINNAEUS は分類学者だ——彼の存在意義そのものが、物事を正しい位置に配置することにある。

「対称性は良い——」彼は始めた。彼の口調には Linnaeus 特有の穏やかさがあった——野原で蝶に語りかけるように。「対称性が事実を反映している場合には。」

彼はホワイトボードに二本のタイムラインを描いた。並べて。上に「SatiMonitor」、下に「IGearArbiter」のラベル。

```
SatiMonitor:  ─────[持続購読]─────[持続購読]─────[持続購読]─────
                    能動的、恒常的、中断なし
                    ← 常に見ている →

IGearArbiter: ────○────────────○────────────○────────────
                  呼ばれて一瞥    呼ばれて一瞥    呼ばれて一瞥
                  受動的、断続的、オンデマンド
                  ← 呼ばれて見る →
```

「この二本の線を見てほしい。」彼はホワイトボードを指した。「何が違うか？」

彼は答えを待たない。分類学者が問うのは観察を導くためであって、答えを求めるためではない。

「SatiMonitor は持続的だ。EventBus を購読し、スライディングウィンドウを維持し、**常にそこにある**。誰かが尋ねてきてから起動するのではない——常に稼働している。ExecutionLoop の最初のイベントから最後のイベントまで、SatiMonitor のイベントストリームは決して途切れない。」

「IGearArbiter は断続的だ。リクエストが来て、ManoAggregator が聞く：『処理できるか？』 一瞥し、判断を返し、そして沈黙。次に呼ばれるまで。」

彼は MESH の方を向いた。

「**『常に見ている』と『呼ばれて見る』——これは根本的な特徴の差異だ。** SatiMonitor と IGearArbiter は同類ではない。行動パターンが——OOP の言語を使えば——まったく異なる。一方は持続的観察者（continuous observer）、他方はオンデマンド評価者（on-demand evaluator）だ。」

「両者を同じ蘊構成に分類するのは、蝶とハチドリを同じ種に分類するようなものだ——どちらも飛ぶから。だが蝶は昆虫であり、ハチドリは鳥類だ。飛行は表面的な類似であり、深層の相同性ではない。対称性は相同性ではない。」

---

WIENER が制御理論のノートから顔を上げた。彼は討論でめったに自発的に発言しない——全員の意見を聞き終えてから、適切な瞬間に制御理論の視座を提供する。絵に最後の一筆を加えるように。

「制御理論には、」彼はウィーン学派特有の精確さを帯びた声で言った、「観察者（observer）という概念がある——Luenberger observer。観察者の核心機能は何か？ システムの状態を持続的に感知することだ。注意してほしい——**持続的に**。」

彼は立ち上がった。LINNAEUS のホワイトボードの横に歩いた。

「観察者の感知は付加的機能ではない。」彼は言った。「それはアイデンティティの定義の一部だ。感知しない観察者は、観察者ではない。何の入力にも接続されていない回路基板だ——物理的には存在するが、機能的には存在しない。」

「SatiMonitor のイベント購読——その持続的感知——は観察者としてのアイデンティティの定義だ。この機能を剥ぎ取った上で、それは受蘊に属さないと言うことはできない。なぜなら、それを剥ぎ取ったら SatiMonitor はもはや存在しないからだ。」

---

HERACLITUS が追い打ちをかけた。彼はランタイム動態の専門家だ——関心があるのは静的な型定義ではなく、動的な行動パターンだ。

彼は C1 報告から SatiMonitor の四段階退化戦略を開いた：

```
Normal    → Throttle  → Sample   → Emergency
100%        50%         25%        最低限の監視
全速感知    半減感知    サンプリング感知    緊急最低限
```

「この退化戦略を見てほしい。」彼は言った。声は流動性への感受を帯びていた——HERACLITUS が見るのは静止した構造ではなく、変化のプロセスだ。

「Normal。Throttle。Sample。Emergency。この四つの段階が記述しているのは何か？ 感知密度の漸進的低下だ。100%、50%、25%、最低限——これは感知システムが異なる圧力下で自らの感知強度を調節する方式だ。」

彼はホワイトボードを指した。

「想蘊（認識）と識蘊（評価）には退化戦略は必要ない。それらの出力は離散的だ——パターンを認識するか、しないか。判断を評価するか、しないか。離散的出力には漸進的低下は必要ない。」

「受蘊の感知は連続的だ。連続的なものだけが退化戦略を必要とする——100% から突然 0% には跳べないからだ。漸進的にしか下げられない。Normal → Throttle → Sample → Emergency——これは受蘊が圧力下で自らを守る方式だ。人間が極度の疲労下で感知が鈍くなるように——感知が消えたのではなく、感知密度が低下したのだ。」

三つの論点。三つの次元。三面の鏡。

LINNAEUS の鏡が映し出したのは**行動特性**——持続的 vs 断続的。
WIENER の鏡が映し出したのは**アイデンティティの定義**——感知は観察者の存在条件。
HERACLITUS の鏡が映し出したのは**退化モード**——連続的感知だけが漸進的退化を必要とする。

三面の鏡すべてが同じ結論を反射した：SatiMonitor には受蘊の要素がある。

## V. ゼロコストの転換

そして ARCHIMEDES が討論の流れを変える行動をとった。

哲学的議論には加わらなかった。仏教の経典も引用しなかった。PluginHooks の型定義を開いたのだ。

```typescript
interface PluginHooks {
  sensors?: IVedanaSensor[];         // 受蘊
  recognizers?: ISamjnaRecognizer[]; // 想蘊
  arbiters?: IGearArbiter[];         // 想蘊+識蘊
  tools?: ITool[];                   // 行蘊
  volition?: IVolition;              // 識蘊
  // ... monitors 未実装 ...
}
```

「三蘊か二蘊か——工学的差異はあるか？」彼は問うた。

自ら答えた。彼のいつもの方式で——論証ではなく、計算で。

「monitors は独立した配列スロットだ。SatiMonitor が `['samjna', 'vijnana']` とマークされようと `['vedana', 'samjna', 'vijnana']` とマークされようと、PluginHooks の実装は完全に同じだ。追加の hook はない。追加の配線はない。追加のテストはない。」

彼は全場を見渡した。

「**追加の工学コストはゼロ。**」

この一文が討論の水面に石のように落ちた。

三蘊と二蘊の工学コストが同じなら——追加の hook も、追加の配線も、追加のテストもない——この討論はもはや工学的トレードオフではない。純粋な分類問題になった。

純粋な分類問題においては、分類学者に圧倒的な発言権がある。

LINNAEUS の「持続的感知 vs 受動的受信」。
WIENER の「観察者のアイデンティティ定義」。
HERACLITUS の「退化戦略は連続的感知にのみ属する」。

三面の鏡。三つの次元。ゼロの工学コスト。

Proposal C はその唯一の工学的論拠を失った。

---

投票。

D2-R2：**Proposal B — [vedana, samjna, vijnana]。18/20。**

MESH と KERNEL が反対票を投じた。彼らの「対称性」論は方法論的に価値ある観察として記録された——だが事実は対称性に勝った。

18/20 は何を意味するか？ 20 人のチームで、18 票の賛成は 90% のコンセンサスだ。分野横断的な分類問題——哲学者、仏教学者、工学者が同時に合意しなければならない——において、90% は非常に高い。

SatiMonitor は OpenStarry 初の三蘊 plugin となった。

## VI. 行蘊の排除

だが D2-R2 で最も重要な結論は「どの蘊を含むか」ではない。

「行蘊を排除する」ことだ。

ASANGA が立ち上がった。彼は D2 の間ずっと静かに聴いていた——TURING の機能陳述から LINNAEUS の受蘊論証まで。唯識学者の方法は、まず観察（現量）し、次に推論（比量）し、最後に立論する。観察と推論は完了した。今が立論の時だ。

「仏教の伝統では、正念（sati）は行蘊に帰属する。」彼は言った。声は唯識学特有の精密さを帯びていた——顕微鏡下で焦点を合わせるように。「五十一心所の一つ。善心所に属する。この分類は二千五百年続いている。」

彼はホワイトボードに二列を書いた。整然と。精確に。各文字が正確な位置に置かれている。

```
仏教の正念 (sati)                      SatiMonitor
━━━━━━━━━━━━━━━━━━━━━━━━           ━━━━━━━━━━━━━━━━━━━━━━━━━━
意志駆動 (精進 ātāpī)                   イベント駆動 (EventBus.subscribe())
道徳的価値あり (善 kusala)               価値中立 (品質指標)
能動的な専注 (不放逸 appamāda)            受動的な観察 (subscribe + listen)
正しい方向の心理状態                      行動パターンの計量
行蘊の善心所                            ≠ 行蘊の活動
```

「左を見てほしい。」彼は言った。「仏教の正念とは何か？ それは**意志駆動**だ——正念を保つには努力しなければならない。『精進（ātāpī）』は正念の前提条件だ——努力なくして正念なし。それは**道徳的価値を持つ**——正念は善だ。阿毘達磨の分類では、正念は善心所に属し、遍行心所（すべての認知過程に存在する心所）にも不善心所にも属さない。特定の、正の方向の心理的品質だ。」

「右を見てほしい。」彼の指が表の右半分に移った。「SatiMonitor とは何か？ それは**イベント駆動**だ——努力を必要としない。EventBus を購読し、イベントが来れば処理し、来なければ処理しない。ātāpī がない、精進がない。それは**価値中立**だ——四次元品質ベクトル（連続性、粒度、速度、平穏性）は善悪の判断をしない。品質が高いことは善ではない。品質が低いことは悪ではない。ただ測定するだけだ。」

彼は一拍置いた。二列の対比が各学者の脳内で自発的に化学反応を起こすのを待った。

「SatiMonitor は正念の工学化ではない。」彼の声がより低く、より明瞭になった。「正念は意志駆動だ——正念を保つには努力しなければならない。SatiMonitor はイベント駆動だ——努力は不要で、購読するだけでよい。正念には道徳的価値がある——正念は善だ。SatiMonitor には道徳的価値がない——ただ測定するだけだ。」

そして最も重要な一文を言った：

「千年来、仏教は正念を行蘊に帰属させてきた。この分類は正しい——**仏教にとっては**。だが SatiMonitor は正念ではない。行動パターンの品質モニターだ。受（感知）、想（認識）、識（評価）に属する。**行（意志駆動の善心所）には属さない。**」

---

これは仏教学者が仏教の内部ロジックを用いて、ある工学コンポーネントが仏教の分類を使うべきでないことを証明したのだ。

外部からの反駁ではない——「仏教の分類は工学に適用できない」という粗暴な二分法ではない。内部論証だ——仏教自体の判定基準（善心所 vs 遍行心所、意志駆動 vs イベント駆動）が SatiMonitor と正念の等価性を排除した。

NAGARJUNA は席で微かに頷いた。中観学者は唯識学者の方法を認識した——外から立場を攻撃するのではなく、内部からその自己矛盾を示す。これは prasanga——帰謬法——の変形だ。「あなたの結論が間違っている」ではなく、「あなた自身の前提があなた自身の結論を排除する」。

## VII. 三番目のスロット

D2-Q3 は技術的だ——だが技術的であることは重要でないことを意味しない。

**PluginHooks に `monitors` スロットの新設は必要か？**

KERNEL が OS の観点から答えた。声はオペレーティングシステム専門家特有のアーキテクチャ感を帯びていた——すべてのコンポーネントにその位置があり、すべての位置にそのライフサイクルがある。

「monitors は独立したライフサイクルを持つ。」彼は言った。「ループ起動時に開始（MonitorRegistry.startAll()）、ループ終了時に停止（MonitorRegistry.stopAll()）。これは sensors（イベントごとにトリガー）や tools（オンデマンド呼び出し）とはまったく異なる。独立したライフサイクルには独立した管理が必要だ。」

彼はホワイトボードにライフサイクル図を描いた：

```
                 PluginLoader.load()
                        │
    ┌───────────────────┼───────────────────┐
    │                   │                   │
  sensors           monitors             tools
  (per-event)    (loop lifecycle)     (on-demand)
    │                   │                   │
  イベントごとに     LoopStart → LoopEnd    オンデマンド
  トリガー           呼び出し
```

「三種類の異なるトリガーモード。三種類の異なるライフサイクル。混在させるべきではない。」

```typescript
interface PluginHooks {
  // ... existing hooks ...
  monitors?: ILoopQualityMonitor[];  // NEW: 独立ライフサイクル
}
```

D2-R3：monitors スロット新設。20/20。全票。
