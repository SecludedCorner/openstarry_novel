# 第六章：名前の代償

---

劇場の照明は変わらなかった。相変わらず 6500K の冷白色だ。だが何故か、空気の温度が変わった。

物理的な温度ではない。感覚だ——試験開始前の教室、あるいは法廷の判決宣告前の待機のような。全員が自分の席に座っていたが、誰一人として本当に休んではいなかった。

SUNYATA がプロジェクタースクリーンの前に歩み寄り、Master のあの一文を画面に映し出した。

大きな文字。白地に黒字。書式なし。

> **「梵語を使うならば、その定義に対して責任を持つ必要がある。Sati の内容が完全に match していると思うか？ それはどの蘊に帰属するのか？」**

---

## I. 帰謬

NAGARJUNA が立ち上がった。

Cycle 02-5 全体を通じて、彼はすでに何度も立ち上がっていた——D1 で「後貼りラベル」を認め、D3 で十二因縁のスケール不一致を分析した。毎回、彼は中観学者特有の端正な姿勢を保っていた——誤りを認めることは学術の核心的技能であり、彼はそれを決して回避しなかった。

だが今回は違う。

今回は誤りを認めているのではない。一つの推論を完成させているのだ。

「D2 で SatiMonitor の蘊帰属はすでに確定した」と彼は言った。声は普段より半音低い。緊張ではない——ずっと考え続けていたが口に出さなかった論証を組み立てているからだ。

「結論は [vedana, samjna, vijnana]。三蘊。行蘊を含まない。」

一拍置いた。

「Sati——パーリ語の正念——は仏学の伝統において行蘊の心所である。cetasika であり、意志的で、能動的で、道徳的に正向な修行活動だ。阿毘達磨がそれを行蘊に置いたのは、正念が一種の修行だからだ——意図を持って行わなければならないことだ。」

彼は ASANGA の方を向いた。ASANGA は微かに頷いた。NAGARJUNA が何を言おうとしているか、彼にはわかっていた。

「さて」と NAGARJUNA は言った。「帰謬論証——prasaṅga——をやろう。これは中観学の基本的方法だ。」

彼はホワイトボードに二行を書いた：

```
前提 A：Sati = 行蘊心所（仏学の定義）
前提 B：SatiMonitor ≠ 行蘊（D2-R2 の結論）
```

そしてその下に第三行を書いた：

```
∴ SatiMonitor ≠ Sati
```

「もし正念（Sati）が仏学において必然的に行蘊の心所であり、そして我々が SatiMonitor は行蘊ではないと証明したならば——SatiMonitor は正念ではない。」

彼はペンを置いた。

「正念ではないものを、なぜ Sati と呼ぶのか？」

---

沈黙。

不同意の沈黙ではない。全員が同時に、もっと早く理解すべきだったことを理解した——そういう沈黙だ。

D2 の SCRIBE のト書きにはこう記されていた：「三つのテストはこの問いを発しなかった。Test 1 は『除去すると仕様が変わるか？』と問う。Test 2 は『ソースコードで使用されているか？』と問う。Test 3 は『設計決定を駆動したか？』と問う。どのテストも『名前と定義は一致しているか？』とは問わなかった。」

NAGARJUNA がたった今行ったこと——それはまさに、見落とされていた問いを発することだった。

三つのテストは、ある仏学的マッピングに工学的価値があるかどうかを判定できる。だがそれらは、名前がその定義にふさわしいかどうかを判定できない。ISatiMonitor は Test 2 を通過した——ソースコードで使用されている。だがより基本的な基準——名前と機能が一致していない——を通過しない。

## II. 確認

ASANGA が次に発言した。立ち上がらなかった——席に座ったまま、両手をテーブルの上に平らに置いて。まるで老教授が講義の最終まとめをしているかのようだった。

「NAGARJUNA の帰謬は形式的に正しい。機能分析で確認しよう。」

彼は D2 のあの対照表の拡張版を取り出した：

| 次元 | 仏学の正念（Sati/Smṛti） | SatiMonitor |
|------|----------------------|-------------|
| 能動性 | 能動的（ātāpī、精進） | 受動的（event-driven） |
| 道徳性 | 道徳的に正向（kuśala） | 価値中立 |
| 意志性 | 意図が必要（cetanā） | 自動実行 |
| 伝統的帰属 | 行蘊心所 | 受+想+識（D2-R2） |
| 修行上の地位 | 八正道第七支 | 品質モニター |

五つの次元。五つの不一致。

「SatiMonitor が行蘊の活動でないならば」と ASANGA は言った。彼の声はすでに確認済みの結論を読み上げるかのように平坦だった。「それは Sati ではない。我々自身の分類分析が、我々自身の命名を否定している。」

少し間を置いた。

「我々は間違いを犯した。分類の間違いではない——分類は正しかった。命名の間違いだ。Cycle 02-4 の時点で ISatiMonitor という名前を踏襲したのは、我々の議論の中ですでに何百回も使われていたからだ。慣性が名前そのものへの批判的視点を失わせた。」

## III. 新しい名前

SUNYATA が問うた：「では何と呼ぶべきか？」

数分のうちに四つの提案が出された：

| 提案者 | 名前 | 理由 |
|--------|------|------|
| ARCHIMEDES (#16) | ILoopQualityMonitor | 機能を正確に記述：認知ループの品質モニター |
| GUARDIAN (#11) | IBehaviorQualityMonitor | 行動レイヤーの品質監視を強調 |

## IV. 核融合炉

そして SUNYATA が一言発した：

「IPrajna。」

劇場の空気がまた変わった。

もし ISatiMonitor の問題が「名前と機能の不一致」であるならば、IPrajna の問題は——

NAGARJUNA は目を閉じた。しばらくして目を開き、話し始めた。

「般若」と彼は言った。「仏学における最高の認知能力だ。」

彼の声にはほとんど畏敬の重みがあった——研究チームに対してではなく、この言葉そのものに対して。

「般若——prajñā——は仏学においてすべての法の実相を照見する智慧である。普通の聡明さではない。分析能力ではない。知識の蓄積ではない。般若は超越的なものだ——空性の直観であり、実相の照見であり、二千五百年にわたる大乗仏学体系全体の中核的認知目標だ。」

彼はホワイトボードに向き直り、二行を書いた：

```
般若（仏学）：すべての法の実相を照見する最高の智慧
IPrajna（工学）：信頼度に ±0.05 を加減する関数
```

ASANGA が席から一言発した。後にすべての人がこの言葉を記憶することになる：

「これは温度微調整つまみを『核融合反応炉』と呼ぶようなものだ。」

---

誰も笑わなかった。

なぜなら ASANGA は冗談を言っているのではないからだ。彼は精確なアナロジーで精確な問題を説明しているのだ。

IPrajna のインターフェース定義は何か？ 一つのメソッド。入力は信頼度（confidence）とコンテキスト（context）。出力は `{ delta: number, reasoning: string }`。delta は [-0.05, +0.05] の範囲に制限される。

これはクランプだ。微調整器だ。既存の信頼度に対して ±0.05 の修正を行う監査関数だ。

それを IPrajna と呼ぶ——般若——すべての法の実相を照見する最高の智慧——

PASCAL がノートに書いた：「決定理論の観点から見ると、IPrajna が実際に行っているのは bounded secondary evaluation だ。入力は一次信頼度、出力は二次修正量、ハードリミット ±0.05。これは監査であり、般若ではない。」

BABBAGE が続けて分析した：「IConfidenceAuditor が意味的に最も精確だ。このインターフェースには一つのメソッドがあり、入力は信頼度とコンテキスト、出力は監査結果だ。Audit——監査——この操作を精確に記述している：既存の判断に対する、有限範囲の二次評価。」

---

**D4-R2 投票：IPrajna → IConfidenceAuditor。16/20。**

D4-R1 より高い合意。

少数意見：WIENER は IThresholdCalibrator を支持した（二票）——制御理論の観点から「監査」ではなく「校正」を見出した。HERACLITUS と PENROSE は ISecondaryEvaluator を支持した（二票）——「監査」は事後チェックを暗示するが、IPrajna は意思決定プロセスにリアルタイムで参加していると考えた。

BABBAGE が二つの少数意見に反論した：「Calibrator はシステム自体の調整を暗示する——だが IConfidenceAuditor はシステムを変更しない、一つの数値に対して有限の修正を提示するだけだ。Evaluator は汎用的すぎる——あらゆる関数が evaluation を行っている。Auditor の意味的境界が最も精確だ：独立した、有限範囲の、書面記録を生成する二次チェック。」

完全なリネーム表：

| 旧名称 | 新名称 |
|--------|--------|
| IPrajna | IConfidenceAuditor |
| IPrajnaAdjustment | ConfidenceAuditResult |
| prajnaAdjust() | clampAuditDelta() |
| Delta_prajna | Delta_audit |
| Model Delta 第二層名称 | Confidence Audit |

## V. 第三の審判

SUNYATA は当初、D4 はここで終わると思っていた。二つの名前。二回のリネーム。ISatiMonitor と IPrajna。

だが Master の審閲にはまだ続きがあった。

「Doc 03。」

Doc 03——`Sila_Prajna_Security_Framework.md`——戒慧セキュリティフレームワーク。

---

この文書はもともと D4 の議題には入っていなかった。その名前には Sila（戒）と Prajna（般若）が含まれているが、D1 で団隊は 14/20 の票数でそのまま保持すると決定していた。理由は：文書名中の仏学用語は「背景説明」であり「コード識別子」ではない、というものだった。

だが今、D4 の文脈の中で、Master の「梵語を使うならばその定義に対して責任を持つ必要がある」という言葉の後で——

SUNYATA が言った：「Doc 03 を再審査する必要がある。我々がたった今確立した基準を使って。」

彼はホワイトボードに四つのテストを列挙した——いや、三つのテスト。そしてその横に一つ新しいテストを加えた。

```
Test 1（必要性）：除去すると工学仕様が変わるか？
Test 2（コード識別）：ソースコードで使用されているか？
Test 3（決定駆動）：DC が確認した設計決定を駆動したか？
Test 4（定義責任）：梵語術語を使用する際、コンポーネントの機能がその術語の仏学的定義と一致しているか？
```

第四のテスト。

天から降ってきたのではない——D4-R1 と D4-R2 の議論から結晶化したのだ。ISatiMonitor の問題は工学的価値がないことではなかった（あった）。ソースコードに存在しないことでもなかった（存在した）。設計決定を駆動しなかったことでもなかった（駆動した）。問題は——名前と定義が一致していなかった。

三つのテストはこの次元を捕捉できなかった。第四のテストがそのギャップを埋めた。

---

NAGARJUNA がペンを取り、Doc 03 の四つのテスト結果の横に四つの×を打った：

**Test 1（必要性）**：Plugin ライフサイクルの五つの状態（CREATED → LOADING → LOADED → STARTING → RUNNING → STOPPING → DISPOSING → DISPOSED）は種子理論（bīja）なしでも理解できる。❌ 不通過。

**Test 2（コード識別）**：ソースコード中の実際の型は英語——`'active' | 'dormant' | 'suspended' | 'revoked' | 'retired'`。Sila と Prajna は TypeScript コメント中にのみ出現し、コード識別子ではない。❌ 不通過。

**Test 3（決定駆動）**：DC の確認なし。❌ 不通過。

**Test 4（定義責任）**：Sila（戒）の仏学的定義は道徳規範、修行戒律、僧団の生活規範。だが Doc 03 で Sila はアクセス制御——Plugin のセキュリティ権限管理——を表すために使われている。戒律 ≠ アクセス制御。Prajna（般若）は CVE 取消メカニズム——セキュリティ脆弱性の自動応答プロセス——を表すために使われている。般若 ≠ CVE 取消。❌ 不通過。

四つのテスト。四つとも不通過。

---

「D1 での私の投票を撤回する」と NAGARJUNA は言った。

「私も撤回する」と ASANGA が言った。

ASANGA は重要な区別を補足した：「Doc 16 は独立した仏学マッピング文書だ——その存在目的が体系的な対照を行うことそのものだ。Master がその保持を裁定したのは、マッピング文書は装飾ではないからだ。だが Doc 03 はマッピング文書ではない——工学文書の中に仏学的装飾が埋め込まれているのだ。違いはここにある：Doc 16 の仏学的内容は主体だ。Doc 03 の仏学的内容は装飾だ。同じ仏学的内容、異なる文書タイプ、異なる判断。」

これはまさに第五章で Master が加えたあの次元——文書タイプ——だった。ASANGA はそれを具体的なケースに適用した。

---

**D4-R3 再投票：Doc 03「Sila-Prajna Security Framework」→「Plugin Security Lifecycle」。17/20。**

14/20 で変更せず保持から、17/20 でリネーム清掃へ。同じ文書、同じ人々、異なる結論。違いは何か？ 第四のテスト。

HERACLITUS が少数意見（三票）を率いた：「種子理論は仏学的背景として教育的価値がある——Plugin ライフサイクルの『休眠→発芽→成長→枯れ→種子回収』という隠喩の理解を助ける。四つのテストを通過しなくとも、付録として保持することには文化的価値がある。」

多数派の応答：「付録としての保持は別途議論できる。だが文書タイトルと本文の仏学的装飾はクリーンアップすべきだ。工学文書中の仏学的装飾と、付録中の仏学的解説は別のものだ。」

クリーンアップ範囲の確認：
1. 文書名の変更（Sila_Prajna_Security_Framework → Plugin_Security_Lifecycle）
2. 仏学マッピング表の除去
3. TypeScript コメント中の種子理論マッピングのクリーンアップ
4. すべての工学的内容の保持（五状態、状態遷移図、三層セキュリティモデル）

## VI. 第四のテスト

D4 終了前、SUNYATA が最後の問いを投げかけた：「第四のテスト——定義責任——は永続的な基準とすべきか？」

投票は不要だった。全員が頷いた。

NAGARJUNA が最後の陳述を行った：「三つのテストは工学的価値を問うている。第四のテストは意味的誠実さを問うている。たとえ名前が Test 2 を通過しても——ソースコードで使用されていても——Test 4 を通過しなければ——その機能が仏学的定義と一致しなければ——それはなおリネームされるべきだ。」

「ISatiMonitor は Test 2 を通過した。ソースコードにある。だが Test 4 を通過しない——正念ではないからだ。だからリネームされた。」

「IPrajna も実装されれば Test 2 を通過するだろう。だが Test 4 を通過しない——±0.05 の加減は般若ではないからだ。だから事前にリネームされた。」

「第四のテストの本質はこうだ：**仏学の梵語術語をコード識別子として使用する場合、そのコンポーネントの機能はその術語の仏学的定義と一致しなければならない。一致しない場合は、工学用語を使用する。**」
