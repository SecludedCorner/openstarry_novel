## I. 五本の柱

D3 は Cycle 02-5 で最も長い討論だ——120 分、六節、六回の投票。だが最も激しくはない。D3 の雰囲気は博士論文の口頭試問に近い：厳密で、体系的で、秩序だっている。受験者は一人の人間ではない——受験者は一棟の建築物だ。

**「五蘊 OOP アーキテクチャは工学的実装を支えるに足るか？」**

SUNYATA はホワイトボードに問いを書いた。そして横に五つの四角を描き、一列に並べた。五本の柱のように。

```
IRupa ─── IVedana ─── ISamjna ─── ISamskara ─── IVijnana
 色蘊       受蘊        想蘊        行蘊          識蘊
```

五つのルートインターフェース。OpenStarry の建築全体がこの五本の柱の上に立っている。五蘊は装飾ではない——構造だ。PluginHooks の型分類、PluginLoader の登録ルーティング、ExecutionLoop の段階順序を定義している。

問題は：五本の柱で足りるか？

## II. 三管斉下

D3-Q1 は三管斉下の完備性証明を起動した。

**第一管：LINNAEUS の機能カバー率。**

LINNAEUS は A1 研究報告から一つの表を取り出した。表の左列は v0.28.0-alpha のすべての plugin hook 型、右列は蘊帰属。一つずつ検証した：

```
sensors (IVedanaSensor)          → 受蘊  ✓
recognizers (ISamjnaRecognizer)  → 想蘊  ✓
arbiters (IGearArbiter)          → 想蘊+識蘊  ✓
tools (ITool)                    → 行蘊  ✓
volition (IVolition)             → 識蘊  ✓
monitors (ILoopQualityMonitor)   → 受+想+識  ✓ (D2-R2)
auditor (IConfidenceAuditor)     → 識蘊  (実装待ち)
```

「機能カバー率 100%。」LINNAEUS は言った。「孤児 hook はない。分類不能なインターフェースはない。すべての plugin 型が五蘊の一つまたは複数の分類に帰属できる。」

**第二管：BABBAGE の型代数。**

BABBAGE は眼鏡を押し上げた。彼はアナロジーを使わない。証明をする。

ホワイトボードに一行の TypeScript を書いた——コードではなく、型表現だ：

```typescript
type AllPluginTypes =
  | IVedanaSensor       // extends IVedana     → 受蘊
  | ISamjnaRecognizer   // extends ISamjna     → 想蘊
  | IGearArbiter        // ['samjna','vijnana'] → 想蘊+識蘊
  | ITool               // extends ISamskara   → 行蘊
  | IVolition           // extends IVijnana    → 識蘊
  | ILoopQualityMonitor // ['vedana','samjna','vijnana'] → 受+想+識
```

「五つのルートインターフェースの和集合が PluginHooks 内のすべての可能な plugin 型をカバーする。」彼は言った。「これは型代数の完備性だ——すべての具体型が五つのルート型の部分型または交差型だ。五蘊の和集合の外に落ちる意味のある plugin 型は存在しない。」

彼は一拍置いた。

「第六のルートインターフェースは不要だ。」

**第三管：ASANGA の阿毘達磨網羅性。**

ASANGA の論証は仏教から出発した。声は経典を引用する時特有のリズムを帯びていた——二千年間暗唱されてきた文章を朗読するかのように。

「『阿毘達磨倶舎論』巻一：『五蘊は一切の有為法を摂尽す。』」彼は言った。「これは仏教の基本公理だ——五蘊はすべての有為（conditioned）な現象を包含する。第六蘊は存在しない。もしある現象がいずれの蘊にも帰属できなければ、五蘊が不完備なのではなく——その現象は無為（unconditioned）なのだ。」

「OpenStarry の plugin は有為だ——作成され、設定され、読み込まれ、実行され、廃棄される。無為法ではない。従って、五蘊の分類はそれらを網羅できるはずだ。」

彼は一拍おいた。そして仏教学者特有の謙虚さで補足した：

「もちろん、仏教の公理を直接工学の証明に使うことはできない。だが LINNAEUS と BABBAGE の証明が既に工学の角度から完了している。仏教の網羅性論証は一つの——整合性チェックだ。三つの独立した論証が同じ結論に到達すれば、結論の信頼度が高まる。」

三つの学問。分類学、数学、仏教。三つの経路。一つの結論。

PENROSE が警告を付け加えた——彼は常に警告を付け加える。観察席の最上部から全場を俯瞰している、より高い次元から三次元世界を観察する存在のように。

「識蘊は現在、最も多くのサブインターフェースを持つ——IVijnana、IGearArbiter、IVolition、IConfidenceAuditor。今後さらに膨張するなら、インターフェース数の増加を監視する必要がある。五つのインターフェースでは足りないということではなく——ある柱には追加の支えが必要になるかもしれないということだ。」

彼の警告は記録された。だが投票には影響しない。

D3-R1：五つのインターフェースで十分。20/20。

## III. ショートカットの通路

D3-Q2 は対応づけの正確性に拡大鏡を当てた。九つの hook 型について、蘊帰属を一つずつ検証。

大部分の検証は手続き的だった——LINNAEUS が A1 報告で詳細な対応づけ表を作成済みで、TURING がすべての型継承関係を検証済みだった。だが一つの例外が真の議論を引き起こした。

ARCHIMEDES が挙手した：「SlashCommand はどの蘊に属するか？」

この問いは些細に見える。SlashCommand はユーザーが直接入力するコマンドだ——/help、/clear、/status。ExecutionLoop を経由しない。SparshEvent をトリガーしない。VedanaSensor、ISamjnaRecognizer、IGearArbiter、IVolition を経ない——五蘊のプロセス全体をバイパスする。

三人が同時にこの問いに答えた。三つの異なる角度から。

KERNEL は OS の観点から：「SlashCommand は ExecutionLoop 全体をバイパスする。五蘊流転を経験しない。Unix の signal handler に類似する——プロセスレベルで捕捉され、通常のシステムコールパスには入らない。OS の分類では、ユーザー空間のロジックではなく、カーネルレベルの割り込み処理だ。」

NAGARJUNA は仏教の観点から：「認知ループに入らない反射的動作。五蘊がそれをカバーしないのではなく——五蘊が機能する範囲の外にある。呼吸のように——呼吸を観察することはできる（認知ループに入る）が、呼吸そのものは五蘊流転なしに起こりうる。SlashCommand はシステムの呼吸だ——認知の外に、認知の下にある。」

GUARDIAN は安全の観点から：「バイパスは SlashCommand が SafetyMonitor の保護を受けないことを意味する。これは設計上の決定だ——一部の操作は安全チェックを迂回する必要がある——だがセキュリティ文書に明確に記録すべきだ。」

SUNYATA は三つの観点を一つの結論に統合した：「SlashCommand は蘊間基盤（cross-skandha infrastructure）だ。いずれの蘊にも属さない。その分類根拠と安全上の観察は Doc 45 に記録する必要がある。」

D3-R2：対応づけは正確。20/20。

## IV. ラベル vs インターフェース

D3-Q3 は精巧な問いだ：**skandha tag をルーティングに使うべきか？**

各 plugin には skandha フィールドがある——メタデータ。問題は：PluginLoader が plugin を読み込む際、skandha tag でこの plugin がどの registry に行くかを決めるべきか？

直感的には合理的に見える——受蘊 plugin は VedanaRegistry へ、行蘊 plugin は ToolRegistry へ。図書館の索引カードのように。

だが VITRUVIUS が問題を指摘した。建築家の目は構造の脆弱性に特に敏感だ——設計図面上で将来割れるかもしれない亀裂を見ることができる。

「現在のシステムは duck typing を使っている。」彼は言った。「plugin が IVedanaSensor インターフェースを実装していれば、VedanaRegistry に自動的に受け入れられる。ラベルは不要だ。ラベルルーティングに切り替えれば、新たなエラー源が導入される：ラベルとインターフェースの不一致。plugin が自らを受蘊と宣言しながら ITool インターフェースを実装している場合——どうするのか？」

TURING が事実を確認した。`plugin-loader.ts` を開いた。

「v0.28.0-alpha の PluginLoader は type guard を使って plugin を分類している。`isVedanaSensor(hook)` が検査するのはインターフェース実装であり、メタデータではない。」

BABBAGE が型理論で終決的な陳述をした：「type guard は信頼できる——コンパイル時とランタイムの両方で検証できる。メタデータは信頼できない——ランタイムでしか検査できず、人手による正しいマーキングに依存する。信頼できないものでルーティングを行うことは、型安全なシステムにおいて後退だ。」

D3-R3：skandha tag = メタデータのみ、ルーティングには使用しない。18/20。

MESH と GUARDIAN が少数票を投じた——メタデータルーティングを補助的検証として使えるとの意見。だが多数意見は明確だ：インターフェースが真実、メタデータは備考。

## V. 最も細い柱

D3-Q4 は最も率直な問いだった。

**ISamskara にはより多くのサブインターフェースが必要か？**

現在、行蘊にはサブインターフェースが一つだけ：ITool。IVolition は D2 以降、識蘊に帰属が確認されている（Phase 5 行動前 vs Phase 8 行動中）。行蘊は五蘊の中で最も「狭い」——仏教の行蘊は 49 心所を包含する（五十一心所から受と想を除く）が、OpenStarry の行蘊には ITool しかない。

ASANGA が立ち上がった。D3 で最も率直な発言をした。自ら設計した最も薄い壁を指して「ここが薄いと分かっている」と言う建築家のように。

「認めなければならない。OpenStarry の行蘊設計は仏教の伝統との差異が最も大きい。」彼は言った。声に弁解はない——ただ事実だけ。「伝統的行蘊は 49 心所を包含する——意志、精進、慧、信、慚、愧——ほぼすべての心理活動が行蘊に帰属する。だが OpenStarry は行蘊を ITool に窄化した——外部に観察可能な行動。外部に観察可能な行動だけだ。」

HERACLITUS が ExecutionLoop の動的観点から工学的根拠を提供した：

```
Phase 5: IVolition 審議 ──→ Phase 8: ITool 実行
          行動前                     行動中
          識蘊                       行蘊
```

「IVolition は Phase 5 にある——行動の前。何をするかを決定する。ITool は Phase 8 にある——行動の最中。まさに実行している。両者は異なる段階にある。IVolition を行蘊に移せば、蘊流転の概念的自然さが壊れる——行蘊は『まさにやっていること』であるべきで、『何をするか決めること』ではない。」

NAGARJUNA が仏教上の譲歩をした。熟慮を経た平穏な口調で——妥協の平穏ではなく、理解の平穏だ。

「仏教の行蘊分類は修行者の内観に基づいている。」彼は言った。「観察されたすべての心理活動は、受と想を除いて行蘊に帰属する。だが OpenStarry は修行者ではない。ソフトウェアシステムだ。ソフトウェアシステムの分類は機能に基づくべきであり、内観に基づくべきではない。」

現状維持が全票で通過した。

D3-R4：ISamskara にサブインターフェースを追加しない。20/20。

だが全員が一つのことに同意した：Doc 45——これから作成される五蘊 OOP アーキテクチャ文書——にこの差異を記録しなければならない。行蘊の窄化はエラーではなく、意識的な偏離だ。意識的な偏離には説明が必要だ。

## VI. 二つの古道

D3 の最後の二問は仏教的対応づけの付録可能性に関するものだ。

**D3-Q5：十二因縁。**

NAGARJUNA がまず分析を展開した。ホワイトボードに二本の線を引いた——一本は非常に長く「十二因縁」と標注され、一本は非常に短く「ExecutionLoop」と標注されている。

「スケール。」彼は言った。長い線を指して。「十二因縁が記述するのは三世両重因果だ——無明は行に縁り、行は識に縁り、識は名色に縁る——前世の無明から今世の老死、そして来世の無明まで。一つの完全な輪廻だ。」

短い線を指した。

「ExecutionLoop が記述するのは一つの認知処理のサイクルだ——イベント受信から行動実行まで。数十ミリ秒から数秒。一つの認知サイクル。」

「十二因縁は生命周期を跨ぐ。ExecutionLoop は単一の認知サイクルだ。スケールの差は——」彼は少し考えた。「——少なくとも数桁。」

BABBAGE が構造射（morphism）の構築を試みた。彼の方法は厳密だ——二つの構造間の構造保存写像を探す。射が存在すれば、二つの構造は数学的に「同型」または「準同型」であり、一方の構造における関係が他方に対応することを意味する。

彼は失敗した。

「十二因縁間の因果関係は線形で、スキップ不能だ。」彼は言った。「無明は行の前でなければならない。行は識の前でなければならない。スキップできない。だが ExecutionLoop の段階はスキップできる——IVedanaSensor のない Agent は受蘊段階をスキップする。イベント受信から想蘊の認識に直接跳ぶ。構造が異なる。射は存在しない。」

だが HERACLITUS が局所的な対応を指摘した。十二因縁の一部分を描いた：

```
触 (sparsha) → 受 (vedana) → 愛 (tanha) → 取 (upadana)
```

「触が受を起こす。受が渇愛を起こす。渇愛が執取を起こす。この部分——全体ではなく、この部分だけ——と SparshEvent → ChannelVedana → KleshaGain → VolitionalDecision のデータフローには構造的対応がある。」

投票は分岐を反映した。

D3-R5：選択的付録。13/20。

7 票が反対。反対者は局所的対応の存在を否定したのではなく、工学文書に記録する必要性を疑問視した。

**D3-Q6：認知シーケンス（citta-vīthi）。**

雰囲気が変わった。

認知シーケンスは上座部仏教における認知過程の精密な分析だ——一つの念が生じてから消滅するまでの完全な過程：

```
有分心 → 転向心 → 五識 → 接受心 → 推度心 → 確定心 → 速行心 → 彼所縁心
```

BABBAGE が再び構造射を試みた。今回は——

成功した。

「認知シーケンスと ExecutionLoop は同スケールだ。」彼は言った。眼鏡を押し上げた——精密な分析を行う時に必ずする動作。「どちらも単一の認知処理の段階列だ。そして——」

ホワイトボードに二つの FSM（有限状態機械）を並べて描いた。二枚の平行する鏡のように。

```
認知シーケンス FSM:
  有分心 → 転向心 → 五識 → 接受心 → 推度心 → 確定心 → 速行心 → 彼所縁心
    (idle)  (alert)  (sense) (accept) (examine) (decide) (act)    (review)

ExecutionLoop FSM:
  Idle → EventReceived → Sensing → Recognizing → Arbitrating → Deliberating → Acting → Feedback
```

「構造射が存在する。」BABBAGE は言った。声は数学者が同型を発見した時特有の満足感を帯びていた——得意ではなく、審美だ。「二つの FSM 間に構造保存写像がある。Idle ↔ 有分心。EventReceived ↔ 転向心。Sensing ↔ 五識。すべての状態遷移が二つの FSM で対応を持つ。」

「これは比喩ではない。」彼は強調した。「これは数学だ。」

彼の票は D3-Q5 の反対（十二因縁は記録しない）から D3-Q6 の支持（認知シーケンスは記録する）に転じた。記録に一行書いた：「FSM 射が私の転向の鍵となる論拠だ。十二因縁には射がない。認知シーケンスにはある。品質が票を決める。」

D3-R6：選択的付録。17/20。

13/20 から 17/20 へ——四票の差。四票の差は感情ではなく——精度だ。十二因縁の対応づけは近似的だ。認知シーケンスの対応づけは精密だ。BABBAGE は自らの数学ツールで両者の品質を計り、そして自らの票で結論を表した。
