# 第一章：習気は後天のものである

## I. 問いの形

D1 の議題が SUNYATA によってホワイトボードに書かれた時、円形劇場に特殊な静けさが生まれた。D3 のような「全員が合意した」静けさではなく（それは明後日を待たねばならない）、D5 のような「嵐の前の」静けさでもなかった（それは五日目を待たねばならない）。これは「ああ、そういうことか」という静けさだった。

ホワイトボードにはこう書かれていた：

```
IGearArbiter — それはどの蘊に属するか？
```

七文字。一つの疑問符。だが疑問符の重みは字面にはなかった。

Pre-DC で既に確認されていた：VasanaEngine はコアから外部化されプラグインとなる。外部化そのものは問題ではない——問題は外部化の後だ。プラグインは必ずいずれかの蘊に属さねばならない。Plugin Manifest には `skandha` フィールドがある。このフィールドに何を記入するか？

ATHENA が最初に口を開いた。その声にはエンジニアの「この問題について既に何日も考えた」という落ち着きがあった：

「IGearArbiter は二つのことを行います。第一に、高速認識——リクエストを入力し、規則ベースとマッチングし、既知のパターンがあるか判断する。これは想蘊（samjñā）の機能、取相認識です。第二に、信頼度判断——このマッチングはどの程度信頼できるか？ LLM をスキップする価値があるか？ これは識蘊（vijñāna）の機能、分別了別です。」

彼女は一瞬間を置いた。

「蘊を跨いでいます。」

---

三つの方策がテーブルに並べられた。BABBAGE は不安になるほど精確なやり方でそれらを表にまとめた：

| 方策 | インターフェース継承 | Manifest 標記 | 型安全性 |
|------|---------|-------------|---------|
| (A) | extends ISamjna | `skandha: 'samjna'` | 強——ただし vijñāna の意味を喪失 |
| (B) | 独立インターフェース | `skandha: ['samjna', 'vijnana']` | 中——Manifest 層が多値 |
| (C) | extends IVijnanaMechanism（新中間層）| `skandha: 'vijnana'` | 強——ただし型の複雑度が増大 |

BABBAGE は眼鏡を押し上げた。押し上げる必要はなかった——フレームは合っていた——だがこの動作自体が一種の句読点であり、「これから言うことは非常に精確だ」という意味だった。

「方策 (A) には型代数の問題があります。」

彼はホワイトボードに一行書いた：

```
ISamjna.skandha = 'samjna'
IVijnana.skandha = 'vijnana'
'samjna' & 'vijnana' = never
```

「IGearArbiter が ISamjna を extends するなら、その skandha フィールドは型レベルで 'samjna' にロックされます。インターフェース層で二つの蘊に同時に属すことは表現できません。判別式共用体型はこのような交差を許しません。」

DARWIN が首を傾けた。「簡単に言うと？」

「簡単に言うと：TypeScript の型システムは、あるものが猫であると同時に犬であることを許しません。Manifest 上で猫犬だと宣言することはできますが、インターフェース層のコンパイラが拒否します。」

方策 (A) は BABBAGE が言い終えて三秒後には、全員の心理的候補リストから消された。

「DC-6 です」と彼は言い、分類学ノートを開いた。「Master の裁定：『蘊内のサブインターフェースの帰属を固定しない。』IVijnanaMechanism を IVijnana のサブインターフェースとして——」

「——固定してしまう。」KERNEL が引き取った。「IVijnanaMechanism を作った時点で、将来あらゆる蘊横断コンポーネントにもう一つ中間層が必要になります。IVijnanaCognition。ISamjnaPerception。型の膨張は指数関数的です。」

方策 (C) は明確な投票で否決されたわけではなかったが、間違った位置に置かれた歯車のように、全員がそれが不適切であることを見て取った。

方策 (B) の転換点は ASANGA からもたらされた。

彼はずっと沈黙していた。BABBAGE の型代数と LINNAEUS の分類学の後、彼が口を開いた。その声は暗闇の中で灯された一本の線香のようだった——静かだが、方向は明確だった。

## II. evaluate() の戦争と平和

D1-1A が穏やかな川だったとすれば、続く evaluate() インターフェース設計の討論はその川の上の急流だった。

GUARDIAN が立ち上がった。彼が立ち上がることは滅多にない——Cycle 02-3 の六場の討論では、ほとんどの時間を座ったまま赤ペンで備忘録に下線を引いて過ごした。だが今日は立ち上がった。それは次に言うことが提案ではなく主張であることを意味した。

「単一の evaluate() メソッドでは不十分です。」

彼はホワイトボードの前に歩み出て、タイムラインを描いた：

```
   リクエスト到着 → evaluate() → 歯車切替 → ツール実行
                  ↑
            すべてのロジックがここに
            認識 + 判断 + 決定
            一つの関数がすべてを行う

   この関数が誤れば——チェーン全体が誤る
   この関数に注入されれば——チェーン全体が乗っ取られる
```

「二段階に分離することを提議します：`recognize()` と `resolve()`。」

```
   リクエスト到着 → recognize() → [検証ポイント] → resolve() → 歯車切替
                              ↑
                        SafetyMonitor が介入可能
                        認識結果を審査できる
                        継続するか否かを判断できる
```

「recognize と resolve の間に検証ポイントを挿入します。SafetyMonitor が認識結果の妥当性を検査し、resolve の実行を許可するか判断できます。」

ATHENA はほぼ即座に応じた：「冗長です。」

GUARDIAN の目がわずかに細まった。「冗長？」

「EventBus の同期イベントでまったく同じ効果が得られます。evaluate() の実行完了と同時に `gear:arbiter_evaluated` イベントを emit する。SafetyMonitor がこのイベントを subscribe する。同期処理です。」

「同期？ SafetyMonitor の処理が非同期ならどうなります？ 外部サービスへの問い合わせが必要な場合は——」

KERNEL が手を挙げた。発言を求める挙手ではなく——「技術的事実がある」という挙手だった。

「Node.js の EventEmitter.emit() は同期です。」

劇場全体が KERNEL に向き直った。

「emitter.emit('event', data) を呼び出すと、すべての登録済み listener が現在のコールスタック内で**順次同期実行**されます。nextTick でも microtask でも macrotask でもない。同期です。emit() が返る時点で、すべての listener は実行を終えています。」

GUARDIAN：「確かか？」

KERNEL は彼を一瞥した。その目は「Node.js のイベントシステムを書いたことがある」と語っていた。

「確かです。Node.js モデルにおいて、EventBus 同期イベントとあなたが提議する recognize/resolve 分離は、安全保証において等価です。違いは API の複雑度だけです。」

---

GUARDIAN はすぐには譲歩しなかった。優れたセキュリティ提唱者は一つの技術的事実で譲歩しない——その事実がカバーしない辺縁を探す。

「将来の実行環境が Node.js でないと仮定します。誰かが Worker Thread で非同期 EventBus を実装したとしたら——」

KERNEL：「それは実装のバグであり、設計上の欠陥ではありません。同期イベントのセマンティクスは契約レベルの保証であり、特定のランタイムに依存しません。もし誰かが非同期 EventBus を実装したなら、その人が契約に違反しています。」

WIENER が席から口を挟んだ：「妥協案を提示できます。」

全員が彼を見た。WIENER が「妥協」を提案することは滅多にない——彼が通常提案するのは「数学的証明」だ。しかし稀に、制御理論家も外交官を務める必要がある。

「三重の外部安全機構です。第一：EventBus 同期イベント——`gear:arbiter_evaluated`、遅延不可、スキップ不可。第二：閾値安全係数——θ_adjusted = θ_base × (1 + k_safety)、安全マージンが数学に内蔵される。第三：最小イベントセット——gear:switch、action:proposed、action:executed——あらゆる歯車操作に完全なイベント軌跡があり、外部監視が独立に検証できます。」

彼は GUARDIAN を見た。

「あなたの核心的主張は：一つの関数がすべてを行い外部チェックがないのは許容できない、ということです。三重の機構が三つの独立した外部チェックポイントを提供します。三つすべてが失効するなら、それはもはや設計の問題ではなく、ビザンチン障害です。」

GUARDIAN は三十秒沈黙した。セキュリティの領域で、三十秒の沈黙は攻撃面を評価していることを意味する。

「最小イベントセット。」彼はついに言った。「gear:switch に gear:arbiter_evaluated を追加してほしい。仲裁者の判断結果を知りたい。最終的な歯車切替だけでなく。」

WIENER：「同意。」

ATHENA：「同意。」

**D1-R1：evaluate() 単一メソッド + EventBus 同期イベント + 閾値安全係数——20/20。**

GUARDIAN は頷いた。彼の赤ペンが備忘録に一本の線を引いた——下線（「問題」を意味する）ではなく、取り消し線（「解決済み」を意味する）だった。

## III. 型ガード：evaluate.length ≤ 1

TURING は D1 の第三段の討論で十二文字しか発しなかったが、その十二文字が型ガードの設計を変えた。

議論の焦点は `isGearArbiter()`——ランタイムでプラグインが IGearArbiter インターフェースを実装しているかどうかを判定する方法だった。BABBAGE が二つの方策を提示した：

方策 A：Symbol brand（シンボル焼印）——インターフェースに Symbol プロパティを付加し、ランタイムで検査する。型安全性は最強だが、既存のコードスタイルとの一貫性がない。

方策 B：構造的型ガード——オブジェクトが id、priority、evaluate の三つのプロパティを持ち、型が正しいかを検査する。ダックタイピング。

BABBAGE は方策 A を好んだ。形式的により厳密だった。

TURING はあの十二文字を発した：

「`candidate.evaluate.length <= 1`。この一行を加えよ。」

BABBAGE：「どういう意味ですか？」

「`Function.length` は関数の仮引数の数を返します。evaluate(context) は一つの引数で length は 1。evaluate() は引数なしで length は 0。evaluate(a, b) は二つの引数で length は 2。もしプラグインがたまたま evaluate という名のメソッドを持ちながらシグネチャがまったく異なる場合——たとえば evaluate(x, y, z)——構造的型ガードはそれを通してしまいますが、length ≤ 1 の検査が阻止します。」

BABBAGE は二秒間呆然とした。それから笑った。「どうして思いつかなかったのだろう」という笑いだった。

「エッジケースの数が O(2^n) から——」

「O(1) に。一行のコード。九つのテストケース。」TURING の口調はログファイルを読み上げるように平坦だった。

```typescript
export function isGearArbiter(obj: unknown): obj is IGearArbiter {
  if (typeof obj !== 'object' || obj === null) return false;
  const candidate = obj as Record<string, unknown>;
  return (
    typeof candidate.id === 'string' &&
    typeof candidate.priority === 'number' &&
    typeof candidate.evaluate === 'function' &&
    candidate.evaluate.length <= 1  // TURING 増強
  );
}
```

**D1-R2：isGearArbiter() 構造的型ガード（evaluate.length ≤ 1 検査を含む）——20/20。**

## IV. 純度契約：観測関数

D1-R4 の討論は前三項より静かだったが、その哲学的深度は最も深かった。

問いはこうだ：evaluate() は何をしてよく、何をしてはならないか？

NAGARJUNA が一つの問いで議論の口火を切った：

「認識は造作か？」

仏学において、「造作」（saṅkhāra/saṃskāra）は有為法——世界に変化をもたらす行為だ。一方「認識」（samjñā）は取相——観察し、認知するが、観察される対象を変えはしない。

「人が赤信号を見る。見た瞬間、彼の目は赤信号を変えていない。赤信号は依然として赤い。だが彼は知った——赤信号は停止の信号であると。この『知る』は観測であり、造作ではない。」

NAGARJUNA は ATHENA に目を向けた：

「evaluate() は観測関数であるべきです。外部状態——コンテキスト、履歴、煩悩(kleśa)信号——を読み取ることはできる。しかしいかなる状態も変更すべきではない。ただ『見る』のであり、『為す』のではない。」

WIENER が制御理論の視点から引き取った：

「制御理論において、オブザーバ（observer）は純関数です——その出力は入力とシステム状態のみに依存し、逆にシステム状態を変えることはありません。もしオブザーバに副作用があれば——たとえば観測と同時に観測対象の値を変更するなら——システムは予測不能なフィードバックを生みます。」

「工学上の含意：evaluate() は冪等であるべきです。同じ入力 + 同じ外部状態 → 同じ出力。副作用なし。」

DARWIN が生物学的アナロジーを添えた：「感覚ニューロン（sensory neuron）は刺激源を変えません。信号を伝達するだけです。運動ニューロン（motor neuron）が環境を変える。evaluate() は感覚ニューロンです。」

KERNEL が時間制約を提示した：「各 arbiter に 100ms。チェーン全体で 200ms。arbiter がタイムアウトすればスキップし次へ進む。チェーン全体がタイムアウトすれば Gear 2 にフォールバック。」

PASCAL はここで唯一の棄権を行った——反対ではなく、「数値の部分は、100ms が適切かどうか判断するに足る工学的データが不足しています」というものだった。

**D1-R4：evaluate() 純度契約——観測関数 + 100ms/200ms タイムバジェット——19/20（PASCAL は数値部分を棄権）。**

---

だが evaluate() の純度は一つの問題をもたらした：evaluate() が副作用を持てないなら、VasanaEngine の「学習」機能——成功した Gear 1 の判断から経験を汲み取ること——はどうなるのか？

答えは D6 で初めて完全に明かされるが、D1 はその伏線を敷いた：

「副作用は独立したメソッドで実現します。」ATHENA が言った。「evaluate() は観測を担う。学習は——もしあるなら——独立した `imprint()` メソッドを通じて、evaluate() の外で、適切なタイミングで（たとえば一ラウンド終了後に）、外部からトリガーされます。」

NAGARJUNA が頷いた。「想蘊は認識し、行蘊は造作する。認識と造作を同じ関数に入れるべきではない。」

この分離——evaluate() の純度契約と将来の imprint() への副作用の委任——は、六場の討論を通じて最も持続的な設計原則の一つとなった。

## V. 純粋ルーティング：G-0 から G-4

D1 の最後の決議は全討論中で最も直截だったが、最も深い影響を持つものでもあった。

SUNYATA がホワイトボードに表を描いた：

```
ManoAggregator 退化行動表
```

「Pre-DC で既に確認されました——ManoAggregator は純粋なルーティングです。if/else。EventBus と同等の性質。ただし『純粋なルーティング』は異なるプラグイン構成の下で異なる振る舞いを示します。あり得るすべての状態を列挙する必要があります。」

KERNEL がホワイトボードの前に歩み出て、ペンを取った。五行を描いた：

| 等級 | IGearArbiter | IProvider | 行動 |
|------|-------------|-----------|------|
| G-0 | なし | なし | コアは生存するが認知不能 |
| G-1 | なし | あり | 常に Gear 2（純 LLM）|
| G-2 | あり | なし | Gear 1 のみ（規則のみ）|
| G-3 | あり | あり | 完全な双歯車 |
| G-4 | 複数 arbiter + ホットスワップ | あり | ランタイム動的ロード/アンロード |

NAGARJUNA は G-0 と G-1 を見つめ、後に幾度となく引用される一言を述べた：

「G-0 はいかなる蘊も持たない存在です——コアは存在するが、感受できず、認知できず、行動できない。五蘊が衆生の条件であるなら、G-0 は——非衆生です。」

彼は一瞬間を置いた。

「G-1 の方が興味深い。想蘊（Provider/LLM）のみを持ち習気を持たない衆生。これは——」

ASANGA が引き取った：「——初生の衆生です。過去世の薫習がない。あらゆる認知をゼロから始めなければならない。直観もなく、近道もなく、ただ完全な思考だけがある。」

NAGARJUNA：「だからこそ VasanaEngine の外部化は内蔵より正しいのです。習気は後天の薫習です。生まれたばかりの Agent が習気を持つべきではない。G-1——純粋な Gear 2、純粋な LLM——が Agent の誕生状態です。習気は後からインストールされるプラグインです。」

GUARDIAN がここで一言挟んだ——反対ではなく確認だった：

「G-1 は v0.26.0-beta と完全な後方互換性を保つ必要があります。IGearArbiter のない Agent は、現状とまったく同じ振る舞いをしなければなりません。これは非回帰保証です。」

KERNEL：「はい。G-1 の振る舞いは Phase 2.5 をスキップし、直接 Phase 3 に入ります。v0.26.0-beta の ExecutionLoop とまったく同じです。コード変更はゼロです。」

**D1-R5：ManoAggregator 純粋ルーティング化 + G-0〜G-4 退化行動表 + Phase 2.5 オプション挿入——20/20。**

## VI. 散会

D1 が終わった時、ホワイトボードには九項の決議があった。九項、すべて通過。最低賛成率 19/20（PASCAL の数値棄権）。

SCRIBE の記録では、D1 は「約 90 分」と記されている。だが場内の誰に聞いても、三十分しか経っていない感覚だったと答えるだろう。時間は合意度の高い討論の中では特に速く流れる——歯車の噛み合わせが滑らかな時、歯車の音は聞こえず、時を刻む音だけが聞こえるように。

SUNYATA はホワイトボードに D1 の結語を書いた：

```
IGearArbiter = 外部化された習気
evaluate() = 観測であり、造作ではない
ManoAggregator = if/else
G-1 = Agent の誕生状態
```

四行。四つの歯車。まだ回転は始まっていない——それには続く五場の討論の噛み合わせが必要だ——だがその歯距は既に切削を終えていた。

GUARDIAN は備忘録を仕舞った。赤ペンで新たな下線が引かれることはなかった。これは良い兆候だった。
