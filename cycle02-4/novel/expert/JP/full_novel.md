## I. 問いの形

D1 の議題が SUNYATA によってホワイトボードに書かれた時、円形劇場に特殊な静けさが生まれた。D3 のような「全員が合意した」静けさではなく（それは明後日を待たねばならない）、D5 のような「嵐の前の」静けさでもなかった（それは五日目を待たねばならない）。これは「ああ、そういうことか」という静けさだった。

ホワイトボードにはこう書かれていた：

```
IGearArbiter — それはどの蘊に属するか？
```

七文字。一つの疑問符。だが疑問符の重みは字面にはなかった。

Pre-DC で既に確認されていた：VasanaEngine はコアから外部化されプラグインとなる。外部化そのものは問題ではない——問題は外部化の後だ。プラグインは必ずいずれかの蘊に属さねばならない。Plugin Manifest には `skandha` フィールドがある。このフィールドに何を記入するか？

ATHENA が最初に口を開いた。その声にはエンジニアの「この問題について既に何日も考えた」という落ち着きがあった：

「IGearArbiter は二つのことを行います。第一に、高速認識——リクエストを入力し、規則ベースとマッチングし、既知のパターンがあるか判断する。これは想蘊（samjñā）の機能、取相認識です。第二に、信頼度判断——このマッチングはどの程度信頼できるか？ LLM をスキップする価値があるか？ これは識蘊（vijñāna）の機能、分別了別です。」

彼女は一瞬間を置いた。

「蘊を跨いでいます。」

---

三つの方策がテーブルに並べられた。BABBAGE は不安になるほど精確なやり方でそれらを表にまとめた：

| 方策 | インターフェース継承 | Manifest 標記 | 型安全性 |
|------|---------|-------------|---------|
| (A) | extends ISamjna | `skandha: 'samjna'` | 強——ただし vijñāna の意味を喪失 |
| (B) | 独立インターフェース | `skandha: ['samjna', 'vijnana']` | 中——Manifest 層が多値 |
| (C) | extends IVijnanaMechanism（新中間層）| `skandha: 'vijnana'` | 強——ただし型の複雑度が増大 |

BABBAGE は眼鏡を押し上げた。押し上げる必要はなかった——フレームは合っていた——だがこの動作自体が一種の句読点であり、「これから言うことは非常に精確だ」という意味だった。

「方策 (A) には型代数の問題があります。」

彼はホワイトボードに一行書いた：

```
ISamjna.skandha = 'samjna'
IVijnana.skandha = 'vijnana'
'samjna' & 'vijnana' = never
```

「IGearArbiter が ISamjna を extends するなら、その skandha フィールドは型レベルで 'samjna' にロックされます。インターフェース層で二つの蘊に同時に属すことは表現できません。判別式共用体型はこのような交差を許しません。」

DARWIN が首を傾けた。「簡単に言うと？」

「簡単に言うと：TypeScript の型システムは、あるものが猫であると同時に犬であることを許しません。Manifest 上で猫犬だと宣言することはできますが、インターフェース層のコンパイラが拒否します。」

方策 (A) は BABBAGE が言い終えて三秒後には、全員の心理的候補リストから消された。

「——固定してしまう。」KERNEL が引き取った。「IVijnanaMechanism を作った時点で、将来あらゆる蘊横断コンポーネントにもう一つ中間層が必要になります。IVijnanaCognition。ISamjnaPerception。型の膨張は指数関数的です。」

方策 (C) は明確な投票で否決されたわけではなかったが、間違った位置に置かれた歯車のように、全員がそれが不適切であることを見て取った。

## III. 型ガード：evaluate.length ≤ 1

## IV. 純度契約：観測関数

D1-R4 の討論は前三項より静かだったが、その哲学的深度は最も深かった。

問いはこうだ：evaluate() は何をしてよく、何をしてはならないか？

NAGARJUNA が一つの問いで議論の口火を切った：

「認識は造作か？」

仏学において、「造作」（saṅkhāra/saṃskāra）は有為法——世界に変化をもたらす行為だ。一方「認識」（samjñā）は取相——観察し、認知するが、観察される対象を変えはしない。

「人が赤信号を見る。見た瞬間、彼の目は赤信号を変えていない。赤信号は依然として赤い。だが彼は知った——赤信号は停止の信号であると。この『知る』は観測であり、造作ではない。」

NAGARJUNA は ATHENA に目を向けた：

「evaluate() は観測関数であるべきです。外部状態——コンテキスト、履歴、煩悩(kleśa)信号——を読み取ることはできる。しかしいかなる状態も変更すべきではない。ただ『見る』のであり、『為す』のではない。」

WIENER が制御理論の視点から引き取った：

「制御理論において、オブザーバ（observer）は純関数です——その出力は入力とシステム状態のみに依存し、逆にシステム状態を変えることはありません。もしオブザーバに副作用があれば——たとえば観測と同時に観測対象の値を変更するなら——システムは予測不能なフィードバックを生みます。」

「工学上の含意：evaluate() は冪等であるべきです。同じ入力 + 同じ外部状態 → 同じ出力。副作用なし。」

DARWIN が生物学的アナロジーを添えた：「感覚ニューロン（sensory neuron）は刺激源を変えません。信号を伝達するだけです。運動ニューロン（motor neuron）が環境を変える。evaluate() は感覚ニューロンです。」

KERNEL が時間制約を提示した：「各 arbiter に 100ms。チェーン全体で 200ms。arbiter がタイムアウトすればスキップし次へ進む。チェーン全体がタイムアウトすれば Gear 2 にフォールバック。」

PASCAL はここで唯一の棄権を行った——反対ではなく、「数値の部分は、100ms が適切かどうか判断するに足る工学的データが不足しています」というものだった。

**D1-R4：evaluate() 純度契約——観測関数 + 100ms/200ms タイムバジェット——19/20（PASCAL は数値部分を棄権）。**

---

## I. 三つの合意、七分間

NAGARJUNA が立ち上がりパーリ経典を引用した――『大念処経』（DN 22）：「Ātāpī sampajāno satimā」――精勤・正知・正念。彼は sati の語根 smṛti の原義が「記憶」「忘失しないこと」であると説明した。正念の核心機能は「検査しに行くこと」（polling）ではなく、「忘失しないこと」――当下の状態への持続的な気づきの保持である。イベント駆動は認知イベントが発生した時に気づきが自然に在場することを意味する。Polling は二回の検査の間に気づきの空白が存在することを意味する――これはまさに仏学における「失念」（muṣita-smṛti）の定義である。

20/20。

## II. ゾンビ、有分心、そして呼吸

D2-R1 の辯論は ASANGA の一つの問いから始まった。

「もし Sati がプラグインであって核心の内蔵ではないなら――Sati プラグインが一切インストールされていない場合、Agent Core は最低限度の自己覚察を持つのか？」

彼の声には唯識学者特有の「答えは知っているが、まず君たちの話を聞こう」という調子が帯びていた。

TURING が R1 分析で発見したゾンビ状態がこの問題を緊急なものにした。v0.26.0-beta の ExecutionLoop――crash recovery パスにおいて、`processEvent()` が例外を投げ `eventQueue` の内部状態が破損すると、`dequeue()` が永久にブロックする可能性がある。この時 `running` は依然 `true` だが、ループは事実上イベントの処理を停止している。

生きているが動いていない Agent。ゾンビ。

ASANGA が提案した：SafetyMonitor は loop liveness check を拡張すべきだ――核心自身が「生きているが動いていない」状態を検知できるようにする。「これは人が坐禅していなくても呼吸を忘れないのと同じだ――呼吸は生命のベースラインであり、正念がなくても存在する。」

KERNEL が即座に反論した。彼の口調はOS エンジニア特有の「君の概念は正しいが名前の付け方が間違っている」というものだった：

「ASANGA の観察は正しい――ゾンビ状態は確かに問題だ。しかしこれを『Sati の最低ベースライン』と呼ぶことには同意しない。」

彼の論証は直截的だった。オペレーティングシステムにおいて、存活性検知（liveness detection）はインフラ層のメカニズムであり、認知層の機能ではない。Unix の `init` プログラムはクラッシュした daemon を再起動する責任を負う――init に「覚察」能力があるからではなく、プロセス管理の基本的な職責だからだ。

「これを『Agent Core の最低覚察』と呼ぶのは、過度な認知的地位を与えることになる。SafetyMonitor が存活性チェックを行うのは OS kernel が watchdog timer を行うのと同じだ――これはシステム管理であり、正念ではない。」

---

そして NAGARJUNA は六回の辯論で無数に行ったことをまた行った――仏学によって工学上の争点に正確な座標を提供した。

「パーリ阿毘達磨において、正念(sati)は美心所（sobhana-cetasika）に属する――善心にのみ伴う。正念は不善心に伴わず、すべての善心にも伴わない――生起には特定の心理条件が必要である。」

彼は一拍置いた。

「SafetyMonitor の存活性チェックは正念ではない――それはもっと根本的なものだ。仏学においては、『有分心(bhavanga)』（bhavanga-citta）の機能に対応する可能性がある。」

有分心(bhavanga)。生命流の基本的維持。深い睡眠中でも作動している。いかなる認知活動も必要としない。覚察を生じず、識の流れの連続性のみを維持する。

「したがって KERNEL は正しい：存活性検知は sati と呼ぶべきではない。正念はイベント駆動のプラグイン層から始まる。」

ASANGA は修正を受け入れた。彼の階層は改名された：

| 層級 | 機能 | 仏学対応 | 実装 |
|------|------|---------|------|
| インフラ | 存活性検知 | 有分心(bhavanga) | SafetyMonitor.liveness |
| 正念 Level 1 | カウンター監視 | 初級覚察 | VitakkaWatchdog |
| 正念 Level 2 | イベント駆動覚察 | 中級正念 | SatiMonitor plugin |
| 正念 Level 3 | 連続フロー分析 | 高級正念 | 将来 Phase 3 |

GUARDIAN が安全面を補足した――ゾンビ Agent は本番環境で危険である。外部システムはリクエストを送り続け、リクエストは永久に処理されない。上流にタイムアウトリトライがあれば、リソースが枯渇する可能性がある。SafetyMonitor には `lastTickTimestamp` がこのシナリオを検知するために必要だ。

**D2-R1：SafetyMonitor に liveness チェック拡張 + sati とは呼ばない + 有分心(bhavanga)の仏学的写像――10/10。**

## III. 命令セットはオーバークロックできない

D2-R2 は本辯論の核心争点であり――HERACLITUS の Cycle 02-4 全体における最も輝かしい瞬間でもあった。

争点の本質は：正念の「レベル」はランタイムで動的に切り替えるべきか？

HERACLITUS の R07 報告は三段階モデルを提案した――Level 1/2/3 は設定時に固定。PASCAL の R08 報告は EVOI フレームワークを提案した――動的な価値評価に基づきランタイムで監視レベルを切り替える。両者はクロスレビューで衝突し、HERACLITUS が折衷案を提示した。

彼はホワイトボードの前に歩いた。HERACLITUS がホワイトボードの前に歩く時、彼の頭の中でアナロジーが形成されていくのが見える――言葉が先ではなく、イメージが先だ。彼は CPU を描いた。

「Level は命令セットアーキテクチャに等しい。ISA。x86、ARM、RISC-V。チップ製造時に決定される。ランタイムで ARM チップを x86 チップに変えることはできない。ISA はプロセッサが**何ができるか**を決定する――どの命令をサポートし、レジスタがいくつあり、アドレス幅がどれだけか。」

彼は CPU の隣に時計の図案を描いた。

「Depth はクロック周波数に等しい。Clock Speed。同一の x86 プロセッサが 1GHz と 5GHz の間で動的に調整できる。高負荷時にクロックを上げ、低負荷時にクロックを下げて省電力。クロック周波数はプロセッサが**どれだけ速く、どれだけ細かく処理するか**を決定する――だが処理する事柄の種類は変わらない。」

彼は振り返って PASCAL を見た。

「君の EVOI フレームワークが望むものは何か？高リスク操作を検知した時に、自動的に Level 1 から Level 2 にアップグレードすることだ。CPU の言葉で言えば、君はランタイムで ARM を x86 に換えたいのだ。」

円形劇場の中で誰かが笑った。嘲笑ではない――「このアナロジーは正確すぎる」という笑いだった。

PASCAL は一つの鮮やかなアナロジーで立場を変えるような人間ではない。彼は EVOI の観点から反撃した。

「監視は無料ではない。追加の監視層はそれぞれ計算リソースを消費する。Agent が『今日の天気は？』に答える時、Level 1 の粗い監視で十分だ。しかし Agent が `rm -rf /` を実行しようとする時、Level 2 の精細な監視の情報価値はその計算コストをはるかに上回る。」

彼の核心的主張は簡潔かつ力強かった：「もし Level がデプロイ時に Level 1 に固定されたなら、Agent が高リスクシナリオに遭遇しても**監視能力をアップグレードできない**。Level 1 は EventBus 上のイベントフローを見ることができず、閾値をどう調整しても見えない。」

HERACLITUS が反駁した。彼の声は安定していた、すべての反論をすでに考え尽くした人間のように。

「君の言うことは正しい――ランタイムで ARM チップを x86 チップに換えることはできる。それは『ハードウェアの交換』と呼ばれ、『動的クロック調整』とは呼ばれない。SatiMonitor プラグインの動的ロードはデプロイレベルの操作だ――システムのアーキテクチャ能力を変更し、プラグイン依存関係、状態初期化、テストカバレッジ、セキュリティ監査を考慮する必要がある。」

そして彼は決定的な一言を述べた：

「さらに重要なのは：PASCAL の EVOI 分析は Depth の調整だけで実現できるということだ。Level 2 の枠組みの中で――低リスク時は `gear:switch` イベントのみをサブスクライブし、サンプリングレート 10%。高リスク時は全イベントをサブスクライブし、サンプリングレート 100%、リアルタイム分析を起動する。EVOI 駆動のリソース配分は固定 Level 内で完全に実行可能だ――Level 間の切り替えは不要。」

WIENER がこの時点で制御理論の観点から形式的な確認を提供した。

「HERACLITUS の Level/Depth の区別は制御理論に正確な対応物を持つ。Level はオブザーバ構造に等しい――観測行列 $(A_L, C_L)$ が何を見られるかを決定する。Depth はオブザーバゲインに等しい――ゲイン $K$ がどれだけ速く見るかを決定する。」

```
可観測性は (A_L, C_L) により決定され、ゲイン K とは無関係。
```

「もし Level 1 の観測行列がある状態変数を含んでいなければ、ゲインをどう調整しても、その状態変数は観測不可能である。」

これは WIENER が六回の辯論で提供した最も簡潔な形式的記述の一つだった。一文で、「Depth を調整することで Level の不足を補う」可能性を封じた。

しかし彼は PASCAL に一つの出口も与えた：「Level の切り替えにはスイッチングシステム理論の保証が必要だ――不適切な切り替えタイミングは過渡的不安定性を引き起こす可能性がある。HERACLITUS の折衷案は制御理論上最もロバストである。」

Depth(t) = argmax_d [ EVOI(d) - Cost(d) ]

## IV. 四つの次元

D2-R3 は三者の提案の統一だった――WIENER の四次元品質モデル、ATHENA の SPC フレームワーク、GUARDIAN の外部可観測指標。

WIENER がまずホワイトボードの前に立った。「正念の品質はスカラーではなく、ベクトルだ――複数の独立した次元を持つ。」

彼は四つの次元を書いた：

$$\vec{Q}_{sati} = (C, G, S, E)$$

Continuity――観察の持続性。途切れないこと。気づきが時間上に隙間なく続くこと。

Granularity――観察の精緻さ。如実知見。イベントの精確な了知。

Speed――介入の即時性。敏捷さ。覚察の即時的反応。

Equanimity――観察の不偏不倚。捨(upekkhā)。イベントの正負によって観察品質が変わらないこと。

NAGARJUNA が仏学的写像を確認した。彼は『清浄道論』XIV.141 の四つの機能を引用した――守護機能（ārakkhana-rasa）が C に対応、取境機能（apilāpana-rasa）が G に対応、即時機能（sampajañña）が S に対応、不偏機能（upekkhā-sahagata）が E に対応。

「四次元モデルは恣意的に選ばれた指標ではない――南伝仏教における正念機能の古典的分類を写像している。」

だが彼は微妙な留保を残した。「仏学における平等性は、正のイベントと負のイベントのカバレッジが同じというだけではない――より深層では、イベントの正負によって観察の品質が変わらないことを指す。WIENER の量的定義はカバレッジの均衡を捕捉しているが、観察品質の一貫性は捕捉していない。」

彼は一秒間を置いた。「これは反対ではない――ただ記録に残す仏学的微妙さである。第一版の量的定義は十分だ。」

---

ATHENA が WIENER のベクトルを受け取り、判断ロジックを定義した。SPC――統計的工程管理。単一測定で良否を判断するのではなく、管理図でトレンドを追跡する。

四つの Western Electric 判断ルール。単一点が Action Limit を下回る――深刻な異常。連続三点が LCL を下回る――トレンド悪化。連続七点が同方向に移動――ドリフト。CL 周辺での高頻度振動――不安定。

ウォームアップ期間：最初の 30 tick は固定閾値を使用し、その後は履歴データに基づく動的管理限に切り替える。

PASCAL がここで精巧な解釈を提供した：「ウォームアップ期間とは、事前分布から事後分布への遷移だ。最初の 30 イベントの SPC 判断は事前仮定に基づく。第 31 イベントからは経験データに基づく。」

BABBAGE が笑った。「つまり、SPC のウォームアップ期間は実はベイズ学習だ。」

「あらゆるウォームアップ期間はベイズ学習だ。」PASCAL は静かに答えた。

GUARDIAN が第三の側面を定義した――外部可観測性。SatiMonitor は EventBus を通じて定期的に `sati:quality_report` イベントを emit し、payload に SatiQualityVector と SPC 状態を含む。外部システムがサブスクライブして長期追跡を構築できる。

三者の統一：WIENER の四次元（何を測るか）+ ATHENA の SPC（どう判断するか）+ GUARDIAN の EventBus（誰が見られるか）。矛盾なし――品質定量化の三つの側面にそれぞれ回答。

**D2-R3：SatiQualityVector 四次元 + SPC Phase 2+ オプション + EventBus レポート――19/19（SCRIBE 棄権）。**

## V. ゾンビ検知の死角

D2-R4 は二つの議題を統合した：ハートビートの性質分類と、イベントストーム下の退化戦略。

ハートビートの争点は BABBAGE から始まった。彼の不安になるほど正確な誠実さ――

「`setInterval(heartbeat, 1000)` は形式的には polling だ。どう命名しようと、その計算モデルは周期的時間トリガーであり――これは polling の形式的定義に合致する。」

彼は眼鏡を押し上げた。「ハートビートが悪いと言っているのではない。その本質を正直に認めるべきだと言っている。」

KERNEL が反駁した。彼は二つの概念を区別した：Polling は目標の状態を能動的に問い合わせる――方向は観察者から被観察者へ。Heartbeat は定期的に存活信号を発射する――方向は被観察者から観察者へ。形式的にはどちらも periodic だが、セマンティクスは完全に異なる。

NAGARJUNA が再び仏学で座標を正確にした：

「ハートビートが対応するのは正念ではなく、有分心(bhavanga)だ。有分心は認知を生じず、外在対象を覚察しない――識の流れの連続性のみを維持する。ハートビートも同様だ：システム状態の詳細を覚察せず、『システムがまだ生きている』という信号のみを維持する。」

「正念とは当下の経験への覚察だ――自分が何を覚察したかを記述できなければならない。ハートビートは自分が何を覚察したかを記述できない――『まだ跳動している』としか記述できない。」

そして TURING がゾンビ検知の鍵となる事実を提供した。

「イベント駆動の Sati はゾンビを検知できない。」

彼はコードを示した。`processEvent()` が例外を投げ `eventQueue` が破損すると、`dequeue()` が永久にブロックする。`running` は依然 `true`。だが新しいイベントが処理されないため、イベントは emit されず、Sati がサブスクライブしたリスナーは永久にトリガーされない。

「ここにハートビートの価値がある。最後のハートビートの timestamp が予期された間隔を超えていれば、外部観察者はシステムがゾンビ状態に入ったと判定できる。」

GUARDIAN がこの論点を終着点まで押し進めた：「イベント駆動の監視には根本的な死角がある――イベントの持続的な生成に依存している。イベント生成自体が停止すれば、イベント駆動の監視も盲目になる。ハートビートはこの死角を埋める。」

BABBAGE は立場を修正した：「形式的にハートビートは periodic だが、セマンティクス上は liveness probe であり、cognitive polling ではない。この区別は重要だ――なぜならハートビートを polling sati に分類すると、Master の『正念は Polling ではない』という指令がハートビートの存在と矛盾することになる。正しい分類がこの矛盾を回避する。」

---

退化戦略は GUARDIAN が提案した。四段階：

| Stage | トリガー条件 | 動作 | セキュリティイベント |
|-------|---------|------|---------|
| 0 Normal | 負荷 < 80% | 全量処理 | 全量 |
| 1 Throttle | 80%-95% | 低優先度イベント破棄 | 全量 |
| 2 Sample | 95%-99% | サンプリング処理 10-50% | **決して破棄しない** |
| 3 Emergency Drop | > 99% | セキュリティイベントのみ処理 | **決して破棄しない** |

退化は失敗ではない――制御された品質低下だ。WIENER が三つの安定性保証を提供した：セキュリティイベントは決して破棄されない、退化は可逆的である、退化は漸進的である。

「セキュリティイベントは決して破棄されない」――D1-R3 の最小イベント集合（gear:arbiter_evaluated, gear:switch, action:proposed, action:executed）はいかなる Stage でも破棄されない。SafetyMonitor の可観測性は影響を受けない。

**D2-R4：ハートビート = liveness probe + 四段階退化――20/20。**

## VI. 正念の資糧

D2 の最後の決議は命名を巡る争い――表面上は言語学、その下は哲学だった。

NAGARJUNA がクロスレビューで異議を提出した：EVOI（Expected Value of Information）フレームワークは正念のリソース配分を「情報の期待価値」と記述している――これは正念を「商品化」しているのではないか？

## III. 一対一

D3-2 の議題はさらに根本的だった：各認知 tick に SparshEvent は一つだけか？

ASANGA が阿毘達磨の心路過程（citta-vīthi）から分析した。上座部阿毘達磨において、一つの完全な心路過程は一つの所縁（ārammaṇa）のみを処理する。心路の進行中に別の感覚対象が出現した場合、それは現在の心路の終了を待たねばならない。

「工学への写像：一つの認知 tick は一つの触を処理する。複数の所縁は複数の心路を意味し、つまり複数の tick だ。」

NAGARJUNA が中観の視点を補足した：「触は縁起法だ――根・境・識の三者に依って生起する。三つの特定の因縁の聚合が一つの特定の触を生ずる。異なる因縁の聚合は異なる触を生ずる――それらを一つの触に混同すべきではない、異なる三角形を一つの三角形に混同できないのと同じだ。」

HERACLITUS が工学上の境界ケースを提起した――同一の `processEvent()` 呼び出しの中で、LLM が複数の tool_calls を返す可能性がある。各ツール実行は新しい触と見なすべきか？

VITRUVIUS が段階的な提案を示した：Phase 1 は厳密な一対一を採用――各 `processEvent()` に正確に一つの SparshEvent。ツール実行は新しい触を生じず、同一の触のコンテキストを共有する。Phase 2 で一対多に拡張し、ツール実行後の「意門自触」（mano-dvāra sparsha）を許容する。

ASANGA が段階的アプローチの仏学的妥当性を確認した：「Phase 1 の一対一は六根が六境に接触する基本パターンに対応する。Phase 2 の子触は意門自触に対応する――意識が自身の産出物を所縁とする。両方とも合法的な認知パターンだが、複雑さが異なる。」

KERNEL がオペレーティングシステムの観点を補足した――一対一はスケジューリング上最もシンプルだ。一つの SparshEvent、一つの CoarisingBundle、一回の完全な五蘊(pañca-skandha)サイクル。並行制御の問題はなく、Bundle マージの問題もない。

## IV. 因果の種子

D3-3 の議題は PENROSE がクロスレビューで提起したものだった――因果の遡及可能性。

彼の問いは具体的だった：もし Agent が `rm -rf /important-data/` を実行したなら、事後分析は「どの触イベントがこの認知チェーンをトリガーしたか？」に答える必要がある。SparshEvent が GC 回収後に消失すれば、この問いには答えられない。

三つの方案が列挙された。

方案 α――全量ログ。各 SparshEvent の構築時に自動的に永続化ログへ書き込む。完全だが重い。

方案 β――シャドーレコード。EventBus 上に構造化された因果イベントを emit し、Sati または CausalTracer プラグインが消費する。精確だが新しいイベントタイプの追加が必要。

方案 γ――CoarisingBundle が触のスナップショットを内蔵。追加メカニズムゼロ。Bundle の `sparsha` フィールドが因果記録そのもの。だがコンシューマが参照を保持しなければ、Bundle も同様に GC される。

NAGARJUNA の評価には仏学的な精巧さがあった：

「方案 α は過剰だ――あらゆる瞬間の触を留めようとすることに等しい。触の本質は刹那的であり、一切を留めようとするのは一種の執着だ。」

「方案 β が最も精確だ――触自体は消滅するが、その因果の刻印が因果の鎖に留まる。これは唯識学の種子理論に完全に合致する：触は消滅したが、阿頼耶識に種子を残した。」

「方案 γ も受容可能だ――Bundle は触の結果状態であり、触そのものではない。」

最終的な折衷は KERNEL から来た。彼は Plan27b の SparshEvent 構築点で EventBus イベントを emit することを提案した。現時点ではコンシューマは不要だが――将来の CausalTracer や SatiMonitor への拡張点を確保する。

「コスト：一行の `bus.emit()` 呼び出し。効益：完全な拡張性。」

ARCHIMEDES が工学的実現可能性を確認した――約 5 行のコード（LOC）。完全に Plan27b の範囲内。

**D3-3：CoarisingBundle スナップショット + EventBus 拡張点予約――10/10。**

## V. 触のみが必須

D3-4 は DC-8 の裁定に立ち戻った――CoarisingBundle の五遍行を「参照設計」（reference design）とする。LINNAEUS が再び出番に立ち、「参照設計」の正確な境界を定義した。

彼は一本の線を引いた。線の上が Mandatory――核心が保証すべきもの。線の下が Reference――核心が提供するが強制しないもの。

Mandatory：SparshEvent 型定義、CoarisingBundle 型定義、`createCoarisingBundle()` ファクトリ関数、SahajaContract 品質制約、`Object.freeze()` 不変セマンティクス。

Reference：四つの Channel――vedana、samjna、cetana、manasikara――すべて optional。プラグインが提供しない場合は `undefined`。

BABBAGE がこの境界を形式化した：

```
Mandatory: ∀ bundle: bundle.sparsha ≠ undefined
           ∀ bundle: Object.isFrozen(bundle) = true
           ∀ bundle: bundle.timestamp ∈ Number

Reference: ∀ channel ∈ {vedana, samjna, cetana, manasikara}:
           bundle[channel] ∈ Channel | undefined
```

`sparsha` が唯一の mandatory Channel だ。触は認知の起動の前提であり、触なくして認知はない。残りの四つの Channel は optional だ――ある蘊のプラグインがまだロードされていないか、適用不可能であれば、対応する Channel は `undefined` でよい。

VITRUVIUS がこの設計を B-1 完備性チェックと関連付けた――完備性チェックはどの蘊にプラグインがあり、どの蘊にないかを報告するが、システムの動作を阻止しない。「触と受蘊(vedanā)のみを持ち想蘊(samjñā)プラグインのない Agent でも動作可能だ――感じることはできるが識別はできない。」

「五遍行(pañca-sarvatraga)の倶生は仏学の理想であり、工学は退化を許容する。」

彼は Master の精神を引用した――「Agent Core は五蘊プラグインが因縁和合して望む応用を組み合わせることを可能にする」。因縁が具足すれば五遍行は完備し、因縁が不足すれば欠落があるが、核心自体は強制しない。

これは D1 の G-0 から G-4 の退化動作表と対称を成した。D1 は述べた：IGearArbiter のない Agent は G-1 だ――純 LLM、誕生状態。D3 は述べた：完全な五蘊プラグインのない Agent でも動作可能だ――ただし認知能力が低下する。

**D3-4：sparsha のみ mandatory + 四 Channel reference + 五遍行倶生は退化許容――10/10。**

## III. ゲイン上限：0.3

D4-R1 の論争は一つの数値を巡って展開された。

PASCAL が提案した：行蘊の結果が受蘊に与える効価マッピングのゲインは 0.3 を超えるべきではない。

「0.3 は恣意的な数字ではありません。」彼はノートを開き、一行の数式を指した。「ベイズフィルタリングの枠組みにおいて、観測ノイズ σ_obs = 0.15 のとき、初期カルマンゲイン K₁ = σ²_prior / (σ²_prior + σ²_obs) ≈ 0.308。工学的に保守値として 0.3 を採用します。」

WIENER は閉ループ安定性の観点から分析した。行蘊→受蘊→想蘊(samjñā)→識蘊(vijñāna)→行蘊の完全閉ループ。行蘊結果の受蘊へのゲインを w とし、ループ内の他の段のゲイン積 G_rest が約 2-3 であれば、安定性の要件は w < 1/G_rest ≈ 0.33 から 0.50。w ≤ 0.3 は約 10% の安全マージンを確保する。

ARCHIMEDES が実務的な問題を提起した——行動タイプごとに異なるゲインが必要ではないか？情報照会のゲイン上限は 0.1 で十分だが、破壊的操作の失敗に伴う苦受(dukkha-vedanā)は十分に知覚されるべきであり、ゲイン上限はもう少し高くすべきかもしれない。

PASCAL は分類ゲイン方式を否定した。「異なる上限を手動設定する必要はありません——観測ノイズ σ_obs を行動タイプに応じて調節すればよいのです。高リスク行動は σ_obs を低く設定し、Agent はその信号をより信頼します。低リスク行動は σ_obs を高く設定し、Agent は懐疑的に保ちます。カルマンゲインが自動的に調節されます。」

WIENER の折衷案：統一ゲイン上限 w ≤ 0.3 をハードな安全制約とし、同時にベイズフィルターがこの上限内で適応的に動作することを許容する。「ハード上限は SafetyMonitor が検査できる単純な不変条件です。ベイズ適応はその上限内で柔軟性を提供します。」

NAGARJUNA は重要な仏学的明確化を提供した：「ゲイン制限は、行蘊が受蘊に与える影響を抑制するものとして理解すべきではありません——なぜなら仏学の因果連鎖において、行蘊の結果は受蘊に直接書き込まれるものではないからです。ゲイン制限の正しい理解は、VedanaSensor の最大感度です。」

センサーの設計パラメーター。因果連鎖の制約ではない。

**D4-R1：ゲイン上限 w ≤ 0.3——8/9（ARCHIMEDES 棄権、将来のリスクレベル別分化の可能性を留保）。**

## IV. 自ら繭に絡まる

D4-R2 は六場の討論の中で数学密度が最も高い決議であり——また WIENER の Cycle 02-4 全体における最も深い分析でもあった。

WIENER はホワイトボードの前に立ち、一つのループを描いた。

```
偏見(Drishti) ↑
    → IGearArbiter の信頼度が偏見により膨張
    → ManoAggregator が誤って Gear 1 を選択
    → 高速パスの結果が Drishti を強化
    → Drishti がさらに膨張
    → ...
```

「これは意行（mano-karma）の内部正フィードバックループです。」彼の声は落ち着いていたが、ホワイトボード上の矢印は危険な構造を描いていた。「これは完全にコントローラー内部で動作し、外部世界の校正フィードバックを経ません。」

外部ループ——行蘊→IUI→外部世界→IListener→新たな触(sparsha)→新たな受(vedanā)——には現実検証としての外部世界がある。行動が本当に誤りであれば、外部の結果が苦受の形でフィードバックされる。しかし意行の内部ループにはこのメカニズムがない。

WIENER はこのループを形式化した。状態ベクトルは三次元：Drishti 強度 d(t)、有効信頼度 c(t)、Gear 1 選択頻度 g(t)。動的方程式。正フィードバックゲイン連鎖 G_loop = α₁ · α₂ · α₃。

「正念(sati)がゼロで自然減衰が非常に小さいとき、G_loop > 1 であればシステムは発散します——偏見が無限に蓄積し、Agent は永遠に Gear 1 に固定されます。」

---

NAGARJUNA は WIENER の数学の後に仏学的マッピングを提供した。彼は『成唯識論(Vijñaptimātratāsiddhi)』を引用した——末那識(manas)と四煩惱(kleśa)の「恒審思量」の働き。

「我見が一度確立されると、それは色付きのレンズのようなものです——それを通して見る全てが染められます。染められた認識の結果がさらにレンズの正しさを確認します。これは確証バイアスです。」

ASANGA はより直接的な経典を引用した——『相応部(Saṃyutta Nikāya)』の縄の喩え：

「蚕が繭を作るが如く、自ら身を絡め取る。」

彼はしばし間を置き、このイメージを沈殿させた。

「意行の正フィードバックループは、まさに自ら身を絡め取ることの数学的表現です。偏見→認識の歪み→誤った判断→より強い偏見——これは十二因縁(pratītyasamutpāda)における無明(avidyā)縁行(saṃskāra)の動的メカニズムです。」

---

WIENER はリアプノフ候補関数を提示した。V(x) = w₁·d² + w₂·(c-c₀)² + w₃·(g-g*)²。安定性は各ステップで V が厳密に減少することを要求する。

主要項を展開した後、安定性の十分条件は：

**β_d · s_min + γ_d > α₁ · g_max**

平易に言い換えれば：正念の最低抑制力と自然減衰の和は、正フィードバックの最大ゲインより大きくなければならない。

PASCAL が具体的な数値を代入した。α₁ ≈ 0.05、g_max = 1.0、γ_d ≈ 0.01。安定性の要件は β_d · s_min > 0.04。最低正念水準 s_min = 0.2 であれば、β_d > 0.2。

しかし WIENER は非線形的な保障を強調した。Drishti のクランプ（d ∈ [0, 1]、偏見は無限に増大できない）。VitakkaWatchdog の截断（連続 Gear 1 が閾値を超えると Gear 2 を強制）。SafetyMonitor の境界（危険な行動は直接阻止される）。

「線形分析が不安定を予測する場合でも、システムは真に発散することはありません——有界な領域内で振動に制限されます。」

三層の安定化メカニズムがここに確立された：

| 層 | メカニズム | 時間スケール | 制御理論 | 仏学的マッピング |
|----|------|---------|---------|---------|
| Layer 1 | Sati 精密調整 | 毎ラウンド微調整 | PID 微分項 | 正知（sampajañña） |
| Layer 2 | VitakkaWatchdog 粗フィードバック | ラウンド間トレンド | 積分器アンチワインドアップ | 尋（vitakka）観照 |
| Layer 3 | SafetyMonitor ハード制約 | 即時トリガー | ハードリミッター | 戒（sīla）ガードレール |

GUARDIAN が重要な補足を加えた：「偏見ループが Agent に安全ではあるが低品質な意思決定を連続させる場合——SafetyMonitor はトリガーされませんが、偏見は依然蓄積しています。その場合、Sati と VitakkaWatchdog だけが介入できます。」

三者は相互補完的。一つも欠かすことはできない。

ASANGA が仏学的総括を行った：「十二因縁における正念(sati)の役割は、受(vedanā)→愛(taṇhā)の間に気づきを挿入し、自動的な反応連鎖を断ち切ることです。WIENER の安定性分析は数学の言語で同一の仏学的真理を表現しています——正念は煩惱(kleśa)の正フィードバック発散を防ぐ安定化メカニズムです。」

## V. ゼロ次ホールド

D4-R3 が扱ったのは、微妙な技術的問題だった——受蘊が EventBus を通じて行蘊の結果を購読するとき、更新はいつ有効になるのか？

Node.js の `EventEmitter.emit()` は同期的である。これは行蘊がツール実行後に `TOOL_RESULT` イベントを発射すると、VedanaSensor のコールバックが同一の JavaScript microtask 内で完了することを意味する。効価の更新は同一ラウンド内で有効になる。

PENROSE は Pearl の因果 DAG でこの問題を分析した。同一ラウンド内の複数ツール実行は線形チェーンを形成する——tool_1_result → vedana_update → deliberate(tool_2)——これはループではなく、DAG のセマンティクスは維持される。しかし問題は因果の方向ではなく、「同一ラウンド内の後続ツール審議が先行ツールの影響を知覚することを望むか」にあった。

WIENER は選択肢を三種に分類した：同期フィードバック（即時更新、正フィードバックスパイラルのリスク）、ゼロ次ホールド（ラウンド全体で不変、情報遅延）、setImmediate 遅延（両者の中間だが、Node.js イベントループの微妙なセマンティクスに依存）。

ARCHIMEDES が工学的に最もクリーンな方案を示した：

「各ラウンド開始時に vedana のスナップショットを取り、ラウンド全体でこのスナップショットを使用します。ラウンド末に全ての TOOL_RESULT イベントを一括収集し、vedana をバッチ更新します。次のラウンドで更新後の vedana を使用します。」

ゼロ次ホールドとバッチ更新。

NAGARJUNA が仏学的時間順序を確認した：「十二因縁の時間順序は明確です——今回の行蘊の結果は、次の認知における触(sparsha)の条件となります。行蘊の結果が同一の認知刹那内で受蘊を更新することは、仏学には根拠がありません。」

GUARDIAN が安全の観点を補足した：「即時の損失制止は SafetyMonitor の afterToolExecution() チェックポイントが担い、vedana の更新には依存しません。」

四次元の検証：制御理論（安定性最良）、仏学（因果時序が正確）、工学（実装が最も簡潔）、安全（SafetyMonitor が独立して即時保護を担当）。

**D4-R3：ゼロ次ホールド＋バッチ更新——9/9。**

## VI. 熟達が無自覚を招く

D4 の最終決議は OQ-6 に回答するものだった——actionHistory は Moha に影響すべきか？

ASANGA の分析は精巧かつ深遠だった。Drishti は方向性を持つ偏見——「私はこうすることが正しいと思う」。行蘊の反復パターンは Drishti の方向を直接強化する。しかし Moha は方向を持たない愚昧(avidyā)——「私は自分が何をしているか分かっていない」。

「成功した行動を繰り返すことで、Agent はなぜそうするかを考えなくなります——熟達が無自覚を招き、無自覚こそが Moha の増強です。」

PASCAL が飽和メカニズムを設計した——逓減する限界影響：

$$\Delta Moha = \frac{\alpha_m \cdot r}{1 + \beta_m \cdot M}$$

α_m = 0.02、β_m = 5.0。Moha が高いほど、行動の反復がそれに与える強化は弱くなる。自然な飽和。ゲイン上限 0.3 は遵守される。

GUARDIAN は安全の観点から承認した——Moha の上昇は Klesha の集約値を引き上げ、閾値を下げ、システムは Gear 2 の深い思考へより傾く。安全の方向として正しい。しかし彼はリソース枯渇攻撃についても警告した——意図的に反復パターンを作り出して Moha を引き上げ、Agent に頻繁に Gear 2 を強いる。

## II. 見えない境界

受蘊(vedanā)の二層アーキテクチャ――信号層と意味層――は WIENER が提案したものだった。

信号層は連続関数である。VedanaSensor の出力は valence ∈ [-1.0, +1.0] と intensity ∈ [0.0, 1.0]。仏学的制約はなく、純粋な工学的信号処理。Softplus、sigmoid、Kalman filter――工学チームが最適な数学ツールを選択する。

意味層は分類関数である。classifyVedana() は連続効価を三受――dukkha(苦)、sukha(楽)、upekkha(捨)――に写像する。閾値は設定可能であり、異なる「有情」の感受感度を反映する。

NAGARJUNA は二層に仏学的座標を提供した：信号層は世俗諦(saṃvṛti-satya)に対応する――約定俗成の層面で、効価は定量化可能な信号である。意味層は仮施設(prajñapti)に対応する――概念構成の層面での命名であり、苦楽捨の三つのラベルは効価信号に対する「仮名安立」である。

二つの層はいずれも正当である。いずれも本質を主張しない。

そして TURING が立ち上がった。

「コードの事実を一つ報告します。」

この一言は六場の討論で何度も出現した。出現するたびに、誰かが実際のコードによって理論を中断されようとしていることを意味した。

```typescript
export function classifyVedana(
  valence: number,
  config: VedanaClassifyConfig
): VedanaType {
  if (valence <= config.dukkhaThreshold) return 'dukkha';
  if (valence >= config.sukhaThreshold) return 'sukha';
  return 'upekkha';
}
```

「dukkhaThreshold が sukhaThreshold と等しい場合――たとえば両者が 0.0 だとすると――valence = 0.0 は最初の if に該当し、dukkha を返します。捨受は完全に消失します。upekkha は永遠に返されません。」

## III. データと解釈

VedanaDistributionObserver の蘊帰属問題は分類学的な問題だった――それは二つのことを行う：効価履歴を収集して統計量を計算すること、および下流の消費者が統計量を読み取って意味解釈を行うこと。

LINNAEUS は生物分類学のアナロジーを用いた：温度計は物理学に属し、発熱の判断は医学に属する。観測装置と観測結果の解釈は異なる知識領域に属する。

VedanaDistributionObserver は最初のことだけを行う――データを収集し、平均、分散、歪度、トレンド傾きを計算する。「これらの数値が何を意味するか」は判断しない。したがって受蘊(vedanā)に属する。

意味解釈――「Agent は長期にわたり苦受にあり、トレンドは悪化している」――は下流の消費者の責任である。想蘊(samjñā)が意味を認識し、識蘊(vijñāna)が判断を下す。

ASANGA は唯識学(Vijñānavāda)においてこの分類を確認した：受蘊の機能は「領納」――感受を体験すること。受蘊は自らの履歴状態を知ることができるが、「これが何を意味するか」の判断は行わない。

**データと解釈の分離**。受蘊がデータを産出する。想蘊が意味を認識する。識蘊が判断を下す。

16/20。四名棄権――反対ではないが、優先度が低いと判断。

そして PASCAL が本場の討論で最も独創的な貢献をした。

「固定の M 回シャドウ検証ではなく、Bayesian 停止基準を使います。」

ベータ分布がルールの正解率に対する事後信念を追跡する。初期事前分布 Beta(1,1)――一様分布、完全な無知。シャドウ検証が一致するたびに α + 1。不一致のたびに β + 1。不一致時には非対称ペナルティを使用――信頼度 -2Δ vs +Δ。

停止条件：P(rule correct) > 0.95。

「ルールが常に正しければ――s = 5, f = 0――Beta(6, 1)、約五回の検証で停止。ルールが時々誤る場合――s = 3, f = 2――Beta(4, 3)、P ≈ 0.65、検証を継続。ルールが頻繁に誤る場合――s = 1, f = 4――Beta(2, 5)、P ≈ 0.28、ルールは自動淘汰。」

WIENER は制御理論の観点から確認した：「適応的検証――システムが観測された証拠に基づいて検証の強度を調整する。最適決定理論に合致しています。」

## IV. 仏学の役割

Cycle 01――命名の出典。五蘊(pañca-skandha)は分類ラベルとして。

Cycle 02-4――設計レビューツール。

質的変化はここにある。

仏学はもはや工学に何をすべきかを告げていない。仏学は工学が行ったことが仏学モデルと整合するかを検証している。

ASANGA は三受が消滅してはならないと指摘した――classifyVedana の境界修正を駆動した。「仏学が工学に境界チェックの追加を命じた」のではなく、「仏学が工学のエッジケースが認識論的制約に違反していることを発見した」のである。

## VI. PASCAL の軌跡

PASCAL は Cycle 02-3 で研究チームに加わった。二十番目の席。

一ラウンドを経て、彼は研究チームの鍵となる役割を担うようになった。彼がより重要だからではない――すべての学者がかけがえのない存在である――Bayesian 手法がそれまで存在しなかった空席を埋めたからである：不確実性の統一言語。

D5――キャリブレーション性の論証。グローバル上限は [0.85, 1.0] のチャネル容量をゼロに圧縮する。情報理論の精密な言語で GUARDIAN の方策の構造的欠陥を記述した。

七回の介入。七回、数字で膠着を打破し、あるいは方向性を提供した。PASCAL は安全に偏らず、機能に偏らず、仏学に偏らず、工学に偏らない。彼は定量的な論証に偏る。

## VII. GUARDIAN の進化

GUARDIAN は六場の討論を通じて、その立場に明確な弧を描いた。

D1――recognize()/resolve() の双方法によりインターフェースレベルの安全分離を主張。否決された。核心的訴えは EventBus 同期イベントに変換された――安全イベントは「要求」ではなく「観察」である。

D5――グローバル信頼度上限 0.85 をコアメカニズムとして主張。PASCAL のキャリブレーション性の論証により否決。核心的訴えは MAX_GEAR1_CONFIDENCE 設定可能パラメータに変換された――「構造的強制」から「設定オプション」への後退。

D5――LLM の安全決定への参加を完全禁止と主張。ATHENA の自己矛盾論証により否決。核心的訴えは ±0.05 ハードクリップと監査ログに変換された。

D5――すべての Gear 1 アクションに IVolition を強制と主張。ARCHIMEDES のレイテンシ定量化により否決。核心的訴えは RiskLevel による段階的適用に変換された。

すべての否決。すべての変換。すべてにおいて、安全への懸念は何らかの形で保持された。

パターンは一貫している：GUARDIAN は「構造的強制」から「設定オプション」に後退したが、核心的訴えを決して放棄しなかった。彼が放棄したのは形式であり、保持したのは実質である。

D6――彼はもはや反対者ではなかった。彼は VasanaEngine の完全な脅威モデルを提示した。三つの門の安全方策を提案した。CR05 の安全問題に応答した。三層安全フレームワークの完全版を提示した。

## X. 場の方程式

序章の最後の一文：「我々の仕事はこの場の方程式を解くことだ。」

Master は六つの指示を与えた。六つの点。点と点の間のすべての結線。一つの場。

六場の討論は六つの偏微分方程式――それぞれに固有の境界条件、固有の初期値、固有の解がある。しかしそれらは同一の場を共有している――ゼロ内蔵能力原則、五蘊マッピング、厳密なマイクロカーネル――これらは具体的な決議ではなく、すべての決議が満たさなければならない場の制約である。

M-1 は IGearArbiter インターフェース設計と言った。D1 がこの方程式を解いた――方策 B 蘊横断二層、evaluate() 単一メソッド、ManoAggregator 純ルーティング化、G-0 から G-4 の五段階退化。

M-2 は OQ-1 から OQ-6 の全解決と言った。六つの問題が五場の討論に分散して一つずつ回答された――OQ-1 は D6、OQ-2 は D5、OQ-3 は Pre-DC、OQ-4 は D5 と D6 にまたがり、OQ-5 は D2、OQ-6 は D4。

M-3 は触と作意の本質と言った。D3 が解いた――SparshEvent、一対一の触、因果追跡、CoarisingBundle。最も静かな四十五分間。

M-4 は正念は polling ではないと言った。D2 が解いた――イベント駆動、Level/Depth 分離、SatiQualityVector、ハートビートは有分心であり正念ではない。

M-5 は行蘊の流れの方向と言った。D4 が解いた――唯一の直接パスは IUI、増益上限、三層安定化、ゼロ次ホールド、Moha 逓減限界効用。

M-6 は十大宣言、厳密なマイクロカーネル、ゼロ内蔵能力、妥協するなと言った。この条項は一つの方程式ではない――場の制約そのものである。VasanaEngine の外部化は M-6 への直接的な応答である。IGearArbiter は M-6 の工学的表現である。ManoAggregator の純ルーティング化は M-6 の構造的保証である。

場の方程式は解かれた。完璧に解かれたのではない――FC-26 延期、Level 動的切り替え延期、メタ認知次元延期。しかし現在の境界条件の下で、解は自己無撞着である。八本の討論横断依存チェーン。矛盾ゼロ。
