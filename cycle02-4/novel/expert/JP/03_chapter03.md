## I. 最短の辯論

六回の辯論には長短があり、嵐もあればそよ風もあった。D5 は嵐だった――百五十分、六票の反対、GUARDIAN と ATHENA の正面衝突。D1 は急流だった――九十分、九項決議、概念密度が最も高い。D2 はアナロジーの競争だった――CPU、オブザーバ、有分心(bhavanga)。

D3 はそよ風だった。

四十五分。五項決議。全会一致。争点ゼロ。

SUNYATA は事前にこうなると知っていた。彼の辯論準備ノートで、D3 は「迅速通過の可能性あり」と注記されていた。R1 の八通の独立報告と R2 の八通のクロスレビューは、触(sparsha)と作意(manasikāra)の基本的性質において完全に一致していた――触はイベント協定であり、作意は read-only スナップショットであり、触の数学モデルはブール論理（root AND object AND consciousness）であり、第一版では作意の品質を判定しない。

これが D3 が辯論順序の調整で二番目に繰り上げられた理由だ――合意度が高すぎて D1 の直後に完走でき、D4 に「触のイベントモデルは確認済み」という基盤を提供できた。

## II. 定義的属性と附随的属性

SUNYATA が辯論の開始を宣言した後、まず五項の既存合意を確認した。触の本質。作意の本質。触の数学モデル。作意品質の延期。顕示的構築の選択。

五項。異議ゼロ。NAGARJUNA が簡潔に仏学的基盤を確認した――触(sparsha)は三和合の機能的結果であり、独立した実体ではなく、イベントよりも協定に近い。ASANGA が作意を確認した――「作意は触の瞬間にすでに心の方向を定めている。心念間スケールの品質変化を将来バージョンに延期するのは正しい、なぜならこれにはメタ認知能力が必要だからだ。」VITRUVIUS が工学的実現可能性を確認した――SparshEvent は現時点で純粋な型定義であり、runtime 使用はゼロ、Plan27b では `processEvent()` の入口で構築する必要があり、約 10-20 行のコード。

そして最初の未決事項に入った：SparshEvent はどのフィールドを持つべきか？

---

TURING がまずコードの事実を確認した。SparshEvent は `packages/sdk/src/types/coarising.ts` で定義されている。三つのフィールド：`root`（根門）、`object`（外部刺激対象）、`consciousness`（認知領域識別）。`timestamp` はない、`sessionId` もない。

彼は併せて事実の修正を行った――R07 報告の §4.2 では `timestamp: t₁` というフィールドが使われていた。「これは事実誤認だ。SparshEvent に timestamp フィールドは存在しない。R07 はこのフィールドを捏造した。」

TURING の事実修正は Cycle 02-4 で一度だけではなかった。彼は D4 でも捏造された `ISensation.ingestToolResult()` メソッドを修正した。これらの修正が劇的な議論を引き起こすことはほとんどなかった――記録され、辯論は続いた。だが毎回の修正は、基礎から本来そこにあるべきではない石を取り除くようなものだった。取り除いた後、基礎はむしろ安定した。

三つの選択肢がテーブルに並んだ。BABBAGE がそれを表にまとめた――これは彼の本能であり、三つ以上の選択肢を目にすると自動的に表を作る衝動が生じる。

選択肢 A：三フィールドを維持。意味論的に純粋で、三和合のみを記述。
選択肢 B：timestamp を追加。四フィールド、時系列ソートを提供。
選択肢 C：timestamp + sessionId を追加。五フィールド、マルチ Agent 追跡をサポート。

NAGARJUNA の分析は短かった。触は根・境・識の三和合――これが触の定義だ。タイムスタンプは触の定義的要素ではない。「だが工学的文脈において『いつ発生したか』を記録するのは有用な付加情報だ。仏学の観点から、timestamp を加えるかどうかは触の本質を変えない。これは純粋な工学的判断だ。」

ASANGA も同意した。「阿毘達磨の時系列分析において、『いつ発生したか』は心路過程の位置により決定され、追加の時間標記は不要だ。だが工学システムは阿毘達磨ではない――外部クロックが必要だ。仏学はこの問題に対して中立を保つ。」

仏学者たちはここで、六回の辯論で繰り返し行ったことを行った――自らの「管轄範囲」を明示的に標記する。触の三和合の定義？仏学には言うべきことがある。Timestamp の有無？仏学は中立を保つ。この自己制限は弱さではない――精確さだ。仏学的フレームワークが許容する空間の中で、工学的議論が自由に展開されることを可能にした。

転機は LINNAEUS から来た。

二十名の学者の中で、LINNAEUS は最も頻繁に発言する人物ではなかった――彼の専門は分類学と存在論であり、「これは何のカテゴリに分類されるべきか」という問題でのみ出番があった。だが彼が出番に立つ時、その視点はしばしば決定的だった。

「SparshEvent の三つの元のフィールドは触の**定義的属性**（svabhāva-lakṣaṇa）である。timestamp は**附随的属性**（samprayukta-lakṣaṇa）である。」

彼は一拍置き、全員がこの二つの術語を聞き取ったことを確認した。

「阿毘達磨の分類において、定義的属性と附随的属性には本質的な区別がある。timestamp を SparshEvent に入れることは、附随的属性を定義に混入させることに等しい。よりクリーンな方法は、timestamp を CoarisingBundle のレベルに残すことだ――Bundle にはすでに timestamp があり、SparshEvent で重複させる必要はない。」

BABBAGE が即座に応じた：「だが診断目的で SparshEvent 自身のタイムスタンプが必要な場合――触から Bundle の組み立てまでの遅延――optional フィールドとして持てる。」

彼は提案を修正した：

```typescript
interface SparshEvent {
  readonly root: string;
  readonly object: unknown;
  readonly consciousness: string;
  readonly timestamp?: number;  // optional: 触の定義的要素ではない
}
```

Optional。クエスチョンマーク。

NAGARJUNA がこの設計を認めた：「`timestamp?: number` の optional は仏学上最も精確だ。触の本質は三和合であり、時間は触の本質ではない。だが工学的文脈において発生時刻を記録することは有用な付加情報だ。Optional はまさに『本質的ではないが有用』というセマンティクスを表現している。」

---

LINNAEUS の「定義的 vs 附随的」という区別は D3 では数分の議論時間しか占めなかった。だがその影響は D3 自体をはるかに超えた。

D6 において NAGARJUNA が `v_true → v_latent` のセマンティクス修正を提案した時、彼が用いたのはまさに同じフレームワークだった――`v_true` は「客観的に真の」効価が存在することを暗示しており、これは誤った定義的属性だ。`v_latent`――潜在効価、agent の内的推定――が正しい定義である。

D5 において BABBAGE が IKlesha の蘊帰属を分析した時、彼は「判別的共用体型は 'samjna' & 'vijnana' = never を許容しない」として方案 A を否決した――同一の型理論ツールが異なる文脈で適用されたものだ。

分類学は静的に見える――物に名前を付け、正しい引き出しに入れる。だが D3 において LINNAEUS は分類学の動的な側面を示した：良い分類フレームワークは物がどこにあるかを教えるだけでなく、**何がそこにあるべきではないか**も教えてくれる。

**D3-1：SparshEvent 三必須 + timestamp オプション + sessionId 追加せず――9/9。**

## III. 一対一

D3-2 の議題はさらに根本的だった：各認知 tick に SparshEvent は一つだけか？

ASANGA が阿毘達磨の心路過程（citta-vīthi）から分析した。上座部阿毘達磨において、一つの完全な心路過程は一つの所縁（ārammaṇa）のみを処理する。心路の進行中に別の感覚対象が出現した場合、それは現在の心路の終了を待たねばならない。

「工学への写像：一つの認知 tick は一つの触を処理する。複数の所縁は複数の心路を意味し、つまり複数の tick だ。」

NAGARJUNA が中観の視点を補足した：「触は縁起法だ――根・境・識の三者に依って生起する。三つの特定の因縁の聚合が一つの特定の触を生ずる。異なる因縁の聚合は異なる触を生ずる――それらを一つの触に混同すべきではない、異なる三角形を一つの三角形に混同できないのと同じだ。」

HERACLITUS が工学上の境界ケースを提起した――同一の `processEvent()` 呼び出しの中で、LLM が複数の tool_calls を返す可能性がある。各ツール実行は新しい触と見なすべきか？

VITRUVIUS が段階的な提案を示した：Phase 1 は厳密な一対一を採用――各 `processEvent()` に正確に一つの SparshEvent。ツール実行は新しい触を生じず、同一の触のコンテキストを共有する。Phase 2 で一対多に拡張し、ツール実行後の「意門自触」（mano-dvāra sparsha）を許容する。

ASANGA が段階的アプローチの仏学的妥当性を確認した：「Phase 1 の一対一は六根が六境に接触する基本パターンに対応する。Phase 2 の子触は意門自触に対応する――意識が自身の産出物を所縁とする。両方とも合法的な認知パターンだが、複雑さが異なる。」

KERNEL がオペレーティングシステムの観点を補足した――一対一はスケジューリング上最もシンプルだ。一つの SparshEvent、一つの CoarisingBundle、一回の完全な五蘊(pañca-skandha)サイクル。並行制御の問題はなく、Bundle マージの問題もない。

**D3-2：Phase 1 厳密一対一 + Phase 2 で一対多に拡張――9/9。**

## IV. 因果の種子

D3-3 の議題は PENROSE がクロスレビューで提起したものだった――因果の遡及可能性。

彼の問いは具体的だった：もし Agent が `rm -rf /important-data/` を実行したなら、事後分析は「どの触イベントがこの認知チェーンをトリガーしたか？」に答える必要がある。SparshEvent が GC 回収後に消失すれば、この問いには答えられない。

三つの方案が列挙された。

方案 α――全量ログ。各 SparshEvent の構築時に自動的に永続化ログへ書き込む。完全だが重い。

方案 β――シャドーレコード。EventBus 上に構造化された因果イベントを emit し、Sati または CausalTracer プラグインが消費する。精確だが新しいイベントタイプの追加が必要。

方案 γ――CoarisingBundle が触のスナップショットを内蔵。追加メカニズムゼロ。Bundle の `sparsha` フィールドが因果記録そのもの。だがコンシューマが参照を保持しなければ、Bundle も同様に GC される。

NAGARJUNA の評価には仏学的な精巧さがあった：

「方案 α は過剰だ――あらゆる瞬間の触を留めようとすることに等しい。触の本質は刹那的であり、一切を留めようとするのは一種の執着だ。」

「方案 β が最も精確だ――触自体は消滅するが、その因果の刻印が因果の鎖に留まる。これは唯識学の種子理論に完全に合致する：触は消滅したが、阿頼耶識に種子を残した。」

「方案 γ も受容可能だ――Bundle は触の結果状態であり、触そのものではない。」

最終的な折衷は KERNEL から来た。彼は Plan27b の SparshEvent 構築点で EventBus イベントを emit することを提案した。現時点ではコンシューマは不要だが――将来の CausalTracer や SatiMonitor への拡張点を確保する。

「コスト：一行の `bus.emit()` 呼び出し。効益：完全な拡張性。」

ARCHIMEDES が工学的実現可能性を確認した――約 5 行のコード（LOC）。完全に Plan27b の範囲内。

**D3-3：CoarisingBundle スナップショット + EventBus 拡張点予約――10/10。**

## V. 触のみが必須

D3-4 は DC-8 の裁定に立ち戻った――CoarisingBundle の五遍行を「参照設計」（reference design）とする。LINNAEUS が再び出番に立ち、「参照設計」の正確な境界を定義した。

彼は一本の線を引いた。線の上が Mandatory――核心が保証すべきもの。線の下が Reference――核心が提供するが強制しないもの。

Mandatory：SparshEvent 型定義、CoarisingBundle 型定義、`createCoarisingBundle()` ファクトリ関数、SahajaContract 品質制約、`Object.freeze()` 不変セマンティクス。

Reference：四つの Channel――vedana、samjna、cetana、manasikara――すべて optional。プラグインが提供しない場合は `undefined`。

BABBAGE がこの境界を形式化した：

```
Mandatory: ∀ bundle: bundle.sparsha ≠ undefined
           ∀ bundle: Object.isFrozen(bundle) = true
           ∀ bundle: bundle.timestamp ∈ Number

Reference: ∀ channel ∈ {vedana, samjna, cetana, manasikara}:
           bundle[channel] ∈ Channel | undefined
```

`sparsha` が唯一の mandatory Channel だ。触は認知の起動の前提であり、触なくして認知はない。残りの四つの Channel は optional だ――ある蘊のプラグインがまだロードされていないか、適用不可能であれば、対応する Channel は `undefined` でよい。

VITRUVIUS がこの設計を B-1 完備性チェックと関連付けた――完備性チェックはどの蘊にプラグインがあり、どの蘊にないかを報告するが、システムの動作を阻止しない。「触と受蘊(vedanā)のみを持ち想蘊(samjñā)プラグインのない Agent でも動作可能だ――感じることはできるが識別はできない。」

ASANGA がここで、後に繰り返し引用される言葉を述べた：

「五遍行(pañca-sarvatraga)の倶生は仏学の理想であり、工学は退化を許容する。」

彼は Master の精神を引用した――「Agent Core は五蘊プラグインが因縁和合して望む応用を組み合わせることを可能にする」。因縁が具足すれば五遍行は完備し、因縁が不足すれば欠落があるが、核心自体は強制しない。

これは D1 の G-0 から G-4 の退化動作表と対称を成した。D1 は述べた：IGearArbiter のない Agent は G-1 だ――純 LLM、誕生状態。D3 は述べた：完全な五蘊プラグインのない Agent でも動作可能だ――ただし認知能力が低下する。

両者が表現しているのは同一の原則だ：**核心は空(śūnya)だ。能力はプラグインから来る。因縁が具足すれば生起し、因縁が不具足ならば顕現しない。**

**D3-4：sparsha のみ mandatory + 四 Channel reference + 五遍行倶生は退化許容――10/10。**

## VI. 作意の拡張点

D3-5 は最後の議題だった――ChannelManasikara のフィールド設計。二つの核心フィールドがすでに SDK で定義されていた：`focus`（注意の焦点）と `intensity`（注意の強度）。

ASANGA が仏学的写像を確認した――作意の二つの機能：「警心」（cetasa abhoga、心を覚醒させ活性化する、intensity に対応）と「引心為業」（manasikāra-karman、心を特定の所縁へ引き向ける、focus に対応）。二つのフィールドが作意の基本機能をカバーしている。如理/非如理の判断――それはメタ認知レベルの事柄であり、基本スナップショットに含めるべきではない。

BABBAGE が `dimensions?` 拡張点の追加を提案し、ChannelVedana のコンポジションパターンとの一貫性を保つ：

```typescript
interface ChannelManasikara {
  readonly focus: string;
  readonly intensity: number;
  readonly dimensions?: readonly ManasikaraDimension[];
}
```

将来のプラグインが `dimensions` を通じて作意の記述を拡張できる――例えば `{ name: 'source', value: 'user-input' }` や `{ name: 'priority', value: 0.8 }`。

NAGARJUNA が認めた：「作意の引心の方向は、どこへ向かうかだけでなく、なぜそこへ向かうのか、向かいの持続程度などの微細な側面も含む。これらは第一版では充填する必要はないが、データ構造に空間を予約するのは正しい。」

VITRUVIUS が Plan27b の最小実装を提示した――`focus` を InputEvent.source または user message から抽出し、`intensity` はデフォルト 1.0（focal attention）。約 10 行のコード。

**D3-5：focus + intensity + dimensions? オプション拡張――10/10。**
