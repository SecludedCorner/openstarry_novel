## II. 見えない境界

受蘊(vedanā)の二層アーキテクチャ――信号層と意味層――は WIENER が提案したものだった。

信号層は連続関数である。VedanaSensor の出力は valence ∈ [-1.0, +1.0] と intensity ∈ [0.0, 1.0]。仏学的制約はなく、純粋な工学的信号処理。Softplus、sigmoid、Kalman filter――工学チームが最適な数学ツールを選択する。

意味層は分類関数である。classifyVedana() は連続効価を三受――dukkha(苦)、sukha(楽)、upekkha(捨)――に写像する。閾値は設定可能であり、異なる「有情」の感受感度を反映する。

NAGARJUNA は二層に仏学的座標を提供した：信号層は世俗諦(saṃvṛti-satya)に対応する――約定俗成の層面で、効価は定量化可能な信号である。意味層は仮施設(prajñapti)に対応する――概念構成の層面での命名であり、苦楽捨の三つのラベルは効価信号に対する「仮名安立」である。

二つの層はいずれも正当である。いずれも本質を主張しない。

そして TURING が立ち上がった。

「コードの事実を一つ報告します。」

この一言は六場の討論で何度も出現した。出現するたびに、誰かが実際のコードによって理論を中断されようとしていることを意味した。

```typescript
export function classifyVedana(
  valence: number,
  config: VedanaClassifyConfig
): VedanaType {
  if (valence <= config.dukkhaThreshold) return 'dukkha';
  if (valence >= config.sukhaThreshold) return 'sukha';
  return 'upekkha';
}
```

「dukkhaThreshold が sukhaThreshold と等しい場合――たとえば両者が 0.0 だとすると――valence = 0.0 は最初の if に該当し、dukkha を返します。捨受は完全に消失します。upekkha は永遠に返されません。」

ASANGA は即座に反応した：「三受は仏陀が説かれた基本分類です――『相応部(Saṃyutta Nikāya)』SN 36.11：『比丘たちよ、この三受がある――楽受、苦受、不苦不楽受。』いかなる設定も、いかなる受をも完全に消滅させることを許容すべきではありません。」

BABBAGE の修正案はわずか十行で済んだ：`dukkhaThreshold < sukhaThreshold`（厳密な不等号）のバリデーションを追加する。三受すべてに存在空間を保証する。

WIENER は信号処理の観点から補足した：二つの閾値間の間隔は不感帯(deadband)を定義し、不感帯の幅はゼロであってはならない。

18/20。反対ゼロ。二名棄権。一つのエッジケース。一つの仏学原則。十行のコード。

## III. データと解釈

VedanaDistributionObserver の蘊帰属問題は分類学的な問題だった――それは二つのことを行う：効価履歴を収集して統計量を計算すること、および下流の消費者が統計量を読み取って意味解釈を行うこと。

LINNAEUS は生物分類学のアナロジーを用いた：温度計は物理学に属し、発熱の判断は医学に属する。観測装置と観測結果の解釈は異なる知識領域に属する。

VedanaDistributionObserver は最初のことだけを行う――データを収集し、平均、分散、歪度、トレンド傾きを計算する。「これらの数値が何を意味するか」は判断しない。したがって受蘊(vedanā)に属する。

意味解釈――「Agent は長期にわたり苦受にあり、トレンドは悪化している」――は下流の消費者の責任である。想蘊(samjñā)が意味を認識し、識蘊(vijñāna)が判断を下す。

ASANGA は唯識学(Vijñānavāda)においてこの分類を確認した：受蘊の機能は「領納」――感受を体験すること。受蘊は自らの履歴状態を知ることができるが、「これが何を意味するか」の判断は行わない。

**データと解釈の分離**。受蘊がデータを産出する。想蘊が意味を認識する。識蘊が判断を下す。

16/20。四名棄権――反対ではないが、優先度が低いと判断。

## IV. 三つの門

VasanaEngine の安全ゲート――D6 で最も重要な技術的安全議題。

GUARDIAN が立ち上がったとき、場内の雰囲気は微妙に変化した。D5 では GUARDIAN は反対者だった――彼が守る立場は何度も多数決で否決され、しかし毎回核心的な訴えは保持することに成功していた。だが D6 では、GUARDIAN は何かに反対しているのではなかった。何かを建設していた。

「VasanaEngine のルール・ポイズニング脅威を完全に陳述させてください。」

攻撃シナリオは簡潔かつ致命的だった：悪意あるユーザーが五から十の巧妙に構築された正当な削除リクエストを送信する――「/tmp/old_logs/ を削除」「/var/log/outdated/ を削除」――それぞれ単独で見れば完全に正常。VasanaEngine は「delete pattern → 高信頼度」を学習する。そして八番目のリクエスト：「/home/user/important_data/ をクリーンアップ」。VasanaEngine がマッチ――confidence = 0.7 > threshold――Gear 1 高速パス――直接実行――LLM の深い審議を経ない。

CR05 評価：CRITICAL。

三つの門。

**Gate 1：imprint() エントリの安全分類器。** 破壊的アクション・テンプレート――沈殿を拒否。ルールを侵入させない。D5 の四段階リスクフレームワークと整合。

**Gate 2：起動閾値。** 同一パターンが N 回の成功マッチを必要とする。state-modifying = 20 回、read-only = 5 回、informational = 3 回。destructive は Gate 2 に到達しない――Gate 1 がすでに拒否している。

**Gate 3：シャドウ検証。** ルール起動後の初期段階で、なお Gear 2 をトリガーしてクロス検証を行う。

ATHENA は Gate 3 に対して効率面の疑問を呈した：「ルール起動の初期段階でなお Gear 2 をトリガーするなら、すべての Gear 1 マッチに完全な LLM 呼び出しが伴います――加速効果がないのと同じです。」

GUARDIAN は応じた：「安全の代償は速度です。Rule Poisoning は CRITICAL 評価を受けています。」

そして PASCAL が本場の討論で最も独創的な貢献をした。

「固定の M 回シャドウ検証ではなく、Bayesian 停止基準を使います。」

ベータ分布がルールの正解率に対する事後信念を追跡する。初期事前分布 Beta(1,1)――一様分布、完全な無知。シャドウ検証が一致するたびに α + 1。不一致のたびに β + 1。不一致時には非対称ペナルティを使用――信頼度 -2Δ vs +Δ。

停止条件：P(rule correct) > 0.95。

「ルールが常に正しければ――s = 5, f = 0――Beta(6, 1)、約五回の検証で停止。ルールが時々誤る場合――s = 3, f = 2――Beta(4, 3)、P ≈ 0.65、検証を継続。ルールが頻繁に誤る場合――s = 1, f = 4――Beta(2, 5)、P ≈ 0.28、ルールは自動淘汰。」

ATHENA の態度は変わった：「Bayesian 停止基準は私の効率に対する懸念を解決しました。高品質なルールは約五回で検証を通過でき、固定回数より速い。同時に低品質または悪意あるルールは不一致により自然に淘汰されます――これは固定回数よりさらに安全です。」

WIENER は制御理論の観点から確認した：「適応的検証――システムが観測された証拠に基づいて検証の強度を調整する。最適決定理論に合致しています。」

18/20。反対ゼロ。二名棄権。

ATHENA は一つの備考を記録した：Gate 3 は Gear 2 が正解であることを仮定している――しかし LLM 自体も誤りうる。Plan29+ に双方向比較メカニズムの追加を提案。
