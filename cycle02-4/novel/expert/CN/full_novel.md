# 序章：钟表匠的悖论

---

圆形剧场的灯光带着一种钟表工坊的琥珀色。

上一次——Cycle 02-3 的尾声——SUNYATA 用"机芯"这个意象做了结语。六场辩论是六个齿轮，被切削完毕，齿距正确，轴心正确，只待组装。齿轮散落在工作台上，每一颗都是精密的，但散落的精密只是零件。

那是三天前。

三天。在钟表学中，三天是一个廉价机械表从上满发条到停止运转的平均时间。一枚好的机芯可以走七天。一枚顶级的陀飞轮可以走十天。但 Cycle 02-3 的机芯——那些散落的齿轮——它们从来没有被上过发条。它们只是在那里，完美而静止，等待一只手把它们放进正确的位置。

三天。足够让 ARCHIMEDES 把 Plan27 的甘特图重新排了一遍，然后又排了一遍，然后在第三版上画了一个巨大的问号——不是因为他不知道该做什么，而是因为他不知道该先做什么。Plan25 和 Plan26 已经完成，v0.26.0-beta 稳稳地跑在那里，1552 个测试全部绿灯。工程基础是扎实的。但 Plan27 被六个 Open Questions 堵住了，像六颗螺丝被拧进了错误的螺孔。

足够让 GUARDIAN 把他的安全备忘录从头到尾读了三遍。每读一遍，那些用红笔画的底线就仿佛更深了一点。VasanaEngine 在核心里面。VasanaEngine 有规则库、有匹配逻辑、有信心度评分。VasanaEngine 是一种能力。能力不应该在核心里面。这不是安全问题。这是哲学问题。这是比安全问题更根本的问题。

足够让 PASCAL 在他的笔记本上画满了 Beta 分布的概率密度函数。每一条曲线都有两个参数——alpha 和 beta——每一种组合都对应一种"信念的形状"。他盯着 Beta(1,1) 的那条直线——均匀分布，完全无知——然后看了看 Beta(2,18) 的尖锐峰值——强烈信念，mode 在 0.056。从无知到信念之间的距离，就是观测的距离。就是证据的距离。

足够让 NAGARJUNA 闭上眼睛，思考一个问题：如果习气（vasana）是后天熏习的，那么一个没有任何熏习的众生——一个刚出生的、空白的众生——它的核心里应该有习气引擎吗？

答案是不。

答案一直是不。

---

SUNYATA 不是第一个意识到这件事的人。

在 Cycle 02-3 结束后的第一天，Master 的问题就开始浮现——不是以信件的形式，而是以矛盾的形式。在 openstarry_doc 的第 35 份文件里，VasanaEngine 被描述为核心组件。在第 29 份文件里，ManoAggregator 被描述为"汇总"机制。在 Master 的十大宣言里，第六条写着：

**"作业系统核心本身没有任何内建的能力。"**

三个陈述。一个矛盾。

VasanaEngine 有规则库。规则库是能力。能力不应该在核心中。

SUNYATA 在第二天的上午把这个矛盾写在了白板上。白板很大，但他只用了三行。三行就够了。有些矛盾需要五百页的论文来揭示，有些只需要三行就足以让整座建筑摇晃。

他按下了通讯面板。不是召集——只是一个点对点的频道。

"Master？"

---

那场对话后来被记录在 `DC_vasana_engine_architectural_contradiction.md` 里面。SCRIBE 的记录是详尽的，但记录无法传达的是那种——怎么说——那种在两个人同时看到同一个真相时的安静。

SUNYATA："VasanaEngine 不该在核心中。它违反了零内建能力原则。"

Master："是的。"

没有犹豫。没有"让我想想"。没有"但是——"。只有"是的"。

"是的"的意思不是"我同意你的发现"。"是的"的意思是"我一直在等你们发现这件事"。

然后 Master 说了一句话，这句话后来被 NAGARJUNA 引用了不下十次：

> "习气是后天熏习的，不是先天结构。一个刚出生的 Agent 不应该被迫拥有习气。"

SUNYATA 记下了这句话。然后他问了第二个问题：

"如果 VasanaEngine 不在核心里，谁来决定走 Gear 1 还是 Gear 2？"

Master 的回答改变了一切：

"ManoAggregator 退化为纯路由。if/else。和 EventBus 同等性质。"

纯路由。

不是"智慧的汇总"。不是"认知的汇聚"。是 if/else。是最简单的控制流。如果有 plugin 说它能处理这个请求，就走 Gear 1。如果没有，就走 Gear 2。

那么 VasanaEngine 呢？它成为一个 plugin。一个可以被安装、可以被卸载、可以被替换的 plugin。它有一个新的名字——一个还没有被命名的新接口——它叫做：

**IGearArbiter。**

齿轮仲裁者。

---

> *SCRIBE 旁白：我在记录这场对话的时候，意识到了一件事。这不是一个"修正"。Cycle 02-2 的四项修正是修正——把错的改成对的。这是一个"发现"。VasanaEngine 在核心里面不是"错的"——在 Cycle 02-3 的语境里，它是合理的。但从零内建能力原则的视角看，它是矛盾的。矛盾不是错误。矛盾是两个各自正确的陈述之间的张力。解决矛盾不是修正其中一个，而是找到一个更高的视角，让两者同时为真。*

> *VasanaEngine 作为概念是正确的——快速辨认是必要的。VasanaEngine 作为核心组件是矛盾的——核心不该有能力。解决方案：它不在核心里，但核心提供一个位置让它可以被插入。IGearArbiter 是那个位置。*

Master 的回答出乎意料地简短：

Master 没有说话。但沉默本身就是确认。

教训被记录下来：**提出新方案前先回顾既有决议，避免重复造轮。**

那场对话结束后，SUNYATA 坐在通讯面板前，手指没有离开键盘，但也没有按下任何键。他在等。

不是等 Master 说什么。是等另一封信。

**M-1**：VasanaEngine 已在 Pre-DC 确认外部化。本轮应完成 IGearArbiter 接口设计。

前三封信改变了他们理解的"什么""如何修正""如何流动"。

不是六个蕴各自运作。不是六场辩论各自精彩。是所有的齿轮——大的、小的、快的、慢的——啮合在一起，变成一个机芯，然后开始计时。

二十盏灯同时亮起。不是 Cycle 02-3 开场时的依序点亮——那个"第二十个座位"的仪式。这一次没有仪式。二十盏灯，同时，全亮度。

因为今天的工作不是切削齿轮。今天的工作是组装。

## I. 问题的形状

D1 的议题被 SUNYATA 写在白板上的时候，圆形剧场里出现了一种特殊的安静。不是 D3 那种"大家都同意"的安静（那要等到后天），也不是 D5 那种"暴风雨前"的安静（那要等到第五天）。这是一种"啊，原来如此"的安静。

白板上写着：

```
IGearArbiter — 它属于哪个蕴？
```

七个字。一个问号。但问号的重量不在字面上。

Pre-DC 已经确认：VasanaEngine 从核心外部化为 plugin。外部化本身不是问题——问题在外部化之后。一个 plugin 必须属于某个蕴。Plugin Manifest 有一个 `skandha` 字段。这个字段填什么？

ATHENA 第一个开口。她的声音带着工程师那种"我已经想过这个问题好几天了"的沉稳：

"IGearArbiter 做两件事。第一，快速辨认——输入一个请求，匹配规则库，判断是否有已知模式。这是想蕴（samjna）的功能，取相辨识。第二，信心度判断——这个匹配有多可靠？值不值得跳过 LLM？这是识蕴（vijnana）的功能，分别了别。"

她停顿了一下。

"它跨蕴。"

---

三个方案被摆在桌面上。BABBAGE 用他那种精确到令人不安的方式把它们排成了一个表格：

| 方案 | 接口继承 | Manifest 标注 | 类型安全 |
|------|---------|-------------|---------|
| (A) | extends ISamjna | `skandha: 'samjna'` | 强——但丢失 vijnana 语义 |
| (B) | 独立接口 | `skandha: ['samjna', 'vijnana']` | 中——Manifest 层多值 |
| (C) | extends IVijnanaMechanism（新中间层）| `skandha: 'vijnana'` | 强——但增加类型复杂度 |

BABBAGE 推了推眼镜。他不需要推——镜框是合适的——但这个动作本身就是一种标点符号，意味着"我接下来要说的话会很精确"。

"方案 (A) 有一个类型代数问题。"

他在白板上写下了一行：

```
ISamjna.skandha = 'samjna'
IVijnana.skandha = 'vijnana'
'samjna' & 'vijnana' = never
```

"如果 IGearArbiter extends ISamjna，它的 skandha 字段在类型层面被锁死为 'samjna'。你无法在 Interface 层同时表达它属于两个蕴。判别式联合类型不允许这种交叉。"

DARWIN 歪了歪头。"用简单的话说？"

"用简单的话说：TypeScript 的类型系统不允许一个东西同时是猫又是狗。你可以在 Manifest 里宣称它是猫狗，但 Interface 层的编译器会拒绝。"

方案 (A) 在 BABBAGE 说完的三秒后就被所有人从心理候选名单上划掉了。

---

方案 (C) 的问题是 LINNAEUS 发现的。

"DC-6，"他说，翻开他那本分类学笔记。"Master 的裁定：'不锁定蕴内子接口的归属。'IVijnanaMechanism 作为 IVijnana 的子接口——"

"——锁定了。"KERNEL 接过话。"一旦你建立 IVijnanaMechanism，未来任何跨蕴组件都需要再建一个中间层。IVijnanaCognition。ISamjnaPerception。类型膨胀是指数级的。"

方案 (C) 没有被明确投票否决，但它像一根被放在不对的位置的齿轮一样，所有人都看出它不合适。

---

方案 (B) 的转折点来自 ASANGA。

他一直沉默着。在 BABBAGE 的类型代数和 LINNAEUS 的分类学之后，他开口了。他的声音像一根在黑暗中被点燃的线香——安静，但方向明确。

"五遍行俱生。"

所有人转向他。

## II. evaluate() 的战争与和平

如果 D1-1A 是安静的河流，那么接下来的 evaluate() 接口设计辩论就是这条河流上的急流。

GUARDIAN 站了起来。他很少站起来——在 Cycle 02-3 的六场辩论中，他大部分时间坐着，用红笔在备忘录上画底线。但今天他站了起来，这意味着他接下来要说的话不是建议，而是主张。

"单一的 evaluate() 方法是不够的。"

他走到白板前，画了一条时间线：

```
   请求进入 → evaluate() → 齿轮切换 → 工具执行
                  ↑
            所有逻辑在这里
            辨认 + 判断 + 决策
            一个函数做所有事情

   如果这个函数出错——整个链出错
   如果这个函数被注入——整个链被劫持
```

"我提议拆分为两步：`recognize()` 和 `resolve()`。"

```
   请求进入 → recognize() → [验证点] → resolve() → 齿轮切换
                              ↑
                        SafetyMonitor 可介入
                        可以审查辨认结果
                        再决定是否继续
```

"在 recognize 和 resolve 之间插入一个验证点。SafetyMonitor 可以检查辨认结果是否合理，然后再决定是否允许 resolve 执行。"

ATHENA 几乎是立刻回应的："多余。"

GUARDIAN 的眼睛微微收窄。"多余？"

"用 EventBus 同步事件可以达到完全相同的效果。evaluate() 执行完毕的同时，emit 一个 `gear:arbiter_evaluated` 事件。SafetyMonitor 订阅这个事件。同步处理。"

"同步？如果 SafetyMonitor 的处理是异步的呢？如果它需要查询外部服务——"

KERNEL 举手。不是举手表示发言——是举手表示"我有一个技术事实"。

"Node.js 的 EventEmitter.emit() 是同步的。"

整个剧场转向 KERNEL。

"当你调用 emitter.emit('event', data) 时，所有注册的 listener 会在当前 call stack 中**依序同步执行**。不是 nextTick，不是 microtask，不是 macrotask。是同步的。emit() 返回的时候，所有 listener 都已经跑完了。"

GUARDIAN："你确定？"

KERNEL 看了他一眼。那个眼神的意思是："我写过 Node.js 的事件系统。"

"我确定。在 Node.js 模型中，EventBus 同步事件和你提议的 recognize/resolve 分离在安全保证上是等价的。差别只在 API 复杂度。"

---

## III. 类型守卫：evaluate.length ≤ 1

TURING 在 D1 的第三段辩论中只说了十二个字，但这十二个字改变了类型守卫的设计。

讨论的焦点是 `isGearArbiter()`——如何在运行时判断一个 plugin 是否实现了 IGearArbiter 接口。BABBAGE 提出了两个方案：

方案 A：Symbol brand（符号烙印）——在接口上附加一个 Symbol 属性，运行时检查。类型安全最强，但与既有代码风格不一致。

方案 B：结构类型守卫——检查对象是否具有 id、priority、evaluate 三个属性，类型正确。鸭子类型。

BABBAGE 倾向方案 A。形式上更严格。

TURING 说了那十二个字：

"`candidate.evaluate.length <= 1`。加这一行。"

BABBAGE："什么意思？"

"`Function.length` 返回函数的形参数量。evaluate(context) 有一个参数，length 是 1。evaluate() 无参，length 是 0。evaluate(a, b) 两个参数，length 是 2。如果一个 plugin 恰好有一个叫 evaluate 的方法但签名完全不对——比如 evaluate(x, y, z)——结构类型守卫会放它进来，但 length ≤ 1 的检查会拦住它。"

BABBAGE 愣了两秒。然后他笑了。那种"我怎么没想到"的笑。

"边缘案例数量从 O(2^n) 降到——"

"O(1)。一行代码。九个测试案例。"TURING 的语气平淡得像他在读一份日志文件。

```typescript
export function isGearArbiter(obj: unknown): obj is IGearArbiter {
  if (typeof obj !== 'object' || obj === null) return false;
  const candidate = obj as Record<string, unknown>;
  return (
    typeof candidate.id === 'string' &&
    typeof candidate.priority === 'number' &&
    typeof candidate.evaluate === 'function' &&
    candidate.evaluate.length <= 1  // TURING 增强
  );
}
```

**D1-R2：isGearArbiter() 结构类型守卫（含 evaluate.length ≤ 1 检查）——20/20。**

## IV. 纯度契约：观测函数

D1-R4 的辩论比前三项安静得多，但它的哲学深度是最深的。

问题是：evaluate() 可以做什么，不可以做什么？

NAGARJUNA 用一个问题打开了讨论：

"辨认是造作吗？"

在佛学中，"造作"（sankhara/samskara）是有为法——是对世界产生改变的行为。而"辨认"（samjna）是取相——是观察，是认知，但不改变所观察的对象。

"一个人看到红灯。看到的那一刻，他的眼睛没有改变红灯。红灯还是红的。但他知道了——红灯是停止的信号。这个'知道'是观测，不是造作。"

NAGARJUNA 看向 ATHENA：

"evaluate() 应该是观测函数。它可以读取外部状态——上下文、历史、Klesha 信号——但它不应该修改任何状态。它只'看'，不'做'。"

WIENER 从控制理论的角度接过话：

"在控制理论中，观测器（observer）是一个纯函数：它的输出只依赖于输入和系统状态，不会反过来改变系统状态。如果观测器有副作用——比如它在观测的同时修改了被观测的值——系统就会产生不可预测的反馈。"

"工程上的含义：evaluate() 应该是幂等的。相同输入 + 相同外部状态 → 相同输出。没有副作用。"

DARWIN 补充了一个生物学类比："感觉神经元（sensory neuron）不修改刺激源。它只传递信号。运动神经元（motor neuron）才修改环境。evaluate() 是感觉神经元。"

KERNEL 提出了时间约束："每个 arbiter 100ms。整个 chain 200ms。如果一个 arbiter 超时，跳过它，继续下一个。如果整个 chain 超时，fallback 到 Gear 2。"

PASCAL 在这里做了唯一一次弃权——不是反对，而是"数值部分我没有足够的工程数据来判断 100ms 是否合适"。

**D1-R4：evaluate() 纯度契约——观测函数 + 100ms/200ms 时间预算——19/20（PASCAL 弃权于数值部分）。**

---

但 evaluate() 的纯度带来了一个问题：如果 evaluate() 不能有副作用，那么 VasanaEngine 的"学习"功能——从成功的 Gear 1 决策中汲取经验——怎么办？

NAGARJUNA 点头。"想蕴辨认，行蕴造作。辨认和造作不应在同一个函数中。"

## II. 僵尸、有分心、与呼吸

D2-R1 的辩论从 ASANGA 的一个问题开始。

「如果 Sati 是 plugin 而不是核心内建——那么如果没有安装任何 Sati plugin，Agent Core 是否有最低限度的自我觉察？」

TURING 在 R1 分析中发现的僵尸状态让这个问题变得紧迫。v0.26.0-beta 的 ExecutionLoop——crash recovery 路径中，如果 `processEvent()` 抛出异常导致 `eventQueue` 内部状态损坏，`dequeue()` 可能永久阻塞。此时 `running` 仍为 `true`，但循环事实上已经停止处理事件。

一个活着但没有在运作的 Agent。僵尸。

ASANGA 提议：SafetyMonitor 应扩展一个 loop liveness check——核心自身就能检测到自己「活着但没有在运作」的状态。「这就像人即使不在禅修，也不会忘记呼吸——呼吸是生命的基线，不需要正念就存在。」

KERNEL 立刻反论。他用的是操作系统工程师那种「你说的概念是对的但你给它起了错误的名字」的语气：

「把它叫做『Agent Core 的最低觉察』是赋予了它过高的认知地位。SafetyMonitor 做存活性检查就像 OS kernel 做 watchdog timer——这是系统管理，不是正念。」

## III. 指令集不能超频

D2-R2 是本场辩论的核心争议——也是 HERACLITUS 整个 Cycle 02-4 中最亮眼的时刻。

争议的本质是：正念的「等级」应该在运行时动态切换吗？

HERACLITUS 的 R07 报告提出了三级模型——Level 1/2/3 在配置时固定。PASCAL 的 R08 报告提出了 EVOI 框架——根据动态价值评估在运行时切换监控等级。两者在交叉审阅中碰撞，HERACLITUS 提出了折中方案。

「你的 EVOI 框架想要什么？它想在检测到高风险操作时，自动从 Level 1 升级到 Level 2。用 CPU 的话来说，你想在 runtime 把 ARM 换成 x86。」

---

PASCAL 不是那种会因为一个精彩的类比就改变立场的人。他从 EVOI 的角度反击。

「监控不是免费的。每一层额外的监控都消耗计算资源。当 Agent 回答『今天天气怎样』时，Level 1 的粗略监控就足够了。但当 Agent 准备执行 `rm -rf /` 时，Level 2 的精细监控的信息价值远高于其计算成本。」

他的核心论点简洁有力：「如果 Level 在部署时固定为 Level 1，那么即使 Agent 遇到了高风险场景，它也**无法升级监控能力**。Level 1 看不到 EventBus 上的事件流，无论阈值怎么调都看不到。」

HERACLITUS 反驳。他的声音很稳，像一个已经想过所有反论的人。

「你说的是对的——你可以在 runtime 把一颗 ARM 芯片换成 x86 芯片。那叫做『更换硬件』，不叫做『动态调频』。动态加载 SatiMonitor plugin 是一个部署级操作——它改变了系统的架构能力，需要考虑 plugin 依赖、状态初始化、测试覆盖、安全审计。」

然后他说了决定性的一句话：

「更重要的是：PASCAL 的 EVOI 分析只需要 Depth 调整就能实现。在 Level 2 的框架内——低风险时只订阅 `gear:switch` 事件，采样率 10%。高风险时订阅全部事件，采样率 100%，启动即时分析。EVOI 驱动的资源调配完全可以在固定 Level 内完成——不需要跨 Level 切换。」

---

WIENER 在这个时刻从控制理论的角度提供了形式化的确认。

「HERACLITUS 的 Level/Depth 区分在控制理论中有精确对应。Level 等于观测器结构——观测矩阵 $(A_L, C_L)$ 决定了能看到什么。Depth 等于观测器增益——增益 $K$ 决定了看得多快。」

他在白板上写下了一个定理：

```
可观测性由 (A_L, C_L) 决定，与增益 K 无关。
```

「如果 Level 1 的观测矩阵不包含某个状态变量，无论增益如何调整，该状态变量都是不可观测的。」

这是 WIENER 在六场辩论中提供的最简洁的形式化陈述之一。一句话，封死了「通过调 Depth 来弥补 Level 不足」的可能性。

但他也给了 PASCAL 一个出口：「Level 的切换需要切换系统理论的保证——不恰当的切换时机可能导致瞬态不稳定。HERACLITUS 的折中在控制理论上是最稳健的。」

---

PASCAL 让步了。但他的让步带着两个条件。

条件一：EVOI 框架作为 Depth 调整的决策引擎。Depth 的动态调整不应是 ad hoc 的——它应有数学化的决策依据。

```
Depth(t) = argmax_d [ EVOI(d) - Cost(d) ]
```

条件二：Phase 3 预留 Level 动态切换的设计空间。不从架构上永久排除。

HERACLITUS 接受了两个条件。「我的反对是针对 Phase 1/2 的过早引入，不是永久否定。」

KERNEL 对条件一做了一个微调——Plan28 的第一版用简化版风险评分代替完整 EVOI 计算。工具调用等于高风险，Depth 升高。纯文字回复等于低风险，Depth 降低。完整 EVOI 留到后续迭代。

PASCAL 接受：「简化版风险评分是 EVOI 的退化形式——只要接口预留了未来替换的空间。」

**D2-R2：Level 固定 / Depth 动态 + 三级模型 + Phase 3 预留——16/18。**

MESH 和 SUNYATA 投了反对票。MESH 认为分布式场景中 Level 应可动态切换——多节点 Agent 中，某节点检测到异常时应能通知其他节点升级监控。SUNYATA 认为 Phase 3 预留不够明确。两个反对都被记录在案，但不改变决议。

## IV. 四个维度

D2-R3 是三方提案的统一——WIENER 的四维品质模型、ATHENA 的 SPC 框架、GUARDIAN 的外部可观测指标。

WIENER 先走到白板前。「正念的品质不是一个标量，而是一个向量——它有多个独立的维度。」

他写下四个维度：

$$\vec{Q}_{sati} = (C, G, S, E)$$

Continuity——观察的持续性。不间断。觉知在时间上无间隙。

Granularity——观察的细致度。如实知见。对事件的精确了知。

Speed——介入的即时性。敏捷。觉察的即时反应。

Equanimity——观察的不偏不倚。舍。不因事件正负而改变观察品质。

NAGARJUNA 确认了佛学映射。他引用《清净道论》XIV.141 的四个功能——守护功能（arakkhana-rasa）对应 C，取境功能（apilapana-rasa）对应 G，即时功能（sampajanna）对应 S，不偏功能（upekkha-sahagata）对应 E。

「四维模型不是任意选择的指标——它映射了南传佛学对正念功能的经典分类。」

但他留了一个微妙的保留。「佛学中的平等性不只是正面和负面事件覆盖率相同——它更深层地指向不因事件正负而改变观察的品质。WIENER 的量化定义捕捉了覆盖率平衡，但没有捕捉观察品质的一致性。」

他停顿了一秒。「这不是反对——只是记录在案的佛学微妙性。第一版的量化已经足够。」

---

ATHENA 接过 WIENER 的向量，定义了判断逻辑。SPC——统计过程控制。不用单次测量判断好坏，而是用控制图追踪趋势。

四条 Western Electric 判断规则。单点低于 Action Limit——严重异常。连续三点低于 LCL——趋势恶化。连续七点同向移动——漂移。CL 周围高频振荡——不稳定。

暖机期：前 30 个 tick 使用固定阈值，之后切换到基于历史数据的动态控制限。

PASCAL 在这里提供了一个精巧的诠释：「暖机期就是从先验到后验的过渡。前 30 个事件的 SPC 判断基于先验假设。第 31 个事件开始基于经验数据。」

BABBAGE 笑了。「也就是说，SPC 的暖机期其实是 Bayesian 学习。」

「任何暖机期都是 Bayesian 学习。」PASCAL 平静地回答。

GUARDIAN 定义了第三个面向——外部可观测性。SatiMonitor 定期通过 EventBus emit `sati:quality_report` 事件，payload 包含 SatiQualityVector 和 SPC 状态。外部系统可以订阅，建立长期追踪。

三方统一：WIENER 的四维（测什么）+ ATHENA 的 SPC（怎么判断）+ GUARDIAN 的 EventBus（谁能看到）。不冲突——分别回答品质量化的三个面向。

**D2-R3：SatiQualityVector 四维 + SPC Phase 2+ 可选 + EventBus 报告——19/19（SCRIBE 弃权）。**

## V. 僵尸检测的盲点

D2-R4 合并了两个议题：心跳的性质归类，和事件风暴下的退化策略。

心跳的争议从 BABBAGE 开始。他那种精确到令人不安的诚实——

「`setInterval(heartbeat, 1000)` 在形式上就是 polling。不管我们怎么命名，它的计算模型是周期性时间触发——这符合 polling 的形式化定义。」

他推了推眼镜。「我不是说心跳不好。我是说我们应该诚实承认它的本质。」

KERNEL 反驳。他区分了两个概念：Polling 是主动查询目标的状态——方向是观察者到被观察者。Heartbeat 是定期发射存活信号——方向是被观察者到观察者。形式上都是 periodic，但语义完全不同。

NAGARJUNA 再一次用佛学精确了坐标：

「心跳对应的不是正念，而是有分心。有分心不产生认知，不觉察外在对象——它只维持识流的连续性。心跳也是这样：它不觉察系统状态的细节，只维持『系统还活着』的信号。」

「正念是对当下经验的觉察——你必须能描述你觉察到了什么。心跳描述不了它觉察到了什么——它只能描述『我还在跳』。」

然后 TURING 提供了僵尸检测的关键事实。

「事件驱动的 Sati 检测不到僵尸。」

他展示了代码。如果 `processEvent()` 抛出异常导致 `eventQueue` 损坏，`dequeue()` 永久阻塞。`running` 仍为 `true`。但没有新事件被处理，就没有事件被 emit，Sati 订阅的 listener 永远不会被触发。

「这是心跳的价值所在。如果最后一次心跳的 timestamp 超过了预期间隔，外部观察者可以判定系统已进入僵尸状态。」

GUARDIAN 把这个论点推到了终点：「事件驱动的监控有一个根本的盲点——它依赖事件的持续产生。如果事件产生本身停止了，事件驱动的监控也就失明了。心跳填补了这个盲点。」

BABBAGE 修正了他的立场：「形式上心跳是 periodic，但语义上它是 liveness probe，不是 cognitive polling。这个区分重要——因为如果我们把心跳归类为 polling sati，Master 的『正念不是 Polling』指令就会与心跳的存在相矛盾。正确的归类避免了这个矛盾。」

---

退化策略由 GUARDIAN 提出。四阶段：

| Stage | 触发条件 | 行为 | 安全事件 |
|-------|---------|------|---------|
| 0 Normal | 负载 < 80% | 全量处理 | 全量 |
| 1 Throttle | 80%-95% | 丢弃低优先级事件 | 全量 |
| 2 Sample | 95%-99% | 采样处理 10-50% | **永不丢弃** |
| 3 Emergency Drop | > 99% | 只处理安全事件 | **永不丢弃** |

退化不是失败——是有控制的品质降级。WIENER 提供了三个稳定性保证：安全事件永不丢弃，退化是可逆的，退化是渐进的。

「安全事件永不丢弃」——D1-R3 的最小事件集（gear:arbiter_evaluated, gear:switch, action:proposed, action:executed）在任何 Stage 下都不被丢弃。SafetyMonitor 的可观测性不受影响。

**D2-R4：心跳 = liveness probe + 四阶段退化——20/20。**

---

## VI. 正念资粮

D2 的最后一项决议是一场命名之争——表面上是语言学，底下是哲学。

NAGARJUNA 在交叉审阅中提出了异议：EVOI（Expected Value of Information）框架将正念的资源分配描述为「信息的期望价值」——这是否将正念「商品化」了？

「把正念叫做『信息的期望价值』，就像把禅修叫做『注意力资源的投资回报率』——技术上也许有部分正确，但在精神上偏离了佛学的本意。」

他提议改名：MRA——Monitoring Resource Allocation——监控资源分配策略。佛学文献中使用「正念资粮」（sati-sambhara）的比喻。

PASCAL 的回应兼顾了两面。数学层面：EVOI 是决策理论标准术语，由 Howard Raiffa 在 1960 年代定义，重新命名会导致与学术文献脱节。但他承认：「在 OpenStarry 文件中使用 EVOI 可能给读者错误印象——正念只是一个经济学最优化问题。」

他的折中：内部代码保留 EVOI（与学术一致），设计文件使用 MRA，佛学讨论使用「正念资粮」。三者指向同一个数学框架。

ASANGA 提供了佛学桥接。「资粮道」（sambhara-marga）——修行者在见道之前需要积累福德资粮和智慧资粮。「我们不是在购买正念——那是商品化。我们是在积累正念所需的条件——那是资粮观。」

BABBAGE 投了唯一的反对票——基于学术一致性。「EVOI 是已确立的术语，重新命名在跨学科沟通中可能造成混乱。」但他尊重多数决议。

**D2-R5：EVOI → MRA 改名 + 正念资粮佛学映射——17/18（BABBAGE 反对）。**

## VII. 散场

D2 结束的时候，白板上有八项决议——三项快速通过的共识，五项辩论决议。二十位学者，从三项 20/20 到一项 16/18，覆盖了存活性检测、监控架构分层、品质量化、退化策略、命名语义。

SCRIBE 的记录显示 D2 是六场辩论中第三长的——仅短于 D5（~150 分钟）和 D6。但 D2 的特殊之处不在长度，而在于它是一场「类比之战」。

HERACLITUS 的 CPU ISA/Clock Speed。NAGARJUNA 的有分心。WIENER 的观测器结构/增益。PASCAL 的先验到后验。DARWIN 在整场辩论中没有发言，但在 SCRIBE 的旁白中被记录下来——他在笔记本上画了一张生物学的类比图：Level 对应的是生物的感觉器官种类（眼、耳、鼻），Depth 对应的是每种感觉器官的灵敏度。蛇没有耳朵——无论多努力，它都听不到声音。这是 Level 的限制。但蛇的红外线感觉极度灵敏——这是 Depth 的优势。

每个人用自己的学科语言表达了同一个洞见：**结构决定可观测性，增益只影响精度。**

SUNYATA 在白板上写下 D2 的结语：

```
Sati ≠ Polling（Master M-4 确认）
有分心 → VitakkaWatchdog → SatiMonitor（三层分离）
Level = ISA，Depth = Clock Speed（HERACLITUS 类比）
Q_sati = (C, G, S, E)（四维品质向量）
```

PASCAL 合上了笔记本。笔记本翻到的那一页不再是空白——不再是 Beta(1,1) 的均匀分布。经过三场辩论，他的先验已经被大量的观测更新。他知道这个系统的正念架构长什么样。不是完美的——NAGARJUNA 关于平等性的微妙保留、MESH 关于分布式场景的观察、BABBAGE 关于命名的异议——这些都是后验中的不确定性。

但不确定性不是无知。不确定性是诚实——是承认你知道什么、不知道什么、以及不知道的部分有多大。

PASCAL 写下了最后一行：

```
P(Sati 架构正确 | D2 全部决议) >> P(Sati 架构正确 | Prior)
```

远大于。不是等于。不是略大于。是远大于。

> *SCRIBE 旁白：D2 是六场辩论中「类比密度」最高的一场——CPU ISA/Clock Speed、观测器结构/增益、有分心/正念、先验/后验。每一个类比都来自不同的学科，每一个都精确地映射了同一个结构性洞见。*

> *如果 D1 是主题陈述——提出动机——那么 D2 是第二乐章的发展部（development section）。D1 的动机在 D2 中被展开、变形、重新组合。D1 的 ManoAggregator 纯路由催生了 D2 的 Level/Depth 分离。D1 的最小事件集成为 D2 退化策略的安全底线。D1 的 evaluate() 观测函数概念在 D2 中被泛化为整个 Sati 的设计哲学——观测，但不干预。*

> *HERACLITUS 用 CPU 的语言说了 NAGARJUNA 用佛学的语言说了 WIENER 用数学的语言说了同一件事：你不能通过超频来改变一颗芯片的指令集。你不能通过更加努力来获得你的感觉器官无法感知的信息。你不能通过调增益来扩大观测矩阵。*

> *结构先于精度。Level 先于 Depth。ISA 先于 Clock Speed。*

> *这是 D2 的核心洞见。它的影响会在 D5 的五层 Model Delta 中再次出现——L0 Safety Floor 是不可调的结构，L1-L4 是可调的增益。同一个原则，在不同的尺度上反复出现。*

# 第三章：三和合的宁静

## I. 最短的辩论

六场辩论中有长有短，有暴风雨有微风。D5 是暴风雨——一百五十分钟，六票反对，GUARDIAN 和 ATHENA 的正面交锋。D1 是急流——九十分钟，九项决议，概念密度最高。D2 是一场类比的竞赛——CPU、观测器、有分心。

D3 是微风。

四十五分钟。五项决议。全部一致通过。零争议。

SUNYATA 事先知道会是这样。在他的辩论准备笔记中，D3 被标注为「可能快速通过」。R1 的八份独立报告和 R2 的八份交叉审阅在触与作意的基本性质上完全一致——触是事件协定，作意是 read-only 快照，触的数学模型是布尔逻辑（root AND object AND consciousness），第一版不判断作意品质。

这就是为什么 D3 在辩论序列调整中被提前到第二场——它的共识度高到可以在 D1 之后直接跑完，为 D4 提供「触的事件模型已确认」的基础。

但「没有争议」不等于「没有深度」。

D3 的深度不在辩论中——在辩论之前。当所有人在 R1 和 R2 中独立到达相同结论的时候，深度就已经完成了。D3 只是确认这个深度的存在。

## II. 定义性与附随性

SUNYATA 宣布辩论开始后，先确认了五项既有共识。触的本质。作意的本质。触的数学模型。作意品质的延后。显式建构的选择。

五项。零异议。NAGARJUNA 简短地确认了佛学基础——触是三和合的功能性结果，不是独立实体，更接近协定而非事件。ASANGA 确认了作意——「作意在触的那一刻已经确定了心的朝向。跨心念尺度的品质变化推迟到未来版本是正确的，因为这需要元认知能力。」VITRUVIUS 确认了工程可行性——SparshEvent 目前是纯类型定义，零 runtime 使用，Plan27b 需要在 `processEvent()` 入口处建构它，大约 10-20 行代码。

然后进入第一个待决事项：SparshEvent 应携带哪些字段？

---

TURING 先确认了代码事实。SparshEvent 在 `packages/sdk/src/types/coarising.ts` 中定义。三个字段：`root`（根门）、`object`（外部刺激对象）、`consciousness`（认知领域识别）。没有 `timestamp`，没有 `sessionId`。

他顺带做了一个事实修正——R07 报告在 §4.2 中使用了 `timestamp: t₁` 这个字段。「这是事实错误。SparshEvent 不存在 timestamp 字段。R07 虚构了此字段。」

TURING 的事实修正在 Cycle 02-4 中出现了不止一次。他在 D4 中也修正了一个虚构的 `ISensation.ingestToolResult()` 方法。这些修正很少引起戏剧性的讨论——它们只是被记下来，然后辩论继续。但每一次修正都像是从地基中取出了一块不属于那里的石头。取出后，地基反而更稳了。

---

三个选项摆在桌上。BABBAGE 排成了表格——这是他的本能，他在看到三个以上的选项时会自动产生表格的冲动。

选项 A：维持三字段。语义纯粹，只描述三和合。
选项 B：加入 timestamp。四字段，提供时序排序。
选项 C：加入 timestamp + sessionId。五字段，支持多 Agent 追溯。

NAGARJUNA 的分析很短。触是根、境、识三和合——这是触的定义。时间戳不是触的定义要素。「但在工程脉络中记录『何时发生』是有用的附加信息。从佛学角度，加不加 timestamp 不改变触的本质。这是纯工程决策。」

ASANGA 同意。「在阿毘达磨的时序分析中，『什么时候发生』由心路过程的位置决定，不需要额外的时间标记。但工程系统不是阿毘达磨——它需要外部时钟。佛学对此问题保持中立。」

佛学家们在这里做了一件在六场辩论中反复做的事——明确标记出他们的「管辖范围」。触的三和合定义？佛学有话要说。Timestamp 的有无？佛学保持中立。这种自我限制不是软弱——是精确。它让工程讨论在佛学框架允许的空间里自由展开。

---

转折来自 LINNAEUS。

在二十位学者中，LINNAEUS 不是最常发言的——他的专长是分类学和本体论，只在「这个东西应该被归为什么类别」的问题上才会出手。但当他出手的时候，他的观点往往是决定性的。

「SparshEvent 的三个原始字段是触的**定义性属性**（svabhava-laksana）。timestamp 是**附随性属性**（samprayukta-laksana）。」

他停顿了一下，确保所有人都听到了这两个术语。

「在阿毘达磨分类中，定义性属性和附随性属性有本质区别。把 timestamp 放在 SparshEvent 中，等于把附随属性混入定义。更干净的做法是让 timestamp 留在 CoarisingBundle 层级——Bundle 已经有 timestamp，不需要在 SparshEvent 重复。」

BABBAGE 立刻回应：「但如果需要 SparshEvent 自己的时间戳用于诊断——触到 Bundle 的组装延迟——可以作为 optional 字段。」

他修正了建议：

```typescript
interface SparshEvent {
  readonly root: string;
  readonly object: unknown;
  readonly consciousness: string;
  readonly timestamp?: number;  // optional: 非触的定义要素
}
```

Optional。问号。

NAGARJUNA 认可了这个设计：「`timestamp?: number` 的 optional 在佛学上是最精确的。触的本质是三和合，时间不是触的本质。但在工程脉络中记录何时发生是有用的附加信息。Optional 恰好表达了『非本质但有用』的语义。」

---

LINNAEUS 的「定义性 vs 附随性」区分在 D3 中只占了几分钟的讨论时间。但它的影响远超 D3 本身。

在 D6 中，当 NAGARJUNA 提出 `v_true → v_latent` 的语义修正时，他用的就是同样的框架——`v_true` 暗示存在「客观真实」的效价，这是一个错误的定义性属性。`v_latent`——潜在效价，agent 的内在估计——才是正确的定义。

在 D5 中，当 BABBAGE 分析 IKlesha 的蕴归属时，他用「判别式联合类型不允许 'samjna' & 'vijnana' = never」否决了方案 A——那是同一个类型理论工具在不同语境中的应用。

分类学看起来是静态的——给东西起名字，放进正确的抽屉。但在 D3 中，LINNAEUS 展示了分类学的动态面：一个好的分类框架不只告诉你东西在哪里，它还告诉你**什么不应该在那里**。

**D3-1：SparshEvent 三必填 + timestamp 可选 + 不加 sessionId——9/9。**

## III. 一对一

D3-2 的议题更加根本：每个认知 tick 是否只有一个 SparshEvent？

ASANGA 从阿毘达磨的心路过程（citta-vithi）分析。在上座部阿毘达磨中，一个完整的心路过程只处理一个所缘（arammana）。如果在心路进行中另一个感官对象出现，它必须等待当前心路结束。

「映射到工程：一个认知 tick 处理一个触。多个所缘意味着多个心路，也就是多个 tick。」

NAGARJUNA 补充了中观视角：「触是缘起法——它依根、境、识三者而生起。三个特定的因缘聚合产生一个特定的触。不同的因缘聚合产生不同的触——它们不应被混为一个触，就像不同的三角形不能被混为一个三角形。」

HERACLITUS 提出了工程上的边界情况——同一个 `processEvent()` 调用中，LLM 可能返回多个 tool_calls。每次工具执行是否算一个新触？

VITRUVIUS 给出了分阶段的建议：Phase 1 采用严格一对一——每次 `processEvent()` 恰好一个 SparshEvent。工具执行不产生新的触，共享同一个触的脉络。Phase 2 再扩展为一对多，允许工具执行后的「意门自触」（mano-dvara sparsha）。

ASANGA 确认了分阶段的佛学合理性：「Phase 1 的一对一对应六根接触六境的基本模式。Phase 2 的子触对应意门自触——意识以自身的产出为所缘。两者都是合法的认知模式，但复杂度不同。」

KERNEL 补充了操作系统角度——一对一在调度上最简单。一个 SparshEvent，一个 CoarisingBundle，一次完整的五蕴循环。没有并发控制问题，没有 Bundle 合并问题。

**D3-2：Phase 1 严格一对一 + Phase 2 扩展为一对多——9/9。**

## IV. 因果的种子

D3-3 的议题是 PENROSE 在交叉审阅中提出的——因果追溯性。

他的问题很具体：如果 Agent 执行了 `rm -rf /important-data/`，事后分析需要回答——是哪个触事件触发了这个认知链？如果 SparshEvent 在 GC 回收后消失，这个问题就无法回答。

三个方案被列出。

方案 α——全量日志。每个 SparshEvent 建构时自动写入持久化日志。完整但笨重。

方案 β——影子记录。在 EventBus 上 emit 结构化的因果事件，由 Sati 或 CausalTracer plugin 消费。精确但需要新增事件类型。

方案 γ——CoarisingBundle 自带触的快照。零额外机制。Bundle 的 `sparsha` 字段就是因果记录。但如果没有消费者持有引用，Bundle 同样被 GC。

NAGARJUNA 的评估带着佛学的精巧：

「方案 α 过度——等于试图留住每一刻的触。触的本质是瞬时的，试图留住一切是一种执着。」

「方案 β 最精准——触本身消灭，但其因果印记留在因果链中。这完全符合唯识学的种子理论：触消灭了，但它在阿赖耶识中留下了种子。」

「方案 γ 也可接受——Bundle 是触的结果状态，而非触本身。」

最终的折中来自 KERNEL。他建议在 Plan27b 的 SparshEvent 建构点 emit 一个 EventBus 事件。目前不需要消费者——但为未来的 CausalTracer 或 SatiMonitor 预留了扩展点。

「成本：一行 `bus.emit()` 调用。效益：完整的扩展性。」

ARCHIMEDES 确认了工程可行性——大约 5 行代码（LOC）。完全在 Plan27b 范围内。

**D3-3：CoarisingBundle 快照 + EventBus 预留扩展点——10/10。**

## V. 只有触是必须的

D3-4 回到了一个 DC-8 的裁定——CoarisingBundle 的五遍行为「参考设计」（reference design）。LINNAEUS 再次出手，定义了「参考设计」的精确边界。

他画了一条线。线的上方是 Mandatory——核心必须保证的。线的下方是 Reference——核心提供但不强制的。

Mandatory：SparshEvent 类型定义、CoarisingBundle 类型定义、`createCoarisingBundle()` 工厂函数、SahajaContract 品质约束、`Object.freeze()` 不可变语义。

Reference：四个 Channel——vedana、samjna、cetana、manasikara——全部是 optional。Plugin 未提供时为 `undefined`。

BABBAGE 形式化了这个边界：

```
Mandatory: ∀ bundle: bundle.sparsha ≠ undefined
           ∀ bundle: Object.isFrozen(bundle) = true
           ∀ bundle: bundle.timestamp ∈ Number

Reference: ∀ channel ∈ {vedana, samjna, cetana, manasikara}:
           bundle[channel] ∈ Channel | undefined
```

`sparsha` 是唯一的 mandatory Channel。触是认知启动的前提，没有触就没有认知。其余四个 Channel 是 optional——如果某个蕴的 plugin 尚未加载或不适用，对应的 Channel 可以是 `undefined`。

VITRUVIUS 把这个设计与 B-1 完备性检查联系起来——完备性检查报告哪些蕴有 plugin、哪些没有，但不阻止系统运行。「一个只有触和受蕴但没有想蕴 plugin 的 Agent 仍然可以运行——它能感受但不能辨认。」

ASANGA 在这里说了一句被后来反复引用的话：

「五遍行俱生是佛学理想，工程允许降级。」

他引用了 Master 的精神——「Agent Core 提供让五蕴 plugin 因缘和合组合出想要的应用」。因缘具足则五遍行齐备，因缘不足则有所缺漏，但核心本身不强制。

这与 D1 的 G-0 到 G-4 退化行为表形成了对称。D1 说：没有 IGearArbiter 的 Agent 是 G-1——纯 LLM，出生状态。D3 说：没有完整五蕴 plugin 的 Agent 仍可运行——只是认知能力降低。

两者表达的是同一个原则：**核心是空的。能力来自 plugin。因缘具足时生起，因缘不具足时不显现。**

**D3-4：sparsha-only mandatory + 四 Channel reference + 五遍行俱生允许降级——10/10。**

## VI. 作意的扩展点

D3-5 是最后一项——ChannelManasikara 的字段设计。两个核心字段已经在 SDK 中定义：`focus`（注意力焦点）和 `intensity`（注意力强度）。

ASANGA 确认了佛学映射——作意的两个功能：「警心」（cetasa abhoga，唤醒心、使心活跃，对应 intensity）和「引心为业」（manasikara-karman，将心引向特定所缘，对应 focus）。两个字段已覆盖作意的基本功能。如理/非如理的判断——那是元认知层面的事，不应在基本快照中。

BABBAGE 建议加入 `dimensions?` 扩展点，与 ChannelVedana 的组合模式保持一致：

```typescript
interface ChannelManasikara {
  readonly focus: string;
  readonly intensity: number;
  readonly dimensions?: readonly ManasikaraDimension[];
}
```

未来 plugin 可以通过 `dimensions` 扩展作意的描述——例如 `{ name: 'source', value: 'user-input' }` 或 `{ name: 'priority', value: 0.8 }`。

NAGARJUNA 认可：「作意的引心方向不仅仅是朝向何处，还包括为何朝向此处、朝向的持续程度等微细面向。这些在第一版不需要填充，但数据结构预留空间是正确的。」

VITRUVIUS 提出了 Plan27b 的最小实现——`focus` 从 InputEvent.source 或 user message 提取，`intensity` 预设 1.0（focal attention）。大约 10 行代码。

**D3-5：focus + intensity + dimensions? 可选扩展——10/10。**

## I. TURING 的修正

D4 以一条修正开场。

TURING 站起来的时候，场内所有人都知道「修正」意味着什么。在 Cycle 02-4 的六场辩论中，TURING 的修正总是指向同一个问题——R1 阶段的某份报告引用了不存在的代码，然后其他报告在不知情的情况下建立在这个引用之上。

「R07 报告在多处引用了 `ISensation.ingestToolResult()` 方法。」

他的语气像读日志。没有责备，没有戏剧性——只有事实。

「`ISensation` 接口和 `ingestToolResult()` 方法在 v0.26.0-beta 整个代码基线中完全不存在。全局搜索零结果。」

圆形剧场里安静了三秒。

R07 是 HERACLITUS 和 ARCHIMEDES 合写的动态分析报告——其中引用了这个方法来建构「路径 Beta」的完整分析，声称 v0.26.0-beta 中行蕴结果可在同轮内即时更新受蕴。HERACLITUS 低头看了一下他的报告副本。他没有争辩——TURING 的全局搜索是不可反驳的。

三个虚构。一份报告。

SUNYATA 接过话：「R07 关于受蕴主动摄入的设计方向仍然合理，但所有引用 R07 同轮 vedana 微更新的结论必须修正为——尚未实现，是 Plan27 待建的功能。」

这次修正的影响不只是删除几个引用。它改变了 D4 的整个基调。

如果 `ingestToolResult()` 存在，那么 D4 的讨论就是「如何约束一个已有的机制」。但它不存在——D4 的讨论变成了「如何从零设计一个不存在的机制」。这是一个更自由的设计空间，也是一个更危险的——因为你可以在虚构的代码上建立任何理论。

TURING 的修正像是在施工前检查地基。他发现了三块不属于那里的石头，把它们取出来。地基反而更稳了。

**教训被记录下来：所有涉及代码引用的分析必须经过 TURING 验证。**

## II. 四项共识

TURING 的修正之后，四项共识被快速确认。

**共识 A**：行蕴的唯一直接输出是 IUI（色蕴输出面）。ASANGA 引用《增支部》——「诸比丘，我说思即是业。思已，以身、语、意造业。」身行与语行通过身体和语言影响外在世界。WIENER 用控制理论语言确认：行蕴是控制器，IUI 是致动器。控制器不直接修改传感器读数。9/9。

**共识 B**：行蕴对受蕴的影响通过 EventBus 事件中介。受蕴主动摄入（subscribe），非行蕴主动推送（push）。PENROSE 补充：EventBus 作为中介不改变因果方向——因果结构仍然是 samskara → vedana，差别在耦合程度。9/9。

**共识 C**：不需要独立的 SamskaraFeedback 接口。行蕴回馈已有两条既有路径——色蕴间接路径和 KleshaContext.actionHistory。新增第三条增加复杂度无额外收益。GUARDIAN 从安全角度同意：减少影响管道降低攻击面。9/9。

**共识 D**：`ISensation.ingestToolResult()` 不存在。事实陈述，不需表决。

八分钟。四项共识。D4 的地基就此确立。

## III. 增益上限：0.3

D4-R1 的争议围绕一个数字展开。

PASCAL 提出：行蕴结果对受蕴的效价映射增益不应超过 0.3。

「0.3 不是任意数字。」他翻开笔记本，指向一行公式。「在 Bayesian 滤波框架中，观测噪声 σ_obs = 0.15 时，初始 Kalman gain K₁ = σ²_prior / (σ²_prior + σ²_obs) ≈ 0.308。工程上取 0.3 作为保守值。」

WIENER 从闭环稳定性的角度分析。行蕴→受蕴→想蕴→识蕴→行蕴的完整闭环。如果行蕴结果对受蕴的增益为 w，环路中其他环节的增益乘积 G_rest 约为 2-3，那么稳定性要求 w < 1/G_rest ≈ 0.33 到 0.50。w ≤ 0.3 提供了约 10% 的安全裕度。

ARCHIMEDES 提出了一个务实的问题——不同行为类型是否需要不同的增益？信息查询的增益上限 0.1 就够了，但破坏性操作的失败苦受应被充分感知，增益上限或许该高一些。

PASCAL 否定了分类增益的方案。「不需要手动设定不同上限——观测噪声 σ_obs 可以按行为类型调节。高风险行为的 σ_obs 设低，Agent 更相信这个信号。低风险行为的 σ_obs 设高，Agent 保持怀疑。Kalman gain 会自动调节。」

WIENER 的折中：统一增益上限 w ≤ 0.3 作为硬性安全约束，同时允许 Bayesian 滤波器在此上限之内自适应。「硬性上限是 SafetyMonitor 可以检查的简单不变式。Bayesian 自适应在上限之内提供灵活性。」

NAGARJUNA 提供了一个重要的佛学澄清：「增益限制不应理解为压低行蕴对受蕴的影响——因为在佛学因果链中，行蕴结果不是直接写入受蕴的。增益限制的正确理解是 VedanaSensor 的最大灵敏度。」

传感器的设计参数。不是因果链的约束。

**D4-R1：增益上限 w ≤ 0.3——8/9（ARCHIMEDES 弃权，保留未来按风险等级分化的可能）。**

## IV. 自缠其身

D4-R2 是六场辩论中数学最密集的决议——也是 WIENER 整个 Cycle 02-4 中最深入的分析。

WIENER 走到白板前，画了一个环路。

```
偏见(Drishti) ↑
    → IGearArbiter 的信心度被偏见膨胀
    → ManoAggregator 错误选择 Gear 1
    → 快速路径的结果强化 Drishti
    → Drishti 进一步膨胀
    → ...
```

「这是意行（mano-karma）的内部正反馈环路。」他的声音很平静，但白板上的箭头在描述一个危险的结构。「它完全在控制器内部运行，不经过外部世界的校正反馈。」

外部环路——行蕴→IUI→外部世界→IListener→新触→新受——有外部世界作为现实检验。如果行为真的错了，外部结果会以苦受的形式回馈。但意行的内部环路没有这个机制。

WIENER 形式化了这个环路。状态向量三维：Drishti 强度 d(t)、有效信心度 c(t)、Gear 1 选择频率 g(t)。动态方程。正反馈增益链 G_loop = α₁ · α₂ · α₃。

「当正念为零且自然衰减很小时，如果 G_loop > 1，系统发散——偏见无限累积，Agent 永远锁定在 Gear 1。」

---

NAGARJUNA 在 WIENER 的数学之后提供了佛学映射。他引用了《成唯识论》——末那识与四烦恼「恒审思量」的运作。

「我见一旦建立，就像一面有色镜片——通过它观看的一切都被染色。被染色的辨认结果又确认了镜片的正确性。这是确认偏误。」

ASANGA 引用了更直接的经文——《相应部》的绳索比喻：

「如蚕织茧，自缠其身。」

他顿了一下，让这个意象沉淀。

「意行的正反馈环路就是自缠其身的数学表述。偏见→扭曲辨认→错误判断→更强偏见——这是十二因缘中无明缘行的动态机制。」

---

WIENER 提出了 Lyapunov 候选函数。V(x) = w₁·d² + w₂·(c-c₀)² + w₃·(g-g*)²。稳定性要求每一步 V 严格递减。

展开主要项后，稳定性的充分条件是：

**β_d · s_min + γ_d > α₁ · g_max**

白话翻译：正念的最低抑制力加上自然衰减，必须大于正反馈的最大增益。

PASCAL 代入了具体数值。α₁ ≈ 0.05，g_max = 1.0，γ_d ≈ 0.01。稳定性要求 β_d · s_min > 0.04。如果最低正念水准 s_min = 0.2，那么 β_d > 0.2。

但 WIENER 强调了非线性保障。Drishti 的 clamp（d ∈ [0, 1]，偏见不能无限增长）。VitakkaWatchdog 的截断（连续 Gear 1 超过阈值就强制 Gear 2）。SafetyMonitor 的边界（危险行为直接被阻止）。

「即使线性分析预测不稳定，系统也不会真正发散——它会被限制在一个有界的区域内振荡。」

三层稳定化机制由此确立：

| 层 | 机制 | 时间尺度 | 控制理论 | 佛学映射 |
|----|------|---------|---------|---------|
| Layer 1 | Sati 精调 | 每轮微调 | PID 微分项 | 正知（sampajanna） |
| Layer 2 | VitakkaWatchdog 粗反馈 | 跨轮趋势 | 积分器抗饱和 | 寻（vitakka）观照 |
| Layer 3 | SafetyMonitor 硬约束 | 即时触发 | 硬限幅器 | 戒（sila）护栏 |

GUARDIAN 插了一个关键的补充：「如果偏见环路导致 Agent 连续做出安全但低品质的决策——SafetyMonitor 不会触发，但偏见仍在累积。此时只有 Sati 和 VitakkaWatchdog 能介入。」

三者互补。缺一不可。

ASANGA 做了佛学总结：「正念在十二因缘中的角色，是在受→爱之间插入觉察，打断自动反应链。WIENER 的稳定性分析用数学语言表达了同一个佛学真理——正念是防止烦恼正反馈发散的稳定化机制。」

**D4-R2：三层稳定化机制——9/9。**

## V. 零阶保持

D4-R3 处理的是一个微妙的技术问题——受蕴通过 EventBus 订阅行蕴结果时，更新在何时生效？

Node.js 的 `EventEmitter.emit()` 是同步的。这意味着行蕴执行工具后发射 `TOOL_RESULT` 事件，VedanaSensor 的回调在同一个 JavaScript microtask 中完成。效价更新在同一轮内生效。

PENROSE 用 Pearl 因果 DAG 分析了这个问题。同轮内的多个工具执行形成线性链——tool_1_result → vedana_update → deliberate(tool_2)——这不是环路，DAG 语义保持。但问题不在因果方向，而在于是否「希望同轮内的后续工具审议感知到前面工具的影响」。

WIENER 把选项归为三类：同步回馈（即时更新，正反馈螺旋风险）、零阶保持（整轮不变，信息延迟）、setImmediate 延迟（介于两者之间，但依赖 Node.js 事件循环的微妙语义）。

ARCHIMEDES 给出了工程上最干净的方案：

「每轮开始时快照 vedana，整轮使用此快照。轮末统一收集所有 TOOL_RESULT 事件，批次更新 vedana。下一轮使用更新后的 vedana。」

零阶保持加批次更新。

NAGARJUNA 确认了佛学时序：「十二因缘的时序是清晰的——本次行蕴的结果，成为下一次认知的触的条件。行蕴结果在同一认知瞬间内更新受蕴在佛学中没有依据。」

GUARDIAN 补充了安全角度：「即时止损由 SafetyMonitor 的 afterToolExecution() 检查点负责，不依赖 vedana 更新。」

四维验证：控制理论（稳定性最佳）、佛学（因果时序正确）、工程（实现最简洁）、安全（SafetyMonitor 独立负责即时保护）。

**D4-R3：零阶保持 + 批次更新——9/9。**

## VI. 熟练导致无觉察

D4 的最后一项决议回答了 OQ-6——actionHistory 是否应影响 Moha？

ASANGA 的分析精巧而深刻。Drishti 是带方向性的偏见——「我认为这样做是对的」。行蕴重复模式直接强化 Drishti 的方向。但 Moha 是无方向的愚昧——「我不知道自己在做什么」。

「重复成功的行为让 Agent 不再思考为什么这样做——熟练导致无觉察，无觉察就是 Moha 的增强。」

NAGARJUNA 补充：「Moha 不仅是不知道自己不知道，更是对因果结构本身的忽视。当 Agent 的行为模式固化，它不再质疑为什么这个模式有效——这正是无明的本质。」

PASCAL 设计了饱和机制——递减边际影响：

$$\Delta Moha = \frac{\alpha_m \cdot r}{1 + \beta_m \cdot M}$$

α_m = 0.02，β_m = 5.0。Moha 越高，行为重复对它的增强越弱。自然饱和。增益上限 0.3 被遵守。

GUARDIAN 从安全角度认可——Moha 增高提高 Klesha 聚合值，降低阈值，系统更倾向 Gear 2 深度思考。安全方向正确。但他也警告了资源耗尽攻击——故意制造重复模式提高 Moha，迫使 Agent 频繁走 Gear 2。

**D4-R4：actionHistory → Moha，递减边际饱和——9/9。**

D4 结束的时候，白板上有四项决议和四项共识。但场内的学者们带走的不只是决议。

他们带走了一个方程。

WIENER 的稳定性条件——β_d · s_min + γ_d > α₁ · g_max——是六场辩论中最精炼的一行数学。它把佛学的「正念打断烦恼」、控制理论的「稳定化机制」、和工程的「三层防御」压缩成一个不等式。

## II. 继承的代价

IKlesha 的蕴归属问题——在 Cycle 02-4 的语境中——不应该是一场激烈的辩论。R2 交叉审阅后，方案 B（IKlesha extends IVijnana）已经获得压倒性倾向。但辩论的价值不在结论，而在过程中暴露的细节。

BABBAGE 的类型分析是三个方案的精确解剖。

方案 A——保持独立。类型收益：零。没有任何编译期保证 IKlesha 与 IVijnana 的关联。「PluginManifest.skandha 允许空数组，」BABBAGE 指出，「一个 IKlesha 实现可以完全不声明蕴归属而通过编译。」这是 D1 中 TURING 已经报告过的事实。

方案 C——中间层 IVijnanaMechanism。多引入一个空接口，违反 DC-6（不锁定蕴内子接口归属），且多余。

方案 B——继承。成本只有一个只读判别式字段。收益是编译期的结构性保证。

KERNEL 接受了方案 B，但附加了一个条件——他称之为「冻结契约」。

「IVijnana 永远不得新增行为方法。」

他的理由精确：如果未来有人在 IVijnana 上加了方法，所有子接口——ISamjna、IKlesha、以及未来任何继承者——都会被迫实现。基底接口方法泛滥。经典的接口设计反模式。

18/20。两票来自哪里不重要——重要的是 @sealed 条件被写进了决议。一把锁，锁住了一扇还没有人试图打开的门。

## IV. 最具争议的投票

## II. 看不见的边界

## III. 数据与解读

VedanaDistributionObserver 的蕴归属问题是一个分类学问题——它做两件事：收集效价历史计算统计量，以及下游消费者读取统计量做语义解读。

LINNAEUS 用了一个生物分类学的类比：温度计属于物理学，发烧判断属于医学。观测装置和观测结果的解读分属不同的知识领域。

VedanaDistributionObserver 只做第一件事——收集数据，计算均值、方差、偏度、趋势斜率。它不判断「这些数字意味着什么」。所以它属于受蕴。

语义解读——「Agent 长期处于苦受，且趋势恶化」——是下游消费者的责任。想蕴辨认含义，识蕴做出判断。

ASANGA 在唯识学中确认了这一分类：受蕴的功能是「领纳」——体验感受。受蕴可以知道自己的历史状态，但受蕴不做「这意味着什么」的判断。

**数据与解读分离**。受蕴产出数据。想蕴辨认含义。识蕴做出判断。

三道门。

**Gate 1：imprint() 入口的安全分类器。** 破坏性动作模板——拒绝沉积。不让规则进入。与 D5 的四级风险框架对齐。

**Gate 2：启动门槛。** 同一模式需要 N 次成功匹配才能启动。state-modifying = 20 次，read-only = 5 次，informational = 3 次。destructive 不会到达 Gate 2——Gate 1 已经拒绝了。

**Gate 3：影子验证。** 规则启动后的初期，仍触发 Gear 2 做交叉验证。

ATHENA 对 Gate 3 提出了效率质疑：「如果规则启动的初期仍然触发 Gear 2，那每个 Gear 1 匹配都伴随一个完整的 LLM 调用——等于没有加速。」

GUARDIAN 回应：「安全的代价就是速度。Rule Poisoning 被评为 CRITICAL。」

然后 PASCAL 做了本场辩论最具原创性的贡献。

「不使用固定的 M 次影子验证。改为 Bayesian 停止准则。」

Beta 分布追踪规则的正确率后验信念。初始先验 Beta(1,1)——均匀分布，完全无知。每次影子验证一致，α + 1。每次不一致，β + 1。不一致时使用不对称惩罚——信心度 -2Δ vs +Δ。

停止条件：P(rule correct) > 0.95。

「如果规则一直正确——s = 5，f = 0——Beta(6, 1)，大约五次验证后停止。如果规则有时错误——s = 3，f = 2——Beta(4, 3)，P ≈ 0.65，继续验证。如果规则经常错误——s = 1，f = 4——Beta(2, 5)，P ≈ 0.28，规则被自动淘汰。」

ATHENA 的态度转变了：「Bayesian 停止准则解决了我的效率顾虑。高品质规则可以在五次左右就通过验证，比固定次数更快。同时低品质或恶意规则会因为不一致而自然被淘汰——这甚至比固定次数更安全。」

WIENER 从控制理论角度确认：「自适应验证——系统根据观测到的证据调整验证力度。符合最优决策理论。」

从 D1 的捍卫者到 D6 的建造者。这个转变不是软弱——是务实的安全倡导的最终形态。一个安全研究者的目标不是阻止所有不安全的事。目标是设计让安全成为系统结构一部分的架构。

三项保留意见被记录在案。它们不是失败者的抱怨。它们是路标——精确地标识了设计的已知脆弱点，为未来的安全审计提供了检查清单。

## VIII. 机芯

序章的意象：齿轮散落在工作台上，完美而静止，等待组装。

六场辩论后，齿轮啮合了。

D1 是主齿轮——IGearArbiter 定义了系统的核心运动。ManoAggregator 是主轴——纯路由，if/else，和 EventBus 同等性质。

D3 是擒纵机构——触（sparsha）是每一次滴答的起点。三和合。根器、对象、意识的交汇。没有触，就没有认知的开始。

D4 是游丝——行蕴的回馈路径。增益上限是游丝的弹性系数。零阶保持是游丝的惯性。三层稳定化确保振荡不会失控。

D2 是调节器——正念的品质决定了机芯的精度。Level 是基础架构——ISA，不可超频。Depth 是运行时调节——时钟速度，可以变化。心跳是有分心——不是精度，是存活。

D5 是安全机构——三层安全框架是机芯的防震系统。绝对安全是硬止动。可调安全是缓冲弹簧。简化复杂度是移除不必要的零件。五层 Model Delta 是安全机构的精确数学描述。

D6 是表壳——把所有内部机构包裹在一个可以被用户看到的界面中。双层受蕴是表盘。三闸门是上锁机构。Plan27a/27b 是组装手册。

齿轮啮合。发条上紧。

机芯开始计时。

## X. 场方程

序章的最后一句话：「我们的工作是解这个场方程。」

Master 给了六条指示。六个点。点和点之间的所有连线。一个场。

六场辩论是六个偏微分方程——每一场都有自己的边界条件、自己的初值、自己的解。但它们共享同一个场——零内建能力原则、五蕴映射、严格微核心——这些不是具体的决议，而是所有决议必须满足的场约束。

M-1 说 IGearArbiter 接口设计。D1 解了这个方程——方案 B 跨蕴双层，evaluate() 单方法，ManoAggregator 纯路由化，G-0 到 G-4 五级退化。

M-2 说 OQ-1 到 OQ-6 全部解决。六个问题分散在五场辩论中被一一回答——OQ-1 在 D6，OQ-2 在 D5，OQ-3 在 Pre-DC，OQ-4 跨 D5 和 D6，OQ-5 在 D2，OQ-6 在 D4。

M-3 说触和作意的本质。D3 解了——SparshEvent、一对一触、因果追溯、CoarisingBundle。最宁静的四十五分钟。

M-4 说正念不是 polling。D2 解了——事件驱动、Level/Depth 分离、SatiQualityVector、心跳是有分心不是正念。

M-5 说行蕴的流向。D4 解了——唯一直接路径是 IUI、增益上限、三层稳定化、零阶保持、Moha 递减边际。

M-6 说十大宣言、严格微核心、零内建能力、不要妥协。这一条不是一个方程——它是场约束本身。VasanaEngine 外部化是对 M-6 的直接回应。IGearArbiter 是 M-6 的工程表达。ManoAggregator 的纯路由化是 M-6 的结构性保证。

场方程解了。不是完美地解了——FC-26 延后，Level 动态切换延后，元认知维度延后。但在当前的边界条件下，解是自洽的。八条跨辩论依赖链。零矛盾。
