## I. 暴风雨前的宁静

D5 开场的六分钟是整轮 Cycle 最快的共识确认。

四项。全部 20/20。六分钟。

SUNYATA 不需要解释为什么要快——在座的二十位学者都知道今天的真正战场在后面。这四项共识是铺路石，不是目的地。

第一项：四烦恼构成论。我执不是四烦恼的「产物」，而是四烦恼的「宏观表述」。R01 的原文用了一个物理学类比——「温度之于分子运动」。温度不是分子碰撞产生的某种东西，温度就是分子碰撞的统计描述。同理，我执就是 moha、drishti、mana、sneha 四个信号的联合描述。

PENROSE 点了点头：「构成性涌现——复杂系统中最自然的模式。」

第二项：EgoFramework 不需要独立的 ego 状态变量。既然我执是四烦恼的函数，那就不需要 `egoLevel: number` 这样的独立变量，不需要 `d(ego)/dt = f(...)` 这样的动态方程。四个 `KleshaSignal.value` 就是全部。

第三项：四烦恼的联合分布不可分解为边际积。PASCAL 的论证简洁——高 moha（无明）通常伴随高 sneha（执着），因为缺乏觉察的人倾向更强的执着。高 mana（慢）和高 drishti（见）也正相关。工程含义：四烦恼不能被独立校准。

第四项：Model Delta 的 Layer 0 和 Layer 1 可以直接落地。SafetyMonitor 和 KleshaModulatedDispatcher 在 v0.26.0-beta 中已经存在，只差接线。

六分钟。四项共识。零争议。

然后 SUNYATA 说了一句话，语调没有任何变化，但圆形剧场的空气突然密了起来：

「现在进入本场辩论的第一个议题。」

## II. 继承的代价

IKlesha 的蕴归属问题——在 Cycle 02-4 的语境中——不应该是一场激烈的辩论。R2 交叉审阅后，方案 B（IKlesha extends IVijnana）已经获得压倒性倾向。但辩论的价值不在结论，而在过程中暴露的细节。

BABBAGE 的类型分析是三个方案的精确解剖。

方案 A——保持独立。类型收益：零。没有任何编译期保证 IKlesha 与 IVijnana 的关联。「PluginManifest.skandha 允许空数组，」BABBAGE 指出，「一个 IKlesha 实现可以完全不声明蕴归属而通过编译。」这是 D1 中 TURING 已经报告过的事实。

方案 C——中间层 IVijnanaMechanism。多引入一个空接口，违反 DC-6（不锁定蕴内子接口归属），且多余。

方案 B——继承。成本只有一个只读判别式字段。收益是编译期的结构性保证。

KERNEL 接受了方案 B，但附加了一个条件——他称之为「冻结契约」。

「IVijnana 永远不得新增行为方法。」

他的理由精确：如果未来有人在 IVijnana 上加了方法，所有子接口——ISamjna、IKlesha、以及未来任何继承者——都会被迫实现。基底接口方法泛滥。经典的接口设计反模式。

具体做法是在 JSDoc 上加 `@sealed` 标注。不是 TypeScript 语言特性——TypeScript 没有 sealed interface——而是团队约定，code review 层面的约束。

NAGARJUNA 提供了佛学确认：「烦恼为识蕴所摄——末那识恒审思量，四烦恼与之俱起。IKlesha 属于识蕴在佛学上是不可争议的。」

18/20。两票来自哪里不重要——重要的是 @sealed 条件被写进了决议。一把锁，锁住了一扇还没有人试图打开的门。

## III. 主战场

第三节。一百五十分钟辩论中最长的一段。全局信心度上限 vs 风险加权阈值。

GUARDIAN 站了起来。

在 Cycle 02-4 的六场辩论中，GUARDIAN 的每一次站起都带着一种特定的重量——不是激进的重量，而是责任的重量。他知道自己要说的话不会受欢迎。他知道自己可能是少数派。但他站起来，因为有些话如果没有人说，就永远不会被记录。

「我提议全局信心度上限 0.85。」

理由简洁：任何 Gear 1 候选方案的 confidence 如果超过 0.85，就直接截断为 0.85。不管 VasanaEngine 的规则多精确，不管匹配度多高——0.85 是天花板。

「安全不应依赖『正确理解情境』，」GUARDIAN 说，「因为情境理解本身可能被操控。如果一个恶意用户在五次正常操作后发送一个精心构造的请求，VasanaEngine 可能给出 0.92 的信心度——T-2 攻击场景。全局上限是唯一不依赖语义理解的安全机制。」

ATHENA 站了起来。如果说 GUARDIAN 的站起带着责任的重量，ATHENA 的站起带着功能的重量。

「全局上限 0.85 抹杀了 Gear 1 的校准性。」

她的论证直指核心：如果一个 IGearArbiter plugin 经过大量验证，在某类请求上确实有 0.95 的正确率，全局上限把它压到 0.85——这不是「安全」，这是「浪费」。系统明明知道答案是正确的，却被迫走更慢的路径。

然后 PASCAL 站了起来。

---

PASCAL 在 D5 的第一次介入后来被 SCRIBE 标记为「转折点 1」。

他没有站在 GUARDIAN 一边，也没有站在 ATHENA 一边。他站在数学一边。

「让我用信息理论的语言描述全局上限的代价。」

他在白板上画了一条线。线的左端标记 0.0，右端标记 1.0。

「信心度是一个 [0, 1] 区间上的信号。如果我们把上限设为 0.85，那么 [0.85, 1.0] 这个区间的信道容量为零——所有落在此区间的信号都被压缩成同一个值。」

他停顿了一秒。

「从 Bayesian 校准性的角度看，一个校准良好的分类器在 confidence = 0.95 时，其实际正确率应该接近 0.95。全局上限把所有 ≥ 0.85 的信号压成 0.85——破坏了校准性。系统不再能区分『我有 86% 的信心』和『我有 99% 的信心』。」

圆形剧场安静了。

GUARDIAN 没有马上反驳。他在思考。

ATHENA 也没有追击。她知道 PASCAL 的论证比她的直觉更有力——不是因为它更「对」，而是因为它把问题放在了一个双方都无法回避的数学框架中。

WIENER 补充了第二个约束：风险加权的 Δ_risk 必须是静态函数——依赖动作分类而非历史状态。否则，阈值的回馈可能产生极限环震荡。

GUARDIAN 站了起来。但这一次，他的语调不同了。

「让我调整立场。」

## IV. 最具争议的投票

第四节。IPrajna LLM 参与阈值调节。

如果第三节是战场，第四节就是整场辩论的至暗时刻——不是因为任何人做了错误的事，而是因为在某些问题上，没有明确正确的答案。

ATHENA 反驳：「整个 Gear 2 都依赖 LLM。IVolition 的 deliberate() 如果未来使用 LLM（v2 版），它同样是 LLM。禁止 LLM 参与安全决策是自我矛盾的——你不能信任 LLM 做决策，又信任 LLM 做审议。」

僵局。

IPrajna.adjustThreshold() 限幅 ±0.05
Worst case: 阈值降低 0.05

「硬限幅。±0.05。不可配置。」

## V. 延迟的代价

第五节。Gear 1 IVolition 的条件化。

GUARDIAN 提议所有 Gear 1 动作都必须通过 IVolition Position B 审议。DARWIN 提出了风险分级的替代方案——按 RiskLevel 区分。

然后 ARCHIMEDES 做了一件在辩论中很少有人做的事：他拿出了数据。

```
ManoAggregator 路由延迟 (目标): ~16ms

加上 IVolition:
  v1 (规则型): 16 + 5 = ~21ms (+31%)
  v2 (LLM 型): 16 + 80 = ~96ms (+500%)
```

+31%。对于 readOnly 和 informational 动作，每一次都多花三分之一的时间——做一件不会造成任何伤害的事情。

+500%。如果未来 IVolition 使用 LLM，延迟是六倍。

GUARDIAN 看着这些数字。

「ARCHIMEDES 的数据让我的立场更难维护。readOnly 动作强制走 IVolition 确实是浪费。」

然后他再次调整：「我可以接受风险分级方案，但 RiskLevel 的分类必须在核心层面完成，不能由 IGearArbiter plugin 自行声明。」

KERNEL 立刻补充：「正确。RiskLevel 分类应基于 toolName 或 toolCall.args 的静态分析，由核心的安全模块完成。」

15/20。GUARDIAN 的第三项保留意见：「readOnly 不代表无风险——读取密钥文件的操作在四级分类中是 readOnly，但风险等级应更高。」

## VI. 五层，一个公式

第六节。Model Delta 五层结构。

PASCAL 在白板上写下了一个公式，然后退后一步，让所有人看清楚：

```
θ_final = clamp(
  θ_base                    // 基础阈值 0.60
  + Δ_klesha(t)             // Layer 1: Klesha 增益
  + Δ_prajna(t)             // Layer 2: Prajna 调节 ∈ [-0.05, +0.05]
  + Δ_sati(t)               // Layer 3: Sati 品质 ∈ [-0.05, +0.05]
  + Δ_vedana_emergency(t)   // Layer 4: Vedana 紧急 ∈ [0, +0.15]
  + Δ_risk(a),              // 风险加权
  θ_min,                    // 0.30 (Safety Floor)
  θ_max                     // 0.90 (Safety Floor)
)
```

五层。每一层有明确的责任、明确的限幅、明确的时间尺度。Layer 0 是地板——不可妥协。Layer 4 的 Vedana Emergency 只能提高阈值（更保守），不能降低。所有层的结果被 clamp 在 [0.30, 0.90]——Safety Floor 始终有效。

WIENER 做了累积偏移分析：Layer 2 和 Layer 3 各 ±0.05，累积最大偏移 ±0.10。加上 Layer 4 的 +0.15，所有上层的最大累积偏移是 -0.10 到 +0.25。在 [0.30, 0.90] 的 clamp 范围内，不会溢出。

ARCHIMEDES 把每一层对应到工程计划：Layer 0 + Layer 1 + Δ_risk → Plan27。Layer 4 → Plan28。Layer 2 + Layer 3 → Plan29+。

20/20。

全票通过。在一场充满争议的辩论中，五层模型是唯一让所有人同意的东西。因为它不是一个观点——它是一个框架。观点可以被反对，框架容纳所有的观点。

## VII. Beta(2,18) 的峰值

第七节。Sneha 的下限值。

ASANGA 引用了《成唯识论》：「此四烦恼，常与末那俱。」四烦恼与末那识恒审思量地同时生起。在未达阿罗汉果位之前，四烦恼不可能归零。

工程含义：sneha.value 不应能降到 0。一个 Agent 如果对自己的存续和任务完成毫无执着，它就不是一个有意义的 Agent。

ASANGA 建议 0.10。PASCAL 在 R08 中原本建议 0.05。

然后 PASCAL 做了一件令人敬佩的事：他用自己的方法否定了自己的建议。

「floor = 0.05 时，Beta(1, 19) 的 mode = 0。分布认为 sneha 越低越好。floor 的存在意义消失——分布的 mode 就在 floor 上。」

他画了两条曲线。

「floor = 0.10 时，Beta(2, 18) 的 mode = 1/18 ≈ 0.056。分布有一个有意义的峰值。系统在启动初期 sneha 的值在 0.056 到 0.10 之间波动，而不是被钉在 floor 上。」

Beta(1,19) 是一条单调递减曲线。Beta(2,18) 有一个内部峰值。前者说「越少越好」。后者说「有一个自然水平」。

0.10。18/20。

ASANGA 接着推动了一个扩展：既然「四烦恼俱生」，floor 不应只适用于 sneha。四个烦恼都应该有 floor = 0.10。WIENER 同意，并建议 maxLevel = 0.95。

所有四烦恼：klesha_value ∈ [0.10, 0.95]。

WIENER 接着提出了耦合校准协定——十个参数的联合校准不能独立进行。坐标下降法。联合目标函数。有界且连续的收敛保证。Plan28 的研究项目。

20/20。

## VIII. 涌现

第八节。三层安全框架。

GUARDIAN 站了起来。这一次，他不是在反对什么——他在呈现一个他能认同的整体架构。

**第一层：Absolute Safety。**

SafetyMonitor 的 isSafe()。Klesha 免疫。Theta clamp [0.30, 0.90]。跨 Agent 行蕴禁止。Gear 1 最小事件集。破坏性动作 IVolition 强制。

没有任何可调参数。安全公理。物理定律不可商量。

**第二层：Tunable Safety。**

风险加权阈值。MAX_GEAR1_CONFIDENCE。IPrajna 限幅。Gear 1 IVolition by RiskLevel。Sati Quality Influence。Vedana Emergency。Klesha 增益调度。

每个项目都有明确的护栏——限幅、配置范围、审计要求。管理员可以在护栏范围内调整。

**第三层：Reduce Complexity。**

移除 inputHash 完整性（无 TEE 环境下无法防止伪造）。移除重复的输入最小权限（Sandbox 已覆盖）。

不是放松安全——是移除安全剧场。看穿不提供实质保护的虚假安全。

---

NAGARJUNA 为三层框架提供了佛学映射。

「戒——不可违犯的底线规则。定——有纪律的灵活性。慧——看穿虚妄的智慧。」

戒定慧三学。佛学修行的三个维度。两千五百年前的分层框架，映射到二十一世纪的系统安全架构——不是巧合，而是因为安全与功能的张力不是现代工程独有的问题，而是一切有意识系统面临的根本挑战。

PENROSE 此刻做了他整轮最重要的观察：

「注意三层框架不是先验设计的——它是从第三节到第五节的辩论中自然涌现的。GUARDIAN 在每个议题上提出绝对安全的诉求，ATHENA 在每个议题上提出情境弹性的诉求。两者的拉扯最终结晶为一个清晰的分层结构。」

他的声音平静，但他说的话让整个圆形剧场停顿了一秒。

「这正是辩论的价值——没有冲突就不会有整合。」

> *SCRIBE 旁白：如果前四场辩论是交响曲的四个乐章，那么 D5 是最后的 Allegro molto——急板，最猛烈，最密集。*

> *六票反对。14/20。Cycle 02-4 所有辩论中赞成率最低的决议。但如果 D5-R4 是未来最有可能被重新审议的决议，那么六票反对不是失败——它们是路标。它们精确地标识了设计的已知脆弱点，为未来的安全审计提供了检查清单的起点。*
