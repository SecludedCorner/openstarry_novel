# 第一章：没有引擎的车也能开

---

## 齿轮仲裁者属于谁？

第一场辩论（D1）的问题很具体：IGearArbiter 这个新接口，属于五蕴中的哪一个？

五蕴是这个系统的五大模块——色（感官）、受（感受）、想（辨认）、行（行动）、识（判断）。每个接口都必须归属于某个蕴。新的齿轮仲裁者归谁管？

三个方案摆在桌上。

方案 A：它属于想蕴（负责辨认的那个蕴）。方案 C：它属于识蕴（负责判断的那个蕴）。方案 B：不定死——让实现的人自己决定。

BABBAGE 用他最擅长的工具——类型代数——把这个问题变成了一道数学题。

「假设 IGearArbiter 同时继承想蕴和识蕴的接口。想蕴的标签是 'samjna'，识蕴的标签是 'vijnana'。在 TypeScript 里，一个值不可能同时是两个不同的字符串。'samjna' AND 'vijnana' = never。空集。不存在。」

就像你不能同时是苹果又是橘子。类型系统不允许。

BABBAGE 的论证只用了三行。但这三行让十九个人同时点头——因为数学不需要你相信。它需要你检查。每个人都可以自己检查那三行。结论不依赖信任，依赖逻辑。

所以方案 A 和方案 C 都有问题——它们把归属定死了。方案 B 不定死归属，让实现者根据自己的逻辑选择。一个基于规则匹配的实现可以自称想蕴；一个基于统计推断的实现可以自称识蕴。

20/20。全票通过。这是 Cycle 02-4 第一个满分决议。

NAGARJUNA 补充了哲学角度：「方案 B 和中观哲学的核心一致。一个法的本质不由它的名字决定，而由它的功能和因缘决定。」翻成白话：你不是因为你的名字才是你——你是因为你做的事才是你。

## 一个方法就够了

GUARDIAN 想要两个方法——evaluate()（纯判断）和 resolve()（纯行动）。安全分离。

KERNEL 摇头。他是操作系统专家——他见过太多因为「安全分离」而变得臃肿不堪的接口。

「不需要。EventBus 的同步事件已经提供了观测点。每次齿轮切换，系统会发射一个事件，任何安全监控器都可以拦截。不需要在接口上多开一道门。」

他补了一句：「如果每个接口都做这种分离，最后整个系统所有接口都有双方法。那不是安全，是仪式。」

最后的设计：evaluate() 一个方法，加上三个同步事件——`gear:switch`（齿轮切换时）、`action:proposed`（行动方案产生后）、`action:executed`（行动完成后）。三个事件形成一条完整的安全链：预防、审查、审计。

GUARDIAN 的方案被否决了，但他的核心诉求——「我要能观察到发生了什么」——以不同的形式被保留了。三个事件 = 三个窗口。GUARDIAN 透过这三个窗口可以看到一切。他不需要自己的门。

这个模式在后续的辩论中反复出现：GUARDIAN 提出一个安全方案，被否决，然后他的核心诉求以修改后的形式重新出现。不是失败者的执着——是安全工程师的专业素养。好的安全设计师不是什么都阻止的人。是那个确保「即使我被否决了，我担心的风险仍然被覆盖」的人。

## 拔掉所有 plugin

D1 最重要的决议：ManoAggregator 变成纯路由器。

以前的 ManoAggregator 像一个交通指挥官——收集各路信号，做出综合判断。现在它变成了一个红绿灯：如果有 IGearArbiter 且信心度够高，绿灯（走 Gear 1 快速路径）；否则红灯（走 Gear 2 让 LLM 仔细想）。

这个改变看起来很小——从「智慧判断」变成「简单判断」。但它意味着核心不再包含任何「能力」。核心只做转发。所有能力都在 plugin 里。

五级退化行为：

| 状态 | 发生了什么 | 系统怎么做 |
|------|----------|----------|
| G-0 | 没装 plugin | 纯 LLM 模式 |
| G-1 | plugin 坏了 | 同上，记录错误 |
| G-2 | plugin 不太确定 | 走 LLM |
| G-3 | plugin 出错 | 走 LLM，记录异常 |
| G-4 | plugin 太慢 | 走 LLM，超时切换 |

G-0 是最重要的——一个什么 plugin 都没装的 Agent，就是一个纯 LLM Agent。它仍然能工作。就像一辆没装导航系统的车——你仍然可以开，只是每个路口都需要自己看路牌。没有人会说一辆没装导航的车是「坏掉的车」。它只是一辆更基本的车。

NAGARJUNA 从佛学角度说了一句很精准的话：「G-0 不是退化。G-0 是原初。一个没有习气的 Agent 不是残缺的——它是初生的。」

零内建能力的终极测试：拔掉所有 plugin，系统仍然活着。20/20。

Gear 1 快速路径还需要一组最小事件——`gear:switch` 和 `action:proposed`。没有这两个事件，快速路径就是一条没有红绿灯的高速公路。太危险了。GUARDIAN 立刻指出了这一点。KERNEL 同意。

还有一个细节：evaluate() 方法有一条严格的「纯度契约」——不能产生副作用、不能修改输入、不能做 I/O。翻成白话：evaluate() 只能看，不能摸。它可以告诉你「我觉得应该走快速路径」，但它不能自己动手。

TURING 搜了一下现有代码——好消息，现有的 VasanaEngine.evaluate() 已经符合这条约束。理论和实践是一致的。

> *SCRIBE 旁白：D1 用了九十分钟。九项决议。最重要的不是任何一项决议，而是一个观念的改变——引擎被拆出来了。核心变得更轻、更简单、更干净。就像整理房间——真正重要的东西反而更容易找到了。*
