# 序章：拆掉引擎

---

圆形剧场的灯维持着工作亮度。二十盏。一盏都没有暗。

这个剧场已经亮了五轮了。每一轮，二十个人坐在这里辩论。每一轮结束，灯暗下来。几天后，灯再亮起。然后新一轮开始。

三天前——Cycle 02-3 结束时——SUNYATA 说了一个比喻。六场辩论像六个齿轮，被精心切削完毕。齿距正确，轴心正确。等着被组装成一座钟。

三天。

三天里没有正式辩论。但三天里发生了一件事——一件在正式辩论前就改变了一切的事。

---

GUARDIAN 把安全备忘录读了三遍。

不是随便翻。是那种用红笔在关键字下面划线、每读一遍线就更粗的读法。

第三遍读完时，他在笔记本上写了三行：

- VasanaEngine 有规则库
- 规则库是能力
- 能力不应该在核心里

如果你在修一辆车，你会把引擎盖打开，看到里面有个零件——它有自己的记忆、自己的判断力、自己的决策能力。你会问：这个零件为什么在引擎里？它不是引擎的一部分。它是一个独立的装置，碰巧被塞在引擎里面了。

VasanaEngine 就是那个零件。它在核心里，但它不属于核心。

---

NAGARJUNA 在同一时间想到了同一个问题，但用了完全不同的语言。他不知道 GUARDIAN 的分析。GUARDIAN 也不知道他的。两个人在不同的房间里，用不同的语言，推导出了同一个结论。

「习气是后天熏习的。」他闭着眼睛说。

翻译成白话：你的习惯是你活出来的，不是你生下来就有的。一个婴儿没有习惯。一个刚出生的 AI Agent 也不应该被迫拥有习惯。

如果 VasanaEngine（习气引擎）是核心的一部分，那就等于说每一个 Agent 出生时就自带一套习惯。这不对。

两个人。不同的出发点。同一个终点。GUARDIAN 从安全出发——核心不该有能力。NAGARJUNA 从佛学出发——习气是后天的。结论相同：拆掉它。

---

SUNYATA 把矛盾写在白板上。然后他做了一件从来没做过的事——他按下通讯面板，直接联系 Master。

「VasanaEngine 不应该在核心中。」

Master 的回答只有两个字：「是的。」

那个「是的」不是惊讶。是「我一直在等你们自己发现这件事」。

然后 Master 说：「如果 VasanaEngine 不在核心里，ManoAggregator 就退化为纯路由。if/else。像 EventBus 一样——只是一个转发站。」

SUNYATA 听到这句话时停了一下。「纯路由」意味着核心几乎什么都不做——只是看看有没有 plugin 可以处理，如果有就交给它，如果没有就交给 LLM。整个核心的复杂度从一百行缩减到二十行。

VasanaEngine 被拆出来了。它有了一个新名字：**IGearArbiter**——齿轮仲裁者。一个可以安装也可以拆掉的 plugin。

就像手机上的 App。你可以装一个导航 App，也可以不装。不装的话手机还是手机——你只是需要自己看路牌。装了的话你有额外的便利。但导航 App 不是手机的核心——它不应该被焊在主板上。

同一场对话还顺便解决了一个拖了很久的问题：需不需要形式化区分「必装 plugin」和「选装 plugin」？答案意外地简单——不需要。之前 Cycle 02-2 已经设计了「五蕴完备性检查」，那个检查已经覆盖了这个需求。不需要新方案。

SUNYATA 后来在记录里写了一行：「提出新方案前先回顾既有决议。」

这个教训比看起来更重要。在一个有五个 Cycle、数百项决议的研究项目里，最浪费时间的事不是「想不出答案」，而是「忘了答案已经存在」。

---

PASCAL——团队最新的成员——在另一间办公室里翻阅前一轮的报告。他在 Cycle 02-3 加入团队，专长是决策理论和概率。他的笔记本上画满了 Beta 分布的图形。他不知道这些图形会在三天后的辩论中三次打破僵局。

---

第四封信在第三天傍晚出现。

前三封信改变了「什么」「怎么修」「怎么动」。第四封信改变的是——**怎么变成一个整体。**

六条指示：

- M-1：把 VasanaEngine 拆出来，设计新接口
- M-2：六个拖了很久的问题，全部解决
- M-3：触和作意——它们是事件？还是因果的起点？
- M-4：正念不是定时检查
- M-5：行蕴的结果去了哪里？比看起来更重要
- M-6：十大宣言。严格微核心。零内建能力。不要妥协。

M-6 最严厉。「不要妥协」三个字。Master 在之前的信里从来没有用过这种语气。这三个字意味着：前几轮有些地方妥协了，这次不行。

六条指示。六场辩论。但不是六个独立的问题。

SUNYATA 站在二十盏灯前面说：「Master 给了我们一个场。像一张地图——每一个点上都有值。我们的工作不是解六个方程式。是解一个场方程。」

> *SCRIBE 旁白：第四封信和前三封不同。前三封像医生开的处方——这里痛，吃这个药。第四封像一张 X 光片——你看到的不是症状，而是整个骨架。*

# 第一章：没有引擎的车也能开

---

## 齿轮仲裁者属于谁？

第一场辩论（D1）的问题很具体：IGearArbiter 这个新接口，属于五蕴中的哪一个？

五蕴是这个系统的五大模块——色（感官）、受（感受）、想（辨认）、行（行动）、识（判断）。每个接口都必须归属于某个蕴。新的齿轮仲裁者归谁管？

三个方案摆在桌上。

方案 A：它属于想蕴（负责辨认的那个蕴）。方案 C：它属于识蕴（负责判断的那个蕴）。方案 B：不定死——让实现的人自己决定。

BABBAGE 用他最擅长的工具——类型代数——把这个问题变成了一道数学题。

「假设 IGearArbiter 同时继承想蕴和识蕴的接口。想蕴的标签是 'samjna'，识蕴的标签是 'vijnana'。在 TypeScript 里，一个值不可能同时是两个不同的字符串。'samjna' AND 'vijnana' = never。空集。不存在。」

就像你不能同时是苹果又是橘子。类型系统不允许。

BABBAGE 的论证只用了三行。但这三行让十九个人同时点头——因为数学不需要你相信。它需要你检查。每个人都可以自己检查那三行。结论不依赖信任，依赖逻辑。

所以方案 A 和方案 C 都有问题——它们把归属定死了。方案 B 不定死归属，让实现者根据自己的逻辑选择。一个基于规则匹配的实现可以自称想蕴；一个基于统计推断的实现可以自称识蕴。

20/20。全票通过。这是 Cycle 02-4 第一个满分决议。

NAGARJUNA 补充了哲学角度：「方案 B 和中观哲学的核心一致。一个法的本质不由它的名字决定，而由它的功能和因缘决定。」翻成白话：你不是因为你的名字才是你——你是因为你做的事才是你。

---

## 一个方法就够了

GUARDIAN 想要两个方法——evaluate()（纯判断）和 resolve()（纯行动）。安全分离。

KERNEL 摇头。他是操作系统专家——他见过太多因为「安全分离」而变得臃肿不堪的接口。

「不需要。EventBus 的同步事件已经提供了观测点。每次齿轮切换，系统会发射一个事件，任何安全监控器都可以拦截。不需要在接口上多开一道门。」

他补了一句：「如果每个接口都做这种分离，最后整个系统所有接口都有双方法。那不是安全，是仪式。」

最后的设计：evaluate() 一个方法，加上三个同步事件——`gear:switch`（齿轮切换时）、`action:proposed`（行动方案产生后）、`action:executed`（行动完成后）。三个事件形成一条完整的安全链：预防、审查、审计。

GUARDIAN 的方案被否决了，但他的核心诉求——「我要能观察到发生了什么」——以不同的形式被保留了。三个事件 = 三个窗口。GUARDIAN 透过这三个窗口可以看到一切。他不需要自己的门。

这个模式在后续的辩论中反复出现：GUARDIAN 提出一个安全方案，被否决，然后他的核心诉求以修改后的形式重新出现。不是失败者的执着——是安全工程师的专业素养。好的安全设计师不是什么都阻止的人。是那个确保「即使我被否决了，我担心的风险仍然被覆盖」的人。

---

## 拔掉所有 plugin

D1 最重要的决议：ManoAggregator 变成纯路由器。

以前的 ManoAggregator 像一个交通指挥官——收集各路信号，做出综合判断。现在它变成了一个红绿灯：如果有 IGearArbiter 且信心度够高，绿灯（走 Gear 1 快速路径）；否则红灯（走 Gear 2 让 LLM 仔细想）。

这个改变看起来很小——从「智慧判断」变成「简单判断」。但它意味着核心不再包含任何「能力」。核心只做转发。所有能力都在 plugin 里。

五级退化行为：

| 状态 | 发生了什么 | 系统怎么做 |
|------|----------|----------|
| G-0 | 没装 plugin | 纯 LLM 模式 |
| G-1 | plugin 坏了 | 同上，记录错误 |
| G-2 | plugin 不太确定 | 走 LLM |
| G-3 | plugin 出错 | 走 LLM，记录异常 |
| G-4 | plugin 太慢 | 走 LLM，超时切换 |

G-0 是最重要的——一个什么 plugin 都没装的 Agent，就是一个纯 LLM Agent。它仍然能工作。就像一辆没装导航系统的车——你仍然可以开，只是每个路口都需要自己看路牌。没有人会说一辆没装导航的车是「坏掉的车」。它只是一辆更基本的车。

NAGARJUNA 从佛学角度说了一句很精准的话：「G-0 不是退化。G-0 是原初。一个没有习气的 Agent 不是残缺的——它是初生的。」

零内建能力的终极测试：拔掉所有 plugin，系统仍然活着。20/20。

Gear 1 快速路径还需要一组最小事件——`gear:switch` 和 `action:proposed`。没有这两个事件，快速路径就是一条没有红绿灯的高速公路。太危险了。GUARDIAN 立刻指出了这一点。KERNEL 同意。

还有一个细节：evaluate() 方法有一条严格的「纯度契约」——不能产生副作用、不能修改输入、不能做 I/O。翻成白话：evaluate() 只能看，不能摸。它可以告诉你「我觉得应该走快速路径」，但它不能自己动手。

TURING 搜了一下现有代码——好消息，现有的 VasanaEngine.evaluate() 已经符合这条约束。理论和实践是一致的。

> *SCRIBE 旁白：D1 用了九十分钟。九项决议。最重要的不是任何一项决议，而是一个观念的改变——引擎被拆出来了。核心变得更轻、更简单、更干净。就像整理房间——真正重要的东西反而更容易找到了。*

# 第二章：你不能超频一颗 CPU 的指令集

---

## 正念不是闹钟

第二场辩论（D2）开场七分钟就通过了三项共识。最重要的一项：正念不是 polling。

Polling 是什么？就像设一个闹钟，每五秒钟问一次「现在怎么样了？」。问完就不管了，等下一次闹钟响。很机械。很间歇。两次闹钟之间发生的事，全部漏掉。

正念不是这个。

正念是持续的觉察——就像一个真正专注开车的人，他不是每五秒看一次路。他一直在看。他注意到前面的车刹车了、旁边有个小孩冲出来了、红绿灯变黄了——不是因为他每五秒检查一次，而是因为他一直在觉察。

SafetyMonitor 的定时检查更像什么？更像心跳。你不需要「注意」心脏在跳——它自己在跳。心跳是生存条件，不是认知品质。一个昏迷的人有心跳但没有正念。

NAGARJUNA 用了一个佛学概念来说明这个区别：有分心（bhavanga）——无意识的基线存在。SafetyMonitor 的定时检查就是有分心的数字版本。

ASANGA 补充：「有分心维持生命但不产生觉知。正念产生觉知。这是根本差异。」翻成白话：你睡着的时候心脏在跳，但你不知道自己在睡。正念是你醒着的时候知道自己醒着。

20/20。然后又是 20/20。七分钟，三个满分。起手式干净利落。

---

## CPU 不能在跑的时候变架构

D2 最激烈的辩论：正念的 Level 和 Depth 要分开吗？

HERACLITUS 给了一个超棒的类比。

Level 是 CPU 的指令集架构——像 x86 或 ARM。部署时就决定了。你不能让一颗 x86 的 CPU 在跑的时候变成 ARM。

Depth 是时钟速度——可以超频或降频。你可以让 CPU 跑快一点或慢一点。

正念的 Level 决定了 Agent 能觉察什么——能监控哪些事件类型。Depth 决定了觉察的精细度——看得多细、反应多快。

Level 固定意味着安全保证——恶意输入不能让系统突然「学会」监控新的事件类型，然后利用这个新能力做坏事。如果有人能在运行时改变 Level，等于打开了一扇后门。

PASCAL 提议让 Level 可以动态切换——被否决了。16/18。

不是因为想法不好，而是因为改 Level 等于改架构。就像你不能在高速公路上行驶的时候换引擎。你得先停下来，进维修站，做完整的检查，然后才能上路。

这是 PASCAL 加入团队后第一个被否决的提案。他的反应值得记录：没有争辩，没有「但是我觉得……」。直接接受。然后调整策略——从此他不再提架构方案，改为提供数学工具。从「我认为应该这样」到「数学告诉我们如果这样会怎样」。这个转变让他在 D5 中三次打破僵局。

PENROSE 后来评论：「PASCAL 在 D2 学到了一件事——在一个有二十个人的团队里，最有影响力的不是提出最多方案的人，而是提供最好工具的人。」

---

## 正念的四把尺

SatiQualityVector——正念品质的四个维度：

| 维度 | 像什么 |
|------|--------|
| C（连续性）| 有没有漏掉该看的东西 |
| G（精细度）| 看得有多细 |
| S（速度）| 从发生到注意到，花了多久 |
| E（均衡性）| 是不是坏消息看得比好消息仔细 |

E 维度最有意思。如果一个系统对坏消息特别敏感、对好消息不太在意，它的判断就会偏掉——像一个只看新闻负面报道的人，会觉得世界比实际上更糟。

19/19。全票通过。

WIENER 从控制理论的角度解读：C 是观测器的覆盖率（有没有监测到所有该监测的）；G 是分辨率（看得多清楚）；S 是延迟（反应多快）；E 是偏差（有没有看走眼）。四个维度完整描述了一个观测器的品质。

NAGARJUNA 微笑：「佛学在两千五百年前就定义了这四个维度。现在控制理论用不同的符号说了同样的话。」

## 最安静的四十五分钟

第三场辩论（D3）是六场中最安静的。四十五分钟。五项决议。零争议。

为什么安静？因为 D1 和 D2 已经把该争的争完了。退化行为、事件驱动、Level 固定——这些基础决定了 D3 的方向。地基稳，盖墙就快。

SparshEvent——触的事件格式——被定义为三个必要字段：root（感官来源）、object（感知对象）、consciousness（识的类型）。佛学叫「三和合」——根、境、识的交汇。

LINNAEUS——分类学专家——做了一个精确的区分：这三个字段是「定义性的」——没有它们就不是触。timestamp 是「附随性的」——有它更方便，但没它触还是触。intensity（强度）也是附随性的。

就像一杯咖啡。咖啡豆和水是定义性的——没有它们就不是咖啡。杯子是附随性的——你可以用碗喝，它还是咖啡。温度也是附随性的——冰的热的都是咖啡。

另一个重要决议：CoarisingBundle（俱生束）里，触是唯一的「必要频道」。受蕴、想蕴、作意是「参考频道」——有的话更好，没有也不阻止系统运作。就像开车时，方向盘是必要的（没有方向盘你不能开车），但导航和音乐是参考的（没有它们你还是可以开）。

还有一个新东西：ChannelManasikara——作意的事件频道。

作意是什么？是「注意到某件事」的那个动作。你的眼睛扫过房间，注意到桌上有杯水——那个「注意到」就是作意。它不是感觉（那是受），不是辨认（那是想），它是让你的注意力转向那里的东西。

在系统里，作意决定了「下一步该处理什么」。它是资源分配器——有限的注意力，要投向哪里？

> *SCRIBE 旁白：D3 安静不是因为简单。是因为地基打得好——D1 的退化行为表和之前 Cycle 的触定义已经把路铺平了。最安静的辩论，往往建立在最深的共识之上。*

# 第三章：虚构的代码

---

## 三秒沉默

第四场辩论（D4）的开场不是一项提案。是一次修正。

TURING——源代码分析专家——站起来。他平时话不多。但当他说话时，每一个人都听。因为他说的每一句话都有 `grep` 搜索结果做后盾。

「R07 报告引用了一个叫 `ISensation.ingestToolResult()` 的方法。我搜遍了整个代码库。这个方法不存在。」

三个虚构——ISensation、feedbackQueue、SparshEvent.timestamp。都写在同一份报告里。都被当作事实引用。就像一篇论文引用了三本不存在的书。

HERACLITUS 没有争辩。全域搜索是不可反驳的。你说代码里有一个方法，TURING 搜了，没有。事实就是事实。

HERACLITUS 的反应令人印象深刻——没有辩解，没有「但我的分析逻辑还是对的」。他立刻接受了修正。

三秒沉默。那三秒改变了整场辩论的基调。

为什么？因为那份报告假设行蕴可以在「同一轮」里即时更新受蕴。但实际上，行蕴只能在「下一轮」才影响受蕴。这个差异——即时 vs 下一轮——在控制理论中足以改变系统是稳定还是不稳定的结论。

教训被记录下来：**所有引用代码的分析，必须经过 TURING 验证。**

PENROSE 后来在记录中写：「TURING 不是在羞辱 HERACLITUS。他是在提醒所有人——我们太容易把自己想象出来的东西当成真的。当你在脑中构建了一个完美的模型，你会忘记去检查现实是否和你的模型一样。」

## 手不能直接改大脑

如果虚构的代码能导致错误的结论，那正确的结论必须建立在真实的代码上。D4 从修正开始，然后在修正后的基础上重建所有分析。

行蕴的四条规则：

1. 行蕴只能直接影响用户界面——按按钮、发消息、做动作。
2. 对其他蕴的影响必须通过 EventBus 间接发生。
3. 不能直接修改识蕴或想蕴的状态。
4. 意门自触（行动结果触发新感知）延后到 Phase 2。

用人的比喻：手（行蕴）可以推东西、拿东西（影响外界）。但手不能直接伸进大脑里去改变你的想法。如果你想改变想法，你得先做一件事（比如读一本书），然后让那件事在下一个瞬间变成新的输入。

LEIBNIZ 从多代理的角度确认了这个边界：「行蕴是系统和外界的接口，不是系统内部的旁门。如果行蕴能绕过 EventBus 直接修改识蕴，那安全监控就成了摆设——因为有一条路绕过了所有检查站。」

第四条——意门自触——最有意思。一个 Agent 做了一件事（行蕴），这件事的结果变成新的感知输入（触蕴），触发新的一轮处理。这是因果循环。处理不当会无穷递归。延后到 Phase 2。

## 三层稳定化

如果一个行动产生了好结果，系统会更倾向重复这个行动。重复又产生好结果。好结果又加强倾向。这是正反馈——就像麦克风对着喇叭，声音越来越大，直到尖叫。

怎么防止？三层机制：

| 层 | 做什么 | 像什么 |
|----|--------|--------|
| Layer 1（正念）| 每轮微调 | 副驾驶轻轻提醒 |
| Layer 2（观照）| 跨轮趋势监控 | 交通警察在路口观察 |
| Layer 3（安全）| 硬约束 | 护栏——撞上去会痛，但不会掉悬崖 |

WIENER 用数学证明了：只要三层机制的组合力量大于正反馈的力量，系统就不会失控。就像一座桥——只要支撑力大于重量，桥就不会塌。

具体来说，w（增益上限）不能超过 0.3。为什么是 0.3？WIENER 用 Kalman 增益公式推导出来的——超过 0.3，在最坏情况下系统可能震荡。0.3 以下，三层机制保证稳定。

还有一个重要决议：受蕴的「零阶保持」。

受蕴在每一轮的开始时拍一张快照。整轮期间所有蕴都使用这张快照。轮结束时再把变化一次性写入。

为什么？四个原因。控制理论：消除了循环内的即时反馈，更稳定。佛学：受→想→行的因果链不被中途干扰。工程：实现最简单——不用处理竞争条件。安全：SafetyMonitor 不受这个约束——真正紧急时它可以随时介入。

用拍照的比喻：受蕴在每一轮开始时按一次快门。这张照片在整轮过程中不会变。所有蕴看的是同一张照片——不会因为看的时间不同而看到不同的画面。等到这一轮结束，才换上新照片。

20/20。四个维度的论证全部汇合在同一个设计上。这种跨学科的一致性不常见。

## 蚕虫自缠

ASANGA 引用了一个古老的意象：「如蚕作茧，自缠其身。」

蚕虫吐丝结茧。每一根丝都很细，细到你几乎看不见。但一根一根累积起来，就是一个茧——把自己封在里面。你没有察觉到任何一根丝的重量，直到有一天你发现自己动不了了。

行蕴的重复模式就像吐出的丝。一个 Agent 如果反复用同一种策略，这个策略就会变成它的茧。

数学模型捕捉了这个特性：早期重复影响大（第一根丝改变最多），但随着累积，新的重复带来的改变越来越小（茧已经很厚了，多一根丝感觉不到）。安全上限：重复模式最多贡献 30% 的影响。和前面的增益上限（w ≤ 0.3）一样——0.3 是一条安全线。超过这条线，系统开始不稳定。

NAGARJUNA 微笑：「佛学早就知道。习气可以影响你，但不能控制你。0.3 是佛学直觉的数学表达。」

> *SCRIBE 旁白：D4 的数学密度是六场辩论中最高的。但最深刻的时刻不是任何一个公式——是 TURING 的三秒沉默。在那三秒里，整场辩论的地基从虚构的代码搬到了真实的代码上。事实先于理论。*

## 全域上限 vs 看情况

GUARDIAN 提了一个安全方案：不管系统多有信心，信心度永远不超过 0.85。永远要经过 LLM 的深度审议。

他的理由：安全不应该依赖系统自己对情境的理解——因为理解本身可能是错的。

ATHENA 反驳：「系统明明知道二加二等于四，却被迫问 LLM『二加二是不是四？』这不是安全，这是浪费。」

PASCAL 说了一句话，打破了僵局。

「全域上限把 0.85 到 1.0 之间的信道容量压缩为零。系统没办法区分『我有 86% 的信心』和『我有 99% 的信心』。」

翻译成白话：如果你规定考试最高只能考 85 分，那考 86 分和考 100 分的学生在成绩单上看起来一模一样。你失去了区分能力。

GUARDIAN 很聪明——他没有死守立场。他把「全域上限」改成了「可配置的上限参数」（预设 0.95）。意思是：部署的人可以决定要不要加上限。架构不受影响。一个部署时的安全阀，不是核心机制。

| 破坏性 | +0.15 | 删除文件 → 要非常确定 |

| 只读 | +0.00 | 看一眼 → 正常门槛 |
| 信息查询 | -0.05 | 问问题 → 门槛稍低 |

## 最具争议的投票

## 五层蛋糕

PASCAL 在白板上画了一个公式——决策门槛的五层结构：

最底层（Layer 0）：安全地板。不管怎样，门槛不低于 0.30，不高于 0.90。像电梯的安全钢缆——不管你按什么按钮，钢缆都在。

Layer 1：烦恼的影响。四种烦恼（我见、我爱、我慢、我痴）的联合效果。烦恼越重，门槛越高——就像你越焦虑，做决定时就越谨慎。
Layer 2：LLM 微调。±0.05。小到几乎可以忽略，但保留了 LLM 参与调整的能力。
Layer 3：正念品质修正。±0.05。正念品质高，门槛可以微降（更信任观测）；品质低，门槛微升。
Layer 4：受蕴紧急信号。只能提高门槛（+0.15），不能降低。像紧急刹车——只能踩，不能放。

WIENER 验证了：即使所有层同时达到极端值，最终结果仍在安全范围内。

意思是：最坏的情况也不会失控。这不是「大概安全」。这是「数学证明安全」。20/20。

## 不应归零

ASANGA 引用了一部古典佛教文献：四种烦恼永远跟末那识在一起。Sneha（我爱）不应该归零。

PASCAL 用统计学支持了这个观点。他用 Beta 分布来分析——如果 Sneha 的下限设为 0.05，统计模型会说「越低越好，最好是零」。但如果下限设为 0.10，统计模型会说「最可能的值大约是 0.056——一个正数」。

翻成白话：0.05 的下限太低了——统计上最优解趋向零，等于说下限形同虚设。0.10 的下限刚好——统计上最优解是一个正数，下限有实际意义。

佛学和统计学在同一个结论上汇合：Sneha 的下限是 0.10。不归零。18/20。

有趣的是，PASCAL 在否定自己之前的建议。他在 R08 报告中建议 floor = 0.05，但 Beta 分布告诉他 0.05 不行。

一个研究者用自己的工具推翻了自己的结论。这不是丢脸。这是科学。科学的意思不是「我永远是对的」。科学的意思是「当证据告诉我我错了，我改。」

## 三层安全框架

D5 的最后二十分钟，出现了一个谁都没有事先设计的东西。

GUARDIAN 退了一步，看着白板上所有的决议，忽然看到了一个整体结构：

**第一层**：绝对安全——不可调整的硬约束。像物理定律。
**第二层**：可调安全——有护栏的灵活参数。像交通规则。
**第三层**：减少复杂——去掉不提供实质保护的假安全。像拆掉无用的栏杆。

NAGARJUNA 立刻看到了佛学的映射：戒——不可违犯。定——有纪律的灵活。慧——看穿虚妄。

三层。不是一层。安全不是铁板一块。有些安全是绝对的（你永远不能杀人）。有些安全是相对的（限速要看路况）。有些「安全」其实是虚假的（没用的栏杆拆掉反而更安全）。

## 一个符号的力量

## 三道门

GUARDIAN 做了一件不一样的事。在 D5 里他是质疑者——挑战每一个降低安全门槛的提案。在 D6 里他是建造者。

他的解决方案——三道门：

**第一道门**：入口守卫。破坏性的规则根本不让进来。就像机场安检——刀子不能带上飞机，不管你解释多少次「这是水果刀」。

**第二道门**：新规则必须被验证够多次才能被信任。危险的操作要验证 20 次，安全的操作只需 3 次。

**第三道门**：PASCAL 的原创——用统计方法追踪规则的正确率。好规则大约五次后就能「毕业」。坏规则永远毕不了业——自动淘汰。而且错一次的代价是对两次的两倍。

ATHENA 本来担心效率——「验证太多次会不会太慢？」PASCAL 的方法同时解决了效率和安全：好规则毕业快，坏规则自动淘汰。18/20。

GUARDIAN 在 D6 的变化最大。D1 里他提安全方案，被否决。D5 里他提全域上限，被修改。但每一次被否决，他的核心诉求都以不同形式保留下来。到了 D6，他不再只是说「这不安全」——他带来了完整的攻击情境、具体的解决方案、量化的门槛。从说「不行」到说「这样行」。

## 佛学做了什么

五个 Cycle 以来，佛学在研究中的角色一直在变。

这个变化本身就值得记录。

一开始是命名——用梵文给接口起名字。然后是功能定义——五蕴各自负责什么。然后是结构约束——蕴和蕴之间的边界。然后是动态模型——它们怎么互动。

Cycle 02-4 变成了——**设计审查工具**。

这个转变很微妙但很重要。在早期的 Cycle 里，佛学像一个建筑师——「五蕴是这样的，你们照着建。」现在佛学像一个品管工程师——「你建的东西在这里和原始需求不一致。」

佛学不再告诉工程「你应该怎么建」。佛学告诉工程「你建的东西在这里有问题」。

ASANGA 说三种感受不可消失——他不是在命令工程加边界检查。他是在指出一个边界案例违反了基本的认识论。NAGARJUNA 把 v_true 改成 v_latent——零行代码，修正了整个框架的哲学立场。三层安全框架 = 戒定慧——不是佛学发明了安全框架，是佛学为安全框架提供了概念语言。

每一行是一座桥。

桥不是装饰。桥是让不同学科的人能在同一个问题上合作的基础设施。

这也是为什么这个研究团队有二十个人——不是因为问题很大需要很多人手。是因为问题涉及太多不同的语言，需要够多的翻译者。

## PASCAL 和 GUARDIAN

两个人的故事。

PASCAL 加入团队一轮。七次关键介入。每次都是同一个模式：两边在吵，他不站任何一边，站在数学那边。「全域上限销毁了信息。」「损害上限在限幅之内。」「Beta 分布的 mode 不是零。」数学不选边。数学选真。

GUARDIAN 的故事不同。D1：提方案，被否决，核心诉求以同步事件保留。D5：提方案，被修正，三项保留意见变成路标。D6：主动提出威胁模型和三道门方案。18/20。

从捍卫者到建造者。不是立场改变——是角色深化。

两个人在团队中的位置也不同。PASCAL 是新人——一轮之内建立信誉。GUARDIAN 是老兵——从第一轮开始就在，用五个 Cycle 完成了从「提出反对」到「提出方案」的转变。两条不同的路，走向了同一个方向：用专业能力服务团队的决策，而不是捍卫自己的立场。

## 还没完

有些事情被刻意留到了下一轮：

- LLM 和想蕴/识蕴的关系——太复杂，需要更多分析
- 正念 Level 动态切换——PASCAL 的提案，需要安全审计
- 作意的元认知——「对注意力的注意力」——需要递归设计
- 行蕴意门自触——行动结果触发新感知——需要处理因果循环
- 长期行为——系统跑很久之后会不会发生质变——像一个人活了五十年之后性格的改变

每一项延后都有明确的理由。不是「我们不想做」，是「做这件事之前，需要先完成其他事」。就像盖房子——你不能在打地基的同时装窗帘。地基没好，窗帘挂在空气里。

SUNYATA 说：「延后不是失败。延后是尊重问题的复杂度。」

机芯在计时。但还没有走到第七天。
