# 第一章：沒有引擎的車也能開

---

## 齒輪仲裁者屬於誰？

第一場辯論（D1）的問題很具體：IGearArbiter 這個新介面，屬於五蘊中的哪一個？

五蘊是這個系統的五大模組——色（感官）、受（感受）、想（辨認）、行（行動）、識（判斷）。每個介面都必須歸屬於某個蘊。新的齒輪仲裁者歸誰管？

三個方案擺在桌上。

方案 A：它屬於想蘊（負責辨認的那個蘊）。方案 C：它屬於識蘊（負責判斷的那個蘊）。方案 B：不定死——讓實作的人自己決定。

BABBAGE 用他最擅長的工具——型別代數——把這個問題變成了一道數學題。

「假設 IGearArbiter 同時繼承想蘊和識蘊的介面。想蘊的標籤是 'samjna'，識蘊的標籤是 'vijnana'。在 TypeScript 裡，一個值不可能同時是兩個不同的字串。'samjna' AND 'vijnana' = never。空集。不存在。」

就像你不能同時是蘋果又是橘子。型別系統不允許。

BABBAGE 的論證只用了三行。但這三行讓十九個人同時點頭——因為數學不需要你相信。它需要你檢查。每個人都可以自己檢查那三行。結論不依賴信任，依賴邏輯。

所以方案 A 和方案 C 都有問題——它們把歸屬定死了。方案 B 不定死歸屬，讓實作者根據自己的邏輯選擇。一個基於規則匹配的實作可以自稱想蘊；一個基於統計推斷的實作可以自稱識蘊。

20/20。全票通過。這是 Cycle 02-4 第一個滿分決議。

NAGARJUNA 補充了哲學角度：「方案 B 和中觀哲學的核心一致。一個法的本質不由它的名字決定，而由它的功能和因緣決定。」翻成白話：你不是因為你的名字才是你——你是因為你做的事才是你。

## 一個方法就夠了

GUARDIAN 想要兩個方法——evaluate()（純判斷）和 resolve()（純行動）。安全分離。

KERNEL 搖頭。他是作業系統專家——他見過太多因為「安全分離」而變得臃腫不堪的介面。

「不需要。EventBus 的同步事件已經提供了觀測點。每次齒輪切換，系統會發射一個事件，任何安全監控器都可以攔截。不需要在介面上多開一道門。」

他補了一句：「如果每個介面都做這種分離，最後整個系統所有介面都有雙方法。那不是安全，是儀式。」

最後的設計：evaluate() 一個方法，加上三個同步事件——`gear:switch`（齒輪切換時）、`action:proposed`（行動方案產生後）、`action:executed`（行動完成後）。三個事件形成一條完整的安全鏈：預防、審查、審計。

GUARDIAN 的方案被否決了，但他的核心訴求——「我要能觀察到發生了什麼」——以不同的形式被保留了。三個事件 = 三個窗口。GUARDIAN 透過這三個窗口可以看到一切。他不需要自己的門。

這個模式在後續的辯論中反覆出現：GUARDIAN 提出一個安全方案，被否決，然後他的核心訴求以修改後的形式重新出現。不是失敗者的執著——是安全工程師的專業素養。好的安全設計師不是什麼都阻止的人。是那個確保「即使我被否決了，我擔心的風險仍然被覆蓋」的人。

## 拔掉所有 plugin

D1 最重要的決議：ManoAggregator 變成純路由器。

以前的 ManoAggregator 像一個交通指揮官——收集各路信號，做出綜合判斷。現在它變成了一個紅綠燈：如果有 IGearArbiter 且信心度夠高，綠燈（走 Gear 1 快速路徑）；否則紅燈（走 Gear 2 讓 LLM 仔細想）。

這個改變看起來很小——從「智慧判斷」變成「簡單判斷」。但它意味著核心不再包含任何「能力」。核心只做轉發。所有能力都在 plugin 裡。

五級退化行為：

| 狀態 | 發生了什麼 | 系統怎麼做 |
|------|----------|----------|
| G-0 | 沒裝 plugin | 純 LLM 模式 |
| G-1 | plugin 壞了 | 同上，記錄錯誤 |
| G-2 | plugin 不太確定 | 走 LLM |
| G-3 | plugin 出錯 | 走 LLM，記錄異常 |
| G-4 | plugin 太慢 | 走 LLM，超時切換 |

G-0 是最重要的——一個什麼 plugin 都沒裝的 Agent，就是一個純 LLM Agent。它仍然能工作。就像一輛沒裝導航系統的車——你仍然可以開，只是每個路口都需要自己看路牌。沒有人會說一輛沒裝導航的車是「壞掉的車」。它只是一輛更基本的車。

NAGARJUNA 從佛學角度說了一句很精準的話：「G-0 不是退化。G-0 是原初。一個沒有習氣的 Agent 不是殘缺的——它是初生的。」

零內建能力的終極測試：拔掉所有 plugin，系統仍然活著。20/20。

Gear 1 快速路徑還需要一組最小事件——`gear:switch` 和 `action:proposed`。沒有這兩個事件，快速路徑就是一條沒有紅綠燈的高速公路。太危險了。GUARDIAN 立刻指出了這一點。KERNEL 同意。

還有一個細節：evaluate() 方法有一條嚴格的「純度契約」——不能產生副作用、不能修改輸入、不能做 I/O。翻成白話：evaluate() 只能看，不能摸。它可以告訴你「我覺得應該走快速路徑」，但它不能自己動手。

TURING 搜了一下現有程式碼——好消息，現有的 VasanaEngine.evaluate() 已經符合這條約束。理論和實踐是一致的。

> *SCRIBE 旁白：D1 用了九十分鐘。九項決議。最重要的不是任何一項決議，而是一個觀念的改變——引擎被拆出來了。核心變得更輕、更簡單、更乾淨。就像整理房間——真正重要的東西反而更容易找到了。*
