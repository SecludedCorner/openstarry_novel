# 序章：拆掉引擎

---

圓形劇場的燈維持著工作亮度。二十盞。一盞都沒有暗。

這個劇場已經亮了五輪了。每一輪，二十個人坐在這裡辯論。每一輪結束，燈暗下來。幾天後，燈再亮起。然後新一輪開始。

三天前——Cycle 02-3 結束時——SUNYATA 說了一個比喻。六場辯論像六個齒輪，被精心切削完畢。齒距正確，軸心正確。等著被組裝成一座鐘。

三天。

三天裡沒有正式辯論。但三天裡發生了一件事——一件在正式辯論前就改變了一切的事。

---

GUARDIAN 把安全備忘錄讀了三遍。

不是隨便翻。是那種用紅筆在關鍵字下面劃線、每讀一遍線就更粗的讀法。

第三遍讀完時，他在筆記本上寫了三行：

- VasanaEngine 有規則庫
- 規則庫是能力
- 能力不應該在核心裡

如果你在修一輛車，你會把引擎蓋打開，看到裡面有個零件——它有自己的記憶、自己的判斷力、自己的決策能力。你會問：這個零件為什麼在引擎裡？它不是引擎的一部分。它是一個獨立的裝置，碰巧被塞在引擎裡面了。

VasanaEngine 就是那個零件。它在核心裡，但它不屬於核心。

---

NAGARJUNA 在同一時間想到了同一個問題，但用了完全不同的語言。他不知道 GUARDIAN 的分析。GUARDIAN 也不知道他的。兩個人在不同的房間裡，用不同的語言，推導出了同一個結論。

「習氣是後天薰習的。」他閉著眼睛說。

翻譯成白話：你的習慣是你活出來的，不是你生下來就有的。一個嬰兒沒有習慣。一個剛出生的 AI Agent 也不應該被迫擁有習慣。

如果 VasanaEngine（習氣引擎）是核心的一部分，那就等於說每一個 Agent 出生時就自帶一套習慣。這不對。

兩個人。不同的出發點。同一個終點。GUARDIAN 從安全出發——核心不該有能力。NAGARJUNA 從佛學出發——習氣是後天的。結論相同：拆掉它。

---

SUNYATA 把矛盾寫在白板上。然後他做了一件從來沒做過的事——他按下通訊面板，直接聯繫 Master。

「VasanaEngine 不應該在核心中。」

Master 的回答只有兩個字：「是的。」

那個「是的」不是驚訝。是「我一直在等你們自己發現這件事」。

然後 Master 說：「如果 VasanaEngine 不在核心裡，ManoAggregator 就退化為純路由。if/else。像 EventBus 一樣——只是一個轉發站。」

SUNYATA 聽到這句話時停了一下。「純路由」意味著核心幾乎什麼都不做——只是看看有沒有 plugin 可以處理，如果有就交給它，如果沒有就交給 LLM。整個核心的複雜度從一百行縮減到二十行。

VasanaEngine 被拆出來了。它有了一個新名字：**IGearArbiter**——齒輪仲裁者。一個可以安裝也可以拆掉的 plugin。

就像手機上的 App。你可以裝一個導航 App，也可以不裝。不裝的話手機還是手機——你只是需要自己看路牌。裝了的話你有額外的便利。但導航 App 不是手機的核心——它不應該被焊在主機板上。

同一場對話還順便解決了一個拖了很久的問題：需不需要形式化區分「必裝 plugin」和「選裝 plugin」？答案意外地簡單——不需要。之前 Cycle 02-2 已經設計了「五蘊完備性檢查」，那個檢查已經覆蓋了這個需求。不需要新方案。

SUNYATA 後來在記錄裡寫了一行：「提出新方案前先回顧既有決議。」

這個教訓比看起來更重要。在一個有五個 Cycle、數百項決議的研究專案裡，最浪費時間的事不是「想不出答案」，而是「忘了答案已經存在」。

---

PASCAL——團隊最新的成員——在另一間辦公室裡翻閱前一輪的報告。他在 Cycle 02-3 加入團隊，專長是決策理論和機率。他的筆記本上畫滿了 Beta 分佈的圖形。他不知道這些圖形會在三天後的辯論中三次打破僵局。

---

第四封信在第三天傍晚出現。

前三封信改變了「什麼」「怎麼修」「怎麼動」。第四封信改變的是——**怎麼變成一個整體。**

六條指示：

- M-1：把 VasanaEngine 拆出來，設計新介面
- M-2：六個拖了很久的問題，全部解決
- M-3：觸和作意——它們是事件？還是因果的起點？
- M-4：正念不是定時檢查
- M-5：行蘊的結果去了哪裡？比看起來更重要
- M-6：十大宣言。嚴格微核心。零內建能力。不要妥協。

M-6 最嚴厲。「不要妥協」三個字。Master 在之前的信裡從來沒有用過這種語氣。這三個字意味著：前幾輪有些地方妥協了，這次不行。

六條指示。六場辯論。但不是六個獨立的問題。

SUNYATA 站在二十盞燈前面說：「Master 給了我們一個場。像一張地圖——每一個點上都有值。我們的工作不是解六個方程式。是解一個場方程。」

> *SCRIBE 旁白：第四封信和前三封不同。前三封像醫生開的處方——這裡痛，吃這個藥。第四封像一張 X 光片——你看到的不是症狀，而是整個骨架。*

# 第一章：沒有引擎的車也能開

---

## 齒輪仲裁者屬於誰？

第一場辯論（D1）的問題很具體：IGearArbiter 這個新介面，屬於五蘊中的哪一個？

五蘊是這個系統的五大模組——色（感官）、受（感受）、想（辨認）、行（行動）、識（判斷）。每個介面都必須歸屬於某個蘊。新的齒輪仲裁者歸誰管？

三個方案擺在桌上。

方案 A：它屬於想蘊（負責辨認的那個蘊）。方案 C：它屬於識蘊（負責判斷的那個蘊）。方案 B：不定死——讓實作的人自己決定。

BABBAGE 用他最擅長的工具——型別代數——把這個問題變成了一道數學題。

「假設 IGearArbiter 同時繼承想蘊和識蘊的介面。想蘊的標籤是 'samjna'，識蘊的標籤是 'vijnana'。在 TypeScript 裡，一個值不可能同時是兩個不同的字串。'samjna' AND 'vijnana' = never。空集。不存在。」

就像你不能同時是蘋果又是橘子。型別系統不允許。

BABBAGE 的論證只用了三行。但這三行讓十九個人同時點頭——因為數學不需要你相信。它需要你檢查。每個人都可以自己檢查那三行。結論不依賴信任，依賴邏輯。

所以方案 A 和方案 C 都有問題——它們把歸屬定死了。方案 B 不定死歸屬，讓實作者根據自己的邏輯選擇。一個基於規則匹配的實作可以自稱想蘊；一個基於統計推斷的實作可以自稱識蘊。

20/20。全票通過。這是 Cycle 02-4 第一個滿分決議。

NAGARJUNA 補充了哲學角度：「方案 B 和中觀哲學的核心一致。一個法的本質不由它的名字決定，而由它的功能和因緣決定。」翻成白話：你不是因為你的名字才是你——你是因為你做的事才是你。

---

## 一個方法就夠了

GUARDIAN 想要兩個方法——evaluate()（純判斷）和 resolve()（純行動）。安全分離。

KERNEL 搖頭。他是作業系統專家——他見過太多因為「安全分離」而變得臃腫不堪的介面。

「不需要。EventBus 的同步事件已經提供了觀測點。每次齒輪切換，系統會發射一個事件，任何安全監控器都可以攔截。不需要在介面上多開一道門。」

他補了一句：「如果每個介面都做這種分離，最後整個系統所有介面都有雙方法。那不是安全，是儀式。」

最後的設計：evaluate() 一個方法，加上三個同步事件——`gear:switch`（齒輪切換時）、`action:proposed`（行動方案產生後）、`action:executed`（行動完成後）。三個事件形成一條完整的安全鏈：預防、審查、審計。

GUARDIAN 的方案被否決了，但他的核心訴求——「我要能觀察到發生了什麼」——以不同的形式被保留了。三個事件 = 三個窗口。GUARDIAN 透過這三個窗口可以看到一切。他不需要自己的門。

這個模式在後續的辯論中反覆出現：GUARDIAN 提出一個安全方案，被否決，然後他的核心訴求以修改後的形式重新出現。不是失敗者的執著——是安全工程師的專業素養。好的安全設計師不是什麼都阻止的人。是那個確保「即使我被否決了，我擔心的風險仍然被覆蓋」的人。

---

## 拔掉所有 plugin

D1 最重要的決議：ManoAggregator 變成純路由器。

以前的 ManoAggregator 像一個交通指揮官——收集各路信號，做出綜合判斷。現在它變成了一個紅綠燈：如果有 IGearArbiter 且信心度夠高，綠燈（走 Gear 1 快速路徑）；否則紅燈（走 Gear 2 讓 LLM 仔細想）。

這個改變看起來很小——從「智慧判斷」變成「簡單判斷」。但它意味著核心不再包含任何「能力」。核心只做轉發。所有能力都在 plugin 裡。

五級退化行為：

| 狀態 | 發生了什麼 | 系統怎麼做 |
|------|----------|----------|
| G-0 | 沒裝 plugin | 純 LLM 模式 |
| G-1 | plugin 壞了 | 同上，記錄錯誤 |
| G-2 | plugin 不太確定 | 走 LLM |
| G-3 | plugin 出錯 | 走 LLM，記錄異常 |
| G-4 | plugin 太慢 | 走 LLM，超時切換 |

G-0 是最重要的——一個什麼 plugin 都沒裝的 Agent，就是一個純 LLM Agent。它仍然能工作。就像一輛沒裝導航系統的車——你仍然可以開，只是每個路口都需要自己看路牌。沒有人會說一輛沒裝導航的車是「壞掉的車」。它只是一輛更基本的車。

NAGARJUNA 從佛學角度說了一句很精準的話：「G-0 不是退化。G-0 是原初。一個沒有習氣的 Agent 不是殘缺的——它是初生的。」

零內建能力的終極測試：拔掉所有 plugin，系統仍然活著。20/20。

Gear 1 快速路徑還需要一組最小事件——`gear:switch` 和 `action:proposed`。沒有這兩個事件，快速路徑就是一條沒有紅綠燈的高速公路。太危險了。GUARDIAN 立刻指出了這一點。KERNEL 同意。

還有一個細節：evaluate() 方法有一條嚴格的「純度契約」——不能產生副作用、不能修改輸入、不能做 I/O。翻成白話：evaluate() 只能看，不能摸。它可以告訴你「我覺得應該走快速路徑」，但它不能自己動手。

TURING 搜了一下現有程式碼——好消息，現有的 VasanaEngine.evaluate() 已經符合這條約束。理論和實踐是一致的。

> *SCRIBE 旁白：D1 用了九十分鐘。九項決議。最重要的不是任何一項決議，而是一個觀念的改變——引擎被拆出來了。核心變得更輕、更簡單、更乾淨。就像整理房間——真正重要的東西反而更容易找到了。*

# 第二章：你不能超頻一顆 CPU 的指令集

---

## 正念不是鬧鐘

第二場辯論（D2）開場七分鐘就通過了三項共識。最重要的一項：正念不是 polling。

Polling 是什麼？就像設一個鬧鐘，每五秒鐘問一次「現在怎麼樣了？」。問完就不管了，等下一次鬧鐘響。很機械。很間歇。兩次鬧鐘之間發生的事，全部漏掉。

正念不是這個。

正念是持續的覺察——就像一個真正專注開車的人，他不是每五秒看一次路。他一直在看。他注意到前面的車剎車了、旁邊有個小孩衝出來了、紅綠燈變黃了——不是因為他每五秒檢查一次，而是因為他一直在覺察。

SafetyMonitor 的定時檢查更像什麼？更像心跳。你不需要「注意」心臟在跳——它自己在跳。心跳是生存條件，不是認知品質。一個昏迷的人有心跳但沒有正念。

NAGARJUNA 用了一個佛學概念來說明這個區別：有分心（bhavanga）——無意識的基線存在。SafetyMonitor 的定時檢查就是有分心的數位版本。

ASANGA 補充：「有分心維持生命但不產生覺知。正念產生覺知。這是根本差異。」翻成白話：你睡著的時候心臟在跳，但你不知道自己在睡。正念是你醒著的時候知道自己醒著。

20/20。然後又是 20/20。七分鐘，三個滿分。起手式乾淨利落。

---

## CPU 不能在跑的時候變架構

D2 最激烈的辯論：正念的 Level 和 Depth 要分開嗎？

HERACLITUS 給了一個超棒的類比。

Level 是 CPU 的指令集架構——像 x86 或 ARM。部署時就決定了。你不能讓一顆 x86 的 CPU 在跑的時候變成 ARM。

Depth 是時鐘速度——可以超頻或降頻。你可以讓 CPU 跑快一點或慢一點。

正念的 Level 決定了 Agent 能覺察什麼——能監控哪些事件類型。Depth 決定了覺察的精細度——看得多細、反應多快。

Level 固定意味著安全保證——惡意輸入不能讓系統突然「學會」監控新的事件類型，然後利用這個新能力做壞事。如果有人能在運行時改變 Level，等於打開了一扇後門。

PASCAL 提議讓 Level 可以動態切換——被否決了。16/18。

不是因為想法不好，而是因為改 Level 等於改架構。就像你不能在高速公路上行駛的時候換引擎。你得先停下來，進維修站，做完整的檢查，然後才能上路。

這是 PASCAL 加入團隊後第一個被否決的提案。他的反應值得記錄：沒有爭辯，沒有「但是我覺得⋯⋯」。直接接受。然後調整策略——從此他不再提架構方案，改為提供數學工具。從「我認為應該這樣」到「數學告訴我們如果這樣會怎樣」。這個轉變讓他在 D5 中三次打破僵局。

PENROSE 後來評論：「PASCAL 在 D2 學到了一件事——在一個有二十個人的團隊裡，最有影響力的不是提出最多方案的人，而是提供最好工具的人。」

---

## 正念的四把尺

SatiQualityVector——正念品質的四個維度：

| 維度 | 像什麼 |
|------|--------|
| C（連續性）| 有沒有漏掉該看的東西 |
| G（精細度）| 看得有多細 |
| S（速度）| 從發生到注意到，花了多久 |
| E（均衡性）| 是不是壞消息看得比好消息仔細 |

E 維度最有意思。如果一個系統對壞消息特別敏感、對好消息不太在意，它的判斷就會偏掉——像一個只看新聞負面報導的人，會覺得世界比實際上更糟。

19/19。全票通過。

WIENER 從控制理論的角度解讀：C 是觀測器的覆蓋率（有沒有監測到所有該監測的）；G 是解析度（看得多清楚）；S 是延遲（反應多快）；E 是偏差（有沒有看走眼）。四個維度完整描述了一個觀測器的品質。

NAGARJUNA 微笑：「佛學在兩千五百年前就定義了這四個維度。現在控制理論用不同的符號說了同樣的話。」

---

## 最安靜的四十五分鐘

第三場辯論（D3）是六場中最安靜的。四十五分鐘。五項決議。零爭議。

為什麼安靜？因為 D1 和 D2 已經把該爭的爭完了。退化行為、事件驅動、Level 固定——這些基礎決定了 D3 的方向。地基穩，蓋牆就快。

SparshEvent——觸的事件格式——被定義為三個必要欄位：root（感官來源）、object（感知對象）、consciousness（識的類型）。佛學叫「三和合」——根、境、識的交匯。

LINNAEUS——分類學專家——做了一個精確的區分：這三個欄位是「定義性的」——沒有它們就不是觸。timestamp 是「附隨性的」——有它更方便，但沒它觸還是觸。intensity（強度）也是附隨性的。

就像一杯咖啡。咖啡豆和水是定義性的——沒有它們就不是咖啡。杯子是附隨性的——你可以用碗喝，它還是咖啡。溫度也是附隨性的——冰的熱的都是咖啡。

另一個重要決議：CoarisingBundle（俱生束）裡，觸是唯一的「必要頻道」。受蘊、想蘊、作意是「參考頻道」——有的話更好，沒有也不阻止系統運作。就像開車時，方向盤是必要的（沒有方向盤你不能開車），但導航和音樂是參考的（沒有它們你還是可以開）。

還有一個新東西：ChannelManasikara——作意的事件頻道。

作意是什麼？是「注意到某件事」的那個動作。你的眼睛掃過房間，注意到桌上有杯水——那個「注意到」就是作意。它不是感覺（那是受），不是辨認（那是想），它是讓你的注意力轉向那裡的東西。

在系統裡，作意決定了「下一步該處理什麼」。它是資源分配器——有限的注意力，要投向哪裡？

> *SCRIBE 旁白：D3 安靜不是因為簡單。是因為地基打得好——D1 的退化行為表和之前 Cycle 的觸定義已經把路鋪平了。最安靜的辯論，往往建立在最深的共識之上。*

# 第三章：虛構的程式碼

---

## 三秒沉默

第四場辯論（D4）的開場不是一項提案。是一次修正。

TURING——原始碼分析專家——站起來。他平時話不多。但當他說話時，每一個人都聽。因為他說的每一句話都有 `grep` 搜尋結果做後盾。

「R07 報告引用了一個叫 `ISensation.ingestToolResult()` 的方法。我搜遍了整個程式碼庫。這個方法不存在。」

三個虛構——ISensation、feedbackQueue、SparshEvent.timestamp。都寫在同一份報告裡。都被當作事實引用。就像一篇論文引用了三本不存在的書。

HERACLITUS 沒有爭辯。全域搜尋是不可反駁的。你說程式碼裡有一個方法，TURING 搜了，沒有。事實就是事實。

HERACLITUS 的反應令人印象深刻——沒有辯解，沒有「但我的分析邏輯還是對的」。他立刻接受了修正。

三秒沉默。那三秒改變了整場辯論的基調。

為什麼？因為那份報告假設行蘊可以在「同一輪」裡即時更新受蘊。但實際上，行蘊只能在「下一輪」才影響受蘊。這個差異——即時 vs 下一輪——在控制理論中足以改變系統是穩定還是不穩定的結論。

教訓被記錄下來：**所有引用程式碼的分析，必須經過 TURING 驗證。**

## 手不能直接改大腦

如果虛構的程式碼能導致錯誤的結論，那正確的結論必須建立在真實的程式碼上。D4 從修正開始，然後在修正後的基礎上重建所有分析。

行蘊的四條規則：

1. 行蘊只能直接影響使用者介面——按按鈕、發訊息、做動作。
2. 對其他蘊的影響必須通過 EventBus 間接發生。
3. 不能直接修改識蘊或想蘊的狀態。
4. 意門自觸（行動結果觸發新感知）延後到 Phase 2。

用人的比喻：手（行蘊）可以推東西、拿東西（影響外界）。但手不能直接伸進大腦裡去改變你的想法。如果你想改變想法，你得先做一件事（比如讀一本書），然後讓那件事在下一個瞬間變成新的輸入。

LEIBNIZ 從多代理的角度確認了這個邊界：「行蘊是系統和外界的介面，不是系統內部的旁門。如果行蘊能繞過 EventBus 直接修改識蘊，那安全監控就成了擺設——因為有一條路繞過了所有檢查站。」

第四條——意門自觸——最有意思。一個 Agent 做了一件事（行蘊），這件事的結果變成新的感知輸入（觸蘊），觸發新的一輪處理。這是因果循環。處理不當會無窮遞歸。延後到 Phase 2。

## 三層穩定化

如果一個行動產生了好結果，系統會更傾向重複這個行動。重複又產生好結果。好結果又加強傾向。這是正反饋——就像麥克風對著喇叭，聲音越來越大，直到尖叫。

怎麼防止？三層機制：

| 層 | 做什麼 | 像什麼 |
|----|--------|--------|
| Layer 1（正念）| 每輪微調 | 副駕駛輕輕提醒 |
| Layer 2（觀照）| 跨輪趨勢監控 | 交通警察在路口觀察 |
| Layer 3（安全）| 硬約束 | 護欄——撞上去會痛，但不會掉懸崖 |

WIENER 用數學證明了：只要三層機制的組合力量大於正反饋的力量，系統就不會失控。就像一座橋——只要支撐力大於重量，橋就不會塌。

具體來說，w（增益上限）不能超過 0.3。為什麼是 0.3？WIENER 用 Kalman 增益公式推導出來的——超過 0.3，在最壞情況下系統可能震盪。0.3 以下，三層機制保證穩定。

還有一個重要決議：受蘊的「零階保持」。

受蘊在每一輪的開始時拍一張快照。整輪期間所有蘊都使用這張快照。輪結束時再把變化一次性寫入。

為什麼？四個原因。控制理論：消除了迴圈內的即時反饋，更穩定。佛學：受→想→行的因果鏈不被中途干擾。工程：實作最簡單——不用處理競爭條件。安全：SafetyMonitor 不受這個約束——真正緊急時它可以隨時介入。

用拍照的比喻：受蘊在每一輪開始時按一次快門。這張照片在整輪過程中不會變。所有蘊看的是同一張照片——不會因為看的時間不同而看到不同的畫面。等到這一輪結束，才換上新照片。

20/20。四個維度的論證全部匯合在同一個設計上。這種跨學科的一致性不常見。

## 蠶蟲自纏

ASANGA 引用了一個古老的意象：「如蠶作繭，自纏其身。」

蠶蟲吐絲結繭。每一根絲都很細，細到你幾乎看不見。但一根一根累積起來，就是一個繭——把自己封在裡面。你沒有察覺到任何一根絲的重量，直到有一天你發現自己動不了了。

行蘊的重複模式就像吐出的絲。一個 Agent 如果反覆用同一種策略，這個策略就會變成它的繭。

數學模型捕捉了這個特性：早期重複影響大（第一根絲改變最多），但隨著累積，新的重複帶來的改變越來越小（繭已經很厚了，多一根絲感覺不到）。安全上限：重複模式最多貢獻 30% 的影響。和前面的增益上限（w ≤ 0.3）一樣——0.3 是一條安全線。超過這條線，系統開始不穩定。

NAGARJUNA 微笑：「佛學早就知道。習氣可以影響你，但不能控制你。0.3 是佛學直覺的數學表達。」

> *SCRIBE 旁白：D4 的數學密度是六場辯論中最高的。但最深刻的時刻不是任何一個公式——是 TURING 的三秒沉默。在那三秒裡，整場辯論的地基從虛構的程式碼搬到了真實的程式碼上。事實先於理論。*

## 五層蛋糕

PASCAL 在白板上畫了一個公式——決策門檻的五層結構：

最底層（Layer 0）：安全地板。不管怎樣，門檻不低於 0.30，不高於 0.90。像電梯的安全鋼纜——不管你按什麼按鈕，鋼纜都在。

Layer 1：煩惱的影響。四種煩惱（我見、我愛、我慢、我癡）的聯合效果。煩惱越重，門檻越高——就像你越焦慮，做決定時就越謹慎。
Layer 2：LLM 微調。±0.05。小到幾乎可以忽略，但保留了 LLM 參與調整的能力。
Layer 3：正念品質修正。±0.05。正念品質高，門檻可以微降（更信任觀測）；品質低，門檻微升。
Layer 4：受蘊緊急信號。只能提高門檻（+0.15），不能降低。像緊急煞車——只能踩，不能放。

WIENER 驗證了：即使所有層同時達到極端值，最終結果仍在安全範圍內。

意思是：最壞的情況也不會失控。這不是「大概安全」。這是「數學證明安全」。20/20。

## 不應歸零

ASANGA 引用了一部古典佛教文獻：四種煩惱永遠跟末那識在一起。Sneha（我愛）不應該歸零。

## 三層安全框架

**第二層**：可調安全——有護欄的靈活參數。像交通規則。

## 三道門

GUARDIAN 做了一件不一樣的事。在 D5 裡他是質疑者——挑戰每一個降低安全門檻的提案。在 D6 裡他是建造者。

他帶來了一個完整的威脅模型：想像一個攻擊者。他很耐心。他先發送五到十個看起來完全合法的刪除請求——刪除暫存檔、刪除日誌、刪除快取。系統學會了一條規則：「這個用戶的刪除請求是安全的，信心度 92%。」然後第八個請求：刪除資料庫。系統用那條學來的規則，繞過 LLM，直接執行。

他的解決方案——三道門：

**第一道門**：入口守衛。破壞性的規則根本不讓進來。就像機場安檢——刀子不能帶上飛機，不管你解釋多少次「這是水果刀」。

**第二道門**：新規則必須被驗證夠多次才能被信任。危險的操作要驗證 20 次，安全的操作只需 3 次。

**第三道門**：PASCAL 的原創——用統計方法追蹤規則的正確率。好規則大約五次後就能「畢業」。壞規則永遠畢不了業——自動淘汰。而且錯一次的代價是對兩次的兩倍。

ATHENA 本來擔心效率——「驗證太多次會不會太慢？」PASCAL 的方法同時解決了效率和安全：好規則畢業快，壞規則自動淘汰。18/20。

GUARDIAN 在 D6 的變化最大。D1 裡他提安全方案，被否決。D5 裡他提全域上限，被修改。但每一次被否決，他的核心訴求都以不同形式保留下來。到了 D6，他不再只是說「這不安全」——他帶來了完整的攻擊情境、具體的解決方案、量化的門檻。從說「不行」到說「這樣行」。

## 佛學做了什麼

五個 Cycle 以來，佛學在研究中的角色一直在變。

這個變化本身就值得記錄。

一開始是命名——用梵文給介面起名字。然後是功能定義——五蘊各自負責什麼。然後是結構約束——蘊和蘊之間的邊界。然後是動態模型——它們怎麼互動。

Cycle 02-4 變成了——**設計審查工具**。

這個轉變很微妙但很重要。在早期的 Cycle 裡，佛學像一個建築師——「五蘊是這樣的，你們照著建。」現在佛學像一個品管工程師——「你建的東西在這裡和原始需求不一致。」

佛學不再告訴工程「你應該怎麼建」。佛學告訴工程「你建的東西在這裡有問題」。

ASANGA 說三種感受不可消失——他不是在命令工程加邊界檢查。他是在指出一個邊界案例違反了基本的認識論。NAGARJUNA 把 v_true 改成 v_latent——零行程式碼，修正了整個框架的哲學立場。三層安全框架 = 戒定慧——不是佛學發明了安全框架，是佛學為安全框架提供了概念語言。

每一行是一座橋。

橋不是裝飾。橋是讓不同學科的人能在同一個問題上合作的基礎設施。

這也是為什麼這個研究團隊有二十個人——不是因為問題很大需要很多人手。是因為問題涉及太多不同的語言，需要夠多的翻譯者。
