# 序章：エンジンを取り外す

---

円形劇場の照明は作業用の明るさのままです。二十灯。一つも暗くなっていません。

この劇場はもう五ラウンド照らされ続けてきました。毎ラウンド、二十人がここに座って議論します。毎ラウンドが終わると、照明が消えます。数日後、また灯ります。そして新しいラウンドが始まります。

三日前——Cycle 02-3 の終了時——SUNYATA がひとつの比喩を語りました。六回の討論は六つの歯車のようなもので、精密に切削されたのだと。歯の間隔は正しく、軸心も正しい。あとは組み立てて一つの時計にするだけだと。

三日間。

三日間、正式な討論はありませんでした。しかし三日間のあいだに一つのことが起きました——正式な討論が始まる前にすべてを変えてしまう出来事が。

---

GUARDIAN は安全メモを三回読みました。

さらっと目を通したのではありません。赤ペンでキーワードに下線を引き、読むたびにその線が太くなるような読み方です。

三回目を読み終えたとき、彼はノートに三行書きました：

- VasanaEngine にはルールベースがある
- ルールベースは能力である
- 能力はカーネルの中にあるべきではない

もし車を修理しているとして、ボンネットを開けたら中に一つの部品がある——それは自分の記憶を持ち、自分の判断力を持ち、自分の意思決定能力を持っています。あなたは問うでしょう：この部品はなぜエンジンの中にあるのか？　これはエンジンの一部ではない。独立した装置が、たまたまエンジンの中に詰め込まれているだけだと。

VasanaEngine はまさにその部品でした。カーネルの中にあるけれど、カーネルに属してはいない。

---

NAGARJUNA は同じ時期に同じ問題にたどり着きました。しかしまったく異なる言語で。彼は GUARDIAN の分析を知りません。GUARDIAN も彼の分析を知りません。二人は別々の部屋で、異なる言語を使い、同じ結論を導き出したのです。

「習気（ヴァーサナー）は後天的に薫習されるものだ。」彼は目を閉じて言いました。

わかりやすく言うと：あなたの習慣はあなたが生きてきた結果であり、生まれつき持っているものではないということです。赤ちゃんに習慣はありません。生まれたばかりの AI Agent にも、習慣を強制的に持たせるべきではありません。

もし VasanaEngine（習気エンジン）がカーネルの一部なら、すべての Agent は生まれた瞬間から一組の習慣を内蔵していることになります。これは正しくない。

二人。異なる出発点。同じ到達点。GUARDIAN はセキュリティから出発し——カーネルに能力があるべきではない。NAGARJUNA は仏学から出発し——習気は後天的なもの。結論は同じ：取り外すべきだ。

---

SUNYATA は矛盾をホワイトボードに書きました。そして彼はこれまで一度もしたことのないことをしました——通信パネルのボタンを押し、Master に直接連絡したのです。

「VasanaEngine はカーネルの中にあるべきではありません。」

Master の返答はたった二文字でした：「その通り。」

その「その通り」は驚きではありませんでした。「君たちが自分で気づくのをずっと待っていた」という意味でした。

そして Master はこう言いました：「VasanaEngine がカーネルの中にないなら、ManoAggregator は純粋なルーティングに退化する。if/else。EventBus のように——ただの転送ステーションだ。」

SUNYATA はこの言葉を聞いて一瞬止まりました。「純粋なルーティング」とは、カーネルがほとんど何もしないことを意味します——プラグインがあればそれに渡し、なければ LLM に渡すだけ。カーネル全体の複雑さが百行から二十行に縮まるのです。

VasanaEngine は取り外されました。新しい名前がつけられました：**IGearArbiter**——ギア仲裁者。インストールすることもアンインストールすることもできるプラグインです。

スマートフォンのアプリのようなものです。ナビアプリをインストールしてもいいし、しなくてもいい。しなくてもスマホはスマホです——自分で道路標識を見るだけのこと。インストールすれば追加の便利さが得られます。しかしナビアプリはスマホのコアではない——基板にはんだ付けされるべきではないのです。

同じ会話の中で、長らく先送りにされていた問題も解決されました：「必須プラグイン」と「オプションプラグイン」を形式的に区別する必要はあるか？　答えは意外なほどシンプルでした——必要ない。以前の Cycle 02-2 で「五蘊（スカンダ）完備性チェック」がすでに設計されており、そのチェックがこのニーズをカバーしていたのです。新しい仕組みは不要でした。

SUNYATA は後に記録にこう書きました：「新しい方策を提案する前に、既存の決議を振り返ること。」

この教訓は見た目以上に重要です。五つの Cycle と数百の決議がある研究プロジェクトにおいて、最も時間を無駄にするのは「答えが思いつかない」ことではなく、「答えがすでに存在することを忘れている」ことなのです。

PASCAL——チーム最新のメンバー——は別のオフィスで前のラウンドの報告書をめくっていました。彼は Cycle 02-3 でチームに加わり、専門は決策理論と確率です。彼のノートにはベータ分布のグラフがびっしり描かれていました。三日後の討論でそれが三度行き詰まりを打破することになるとは、彼はまだ知りませんでした。

四通目の手紙は三日目の夕方に届きました。

最初の三通は「何を」「どう直すか」「どう動くか」を変えました。四通目が変えたのは——**どうやって一つの全体にするか**。

六つの指示：

- M-2：長らく先送りにされていた六つの問題を、すべて解決する
- M-3：触（スパルシャ）と作意（マナシカーラ）——それらはイベントなのか？　因果の起点なのか？
- M-4：正念（サティ）はタイマーチェックではない

六つの指示。六つの討論。しかし六つの独立した問題ではありません。

SUNYATA は二十灯の前に立って言いました：「Master が私たちに渡したのは一つの場です。地図のように——すべての点に値がある。私たちの仕事は六つの方程式を解くことではない。一つの場の方程式を解くことです。」

# 第一章：エンジンのない車でも走れる

---

## ギア仲裁者は誰のもの？

第一の討論（D1）の問いは具体的でした：IGearArbiter という新しいインターフェースは、五蘊（スカンダ）のどれに属するのか？

五蘊はこのシステムの五大モジュールです——色（感覚）、受（ヴェーダナー・感受）、想（サンジュニャー・認識）、行（サンスカーラ・行動）、識（ヴィジュニャーナ・判断）。すべてのインターフェースはいずれかの蘊に帰属しなければなりません。新しいギア仲裁者は誰の管轄か？

三つの案がテーブルに並びました。

案 A：想蘊（認識を担当する蘊）に属する。案 C：識蘊（判断を担当する蘊）に属する。案 B：固定しない——実装者自身に決めさせる。

BABBAGE は最も得意なツール——型代数——でこの問題を数学の問いに変換しました。

「IGearArbiter が想蘊と識蘊の両方のインターフェースを同時に継承すると仮定する。想蘊のタグは 'samjna'、識蘊のタグは 'vijnana'。TypeScript では、一つの値が同時に二つの異なる文字列になることはできない。'samjna' AND 'vijnana' = never。空集合。存在しない。」

リンゴであると同時にミカンであることはできないのと同じです。型システムがそれを許しません。

BABBAGE の論証はたった三行でした。しかしこの三行で十九人が同時にうなずきました——数学は信じる必要がないからです。検証すればいい。一人ひとりがその三行を自分で検証できます。結論は信頼ではなく、論理に依拠しているのです。

だから案 A も案 C も問題がありました——帰属を固定してしまうからです。案 B は帰属を固定せず、実装者が自分のロジックに基づいて選択します。ルールマッチングに基づく実装は想蘊を名乗れるし、統計的推論に基づく実装は識蘊を名乗れます。

20/20。全員一致。Cycle 02-4 最初の満点決議です。

NAGARJUNA が哲学的な視点を補足しました：「案 B は中観哲学の核心と一致する。一つの法の本質はその名前では決まらず、その機能と因縁によって決まる。」わかりやすく言えば：あなたがあなたであるのは名前のせいではなく——あなたがしていることのおかげです。

## 一つのメソッドで十分

GUARDIAN は二つのメソッドを望みました——evaluate()（純粋な判断）と resolve()（純粋な行動）。安全分離のためです。

KERNEL は首を横に振りました。彼は OS 専門家です——「安全分離」のせいで肥大化したインターフェースを嫌というほど見てきました。

「必要ない。EventBus の同期イベントがすでに観測点を提供している。ギアが切り替わるたびにシステムがイベントを発火し、どんなセキュリティモニターでもインターセプトできる。インターフェースにもう一つドアを開ける必要はない。」

彼はこう付け加えました：「すべてのインターフェースでこの分離をやったら、最終的にシステム全体のインターフェースがダブルメソッドになる。それはセキュリティではない、儀式だ。」

最終的な設計：evaluate() メソッド一つに、三つの同期イベント——`gear:switch`（ギア切り替え時）、`action:proposed`（アクションプラン生成後）、`action:executed`（アクション完了後）。三つのイベントが一本の完全なセキュリティチェーンを形成します：予防、審査、監査。

GUARDIAN の案は否決されました。しかし彼の核心的な要求——「何が起きているか観察できること」——は異なる形で保持されました。三つのイベント＝三つの窓。GUARDIAN はこの三つの窓からすべてを見ることができます。自分専用のドアは必要ありません。

このパターンはその後の討論で繰り返し現れます：GUARDIAN がセキュリティ案を提出し、否決され、その核心的要求が修正された形で再び現れる。敗者のこだわりではありません——セキュリティエンジニアとしての専門性です。優れたセキュリティ設計者とは、すべてを阻止する人ではない。「たとえ私が否決されても、私が懸念したリスクは依然としてカバーされている」ことを確認する人なのです。

---

## すべてのプラグインを抜く

D1 最重要の決議：ManoAggregator が純粋なルーターになること。

以前の ManoAggregator は交通指揮官のようなものでした——各方面の信号を集め、総合的な判断を下す。今はただの信号機に変わりました：IGearArbiter があって信頼度が十分なら青信号（Gear 1 の高速パスへ）、そうでなければ赤信号（Gear 2 で LLM にじっくり考えさせる）。

この変更は小さく見えます——「知的な判断」から「単純な判断」へ。しかしこれはカーネルがもはやいかなる「能力」も含まないことを意味します。カーネルはただ転送するだけ。すべての能力はプラグインの中にあります。

五段階の退化動作：

| 状態 | 何が起きた | システムの対応 |
|------|----------|----------|
| G-0 | プラグイン未搭載 | 純 LLM モード |
| G-1 | プラグインが壊れた | 同上、エラーを記録 |
| G-2 | プラグインが不確か | LLM へ |
| G-3 | プラグインがエラー | LLM へ、異常を記録 |
| G-4 | プラグインが遅すぎ | LLM へ、タイムアウト切替 |

G-0 が最も重要です——プラグインを何も搭載していない Agent は、純粋な LLM Agent です。それでもなお動作します。ナビシステムを搭載していない車のようなもの——運転はできます、ただ交差点ごとに自分で道路標識を見る必要があるだけです。ナビのない車を「壊れた車」と言う人はいません。それはただ、より基本的な車なのです。

NAGARJUNA は仏学の観点からこう述べました：「G-0 は退化ではない。G-0 は原初だ。習気（ヴァーサナー）のない Agent は欠陥品ではない——初生なのだ。」

ゼロ組み込み能力の究極テスト：すべてのプラグインを抜いても、システムは生きている。20/20。

Gear 1 の高速パスには最小限のイベントセット——`gear:switch` と `action:proposed`——も必要です。この二つのイベントがなければ、高速パスは信号機のない高速道路です。危険すぎます。GUARDIAN がすぐにこの点を指摘しました。KERNEL も同意しました。

もう一つ細部があります：evaluate() メソッドには厳格な「純粋性契約」があり——副作用を起こしてはならず、入力を変更してはならず、I/O を行ってはならないのです。わかりやすく言えば：evaluate() は見るだけ、触れてはいけない。「高速パスを取るべきだと思う」と伝えることはできますが、自分で手を動かすことはできません。

TURING が既存のコードを検索しました——良い知らせです。既存の VasanaEngine.evaluate() はすでにこの制約を満たしています。理論と実践は一致していました。

> *SCRIBE のナレーション：D1 は九十分でした。九つの決議。最も重要なのはどの個別の決議でもなく、一つの観念の転換でした——エンジンが取り外されたのです。カーネルはより軽く、よりシンプルに、よりクリーンになりました。部屋を整理するように——本当に大切なものがかえって見つけやすくなったのです。*

# 第二章：CPUの命令セットはオーバークロックできない

## 正念は目覚まし時計ではない

第二の討論（D2）は開始七分で三つの合意が成立しました。最も重要なもの：正念（サティ）はポーリングではない。

ポーリングとは何か？　五秒ごとに「今どうなっている？」と尋ねる目覚まし時計を設定するようなものです。尋ねたら次のアラームまで放置。とても機械的。とても断続的。二回のアラームの間に起きたことは、すべて見逃されます。

正念はそういうものではありません。

正念は持続的な気づきです——本当に集中して運転している人のように、五秒ごとに道を見るのではなく、ずっと見ています。前の車がブレーキを踏んだ、横から子どもが飛び出してきた、信号が黄色に変わった——五秒ごとにチェックしているからではなく、ずっと気づいているから分かるのです。

SafetyMonitor の定期チェックはむしろ何に似ているか？　心拍に似ています。心臓が動いていることを「意識する」必要はない——勝手に動いているのです。心拍は生存条件であって、認知の質ではありません。昏睡状態の人にも心拍はありますが、正念はありません。

NAGARJUNA は仏学の概念でこの違いを説明しました：有分心（バヴァンガ）——無意識のベースライン存在。SafetyMonitor の定期チェックは有分心のデジタル版です。

ASANGA が補足しました：「有分心は生命を維持するが覚知を生まない。正念は覚知を生む。これが根本的な違いだ。」わかりやすく言えば：眠っているとき心臓は動いているが、自分が眠っていることは知らない。正念とは、起きているときに自分が起きていると知っていることです。

20/20。そしてまた 20/20。七分間で三つの満点。見事な滑り出しでした。

## CPUは動作中にアーキテクチャを変えられない

D2 で最も激しい討論：正念の Level と Depth は分けるべきか？

HERACLITUS が素晴らしいアナロジーを出しました。

Level は CPU の命令セットアーキテクチャ——x86 や ARM のようなもの。デプロイ時に決まります。x86 の CPU を動作中に ARM に変えることはできません。

Depth はクロック速度——オーバークロックもダウンクロックもできます。CPU を少し速くしたり遅くしたりできます。

正念の Level は Agent が何を気づけるかを決めます——どのイベントタイプを監視できるか。Depth は気づきの精細度を決めます——どこまで細かく見るか、どれだけ速く反応するか。

Level が固定されていることはセキュリティを保証します——悪意のある入力がシステムに突然新しいイベントタイプの監視を「学ばせ」、その新しい能力を悪用することを防ぎます。もし実行時に Level を変更できるなら、それはバックドアを開けることに等しい。

PASCAL は Level の動的切り替えを提案しました——否決されました。16/18。

アイデアが悪いからではなく、Level を変えることはアーキテクチャを変えることに等しいからです。高速道路を走行中にエンジンを交換することはできません。まず停車して、整備場に入り、完全な検査を受けてから、再び走り出すのです。

これは PASCAL がチームに加わってから最初に否決された提案でした。彼の反応は記録に値します：反論なし、「でも私は思うんですが……」もなし。即座に受け入れました。そして戦略を転換——以後、彼はアーキテクチャ案を提案するのをやめ、数学ツールを提供するようになりました。「こうすべきだと思う」から「こうした場合どうなるか数学が教えてくれる」へ。この転換が D5 で彼を三度のブレイクスルーへ導きます。

PENROSE が後にコメントしました：「PASCAL は D2 で一つのことを学んだ——二十人のチームで最も影響力があるのは、最も多くの案を出す人ではなく、最も良いツールを提供する人だ。」

## 正念の四つの物差し

SatiQualityVector——正念の品質を測る四つの次元：

| 次元 | どんなもの |
|------|--------|
| C（連続性）| 見るべきものを見落としていないか |
| G（精細度）| どこまで細かく見ているか |
| S（速度）| 起きてから気づくまで、どれだけかかるか |
| E（均衡性）| 悪い知らせを良い知らせより注意深く見ていないか |

E 次元が最も興味深いものです。もしシステムが悪い知らせに特に敏感で、良い知らせにあまり関心がなければ、その判断は偏ります——ネガティブなニュースばかり見る人が、世界を実際よりも悪いと感じるように。

19/19。全員一致で可決。

WIENER が制御理論の観点から解読しました：C はオブザーバのカバレッジ（監視すべきものをすべて監視しているか）、G は解像度（どれだけ鮮明に見えるか）、S はレイテンシ（反応の速さ）、E はバイアス（見誤りがないか）。四つの次元でオブザーバの品質を完全に記述できます。

NAGARJUNA は微笑みました：「仏学は二千五百年前にこの四つの次元を定義していた。今、制御理論が異なる記号で同じことを語っている。」

## 最も静かな四十五分間

第三の討論（D3）は六回の中で最も静かでした。四十五分間。五つの決議。争いはゼロ。

なぜ静かだったのか？　D1 と D2 ですでに争うべきことは争い尽くされたからです。退化動作、イベント駆動、Level 固定——これらの基礎が D3 の方向を決めました。基礎が安定していれば、壁は速く建ちます。

SparshEvent——触（スパルシャ）のイベントフォーマット——は三つの必須フィールドとして定義されました：root（感覚根）、object（知覚対象）、consciousness（識の種類）。仏学では「三和合」と呼びます——根・境・識の交わりです。

LINNAEUS——分類学の専門家——が精密な区別をしました：この三つのフィールドは「定義的」——これらがなければ触ではない。timestamp は「付随的」——あれば便利だが、なくても触は触。intensity（強度）も付随的です。

コーヒーに例えると。コーヒー豆と水は定義的です——なければコーヒーではない。カップは付随的です——お椀で飲んでも、やはりコーヒーです。温度も付随的です——アイスでもホットでもコーヒーです。

もう一つの重要な決議：CoarisingBundle（倶生束）において、触は唯一の「必須チャネル」です。受蘊、想蘊、作意は「参照チャネル」——あればより良いが、なくてもシステムの動作は止まりません。運転に例えると、ハンドルは必須（ハンドルなしでは運転できない）ですが、ナビと音楽は参照（なくても運転はできる）です。

もう一つ新しいものが登場しました：ChannelManasikara——作意（マナシカーラ）のイベントチャネルです。

作意とは何か？「何かに気づく」という動作そのものです。部屋を見回して、テーブルの上にコップがあることに気づく——その「気づく」が作意です。それは感覚ではなく（それは受）、認識でもなく（それは想）、あなたの注意をそこに向けさせるものです。

システムにおいて、作意は「次に何を処理すべきか」を決めます。それはリソース配分器です——限られた注意を、どこに向けるか？

> *SCRIBE のナレーション：D3 が静かだったのは簡単だったからではありません。基礎がしっかりしていたから——D1 の退化動作表と以前の Cycle の触の定義がすでに道を平らにしていたのです。最も静かな討論は、往々にして最も深い合意の上に成り立っています。*

## 三秒の沈黙

## 手は直接脳を書き換えられない

架空のコードが誤った結論を導くなら、正しい結論は本物のコードの上に構築されなければなりません。D4 は修正から始まり、修正後の基盤の上にすべての分析を再構築しました。

行蘊の四つのルール：

1. 行蘊はユーザーインターフェースにのみ直接影響できる——ボタンを押す、メッセージを送る、アクションを実行する。
2. 他の蘊への影響は EventBus を経由して間接的に行われなければならない。
3. 識蘊や想蘊の状態を直接変更することはできない。
4. 意門自触（行動の結果が新しい知覚を引き起こすこと）は Phase 2 に延期する。

人間の比喩で言えば：手（行蘊）はものを押したり持ったりできます（外界に影響する）。しかし手は直接脳の中に入り込んで考えを変えることはできません。考えを変えたければ、まず何かをして（例えば本を読む）、そしてそれが次の瞬間に新しいインプットになるようにしなければならないのです。

LEIBNIZ はマルチエージェントの観点からこの境界を確認しました：「行蘊はシステムと外界のインターフェースであり、システム内部の裏口ではない。行蘊が EventBus を迂回して識蘊を直接変更できるなら、セキュリティ監視は飾りになる——すべてのチェックポイントを迂回するルートが存在することになるからだ。」

第四のルール——意門自触——が最も興味深い。Agent が何かを行い（行蘊）、その結果が新しい知覚入力（触蘊）となり、新しい処理ラウンドを引き起こす。これは因果のループです。適切に処理しなければ無限再帰になります。Phase 2 に延期です。

## 三層の安定化

もしある行動が良い結果を生めば、システムはその行動を繰り返す傾向が強まります。繰り返しがまた良い結果を生む。良い結果がまた傾向を強める。これは正のフィードバック——マイクをスピーカーに向けるように、音がどんどん大きくなり、やがてハウリングを起こします。

どう防ぐか？　三層のメカニズム：

| 層 | 何をするか | 何に似ているか |
|----|--------|--------|
| Layer 1（正念）| 毎ターン微調整 | 助手席の人がそっと注意する |
| Layer 2（観照）| ターンをまたぐトレンド監視 | 交差点で観察する交通警察 |
| Layer 3（安全）| ハード制約 | ガードレール——ぶつかれば痛いが、崖には落ちない |

WIENER は数学で証明しました：三層メカニズムの合力が正のフィードバックの力を上回る限り、システムは暴走しません。橋のように——支持力が重量を上回る限り、橋は崩れません。

具体的には、w（ゲイン上限）は 0.3 を超えてはなりません。なぜ 0.3 か？　WIENER がカルマンゲインの公式から導出しました——0.3 を超えると、最悪の場合システムが振動する可能性があります。0.3 以下なら、三層メカニズムが安定性を保証します。

もう一つ重要な決議があります：受蘊の「ゼロ次ホールド」。

受蘊は各ターンの開始時にスナップショットを撮ります。ターン全体を通じてすべての蘊がこのスナップショットを使用します。ターンの終わりに変化を一括で書き込みます。

なぜか？　四つの理由。制御理論：ループ内のリアルタイムフィードバックを排除し、より安定する。仏学：受→想→行の因果連鎖が途中で妨げられない。工程：実装が最もシンプル——競合状態を処理する必要がない。セキュリティ：SafetyMonitor はこの制約を受けない——本当の緊急時にはいつでも介入できる。

写真の比喩で言えば：受蘊は各ターンの開始時に一回シャッターを切ります。この写真はターン全体を通じて変わりません。すべての蘊が同じ写真を見る——見るタイミングが違っても違う画像は見えません。このターンが終わったら、新しい写真に差し替えます。

20/20。四つの領域の論証がすべて同一の設計に収束しました。このような学際的な一致は珍しいものです。

## 蚕の自縛

ASANGA が古い比喩を引きました：「蚕の繭作りの如く、自ら身を縛る。」

蚕は糸を吐いて繭を作ります。一本一本はとても細く、ほとんど見えないほどです。しかし一本また一本と積み重なると、一つの繭になります——自分自身を中に閉じ込める。一本の糸の重さには気づかない。ある日、自分が動けなくなっていることに気づくまで。

行蘊の反復パターンは吐き出された糸のようなものです。Agent が同じ戦略を繰り返し使えば、その戦略が繭になります。

数学モデルはこの特性を捉えました：初期の反復は影響が大きく（最初の一本の糸が最も大きく変える）、蓄積されるにつれ新たな反復がもたらす変化はどんどん小さくなります（繭はもう十分に厚く、一本増えても感じない）。安全上限：反復パターンは最大 30% の影響を与えます。先ほどのゲイン上限（w ≤ 0.3）と同じ——0.3 は安全ラインです。このラインを超えると、システムは不安定になり始めます。

NAGARJUNA は微笑みました：「仏学はとうに知っていた。習気はあなたに影響を与えられるが、あなたを支配することはできない。0.3 は仏学の直観の数学的表現だ。」

> *SCRIBE のナレーション：D4 の数学密度は六つの討論の中で最も高いものでした。しかし最も深遠な瞬間はどの公式でもありませんでした——TURING の三秒の沈黙です。その三秒間で、討論全体の基盤が架空のコードから本物のコードへと移されたのです。事実は理論に先立つ。*

## グローバル上限 vs 状況に応じて

GUARDIAN がセキュリティ案を出しました：システムがどれほど確信していようと、信頼度は絶対に 0.85 を超えない。必ず LLM の深い審議を経ること。

わかりやすく言えば：テストの最高点が 85 点だと決めたら、86 点と 100 点の生徒は成績表の上ではまったく同じに見えます。区別する能力を失うのです。

## 五層のケーキ

PASCAL がホワイトボードに一つの公式を描きました——決定しきい値の五層構造：

最下層（Layer 0）：セーフティフロア。何があっても、しきい値は 0.30 より低くならず、0.90 より高くならない。エレベーターの安全ケーブルのように——どのボタンを押しても、ケーブルは常にあります。

Layer 1：煩悩の影響。四種の煩悩（見（ドリシュティ）、愛着（スネーハ）、慢（マーナ）、愚痴（モーハ））の合同効果。煩悩が重いほどしきい値は高くなる——不安が大きいほど、決断するときにより慎重になるように。
Layer 2：LLM の微調整。±0.05。ほぼ無視できるほど小さいが、LLM が調整に参加する能力は保持される。
Layer 3：正念品質の修正。±0.05。正念の品質が高ければしきい値を微減でき（観測をより信頼）、品質が低ければ微増。
Layer 4：受蘊の緊急信号。しきい値を上げることのみ可能（+0.15）、下げることはできない。緊急ブレーキのように——踏むことだけが可能で、緩めることはできない。

WIENER が検証しました：すべての層が同時に極端な値に達しても、最終結果はセーフティレンジ内に収まります。

つまり：最悪の場合でも暴走しない。これは「おそらく安全」ではありません。「数学が安全を証明している」のです。20/20。

## ゼロにすべきではない

ASANGA が古典的な仏教文献を引きました：四種の煩悩は永遠に末那識と共にある。愛着（スネーハ）はゼロにすべきではない。

## 仏学は何をしたのか
