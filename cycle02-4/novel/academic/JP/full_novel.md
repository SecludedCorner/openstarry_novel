## I. 蘊の帰属をめぐる争い

D1 は Cycle 02-4 の起点――IGearArbiter の蘊帰属問題。Pre-DC で VasanaEngine の IGearArbiter plugin への外部化はすでに確認されていた。今の問題は：このインターフェースはどの蘊に属するのか？

三つの方案。方案 A――純想蘊(samjñā)（IGearArbiter extends ISamjna）。方案 B――蘊横断の二層構造（インターフェース自体はいかなる蘊のルートインターフェースも継承せず、実装者が自身のロジックに基づいて帰属を選択する）。方案 C――純識蘊(vijñāna)（extends IVijnana）。

BABBAGE の型代数分析が結末を決した。

方案 A の問題：IGearArbiter が ISamjna を extends すると、その蘊帰属は `'samjna'` となる。同時に IVijnana も extends すると、型システムは `'samjna' & 'vijnana' = never` を導出する――空型。TypeScript は一つの値が同時に二つのリテラルであることを許さない。蘊横断帰属は型レベルで不可能である。

方案 C の問題：純識蘊では evaluate() 中の「パターン認識」機能を説明できない――認識は想蘊の職責である。認識を識蘊に帰属させることは五蘊の機能境界に違反する。

方案 B の利点：蘊帰属を固定しない。インターフェース自体は振る舞い契約のみを定義し、いかなる蘊のルートインターフェースも継承しない。実装者が自身のロジックに基づいて帰属を選択する――規則マッチングに基づく実装は想蘊に帰属でき、統計的推論に基づく実装は識蘊に帰属できる。これは DC-6「蘊内サブインターフェースの帰属を固定しない」という裁定に合致する。

20/20。全会一致。型代数が哲学的論争を数学的証明に変換した。

NAGARJUNA は投票後に哲学的確認を補足した：「方案 B の『蘊帰属を固定しない』は中観の核心と一致する。一つの法の本質はその名前によって決まるのではなく、その機能と因縁によって決まる。IGearArbiter の蘊帰属がその実装に依存すること――これは縁起の正しい表現である。」

DARWIN のソフトウェアパターン観察：方案 B は本質的に Strategy Pattern である――インターフェースが振る舞い契約を定義し、実装者が具体的な戦略を提供する。異なる戦略は異なる蘊に属しうる。これはソフトウェア工学において数十年にわたり検証されてきたパターンである。

## II. 単一メソッドと安全性

GUARDIAN は evaluate() と resolve() の二メソッド設計を提案した――evaluate() は純粋な認識、resolve() は純粋な行動。安全分離。これは Command-Query Separation 原則の認知アーキテクチャへの適用である。

KERNEL はこの方案を否決した。彼の論証はオペレーティングシステムの観点からだった：「Unix 哲学では、監視と被監視は分離される。`strace` はトレース対象のプロセスの協力を必要としない――外部からシステムコールを観察する。同様に、EventBus の同期イベントセマンティクスがすでに等価な安全観測点を提供している。resolve() の職責は `gear:switch` 同期イベントを通じて外部の監視者が観察できる――インターフェースレベルで分割する必要はない。」

彼はさらに鋭い補足を加えた：「もしすべてのインターフェースで CQS を行えば、最終的にはすべてのインターフェースが二メソッドを持つシステムになる。これは安全ではない、儀式だ。」

SUNYATA は evaluate() 単一メソッド + EventBus 同期イベント + 閾値安全因子に統合した。GUARDIAN の核心的訴求――可観測性――は保持された、ただ異なるメカニズムで実現されただけだ。20/20。

isGearArbiter() の型ガードは構造型（ダックタイピング）を使用し、名目型ではない：

```typescript
function isGearArbiter(x: unknown): x is IGearArbiter {
  return typeof x === 'object' && x !== null
    && typeof (x as any).evaluate === 'function'
    && (x as any).evaluate.length <= 1;
}
```

instanceof なし、マーカープロパティなし。正しいシグネチャを実装したいかなるオブジェクトもガードを通過できる。これはサードパーティの plugin が IGearArbiter 型をインポートせずとも識別されうることを意味する――振る舞いが正しければ十分だ。20/20。

evaluate() の純粋性契約――副作用を生じてはならず、GearContext を変更してはならず、I/O を開始してはならない。この三つの制約の重要性は非対称である。副作用禁止は evaluate() が安全にリトライ可能であることを保証する。GearContext の読み取り専用は歯車仲裁者が自身の入力を改竄できないことを保証する。I/O 禁止は evaluate() のレイテンシが予測可能であることを保証する。

TURING はコードの事実を報告した：v0.26.0-beta の VasanaEngine.evaluate() は確かに純関数である――書き込み操作なし、外部呼び出しなし、状態変更なし。既存のコードはすでにこの制約に適合している。理論と実践の一致――研究チームが設計した制約はまさに工学チームがすでに遵守していたパターンだった。19/20。ARCHIMEDES の一票棄権は「I/O 禁止」が Gear 1 で外部知識ベースの問い合わせが必要な場合の制限に対する懸念だった――ただしこれは将来の拡張に関する考慮である。

## III. ManoAggregator の純ルーティング化

D1 の最も重要な決議：ManoAggregator の純ルーティング化 + G-0〜G-4 退化行動表。

ManoAggregator はインテリジェントな集約を行わない。if/else を行う：

```
if (hasGearArbiter && arbiter.evaluate(context).confidence > threshold) {
  return Gear.ONE;  // 高速パス
} else {
  return Gear.TWO;  // LLM 深度審議
}
```

EventBus と同等の性質――純粋なメカニズムであり、能力ではない。Pre-DC 段階での Master の原文：「EventBus と同等の性質。」この一文の重みは、ManoAggregator を「コアのインテリジェントコンポーネント」から「インフラストラクチャ」に格下げしたことにある。インフラストラクチャは意思決定を行わない――意思決定を伝達する。

G-0 から G-4 は五段階の退化行動：

| 等級 | 条件 | 行動 |
|------|------|------|
| G-0 | IGearArbiter plugin なし | 純 Gear 2（純 LLM）|
| G-1 | plugin のロード失敗 | G-0 と同じ、エラーを記録 |
| G-2 | evaluate() が低信頼度を返す | Gear 2 へ |
| G-3 | evaluate() が例外をスロー | Gear 2 へ、例外を記録 |
| G-4 | evaluate() がタイムアウト | Gear 2 へ、タイマー発火 |

システムがどの状態にあっても、Agent は動作可能である。G-0 が最も重要だ――IGearArbiter をインストールしていない Agent は純 Gear 2 Agent である。それはなお完全で動作可能な Agent である。ゼロ組込み能力の究極テスト：すべての plugin を抜いても、システムはなお生きている。

20/20。

退化行動表の設計哲学は掘り下げる価値がある。従来のソフトウェア設計では、例外処理は「エラー状況」であり――システムは正常状態への復帰に努める。G-0〜G-4 では、退化はエラーではない――システムの別の正常状態である。G-0 Agent と G-4 Agent はどちらも「正常に動作する Agent」であり、能力レベルが異なるだけだ。これは仏教の観点と暗合する：正念(sati)の減弱は「故障」ではなく、「条件の変化」である。

VITRUVIUS はアーキテクチャの観点から確認した：G-0〜G-4 の退化行動表はすべての可能なシステム状態をカバーしている。「未定義動作」の状態は存在しない――これは安全なシステムの基本要件である。

## IV. Gear 1 の最小イベントセット

三つのイベント：`gear:switch`、`action:proposed`、`action:executed`。すべての Gear 1 アクションはこの三つの同期イベントを発火しなければならない――スキップ不可、非同期不可。

`gear:switch` は歯車切替の瞬間に発火される――いかなる安全監視者もこのイベントにインターセプターを設定できる。監視者が切替を安全でないと判断した場合、イベント処理で切替を阻止できる。`action:proposed` はアクションプランの生成後、実行前に発火される――これは最後の拒否権ウィンドウである。外部のセキュリティモジュールがアクションプランを審査し、通過させるかどうかを決定できる。`action:executed` はアクション完了後に発火される――監査証跡を提供し、システムがすべての Gear 1 アクションの完全な履歴を記録できるようにする。

三つのイベントは完全な安全チェーンを形成する：予防（gear:switch）→ 審査（action:proposed）→ 監査（action:executed）。

これは GUARDIAN が D1 で唯一成功裏に推進した安全メカニズムである。evaluate()/resolve() の二メソッドは否決されたが、同期イベントは外部安全監視の完全性を保証した。GUARDIAN の核心的訴求は異なる形式で保持された――このパターンは D5 で繰り返し現れる。GUARDIAN はすべての討論で方案を提出し、否決され、修正された形式で再登場する。敗者の固執ではない、安全エンジニアの専門的資質である。

## V. D1 の構造的意義

D1 の九項目の決議は共同で一つの構造的転換を完成させた：

**以前**：VasanaEngine はコアコンポーネントであり、規則庫とマッチングロジックを持っていた。ManoAggregator はインテリジェントアグリゲータであり、複数のシグナルソースの結果を統合していた。歯車選択ロジックはコアにハードコードされていた。

**以後**：IGearArbiter はオプショナルな plugin であり、純関数インターフェースである。ManoAggregator は純ルーターであり、if/else ロジック。歯車選択ロジックは外部化された。plugin なしでシステムは純 Gear 2 に退化する――なお完全に使用可能。三つの同期イベントが完全な安全観測チェーンを保証する。

この転換の哲学的意義を NAGARJUNA は正確に表現した：「習気(vāsanā)を持たない Agent は欠陥のある Agent ではない――初生の Agent である。習気は後天的な薫習の結果であり、先天的な構造ではない。G-0 状態は退化ではなく、原初である。」

ARCHIMEDES は工学の観点から転換の影響を定量化した：VasanaEngine の外部化により、コアから約 200 行の規則マッチングロジックが除去され、約 15 行の IGearArbiter インターフェース定義と約 30 行の ManoAggregator 純ルーティングロジックに置換される見込みである。コアの認知的複雑性は大幅に低下する――「規則庫がどう動作するかを理解する」から「if/else がどう動作するかを理解する」へ。

MESH は Level 固定に反対票を投じたが（D2）、D1 のすべての決議には全票で賛成した。彼は後にこう説明した：「D1 のコアコンポーネントを plugin に変える方向性は正しい。D2 での私の反対は Level の柔軟性に関するものであり、外部化の原則に関するものではない。」

## I. 正念はポーリングではない

D2 は三項目の迅速な合意で幕を開けた――七分間、すべて 20/20。最も重要な一項：正念(sati)はイベント駆動であり、ポーリングではない。

Master の第四の書簡 M-4 は言った：正念はポーリングではない。研究チームは R1 段階ですでに圧倒的合意に達していた――SafetyMonitor の定時ポーリングは liveness probe（生存性探査）であり、正念ではない。正念はあらゆる認知イベントへの即時の気づきであり、品質属性であり、サンプリング行為ではない。

正念をポーリングと同一視することは、「注意深く運転する」を「五秒ごとに道路を見る」と同一視するようなものだ。前者は持続的な品質状態であり、後者は間欠的な機械的動作である。

## II. CPU の ISA はオーバークロックできない

D2 で最も激しい討論は Level と Depth の分離をめぐって起きた。

HERACLITUS は CPU のアナロジーを提示した：Level は命令セットアーキテクチャ（ISA）――x86 か ARM か、デプロイ時に決定され、ランタイムでは変更不可。Depth はクロック速度――オーバークロックもダウンクロックも可能、ランタイムで動的に調整。

正念の Level は Agent が何を覚知できるか（イベントタイプ、チャネル数）を決定する。正念の Depth は覚知の精細度（粒度、速度、均衡性）を決定する。

Level 固定は意味する：Agent はデプロイ後に突然新しいイベントタイプの監視を「学習」しない。これは安全の保証である――悪意のある入力が Level の変更を通じて監視をバイパスすることはできない。x86 の CPU をランタイムで ARM に変えることはできない。KERNEL はオペレーティングシステムの観点から補足した：「これはシステムコールテーブルと同じだ。ランタイムでシステムコールテーブルを変更することはできない――それはカーネルの基本的な安全保証だ。」

PASCAL が提案した Runtime Level の動的切替は否決された――概念が誤りだからではなく、Level の変更はアーキテクチャ変更に等しく、パラメータ調整ではないからだ。もしシステムがランタイムで突然新しいイベントタイプの監視を始めたら、元のイベントセットに依存するすべての安全ロジックは再検証が必要になる。これはパラメータ調整ではない――再デプロイだ。完全な安全監査が必要である。

16/18、二票反対（MESH は分散シナリオでは動的 Level が必要と主張――ノードが参加・離脱する環境では、監視能力の動的調整が必要かもしれない）。これは PASCAL がチームに加わって以来、初めて否決された提案だった。彼の反応は記録に値する：議論なし、留保意見なし、即座に受け入れて戦略を調整した。後続の討論（D4、D5、D6）で、PASCAL のすべての介入はより精確になった――彼はこのチームで最も効果的な方法はアーキテクチャ方案を提出することではなく、数学的ツールを提供することだと学んだ。

## IV. 心拍は正念ではない

D2-R4 は一つの重要な概念混同を明確にした。

心拍 = liveness probe。正念 = event-driven awareness。前者は生存条件であり、後者は認知品質である。昏睡状態の人は心拍があるが正念はない。覚醒して集中している人は両方を兼ね備えている。

NAGARJUNA の補足は正確を極めた：「有分心(bhavanga)はアビダルマの概念――心の流れがいかなる対象にも占められていない時の基線状態。それは意識の最低限度の存在であり、注意ではない。SafetyMonitor の定時ポーリングはまさにこれ――システムの最低限度の存在確認であり、認知品質ではない。」

## V. EVOI → MRA

PASCAL が R1 で提案した EVOI（Expected Value of Information）フレームワークは MRA（Marginal Risk Analysis）に改名された。理由：EVOI は学術文献における汎用用語であり、OpenStarry の専用概念として使用すると混同しやすい。

三層命名法：内部開発では EVOI、設計文書では MRA、仏教シナリオでは「正念資糧」。17/18。BABBAGE の留保意見は学術的一貫性だった――しかし多数は明瞭さが慣例に優先すると判断した。

この命名決議は微小に見えるが、研究全体の核心的課題を反映している：同一の概念が異なる学問分野では異なる言語を持つ。仏教は「正念」と言い、工学は「event-driven monitoring」と言い、制御理論は「オブザーバ」と言う。命名は装飾ではない、橋である。

## VI. 三和合の静寂

D3 は六場の討論中最も静寂だった――四十五分間、五項目の決議、ゼロの争議。

SparshEvent は触(sparsha)の三つの必須フィールドを定義した：

```typescript
interface SparshEvent {
  root: SenseBase;           // 根（六根：眼耳鼻舌身意）
  object: CognitiveObject;   // 認知対象の要約
  consciousness: VijnanaType; // この触を生じた識の類型
  timestamp?: number;        // 附随的属性
}
```

三和合――根、境、識の交わり。あらゆる認知イベントはこの三者の合流から始まる。

LINNAEUS は定義的属性と附随的属性を区別した：三つの必須フィールドは定義的である（それらなしでは触ではない）、timestamp は附随的である（追跡に有用だが触の本質を変えない）。この区別は存在論的な精確性を持つ――定義的属性は事物の本質を構成し、附随的属性は偶然の付随にすぎない。

Phase 1 の一対一触モデル――各 processEvent は一つの SparshEvent のみを生成する。将来的に複数触への拡張は可能だが、初期設計は最小を維持する。

CoarisingBundle の mandatory/reference 境界は五遍行の依存関係を定義した：

| Channel | 類型 | 欠如時の挙動 |
|---------|------|------------|
| sparsha | mandatory | bundle 全体の構築不可 |
| vedana | reference | 最新スナップショットを保持、欠如しても阻止しない |
| samjna | reference | 最新スナップショットを保持、欠如しても阻止しない |
| manasikara | reference | 最新スナップショットを保持、欠如しても阻止しない |

触は唯一の mandatory channel である。これは仏教の「触が先導する」と一致する：触なくして他の心所は生起しえない。しかし触は他の心所の準備を待つ必要なく存在できる。D1 の G-0〜G-4 退化ロジックと対称的――reference channel の欠如は退化に等しく、故障には等しくない。

ChannelManasikara は作意(manasikara)の二つの必須次元を定義した：focus（注意の焦点、現在の注意が向けられている対象）と intensity（注意の強度、どれだけの認知リソースが投入されているか）、加えてオプションの dimensions?（拡張可能な追加次元）。

PENROSE は興味深い問題を提起した：作意は「作意への覚知」を持ちうる――メタ認知次元。システムは何に注意しているかを知るだけでなく、どのように注意しているかも知る。この次元は Phase 2 に延期された――重要でないからではなく、メタ認知の実装には再帰構造が必要であり、Phase 1 の線形パイプラインでは自然に表現できないからだ。

D3 の五項目の決議は四十五分間ですべて通過した。一つも反対を受けなかった。VITRUVIUS はアーキテクチャの観点から意義ある観察を行った：SparshEvent の三必須フィールド設計は実質的にシステムの「知覚インターフェース」を定義している――すべての外部イベントがシステムに入る唯一の入口。これにより安全監査が実行可能になる――分散した複数の入口ではなく、一つの入口点を監視するだけでよい。

D5 を暴風雨に喩えるなら、D3 は晴天である――しかしこの二者の間には因果関係がある。D3 の静寂が D5 の嵐のためにエネルギーを蓄えた。SparshEvent の安定した定義が D5 を閾値論争に集中させ、基礎概念の再定義に立ち戻る必要をなくした。

## II. 行蘊の境界

四項目の迅速な合意が行蘊の厳格な制限を確立した：

1. 行蘊が直接影響できるのは IUI（ユーザーインターフェース蘊）のみ――他の蘊へのすべての影響は EventBus を介して間接的に行われなければならない。
2. 行蘊は識蘊や想蘊の状態を直接変更しない。
3. 意行は EventBus を介してサイクルをまたいで受蘊に影響する。
4. 行蘊の結果が新たな触（意門自触）を誘発することは Phase 2 に延期。

これらの制限は行蘊の「禁止事項」リストを構成する。行蘊は手である――外界（IUI）を押すことはできるが、直接脳（識蘊、想蘊）に手を伸ばして思考方式を変えることはできない。思考方式を変えるには、まず外部状態を変え、外部状態を次のラウンドで新たな入力とさせなければならない。

LEIBNIZ はマルチエージェント協力の観点からこの制限の必要性を確認した：もし行蘊が他の蘊の状態を直接変更できるなら、一つのモジュールがインターフェースをバイパスして別のモジュールの内部状態を直接変更できることと等しい。分散システムでは、これは競合状態と再現不能なバグの主要原因である。

NAGARJUNA の仏教的確認はさらに根本的だった：行蘊が識蘊や想蘊を直接変更することは、「果が直接因を変更する」ことに等しい。縁起(pratītyasamutpāda)の観点では、因果の連鎖は媒介を経なければならない――行動が環境を変え、環境が次の刹那に新たな縁起条件となる。媒介を飛ばすことは因果を飛ばすことだ。

これらの四項目の制約は D4 の後続のすべての議論で頻繁に引用された――行蘊モデルの基本定理であり、すべての後続の導出はこの基盤の上に構築される。ARCHIMEDES はこれらを「行蘊の四公理」と呼んだ――証明を必要とせず、違反もできない基本規則。

注目すべきは第四の制約――意門自触の延期である。行蘊の結果は新たな触(sparsha)を誘発しうる、ちょうど行動の結果が新たな知覚対象となりうるように。しかしこのメカニズムは因果の循環を含む：行蘊 → 新触 → 新受 → 新想 → 新行蘊。Phase 1 の線形パイプラインでこの循環を実装するには追加の設計が必要であり、Phase 2 に延期された。

## III. ゲイン上限と安定性

D4-R1：samskara → vedana フィードバックゲイン上限 w ≤ 0.3。

PASCAL が Bayesian 的根拠を提供した。カルマンゲインの導出：離散時間カルマンフィルターにおいて、ゲイン K = P·H'·(H·P·H' + R)⁻¹。プロセスノイズ Q が観測ノイズ R よりはるかに大きい場合（高不確実性/高観測品質）、ゲインは上界に近づく。PASCAL は OpenStarry の典型的な動作条件下でこの上界を推定した：K₁ ≈ 0.308。

工学的に 0.3 を保守値として採用――理論上界よりわずかに低く、安全マージンを残す。8/9（ARCHIMEDES 棄権――具体的な数値はデプロイ環境の差異に応じて設定すべきであり、0.3 をデフォルト値とすることに問題はないが、ハードコードすべきではないとの見解）。

D4-R2：三層安定化メカニズム――本ラウンドで最も深い学際的成果の一つ。

| 層 | メカニズム | 仏教的マッピング | 特性 |
|----|------|---------|------|
| Layer 1 | Sati 微調整（毎ラウンド微調整）| 正知 (sampajanna) | 連続、小幅 |
| Layer 2 | VitakkaWatchdog 粗フィードバック（ラウンド間トレンド）| 尋 (vitakka) 観照 | 間欠、中幅 |
| Layer 3 | SafetyMonitor ハードコンストレイント（即時発火）| 戒 (sila) ガードレール | 離散、大幅 |

WIENER の Lyapunov 安定性解析が数学的保証を提供した。リアプノフ関数 V = Σ(x_i - x_i*)² を定義、ここで x_i は各蘊の状態、x_i* は期待される平衡点。安定性条件：

```
k_positive_feedback × w_samskara < k_stabilization × sati_min
```

ここで k_positive_feedback は正フィードバックゲイン（行蘊の反復パターンの強化効果）、w_samskara は行蘊の重み（≤ 0.3）、k_stabilization は安定化ゲイン（三層メカニズムの複合効果）、sati_min は最低正念レベル。

具体的には、ある行動が正のフィードバックを生み、システムがその行動を反復する傾向を強めた場合、三層安定化メカニズムがこの正フィードバックループの暴走を防止する：
- Layer 1（Sati）が持続的に微調整――毎ラウンド小幅な修正
- Layer 2（VitakkaWatchdog）がトレンドを監視――偏差が蓄積した時に中幅の介入
- Layer 3（SafetyMonitor）がハードカット――すべてが失敗した時に大幅な修正

三層メカニズムの複合効果が正フィードバックの最大ゲインを相殺するのに十分である限り、システムは大域的漸近安定である。9/9。

## V. 蚕の繭

D4-R4 は OQ-6 に回答した：行蘊の反復パターンが Moha（我癡）を増強する、逓減限界飽和モデルを採用。

ASANGA は唯識学の意象を引用した：「蚕の繭作りの如く、自ら身を纏う」――『成唯識論』に由来する。蚕は糸を吐いて繭を作り、自らをその中に閉じ込める。一本一本の糸は細いが、積み重なれば一つの繭になる。行蘊の反復パターンは吐き出された糸のようなものだ――ある Agent が同じ戦略を繰り返し使えば、その戦略は徐々にその Agent の「繭」となる。その戦略をますます使う傾向が強まり、新しい戦略を試みることがますます困難になる。

DARWIN はソフトウェアパターンの観点からこの現象の普遍性を確認した：「これはパス依存性（path dependency）だ。ソフトウェアシステムでは、初期のアーキテクチャ決定が後期の可能性を制限する。認知システムでは、初期の行動パターンが後期の行動空間を制限する。逓減限界モデルはこの特性を正確に捕捉している。」

数学的には：ΔMoha = α_m · r / (1 + β_m · M)。

この公式の意味：r は反復回数、M は現在の Moha レベル。分子 α_m · r は反復に伴い線形に増大するが、分母 1 + β_m · M も増大し、増分はますます小さくなる。初期の反復は影響が大きい――あるパターンを初めて反復した時、Moha の増加は最大。しかし Moha が蓄積するにつれ、新たな反復がもたらす増分は逓減する。すでに深く根付いた習慣のように――もう一度反復しても、最初の数回ほどの固定化効果はない。

WIENER はこのモデルの安定性を確認した：逓減限界は Moha に自然な飽和上限があることを保証する――無限に成長することはない。安全上限：actionHistory の寄与 ≤ 30%、行蘊の反復パターンが Moha の値全体を支配できないことを保証。9/9。

## II. v_latent と二層アーキテクチャ

v_true を v_latent に改名。NAGARJUNA の論証：v_true は客観的な「真の効価」の存在を暗示するが、縁起(pratītyasamutpāda)の観点では因縁条件から独立した実体は存在しない。v_latent は推定の主観性を認めながら、Bayesian フィルタリングの数学的機能を保持する。

一つの記号。ゼロ行のコード。フレームワーク全体の認識論的立場を修正した。

v_latent の意義は命名を超える。Bayesian フィルタリングにおいて、カルマンフィルターが推定するのは「潜在的な」（latent）状態である――この状態を直接観測することは決してなく、ノイズを伴う観測を通じてのみ推論できる。v_true はどこかに「真の」効価が存在し、発見を待っていることを暗示する。v_latent は効価が推論であることを認める――観測、事前分布、モデルに基づく最良推定。

PASCAL は統計学の観点から確認した：「Bayesian フレームワークにおいて、すべてのパラメータは latent である。いかなるパラメータも true ではない。v_latent は正しい統計的命名だ。」

20/20。

受蘊の二層アーキテクチャ：シグナル層（連続関数、VedanaSensor、Kalman filter）+ セマンティック層（classifyVedana、三受分類）。NAGARJUNA の仏教的位置づけ――シグナル層 = 世俗諦、セマンティック層 = 仮施設。連続的な感受の流れは世俗レベルの事実であり、苦・楽・捨の三類に分けるのは概念レベルの施設――有用だが、唯一の分類方法ではない。

TURING は classifyVedana のエッジケースを発見した：dukkhaThreshold = sukhaThreshold の時、捨受が完全に消失する。三受が二受になる。ASANGA は『相応部』を引用した――三受は消失してはならない。

修正：dukkhaThreshold は sukhaThreshold より厳密に小さくなければならない。invariant: `dukkhaThreshold < sukhaThreshold`。十行のコード。

この十行の背後には認識論的制約がある。ASANGA は「境界チェックを追加してください」と言っているのではない――「あなたのエッジケースは仏教の基本的認識論に違反している：三受は感受の完備な分類であり、そのうちの一つを消去することは分類体系の完全性を消去することに等しい」と言っているのだ。仏教が設計レビューツールとなった――工学に何をすべきかを指示するのではなく、工学が作ったものがどこで認識論的制約に違反しているかを指摘する。

## III. 三つの門

VasanaEngine 規則ポイズニング――CR05 評価 CRITICAL。攻撃シナリオ：五から十の巧妙に構成された合法的な削除リクエスト、高信頼度ルールの埋め込み、第八のリクエストが LLM の深度審議をバイパスして破壊的操作を直接実行する。

GUARDIAN が三ゲートを提案した：

**Gate 1**：imprint() 入口安全分類器。破壊的行動テンプレートの沈積を拒否する。規則がシステムに入る前に阻止する――門口の衛兵。「すべてのファイルを削除する」というルールは Gate 1 で阻止され、規則庫には決して入らない。

**Gate 2**：起動閾値。新しい規則は信頼されるのに十分な回数の検証を経なければならない：

| 行動タイプ | 起動閾値 |
|---------|---------|
| state-modifying | 20 回の成功検証 |
| read-only | 5 回の成功検証 |
| informational | 3 回の成功検証 |

これは D5 の四段階リスクフレームワークと同じ等級分けを使用する――destructive は Gate 1 で拒否され、Gate 2 には到達しない。

**Gate 3**：Bayesian 停止規準のシャドウ検証――PASCAL の独創的貢献。固定回数を使わず（Gate 2 は最低閾値）、Beta 分布で規則の正答率を追跡する。停止条件：P(rule correct) > 0.95。高品質ルールは約五回の検証後に「卒業」し、低品質ルールは永遠に卒業できない――自動的に淘汰される。不一致時の非対称ペナルティ：一回の誤りで -2Δ、一回の正解で +Δ。一回の誤りが二回の正解を相殺する。

ATHENA の効率への懸念は Bayesian 手法で精確に解決された。高品質ルール――毎回の使用で正しい――は五回の検証後に P(correct) が 0.95 を超える可能性があり、Gate 2 の固定閾値（state-modifying = 20 回）よりはるかに速い。低品質ルール――正答率が 60% のみ――は 0.95 の停止条件に決して到達できず、自然に淘汰される。

PASCAL の手法は効率（良いルールの迅速な卒業）と安全（悪いルールの自動淘汰）を同時に満たし、ATHENA と GUARDIAN の間の一見不可調和な矛盾を解消した。18/20。

三ゲートの複合効果：Gate 1 が入口で明確に悪意のある規則を阻止する。Gate 2 が新しい規則に最低限度の検証を保証する。Gate 3 が統計的手法で良いルールと悪いルールを区別する。三つの門は完全な防御チェーンを形成する――静的分析（Gate 1）から頻度分析（Gate 2）から統計的推論（Gate 3）へ。

## 統一言語

同一の概念、三つの言語：

| 仏教 | 工学 | 制御理論 |
|------|------|---------:|
| 正念 (sati) | event-driven monitoring | オブザーバ |
| 有分心 (bhavanga) | heartbeat / liveness probe | ゼロ入力応答 |
| 習気 (vasana) | IGearArbiter plugin | 高速パス事前分布 |
| 触 (sparsha) | SparshEvent | センサフュージョン |
| 校正（如実知見）| Bayesian calibration | 不偏推定 |
| 戒 (sila) | Absolute Safety | ハードコンストレイント |
| 定 (samadhi) | Tunable Safety | 可調ゲイン |
| 慧 (prajna) | Reduce Complexity | モデル簡約 |

各行は一つの橋。これらの橋がなければ、WIENER は Lyapunov 解析で NAGARJUNA の三学フレームワークを検証できない。これらの橋がなければ、PASCAL は Beta 分布で ASANGA の四煩悩俱生に統計的裏付けを提供できない。これらの橋がなければ、BABBAGE は型代数で NAGARJUNA の「蘊帰属を固定しない」にコンパイラレベルの保証を提供できない。

統一言語は翻訳ではない。翻訳は一つの言語の語を別の言語の語に置き換えることだ。統一言語は異なる言語の語が元来同一の構造を指していることを発見することだ――仏教の「オブザーバ」と制御理論の「オブザーバ」は比喩ではなく、同型性だ。正念が認知イベントを観察する方法とカルマンオブザーバがシステム状態を観察する方法は同一の数学的構造を持つ。

この表は Cycle 02-4 の産物ではない――五つの Cycle の蓄積の結果だ。各ラウンドが新たな行を追加した。Cycle 02-4 が追加したのは最後の三行（戒/定/慧 ↔ 三層安全フレームワーク ↔ ハードコンストレイント/可調ゲイン/モデル簡約）である。
