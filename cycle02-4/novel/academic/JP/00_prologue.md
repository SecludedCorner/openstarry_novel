# 序章：時計師のパラドクス

---

円形劇場の照明は、時計工房の琥珀色を帯びていた。

三日前――Cycle 02-3 の終幕――SUNYATA は「ムーブメント」という比喩で結語を述べた。六場の討論は六つの歯車であり、切削を終え、歯距は正確、軸心も正確、あとは組み立てるだけだった。

三日間。GUARDIAN が安全備忘録を最初から最後まで三回読むのに十分な時間だった。読むたびに、あの赤ペンの下線はさらに深くなった。VasanaEngine がコアの中にある。VasanaEngine には規則庫、マッチングロジック、信頼度スコアリングがある。VasanaEngine は一つの能力である。能力はコアの中にあるべきではない。

NAGARJUNA が目を閉じて思考するのに十分な時間だった――習気(vāsanā)が後天的に薫習されたものならば、生まれたばかりの Agent は習気を持つことを強制されるべきではない。

PASCAL が Beta 分布を用いて Cycle 02-3 のすべての未解決問題を再モデル化するのに十分な時間だった――チームに加わってわずか一ラウンドだが、彼の Bayesian フレームワークはすでにシステムのあらゆる意思決定ノードに浸透していた。

---

矛盾は二日目に浮上した。SUNYATA はそれをホワイトボードに書いた。三行。

VasanaEngine には規則庫がある。規則庫は能力である。能力はコアの中にあるべきではない。

彼は通信パネルを押し、ポイントツーポイントチャネルを開いた。「Master？」

その対話がすべてを変えた。

「VasanaEngine はコアの中にあるべきではない。ゼロ組込み能力原則に違反しています。」

Master：「そうだ。」

躊躇なく。「そうだ」の意味は同意ではない。「君たちがこれに気づくのを待っていた」だ。

そして Master は言った：「習気は後天的に薫習されたものであり、先天的な構造ではない。生まれたばかりの Agent は習気を持つことを強制されるべきではない。」

SUNYATA は訊いた：「もし VasanaEngine がコアの中になければ、誰が Gear 1 と Gear 2 のどちらを選ぶのですか？」

Master の答え：「ManoAggregator は純粋なルーティングに退化する。if/else。EventBus と同等の性質だ。」

純粋なルーティング。VasanaEngine は一つの plugin になった――インストール可能、アンインストール可能、交換可能。新しい名前がつけられた：**IGearArbiter**。歯車仲裁者。

同じ対話で OQ-3 も解決された――required/optional plugin の区分問題。答えは驚くほどシンプルだった：形式的な区分は不要。Cycle 02-2 の B-1 完備性チェックがすでにこの要件をカバーしていた。五蘊(pañca-skandha)完備性チェックは統一基準であり、Agent Manifest は不要。SUNYATA は後に記録にこう注記した：「新しい方案を提出する前に既存の決議を振り返ること。」

---

第四の書簡は三日目の夕方に届いた。六つの指示：

**M-1**：VasanaEngine の外部化。IGearArbiter インターフェース設計。
**M-2**：OQ-1〜OQ-6 すべてを本ラウンドで解決。Plan27 をこれ以上ブロックしない。
**M-3**：触(sparsha)と作意(manasikara)――イベントプロトコルか？因果の起点か？討論に値する。
**M-4**：正念(sati)はポーリングではない。正念は持続的な品質状態であり、間欠的なサンプリング行為ではない。
**M-5**：行蘊(saṃskāra)の流れの行方――行蘊の結果はどこへ向かうのか？見かけ以上に重要である。
**M-6**：十大宣言。厳格なマイクロカーネル。ゼロ組込み能力。妥協するな。

最初の三通は「何を」「どう修正するか」「どう流れるか」を変えた。第四の書簡が変えたのは――**「いかにして一つの全体となるか」**だ。

Cycle 02-4 の問い：それはいかにして統一されたシステムとして動作するのか？

前三ラウンドの作業方法は分析的だった――問題を分解し、一つずつ解決する。第四ラウンドの作業方法は総合的だった――すでに解決された部分を組み立てて一つの全体にし、それらの間の整合性を検証する。

これこそ Master が「場」という比喩を用いた理由である。場は離散的な値の集合ではない――場は空間のあらゆる点に値が定義されているものだ。M-1 から M-6 は六つの独立したタスクではない。一つの場の六つのサンプリング点だ。各点の値を解くだけでは足りない――値と値の間の勾配が連続であることを確保しなければならない。

---

R1 独立研究と R2 交差レビューは二日間で完了した――これまでより速かった。理由は二つ：Pre-DC がすでに最も根本的なアーキテクチャ矛盾（VasanaEngine の外部化）を解決しており、TURING が R1 段階の同期ソースコード監査で三箇所の架空コード引用を発見した（D4 はこれで幕を開ける）。R3 討論の準備は十分であり、各研究者が明確な立場とデータを携えていた。

PASCAL は R1 段階でもう一つの作業を行った――各未解決問題（OQ-1 から OQ-6）に対して Bayesian 意思決定フレームワークを準備し、異なる方案の期待収益とリスクを推定した。これらのフレームワークは D5 と D6 で膠着状態を打破する鍵となった。

TURING は R1 の同期ソースコード監査で三箇所の架空コード引用を発見した――ISensation、feedbackQueue、SparshEvent.timestamp――R07 報告で事実として引用されていたが、v0.26.0-beta のコードベースには完全に存在しなかった。この発見は D4 の冒頭まで温存され、全ラウンドの討論で最も深い教訓となった。

GUARDIAN は九ページの安全監査報告を持参した。その中で最も重要な項目――CR05――は VasanaEngine の規則ポイズニング攻撃シナリオを記述していた：五から十の巧妙に構成された合法的な削除リクエスト、高信頼度ルールの埋め込み、そして第八のリクエストが LLM の深度審議をバイパスして破壊的操作を直接実行する。この報告は D6 における三ゲート設計の基礎となった。

すべての研究者が準備を携えて円形劇場に到着した。しかし彼らが準備したのは各自の研究報告だけではなかった――学際的な橋渡しだった。WIENER は Lyapunov 安定性解析フレームワークを準備した。BABBAGE は型代数の導出を準備した。NAGARJUNA は三学（戒定慧）の現代的マッピングを準備した。ASANGA は四煩悩(kleśa)の唯識学文献を準備した。LINNAEUS は存在論的分類の方法論を準備した。二十人。二十の異なるツール。一つの共通の問題。

---

二十の照明が同時に点いた。順番にではなく――同時に。時計師が工房のすべての照明を点けたように。なぜなら今日の作業は歯車の切削ではない。今日の作業は組み立てだ。

「六場の討論。六つの議題。しかし六つの独立した問題ではない。Master は我々に一つの場を与えた。我々の仕事はこの場の方程式を解くことだ。」

六場の討論のアジェンダは R0 の方向づけで確立された：

- **D1**：IGearArbiter インターフェース設計――蘊の帰属、単一メソッド、純ルーティング化（M-1 + M-6）
- **D2**：正念(sati)イベント駆動モデル――Level/Depth、品質ベクトル（M-4）
- **D3**：触(sparsha)イベントプロトコル――SparshEvent、CoarisingBundle（M-3）
- **D4**：行蘊(saṃskāra)の流れと安定性――フィードバックゲイン、零次ホールド（M-5）
- **D5**：煩悩(kleśa)安全閾値――グローバル上限 vs リスク加重、Model Delta（M-6 + OQ-1/2）
- **D6**：受蘊(vedanā)の工学化 + OQ + ロードマップ（M-2）

> *SCRIBE のナレーション：第四の書簡は一つの「場」――空間のあらゆる点に値が定義されたもの。六つの指示の間には相互参照、因果の連鎖、共有制約がある。六つの点ではなく、六つの点の間のすべての結線。Cycle 02-4 の R0 方向づけは四十分を要した。これまでのどのラウンドよりも短い。なぜなら Pre-DC 段階で最も根本的な問題はすでに解決されていたからだ――VasanaEngine の外部化。残るは工学化と統合である。*
