## 佛學的角色

五個 Cycle 的演進軌跡：命名來源 → 功能定義 → 結構約束 → 動態模型 → **設計審查工具**。

Cycle 02-4 的質變：佛學不再告訴工程應該做什麼。佛學驗證工程做的是否與佛學模型一致。

ASANGA 指出三受不可消失——不是「命令加邊界檢查」，而是「發現邊界案例違反認識論約束」。NAGARJUNA 確認雙層架構——不是「設計」，而是「認證」。信號層是世俗諦，語義層是假施設。世俗諦/勝義諦預構了三層安全框架——不是「發明」，而是「提供概念語言」。v_true → v_latent 的重命名——零行程式碼，修正了整個框架的認識論立場。

佛學成為了設計審查工具。不是告訴你怎麼建造。是告訴你建造的東西是否在認識論上自洽。

BABBAGE 的角色同樣值得回顧。他在 D1 和 D5 中兩次使用型別代數作為決定性工具——把哲學爭論轉化為型別系統的推導，讓選擇變成必然。LINNAEUS 在 D3 中用本體論區分了定義性和附隨性屬性。WIENER 在 D4 和 D5 中用 Lyapunov 分析和累積偏移分析提供了穩定性保證。每一位研究者的專業工具都在正確的時刻被部署到正確的問題上。

## 統一語言

同一個概念，三種語言：

| 佛學 | 工程 | 控制理論 |
|------|------|---------:|
| 正念 (sati) | event-driven monitoring | 觀測器 |
| 有分心 (bhavanga) | heartbeat / liveness probe | 零輸入響應 |
| 習氣 (vasana) | IGearArbiter plugin | 快速路徑先驗 |
| 觸 (sparsha) | SparshEvent | 感測器融合 |
| 校準 (如實知見) | Bayesian calibration | 無偏估計 |
| 戒 (sila) | Absolute Safety | 硬約束 |
| 定 (samadhi) | Tunable Safety | 可調增益 |
| 慧 (prajna) | Reduce Complexity | 模型簡化 |

每一行是一座橋。沒有這些橋，WIENER 無法用 Lyapunov 分析驗證 NAGARJUNA 的三學框架。沒有這些橋，PASCAL 無法用 Beta 分佈為 ASANGA 的四煩惱俱生提供統計支撐。沒有這些橋，BABBAGE 無法用型別代數為 NAGARJUNA 的「不鎖定蘊歸屬」提供編譯器級保證。

統一語言不是翻譯。翻譯是把一種語言的詞換成另一種語言的詞。統一語言是發現不同語言的詞原本就指向同一個結構——佛學的「觀測器」和控制理論的「觀測器」不是隱喻，是同構。正念觀察認知事件的方式和 Kalman 觀測器觀察系統狀態的方式具有相同的數學結構。

這張表不是 Cycle 02-4 的產物——它是五個 Cycle 累積的結果。每一輪增加新的行。Cycle 02-4 增加了最後三行（戒/定/慧 ↔ 三層安全框架 ↔ 硬約束/可調增益/模型簡化）。

## 還沒有結束

五項延後事項，每一項都有明確的原因：

- **FC-26**：LLM 與想蘊/識蘊的交織——需要更深入的架構分析
- **Level 動態切換**：PASCAL 在 D2 提出，被否決——需要完整的安全審計才能重新考慮
- **ChannelManasikara 元認知維度**：作意的「對作意的覺察」——概念上清楚，工程上需要更多設計
- **行蘊意門自觸**：行蘊結果觸發新觸——因果循環的處理需要謹慎
- **長期漸現行為**：相變、分岔——需要更長時間的模擬和分析

延後不是遺忘。每一項都被記錄在路線圖中，帶有明確的前置條件和解除條件。FC-26 需要 Plan28 的安全強化完成後才能啟動。Level 動態切換需要三層安全框架的完整實作和壓力測試。元認知需要遞歸結構的設計——這可能需要一個完整的 Cycle。

ARCHIMEDES 在 D6 結束時做了一個估計：Plan27a + Plan27b 的工程量約 725-1070 LOC。如果以 Cycle 02-3 的 Plan25/Plan26 的交付速度為基準，約需兩到三週。Plan28 和 Plan29+ 需要更多的設計工作——它們涉及安全和學習，不能靠快速迭代完成。Master 的核心價值在這裡起作用：「踏實優先於速度。」

機芯在計時。但還沒有走到第七天。

## 場方程

M-1 → D1（IGearArbiter）。M-2 → D6（OQ 全部回答）。M-3 → D3（SparshEvent）。M-4 → D2（事件驅動正念）。M-5 → D4（行蘊流向 + 穩定性）。M-6 → 場約束本身（VasanaEngine 外部化、ManoAggregator 純路由、IGearArbiter 介面）。

六個方程。一個自洽的解。八條依賴鏈。零矛盾。

這個「場方程」的解不是預先設計的。它是從六場辯論的辯證過程中湧現的。M-1 給出了 IGearArbiter 的需求，但 IGearArbiter 的蘊歸屬（D1）取決於型別系統的約束，而型別系統的約束（BABBAGE）反過來影響了 IKlesha 的繼承結構（D5）。M-5 給出了行蘊流向的問題，但答案（D4）依賴於 TURING 對虛構程式碼的修正，而這個修正改變了穩定性分析的前提，最終影響了 D6 的零階保持設計。

因果網絡。不是因果鏈。
