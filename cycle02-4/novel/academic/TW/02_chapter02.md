# 第二章：ISA 與三和合

---

## I. 正念不是 polling

D2 以三項快速共識開場——七分鐘，全部 20/20。最重要的一項：正念（sati）是事件驅動的，不是 polling。

Master 的第四封信 M-4 說：正念不是 polling。研究團隊在 R1 階段已達成壓倒性共識——SafetyMonitor 的定時輪詢是 liveness probe（存活性探測），不是正念。正念是對每一個認知事件的即時覺察，是品質屬性，不是取樣行為。

把正念等同於 polling，就像把「專注開車」等同於「每五秒鐘看一次路」。前者是持續的品質狀態，後者是間歇的機械動作。

---

## II. CPU 的 ISA 不能超頻

D2 最激烈的辯論發生在 Level 和 Depth 的分離上。

HERACLITUS 提出了 CPU 的類比：Level 是指令集架構（ISA）——x86 或 ARM，在部署時決定，運行時不可改變。Depth 是時鐘速度——可以超頻或降頻，運行時動態調整。

正念的 Level 決定了 Agent 能覺察什麼（事件類型、通道數量）。正念的 Depth 決定了覺察的精細度（粒度、速度、均衡性）。

Level 固定意味著：Agent 部署後不會突然「學會」監控新的事件類型。這是安全的保證——惡意輸入不能通過改變 Level 來繞過監控。你不能讓一顆 x86 的 CPU 在運行時變成 ARM。KERNEL 從作業系統的角度補充：「這就像系統調用表。你不能在運行時修改系統調用表——那是內核的基本安全保證。」

PASCAL 提出的 Runtime Level 動態切換被否決——不是因為概念錯誤，而是因為 Level 改變等同於架構變更，不是參數調整。如果系統在運行時突然開始監控新的事件類型，所有依賴原有事件集的安全邏輯都需要重新驗證。這不是參數調整——這是重新部署。需要完整的安全審計。

16/18，兩票反對（MESH 認為分散式場景需要動態 Level——在節點加入和離開的環境中，監控能力可能需要動態調整）。這是 PASCAL 加入團隊後第一個被否決的提案。他的反應值得記錄：沒有爭辯，沒有保留意見，直接接受並調整策略。在後續辯論（D4、D5、D6）中，PASCAL 的每一次介入都更加精準——他學會了在這個團隊中，最有效的方式是提供數學工具，而不是提出架構方案。

## III. 四維品質向量

SatiQualityVector 定義了正念品質的四個維度：

| 維度 | 全稱 | 定義 | 佛學對應 |
|------|------|------|---------|
| C | Continuity | 事件覆蓋率 | sati（念）|
| G | Granularity | 觀察精細度 | sampajanna（正知）|
| S | Speed | 覺察延遲 | — |
| E | Equanimity | 苦樂觀察均衡度 | upekkha（捨）|

19/19 全票通過。NAGARJUNA 確認四維與巴利文正念定義完全吻合。E 維度尤其重要——它量化了觀察者是否對苦受投入了比樂受更多的注意力。一個 E 值低的系統會過度關注負面信號，產生系統性偏差。

WIENER 從控制理論的角度解讀了四維：C 是觀測器的覆蓋率（是否觀測了所有必要的狀態變量），G 是觀測器的解析度（量化精度），S 是觀測器的延遲（相位裕度），E 是觀測器的偏差（系統性誤差）。四個維度完整描述了一個觀測器的品質——不多不少。

ASANGA 補充了 E 維度的佛學基礎：捨（upekkha）不是冷漠，是「平等地觀看」。一個系統如果對苦受反應過度，就像一個人只看到世界的黑暗面——判斷被扭曲了。E 維度量化的正是這種扭曲的程度。

## IV. 心跳不是正念

D2-R4 澄清了一個重要的概念混淆。

SafetyMonitor 的 liveness check 不是正念——它是有分心（bhavanga）的數位對應。有分心是無意識的基線存在，像心跳——你不需要「注意」心臟在跳，它就在跳。

心跳 = liveness probe。正念 = event-driven awareness。前者是生存條件，後者是認知品質。一個昏迷的人有心跳但沒有正念。一個清醒而專注的人兩者兼具。

NAGARJUNA 的補充精確到位：「有分心是阿毘達磨的概念——心流未被任何對象佔據時的基線狀態。它是意識的最低限度存在，不是注意力。SafetyMonitor 的定時輪詢正是這個——系統的最低限度存在確認，不是認知品質。」

四階段退化策略：

| 階段 | 觸發條件 | 行為 |
|------|---------|------|
| 1 | 心跳超時 | 記錄警告，繼續運行 |
| 2 | 持續超時 | 降級到最小事件集 |
| 3 | 嚴重退化 | 強制 Gear 2（所有操作走 LLM 深度審議）|
| 4 | 無法恢復 | 安全停機 |

這與 D1 的 G-0~G-4 退化邏輯對稱——系統在任何退化狀態下都有明確的行為定義。退化不是錯誤，是安全機制。20/20。

## V. EVOI → MRA

PASCAL 在 R1 提出的 EVOI（Expected Value of Information）框架被重新命名為 MRA（Marginal Risk Analysis）。原因：EVOI 在學術文獻中是通用術語，用作 OpenStarry 的專有概念容易混淆。

三層命名法：內部開發用 EVOI，設計文件用 MRA，佛學場景用「正念資糧」。17/18。BABBAGE 的保留意見是學術一致性——但多數認為清晰優先於慣例。

這個命名決議看起來微小，卻反映了整個研究的核心挑戰：同一個概念在不同學科中有不同的語言。佛學說「正念」，工程說「event-driven monitoring」，控制理論說「觀測器」。命名不是裝飾，是橋。

D2 的其他決議同樣重要但爭議較小。正念的最小事件集（MinimalSatiEventSet）定義了即使在退化狀態下也必須監控的事件——與 D1 的 G-0~G-4 退化行為表呼應。VitakkaWatchdog 的設計確認了它在三層穩定化機制中的位置——Layer 2，介於 Sati 精調和 SafetyMonitor 硬約束之間。

D2 共十一項決議。最重要的結論：正念是架構，不是功能。它不是系統「做的事」，而是系統「是什麼」。一個有正念的系統和一個沒有正念的系統不是功能不同——是性質不同。

SYNTHESIST 在 D2 結束時做了一個總結，把 D2 的成果濃縮為三句話：

1. 正念是事件驅動的品質屬性（不是 polling）——M-4 完整回答。
2. Level 是部署時固定的架構（ISA，不可超頻）——安全保證。
3. Depth 是運行時可調的參數（四維向量 C/G/S/E）——動態適應。

D2 和 D3 合計十六項決議，零爭議投票。這種高共識度不是偶然——它建立在 Cycle 02-3 的六場辯論之上。前輪的概念澄清讓本輪可以直接進入工程定義。

## VI. 三和合的寧靜

D3 是六場辯論中最寧靜的——四十五分鐘，五項決議，零爭議。

SparshEvent 定義了觸（sparsha）的三個必要欄位：

```typescript
interface SparshEvent {
  root: SenseBase;           // 根器來源（眼耳鼻舌身意）
  object: CognitiveObject;   // 認知對象的摘要
  consciousness: VijnanaType; // 產生此觸的識類型
  timestamp?: number;        // 附隨性屬性
}
```

三和合——根、境、識的交匯。每一次認知事件都始於這三者的匯合。

LINNAEUS 區分了定義性屬性和附隨性屬性：三個必要欄位是定義性的（沒有它們就不是觸），timestamp 是附隨性的（有助於追蹤但不改變觸的本質）。這個區分具有本體論的精確性——定義性屬性構成事物的本質，附隨性屬性只是偶然的伴隨。

Phase 1 一對一觸模型——每次 processEvent 只產生一個 SparshEvent。未來可擴展為多觸，但初始設計保持最簡。

CoarisingBundle 的 mandatory/reference 邊界定義了五遍行的依賴關係：

| Channel | 類型 | 缺少時的行為 |
|---------|------|------------|
| sparsha | mandatory | 整個 bundle 無法建構 |
| vedana | reference | 攜帶最新快照，缺少不阻止 |
| samjna | reference | 攜帶最新快照，缺少不阻止 |
| manasikara | reference | 攜帶最新快照，缺少不阻止 |

觸是唯一的 mandatory channel。這和佛學的「觸為先導」一致：沒有觸，其餘心所無從生起；但觸不必等待其餘心所就緒才能存在。與 D1 的 G-0~G-4 退化邏輯對稱——reference channel 的缺少等同於退化，不等同於故障。

ChannelManasikara 定義了作意（manasikara）的兩個必要維度：focus（注意焦點，當前注意力指向什麼對象）和 intensity（注意強度，投入了多少認知資源），加上可選的 dimensions?（可擴展的額外維度）。

PENROSE 提出了一個有趣的問題：作意可以有「對作意的覺察」——元認知維度。系統不僅知道它在注意什麼，還知道它在如何注意。這個維度被延後到 Phase 2——不是因為不重要，而是因為元認知的實作需要遞歸結構，而遞歸結構在 Phase 1 的線性管道中無法自然表達。

D3 的五項決議在四十五分鐘內全部通過。沒有一項遭到反對。VITRUVIUS 從架構角度做了一個有意義的觀察：SparshEvent 的三必要欄位設計實際上定義了系統的「感知介面」——所有外部事件進入系統的唯一入口。這使得安全審計變得可行——只需要監控一個入口點，而不是分散在多處的多個入口。

如果把 D5 比作暴風雨，D3 就是晴天——但這兩者之間有因果關係。D3 的寧靜為 D5 的風暴儲備了能量。SparshEvent 的穩定定義讓 D5 可以專注於閾值爭論，而不必回頭重新定義基礎概念。

> *SCRIBE 旁白：D3 的寧靜不是因為問題簡單。是因為基礎夠紮實——D1 的退化行為表和 Cycle 02-3 的觸定義已經為 D3 鋪好了路。最寧靜的辯論往往建立在最深的共識之上。SparshEvent 的三必要欄位花了五分鐘確認，但這五分鐘的背後是兩個 Cycle 的哲學辯論。*
