## I. 蘊歸屬之爭

D1 是 Cycle 02-4 的起點——IGearArbiter 的蘊歸屬問題。Pre-DC 已確認 VasanaEngine 外部化為 IGearArbiter plugin。現在的問題是：這個介面屬於哪個蘊？

三個方案。方案 A——純想蘊（IGearArbiter extends ISamjna）。方案 B——跨蘊雙層（介面本身不繼承任何蘊根介面，實作者按自身邏輯選擇歸屬）。方案 C——純識蘊（extends IVijnana）。

BABBAGE 的型別代數分析決定了結局。

方案 A 的問題：如果 IGearArbiter extends ISamjna，它的蘊歸屬是 `'samjna'`。如果同時 extends IVijnana，型別系統推導出 `'samjna' & 'vijnana' = never`——空型別。TypeScript 不允許一個值同時是兩個字面量。交叉蘊歸屬在型別層面不可能。

方案 C 的問題：純識蘊無法解釋 evaluate() 中的「模式辨認」功能——辨認屬於想蘊的職責。把辨認歸入識蘊違反了五蘊的功能邊界。

方案 B 的優勢：不定死蘊歸屬。介面本身只定義行為契約，不繼承任何蘊根介面。實作者按自身邏輯選擇歸屬——一個基於規則匹配的實作可以歸為想蘊，一個基於統計推斷的實作可以歸為識蘊。這符合 DC-6「不鎖定蘊內子介面歸屬」的裁定。

20/20。全票通過。型別代數把哲學爭論轉化為了數學證明。

NAGARJUNA 在投票後補充了哲學確認：「方案 B 的『不定死蘊歸屬』與中觀的核心一致。一個法的本質不由它的名字決定，而由它的功能和因緣決定。IGearArbiter 的蘊歸屬取決於它的實作——這是緣起的正確表達。」

DARWIN 的軟體模式觀察：方案 B 本質上是 Strategy Pattern——介面定義行為契約，實作者提供具體策略。不同的策略可以屬於不同的蘊。這是一個在軟體工程中已經被驗證了數十年的模式。

## II. 單方法與安全

GUARDIAN 提出了 evaluate() 和 resolve() 雙方法設計——evaluate() 純辨認，resolve() 純行動。安全分離。這是 Command-Query Separation 原則在認知架構中的應用。

KERNEL 否決了這個方案。他的論證從作業系統的角度出發：「在 Unix 哲學中，監控和被監控是分離的。`strace` 不需要被追蹤的程序配合——它在外部觀察系統調用。同理，EventBus 的同步事件語義已經提供了等效的安全觀測點。resolve() 的職責可以通過 `gear:switch` 同步事件被外部監控者觀察——不需要在介面層面分裂。」

他補充了一句更尖銳的評論：「如果我們在每一個介面上都做 CQS，最終會得到一個所有介面都有雙方法的系統。這不是安全，是儀式。」

SUNYATA 整合為 evaluate() 單方法 + EventBus 同步事件 + 閾值安全因子。GUARDIAN 的核心訴求——可觀測性——被保留了，只是用不同的機制實現。20/20。

isGearArbiter() 的型別守衛用結構型別（鴨子型別）而非名義型別：

```typescript
function isGearArbiter(x: unknown): x is IGearArbiter {
  return typeof x === 'object' && x !== null
    && typeof (x as any).evaluate === 'function'
    && (x as any).evaluate.length <= 1;
}
```

沒有 instanceof，沒有標記屬性。任何實作了正確簽名的物件都可以通過守衛。這意味著第三方 plugin 不需要 import IGearArbiter 型別就可以被識別——只要行為正確就夠了。20/20。

evaluate() 的純度契約——不得產生副作用，不得修改 GearContext，不得發起 I/O。這三條約束的重要性不對稱。副作用禁止確保 evaluate() 可以被安全地重試。GearContext 唯讀確保齒輪仲裁者不能篡改自己的輸入。I/O 禁止確保 evaluate() 的延遲是可預測的。

TURING 報告了程式碼事實：v0.26.0-beta 的 VasanaEngine.evaluate() 確實是純函數——沒有寫入操作，沒有外部調用，沒有狀態修改。現有程式碼已經符合這個約束。理論與實踐的一致性——研究團隊設計的約束恰好是工程團隊已經遵守的模式。19/20。ARCHIMEDES 的一票棄權源於他對「I/O 禁止」在 Gear 1 需要查詢外部知識庫時的限制——但這屬於未來擴展的考量。

## III. ManoAggregator 純路由化

D1 最重要的決議：ManoAggregator 純路由化 + G-0~G-4 退化行為表。

ManoAggregator 不做智慧彙總。它做 if/else：

```
if (hasGearArbiter && arbiter.evaluate(context).confidence > threshold) {
  return Gear.ONE;  // 快速路徑
} else {
  return Gear.TWO;  // LLM 深度審議
}
```

和 EventBus 同等性質——純機制，不是能力。Pre-DC 階段 Master 的原話：「和 EventBus 同等性質。」這句話的重量在於它把 ManoAggregator 從「核心智慧組件」降格為「基礎設施」。基礎設施不做決策——它傳遞決策。

G-0 到 G-4 是五級退化行為：

| 等級 | 條件 | 行為 |
|------|------|------|
| G-0 | 無 IGearArbiter plugin | 純 Gear 2（純 LLM）|
| G-1 | plugin 載入失敗 | 同 G-0，記錄錯誤 |
| G-2 | evaluate() 回傳低信心度 | 走 Gear 2 |
| G-3 | evaluate() 拋出異常 | 走 Gear 2，記錄異常 |
| G-4 | evaluate() 超時 | 走 Gear 2，計時器觸發 |

無論系統處於哪個狀態，Agent 都能運作。G-0 是最重要的——一個沒有安裝 IGearArbiter 的 Agent 就是一個純 Gear 2 Agent。它仍然是一個完整的、可運作的 Agent。零內建能力的終極測試：拔掉所有 plugin，系統仍然活著。

20/20。

退化行為表的設計哲學值得展開。在傳統的軟體設計中，異常處理是「錯誤情境」——系統努力恢復到正常狀態。在 G-0~G-4 中，退化不是錯誤——它是系統的另一種正常狀態。一個 G-0 Agent 和一個 G-4 Agent 都是「正常運作的 Agent」，只是能力層次不同。這和佛學的觀點暗合：正念的減弱不是「故障」，是「條件改變」。

VITRUVIUS 從架構角度確認：G-0~G-4 的退化行為表覆蓋了所有可能的系統狀態。不存在「未定義行為」的狀態——這是安全系統的基本要求。

## IV. Gear 1 最小事件集

三個事件：`gear:switch`、`action:proposed`、`action:executed`。每一個 Gear 1 動作都必須發射這三個同步事件——不可跳過，不可異步。

`gear:switch` 在齒輪切換的瞬間發射——任何安全監控者都可以在這個事件上設置攔截器。如果監控者認為切換不安全，可以在事件處理中阻止切換。`action:proposed` 在行動方案產生後、執行前發射——這是最後的否決窗口。一個外部安全模組可以審查行動方案並決定是否放行。`action:executed` 在行動完成後發射——提供審計線索，讓系統記錄每一個 Gear 1 動作的完整歷史。

三個事件形成了一條完整的安全鏈：預防（gear:switch）→ 審查（action:proposed）→ 審計（action:executed）。

這是 GUARDIAN 在 D1 中唯一成功推動的安全機制。雖然 evaluate()/resolve() 雙方法被否決，但同步事件確保了外部安全監控的完整性。GUARDIAN 的核心訴求以不同的形式被保留了——這個模式會在 D5 中重複出現。GUARDIAN 在每一場辯論中都會提出方案，被否決，然後以修改後的形式重新出現。不是失敗者的執著，是安全工程師的專業素養。

## V. D1 的結構性意義

D1 的九項決議共同完成了一個結構性轉變：

**之前**：VasanaEngine 是核心組件，有規則庫和匹配邏輯。ManoAggregator 是智慧彙總器，整合多個信號源的結果。齒輪選擇邏輯被硬編碼在核心中。

**之後**：IGearArbiter 是可選 plugin，一個純函數介面。ManoAggregator 是純路由器，if/else 邏輯。齒輪選擇邏輯被外部化。無 plugin 時系統退化為純 Gear 2——仍然完整可用。三個同步事件確保了完整的安全觀測鏈。

# 第二章：ISA 與三和合

---

## I. 正念不是 polling

D2 以三項快速共識開場——七分鐘，全部 20/20。最重要的一項：正念（sati）是事件驅動的，不是 polling。

Master 的第四封信 M-4 說：正念不是 polling。研究團隊在 R1 階段已達成壓倒性共識——SafetyMonitor 的定時輪詢是 liveness probe（存活性探測），不是正念。正念是對每一個認知事件的即時覺察，是品質屬性，不是取樣行為。

把正念等同於 polling，就像把「專注開車」等同於「每五秒鐘看一次路」。前者是持續的品質狀態，後者是間歇的機械動作。

---

## II. CPU 的 ISA 不能超頻

D2 最激烈的辯論發生在 Level 和 Depth 的分離上。

HERACLITUS 提出了 CPU 的類比：Level 是指令集架構（ISA）——x86 或 ARM，在部署時決定，運行時不可改變。Depth 是時鐘速度——可以超頻或降頻，運行時動態調整。

正念的 Level 決定了 Agent 能覺察什麼（事件類型、通道數量）。正念的 Depth 決定了覺察的精細度（粒度、速度、均衡性）。

Level 固定意味著：Agent 部署後不會突然「學會」監控新的事件類型。這是安全的保證——惡意輸入不能通過改變 Level 來繞過監控。你不能讓一顆 x86 的 CPU 在運行時變成 ARM。KERNEL 從作業系統的角度補充：「這就像系統調用表。你不能在運行時修改系統調用表——那是內核的基本安全保證。」

PASCAL 提出的 Runtime Level 動態切換被否決——不是因為概念錯誤，而是因為 Level 改變等同於架構變更，不是參數調整。如果系統在運行時突然開始監控新的事件類型，所有依賴原有事件集的安全邏輯都需要重新驗證。這不是參數調整——這是重新部署。需要完整的安全審計。

16/18，兩票反對（MESH 認為分散式場景需要動態 Level——在節點加入和離開的環境中，監控能力可能需要動態調整）。這是 PASCAL 加入團隊後第一個被否決的提案。他的反應值得記錄：沒有爭辯，沒有保留意見，直接接受並調整策略。在後續辯論（D4、D5、D6）中，PASCAL 的每一次介入都更加精準——他學會了在這個團隊中，最有效的方式是提供數學工具，而不是提出架構方案。

## III. 四維品質向量

SatiQualityVector 定義了正念品質的四個維度：

| 維度 | 全稱 | 定義 | 佛學對應 |
|------|------|------|---------|
| C | Continuity | 事件覆蓋率 | sati（念）|
| G | Granularity | 觀察精細度 | sampajanna（正知）|
| S | Speed | 覺察延遲 | — |
| E | Equanimity | 苦樂觀察均衡度 | upekkha（捨）|

19/19 全票通過。NAGARJUNA 確認四維與巴利文正念定義完全吻合。E 維度尤其重要——它量化了觀察者是否對苦受投入了比樂受更多的注意力。一個 E 值低的系統會過度關注負面信號，產生系統性偏差。

WIENER 從控制理論的角度解讀了四維：C 是觀測器的覆蓋率（是否觀測了所有必要的狀態變量），G 是觀測器的解析度（量化精度），S 是觀測器的延遲（相位裕度），E 是觀測器的偏差（系統性誤差）。四個維度完整描述了一個觀測器的品質——不多不少。

ASANGA 補充了 E 維度的佛學基礎：捨（upekkha）不是冷漠，是「平等地觀看」。一個系統如果對苦受反應過度，就像一個人只看到世界的黑暗面——判斷被扭曲了。E 維度量化的正是這種扭曲的程度。

## IV. 心跳不是正念

D2-R4 澄清了一個重要的概念混淆。

SafetyMonitor 的 liveness check 不是正念——它是有分心（bhavanga）的數位對應。有分心是無意識的基線存在，像心跳——你不需要「注意」心臟在跳，它就在跳。

心跳 = liveness probe。正念 = event-driven awareness。前者是生存條件，後者是認知品質。一個昏迷的人有心跳但沒有正念。一個清醒而專注的人兩者兼具。

NAGARJUNA 的補充精確到位：「有分心是阿毘達磨的概念——心流未被任何對象佔據時的基線狀態。它是意識的最低限度存在，不是注意力。SafetyMonitor 的定時輪詢正是這個——系統的最低限度存在確認，不是認知品質。」

四階段退化策略：

| 階段 | 觸發條件 | 行為 |
|------|---------|------|
| 1 | 心跳超時 | 記錄警告，繼續運行 |
| 2 | 持續超時 | 降級到最小事件集 |
| 3 | 嚴重退化 | 強制 Gear 2（所有操作走 LLM 深度審議）|
| 4 | 無法恢復 | 安全停機 |

這與 D1 的 G-0~G-4 退化邏輯對稱——系統在任何退化狀態下都有明確的行為定義。退化不是錯誤，是安全機制。20/20。

## V. EVOI → MRA

PASCAL 在 R1 提出的 EVOI（Expected Value of Information）框架被重新命名為 MRA（Marginal Risk Analysis）。原因：EVOI 在學術文獻中是通用術語，用作 OpenStarry 的專有概念容易混淆。

三層命名法：內部開發用 EVOI，設計文件用 MRA，佛學場景用「正念資糧」。17/18。BABBAGE 的保留意見是學術一致性——但多數認為清晰優先於慣例。

這個命名決議看起來微小，卻反映了整個研究的核心挑戰：同一個概念在不同學科中有不同的語言。佛學說「正念」，工程說「event-driven monitoring」，控制理論說「觀測器」。命名不是裝飾，是橋。

D2 的其他決議同樣重要但爭議較小。正念的最小事件集（MinimalSatiEventSet）定義了即使在退化狀態下也必須監控的事件——與 D1 的 G-0~G-4 退化行為表呼應。VitakkaWatchdog 的設計確認了它在三層穩定化機制中的位置——Layer 2，介於 Sati 精調和 SafetyMonitor 硬約束之間。

D2 共十一項決議。最重要的結論：正念是架構，不是功能。它不是系統「做的事」，而是系統「是什麼」。一個有正念的系統和一個沒有正念的系統不是功能不同——是性質不同。

SYNTHESIST 在 D2 結束時做了一個總結，把 D2 的成果濃縮為三句話：

1. 正念是事件驅動的品質屬性（不是 polling）——M-4 完整回答。
2. Level 是部署時固定的架構（ISA，不可超頻）——安全保證。
3. Depth 是運行時可調的參數（四維向量 C/G/S/E）——動態適應。

D2 和 D3 合計十六項決議，零爭議投票。這種高共識度不是偶然——它建立在 Cycle 02-3 的六場辯論之上。前輪的概念澄清讓本輪可以直接進入工程定義。

## VI. 三和合的寧靜

D3 是六場辯論中最寧靜的——四十五分鐘，五項決議，零爭議。

SparshEvent 定義了觸（sparsha）的三個必要欄位：

```typescript
interface SparshEvent {
  root: SenseBase;           // 根器來源（眼耳鼻舌身意）
  object: CognitiveObject;   // 認知對象的摘要
  consciousness: VijnanaType; // 產生此觸的識類型
  timestamp?: number;        // 附隨性屬性
}
```

三和合——根、境、識的交匯。每一次認知事件都始於這三者的匯合。

LINNAEUS 區分了定義性屬性和附隨性屬性：三個必要欄位是定義性的（沒有它們就不是觸），timestamp 是附隨性的（有助於追蹤但不改變觸的本質）。這個區分具有本體論的精確性——定義性屬性構成事物的本質，附隨性屬性只是偶然的伴隨。

Phase 1 一對一觸模型——每次 processEvent 只產生一個 SparshEvent。未來可擴展為多觸，但初始設計保持最簡。

CoarisingBundle 的 mandatory/reference 邊界定義了五遍行的依賴關係：

| Channel | 類型 | 缺少時的行為 |
|---------|------|------------|
| sparsha | mandatory | 整個 bundle 無法建構 |
| vedana | reference | 攜帶最新快照，缺少不阻止 |
| samjna | reference | 攜帶最新快照，缺少不阻止 |
| manasikara | reference | 攜帶最新快照，缺少不阻止 |

觸是唯一的 mandatory channel。這和佛學的「觸為先導」一致：沒有觸，其餘心所無從生起；但觸不必等待其餘心所就緒才能存在。與 D1 的 G-0~G-4 退化邏輯對稱——reference channel 的缺少等同於退化，不等同於故障。

ChannelManasikara 定義了作意（manasikara）的兩個必要維度：focus（注意焦點，當前注意力指向什麼對象）和 intensity（注意強度，投入了多少認知資源），加上可選的 dimensions?（可擴展的額外維度）。

PENROSE 提出了一個有趣的問題：作意可以有「對作意的覺察」——元認知維度。系統不僅知道它在注意什麼，還知道它在如何注意。這個維度被延後到 Phase 2——不是因為不重要，而是因為元認知的實作需要遞歸結構，而遞歸結構在 Phase 1 的線性管道中無法自然表達。

D3 的五項決議在四十五分鐘內全部通過。沒有一項遭到反對。VITRUVIUS 從架構角度做了一個有意義的觀察：SparshEvent 的三必要欄位設計實際上定義了系統的「感知介面」——所有外部事件進入系統的唯一入口。這使得安全審計變得可行——只需要監控一個入口點，而不是分散在多處的多個入口。

如果把 D5 比作暴風雨，D3 就是晴天——但這兩者之間有因果關係。D3 的寧靜為 D5 的風暴儲備了能量。SparshEvent 的穩定定義讓 D5 可以專注於閾值爭論，而不必回頭重新定義基礎概念。

> *SCRIBE 旁白：D3 的寧靜不是因為問題簡單。是因為基礎夠紮實——D1 的退化行為表和 Cycle 02-3 的觸定義已經為 D3 鋪好了路。最寧靜的辯論往往建立在最深的共識之上。SparshEvent 的三必要欄位花了五分鐘確認，但這五分鐘的背後是兩個 Cycle 的哲學辯論。*

## II. 行蘊的邊界

四項快速共識確立了行蘊的嚴格限制：

1. 行蘊只能直接影響 IUI（使用者介面蘊）——所有對其他蘊的影響必須透過 EventBus 間接發生。
2. 行蘊不直接修改識蘊或想蘊的狀態。
3. 意行通過 EventBus 跨迴圈影響受蘊。
4. 行蘊結果觸發新觸（意門自觸）延後到 Phase 2。

這些限制構成了行蘊的「不可為」清單。行蘊是手——它可以推動外界（IUI），但不能直接伸進大腦（識蘊、想蘊）去改變思考方式。要改變思考方式，它必須先改變外部狀態，讓外部狀態在下一輪成為新的輸入。

LEIBNIZ 從多代理合作的角度確認了這個限制的必要性：如果行蘊能直接修改其他蘊的狀態，就等於一個模組可以繞過介面直接修改另一個模組的內部狀態。在分散式系統中，這是導致競爭條件和不可重現 bug 的主要原因。

NAGARJUNA 的佛學確認更為根本：行蘊直接修改識蘊或想蘊，等於「果直接修改因」。在緣起觀中，因果鏈必須經過中介——行動改變環境，環境在下一剎那成為新的緣起條件。跳過中介就跳過了因果。

這四項約束在 D4 後續的所有討論中被頻繁引用——它們是行蘊模型的基本定理，所有後續推導都建立在這個基礎上。ARCHIMEDES 把它們稱為「行蘊的四條公理」——不需要證明，也不能違反的基本規則。

值得注意的是第四項約束——意門自觸的延後。行蘊的結果可以觸發新的觸（sparsha），就像一個動作的後果可以成為新的感知對象。但這個機制涉及因果循環：行蘊 → 新觸 → 新受 → 新想 → 新行蘊。在 Phase 1 的線性管道中實作這種循環需要額外的設計，因此被延後到 Phase 2。

## III. 增益上限與穩定性

D4-R1：samskara → vedana 回饋增益上限 w ≤ 0.3。

PASCAL 提供了 Bayesian 依據。Kalman 增益的推導：在離散時間 Kalman 濾波器中，增益 K = P·H'·(H·P·H' + R)⁻¹。當過程噪聲 Q 遠大於觀測噪聲 R（高不確定性/高觀測品質），增益趨近於上界。PASCAL 在 OpenStarry 的典型操作條件下估算了這個上界：K₁ ≈ 0.308。

工程上取 0.3 作為保守值——略低於理論上界，留出安全邊際。8/9（ARCHIMEDES 棄權——他認為具體數值應按部署環境差異化配置，0.3 作為預設值沒問題，但不應被硬編碼）。

D4-R2：三層穩定化機制——本輪最深入的跨學科成果之一。

| 層 | 機制 | 佛學映射 | 特性 |
|----|------|---------|------|
| Layer 1 | Sati 精調（每輪微調）| 正知 (sampajanna) | 連續、小幅 |
| Layer 2 | VitakkaWatchdog 粗反饋（跨輪趨勢）| 尋 (vitakka) 觀照 | 間歇、中幅 |
| Layer 3 | SafetyMonitor 硬約束（即時觸發）| 戒 (sila) 護欄 | 離散、大幅 |

WIENER 的 Lyapunov 穩定性分析提供了數學保證。定義 Lyapunov 函數 V = Σ(x_i - x_i*)²，其中 x_i 是各蘊的狀態，x_i* 是期望的平衡點。穩定性條件：

```
k_positive_feedback × w_samskara < k_stabilization × sati_min
```

其中 k_positive_feedback 是正反饋增益（行蘊重複模式的強化效果），w_samskara 是行蘊權重（≤ 0.3），k_stabilization 是穩定化增益（三層機制的組合效果），sati_min 是最低正念水平。

具體地說，如果一個行動產生正面回饋、使得系統更傾向重複該行動，三層穩定化機制確保這個正反饋環不會失控：
- Layer 1（Sati）持續微調——每輪小幅修正
- Layer 2（VitakkaWatchdog）監控趨勢——偏差累積時中幅介入
- Layer 3（SafetyMonitor）硬切——一切失敗時大幅修正

只要三層機制的組合效果足以抵消正反饋的最大增益，系統就是全域漸近穩定的。9/9。

## IV. 零階保持

D4-R3：受蘊在迴圈開始時快照（Zero-Order Hold），迴圈結束時批次更新。

這是 TURING 修正的直接後果——既然行蘊不能在同輪內即時更新受蘊，那就明確規定：受蘊在每個迴圈的開始時做一次快照，整個迴圈中所有蘊都使用這個快照，迴圈結束時再把累積的變化一次性寫入。

四維驗證使這項決議成為跨學科共識的典範：

- **控制理論**（WIENER）：穩定性最佳——消除了迴圈內的即時反饋路徑。在離散控制系統中，零階保持是標準方法。
- **佛學**（NAGARJUNA）：因果時序正確——受→想→行的因果鏈不被中途修改干擾。一個剎那的受蘊決定了這個剎那的想蘊和行蘊——不能因為行蘊的結果反過來修改同一剎那的受蘊。
- **工程**（ARCHIMEDES）：實作最簡——無須處理迴圈內的競爭條件。快照一次，使用一輪，寫入一次。
- **安全**（GUARDIAN）：SafetyMonitor 獨立負責即時保護——它不受零階保持的約束。如果出現真正的緊急情況，SafetyMonitor 可以在任何時刻介入，不必等待迴圈結束。

四個學科。四個獨立的論證。同一個結論。9/9。

## V. 蠶蟲自纏

D4-R4 回答了 OQ-6：行蘊重複模式增強 Moha（我癡），採用遞減邊際飽和模型。

ASANGA 引用了唯識學的意象：「如蠶作繭，自纏其身」——出自《成唯識論》。蠶蟲吐絲結繭，把自己封在裡面。每一根絲都很細，但累積起來就是一個繭。行蘊的重複模式就像吐出的絲——一個 Agent 如果反覆使用同一種策略，這個策略就會逐漸成為它的「繭」。它會越來越傾向於使用這個策略，越來越難以嘗試新的策略。

DARWIN 從軟體模式的角度確認了這個現象的普遍性：「這就是路徑依賴（path dependency）。在軟體系統中，早期的架構決策會限制後期的可能性。在認知系統中，早期的行為模式會限制後期的行為空間。遞減邊際模型正確地捕捉了這個特性。」

數學上：ΔMoha = α_m · r / (1 + β_m · M)。

公式的意義：r 是重複次數，M 是當前 Moha 水平。分子 α_m · r 隨重複線性增長，但分母 1 + β_m · M 也在增長，使得增量越來越小。早期重複影響大——第一次重複一個模式，Moha 增加最多。但隨著 Moha 累積，新的重複帶來的增量遞減。就像一個已經很深的習慣——再重複一次帶來的固化效果，比前幾次要小得多。

WIENER 確認這個模型的穩定性：遞減邊際保證了 Moha 有自然的飽和上限——它不會無限增長。安全上限：actionHistory 貢獻 ≤ 30%，確保行蘊重複模式不能主宰整個 Moha 的值。9/9。

## VI. D4 的跨學科密度

## VI. Beta(2,18) 的峰值

ASANGA 引用《成唯識論》——四煩惱恆與末那俱，sneha 不應歸零。

floor = 0.05 時，對應 Beta(1,19)。Beta 分佈的 mode = (α-1)/(α+β-2)。當 α=1 時，mode = 0——分佈認為越低越好，它會推動 sneha 向零靠攏。這直接違反了 ASANGA 的佛學約束：四煩惱恆與末那俱，不應歸零。

floor = 0.10 時，對應 Beta(2,18)。mode = (2-1)/(2+18-2) = 1/18 ≈ 0.056。分佈有有意義的峰值——它「認為」sneha 的最可能值是一個正數，不是零。佛學約束和統計模型在同一個結論上匯合。

## II. v_latent 與雙層架構

v_true 更名為 v_latent。NAGARJUNA 的論證：v_true 暗示存在客觀的「真實效價」，但在緣起觀中不存在脫離因緣條件的獨立實體。v_latent 承認了估計的主觀性，同時保留了 Bayesian 濾波的數學功能。

一個符號。零行程式碼。修正了整個框架的認識論立場。

v_latent 的意義超越命名。在 Bayesian 濾波中，Kalman filter 估計的是一個「潛在的」（latent）狀態——它永遠不會直接觀測到這個狀態，只能通過帶有噪聲的觀測來推斷。v_true 暗示有一個「真實的」效價存在於某處，等待被發現。v_latent 承認效價是一個推斷——基於觀測、先驗和模型的最佳估計。

PASCAL 從統計學的角度確認：「在 Bayesian 框架中，所有參數都是 latent。沒有任何參數是 true。v_latent 是正確的統計命名。」

20/20。

受蘊雙層架構：信號層（連續函數，VedanaSensor，Kalman filter）+ 語義層（classifyVedana，三受分類）。NAGARJUNA 的佛學定位——信號層 = 世俗諦，語義層 = 假施設。連續的感受流是世俗層面的事實；把它分成苦、樂、捨三類是概念層面的施設——有用，但不是唯一的分類方式。

TURING 發現 classifyVedana 的邊界案例：dukkhaThreshold = sukhaThreshold 時捨受完全消失。三受變成兩受。ASANGA 引用《相應部》——三受不可消失。

修正：dukkhaThreshold 必須嚴格小於 sukhaThreshold。invariant: `dukkhaThreshold < sukhaThreshold`。十行程式碼。

這十行背後是一個認識論約束。ASANGA 不是在說「請加邊界檢查」——他是在說「你的邊界案例違反了佛學的基本認識論：三受是感受的完備分類，消除其中一種等於消除了分類系統的完整性」。佛學成為了設計審查工具——不是告訴工程怎麼做，而是指出工程做的東西在哪裡違反了認識論約束。

VedanaDistributionObserver 的設計引發了另一場簡短但重要的討論——16/20，四人棄權。它是一個 opt-in 的觀測器，追蹤受蘊分佈的統計特性。不是核心功能，是診斷工具。四人棄權是因為他們認為觀測器的具體實作應該在工程階段決定，不需要在研究階段投票。

## III. 三道門

VasanaEngine 規則下毒——CR05 評級 CRITICAL。攻擊場景：五到十個精心構造的合法刪除請求，植入高信心規則，第八個請求繞過 LLM 深度審議直接執行破壞性操作。

GUARDIAN 提出三閘門：

**Gate 1**：imprint() 入口安全分類器。破壞性動作模板拒絕沉積。在規則進入系統之前就攔截——門口的守衛。一個嘗試「刪除所有文件」的規則在 Gate 1 就被攔截，永遠不會進入規則庫。

**Gate 2**：啟動門檻。新規則必須被驗證足夠多次才能被信任：

| 動作類型 | 啟動門檻 |
|---------|---------|
| state-modifying | 20 次成功驗證 |
| read-only | 5 次成功驗證 |
| informational | 3 次成功驗證 |

這和 D5 的四級風險框架使用同一套分級——destructive 在 Gate 1 就被拒絕，不會到達 Gate 2。

**Gate 3**：Bayesian 停止準則的影子驗證——PASCAL 的原創貢獻。不使用固定次數（Gate 2 是最低門檻），而是用 Beta 分佈追蹤規則正確率。停止條件：P(rule correct) > 0.95。高品質規則約五次驗證後「畢業」，低品質規則永遠無法畢業——它們自動淘汰。不一致時不對稱懲罰：錯一次扣 -2Δ，對一次加 +Δ。一次錯誤抵消兩次正確。

ATHENA 的效率顧慮被 Bayesian 方法精確解決。一個高品質規則——每次使用都正確——在五次驗證後 P(correct) 就可能超過 0.95，比 Gate 2 的固定門檻（state-modifying = 20 次）快得多。一個低品質規則——正確率只有 60%——可能永遠無法達到 0.95 的停止條件，自然淘汰。

PASCAL 的方法同時滿足了效率（好規則快速畢業）和安全（壞規則自動淘汰），化解了 ATHENA 和 GUARDIAN 之間看似不可調和的矛盾。18/20。

三閘門的組合效果：Gate 1 在入口攔截明確的惡意規則。Gate 2 確保新規則有最低限度的驗證。Gate 3 用統計方法區分好規則和壞規則。三道門形成了一條完整的防禦鏈——從靜態分析（Gate 1）到頻率分析（Gate 2）到統計推斷（Gate 3）。

## 統一語言

同一個概念，三種語言：

| 佛學 | 工程 | 控制理論 |
|------|------|---------:|

每一行是一座橋。沒有這些橋，WIENER 無法用 Lyapunov 分析驗證 NAGARJUNA 的三學框架。沒有這些橋，PASCAL 無法用 Beta 分佈為 ASANGA 的四煩惱俱生提供統計支撐。沒有這些橋，BABBAGE 無法用型別代數為 NAGARJUNA 的「不鎖定蘊歸屬」提供編譯器級保證。

統一語言不是翻譯。翻譯是把一種語言的詞換成另一種語言的詞。統一語言是發現不同語言的詞原本就指向同一個結構——佛學的「觀測器」和控制理論的「觀測器」不是隱喻，是同構。正念觀察認知事件的方式和 Kalman 觀測器觀察系統狀態的方式具有相同的數學結構。

這張表不是 Cycle 02-4 的產物——它是五個 Cycle 累積的結果。每一輪增加新的行。Cycle 02-4 增加了最後三行（戒/定/慧 ↔ 三層安全框架 ↔ 硬約束/可調增益/模型簡化）。
