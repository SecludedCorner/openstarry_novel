# 第二章：ISA 与三和合

---

## I. 正念不是 polling

D2 以三项快速共识开场——七分钟，全部 20/20。最重要的一项：正念（sati）是事件驱动的，不是 polling。

Master 的第四封信 M-4 说：正念不是 polling。研究团队在 R1 阶段已达成压倒性共识——SafetyMonitor 的定时轮询是 liveness probe（存活性探测），不是正念。正念是对每一个认知事件的即时觉察，是品质属性，不是取样行为。

把正念等同于 polling，就像把"专注开车"等同于"每五秒钟看一次路"。前者是持续的品质状态，后者是间歇的机械动作。

---

## II. CPU 的 ISA 不能超频

D2 最激烈的辩论发生在 Level 和 Depth 的分离上。

HERACLITUS 提出了 CPU 的类比：Level 是指令集架构（ISA）——x86 或 ARM，在部署时决定，运行时不可改变。Depth 是时钟速度——可以超频或降频，运行时动态调整。

正念的 Level 决定了 Agent 能觉察什么（事件类型、通道数量）。正念的 Depth 决定了觉察的精细度（粒度、速度、均衡性）。

Level 固定意味着：Agent 部署后不会突然"学会"监控新的事件类型。这是安全的保证——恶意输入不能通过改变 Level 来绕过监控。你不能让一颗 x86 的 CPU 在运行时变成 ARM。KERNEL 从操作系统的角度补充："这就像系统调用表。你不能在运行时修改系统调用表——那是内核的基本安全保证。"

PASCAL 提出的 Runtime Level 动态切换被否决——不是因为概念错误，而是因为 Level 改变等同于架构变更，不是参数调整。如果系统在运行时突然开始监控新的事件类型，所有依赖原有事件集的安全逻辑都需要重新验证。这不是参数调整——这是重新部署。需要完整的安全审计。

16/18，两票反对（MESH 认为分散式场景需要动态 Level——在节点加入和离开的环境中，监控能力可能需要动态调整）。这是 PASCAL 加入团队后第一个被否决的提案。他的反应值得记录：没有争辩，没有保留意见，直接接受并调整策略。在后续辩论（D4、D5、D6）中，PASCAL 的每一次介入都更加精准——他学会了在这个团队中，最有效的方式是提供数学工具，而不是提出架构方案。

## III. 四维品质向量

SatiQualityVector 定义了正念品质的四个维度：

| 维度 | 全称 | 定义 | 佛学对应 |
|------|------|------|---------|
| C | Continuity | 事件覆盖率 | sati（念）|
| G | Granularity | 观察精细度 | sampajanna（正知）|
| S | Speed | 觉察延迟 | — |
| E | Equanimity | 苦乐观察均衡度 | upekkha（舍）|

19/19 全票通过。NAGARJUNA 确认四维与巴利文正念定义完全吻合。E 维度尤其重要——它量化了观察者是否对苦受投入了比乐受更多的注意力。一个 E 值低的系统会过度关注负面信号，产生系统性偏差。

WIENER 从控制理论的角度解读了四维：C 是观测器的覆盖率（是否观测了所有必要的状态变量），G 是观测器的解析度（量化精度），S 是观测器的延迟（相位裕度），E 是观测器的偏差（系统性误差）。四个维度完整描述了一个观测器的品质——不多不少。

ASANGA 补充了 E 维度的佛学基础：舍（upekkha）不是冷漠，是"平等地观看"。一个系统如果对苦受反应过度，就像一个人只看到世界的黑暗面——判断被扭曲了。E 维度量化的正是这种扭曲的程度。

## IV. 心跳不是正念

D2-R4 澄清了一个重要的概念混淆。

SafetyMonitor 的 liveness check 不是正念——它是有分心（bhavanga）的数字对应。有分心是无意识的基线存在，像心跳——你不需要"注意"心脏在跳，它就在跳。

心跳 = liveness probe。正念 = event-driven awareness。前者是生存条件，后者是认知品质。一个昏迷的人有心跳但没有正念。一个清醒而专注的人两者兼具。

NAGARJUNA 的补充精确到位："有分心是阿毗达磨的概念——心流未被任何对象占据时的基线状态。它是意识的最低限度存在，不是注意力。SafetyMonitor 的定时轮询正是这个——系统的最低限度存在确认，不是认知品质。"

四阶段退化策略：

| 阶段 | 触发条件 | 行为 |
|------|---------|------|
| 1 | 心跳超时 | 记录警告，继续运行 |
| 2 | 持续超时 | 降级到最小事件集 |
| 3 | 严重退化 | 强制 Gear 2（所有操作走 LLM 深度审议）|
| 4 | 无法恢复 | 安全停机 |

这与 D1 的 G-0~G-4 退化逻辑对称——系统在任何退化状态下都有明确的行为定义。退化不是错误，是安全机制。20/20。

## V. EVOI → MRA

PASCAL 在 R1 提出的 EVOI（Expected Value of Information）框架被重新命名为 MRA（Marginal Risk Analysis）。原因：EVOI 在学术文献中是通用术语，用作 OpenStarry 的专有概念容易混淆。

三层命名法：内部开发用 EVOI，设计文件用 MRA，佛学场景用"正念资粮"。17/18。BABBAGE 的保留意见是学术一致性——但多数认为清晰优先于惯例。

这个命名决议看起来微小，却反映了整个研究的核心挑战：同一个概念在不同学科中有不同的语言。佛学说"正念"，工程说"event-driven monitoring"，控制理论说"观测器"。命名不是装饰，是桥。

D2 的其他决议同样重要但争议较小。正念的最小事件集（MinimalSatiEventSet）定义了即使在退化状态下也必须监控的事件——与 D1 的 G-0~G-4 退化行为表呼应。VitakkaWatchdog 的设计确认了它在三层稳定化机制中的位置——Layer 2，介于 Sati 精调和 SafetyMonitor 硬约束之间。

D2 共十一项决议。最重要的结论：正念是架构，不是功能。它不是系统"做的事"，而是系统"是什么"。一个有正念的系统和一个没有正念的系统不是功能不同——是性质不同。

SYNTHESIST 在 D2 结束时做了一个总结，把 D2 的成果浓缩为三句话：

1. 正念是事件驱动的品质属性（不是 polling）——M-4 完整回答。
2. Level 是部署时固定的架构（ISA，不可超频）——安全保证。
3. Depth 是运行时可调的参数（四维向量 C/G/S/E）——动态适应。

D2 和 D3 合计十六项决议，零争议投票。这种高共识度不是偶然——它建立在 Cycle 02-3 的六场辩论之上。前轮的概念澄清让本轮可以直接进入工程定义。

## VI. 三和合的宁静

D3 是六场辩论中最宁静的——四十五分钟，五项决议，零争议。

SparshEvent 定义了触（sparsha）的三个必要字段：

```typescript
interface SparshEvent {
  root: SenseBase;           // 根器来源（眼耳鼻舌身意）
  object: CognitiveObject;   // 认知对象的摘要
  consciousness: VijnanaType; // 产生此触的识类型
  timestamp?: number;        // 附随性属性
}
```

三和合——根、境、识的交汇。每一次认知事件都始于这三者的汇合。

LINNAEUS 区分了定义性属性和附随性属性：三个必要字段是定义性的（没有它们就不是触），timestamp 是附随性的（有助于追踪但不改变触的本质）。这个区分具有本体论的精确性——定义性属性构成事物的本质，附随性属性只是偶然的伴随。

Phase 1 一对一触模型——每次 processEvent 只产生一个 SparshEvent。未来可扩展为多触，但初始设计保持最简。

CoarisingBundle 的 mandatory/reference 边界定义了五遍行的依赖关系：

| Channel | 类型 | 缺少时的行为 |
|---------|------|------------|
| sparsha | mandatory | 整个 bundle 无法建构 |
| vedana | reference | 携带最新快照，缺少不阻止 |
| samjna | reference | 携带最新快照，缺少不阻止 |
| manasikara | reference | 携带最新快照，缺少不阻止 |

触是唯一的 mandatory channel。这和佛学的"触为先导"一致：没有触，其余心所无从生起；但触不必等待其余心所就绪才能存在。与 D1 的 G-0~G-4 退化逻辑对称——reference channel 的缺少等同于退化，不等同于故障。

ChannelManasikara 定义了作意（manasikara）的两个必要维度：focus（注意焦点，当前注意力指向什么对象）和 intensity（注意强度，投入了多少认知资源），加上可选的 dimensions?（可扩展的额外维度）。

PENROSE 提出了一个有趣的问题：作意可以有"对作意的觉察"——元认知维度。系统不仅知道它在注意什么，还知道它在如何注意。这个维度被延后到 Phase 2——不是因为不重要，而是因为元认知的实现需要递归结构，而递归结构在 Phase 1 的线性管道中无法自然表达。

D3 的五项决议在四十五分钟内全部通过。没有一项遭到反对。VITRUVIUS 从架构角度做了一个有意义的观察：SparshEvent 的三必要字段设计实际上定义了系统的"感知接口"——所有外部事件进入系统的唯一入口。这使得安全审计变得可行——只需要监控一个入口点，而不是分散在多处的多个入口。

如果把 D5 比作暴风雨，D3 就是晴天——但这两者之间有因果关系。D3 的宁静为 D5 的风暴储备了能量。SparshEvent 的稳定定义让 D5 可以专注于阈值争论，而不必回头重新定义基础概念。

> *SCRIBE 旁白：D3 的宁静不是因为问题简单。是因为基础够扎实——D1 的退化行为表和 Cycle 02-3 的触定义已经为 D3 铺好了路。最宁静的辩论往往建立在最深的共识之上。SparshEvent 的三必要字段花了五分钟确认，但这五分钟的背后是两个 Cycle 的哲学辩论。*
