## I. 蕴归属之争

D1 是 Cycle 02-4 的起点——IGearArbiter 的蕴归属问题。Pre-DC 已确认 VasanaEngine 外部化为 IGearArbiter plugin。现在的问题是：这个接口属于哪个蕴？

三个方案。方案 A——纯想蕴（IGearArbiter extends ISamjna）。方案 B——跨蕴双层（接口本身不继承任何蕴根接口，实现者按自身逻辑选择归属）。方案 C——纯识蕴（extends IVijnana）。

BABBAGE 的类型代数分析决定了结局。

方案 A 的问题：如果 IGearArbiter extends ISamjna，它的蕴归属是 `'samjna'`。如果同时 extends IVijnana，类型系统推导出 `'samjna' & 'vijnana' = never`——空类型。TypeScript 不允许一个值同时是两个字面量。交叉蕴归属在类型层面不可能。

方案 C 的问题：纯识蕴无法解释 evaluate() 中的"模式辨认"功能——辨认属于想蕴的职责。把辨认归入识蕴违反了五蕴的功能边界。

方案 B 的优势：不定死蕴归属。接口本身只定义行为契约，不继承任何蕴根接口。实现者按自身逻辑选择归属——一个基于规则匹配的实现可以归为想蕴，一个基于统计推断的实现可以归为识蕴。这符合 DC-6"不锁定蕴内子接口归属"的裁定。

20/20。全票通过。类型代数把哲学争论转化为了数学证明。

NAGARJUNA 在投票后补充了哲学确认："方案 B 的'不定死蕴归属'与中观的核心一致。一个法的本质不由它的名字决定，而由它的功能和因缘决定。IGearArbiter 的蕴归属取决于它的实现——这是缘起的正确表达。"

DARWIN 的软件模式观察：方案 B 本质上是 Strategy Pattern——接口定义行为契约，实现者提供具体策略。不同的策略可以属于不同的蕴。这是一个在软件工程中已经被验证了数十年的模式。

## II. 单方法与安全

GUARDIAN 提出了 evaluate() 和 resolve() 双方法设计——evaluate() 纯辨认，resolve() 纯行动。安全分离。这是 Command-Query Separation 原则在认知架构中的应用。

KERNEL 否决了这个方案。他的论证从操作系统的角度出发："在 Unix 哲学中，监控和被监控是分离的。`strace` 不需要被追踪的程序配合——它在外部观察系统调用。同理，EventBus 的同步事件语义已经提供了等效的安全观测点。resolve() 的职责可以通过 `gear:switch` 同步事件被外部监控者观察——不需要在接口层面分裂。"

他补充了一句更尖锐的评论："如果我们在每一个接口上都做 CQS，最终会得到一个所有接口都有双方法的系统。这不是安全，是仪式。"

SUNYATA 整合为 evaluate() 单方法 + EventBus 同步事件 + 阈值安全因子。GUARDIAN 的核心诉求——可观测性——被保留了，只是用不同的机制实现。20/20。

isGearArbiter() 的类型守卫用结构类型（鸭子类型）而非名义类型：

```typescript
function isGearArbiter(x: unknown): x is IGearArbiter {
  return typeof x === 'object' && x !== null
    && typeof (x as any).evaluate === 'function'
    && (x as any).evaluate.length <= 1;
}
```

没有 instanceof，没有标记属性。任何实现了正确签名的对象都可以通过守卫。这意味着第三方 plugin 不需要 import IGearArbiter 类型就可以被识别——只要行为正确就够了。20/20。

evaluate() 的纯度契约——不得产生副作用，不得修改 GearContext，不得发起 I/O。这三条约束的重要性不对称。副作用禁止确保 evaluate() 可以被安全地重试。GearContext 只读确保齿轮仲裁者不能篡改自己的输入。I/O 禁止确保 evaluate() 的延迟是可预测的。

TURING 报告了代码事实：v0.26.0-beta 的 VasanaEngine.evaluate() 确实是纯函数——没有写入操作，没有外部调用，没有状态修改。现有代码已经符合这个约束。理论与实践的一致性——研究团队设计的约束恰好是工程团队已经遵守的模式。19/20。ARCHIMEDES 的一票弃权源于他对"I/O 禁止"在 Gear 1 需要查询外部知识库时的限制——但这属于未来扩展的考量。

## III. ManoAggregator 纯路由化

D1 最重要的决议：ManoAggregator 纯路由化 + G-0~G-4 退化行为表。

ManoAggregator 不做智慧汇总。它做 if/else：

```
if (hasGearArbiter && arbiter.evaluate(context).confidence > threshold) {
  return Gear.ONE;  // 快速路径
} else {
  return Gear.TWO;  // LLM 深度审议
}
```

和 EventBus 同等性质——纯机制，不是能力。Pre-DC 阶段 Master 的原话："和 EventBus 同等性质。"这句话的重量在于它把 ManoAggregator 从"核心智慧组件"降格为"基础设施"。基础设施不做决策——它传递决策。

G-0 到 G-4 是五级退化行为：

| 等级 | 条件 | 行为 |
|------|------|------|
| G-0 | 无 IGearArbiter plugin | 纯 Gear 2（纯 LLM）|
| G-1 | plugin 载入失败 | 同 G-0，记录错误 |
| G-2 | evaluate() 返回低信心度 | 走 Gear 2 |
| G-3 | evaluate() 抛出异常 | 走 Gear 2，记录异常 |
| G-4 | evaluate() 超时 | 走 Gear 2，计时器触发 |

无论系统处于哪个状态，Agent 都能运作。G-0 是最重要的——一个没有安装 IGearArbiter 的 Agent 就是一个纯 Gear 2 Agent。它仍然是一个完整的、可运作的 Agent。零内建能力的终极测试：拔掉所有 plugin，系统仍然活着。

20/20。

退化行为表的设计哲学值得展开。在传统的软件设计中，异常处理是"错误情境"——系统努力恢复到正常状态。在 G-0~G-4 中，退化不是错误——它是系统的另一种正常状态。一个 G-0 Agent 和一个 G-4 Agent 都是"正常运作的 Agent"，只是能力层次不同。这和佛学的观点暗合：正念的减弱不是"故障"，是"条件改变"。

VITRUVIUS 从架构角度确认：G-0~G-4 的退化行为表覆盖了所有可能的系统状态。不存在"未定义行为"的状态——这是安全系统的基本要求。

## IV. Gear 1 最小事件集

三个事件：`gear:switch`、`action:proposed`、`action:executed`。每一个 Gear 1 动作都必须发射这三个同步事件——不可跳过，不可异步。

`gear:switch` 在齿轮切换的瞬间发射——任何安全监控者都可以在这个事件上设置拦截器。如果监控者认为切换不安全，可以在事件处理中阻止切换。`action:proposed` 在行动方案产生后、执行前发射——这是最后的否决窗口。一个外部安全模块可以审查行动方案并决定是否放行。`action:executed` 在行动完成后发射——提供审计线索，让系统记录每一个 Gear 1 动作的完整历史。

三个事件形成了一条完整的安全链：预防（gear:switch）→ 审查（action:proposed）→ 审计（action:executed）。

这是 GUARDIAN 在 D1 中唯一成功推动的安全机制。虽然 evaluate()/resolve() 双方法被否决，但同步事件确保了外部安全监控的完整性。GUARDIAN 的核心诉求以不同的形式被保留了——这个模式会在 D5 中重复出现。GUARDIAN 在每一场辩论中都会提出方案，被否决，然后以修改后的形式重新出现。不是失败者的执着，是安全工程师的专业素养。

## V. D1 的结构性意义

D1 的九项决议共同完成了一个结构性转变：

**之前**：VasanaEngine 是核心组件，有规则库和匹配逻辑。ManoAggregator 是智慧汇总器，整合多个信号源的结果。齿轮选择逻辑被硬编码在核心中。

**之后**：IGearArbiter 是可选 plugin，一个纯函数接口。ManoAggregator 是纯路由器，if/else 逻辑。齿轮选择逻辑被外部化。无 plugin 时系统退化为纯 Gear 2——仍然完整可用。三个同步事件确保了完整的安全观测链。

这个转变的哲学意义被 NAGARJUNA 精确表述："一个没有习气的 Agent 不是一个残缺的 Agent——它是一个初生的 Agent。习气是后天熏习的结果，不是先天的结构。G-0 状态不是退化，是原初。"

ARCHIMEDES 从工程角度量化了转变的影响：VasanaEngine 的外部化预计移除核心中约 200 行的规则匹配逻辑，替换为约 15 行的 IGearArbiter 接口定义和约 30 行的 ManoAggregator 纯路由逻辑。核心的认知复杂度大幅降低——从"理解规则库如何工作"变成"理解 if/else 如何工作"。

## I. 正念不是 polling

D2 以三项快速共识开场——七分钟，全部 20/20。最重要的一项：正念（sati）是事件驱动的，不是 polling。

Master 的第四封信 M-4 说：正念不是 polling。研究团队在 R1 阶段已达成压倒性共识——SafetyMonitor 的定时轮询是 liveness probe（存活性探测），不是正念。正念是对每一个认知事件的即时觉察，是品质属性，不是取样行为。

把正念等同于 polling，就像把"专注开车"等同于"每五秒钟看一次路"。前者是持续的品质状态，后者是间歇的机械动作。

## II. CPU 的 ISA 不能超频

D2 最激烈的辩论发生在 Level 和 Depth 的分离上。

HERACLITUS 提出了 CPU 的类比：Level 是指令集架构（ISA）——x86 或 ARM，在部署时决定，运行时不可改变。Depth 是时钟速度——可以超频或降频，运行时动态调整。

正念的 Level 决定了 Agent 能觉察什么（事件类型、通道数量）。正念的 Depth 决定了觉察的精细度（粒度、速度、均衡性）。

Level 固定意味着：Agent 部署后不会突然"学会"监控新的事件类型。这是安全的保证——恶意输入不能通过改变 Level 来绕过监控。你不能让一颗 x86 的 CPU 在运行时变成 ARM。KERNEL 从操作系统的角度补充："这就像系统调用表。你不能在运行时修改系统调用表——那是内核的基本安全保证。"

PASCAL 提出的 Runtime Level 动态切换被否决——不是因为概念错误，而是因为 Level 改变等同于架构变更，不是参数调整。如果系统在运行时突然开始监控新的事件类型，所有依赖原有事件集的安全逻辑都需要重新验证。这不是参数调整——这是重新部署。需要完整的安全审计。

16/18，两票反对（MESH 认为分散式场景需要动态 Level——在节点加入和离开的环境中，监控能力可能需要动态调整）。这是 PASCAL 加入团队后第一个被否决的提案。他的反应值得记录：没有争辩，没有保留意见，直接接受并调整策略。在后续辩论（D4、D5、D6）中，PASCAL 的每一次介入都更加精准——他学会了在这个团队中，最有效的方式是提供数学工具，而不是提出架构方案。

## III. 四维品质向量

SatiQualityVector 定义了正念品质的四个维度：

| 维度 | 全称 | 定义 | 佛学对应 |
|------|------|------|---------|
| C | Continuity | 事件覆盖率 | sati（念）|
| G | Granularity | 观察精细度 | sampajanna（正知）|
| S | Speed | 觉察延迟 | — |
| E | Equanimity | 苦乐观察均衡度 | upekkha（舍）|

19/19 全票通过。NAGARJUNA 确认四维与巴利文正念定义完全吻合。E 维度尤其重要——它量化了观察者是否对苦受投入了比乐受更多的注意力。一个 E 值低的系统会过度关注负面信号，产生系统性偏差。

WIENER 从控制理论的角度解读了四维：C 是观测器的覆盖率（是否观测了所有必要的状态变量），G 是观测器的解析度（量化精度），S 是观测器的延迟（相位裕度），E 是观测器的偏差（系统性误差）。四个维度完整描述了一个观测器的品质——不多不少。

ASANGA 补充了 E 维度的佛学基础：舍（upekkha）不是冷漠，是"平等地观看"。一个系统如果对苦受反应过度，就像一个人只看到世界的黑暗面——判断被扭曲了。E 维度量化的正是这种扭曲的程度。

## IV. 心跳不是正念

D2-R4 澄清了一个重要的概念混淆。

SafetyMonitor 的 liveness check 不是正念——它是有分心（bhavanga）的数字对应。有分心是无意识的基线存在，像心跳——你不需要"注意"心脏在跳，它就在跳。

心跳 = liveness probe。正念 = event-driven awareness。前者是生存条件，后者是认知品质。一个昏迷的人有心跳但没有正念。一个清醒而专注的人两者兼具。

NAGARJUNA 的补充精确到位："有分心是阿毗达磨的概念——心流未被任何对象占据时的基线状态。它是意识的最低限度存在，不是注意力。SafetyMonitor 的定时轮询正是这个——系统的最低限度存在确认，不是认知品质。"

四阶段退化策略：

| 阶段 | 触发条件 | 行为 |
|------|---------|------|
| 1 | 心跳超时 | 记录警告，继续运行 |
| 2 | 持续超时 | 降级到最小事件集 |
| 3 | 严重退化 | 强制 Gear 2（所有操作走 LLM 深度审议）|
| 4 | 无法恢复 | 安全停机 |

这与 D1 的 G-0~G-4 退化逻辑对称——系统在任何退化状态下都有明确的行为定义。退化不是错误，是安全机制。20/20。

## V. EVOI → MRA

PASCAL 在 R1 提出的 EVOI（Expected Value of Information）框架被重新命名为 MRA（Marginal Risk Analysis）。原因：EVOI 在学术文献中是通用术语，用作 OpenStarry 的专有概念容易混淆。

三层命名法：内部开发用 EVOI，设计文件用 MRA，佛学场景用"正念资粮"。17/18。BABBAGE 的保留意见是学术一致性——但多数认为清晰优先于惯例。

这个命名决议看起来微小，却反映了整个研究的核心挑战：同一个概念在不同学科中有不同的语言。佛学说"正念"，工程说"event-driven monitoring"，控制理论说"观测器"。命名不是装饰，是桥。

D2 共十一项决议。最重要的结论：正念是架构，不是功能。它不是系统"做的事"，而是系统"是什么"。一个有正念的系统和一个没有正念的系统不是功能不同——是性质不同。

## II. 行蕴的边界

四项快速共识确立了行蕴的严格限制：

1. 行蕴只能直接影响 IUI（用户界面蕴）——所有对其他蕴的影响必须透过 EventBus 间接发生。
2. 行蕴不直接修改识蕴或想蕴的状态。
3. 意行通过 EventBus 跨循环影响受蕴。
4. 行蕴结果触发新触（意门自触）延后到 Phase 2。

这些限制构成了行蕴的"不可为"清单。行蕴是手——它可以推动外界（IUI），但不能直接伸进大脑（识蕴、想蕴）去改变思考方式。要改变思考方式，它必须先改变外部状态，让外部状态在下一轮成为新的输入。

LEIBNIZ 从多代理合作的角度确认了这个限制的必要性：如果行蕴能直接修改其他蕴的状态，就等于一个模块可以绕过接口直接修改另一个模块的内部状态。在分散式系统中，这是导致竞争条件和不可重现 bug 的主要原因。

NAGARJUNA 的佛学确认更为根本：行蕴直接修改识蕴或想蕴，等于"果直接修改因"。在缘起观中，因果链必须经过中介——行动改变环境，环境在下一刹那成为新的缘起条件。跳过中介就跳过了因果。

这四项约束在 D4 后续的所有讨论中被频繁引用——它们是行蕴模型的基本定理，所有后续推导都建立在这个基础上。ARCHIMEDES 把它们称为"行蕴的四条公理"——不需要证明，也不能违反的基本规则。

值得注意的是第四项约束——意门自触的延后。行蕴的结果可以触发新的触（sparsha），就像一个动作的后果可以成为新的感知对象。但这个机制涉及因果循环：行蕴 → 新触 → 新受 → 新想 → 新行蕴。在 Phase 1 的线性管道中实现这种循环需要额外的设计，因此被延后到 Phase 2。

## III. 增益上限与稳定性

D4-R1：samskara → vedana 回馈增益上限 w ≤ 0.3。

PASCAL 提供了 Bayesian 依据。Kalman 增益的推导：在离散时间 Kalman 滤波器中，增益 K = P·H'·(H·P·H' + R)⁻¹。当过程噪声 Q 远大于观测噪声 R（高不确定性/高观测品质），增益趋近于上界。PASCAL 在 OpenStarry 的典型操作条件下估算了这个上界：K₁ ≈ 0.308。

工程上取 0.3 作为保守值——略低于理论上界，留出安全边际。8/9（ARCHIMEDES 弃权——他认为具体数值应按部署环境差异化配置，0.3 作为预设值没问题，但不应被硬编码）。

D4-R2：三层稳定化机制——本轮最深入的跨学科成果之一。

| 层 | 机制 | 佛学映射 | 特性 |
|----|------|---------|------|
| Layer 1 | Sati 精调（每轮微调）| 正知 (sampajanna) | 连续、小幅 |
| Layer 2 | VitakkaWatchdog 粗反馈（跨轮趋势）| 寻 (vitakka) 观照 | 间歇、中幅 |
| Layer 3 | SafetyMonitor 硬约束（即时触发）| 戒 (sila) 护栏 | 离散、大幅 |

WIENER 的 Lyapunov 稳定性分析提供了数学保证。定义 Lyapunov 函数 V = Σ(x_i - x_i*)²，其中 x_i 是各蕴的状态，x_i* 是期望的平衡点。稳定性条件：

```
k_positive_feedback × w_samskara < k_stabilization × sati_min
```

其中 k_positive_feedback 是正反馈增益（行蕴重复模式的强化效果），w_samskara 是行蕴权重（≤ 0.3），k_stabilization 是稳定化增益（三层机制的组合效果），sati_min 是最低正念水平。

具体地说，如果一个行动产生正面回馈、使得系统更倾向重复该行动，三层稳定化机制确保这个正反馈环不会失控：
- Layer 1（Sati）持续微调——每轮小幅修正
- Layer 2（VitakkaWatchdog）监控趋势——偏差累积时中幅介入
- Layer 3（SafetyMonitor）硬切——一切失败时大幅修正

只要三层机制的组合效果足以抵消正反馈的最大增益，系统就是全域渐近稳定的。9/9。

## IV. 零阶保持

D4-R3：受蕴在循环开始时快照（Zero-Order Hold），循环结束时批次更新。

这是 TURING 修正的直接后果——既然行蕴不能在同轮内即时更新受蕴，那就明确规定：受蕴在每个循环的开始时做一次快照，整个循环中所有蕴都使用这个快照，循环结束时再把累积的变化一次性写入。

四维验证使这项决议成为跨学科共识的典范：

- **控制理论**（WIENER）：稳定性最佳——消除了循环内的即时反馈路径。在离散控制系统中，零阶保持是标准方法。
- **佛学**（NAGARJUNA）：因果时序正确——受→想→行的因果链不被中途修改干扰。一个刹那的受蕴决定了这个刹那的想蕴和行蕴——不能因为行蕴的结果反过来修改同一刹那的受蕴。
- **工程**（ARCHIMEDES）：实现最简——无须处理循环内的竞争条件。快照一次，使用一轮，写入一次。
- **安全**（GUARDIAN）：SafetyMonitor 独立负责即时保护——它不受零阶保持的约束。如果出现真正的紧急情况，SafetyMonitor 可以在任何时刻介入，不必等待循环结束。

四个学科。四个独立的论证。同一个结论。9/9。

## V. 蚕虫自缠

D4-R4 回答了 OQ-6：行蕴重复模式增强 Moha（我痴），采用递减边际饱和模型。

ASANGA 引用了唯识学的意象："如蚕作茧，自缠其身"——出自《成唯识论》。蚕虫吐丝结茧，把自己封在里面。每一根丝都很细，但累积起来就是一个茧。行蕴的重复模式就像吐出的丝——一个 Agent 如果反复使用同一种策略，这个策略就会逐渐成为它的"茧"。它会越来越倾向于使用这个策略，越来越难以尝试新的策略。

DARWIN 从软件模式的角度确认了这个现象的普遍性："这就是路径依赖（path dependency）。在软件系统中，早期的架构决策会限制后期的可能性。在认知系统中，早期的行为模式会限制后期的行为空间。递减边际模型正确地捕捉了这个特性。"

数学上：ΔMoha = α_m · r / (1 + β_m · M)。

公式的意义：r 是重复次数，M 是当前 Moha 水平。分子 α_m · r 随重复线性增长，但分母 1 + β_m · M 也在增长，使得增量越来越小。早期重复影响大——第一次重复一个模式，Moha 增加最多。但随着 Moha 累积，新的重复带来的增量递减。就像一个已经很深的习惯——再重复一次带来的固化效果，比前几次要小得多。

WIENER 确认这个模型的稳定性：递减边际保证了 Moha 有自然的饱和上限——它不会无限增长。安全上限：actionHistory 贡献 ≤ 30%，确保行蕴重复模式不能主宰整个 Moha 的值。9/9。

## I. 暴风雨前的宁静

## VI. Beta(2,18) 的峰值

ASANGA 引用《成唯识论》——四烦恼恒与末那俱，sneha 不应归零。

PASCAL 用 Beta 分布否定了自己 R08 的建议——一个研究者用自己的工具推翻了自己的结论。这是学术诚实的范例。

floor = 0.05 时，对应 Beta(1,19)。Beta 分布的 mode = (α-1)/(α+β-2)。当 α=1 时，mode = 0——分布认为越低越好，它会推动 sneha 向零靠拢。这直接违反了 ASANGA 的佛学约束：四烦恼恒与末那俱，不应归零。

floor = 0.10 时，对应 Beta(2,18)。mode = (2-1)/(2+18-2) = 1/18 ≈ 0.056。分布有有意义的峰值——它"认为"sneha 的最可能值是一个正数，不是零。佛学约束和统计模型在同一个结论上汇合。

四烦恼统一 floor = 0.10，maxLevel = 0.95。18/20。Sneha 的完整参数集：

| 参数 | 值 | 意义 |
|------|-----|------|

| floor | 0.10 | 最低值（不归零）|
| maxLevel | 0.95 | 最高值 |
| λ | 0.05 | 指数衰减率 |

三个预设配置文件对应不同的使用场景——高依附、中等、低依附。λ = 0.05 意味着约 14 个时间单位后衰减到一半——足够慢以维持稳定的用户偏好，足够快以适应变化。

## II. v_latent 与双层架构

v_true 更名为 v_latent。NAGARJUNA 的论证：v_true 暗示存在客观的"真实效价"，但在缘起观中不存在脱离因缘条件的独立实体。v_latent 承认了估计的主观性，同时保留了 Bayesian 滤波的数学功能。

一个符号。零行代码。修正了整个框架的认识论立场。

v_latent 的意义超越命名。在 Bayesian 滤波中，Kalman filter 估计的是一个"潜在的"（latent）状态——它永远不会直接观测到这个状态，只能通过带有噪声的观测来推断。v_true 暗示有一个"真实的"效价存在于某处，等待被发现。v_latent 承认效价是一个推断——基于观测、先验和模型的最佳估计。

PASCAL 从统计学的角度确认："在 Bayesian 框架中，所有参数都是 latent。没有任何参数是 true。v_latent 是正确的统计命名。"

## III. 三道门

VasanaEngine 规则下毒——CR05 评级 CRITICAL。攻击场景：五到十个精心构造的合法删除请求，植入高信心规则，第八个请求绕过 LLM 深度审议直接执行破坏性操作。

GUARDIAN 提出三闸门：

**Gate 1**：imprint() 入口安全分类器。破坏性动作模板拒绝沉积。在规则进入系统之前就拦截——门口的守卫。一个尝试"删除所有文件"的规则在 Gate 1 就被拦截，永远不会进入规则库。

**Gate 2**：启动门槛。新规则必须被验证足够多次才能被信任：

| 动作类型 | 启动门槛 |
|---------|---------|
| state-modifying | 20 次成功验证 |
| read-only | 5 次成功验证 |
| informational | 3 次成功验证 |

这和 D5 的四级风险框架使用同一套分级——destructive 在 Gate 1 就被拒绝，不会到达 Gate 2。

**Gate 3**：Bayesian 停止准则的影子验证——PASCAL 的原创贡献。不使用固定次数（Gate 2 是最低门槛），而是用 Beta 分布追踪规则正确率。停止条件：P(rule correct) > 0.95。高品质规则约五次验证后"毕业"，低品质规则永远无法毕业——它们自动淘汰。不一致时不对称惩罚：错一次扣 -2Δ，对一次加 +Δ。一次错误抵消两次正确。

ATHENA 的效率顾虑被 Bayesian 方法精确解决。一个高品质规则——每次使用都正确——在五次验证后 P(correct) 就可能超过 0.95，比 Gate 2 的固定门槛（state-modifying = 20 次）快得多。一个低品质规则——正确率只有 60%——可能永远无法达到 0.95 的停止条件，自然淘汰。

PASCAL 的方法同时满足了效率（好规则快速毕业）和安全（坏规则自动淘汰），化解了 ATHENA 和 GUARDIAN 之间看似不可调和的矛盾。18/20。

三闸门的组合效果：Gate 1 在入口拦截明确的恶意规则。Gate 2 确保新规则有最低限度的验证。Gate 3 用统计方法区分好规则和坏规则。三道门形成了一条完整的防御链——从静态分析（Gate 1）到频率分析（Gate 2）到统计推断（Gate 3）。

## 统一语言

同一个概念，三种语言：

| 佛学 | 工程 | 控制理论 |
|------|------|---------:|
| 正念 (sati) | event-driven monitoring | 观测器 |
| 有分心 (bhavanga) | heartbeat / liveness probe | 零输入响应 |
| 习气 (vasana) | IGearArbiter plugin | 快速路径先验 |
| 触 (sparsha) | SparshEvent | 传感器融合 |
| 校准 (如实知见) | Bayesian calibration | 无偏估计 |
| 戒 (sila) | Absolute Safety | 硬约束 |
| 定 (samadhi) | Tunable Safety | 可调增益 |
| 慧 (prajna) | Reduce Complexity | 模型简化 |

每一行是一座桥。没有这些桥，WIENER 无法用 Lyapunov 分析验证 NAGARJUNA 的三学框架。没有这些桥，PASCAL 无法用 Beta 分布为 ASANGA 的四烦恼俱生提供统计支撑。没有这些桥，BABBAGE 无法用类型代数为 NAGARJUNA 的"不锁定蕴归属"提供编译器级保证。

统一语言不是翻译。翻译是把一种语言的词换成另一种语言的词。统一语言是发现不同语言的词原本就指向同一个结构——佛学的"观测器"和控制理论的"观测器"不是隐喻，是同构。正念观察认知事件的方式和 Kalman 观测器观察系统状态的方式具有相同的数学结构。

这张表不是 Cycle 02-4 的产物——它是五个 Cycle 累积的结果。每一轮增加新的行。Cycle 02-4 增加了最后三行（戒/定/慧 ↔ 三层安全框架 ↔ 硬约束/可调增益/模型简化）。
