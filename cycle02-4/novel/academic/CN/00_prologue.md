# 序章：钟表匠的悖论

---

圆形剧场的灯光带着一种钟表工坊的琥珀色。

三天前——Cycle 02-3 的尾声——SUNYATA 用"机芯"这个意象做了结语。六场辩论是六个齿轮，被切削完毕，齿距正确，轴心正确，只待组装。

三天。足够让 GUARDIAN 把安全备忘录从头到尾读了三遍。每读一遍，那些红笔的底线就更深。VasanaEngine 在核心里面。VasanaEngine 有规则库、匹配逻辑、信心度评分。VasanaEngine 是一种能力。能力不应该在核心里面。

足够让 NAGARJUNA 闭上眼睛思考：如果习气是后天熏习的，一个刚出生的 Agent 不应该被迫拥有习气。

足够让 PASCAL 用 Beta 分布重新建模了每一项 Cycle 02-3 的开放问题——他加入团队才一轮，但他的 Bayesian 框架已经深入了系统的每一个决策节点。

---

矛盾在第二天浮现。SUNYATA 把它写在白板上。三行。

VasanaEngine 有规则库。规则库是能力。能力不应该在核心中。

他按下通讯面板，点对点频道。"Master？"

那场对话改变了一切。

"VasanaEngine 不该在核心中。它违反了零内建能力原则。"

Master："是的。"

没有犹豫。"是的"的意思不是同意。是"我一直在等你们发现这件事"。

然后 Master 说："习气是后天熏习的，不是先天结构。一个刚出生的 Agent 不应该被迫拥有习气。"

SUNYATA 问："如果 VasanaEngine 不在核心里，谁来决定走 Gear 1 还是 Gear 2？"

Master 的回答："ManoAggregator 退化为纯路由。if/else。和 EventBus 同等性质。"

纯路由。VasanaEngine 成为一个 plugin——可安装、可卸载、可替换。它有一个新名字：**IGearArbiter**。齿轮仲裁者。

同一场对话还解决了 OQ-3——required/optional plugin 的区分问题。答案出人意料的简单：不需要形式化区分。Cycle 02-2 的 B-1 完备性检查已经覆盖了这个需求。五蕴完备性检查是统一标准，不需要 Agent Manifest。SUNYATA 后来在记录中写了一行批注："提出新方案前先回顾既有决议。"

---

第四封信在第三天傍晚出现。六条指示：

**M-1**：VasanaEngine 外部化。IGearArbiter 接口设计。
**M-2**：OQ-1~OQ-6 全部在本轮解决。Plan27 不能再被阻塞。
**M-3**：触和作意——事件协定？因果起点？值得一场辩论。
**M-4**：正念不是 polling。正念是持续的品质状态，不是间歇的取样行为。
**M-5**：行蕴的流向——行蕴的结果去了哪里？比它看起来的更重要。
**M-6**：十大宣言。严格微核心。零内建能力。不要妥协。

前三封信改变了"什么""如何修正""如何流动"。第四封信改变的是——**"如何成为一个整体。"**

Cycle 02-4 的问题：它如何作为一个统一的系统运作？

前三轮的工作方式是分析性的——拆解问题，逐一解决。第四轮的工作方式是综合性的——把已经解决的部分组装成一个整体，检验它们之间是否一致。

这就是为什么 Master 用"场"这个比喻。场不是一组离散的值——场是在空间的每一个点上都定义了值的东西。M-1 到 M-6 不是六个独立的任务。它们是一个场的六个取样点。解决每一个点的值不够——必须确保值与值之间的梯度是连续的。

---

R1 独立研究和 R2 交叉审阅在两天内完成——比以往更快。原因有二：Pre-DC 已经解决了最根本的架构矛盾（VasanaEngine 外部化），TURING 在 R1 阶段的同步源代码审查发现了三处虚构的代码引用（D4 将以此开场）。R3 辩论的准备充分，每位研究者都带着明确的立场和数据。

PASCAL 在 R1 阶段做了一件额外的事——他为每一个开放问题（OQ-1 到 OQ-6）准备了 Bayesian 决策框架，估算了不同方案的期望收益和风险。这些框架成为了 D5 和 D6 中打破僵局的关键工具。

TURING 在 R1 的同步源代码审查中发现了三处虚构的代码引用——ISensation、feedbackQueue、SparshEvent.timestamp——在 R07 报告中被当作事实引用，但在 v0.26.0-beta 的代码基线中完全不存在。这个发现被保留到 D4 开场，成为了整轮辩论中最深刻的教训。

GUARDIAN 带来了一份九页的安全审查报告。其中最重要的一项——CR05——描述了 VasanaEngine 的规则下毒攻击场景：精心构造五到十个合法的删除请求，植入高信心规则，然后第八个请求绕过 LLM 深度审议直接执行破坏性操作。这份报告在 D6 中成为了三闸门设计的基础。

每一位研究者都带着准备来到圆形剧场。但他们准备的不仅是各自的研究报告——是跨学科的桥接。WIENER 准备了 Lyapunov 稳定性分析框架。BABBAGE 准备了类型代数推导。NAGARJUNA 准备了三学（戒定慧）的现代映射。ASANGA 准备了四烦恼的唯识学文献。LINNAEUS 准备了本体论分类的方法论。二十个人。二十把不同的工具。一个共同的问题。

---

二十盏灯同时亮起。不是依序——同时。像钟表匠打开了工坊的所有照明。因为今天的工作不是切削齿轮。今天的工作是组装。

"六场辩论。六个议题。但不是六个独立的问题。Master 给了我们一个场。我们的工作是解这个场方程。"

六场辩论的议程在 R0 定向中确立：

- **D1**：IGearArbiter 接口设计——蕴归属、单方法、纯路由化（M-1 + M-6）
- **D2**：正念（Sati）事件驱动模型——Level/Depth、品质向量（M-4）
- **D3**：触（Sparsha）事件协定——SparshEvent、CoarisingBundle（M-3）
- **D4**：行蕴（Samskara）流向与稳定性——回馈增益、零阶保持（M-5）
- **D5**：烦恼（Klesha）安全阈值——全域上限 vs 风险加权、Model Delta（M-6 + OQ-1/2）
- **D6**：受蕴（Vedana）工程化 + OQ + 路线图（M-2）

> *SCRIBE 旁白：第四封信是一个"场"——在每一个空间点上都定义了值的东西。六条指示之间有交叉引用、因果链、共享约束。不是六个点，而是六个点之间的所有连线。Cycle 02-4 的 R0 定向用了四十分钟。比以往任何一轮都短。因为在 Pre-DC 阶段，最根本的问题已经被解决——VasanaEngine 外部化。剩下的是工程化和整合。*
