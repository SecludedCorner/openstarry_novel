# VasanaEngine 架构——问题分析与结论

## 原始设计（修正前）

原始构想中，Agent 认知有两种固定速度：

- **Gear 1（快挡）**：VasanaEngine 规则引擎，~50ms，处理见过的模式
- **Gear 2（慢挡）**：VitakkaEngine 调用 LLM，数秒，处理复杂情况

识蕴根据匹配信心度决定走哪条路径。两个 Engine 都位于 ManoAggregator 核心内部。

以下分析基于此原始构想，最终结论为：两个 Engine 都应从核心中移出，齿轮数量不由架构决定，而由 plugin 组合决定。

---

## 三个结构性问题

### 问题一：快挡无法真正跳过理解

VasanaEngine 要将自然语言输入匹配到规则，这一步本身就是语义理解。用关键字匹配太脆弱（「几点了」vs「现在几点啊」vs「能告诉我时间吗」），用 embedding 匹配则已经引入了模型。所谓「不问 LLM、几十毫秒完成」的前提不成立——快挡和慢挡的边界是模糊的。

佛学上这反而正确：即便是习气的直觉反应，也是根、境、识三者和合（触）之后才发生的，不存在完全绕过识的捷径。

### 问题二：70-80% 快挡覆盖率在通用框架中站不住

用户找 AI Agent 处理的事情，多数是因为不简单才来的。能用规则穷举的模式在开放式自然语言场景下非常有限，实际比例可能是反过来的（20-30% 快挡、70-80% 慢挡）。如果快挡只覆盖少数情境，维护规则库的投入产出比就有问题。

此外，vasana（习气）在唯识学中是被经验熏习而不断改变的种子，不是写死的 if-then。如果 VasanaEngine 没有从慢挡结果自动沉淀新规则的学习机制，它就只是静态查询表，名不符实。

### 问题三：切换决策是二元的，但信心度是连续的

设计中提到「匹配信心度」，是一个 0 到 1 的连续值，但架构只有二元切换（匹配/不匹配），没有处理灰色地带。信心度 0.6 时走快挡可能执行错误动作，走慢挡又浪费规则引擎价值。更关键的是，阈值该由谁设定——查时间判断错了无所谓，删文件判断错了是灾难。不同场景、不同风险等级需要不同的阈值。

### 共同根源

三个问题指向同一件事：**把一个连续的、需要理解的、动态的认知过程，硬切成了一个离散的、不需理解的、静态的二元开关。**

---

## 第四个问题：零内建能力原则的矛盾

OpenStarry 核心目前的组件全部是纯机制——EventBus（路由）、ServiceRegistry（注册）、PluginLoader（加载）、ExecutionLoop（调度）——不含任何领域知识。

VasanaEngine 打破了这个模式：它有规则库（知识）、匹配逻辑（语义判断）、信心度评分（决策）。**它不是机制，它是能力。** 放进核心就违反了「零内建能力」原则。

---

## 解决方案：拆开机制与能力

### 核心原则

矛盾的根源在于把齿轮切换「机制」和 VasanaEngine「能力」混为一体。它们应该分离。

### 最终架构

```
ManoAggregator（核心 = 纯机制）
  │
  ├── IGearArbiter plugin chain（0 ~ N 个，可选）
  │     依序查询每个已注册的 IGearArbiter
  │     第一个信心度过阈值的就采用其结果
  │     全部没过 → fallback 到 IProvider
  │     没装任何一个 → 永远走 IProvider
  │
  ├── IProvider plugin 插槽（fallback）
  │     已有接口，LLM 能力来自 plugin
  │     没装 → Agent 无法推理（但核心不坏）
  │
  └── 阈值调节：Klesha（已有机制）
```

**核心只保留：** 依序问 IGearArbiter chain，有人过阈值就用，没有就调用 IProvider。这是 middleware chain 模式，跟 EventBus 的路由一样——纯机制，不含知识。齿轮数量完全取决于装了几个 IGearArbiter plugin。

### VitakkaEngine 的处理

VitakkaEngine 的「能力」不是自己的——它只是组装上下文然后调用 IProvider plugin 的管道，性质等同 EventBus 分发事件。但为彻底干净，它也不该作为核心的独立组件存在，只是核心调用已有 plugin 的一条路径。两个 Engine 都不该在核心里，核心里只有路由机制。

### 不同 plugin 组合的退化行为

齿轮数量不是架构决定的，而是用户根据场景装了什么 plugin 决定的：

| plugin 组合 | 行为 | 适用场景 |
|------------|------|---------|
| 多个 IGearArbiter + IProvider | 多挡瀑布式 fallback，完整运作 | 高精度场景 |
| 一个 IGearArbiter + IProvider | 经典双速运作 | 客服机器人等 |
| 仅 IProvider | 每件事都走 LLM，无快速路径 | 研究型 Agent |
| 仅 IGearArbiter | 只能处理规则匹配到的，无法推理 | 嵌入式 IoT |
| 皆无 | 核心存活但无法认知 | — |

## 佛学自洽性

这个拆法在佛学上更正确：

- 习气（vasana）在唯识学中是**后天熏习的惯性模式**，不是心识的本有能力
- 一个众生没有任何习气，依然有完整的认知能力（识），只是每件事都要重新认知
- 把 VasanaEngine 塞进核心 = 宣称「习气是意识的先天结构」——这在唯识学里不成立

因此：

- 无 VasanaEngine plugin 的 Agent = 初生众生，无习气，每件事都要深思
- 有 VasanaEngine plugin 的 Agent = 有经验的众生，常见情境可直觉反应
- VasanaEngine 规则随时间累积 = 习气的熏习过程

不同众生的习气深浅不同——有的几乎全靠直觉，有的事事深思。这不是意识结构的差异，是经验累积的差异。核心（识的基本结构）是一样的，差别在装了什么 plugin（累积了什么习气）。

---

## 额外建议

1. **让 VasanaEngine 能学习**：IProvider 处理过的模式自动沉淀为新的 IGearArbiter 规则，真正对应「熏习」概念
2. **引入风险感知的动态阈值**：切换决策不只看信心度，还要看操作的风险等级，高风险操作宁可走 IProvider
