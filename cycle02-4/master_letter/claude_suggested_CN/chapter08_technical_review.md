## 五项深入讨论及结论

### 第一项：烦恼与智慧同时影响阈值的合成规则

**初步问题**：受蕴的信号同时流向烦恼模块和智慧模块，两者可能对同一个阈值产生矛盾的调整，冲突解决机制是什么？

**讨论**：Master 指出「烦恼即菩提，一体两面」。烦恼与智慧不是两个独立实体在拔河，而是同一个 Klesha 的不同面向。差别在于运行时的状态——无明驱动时展现为烦恼（收敛、执着），觉察介入后展现为智慧（开放、放下）。因此不存在「两股力量需要合成」的问题。

Master 进一步指出，烦恼经过**淬炼、经验、学习、思考、良善**之后才转化为智慧。这个「经过」的过程在工程上是一个重要的未来研究方向——**Klesha 的经验性转化机制**，涉及记忆策略如何保留淬炼结果、Klesha 参数如何随经验动态调整、方向性由谁定义等问题。

此外，烦恼与智慧都属于上层概念，底层有大量具体的场景应用需要设计。

**结论**：不是第八章的缺口。「烦恼即菩提」的一体两面在宣言层不需要额外定义合成规则。Klesha 的经验性转化机制列为未来研究方向，具体行为留给场景级的 Plugin 设计文档。

---

### 第二项：三受到行为的映射缺少梯度信息

**初步问题**：第八章的工程对照只映射了三路分支（苦→修正、乐→强化、舍→维持），但 Debate 5 已确立连续测量模型。IVolition 需要的是离散标签还是包含连续值和趋势的完整数据？

**讨论**：Master 指出三受也是很上层的概念，底层的实现会连接到不同的传感器。回到 cycle02-2 的确认——「VedanaPlugin 只是基底，三受继承自它，然后有一些实体传感器会需要实现」。

实际数据流为：**传感器（底层）→ raw data → 受蕴子类转化为连续值 → 连续值附带三受标签 → 上层消费**。梯度、趋势、连续值是在传感器到受蕴子类这一段处理的，不是宣言该管的层级。宣言说「三受」就像说「温度有冷热适中」，精度和采样频率是传感器 Plugin 自己的规格。

**结论**：不是第八章的缺口。宣言不需要修改。梯度信息属于各传感器 Plugin 的实现细节，应在各 Plugin 的独立 MD 文档中描述。

---

### 第三项：循环的出口条件与因果生命周期的「寂灭」阶段脱节

**初步问题**：宣言说「每一次行动都重塑了触的世界，开启新的循环」，描述的是永续运转。但 Agent 需要有停止的能力。循环的终止条件在哪里？

**讨论**：确认两条路径都需要：

- **外部终止**：协调层作为独立 daemon，具备强制终止 agent 的权力（cycle02-2 已确认协调层有绝对权力介入删除出错的 agent）。
- **内部终止**：Agent Core 自身需要有主动停止的能力。

关于内部终止的判断逻辑，Master 明确指出：**要看任务是否完成，会看相关 Plugin 怎么设计。** 主动终止不是 Core 写死的逻辑，而是由 Plugin 的设计来决定。这完全吻合宣言 #2「一切皆 Plugin」的原则。

不同场景的 agent 终止逻辑完全不同——回答问题的 agent 可能用户说「谢谢」就结束；监控服务器的 agent 可能永远不会主动终止；跑批次任务的 agent 可能处理完最后一项就停。

**结论**：不是第八章的缺口。Core 需要提供终止的接口（cease 的 hook 或信号通道），具体的终止判断逻辑属于 Plugin 层的设计。此接口规格应写入 ExecutionLoop 的文档中。

---

### 第四项：安全监控在循环中的位置与受蕴「观察不干预」的张力

**初步问题**：GUARDIAN 投票时说「观察不干预确保受蕴不能绕过安全闸门」，但安全监控在宣言描述的因果链中的具体插入点（受蕴之后？IVolition 之前？行动执行之前？）没有定义。

**讨论**：Master 指出 Agent Core 很多时候是看 Plugin 怎么设计的。安全监控的位置由 Plugin 组合决定，不是 Core 写死的。

Master 在 cycle02-2 已经给出了多层安全的方向：
- 协调层设定法则约束（天条天规）
- init 时加载的安全 Plugin
- 运行期的觉察 Plugin

Core 的责任是在启动时做完备性检查——五蕴是否齐全、必要的安全 Plugin 是否存在——通过了才启动。启动的时候需要完备才可以启动，除非是在测试或特别的情况（对应 Q1 裁定的 dev mode）。

**结论**：不是第八章的缺口。安全监控的插入点是 Plugin 设计层面的事。Core 负责启动时的完备性检查，具体的拦截逻辑由各安全 Plugin 的 MD 文档定义。

---

### 第五项：受蕴「对内有状态、对外只读」的类型约束

**初步问题**：受蕴要做连续测量就需要保留历史值（趋势、移动平均），所以它对内有状态。但对外发布的数据应该是只读的。这种「对内可变、对外不可变」的双面约束，在 SDK 类型层面和十大宣言之间是否有冲突？

**讨论**：确认与宣言 #7（微内核与绝对纯净）和 #9（可插拔的记忆策略）存在潜在张力。

三个方向被提出：
1. **状态持久化作为 SDK 接口**：定义 `IStateful` 接口，Plugin 自行实现，Core 只通知不执行
2. **区分工作状态与记忆**：Plugin 内部的工作状态 vs agent 的长期记忆，走不同通道
3. **对外发布的类型用 Readonly wrapper**：SDK 类型设计层面的约束

Master 给出了关键方向：**每一个 agent 要使用某个 Plugin 的时候，都有自己的空间。** 这确立了架构：

- **状态空间的分配**由协调层负责。协调层在建立 agent 时，为该 agent 的每个 Plugin 分配独立的存储空间。符合宣言 #5「目录结构即权限」——agent 的运行位置决定权限边界，Plugin 的状态自然落在该边界内。
- **要不要用这个空间**由 Plugin 自己决定。有些 Plugin 需要保留历史（受蕴传感器），有些不需要（纯计算的工具 Plugin）。
- **Core 完全不介入**。Core 不知道此空间的存在，不负责读写，维持绝对纯净。

**宣言冲突解消**：
- **#7 未被违反**：Core 没有多任何能力，状态空间是协调层提供的基础设施，与 Core 无关。
- **#9 未产生冲突**：Plugin 的工作状态（受蕴历史值）和 agent 的记忆策略（Context Management）是两套系统。记忆策略管 LLM 对话的上下文，Plugin 状态管各 Plugin 自己的内部数据，各走各的通道。

**结论**：不违反十大宣言。协调层为每个 agent 的每个 Plugin 提供独立状态空间，Plugin 自行决定使用方式，Core 不介入。此规格应写入 Agent Coordination Layer 的文档中。

## 总结

第八章在宣言层面**没有技术缺口**。它的核心产出——宣言 #6 的文本、六场辩论的一致性验证（六比六零冲突）、八句话八个工程对应——在其负责的层级上是完备的。

初步审议提出的五个技术问题，经过与 Master's Letter 比对和逐项讨论后，全部归入下游工作：

| 问题 | 归属 |
|------|------|
| 烦恼与智慧的合成规则 | 场景级 Plugin 设计文档 + 未来研究方向（Klesha 经验性转化机制） |
| 三受的梯度信息 | 各传感器 Plugin 的独立 MD 文档 |
| 循环的出口条件 | ExecutionLoop 文档中定义 cease 接口，终止判断由 Plugin 设计 |
| 安全监控的插入点 | 各安全 Plugin 的 MD 文档，Core 负责启动时完备性检查 |
| 受蕴对内有状态对外只读 | Agent Coordination Layer 文档（状态空间分配）+ SDK 类型设计 |
