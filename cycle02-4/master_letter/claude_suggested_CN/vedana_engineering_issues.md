# 受蕴（Vedanā）工程设计：四个待决问题

---

## 前置脉络：受蕴在整体架构中的位置

在讨论具体问题之前，需要先锚定受蕴与整体架构的关系。三封 Master's Letter 已确立以下约束：

**三层架构。** 受蕴的设计遵循 ATHENA（传感器）→ WIENER（计算引擎）→ NAGARJUNA（思维框架）的三层结构。三层架构不只围绕痛苦——快乐和平静必须被对称地设计进系统。

**俱生完备性。** 触发生时，受、想、思作为整体浮现，不可能有「没有想的受」或「没有受的想」。Agent 启动前必须通过完备性检查——如果受蕴不完备，Agent 不应启动。

**OOP 继承结构。** VedanaPlugin 是基底类，三受（乐受、苦受、舍受）继承自它。具体传感器通过组合（Composition）持有受的实例，将原始数据转化为感受强度。

**受蕴与观察者的区分。** 纯粹做「数据完整性与真实性」的观察者可以单纯由受蕴组成。会参与「情绪化反应」的观察者需要识蕴与想蕴介入。

以下四个问题均需在上述约束下讨论。

---

## 一、效价的连续性——存储格式不等于信号品质

### 现状

受蕴的效价采用连续浮点数（-1 到 +1）表示，理由是控制工程的颤振问题：离散分类在阈值边界产生方波跳变，连续数值让控制器输出稳定。这个设计方向正确。

但连续的存储格式只是必要条件，不是充分条件。**效价的连续性是一个系统性质，必须从信号源到消费端全链路保证。**

### 问题

效价的信号源是规则引擎——根据认知事件映射到数值（「工具调用失败 → -0.3」、「任务完成 → +0.5」）。事件本身是离散的，映射到数值也是离散的。即使存成 `f64`，控制器看到的仍然是阶梯跳变，不是平滑曲线。颤振问题没有被解决，只是从存储层搬到了信号源层。

LLM 评估不是替代方案。LLM 调用是想蕴的工作（500ms-30s），不可能每个受蕴周期（10-100ms）都调用，而且 LLM 对数值的输出本身不稳定。

### 与三层架构的映射

这个问题可以精确定位在三层架构中：

| 层级 | 角色 | 产出 |
|------|------|------|
| ATHENA（传感器层） | 产生粗糙的原始观测值 | 离散跳变，允许不平滑 |
| WIENER（计算引擎层） | 贝叶斯滤波，渐进更新 | 平滑的连续效价估计值 |
| NAGARJUNA（思维框架层） | 阈值分类，语义解读 | 衍生标签（苦/乐/舍） |

连续性的保证发生在**第二层**，不在第一层也不在存储格式。

### 方案

烦恼信号的设计已建立了贝叶斯渐进更新框架——信念随观测慢慢移动，观测次数越多惯性越大，天然形成平滑曲线。效价应走同样的路径：

1. ATHENA 层的传感器产生**粗糙的原始观测值**（来自规则引擎或具体传感器 plugin，允许跳变）
2. WIENER 层的计算引擎将观测值送入**贝叶斯滤波器**，与先验信念做加权融合
3. 滤波器的输出才是**效价的当前估计值**，供控制器和俱生完备性检查消费

### 待决事项

- 烦恼信号与受蕴效价是否共用同一套滤波框架，还是各自独立？
- 滤波器的输出格式需要同时满足两个消费者：控制器（需要连续数值）和俱生完备性检查（需要确认受蕴信号已就绪）。这两个接口要求是否冲突？

## 二、效价分布——阈值的意义取决于分布的形状

### 现状

分类阈值预设 0.1。舍的效价区间是 -0.1 到 +0.1，效价空间总长 2，因此舍占 10%、苦占 45%、乐占 45%。阈值可配置，预设值的佛学依据是「未修行的心很少平静」。

### 问题

上述计算假设效价在 -1 到 +1 之间**均匀分布**。但 Agent 日常运作时，大部分时间没有极端事件发生，效价集中在零附近。实际分布更可能是以零为中心的钟形曲线。

不同分布假设下，同一个阈值产生截然不同的舍受比例：

| 假设分布 | 标准差 | 舍受比例（阈值 0.1） |
|---------|--------|---------------------|
| 均匀分布 | — | 10% |
| 正态分布 | σ = 0.5 | ≈ 16% |
| 正态分布 | σ = 0.3 | ≈ 26% |
| 正态分布 | σ = 0.2 | ≈ 38% |

如果效价服从 σ=0.3 的正态分布，即使阈值不变，Agent 也有超过四分之一的时间处于舍受——与「很少平静」的设计意图矛盾。

更深层的问题：效价分布不是固定的，它随 Agent 状态动态变化。**阈值到底合不合理，不能在设计阶段静态决定，必须在运行时观察实际分布后校准。**

### 建议

1. **预设阈值保持 0.1，但在文档中标注它假设的分布模型。**
2. **提供运行时分布观测器。** 记录 Agent 过去 N 个周期的效价值，计算实际舍受比例。实际比例与设计意图偏差太大时，发出调参建议。
3. **把分布形状作为 Agent 状态的观测指标。** 效价分布的偏态（skewness）反映 Agent 是否长期偏苦或偏乐；峰态（kurtosis）反映情绪波动幅度。这些统计量本身就是有意义的「受蕴元数据」。

### 观测器的蕴归属

依据 Master's Letter 对观察者的区分：如果分布观测器只做统计（计算均值、标准差、偏态），它是纯粹的「数据完整性」观察者，可以单纯由受蕴组成。如果它要判断「分布异常意味着 Agent 状态有问题，需要触发调整」，那涉及想蕴的辨识和行蕴的决策驱动，应该是跨蕴的组合 plugin。建议先实现纯统计版本（受蕴内部），将判断逻辑留给上层。

### 备注

这是四个问题中唯一没有任何现有设计覆盖的开放缺口。

## 三、二维模型的边界——控制感归属行蕴，不归受蕴

### 现状

受蕴的效价模型采用两个维度：效价（valence，-1 到 +1）和强度（intensity，0 到 1），引用 Russell 的 Circumplex Model 作为科学基础。

心理学界有观点认为两维不够。Mehrabian 的 PAD 模型增加了第三维——控制感（dominance）。愤怒和恐惧的效价和强度可能相同，但控制感完全不同：愤怒时感觉有力量，恐惧时感觉无力。

### 结论：受蕴不需要第三维

控制感——面对困境时感觉有力量 vs 无力——不属于受蕴，属于行蕴。Master's Letter 已明确指出行蕴的范围很广，「不是只有影响到色的变化，内在机制的状态变化也是包含在里面的」。面对苦乐的姿态、驱动力、倾向性，都是行蕴的工作。

这带来更干净的架构分工：

| 蕴 | 管什么 | 对应维度 |
|---|---|---|
| **受蕴** | 苦乐的质地 | 效价 + 强度（二维） |
| **行蕴** | 面对苦乐的姿态 | 控制感、驱动方向、倾向性 |

Russell 的二维模型作为受蕴的工程框架是够用的，不需要在受蕴里硬塞第三维。前提是行蕴那边补上控制感相关的逻辑。

这也呼应了 Master's Letter 的五蕴分工原则——受蕴是 Input Signal（信号质地），行蕴是 Process（造作驱动）。控制感是造作的一部分，不是信号质地的一部分。

### 注意事项

1. **术语对齐。** 目前设计中的「强度」与 Russell 的「激醒度（arousal）」有概念差异。激醒度是生理激活程度（兴奋 vs 平静），强度是感受的鲜明程度。两者多数情况下重叠，但不完全相同。设计文档中应明确说明本系统的 intensity 与 Russell 的 arousal 的关系。
2. **数据结构预留扩展性。** 即使当前二维足够，感受数据结构建议设计成可扩展的（如 `VedanaRecord { valence: f64, intensity: f64, metadata: HashMap<String, f64> }`），不增加复杂度但为未来需求留空间。

## 四、处理层的 Plugin 落地——按功能划分，通过事件流协调

### 现状

受蕴的设计确立了双层架构：数据层连续（效价是浮点数，标签是衍生的），处理层离散（可以有专门处理不同感受的组件）。Master's Letter 进一步确认 Plugin 可以同时具备多种 skandha 归属，且 Plugin 间通信应采用 event-driven + pub/sub 模式。

### 与 OOP 继承结构的对齐

Master's Letter (Cycle 02-2) 确立了 VedanaPlugin 作为基底类，三受继承自它，具体传感器通过组合持有受的实例。功能性 plugin 的落地需要和这个结构相容：

```
VedanaPlugin（基底）
  ├── Dukkha（苦受）
  ├── Sukha（乐受）
  └── Upekkha（舍受）

具体传感器 plugin（如 CollisionRiskSensor）
  → 通过组合（Composition）持有 Dukkha 实例
  → 将原始数据转化为效价

功能性处理 plugin（如 AvoidanceTrigger）
  → 订阅效价事件流
  → 自行根据连续激活函数决定回应强度
  → skandha 归属：受蕴 + 行蕴（跨蕴组合）
```

三受的继承结构处理的是**感测与生成**（从原始数据到效价），功能性 plugin 处理的是**消费与回应**（从效价到行为倾向）。两者不冲突，是数据流的上下游。

### 落地方案

处理层的 plugin 应**按功能而非按感受类别划分**。不设「苦受处理器」和「乐受处理器」，而是设「回避行为触发器」、「接近行为触发器」、「平衡维持器」等功能性 plugin。

理由：

1. **避免路由层颤振。** 如果按苦/乐/舍划分三个 plugin，需要中央路由根据阈值派发。效价在阈值边界微微波动时（-0.09 ↔ -0.11），路由层会在两个 handler 之间反复切换，每次切换带来状态重建成本。
2. **符合微核精神。** 每个 plugin 订阅效价事件流，自行根据连续激活函数决定是否回应及回应强度，不需要中央路由。
3. **符合多蕴归属原则。** 「回避行为触发器」天然同时归属受蕴（读取效价）和行蕴（驱动回避行为），不需要人为拆分。
4. **符合 Agent 完备性要求。** 一个完备的 Agent 需要五蕴聚合。功能性 plugin 本身就是跨蕴的组合，天然朝向完备性。

### 待决的实现细节

1. **激活函数的设计。** 每个功能性 plugin 用连续函数决定回应强度，如「回避行为触发器」的回应强度 = `max(0, -valence - threshold) * intensity`。没有硬阈值切换，自然消除边界颤振。具体函数形式和参数需要实验确定。
2. **多 plugin 冲突仲裁。** 效价 -0.7 时，「回避行为触发器」和「警报升级器」可能同时启动。通过 pub/sub 可以让它们各自独立回应，但最终需要仲裁机制——这应该是行蕴 IVolition 的 deliberate() 两阶段审议的工作（全局审议阶段评估多个 plugin 的建议是否冲突）。
3. **迟滞（hysteresis）机制。** 即使用连续激活函数，plugin 的启动和停止之间仍建议加入迟滞：启动阈值和停止阈值设不同值，避免在边界附近反复启停。
4. **传感器 plugin 的 skandha 归属。** 依据 Master's Letter 的超声波传感器范例，传感器通过组合持有三受实例，将 Raw Data 转为效价。传感器 plugin 的 skandha 字段应标记为色蕴（物质感测）+ 受蕴（感受生成），即多蕴归属。

## 总结

| 问题 | 结论 | 下一步 |
|------|------|--------|
| 效价连续性 | 连续性由 WIENER 层的滤波器保证，不由数据类型保证 | 决定效价是否共用烦恼信号的贝叶斯滤波框架；确认滤波器输出同时满足控制器和俱生完备性检查 |
| 效价分布与阈值 | 阈值意义取决于分布形状，需运行时校准 | 设计运行时分布观测器（纯统计版，受蕴内部），标注阈值的分布前提 |
| 二维模型边界 | 二维够用，控制感归行蕴 | 术语对齐，数据结构预留扩展性 |
| 处理层 plugin 落地 | 按功能划分，事件流协调，连续激活函数 | 激活函数设计、冲突仲裁、迟滞机制、传感器 skandha 归属确认 |

**优先级建议：** 问题一的缺口最小但最关键——差的只是把烦恼信号的滤波框架明确延伸到受蕴效价，一旦确认就可以写进接口定义。问题二是唯一的完全开放缺口，但不阻塞实现，可以在运行时迭代。问题三和四的架构方向已定，进入实现细节阶段。
