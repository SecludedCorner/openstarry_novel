# VasanaEngine 架構——問題分析與結論

## 原始設計（修正前）

原始構想中，Agent 認知有兩種固定速度：

- **Gear 1（快檔）**：VasanaEngine 規則引擎，~50ms，處理見過的模式
- **Gear 2（慢檔）**：VitakkaEngine 呼叫 LLM，數秒，處理複雜情況

識蘊根據匹配信心度決定走哪條路徑。兩個 Engine 都位於 ManoAggregator 核心內部。

以下分析基於此原始構想，最終結論為：兩個 Engine 都應從核心中移出，齒輪數量不由架構決定，而由 plugin 組合決定。

---

## 三個結構性問題

### 問題一：快檔無法真正跳過理解

VasanaEngine 要將自然語言輸入匹配到規則，這一步本身就是語意理解。用關鍵字匹配太脆弱（「幾點了」vs「現在幾點啊」vs「能告訴我時間嗎」），用 embedding 匹配則已經引入了模型。所謂「不問 LLM、幾十毫秒完成」的前提不成立——快檔和慢檔的邊界是模糊的。

佛學上這反而正確：即便是習氣的直覺反應，也是根、境、識三者和合（觸）之後才發生的，不存在完全繞過識的捷徑。

### 問題二：70-80% 快檔覆蓋率在通用框架中站不住

使用者找 AI Agent 處理的事情，多數是因為不簡單才來的。能用規則窮舉的模式在開放式自然語言場景下非常有限，實際比例可能是反過來的（20-30% 快檔、70-80% 慢檔）。如果快檔只覆蓋少數情境，維護規則庫的投入產出比就有問題。

此外，vasana（習氣）在唯識學中是被經驗薰習而不斷改變的種子，不是寫死的 if-then。如果 VasanaEngine 沒有從慢檔結果自動沉澱新規則的學習機制，它就只是靜態查詢表，名不符實。

### 問題三：切換決策是二元的，但信心度是連續的

設計中提到「匹配信心度」，是一個 0 到 1 的連續值，但架構只有二元切換（匹配/不匹配），沒有處理灰色地帶。信心度 0.6 時走快檔可能執行錯誤動作，走慢檔又浪費規則引擎價值。更關鍵的是，閾值該由誰設定——查時間判斷錯了無所謂，刪檔案判斷錯了是災難。不同場景、不同風險等級需要不同的閾值。

### 共同根源

三個問題指向同一件事：**把一個連續的、需要理解的、動態的認知過程，硬切成了一個離散的、不需理解的、靜態的二元開關。**

---

## 第四個問題：零內建能力原則的矛盾

OpenStarry 核心目前的組件全部是純機制——EventBus（路由）、ServiceRegistry（註冊）、PluginLoader（載入）、ExecutionLoop（排程）——不含任何領域知識。

VasanaEngine 打破了這個模式：它有規則庫（知識）、匹配邏輯（語義判斷）、信心度評分（決策）。**它不是機制，它是能力。** 放進核心就違反了「零內建能力」原則。

---

## 解決方案：拆開機制與能力

### 核心原則

矛盾的根源在於把齒輪切換「機制」和 VasanaEngine「能力」混為一體。它們應該分離。

### 最終架構

```
ManoAggregator（核心 = 純機制）
  │
  ├── IGearArbiter plugin chain（0 ~ N 個，可選）
  │     依序查詢每個已註冊的 IGearArbiter
  │     第一個信心度過閾值的就採用其結果
  │     全部沒過 → fallback 到 IProvider
  │     沒裝任何一個 → 永遠走 IProvider
  │
  ├── IProvider plugin 插槽（fallback）
  │     已有介面，LLM 能力來自 plugin
  │     沒裝 → Agent 無法推理（但核心不壞）
  │
  └── 閾值調節：Klesha（已有機制）
```

**核心只保留：** 依序問 IGearArbiter chain，有人過閾值就用，沒有就呼叫 IProvider。這是 middleware chain 模式，跟 EventBus 的路由一樣——純機制，不含知識。齒輪數量完全取決於裝了幾個 IGearArbiter plugin。

### VitakkaEngine 的處理

VitakkaEngine 的「能力」不是自己的——它只是組裝上下文然後呼叫 IProvider plugin 的管道，性質等同 EventBus 分發事件。但為徹底乾淨，它也不該作為核心的獨立組件存在，只是核心呼叫已有 plugin 的一條路徑。兩個 Engine 都不該在核心裡，核心裡只有路由機制。

### 不同 plugin 組合的退化行為

齒輪數量不是架構決定的，而是使用者根據場景裝了什麼 plugin 決定的：

| plugin 組合 | 行為 | 適用場景 |
|------------|------|---------|
| 多個 IGearArbiter + IProvider | 多檔瀑布式 fallback，完整運作 | 高精度場景 |
| 一個 IGearArbiter + IProvider | 經典雙速運作 | 客服機器人等 |
| 僅 IProvider | 每件事都走 LLM，無快速路徑 | 研究型 Agent |
| 僅 IGearArbiter | 只能處理規則匹配到的，無法推理 | 嵌入式 IoT |
| 皆無 | 核心存活但無法認知 | — |

## 佛學自洽性

這個拆法在佛學上更正確：

- 習氣（vasana）在唯識學中是**後天薰習的慣性模式**，不是心識的本有能力
- 一個眾生沒有任何習氣，依然有完整的認知能力（識），只是每件事都要重新認知
- 把 VasanaEngine 塞進核心 = 宣稱「習氣是意識的先天結構」——這在唯識學裡不成立

因此：

- 無 VasanaEngine plugin 的 Agent = 初生眾生，無習氣，每件事都要深思
- 有 VasanaEngine plugin 的 Agent = 有經驗的眾生，常見情境可直覺反應
- VasanaEngine 規則隨時間累積 = 習氣的薰習過程

不同眾生的習氣深淺不同——有的幾乎全靠直覺，有的事事深思。這不是意識結構的差異，是經驗累積的差異。核心（識的基本結構）是一樣的，差別在裝了什麼 plugin（累積了什麼習氣）。

---

## 額外建議

1. **讓 VasanaEngine 能學習**：IProvider 處理過的模式自動沉澱為新的 IGearArbiter 規則，真正對應「薰習」概念
2. **引入風險感知的動態閾值**：切換決策不只看信心度，還要看操作的風險等級，高風險操作寧可走 IProvider
