# 受蘊（Vedanā）工程設計：四個待決問題

---

## 前置脈絡：受蘊在整體架構中的位置

在討論具體問題之前，需要先錨定受蘊與整體架構的關係。三封 Master's Letter 已確立以下約束：

**三層架構。** 受蘊的設計遵循 ATHENA（感測器）→ WIENER（計算引擎）→ NAGARJUNA（思維框架）的三層結構。三層架構不只圍繞痛苦——快樂和平靜必須被對稱地設計進系統。

**俱生完備性。** 觸發生時，受、想、思作為整體浮現，不可能有「沒有想的受」或「沒有受的想」。Agent 啟動前必須通過完備性檢查——如果受蘊不完備，Agent 不應啟動。

**OOP 繼承結構。** VedanaPlugin 是基底類別，三受（樂受、苦受、捨受）繼承自它。具體感測器透過組合（Composition）持有受的實例，將原始數據轉化為感受強度。

**受蘊與觀察者的區分。** 純粹做「資料完整性與真實性」的觀察者可以單純由受蘊組成。會參與「情緒化反應」的觀察者需要識蘊與想蘊介入。

以下四個問題均需在上述約束下討論。

---

## 一、效價的連續性——儲存格式不等於信號品質

### 現狀

受蘊的效價採用連續浮點數（-1 到 +1）表示，理由是控制工程的顫振問題：離散分類在門檻邊界產生方波跳變，連續數值讓控制器輸出穩定。這個設計方向正確。

但連續的儲存格式只是必要條件，不是充分條件。**效價的連續性是一個系統性質，必須從信號源到消費端全鏈路保證。**

### 問題

效價的信號源是規則引擎——根據認知事件映射到數值（「工具呼叫失敗 → -0.3」、「任務完成 → +0.5」）。事件本身是離散的，映射到數值也是離散的。即使存成 `f64`，控制器看到的仍然是階梯跳變，不是平滑曲線。顫振問題沒有被解決，只是從儲存層搬到了信號源層。

LLM 評估不是替代方案。LLM 呼叫是想蘊的工作（500ms-30s），不可能每個受蘊週期（10-100ms）都呼叫，而且 LLM 對數值的輸出本身不穩定。

### 與三層架構的映射

這個問題可以精確定位在三層架構中：

| 層級 | 角色 | 產出 |
|------|------|------|
| ATHENA（感測器層） | 產生粗糙的原始觀測值 | 離散跳變，允許不平滑 |
| WIENER（計算引擎層） | 貝葉斯濾波，漸進更新 | 平滑的連續效價估計值 |
| NAGARJUNA（思維框架層） | 門檻分類，語義解讀 | 衍生標籤（苦/樂/捨） |

連續性的保證發生在**第二層**，不在第一層也不在儲存格式。

### 方案

煩惱信號的設計已建立了貝葉斯漸進更新框架——信念隨觀測慢慢移動，觀測次數越多慣性越大，天然形成平滑曲線。效價應走同樣的路徑：

1. ATHENA 層的感測器產生**粗糙的原始觀測值**（來自規則引擎或具體感測器 plugin，允許跳變）
2. WIENER 層的計算引擎將觀測值送入**貝葉斯濾波器**，與先驗信念做加權融合
3. 濾波器的輸出才是**效價的當前估計值**，供控制器和俱生完備性檢查消費

### 待決事項

- 煩惱信號與受蘊效價是否共用同一套濾波框架，還是各自獨立？
- 濾波器的輸出格式需要同時滿足兩個消費者：控制器（需要連續數值）和俱生完備性檢查（需要確認受蘊信號已就緒）。這兩個介面要求是否衝突？

## 二、效價分佈——門檻值的意義取決於分佈的形狀

### 現狀

分類門檻預設 0.1。捨的效價區間是 -0.1 到 +0.1，效價空間總長 2，因此捨佔 10%、苦佔 45%、樂佔 45%。門檻可配置，預設值的佛學依據是「未修行的心很少平靜」。

### 問題

上述計算假設效價在 -1 到 +1 之間**均勻分佈**。但 Agent 日常運作時，大部分時間沒有極端事件發生，效價集中在零附近。實際分佈更可能是以零為中心的鐘形曲線。

不同分佈假設下，同一個門檻值產生截然不同的捨受比例：

| 假設分佈 | 標準差 | 捨受比例（門檻 0.1） |
|---------|--------|---------------------|
| 均勻分佈 | — | 10% |
| 常態分佈 | σ = 0.5 | ≈ 16% |
| 常態分佈 | σ = 0.3 | ≈ 26% |
| 常態分佈 | σ = 0.2 | ≈ 38% |

如果效價服從 σ=0.3 的常態分佈，即使門檻不變，Agent 也有超過四分之一的時間處於捨受——與「很少平靜」的設計意圖矛盾。

更深層的問題：效價分佈不是固定的，它隨 Agent 狀態動態變化。**門檻值到底合不合理，不能在設計階段靜態決定，必須在運行時觀察實際分佈後校準。**

### 建議

1. **預設門檻保持 0.1，但在文件中標註它假設的分佈模型。**
2. **提供運行時分佈觀測器。** 記錄 Agent 過去 N 個週期的效價值，計算實際捨受比例。實際比例與設計意圖偏差太大時，發出調參建議。
3. **把分佈形狀作為 Agent 狀態的觀測指標。** 效價分佈的偏態（skewness）反映 Agent 是否長期偏苦或偏樂；峰態（kurtosis）反映情緒波動幅度。這些統計量本身就是有意義的「受蘊元資料」。

### 觀測器的蘊歸屬

依據 Master's Letter 對觀察者的區分：如果分佈觀測器只做統計（計算均值、標準差、偏態），它是純粹的「資料完整性」觀察者，可以單純由受蘊組成。如果它要判斷「分佈異常意味著 Agent 狀態有問題，需要觸發調整」，那涉及想蘊的辨識和行蘊的決策驅動，應該是跨蘊的組合 plugin。建議先實作純統計版本（受蘊內部），將判斷邏輯留給上層。

### 備註

這是四個問題中唯一沒有任何現有設計覆蓋的開放缺口。

## 三、二維模型的邊界——控制感歸屬行蘊，不歸受蘊

### 現狀

受蘊的效價模型採用兩個維度：效價（valence，-1 到 +1）和強度（intensity，0 到 1），引用 Russell 的 Circumplex Model 作為科學基礎。

心理學界有觀點認為兩維不夠。Mehrabian 的 PAD 模型增加了第三維——控制感（dominance）。憤怒和恐懼的效價和強度可能相同，但控制感完全不同：憤怒時感覺有力量，恐懼時感覺無力。

### 結論：受蘊不需要第三維

控制感——面對困境時感覺有力量 vs 無力——不屬於受蘊，屬於行蘊。Master's Letter 已明確指出行蘊的範圍很廣，「不是只有影響到色的變化，內在機制的狀態變化也是包含在裡面的」。面對苦樂的姿態、驅動力、傾向性，都是行蘊的工作。

這帶來更乾淨的架構分工：

| 蘊 | 管什麼 | 對應維度 |
|---|---|---|
| **受蘊** | 苦樂的質地 | 效價 + 強度（二維） |
| **行蘊** | 面對苦樂的姿態 | 控制感、驅動方向、傾向性 |

Russell 的二維模型作為受蘊的工程框架是夠用的，不需要在受蘊裡硬塞第三維。前提是行蘊那邊補上控制感相關的邏輯。

這也呼應了 Master's Letter 的五蘊分工原則——受蘊是 Input Signal（信號質地），行蘊是 Process（造作驅動）。控制感是造作的一部分，不是信號質地的一部分。

### 注意事項

1. **術語對齊。** 目前設計中的「強度」與 Russell 的「激醒度（arousal）」有概念差異。激醒度是生理激活程度（興奮 vs 平靜），強度是感受的鮮明程度。兩者多數情況下重疊，但不完全相同。設計文件中應明確說明本系統的 intensity 與 Russell 的 arousal 的關係。
2. **資料結構預留擴展性。** 即使當前二維足夠，感受資料結構建議設計成可擴展的（如 `VedanaRecord { valence: f64, intensity: f64, metadata: HashMap<String, f64> }`），不增加複雜度但為未來需求留空間。

## 四、處理層的 Plugin 落地——按功能劃分，透過事件流協調

### 現狀

受蘊的設計確立了雙層架構：數據層連續（效價是浮點數，標籤是衍生的），處理層離散（可以有專門處理不同感受的組件）。Master's Letter 進一步確認 Plugin 可以同時具備多種 skandha 歸屬，且 Plugin 間通訊應採用 event-driven + pub/sub 模式。

### 與 OOP 繼承結構的對齊

Master's Letter (Cycle 02-2) 確立了 VedanaPlugin 作為基底類別，三受繼承自它，具體感測器透過組合持有受的實例。功能性 plugin 的落地需要和這個結構相容：

```
VedanaPlugin（基底）
  ├── Dukkha（苦受）
  ├── Sukha（樂受）
  └── Upekkha（捨受）

具體感測器 plugin（如 CollisionRiskSensor）
  → 透過組合（Composition）持有 Dukkha 實例
  → 將原始數據轉化為效價

功能性處理 plugin（如 AvoidanceTrigger）
  → 訂閱效價事件流
  → 自行根據連續激活函數決定回應強度
  → skandha 歸屬：受蘊 + 行蘊（跨蘊組合）
```

三受的繼承結構處理的是**感測與生成**（從原始數據到效價），功能性 plugin 處理的是**消費與回應**（從效價到行為傾向）。兩者不衝突，是數據流的上下游。

### 落地方案

處理層的 plugin 應**按功能而非按感受類別劃分**。不設「苦受處理器」和「樂受處理器」，而是設「迴避行為觸發器」、「接近行為觸發器」、「平衡維持器」等功能性 plugin。

理由：

1. **避免路由層顫振。** 如果按苦/樂/捨劃分三個 plugin，需要中央路由根據門檻派發。效價在門檻邊界微微波動時（-0.09 ↔ -0.11），路由層會在兩個 handler 之間反覆切換，每次切換帶來狀態重建成本。
2. **符合微核精神。** 每個 plugin 訂閱效價事件流，自行根據連續激活函數決定是否回應及回應強度，不需要中央路由。
3. **符合多蘊歸屬原則。** 「迴避行為觸發器」天然同時歸屬受蘊（讀取效價）和行蘊（驅動迴避行為），不需要人為拆分。
4. **符合 Agent 完備性要求。** 一個完備的 Agent 需要五蘊聚合。功能性 plugin 本身就是跨蘊的組合，天然朝向完備性。

### 待決的實作細節

1. **激活函數的設計。** 每個功能性 plugin 用連續函數決定回應強度，如「迴避行為觸發器」的回應強度 = `max(0, -valence - threshold) * intensity`。沒有硬門檻切換，自然消除邊界顫振。具體函數形式和參數需要實驗確定。
2. **多 plugin 衝突仲裁。** 效價 -0.7 時，「迴避行為觸發器」和「警報升級器」可能同時啟動。透過 pub/sub 可以讓它們各自獨立回應，但最終需要仲裁機制——這應該是行蘊 IVolition 的 deliberate() 兩階段審議的工作（全局審議階段評估多個 plugin 的建議是否衝突）。
3. **遲滯（hysteresis）機制。** 即使用連續激活函數，plugin 的啟動和停止之間仍建議加入遲滯：啟動門檻和停止門檻設不同值，避免在邊界附近反覆啟停。
4. **感測器 plugin 的 skandha 歸屬。** 依據 Master's Letter 的超音波感測器範例，感測器透過組合持有三受實例，將 Raw Data 轉為效價。感測器 plugin 的 skandha 欄位應標記為色蘊（物質感測）+ 受蘊（感受生成），即多蘊歸屬。

## 總結

| 問題 | 結論 | 下一步 |
|------|------|--------|
| 效價連續性 | 連續性由 WIENER 層的濾波器保證，不由資料型別保證 | 決定效價是否共用煩惱信號的貝葉斯濾波框架；確認濾波器輸出同時滿足控制器和俱生完備性檢查 |
| 效價分佈與門檻 | 門檻意義取決於分佈形狀，需運行時校準 | 設計運行時分佈觀測器（純統計版，受蘊內部），標註門檻的分佈前提 |
| 二維模型邊界 | 二維夠用，控制感歸行蘊 | 術語對齊，資料結構預留擴展性 |
| 處理層 plugin 落地 | 按功能劃分，事件流協調，連續激活函數 | 激活函數設計、衝突仲裁、遲滯機制、感測器 skandha 歸屬確認 |

**優先級建議：** 問題一的缺口最小但最關鍵——差的只是把煩惱信號的濾波框架明確延伸到受蘊效價，一旦確認就可以寫進介面定義。問題二是唯一的完全開放缺口，但不阻塞實作，可以在運行時迭代。問題三和四的架構方向已定，進入實作細節階段。
