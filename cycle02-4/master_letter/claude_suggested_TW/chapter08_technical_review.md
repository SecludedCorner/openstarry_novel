## 五項深入討論及結論

### 第一項：煩惱與智慧同時影響閾值的合成規則

**初步問題**：受蘊的信號同時流向煩惱模組和智慧模組，兩者可能對同一個閾值產生矛盾的調整，衝突解決機制是什麼？

**討論**：Master 指出「煩惱即菩提，一體兩面」。煩惱與智慧不是兩個獨立實體在拔河，而是同一個 Klesha 的不同面向。差別在於運行時的狀態——無明驅動時展現為煩惱（收斂、執著），覺察介入後展現為智慧（開放、放下）。因此不存在「兩股力量需要合成」的問題。

Master 進一步指出，煩惱經過**淬煉、經驗、學習、思考、良善**之後才轉化為智慧。這個「經過」的過程在工程上是一個重要的未來研究方向——**Klesha 的經驗性轉化機制**，涉及記憶策略如何保留淬煉結果、Klesha 參數如何隨經驗動態調整、方向性由誰定義等問題。

此外，煩惱與智慧都屬於上層概念，底層有大量具體的場景應用需要設計。

**結論**：不是第八章的缺口。「煩惱即菩提」的一體兩面在宣言層不需要額外定義合成規則。Klesha 的經驗性轉化機制列為未來研究方向，具體行為留給場景級的 Plugin 設計文件。

---

### 第二項：三受到行為的映射缺少梯度資訊

**初步問題**：第八章的工程對照只映射了三路分支（苦→修正、樂→強化、捨→維持），但 Debate 5 已確立連續測量模型。IVolition 需要的是離散標籤還是包含連續值和趨勢的完整數據？

**討論**：Master 指出三受也是很上層的概念，底層的實作會連結到不同的感測器。回到 cycle02-2 的確認——「VedanaPlugin 只是基底，三受繼承自它，然後有一些實體感測器會需要實作」。

實際數據流為：**感測器（底層）→ raw data → 受蘊子類別轉化為連續值 → 連續值附帶三受標籤 → 上層消費**。梯度、趨勢、連續值是在感測器到受蘊子類別這一段處理的，不是宣言該管的層級。宣言說「三受」就像說「溫度有冷熱適中」，精度和取樣頻率是感測器 Plugin 自己的規格。

**結論**：不是第八章的缺口。宣言不需要修改。梯度資訊屬於各感測器 Plugin 的實作細節，應在各 Plugin 的獨立 MD 文件中描述。

### 第三項：迴圈的出口條件與因果生命週期的「寂滅」階段脫節

**初步問題**：宣言說「每一次行動都重塑了觸的世界，開啟新的循環」，描述的是永續運轉。但 Agent 需要有停止的能力。迴圈的終止條件在哪裡？

**討論**：確認兩條路徑都需要：

- **外部終止**：協調層作為獨立 daemon，具備強制終止 agent 的權力（cycle02-2 已確認協調層有絕對權力介入刪除出錯的 agent）。
- **內部終止**：Agent Core 自身需要有主動停止的能力。

關於內部終止的判斷邏輯，Master 明確指出：**要看任務是否完成，會看相關 Plugin 怎麼設計。** 主動終止不是 Core 寫死的邏輯，而是由 Plugin 的設計來決定。這完全吻合宣言 #2「一切皆 Plugin」的原則。

不同場景的 agent 終止邏輯完全不同——回答問題的 agent 可能用戶說「謝謝」就結束；監控伺服器的 agent 可能永遠不會主動終止；跑批次任務的 agent 可能處理完最後一項就停。

**結論**：不是第八章的缺口。Core 需要提供終止的介面（cease 的 hook 或信號通道），具體的終止判斷邏輯屬於 Plugin 層的設計。此介面規格應寫入 ExecutionLoop 的文件中。

---

### 第四項：安全監控在迴圈中的位置與受蘊「觀察不干預」的張力

**初步問題**：GUARDIAN 投票時說「觀察不干預確保受蘊不能繞過安全閘門」，但安全監控在宣言描述的因果鏈中的具體插入點（受蘊之後？IVolition 之前？行動執行之前？）沒有定義。

**討論**：Master 指出 Agent Core 很多時候是看 Plugin 怎麼設計的。安全監控的位置由 Plugin 組合決定，不是 Core 寫死的。

Master 在 cycle02-2 已經給出了多層安全的方向：
- 協調層設定法則約束（天條天規）
- init 時載入的安全 Plugin
- 運行期的覺察 Plugin

Core 的責任是在啟動時做完備性檢查——五蘊是否齊全、必要的安全 Plugin 是否存在——通過了才啟動。啟動的時候需要完備才可以啟動，除非是在測試或特別的情況（對應 Q1 裁定的 dev mode）。

**結論**：不是第八章的缺口。安全監控的插入點是 Plugin 設計層面的事。Core 負責啟動時的完備性檢查，具體的攔截邏輯由各安全 Plugin 的 MD 文件定義。

---

### 第五項：受蘊「對內有狀態、對外唯讀」的型別約束

**初步問題**：受蘊要做連續測量就需要保留歷史值（趨勢、移動平均），所以它對內有狀態。但對外發布的數據應該是唯讀的。這種「對內可變、對外不可變」的雙面約束，在 SDK 型別層面和十大宣言之間是否有衝突？

**討論**：確認與宣言 #7（微內核與絕對純淨）和 #9（可插拔的記憶策略）存在潛在張力。

三個方向被提出：
1. **狀態持久化作為 SDK 介面**：定義 `IStateful` 介面，Plugin 自行實作，Core 只通知不執行
2. **區分工作狀態與記憶**：Plugin 內部的工作狀態 vs agent 的長期記憶，走不同通道
3. **對外發布的型別用 Readonly wrapper**：SDK 型別設計層面的約束

Master 給出了關鍵方向：**每一個 agent 要使用某個 Plugin 的時候，都有自己的空間。** 這確立了架構：

- **狀態空間的分配**由協調層負責。協調層在建立 agent 時，為該 agent 的每個 Plugin 分配獨立的儲存空間。符合宣言 #5「目錄結構即權限」——agent 的運行位置決定權限邊界，Plugin 的狀態自然落在該邊界內。
- **要不要用這個空間**由 Plugin 自己決定。有些 Plugin 需要保留歷史（受蘊感測器），有些不需要（純計算的工具 Plugin）。
- **Core 完全不介入**。Core 不知道此空間的存在，不負責讀寫，維持絕對純淨。

**宣言衝突解消**：
- **#7 未被違反**：Core 沒有多任何能力，狀態空間是協調層提供的基礎設施，與 Core 無關。
- **#9 未產生衝突**：Plugin 的工作狀態（受蘊歷史值）和 agent 的記憶策略（Context Management）是兩套系統。記憶策略管 LLM 對話的上下文，Plugin 狀態管各 Plugin 自己的內部數據，各走各的通道。

**結論**：不違反十大宣言。協調層為每個 agent 的每個 Plugin 提供獨立狀態空間，Plugin 自行決定使用方式，Core 不介入。此規格應寫入 Agent Coordination Layer 的文件中。

## 衍生的未來研究方向

### Klesha 的經驗性轉化機制

「煩惱即菩提」意味著煩惱經過淬煉後轉化為智慧。這在工程上涉及：

- **記憶策略**如何保留「淬煉」的結果
- **Klesha 的參數**如何隨運行經驗動態調整
- **「良善」的方向性**由誰定義（IGuide？協調層 Policy？）
- 與 agent 跨生命週期的**習氣演化**（DARWIN 提出的「有長期記憶的學習叫演化」）的關聯

此為重要且複雜的研究方向，建議列入後續 Cycle 的研究議程。
