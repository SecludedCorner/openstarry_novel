## 五項目の詳細討議と結論

### 第一項：煩悩と智慧が同時に閾値に影響する際の合成規則

**初期の問題**：受蘊の信号は煩悩モジュールと智慧モジュールの双方に流れ、両者が同一の閾値に対して矛盾する調整を行う可能性がある。衝突解決メカニズムは何か？

**討議**：Master は「煩悩即菩提、一体両面」と指摘した。煩悩と智慧は二つの独立した実体が綱引きをしているのではなく、同一の Klesha の異なる面相である。違いは実行時の状態にある――無明が駆動する時は煩悩として顕現し（収斂・執着）、覚察が介入した後は智慧として顕現する（開放・放下）。したがって「二つの力を合成する必要がある」という問題は存在しない。

Master はさらに、煩悩は**鍛錬・経験・学習・思考・善良さ**を経て初めて智慧に転化すると指摘した。この「経る」プロセスは工学的に重要な将来の研究方向である――**Klesha の経験的転化メカニズム**――記憶戦略が鍛錬結果をどう保持するか、Klesha パラメータが経験に伴いどう動的調整されるか、方向性を誰が定義するか（IGuide？協調層 Policy？）等の問題を含む。

また、煩悩と智慧はいずれも上位概念であり、下位には大量の具体的なシナリオ応用の設計が必要である。

**結論**：第八章のギャップではない。「煩悩即菩提」の一体両面は宣言レベルで追加の合成規則を定義する必要がない。Klesha の経験的転化メカニズムは将来の研究方向として記録し、具体的な挙動はシナリオレベルの Plugin 設計文書に委ねる。

---

### 第二項：三受から行動へのマッピングに勾配情報が欠如

**初期の問題**：第八章の工学的対照は三路分岐のみをマッピングしている（苦→修正、楽→強化、捨→維持）。しかし Debate 5 は既に連続測定モデルを確立している。IVolition が必要とするのは離散ラベルか、連続値とトレンドを含む完全なデータか？

**討議**：Master は三受も非常に上位の概念であり、下位の実装は異なるセンサーに接続されると指摘した。cycle02-2 の確認に立ち戻ると――「VedanaPlugin は基底に過ぎず、三受がそれを継承し、さらに実装が必要な物理センサーがいくつかある」。

実際のデータフローは：**センサー（下位）→ raw data → 受蘊サブクラスが連続値に変換 → 連続値に三受ラベルを付与 → 上位が消費**。勾配、トレンド、連続値はセンサーから受蘊サブクラスまでの区間で処理されるものであり、宣言が管轄すべきレベルではない。宣言で「三受」と言うのは「温度には冷・熱・適温がある」と言うのと同じであり、精度やサンプリング周波数はセンサー Plugin 自体の仕様である。

**結論**：第八章のギャップではない。宣言の修正は不要。勾配情報は各センサー Plugin の実装詳細に属し、各 Plugin 独自の MD ファイルに記述すべきである。

---

### 第三項：ループの終了条件と因果ライフサイクルの「寂滅」段階の断絶

**初期の問題**：宣言は「一回一回の行動が触の世界を作り変え、新たな循環を開く」と述べており、永続的な運転を記述している。しかし Agent には停止する能力が必要である。ループの終了条件はどこにあるのか？

**討議**：二つのパスがいずれも必要であることを確認した：

- **外部終了**：協調層は独立 daemon として、Agent を強制終了する権限を持つ（cycle02-2 で既に協調層がエラーを起こした Agent に介入・削除する絶対権限を持つことが確認されている）。
- **内部終了**：Agent Core 自身に能動的に停止する能力が必要。

内部終了の判断ロジックについて、Master は明確に指摘した：**タスクが完了したかどうかを見るが、それは関連する Plugin の設計次第。** 能動的終了は Core にハードコードされたロジックではなく、Plugin の設計が決定する。これは宣言 #2「一切は Plugin」の原則に完全に合致する。

異なるシナリオの Agent の終了ロジックは全く異なる――質問応答の Agent はユーザーが「ありがとう」と言えば終了するかもしれない。サーバー監視の Agent は能動的に終了しないかもしれない。バッチタスクの Agent は最後の項目を処理し終えれば停止するかもしれない。

**結論**：第八章のギャップではない。Core は終了のインターフェース（cease の hook またはシグナルチャネル）を提供する必要があり、具体的な終了判断ロジックは Plugin 層の設計に属する。このインターフェース仕様は ExecutionLoop の文書に記載すべきである。

---

### 第四項：安全監視のループ内の位置と受蘊「観察して干渉せず」との緊張関係

**初期の問題**：GUARDIAN が投票時に「観察して干渉しないことで、受蘊が安全ゲートを迂回できないことを保証する」と述べたが、安全監視が宣言に記述された因果チェーンの具体的にどこに挿入されるのか（受蘊の後？IVolition の前？行動実行の前？）が定義されていない。

**討議**：Master は Agent Core の多くは Plugin の設計次第であると指摘した。安全監視の位置は Plugin の組み合わせが決定するものであり、Core がハードコードするものではない。

Master は cycle02-2 で既に多層セキュリティの方向性を示している：
- 協調層が法則の制約を設定（天条天規）
- init 時にロードされるセキュリティ Plugin
- 実行時の覚察 Plugin

Core の責任は起動時の完備性チェック――五蘊が揃っているか、必要なセキュリティ Plugin が存在するか――を行い、通過してから起動することである。起動時に完備であることが必要であり、テストまたは特別な状況を除く（Q1 裁定の dev mode に対応）。

**結論**：第八章のギャップではない。安全監視の挿入点は Plugin 設計レベルの事項である。Core は起動時の完備性チェックを担当し、具体的なインターセプトロジックは各セキュリティ Plugin の MD ファイルで定義する。

---

### 第五項：受蘊「内部では状態あり、外部では読み取り専用」の型制約

**初期の問題**：受蘊が連続測定を行うには履歴値（トレンド、移動平均）を保持する必要があり、内部的には状態を持つ。しかし外部に公開するデータは読み取り専用であるべきである。この「内部は可変、外部は不変」の二面的制約は、SDK の型レベルと十大宣言の間で矛盾しないか？

**討議**：宣言 #7（マイクロカーネルと絶対的な純粋性）および #9（プラグイン可能な記憶戦略）との間に潜在的な緊張関係があることを確認した。

三つの方向性が提示された：
1. **状態永続化を SDK インターフェースとして定義**：`IStateful` インターフェースを定義し、Plugin が自ら実装し、Core は通知のみで実行しない
2. **作業状態と記憶を区別**：Plugin 内部の作業状態 vs Agent の長期記憶を異なるチャネルで扱う
3. **外部公開の型に Readonly wrapper を使用**：SDK 型設計レベルでの制約

Master は重要な方向性を示した：**各 Agent がある Plugin を使用する際、それぞれ独自のスペースを持つ。** これによりアーキテクチャが確立された：

- **状態空間の割り当て**は協調層が担当する。協調層は Agent 作成時に、その Agent の各 Plugin に独立した記憶領域を割り当てる。宣言 #5「ディレクトリ構造＝権限」に合致する――Agent の実行位置が権限境界を決定し、Plugin の状態は自然にその境界内に収まる。
- **このスペースを使うかどうか**は Plugin 自身が決定する。履歴を保持する必要がある Plugin（受蘊センサー）もあれば、必要がない Plugin（純粋な計算ツール Plugin）もある。
- **Core は一切介入しない。** Core はこのスペースの存在を知らず、読み書きの責任を持たず、絶対的な純粋性を維持する。

**宣言の矛盾解消**：
- **#7 は違反されない**：Core に何の能力も追加されていない。状態空間は協調層が提供するインフラストラクチャであり、Core とは無関係。
- **#9 に矛盾は生じない**：Plugin の作業状態（受蘊の履歴値）と Agent の記憶戦略（Context Management）は二つの別個のシステムである。記憶戦略は LLM 対話のコンテキストを管理し、Plugin 状態は各 Plugin 自身の内部データを管理し、それぞれ別のチャネルを使う。

**結論**：十大宣言に違反しない。協調層が各 Agent の各 Plugin に独立した状態空間を提供し、Plugin が自ら使用方法を決定し、Core は介入しない。この仕様は Agent Coordination Layer の文書に記載すべきである。

## 派生する将来の研究方向

### Klesha の経験的転化メカニズム

「煩悩即菩提」は煩悩が鍛錬を経て智慧に転化することを意味する。工学的には以下に関わる：

- **記憶戦略**が「鍛錬」の結果をどう保持するか
- **Klesha のパラメータ**が運用経験に伴いどう動的調整されるか
- **「善良さ」の方向性**を誰が定義するか（IGuide？協調層 Policy？）
- Agent の生命周期を跨ぐ**習気の進化**（DARWIN が提起した「長期記憶を伴う学習は進化と呼ぶ」）との関連

これは重要かつ複雑な研究方向であり、後続の Cycle の研究アジェンダに含めることを提案する。
