# VasanaEngine アーキテクチャ――問題分析と結論

## 元の設計（修正前）

元の構想では、Agent の認知には二つの固定速度があった：

- **Gear 1（高速）**：VasanaEngine ルールエンジン、約50ms、既知のパターンを処理
- **Gear 2（低速）**：VitakkaEngine が LLM を呼び出し、数秒、複雑な状況を処理

識蘊（vijnana-skandha）がマッチング信頼度に基づいてどちらの経路を取るかを決定する。二つの Engine はいずれも ManoAggregator コア内部に位置していた。

以下の分析はこの元の構想に基づく。最終結論は：二つの Engine はいずれもコアから取り出すべきであり、ギア数はアーキテクチャが決めるのではなく、Plugin の組み合わせが決める、というものである。

---

## 三つの構造的問題

### 問題一：高速ギアは理解を真にスキップできない

VasanaEngine が自然言語入力をルールにマッチングさせる――このステップ自体が意味理解である。キーワードマッチングは脆弱すぎ（「何時？」vs「今何時ですか」vs「時間を教えてもらえますか」）、Embedding マッチングを使えば既にモデルが介在している。いわゆる「LLM に問い合わせず、数十ミリ秒で完了する」前提は成立しない――高速ギアと低速ギアの境界は曖昧である。

仏学的にはこれはむしろ正しい：習気（vasana）による直覚的反応であっても、根・境・識三者の和合（触、sparsha）の後に初めて発生するものであり、識を完全に迂回する近道は存在しない。

### 問題二：70-80%の高速ギアカバー率は汎用フレームワークでは成立しない

ユーザーが AI Agent に処理を依頼する事柄は、大半が簡単ではないからこそ依頼するのである。ルールで網羅できるパターンは開放的な自然言語シナリオでは極めて限定的であり、実際の比率はおそらく逆（20-30%が高速ギア、70-80%が低速ギア）である。高速ギアが少数の状況しかカバーしないなら、ルールベースの維持に対する投入産出比に問題がある。

さらに、習気（vasana）は唯識学（Vijnaptimatrata）において経験によって薫習され絶えず変化する種子であり、固定的な if-then ではない。VasanaEngine に低速ギアの結果から新規ルールを自動的に沈殿させる学習メカニズムがなければ、それは静的な検索テーブルに過ぎず、名に値しない。

### 問題三：切り替え判断は二値だが、信頼度は連続的

設計では「マッチング信頼度」に言及しているが、これは 0 から 1 の連続値である。しかしアーキテクチャには二値の切り替え（マッチ/非マッチ）しかなく、グレーゾーンの処理がない。信頼度 0.6 で高速ギアを選べば誤った動作を実行する可能性があり、低速ギアを選べばルールエンジンの価値を無駄にする。さらに重要なのは、閾値を誰が設定するかである――時刻の確認を誤判断しても問題ないが、ファイル削除を誤判断すれば災害となる。異なるシナリオ、異なるリスクレベルには異なる閾値が必要である。

### 共通の根本原因

三つの問題は同じ一点を指し示している：**連続的で、理解を要し、動態的な認知プロセスを、離散的で、理解不要で、静的な二値スイッチに無理やり切り分けたこと。**

## 第四の問題：ゼロ内蔵能力原則との矛盾

OpenStarry コアの現在のコンポーネントはすべて純粋なメカニズムである――EventBus（ルーティング）、ServiceRegistry（登録）、PluginLoader（ロード）、ExecutionLoop（スケジューリング）――ドメイン知識を一切含まない。

VasanaEngine はこのパターンを破る：ルールベース（知識）、マッチングロジック（意味判断）、信頼度スコアリング（意思決定）を持つ。**それはメカニズムではなく、能力である。** コアに入れれば「ゼロ内蔵能力」原則に違反する。

---

## 解決策：メカニズムと能力の分離

### 核心原則

矛盾の根源は、ギア切り替え「メカニズム」と VasanaEngine「能力」を一体にしたことにある。両者は分離すべきである。

### 最終アーキテクチャ

```
ManoAggregator（コア = 純メカニズム）
  │
  ├── IGearArbiter plugin chain（0 ~ N 個、オプション）
  │     登録済みの各 IGearArbiter に順次問い合わせ
  │     最初に信頼度が閾値を超えたものの結果を採用
  │     全て閾値未満 → IProvider にフォールバック
  │     一つも装着していない → 常に IProvider を経由
  │
  ├── IProvider plugin スロット（フォールバック）
  │     既存インターフェース、LLM 能力は Plugin から提供
  │     未装着 → Agent は推論不能（ただしコアは壊れない）
  │
  └── 閾値調節：Klesha（既存メカニズム）
```

**コアが保持するのは：** IGearArbiter chain に順次問い合わせ、閾値を超えたものがあればそれを採用し、なければ IProvider を呼び出す――これだけである。これは middleware chain パターンであり、EventBus のルーティングと同じ――純粋なメカニズムであり、知識を含まない。ギア数は装着した IGearArbiter Plugin の数によって完全に決まる。

### VitakkaEngine の処理

VitakkaEngine の「能力」は自身のものではない――コンテキストを組み立てて IProvider Plugin を呼び出すパイプラインに過ぎず、性質は EventBus がイベントを配信するのと同等である。しかし徹底的にクリーンにするため、これもコアの独立コンポーネントとして存在すべきではなく、コアが既存の Plugin を呼び出す一つの経路に過ぎない。二つの Engine はいずれもコアに置くべきではなく、コアにはルーティングメカニズムのみが存在する。

### 異なる Plugin 組み合わせでの縮退動作

ギア数はアーキテクチャが決めるのではなく、ユーザーがシナリオに応じて装着した Plugin が決める：

| Plugin 組み合わせ | 動作 | 適用シナリオ |
|------------------|------|-------------|
| 複数の IGearArbiter + IProvider | 多段カスケード式フォールバック、完全動作 | 高精度シナリオ |
| 一つの IGearArbiter + IProvider | 古典的二速動作 | カスタマーサポートボット等 |
| IProvider のみ | 全て LLM 経由、高速パスなし | 研究型 Agent |
| IGearArbiter のみ | ルールマッチできるもののみ処理、推論不能 | 組み込み IoT |
| いずれもなし | コアは生存するが認知不能 | ― |

## 仏学的整合性

この分離は仏学的にもより正しい：

- 習気（vasana）は唯識学において**後天的に薫習された慣性パターン**であり、心識の本有の能力ではない
- 一つの衆生に何の習気もなくとも、完全な認知能力（識）は備わっている――ただ全てのことを新たに認知しなければならないだけである
- VasanaEngine をコアに入れること＝「習気は意識の先天的構造である」と宣言すること――これは唯識学では成立しない

したがって：

- VasanaEngine Plugin なしの Agent ＝ 初生の衆生、習気なし、全てのことを深く考える
- VasanaEngine Plugin ありの Agent ＝ 経験ある衆生、よくある状況には直覚的に反応できる
- VasanaEngine のルールが時間と共に蓄積される＝ 習気の薫習プロセス

衆生によって習気の深浅は異なる――ほぼ全て直覚で動く者もいれば、全て深く考える者もいる。これは意識構造の違いではなく、経験蓄積の違いである。コア（識の基本構造）は同じであり、違いは何の Plugin を装着しているか（何の習気を蓄積しているか）にある。

---

## 追加提案

1. **VasanaEngine に学習させる**：IProvider が処理したパターンを自動的に新しい IGearArbiter ルールとして沈殿させ、真に「薫習」の概念に対応させる
2. **リスク認識に基づく動的閾値を導入する**：切り替え判断は信頼度だけでなく、操作のリスクレベルも考慮し、高リスク操作では IProvider を経由する方を選ぶ
