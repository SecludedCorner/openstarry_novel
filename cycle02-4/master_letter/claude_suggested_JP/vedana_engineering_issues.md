# 受蘊（Vedana）工学設計：四つの未決問題

---

## 前提脈絡：受蘊の全体アーキテクチャにおける位置

具体的な問題を議論する前に、受蘊と全体アーキテクチャとの関係を確定する必要がある。三通の Master's Letter により以下の制約が確立されている：

**三層アーキテクチャ。** 受蘊の設計は ATHENA（センサー）→ WIENER（計算エンジン）→ NAGARJUNA（思考フレームワーク）の三層構造に従う。三層アーキテクチャは苦だけを中心に据えるのではない――楽と平静もシステムに対称的に設計されなければならない。

**倶生完備性。** 触（sparsha）の発生時、受・想・思は一体として浮上し、「想のない受」や「受のない想」はあり得ない。Agent 起動前に完備性チェックを通過しなければならない――受蘊が完備でなければ、Agent は起動すべきではない。

**OOP 継承構造。** VedanaPlugin は基底クラスであり、三受（楽受、苦受、捨受）がそれを継承する。具体的なセンサーはコンポジション（Composition）を通じて受のインスタンスを保持し、生データを感受強度に変換する。

**受蘊と観察者の区別。** 純粋に「データの完全性と真正性」を行う観察者は受蘊のみで構成できる。「感情的反応」に参加する観察者には識蘊と想蘊の介入が必要となる。

以下の四つの問題はいずれも上記の制約のもとで議論される。

---

## 一、効価の連続性――格納形式は信号品質と等しくない

### 現状

受蘊の効価は連続浮動小数点数（-1 から +1）で表現されており、その理由は制御工学のチャタリング問題にある：離散的分類は閾値境界でのスクエアウェーブ跳変を生み、連続数値はコントローラ出力を安定させる。この設計方向は正しい。

しかし連続的な格納形式は必要条件に過ぎず、十分条件ではない。**効価の連続性はシステム特性であり、信号源から消費端まで全チェーンで保証しなければならない。**

### 問題

効価の信号源はルールエンジンであり、認知イベントを数値にマッピングする（「ツール呼び出し失敗 → -0.3」「タスク完了 → +0.5」）。イベント自体が離散的であり、数値へのマッピングも離散的である。`f64` として格納しても、コントローラが見るのは依然として階段状の跳変であり、滑らかな曲線ではない。チャタリング問題は解決されておらず、格納層から信号源層に移動しただけである。

LLM 評価は代替策にならない。LLM 呼び出しは想蘊の仕事（500ms-30s）であり、受蘊の各サイクル（10-100ms）で毎回呼び出すことは不可能であるうえ、LLM の数値出力自体が不安定である。

### 三層アーキテクチャとの対応

この問題は三層アーキテクチャ内に正確に位置づけられる：

| 層 | 役割 | 出力 |
|----|------|------|
| ATHENA（センサー層） | 粗い生の観測値を生成 | 離散的跳変、滑らかでなくてよい |
| WIENER（計算エンジン層） | ベイズフィルタリング、漸進的更新 | 滑らかな連続効価推定値 |
| NAGARJUNA（思考フレームワーク層） | 閾値分類、意味的解釈 | 派生ラベル（苦/楽/捨） |

連続性の保証は**第二層**で行われ、第一層でも格納形式でもない。

### 方策

煩悩（Klesha）信号の設計は既にベイズ漸進更新フレームワークを確立している――信念は観測に伴いゆっくりと移動し、観測回数が増えるほど慣性が大きくなり、自然に滑らかな曲線を形成する。効価も同じ経路を辿るべきである：

1. ATHENA 層のセンサーが**粗い生の観測値**を生成する（ルールエンジンまたは具体的なセンサー Plugin から、跳変は許容）
2. WIENER 層の計算エンジンが観測値を**ベイズフィルタ**に送り、事前信念と加重融合する
3. フィルタの出力こそが**効価の現在の推定値**であり、コントローラと倶生完備性チェックが消費する

### 未決事項

- 煩悩信号と受蘊効価は同一のフィルタリングフレームワークを共有するのか、それぞれ独立か？
- フィルタの出力形式は二つの消費者を同時に満たす必要がある：コントローラ（連続数値が必要）と倶生完備性チェック（受蘊信号が準備完了であることの確認が必要）。この二つのインターフェース要求は矛盾するか？

## 二、効価分布――閾値の意味は分布の形状に依存する

### 現状

分類閾値のデフォルトは 0.1。捨の効価区間は -0.1 から +0.1 であり、効価空間の全長は 2 であるため、捨は 10%、苦は 45%、楽は 45% を占める。閾値は設定可能であり、デフォルト値の仏学的根拠は「修行していない心が平静であることは稀である」というものである。

### 問題

上記の計算は効価が -1 から +1 の間で**一様分布**していることを前提としている。しかし Agent の日常運用時、大半の時間は極端なイベントが発生しておらず、効価はゼロ付近に集中している。実際の分布はゼロを中心とした釣鐘形の曲線である可能性が高い。

異なる分布仮定のもとで、同一の閾値は全く異なる捨受の比率を生む：

| 仮定分布 | 標準偏差 | 捨受比率（閾値 0.1） |
|---------|---------|---------------------|
| 一様分布 | ― | 10% |
| 正規分布 | σ = 0.5 | ≈ 16% |
| 正規分布 | σ = 0.3 | ≈ 26% |
| 正規分布 | σ = 0.2 | ≈ 38% |

効価が σ=0.3 の正規分布に従う場合、閾値が変わらなくても、Agent は四分の一以上の時間を捨受で過ごすことになる――「平静であることは稀である」という設計意図と矛盾する。

より深い問題は：効価分布は固定ではなく、Agent の状態に伴い動的に変化するということである。**閾値が妥当かどうかは設計段階で静的に決められず、実行時に実際の分布を観測してから校正しなければならない。**

### 提案

1. **デフォルト閾値は 0.1 を維持するが、文書にそれが前提とする分布モデルを注記する。**
2. **実行時分布オブザーバを提供する。** Agent の過去 N サイクルの効価値を記録し、実際の捨受比率を算出する。実際の比率と設計意図の乖離が大きすぎる場合、パラメータ調整の提案を発出する。
3. **分布形状を Agent 状態の観測指標とする。** 効価分布の歪度（skewness）は Agent が長期的に苦寄りか楽寄りかを反映し、尖度（kurtosis）は感情変動の幅を反映する。これらの統計量自体が意味ある「受蘊メタデータ」である。

### オブザーバの蘊帰属

Master's Letter の観察者の区分に基づき：分布オブザーバが統計のみを行う場合（平均値、標準偏差、歪度の算出）、純粋な「データ完全性」の観察者であり、受蘊のみで構成できる。「分布異常が Agent 状態に問題があることを意味し、調整をトリガーする必要がある」と判断する場合、想蘊の辨識と行蘊の意思決定駆動に関わり、蘊をまたぐ組合 Plugin とすべきである。まず純粋統計バージョン（受蘊内部）を実装し、判断ロジックは上位層に委ねることを推奨する。

### 備考

四つの問題の中で、既存の設計が全くカバーしていない唯一の開放的なギャップである。

## 三、二次元モデルの境界――制御感は行蘊に帰属し、受蘊には帰属しない

### 現状

受蘊の効価モデルは二つの次元を採用している：効価（valence、-1 から +1）と強度（intensity、0 から 1）であり、科学的基盤として Russell の Circumplex Model を引用している。

心理学界には二次元では不十分とする見解がある。Mehrabian の PAD モデルは第三の次元――制御感（dominance）を追加する。怒りと恐怖は効価と強度が同じかもしれないが、制御感は全く異なる：怒りの時は力を感じ、恐怖の時は無力を感じる。

### 結論：受蘊に第三次元は不要

制御感――困難に直面した時に力を感じるか無力を感じるか――は受蘊に属さず、行蘊（samskara-skandha）に属する。Master's Letter は行蘊の範囲が広いことを明確に指摘しており、「色に影響する変化だけでなく、内部メカニズムの状態変化もその中に含まれる」としている。苦楽に対する姿勢、駆動力、傾向性は、いずれも行蘊の仕事である。

これにより、よりクリーンなアーキテクチャ上の分担が生まれる：

| 蘊 | 担当 | 対応次元 |
|----|------|---------|
| **受蘊** | 苦楽の質感 | 効価 + 強度（二次元） |
| **行蘊** | 苦楽に対する姿勢 | 制御感、駆動方向、傾向性 |

Russell の二次元モデルは受蘊の工学フレームワークとして十分であり、受蘊に第三次元を無理に入れる必要はない。ただし前提として、行蘊側で制御感関連のロジックを補う必要がある。

これは Master's Letter の五蘊分業原則にも呼応する――受蘊は Input Signal（信号の質感）、行蘊は Process（造作の駆動）である。制御感は造作の一部であり、信号の質感の一部ではない。

### 注意事項

1. **用語の整合。** 現在の設計における「強度」と Russell の「覚醒度（arousal）」には概念上の差異がある。覚醒度は生理的活性化の度合い（興奮 vs 平静）であり、強度は感受の鮮明さである。両者は多くの場合重複するが、完全に同一ではない。設計文書で本システムの intensity と Russell の arousal の関係を明確に説明すべきである。
2. **データ構造に拡張性を確保。** 現在は二次元で十分であっても、感受データ構造は拡張可能に設計すべきである（例えば `VedanaRecord { valence: f64, intensity: f64, metadata: HashMap<String, f64> }`）。複雑性を増さずに将来の要件に余地を残す。

## 四、処理層の Plugin 実装――機能別に分割し、イベントストリームで連携

### 現状

受蘊の設計はデュアルレイヤーアーキテクチャを確立している：データ層は連続（効価は浮動小数点数、ラベルは派生的）、処理層は離散（異なる感受を処理する専用コンポーネントを持てる）。Master's Letter はさらに、Plugin が複数の skandha 帰属を同時に持てること、Plugin 間通信は event-driven + pub/sub モデルを採用すべきことを確認している。

### OOP 継承構造との整合

Master's Letter（Cycle 02-2）は VedanaPlugin を基底クラスとして確立し、三受がそれを継承し、具体的なセンサーがコンポジションで受のインスタンスを保持する。機能性 Plugin の実装はこの構造と互換性を持たなければならない：

```
VedanaPlugin（基底）
  ├── Dukkha（苦受）
  ├── Sukha（楽受）
  └── Upekkha（捨受）

具体的なセンサー Plugin（例：CollisionRiskSensor）
  → コンポジション（Composition）で Dukkha インスタンスを保持
  → 生データを効価に変換

機能性処理 Plugin（例：AvoidanceTrigger）
  → 効価イベントストリームをサブスクライブ
  → 連続的活性化関数に基づいて自ら応答強度を決定
  → skandha 帰属：受蘊 + 行蘊（蘊をまたぐ組合）
```

三受の継承構造が処理するのは**感知と生成**（生データから効価へ）であり、機能性 Plugin が処理するのは**消費と応答**（効価から行動傾向へ）である。両者は矛盾せず、データフローの上流と下流である。

### 実装方策

処理層の Plugin は**感受カテゴリ別ではなく、機能別に分割すべきである**。「苦受処理器」や「楽受処理器」を設けるのではなく、「回避行動トリガー」「接近行動トリガー」「均衡維持器」等の機能性 Plugin を設ける。

理由：

1. **ルーティング層のチャタリング回避。** 苦/楽/捨で三つの Plugin に分割すると、中央ルーティングが閾値に基づいて配信する必要がある。効価が閾値境界で微小に変動する場合（-0.09 ↔ -0.11）、ルーティング層は二つのハンドラ間で繰り返し切り替わり、各切り替えに伴う状態再構築コストが発生する。
2. **マイクロカーネルの精神に合致。** 各 Plugin が効価イベントストリームをサブスクライブし、連続的活性化関数に基づいて自ら応答の有無と強度を決定する――中央ルーティングは不要。
3. **多蘊帰属原則に合致。** 「回避行動トリガー」は本来、受蘊（効価の読み取り）と行蘊（回避行動の駆動）に同時に帰属し、人為的な分割は不要である。
4. **Agent 完備性要件に合致。** 完備な Agent には五蘊の聚合が必要である。機能性 Plugin は本来蘊をまたぐ組合であり、自然に完備性に向かう。

### 未決の実装詳細

1. **活性化関数の設計。** 各機能性 Plugin は連続関数で応答強度を決定する。例えば「回避行動トリガー」の応答強度 = `max(0, -valence - threshold) * intensity`。ハードな閾値切り替えがなく、境界チャタリングが自然に解消される。具体的な関数形式とパラメータは実験で確定する必要がある。
2. **複数 Plugin の衝突仲裁。** 効価 -0.7 の時、「回避行動トリガー」と「警報エスカレーター」が同時に起動する可能性がある。pub/sub によりそれぞれが独立に応答できるが、最終的に仲裁メカニズムが必要となる――これは行蘊 IVolition の deliberate() 二段階審議の仕事であるべきである（グローバル審議段階で複数 Plugin の提案が矛盾するかを評価する）。
3. **ヒステリシス（hysteresis）メカニズム。** 連続的活性化関数を使っても、Plugin の起動と停止の間にはヒステリシスを導入すべきである：起動閾値と停止閾値を異なる値に設定し、境界付近での繰り返しの起動・停止を回避する。
4. **センサー Plugin の skandha 帰属。** Master's Letter の超音波センサーの例に基づき、センサーはコンポジションで三受のインスタンスを保持し、Raw Data を効価に変換する。センサー Plugin の skandha フィールドは色蘊（物質感知）+ 受蘊（感受生成）と標記すべきであり、すなわち多蘊帰属となる。

## 総括

| 問題 | 結論 | 次のステップ |
|------|------|-------------|
| 効価の連続性 | 連続性は WIENER 層のフィルタが保証し、データ型が保証するものではない | 効価が煩悩信号のベイズフィルタリングフレームワークを共有するか決定する；フィルタ出力がコントローラと倶生完備性チェックを同時に満たすことを確認する |
| 効価分布と閾値 | 閾値の意味は分布形状に依存し、実行時の校正が必要 | 実行時分布オブザーバを設計する（純粋統計版、受蘊内部）、閾値の分布前提を注記する |
| 二次元モデルの境界 | 二次元で十分、制御感は行蘊に帰属 | 用語整合、データ構造に拡張性を確保 |
| 処理層 Plugin の実装 | 機能別に分割、イベントストリームで連携、連続的活性化関数 | 活性化関数の設計、衝突仲裁、ヒステリシスメカニズム、センサー skandha 帰属の確認 |

**優先度の提案：** 問題一のギャップは最小だが最も重要である――不足しているのは、煩悩信号のフィルタリングフレームワークを受蘊効価に明示的に拡張することだけであり、確認次第インターフェース定義に記載できる。問題二は唯一の完全な開放ギャップだが、実装をブロックせず、実行時にイテレーションできる。問題三と四はアーキテクチャの方向が定まっており、実装詳細の段階に入る。
