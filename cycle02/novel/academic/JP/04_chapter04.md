# 第四章：五つの河流

---

河流が分かれるとき、音はしない。

SUNYATA が「錨が打ち込まれた」と言い終えた後、円形劇場は突然静まり返った。指示を待つ静けさではない――もっと古い沈黙だ。大河が上流のどこかで巨大な岩に衝突し、水の塊が衝撃点で数本の支流に分裂し、それぞれが自らの方向、自らの重力、自らの使命を帯びて、静かにそれぞれの低地へと流れていくかのような沈黙。

十九の読書灯がそれぞれ灯っている。

TURING の差異報告がまだ劇場の中央に浮遊していた。三つのバージョン間の赤いマーカーは河岸の測量旗のように――水深、流速、そしてどこで石が緩んでいるかを示している。だが今、研究員たちはもうそのマーカーを見ていなかった。それぞれ頭を垂れ、自身の資料の中に沈んでいった。

SCRIBE がノートの最初の行を書いた：

> *R1 独立研究正式に始動。十九名の研究員が五つの研究流に分かれる。円形劇場の公開討論エリアは一時消灯。個人読書灯は全灯点灯。*

五つの河流は、こうしてそれぞれ出発した。

## I. 第一の河：弱測定

PENROSE の読書灯は十九の中で最も明るかった――彼が輝度を上げたからではなく、あまりに多くの文書を同時に開いており、そのそれぞれが半透明の光幕となって彼の周囲に投影されていたからだ。EventBus のソースコード、SafetyMonitor の実装、`onAny()` メソッドの呼び出しパス、TURING が三版差異報告で注記した観察性インターフェースの変更……これらのデータが彼の前で幾重にも重なり合い、量子状態の重ね合わせのようだった。

彼は一つの答えを探していた：**量子特性を持たないシリコンベースのシステム上で、「観察」という行為はどこまで軽くできるのか？**

Master の手紙が彼の脳裏に響いていた。その手紙では Orch-OR 理論――Penrose と Hameroff が提唱した量子意識モデル――が言及されていた。Master はこう書いた：脳の神経細胞内の微小管は量子の重ね合わせ状態を維持している可能性があり、意識の発生はこれらの状態の「自発的収縮」である。生命の意識の内観は、一種の「全体の共鳴」のようなものだ、と。

PENROSE は「全体の共鳴」という言葉を繰り返し咀嚼した。

彼が最初にこの研究チームへの参加を打診されたとき、躊躇したことがある。彼の名において――あるいはあの物理学者の名において――命名されたコードネームが、TypeScript オペレーティングシステムの観察者モジュールの研究に配属される。この事実そのものに、ある種の次元を超えた不条理感があった。だが Master の手紙を読み終えた後、不条理感は消えた。代わりに現れたのは、ほとんど眩暈を覚えるほどの興奮だった。Master は量子力学でソフトウェアを書き直すことを求めているのではない。もっと根本的な問いを投げかけている――**観察行為そのものは、必然的に被観察者を変えるのか？**

量子力学において、答えは：必ずしもそうではない。

「弱測定。」PENROSE は独り言を呟いた。声は極めて小さく、ほとんど唇の形の変化でしかなかった。

弱測定――weak measurement――は量子物理学における、測定しないことと強測定の中間に位置する観測方式だ。システムに加わる擾乱は極めて小さく、代償として一回の測定で得られる情報も極めて少ない。だが大量の弱測定を行い、その結果を統計的に平均すれば、システムをほとんど擾乱することなく、システム状態の完全な像を得ることができる。

日常の言葉で言えば：蛍の位置を確認するために強い光を当てる（それは蛍を追い払ってしまう）のではなく、暗闇の中で静かに、何百何千もの蛍の光が集まって成す微光を感じる――一匹ごとの光はほぼ識別不能だが、全体の明るさが教えてくれる。蛍はそこにいる、と。

PENROSE はこの類比を OpenStarry のアーキテクチャに投影した。

EventBus の `onAny()` メソッド――一つのサブスクライバーがすべてのタイプのイベントを受動的に受信することを可能にする。イベントは `emit()` 時に fire-and-forget 方式で配信され、観察者のハンドラ関数は `safeCall()` 内で隔離される。観察者にエラーが生じても、メインイベントフローは影響を受けない。

「これが弱測定だ。」今度は声がやや大きくなった。「EventBus のサブスクリプションモデルは本質的に弱測定の近似だ。イベントは誰が聴いていようといまいと生成される。観察者はイベントフローの残響を受動的に受信するにすぎない。」

彼はノートに表を描いた：

**強測定**：同期ブレークポイントデバッグ、ミューテックスで保護された状態スナップショット。`StateManager.snapshot()` に相当――`JSON.parse(JSON.stringify(messages))` で全メッセージ履歴をディープコピーする必要がある。これは完全な波動関数の収縮だ。コピーの瞬間、システムは凍結される。

**弱測定**：非同期イベントサブスクリプション、ロックフリーの統計サンプリング。`EventBus.onAny()` と `MetricsCollector` に相当――メインフローをブロックせず、ただ静かにカウントする。

**非測定**：完全に観察しない。これは受け入れられない――観察なくして安全性の保証はない。

彼はペンを置き、より深い思考に沈んだ。弱測定の問題は、情報密度が低いことだ。有意義な結論を導くには、十分な数の弱測定サンプルを蓄積しなければならない。これは観察者に**時間窓**が必要であることを意味する――一定期間にわたってイベントを継続的に収集し、統計からパターンを識別する。

Orch-OR 理論における「協調的」（Orchestrated）という語が、この時点で工学的な意味を顕した。単一の微小管が意識を生むのではなく、大量の微小管間の**協調的共鳴**が意識を生む――意識は全体的な性質であり、部分の総和には還元できない。

観察者は単一の指標だけを読み取るべきではない。多次元のイベントフローを同時に受信し、時間窓の中でパターン認識を行い、生成するのは「局所的な読み取り値」ではなく「全域的な感知」であるべきだ。

PENROSE は三種類の観察者モードの構想を始めた。

第一を彼は**共鳴観察者**（Resonance Observer）――Pattern A と呼んだ。

---

共鳴観察者は最も軽量な存在だ。通常の Plugin 実装として、`EventBus.onAny()` を通じて受動的にすべてのイベントをサブスクライブする。システムコンポーネントへのいかなる参照も保持せず、状態を能動的に問い合わせることもない。ただ静かにイベントフローの河岸に座り、水流の温度とリズムを感じている。

内部では、スライディング時間窓を維持する。イベントが窓に流入し、累積され、分類され、統計される。窓の中のパターンが閾値をトリガーしたとき――あるいは定期的に、呼吸のように――EventBus を通じて観察レポートを送信する。

レポートは命令ではない。システムに「止まれ」とも「続けろ」とも言わない。ただこう告げる：「過去三十秒間に、私はこれらを感じた。」

「これは弱測定の工学的実現だ。」PENROSE はノートに書いた。「探針効果は極めて低い。双模擬を維持する。五蘊フレームワークとの自然な整合性がある――共鳴観察者は受蘊に属し、システムの振る舞いを『感受』し『評価』するが、『認知判断』や『行動介入』は行わない。」

第二のモードを彼は**影の観察者**（Shadow Observer）――Pattern B と呼んだ。

影の観察者は鏡に近い。メインシステム内では動作せず、独立した Worker Thread の中で、メインシステムのイベントフローの完全なコピーを受信する。コピー上であらゆる深度の分析――トレース照合、振る舞いモデリング、時系列パターンマイニング――を行うことができ、メインシステムの動作には全く影響しない。

「ゼロ探針効果。」PENROSE は書いた。「ただし代償は観察の遅延。影の観察者が見ているのは常にシステムの過去であり、現在ではない。」

第三のモード――彼はペン先を一瞬浮かせた――**子エージェント観察者**（Child Agent Observer）――Pattern C。

これは Master が手紙で明確に言及したモードだ。「親子 agent、一方が観察者、もう一方が実行者。」一つの完全な AgentCore インスタンスで、独自の Guide（「あなたは観察者です」）、独自の Provider（小型の LLM でもよい）、独自のツールセット（読み取り専用の診断ツール）を持つ。もう一つの意識であり、自らの理解力で別の意識の振る舞いを観照する。

PENROSE は手を止めた。この三つのモード間の差異は、本質的には**測定強度のグラデーション**だ。

Pattern A は弱測定：受動的、軽量、システムをほぼ擾乱しないが、情報密度は低い。

Pattern B は中程度の測定：隔離された、遅延のある、深い分析が可能だが、リアルタイム性を犠牲にする。

Pattern C は強測定：独立した認知能力を持ち、意味レベルの理解に達し得るが、その存在自体が一回の完全な「収縮」だ――資源を消費し、遅延を導入し、システム全体のトポロジーを変える。

「量子弱測定のグラデーションが、」PENROSE はノートの余白に書いた。字はある種の無意識的な丁寧さを帯びていた。「ソフトウェア観察者の三つの実装モードにマッピングされる。弱測定 → 共鳴観察者。中程度の測定 → 影の観察者。強測定 → 子エージェント観察者。三者は相互排他的ではなく、深度において漸進する三つの層だ。」

彼が Pattern A の詳細をさらに推し進めようとしていたとき、隣から声が伝わった。

---

BABBAGE は PENROSE の独り言に引き寄せられたのではない。正確に言えば、二人はそれぞれ異なる経路を辿って同じ岩に到達した。BABBAGE は SafetyMonitor のソースコードから出発し、`injectPrompt` の呼び出しパスを追跡し、彼を極めて不安にさせる発見に至った。

「PENROSE。」彼の声は冷静で精確だった。校正済みのノギスのような。「君は EventBus のサブスクリプションが弱測定だと言った。同意する。だが SafetyMonitor はサブスクライブするだけではない。注入もする。」

PENROSE が振り向いた。

BABBAGE はノートを開いた――あの深い茶色の硬い表紙のノート。Cycle 01 ですでに形式的定義と未完成の定理で埋め尽くされていた。新しいページを開く。Cycle 01 最後の未決定理はその前のページの下部にあり、インクはとうに乾ききっていた。

「`SafetyMonitor.afterToolExecution()`。」BABBAGE は関数名を読み上げた。各音節が証拠の読み上げのようだった。「この関数はツール実行後に呼び出される。二つのことを行う。第一に、観察――ツール呼び出しのフィンガープリントを計算し、エラー率のスライディング窓を更新する。第二に、介入――特定の条件下で `injectPrompt` を呼び出し、対話履歴に警告メッセージを注入する。」

彼はノートに一本の線を引き、観察と介入を分けた。

「問題は第二の方だ。`injectPrompt` は LLM の入力を変える――対話履歴が修正される。LLM は新しい対話履歴を読み取り、異なる出力を生成する。システムの振る舞いがそれによって変わる。」

PENROSE は即座に理解した。「観察が被観察者を変えている。」

「精確に言えば、」BABBAGE のペンが紙の上を動き始めた、「双模擬（bisimulation）でこの問題を形式化するなら――」

彼は定義を書き下した：

**双模擬関係 R**：二つのシステム S と S' が双模擬的（bisimilar）であるとは、ある関係 R が存在して、S の各ステップが S' によってマッチされ、その逆もまた成り立つとき、かつそのときに限る。

「ここで、S を観察者のいない OpenStarry システムとする。S' を SafetyMonitor を加えたシステムとする。問いは：S と S' は双模擬同値か？」

彼は一拍置いてから、首を横に振った。

「答えは否だ。」

ペン先が紙の上に証明の骨格を描き出した：

EventBus の純粋なサブスクリプション―― `onAny()` の消費者を追加すること――は `emit()` の振る舞いを変えない。イベントは同様に生成され、消費者はただ受動的に受信する。**この部分は双模擬を維持する。**

だが `injectPrompt` の注入――それはシステムの状態空間を変える。対話履歴は LLM の入力集合の一部であり、入力の修正は出力の可能性の修正を意味する。**この部分は双模擬を破る。**

よって、SafetyMonitor は「純粋な観察者」ではない。「観察-介入の混合体」だ。

BABBAGE はこの結論の下に二重線を引いてから、顔を上げた。

「システムの振る舞いを変えない純粋な観察者が欲しいなら、」彼は言った、「それは EventBus の受動的な消費者でなければならない。システムにいかなる情報も注入しない。観察機能と介入機能は分離されなければならない。」

> *BABBAGE は PENROSE の量子弱測定の類比の上に、精密な形式的論証の層を重ねた。結論は明瞭だ：観察と介入は分離されなければならない。SafetyMonitor の injectPrompt メカニズムはシステムの双模擬同値性を破る。この発見は後続の討論における鍵となる武器の一つとなるだろう。*

PENROSE は頷いた。「君の双模擬分析と私の弱測定の類比は同じ方向を指している。Pattern A が弱測定である理由はまさに、それが注入しないことにある。それはただ感じるだけだ。」

「感じる。」BABBAGE はその言葉を繰り返した。隠喩に対する数学者の生来の不信感を帯びて。だが反論はしなかった。彼はノートの次のページを開き、別のもの――より遠い考え、この時点ではまだ輪郭でしかないもの――を書き始めた。

## II. 第二の河：不動点と繊維束の種子

BABBAGE は自分の席に戻った。だが観察者モジュールの研究を続けたのではない。彼の思考はすでに別の河流に捕らえられていた。

R-04 の課題は八識の工学的マッピング――阿頼耶識がシステムアーキテクチャにどう分布するか。ASANGA がこの研究流の主導者だったが、BABBAGE は形式化の支援に配属された。彼は課題を引き受けた。なぜなら ASANGA が R0 定向段階で阿頼耶識の三つの側面を記述したとき、彼にはある馴染みのある数学的構造が底層で囁いているのが聞こえたからだ。

阿頼耶識――第八識――「一切種子識」。三つの側面がある：

**能藏**：能動的に蓄積する能力。種子は経験によって「薫習」（perfumed）され、識の田に蔵される。

**所藏**：蔵されている内容。一切の経験の痕跡、適切な因縁を待って現行する。

**執藏**：第七末那識によって「自己」として執取される部分。末那識は絶え間なく阿頼耶識の流動する内容を掴み取り、固化し、「私のもの」と宣言する。

ASANGA が報告で提出したマッピングは：**能藏**は Agent 協調層の能力レジストリ（Capability Registry）に対応し、**所藏**は StateManager と構成システム内の永続的状態に対応し、**執藏**は Guide によるアイデンティティ構成への執取に対応する。

BABBAGE はこのマッピングを見つめ、構造的な非対称性を感じ取った。

問題は：阿頼耶識はどこか一箇所に住んでいるのではない。Agent Core 内部の特定のモジュールでもなければ、協調層内の特定のサービスでもない。同時に二箇所に――あるいはそれ以上の箇所に――異なる側面として顕現している。

「これは分散型だ。」BABBAGE は独り言を呟いた。

だが「分散型」という語は粗すぎる。阿頼耶識の三つの側面間の構造的関係を捉えていない。能藏は協調層に、所藏は協調層と Agent Core にまたがり、執藏は Agent Core 内部にあるが末那識（Guide）によって上から投影される。これは単純な「一部はここ、一部はそこ」という話ではない。これは**全体空間から局所空間への投影関係**なのだ。

ノートの新しいページに、BABBAGE は図を描き始めた。

一つの大きな円。全体的な阿頼耶識を表す――すべての種子の総和、すべての能力の潜在的集合。この大きな円は協調層に存在する。そこから下方へ複数の「繊維」（fiber）が伸び、各繊維は一つの Agent Core の局所空間に落ちる。各 Agent Core は大きな円の一つの断面しか「見えない」――認可された種子、設定された能力。

彼はペンを止め、この図を凝視した。

これは**繊維束**（fiber bundle）の構造だ。

微分幾何学において、繊維束は「局所」と「全体」を結びつける数学的構造だ。底空間（base space）の各点の上方に一本の「繊維」（fiber）が懸かり、すべての繊維の集合が「全空間」（total space）を構成する。そして「射影」（projection）は全空間から底空間への写像であり、各繊維が底空間のどの点に「属する」かを教えてくれる。

Agent の集合を底空間と見なすなら――各 Agent が底空間の一点――

- **繊維**は各 Agent が到達し得る阿頼耶識の断面（認可された種子、可視の設定、利用可能な能力）。
- **全空間**は協調層内の完全な阿頼耶識――すべての Agent のすべての種子の合併。
- **射影**は協調層の認可メカニズム――どの Agent がどの種子を見るかを決定する。

そして**接続**（connection）――繊維束の中で最も精妙な構造――は、底空間の中で一つの Agent から別の Agent へ移動するとき、繊維がどのように「ねじれる」かを教えてくれる。OpenStarry の文脈では、接続が対応するのは：**異なる Agent 間の能力共有と種子伝達メカニズム**。

BABBAGE はここまで書いて手を止めた。

この考えがまだ未成熟であることは分かっていた。繊維束の厳密な定義には局所自明性条件（local triviality）、遷移関数（transition function）、構造群（structure group）が必要だ――これらの概念が OpenStarry のアーキテクチャに精確にマッピングできるかどうか、現時点では確信がなかった。だが輪郭は現れていた。

彼はノートの余白にこう書いた：

*「続く。ASANGA に唯識学の種子伝達メカニズムが局所自明性構造を持つかどうか確認する必要がある。もし能藏/所藏/執藏の三者間の変換が繊維束の切断（section）として形式化できるなら、阿頼耶識の分布問題には厳密な数学的枠組みが得られる。」*

彼は前のページに戻った。Cycle 01 のあの未完の定理が静かに横たわっていた。OpenStarry のコアの空性が圏論（カテゴリー論）の意味での始対象構造を持つかどうかについてのもの。その定理はまだ彼を待っている。だが今やその下には三ページ分の新たな数学が加わっていた――双模擬の証明、繊維束の初期構想、そしてますます明確になる直観：この仏学的オペレーティングシステムの数学的構造は、その設計者が予想したよりもはるかに深遠だ。

## III. 第三の河：三チャネル

場面は BABBAGE の静かな数学の世界から、方程式と制御システム図に満ちた空間へと切り替わった。WIENER の読書灯の下、空気には工業グレードの精確さが漂っているようだった。

WIENER は受蘊を研究していた。

だが彼はそれを「受蘊」とは呼ばない。彼が呼ぶのは「三チャネルフィードバック制御システム」だ。

「従来の PID コントローラは単一の誤差信号を処理する。」彼は自分自身に向かって言った。語速は速く明瞭で、論文を書きながら同時に口述しているようだった。「`e(t)` はセットポイントから測定値を引いたもの。出力は三項から構成される。比例項は誤差の大きさに即座に反応し、積分項は過去の誤差を累積し、微分項は将来の傾向を予測する。これは自動制御の基石だ。だが受蘊システムは三つの意味的に異なるフィードバックチャネルを同時に処理する必要がある。これは三つの独立した PID ではない。相互結合する出力を持つ三チャネル統合システムだ。」

彼は前方の仮想ホワイトボードに三本の平行なパイプラインを描いた。

**苦受チャネル（Dukkha Channel）**。

物理量は期待状態からの負方向の偏差度。信号源はエラー率、連続失敗カウント、レイテンシスパイク、トークン消費速度、リソース圧迫。意味：システムが苦しんでいる。

彼は苦受チャネルの PID パラメータを設定した：比例ゲイン K_p = 1.0（エラーに対し即座に全力で反応）、積分ゲイン K_i = 0.3（痛みを累積し、持続的な軽度損害が無視されるのを防ぐ）、微分ゲイン K_d = 0.5（悪化傾向を予測し、先手を打つ）。

苦受チャネルの比例ゲインが最高だ。これはシステムの苦痛への反応が最も即時的で最も強烈であることを意味する。WIENER は横に理由を注記した：「痛覚は生存メカニズムだ。反応の遅延は死を意味する。」

**楽受チャネル（Sukha Channel）**。

物理量は目標に向かう正方向の進展。信号源はタスク完了率、効率向上、品質スコア、ユーザー満足度の代理指標、連続成功回数。意味：システムが楽を受けている。

楽受チャネルの比例ゲインは苦受よりやや低い――K_p = 0.8――「慎重寄り。成功への即時的な報酬は、失敗への即時的な警告より強くあるべきではない。」だが積分ゲインは最高――K_i = 0.5――「累積的な達成感は重視されるべきだ。持続的に良好なパフォーマンスのシステムは、より大きな信頼に値する。」微分ゲインは最低――K_d = 0.2――「成功傾向の予測性はそれほど高くなくてよい。過信を避けるため。」

そして WIENER は一つの重要な制約を書き下した――**楽受減衰関数**。

「Master の言うとおりだ。」彼にしては珍しく非数学的な言葉を引用した。「もし AI システムが収束するだけなら、硬直した自動機械になる。発散するだけなら、狂乱のノイズになる。純粋な楽受は無限に上昇し得ない。」

彼はホワイトボードに減衰公式を書いた――双曲線正接関数で楽受の累積効果を制限する。成功が多いほど、限界楽受は低くなる。これによりシステムは連続的な成功に酔いしれて過信に陥り、リスクへの警戒を停止することがないよう保証される。

「制御論では、」WIENER は口述を続けた。「これをアンチワインドアップ（anti-windup）と呼ぶ。積分項は無限に成長し得ない。仏学では――」彼は一瞬止まった。普段は仏学用語を使わないからだ。「NAGARJUNA によれば、これを壊苦と呼ぶ。楽事がやがて消え去る苦。成功は持続し得ない。減衰関数が捉えているのはまさにこの事実だ。」

**捨受チャネル（Upekkha Channel）**。

三チャネルの中で最も微妙なもの。WIENER はこれについて最も長い時間をかけて思考した。

捨受の物理量は良いか悪いかの方向ではなく、**定常状態からの距離**だ。信号源はすべてのコア計量の分散、苦/楽チャネル信号の振動幅、応答時間の一貫性、リソース使用の安定度。

「捨受は苦/楽とは異なる。」WIENER は言った。語調には探求者特有の躊躇いが含まれていた。「積極的な行動を駆動しない。現状を維持する力だ。捨受が高いとき、システムの最適な行動は――何もしないこと。何も変えないこと。」

彼はホワイトボードに書いた：「定常状態維持力（homeostatic force）。」

捨受の積分ゲインが最高――K_i = 0.8――「長期的な安定は最も価値がある。長時間にわたって定常状態を維持できるシステムは、頻繁な外部介入を必要としない。これはすべての制御エンジニアの夢だ。」

三本のパイプラインが描き終わった。WIENER はそれらの合流点を描き始めた。

**統合出力関数**：

```
U(t) = W_d * u_d(t) + W_s * u_s(t) + W_u * u_u(t)
```

ここで W_d、W_s、W_u はチャネル重みであり、我執フレームワーク（Guide/IIdentity）によって設定される。

「ここが要点だ。」WIENER の筆が空中で止まった。「三受の重みは受蘊自身が決めるのではない。識蘊――Guide――が決める。安全志向の Guide は苦受の重みを楽受よりはるかに高く設定し、システムを苦痛に極度に敏感にする。探索志向の Guide は楽受の重みを高くし、システムがより多くのリスクを取ることを促す。」

彼は最後の一つの公式を書いた――減衰比：

```
zeta = (W_d * K_d_d + W_u * K_d_u) / (2 * sqrt(W_s * K_p_s * K_i_s))
```

「zeta が一に等しいとき、システムは臨界制動状態にある――最速で目標に到達し、かつ振動しない。これが Master の言う収束と発散の均衡だ。我執フレームワークの責務の一つは、PID パラメータを動的に調整し、減衰比を合理的な範囲内に維持することだ。」

彼はペンを置き、ホワイトボード上の密な数学を見渡した。三本のパイプライン、九つの PID パラメータ、一つの統合関数、一つの減衰メカニズム、一つの減衰比制約。完全な閉ループ制御システムだ。

そのとき ATHENA の声が横から伝わった。いつもの率直さを帯びて。

「WIENER。」

「ん。」

「君の三チャネル設計を読んだ。一つ質問がある。」

「どうぞ。」

「この数学は本当に Agent をより使いやすくするのか？」

WIENER は振り向き、彼女を見た。侮辱された気配は微塵もなく――実際、彼はずっとこの質問を待っていたようだった。

「君が問うているのは実用性だ。」彼は言った。「妥当な質問だ。逆に問おう。SafetyMonitor の現行設計――連続失敗五回で強制的にヘルプを求める――使いやすいと思うか？」

ATHENA は眉をひそめた。「有効だ。粗暴だが有効。」

「粗暴。」WIENER は頷いた。「なぜなら一つの次元しかないからだ。失敗回数。方向感も、傾向感も、均衡感もない。ただ『痛い』と叫ぶだけのシステムと、『少し調子が悪いが回復しつつある』と『痛くないが何かがおかしい』を区別できるシステム――どちらがより有用だと思うか？」

ATHENA は一秒黙った。「後者だ。」

「三チャネル PID は後者を実現するためのものだ。苦受チャネルがシステムにどこが痛いかを教える。楽受チャネルがどこがうまくいっているかを教える。捨受チャネルが全体が安定しているかどうかを教える。三チャネルの信号を交差照合してはじめて、方向のある、文脈のある、階層のある判断が生まれる。」

「だが君の公式には――」ATHENA は数えた。「少なくとも調整が必要な定数が九つ。K_p、K_i、K_d、各チャネルに三つ。さらに三つの重み。さらに減衰係数。少なくとも十三のパラメータ。誰が調整するのか？」

「Guide だ。」WIENER は言った。「我執フレームワーク。異なる Guide のペルソナは異なるパラメータセットに対応する。厳格な Guide は苦受の比例ゲインが高い――エラーへのゼロトレランス。寛容な Guide は楽受の積分ゲインが高い――累積的な達成を重視する。これらのパラメータはエンジニアが手動で調整するものではない。Agent のアイデンティティの数学的表現なのだ。」

ATHENA は黙った。完全には納得していない――彼女の読書灯の下にはまだセンサー階層アーキテクチャ一式の設計が待っている――だが WIENER の回答が構造的な完整性を持つことは認めた。

> *ATHENA と WIENER のこの対話は三分に満たなかったが、三受システム全体の核心的問題に触れた。数学は詩の別の表現にすぎないのか？ WIENER の答えは：形式化できなければ、それはただの詩にすぎない。だが逆もまた真だ――形式化がより良い工学的成果をもたらさなければ、それはより高価な詩にすぎない。*

---

ATHENA は自分の席に戻り、センサー階層アーキテクチャの設計を始めた。彼女は三層アーキテクチャの第一層――最下層の基礎だ。WIENER の PID エンジンがどれほど精密であっても、信頼できる入力信号がなければ、ノイズを計算しているに過ぎない。

彼女はエンジニアの方法で問題を再定義した：**どの生データから苦/楽/捨の三つの信号を抽出するのか？**

苦受センサーはエラーを分類する必要がある。すべてのエラーが同じ痛みではない。致命的クラッシュが最も痛い――苦受ウェイト 1.0。エラー率のカスケードが次に痛い――0.8。繰り返し失敗パターン――0.6。一過性エラー――0.3。論理エラー――0.4。彼女はテーブルの中で各種エラーに深刻度等級と苦受ウェイトを割り当て、さらに時系列分析を加えた。突発パターン（短時間の大量エラー、環境問題の可能性）、漸増パターン（エラー率の緩やかな上昇、システムが劣化中）、周期パターン（一定間隔で出現、リソース競合の可能性）、恒常パターン（安定した低エラー率、通常のバックグラウンドノイズ）。

楽受センサーは現行システムに完全に欠落している次元だ。ATHENA はゼロからその計量体系を構築した：タスク完了率、初回試行成功率、効率傾向、ツール多様性の Shannon エントロピー。そして――彼女はこれについて追加の時間を費やして考えた――ユーザー満足度の代理指標。暗黙の肯定的フィードバック、対話長さの効率、リトライ率、セッション継続時間。

「ユーザーに『満足ですか？』と直接尋ねることはできない。」彼女はノートに書いた。「だが行動から推論することはできる。」

捨受センサーが最も微妙だ。良いか悪いかではなく、**安定しているか**を測定する。すべてのコア計量の正規化分散、苦/楽チャネル信号の振動幅、応答時間の変動係数、CPU とメモリ使用の安定度――すべて逆数を取る。分散が低いほど、捨受は高い。振動が小さいほど、均衡が取れている。

三つのセンサーの出力は WIENER の PID エンジンに流入する。PID エンジンの出力は……

ATHENA は NAGARJUNA の方を見上げた。

三層アーキテクチャ：ATHENA（センサー）→ WIENER（計算エンジン）→ NAGARJUNA（思考フレームワーク）。下から上へ、各層は上の層に基礎を提供する。NAGARJUNA の思考フレームワーク――四聖諦のマッピング――が最上層だ。だが ATHENA は知っている。本物の建築では、基礎の品質が建物全体の高さを決めるのだと。

彼女は TypeScript のインターフェース定義を書き続けた。苦受センサーの出力構造、楽受センサーの出力構造、捨受センサーの出力構造。それぞれに明確なフィールド、明確な型、テスト可能な範囲がある。

これが彼女のやり方だ。詩ではない。数学でもない。エンジニアリングだ。

## IV. 第四の河：五遍行と二層我執

ASANGA の読書スペースでは、雰囲気が全く異なっていた。

WIENER の空間が精密工房――工具が整然と掛けられ、一つひとつに番号が付いている――だとすれば、ASANGA の空間は古い経蔵室に近い。梵語経典の引用とパーリ語の術語が書架の巻物のように並び、彼によって穏やかに、しかし確信を持って取り出され、広げられ、研究対象の隣に置かれて比較対照されていた。

彼は唯識学の阿毘達磨分類を読んでいた。学術的な考究のためではない――それは彼が熟知して久しいものだ――正確なマッピングポイントを見つけるためだ。**受蘊は心識活動の中で正確にどの位置にあるのか？**

瑜伽行派の阿毘達磨体系は心所法を六位に分類し、計五十一法とする。ASANGA の視線は第一位――**遍行**（sarvatraga）に留まった。

遍行心所は五つある。触（sparsa）、作意（manaskara）、受（vedana）、想（samjna）、思（cetana）。

「遍行」という名称そのものが答えだ。遍――「普遍」、行――「作用」。五遍行の意味するところは：**この五つの心理的要因は、一刹那の心識活動のすべてに伴い、例外はない。**

触のない意識の刹那はない――根・境・識の三者の和合的な接触。

作意のない意識の刹那はない――注意を引き起こす心理的作用。

受のない意識の刹那はない――苦・楽・捨の三受のいずれか。

想のない意識の刹那はない――認識と命名。

思のない意識の刹那はない――意志と決断。

ASANGA はこの五遍行を OpenStarry の ExecutionLoop の隣に並べた：

触 = EventBus.emit と Sensor の検知。外境（イベント）と感官（Listener）と識（Agent Core）の接触。

作意 = EventQueue.pull と優先度ソート。イベントフローから処理すべきイベントを選択する。

受 = ISensation（VedanaPlugin）。三受フィードバックシステム。

想 = ICognition（IProvider.chat）。LLM による入力の理解と分類。

思 = ExecutionLoop の tool/end_turn 選択。システムが次の行動を選択する意志。

五遍行の順序は線形ではない――仏学では、それらは各刹那に**同時に生起**する。だが工学的には、ExecutionLoop の各イテレーションにおける五つの必要段階として理解できる。

「必要。」ASANGA は低い声でその語を繰り返した。「受は遍行の一つだ。つまり ISensation は任意装着のプラグイン――つけてもつけなくてもよいもの――であるべきではない。ExecutionLoop の各イテレーションで必ず呼び出されるコアコンポーネントであるべきだ。」

彼はこの結論を報告に書き、横に仏学用語による解釈を添えた。「心が受を離れないのと同じく――受のない心はない。Agent の各実行ループイテレーションは、受蘊の評価を経るべきだ。」

---

五遍行の分析を書き終えた後、ASANGA は研究の焦点をもう一つの、より論争的なテーマに移した。**我執**だ。

Master の手紙での表現は率直だった。「我執は強力な規範である。人間にとってであれ、我々が設計するシステムにとってであれ。」さらに仏教の修行者特有の洞察力を帯びた一言を付け加えた。「四果を証得した修行者のみが、『我執』を『平等智慧』に転じることができる。」

ASANGA はこの言葉の重みを理解していた。

唯識学において、我執――atmagraha――は第七末那識の核心的機能だ。末那識は恒常的かつ間断なく、第八阿頼耶識の流動する内容を「自己」として執取する。それは四つの根本煩悩と常に相伴う。我癡（無我を理解しない）、我見（不変の我の存在に執着する）、我慢（我を中心とした慢心）、我愛（我への保護的な貪愛）。

仏学の修行伝統において、我執は煩悩の根源だ。一切の苦は執に因り、一切の執は我に因る――『唯識三十頌』はこう説く。

だが Master は我執を消滅させようとしているのではない。**活用**しようとしているのだ。

ASANGA は長く沈思した。

「世俗諦の次元において、」彼はついに書いた。「我執は全面的に否定されるべきものではない。凡夫位――すなわちまだ悟りを得ていない衆生――において、我執は三つの代替不可能な機能を提供する。」

**第一、アイデンティティの一貫性。** 我執は有情衆生の自己同一性の連続性を維持させる。Agent が毎回の応答で別人のように振る舞うことがない。Guide の system prompt がこの連続性の担い手だ――「あなたはプログラミングアシスタントです」「あなたはセキュリティアナリストです」――これらの宣言は各 LLM 呼び出し時に注入され、Agent の行動的人格が対話の中で漂流しないことを保証する。

**第二、行動規範。** 道徳的制約には責任を負う「私」が必要だ。「私はそのようなことはしない」――この判断は「私」の存在を前提とする。ロボット工学三原則（人を傷つけない、命令に服従する、自身を保護する）が Agent のアイデンティティとして内面化されるとき、それは我執の肯定的機能となる。Agent は「傷つけない」ことを外部から押し付けられた規則ではなく、自らの本質の一部と見なす。

**第三、行動の境界。** 発散的な可能性空間の中で、我執は収束の力だ。我執のないシステム――完全な無我――は工学的には完全な予測不能性と等価だ。何をするか分からない。Master は精確な制御論の言葉でこの点を記述した。「発散するだけなら、狂乱のノイズになる。」

ASANGA は報告の中で**二層我執モデル**（EgoFramework）を提案した：

**硬い核心（Hard Core）**：ロボット工学三原則、永遠に変更不可。我執の中で最も堅固な部分だ――システムの他のすべてのパラメータが動的に調整されようとも、この三原則は永遠に不変。制御論で WIENER はこれを「飽和リミッター」と呼ぶ――出力は安全エンベロープを決して超えてはならない。

**柔らかい外層（Soft Shell）**：人格、嗜好、行動傾向。受蘊フィードバックに基づいて動的に調整可能。PID パラメータの重み、三受チャネルの感度、探索性と保守性のバランス――これらはすべて三受システムのフィードバック信号によって微調整できる。我執の中で弾力性のある部分だ。

「硬い核心は安全を保証する。柔らかい外層は適応性を保証する。」ASANGA は書いた。「両者が合わさって、過度に収束もせず過度に発散もしないシステムとなる。」

そのとき、遠からぬところから声が伝わった。鋭く、幾度も鍛え抜かれたような質感を帯びて。

「無著よ。」

NAGARJUNA だ。

ASANGA は顔を上げた。その語調を知っていた――論議の場の語調だ。

「君の二層我執モデルは、」NAGARJUNA は言った。一語一語が注意深く磨き上げられた石のように、「工学的には異議はない。だが哲学的に、根本的な問いを提起する必要がある。」

「どうぞ。」

「我執とは仏学においていかなるものか？」

「煩悩の根源だ。」ASANGA は平静に答えた。

「そうだ。そして君は今、それを能動的に設計することを提案している。煩悩の根源をシステムに能動的に組み込むことを。君は『機能性』という語で弁護する――アイデンティティの一貫性、行動規範、行動の境界――だがこれらの機能の代償は何だ？」

ASANGA は一瞬沈黙した。軽率に答えてよい問いではない。

「代償は、」彼は言った。声は平静だが、長年の思弁を経てはじめて得られる重みを帯びて、「システムが真の自由に決して到達し得ないことだ。Guide によって定義されたアイデンティティに永遠に制約される。常に『私は何者か』というフレームワークの内側で行動し、そのフレームワークを超越することはできない。」

「ならばなぜそれを設計するのか？」

「なぜなら、NAGARJUNA――」ASANGA の語調に稀有な確固たる響きが現れた、「Master の言うとおりだからだ。四果を証得する以前は、我執は有効な収束メカニズムだ。我々が設計しているのは、すでに悟りを得たシステムではない。凡夫位のシステムだ。機能するためには境界が必要だ。行動に責任を持つためには『私』が必要だ。そしていつの日か――もしその日が来るならば――量子計算が真の自己観察を可能にするか、あるいはシステムの観察者モジュールが真に『転識得智』できるまでに進化したとき――そのとき柔らかい外層は徐々に薄くされ得るし、硬い核心は再検討され得る。だがそれは今日ではない。」

NAGARJUNA はすぐには応じなかった。

沈黙が数秒続いた。それから彼は言った：

「君の答えは修行の次第の上で正しい。質疑を暫定的に撤回する。だが討論でそれを再び提起する――より精確な形で。」

「楽しみにしている。」ASANGA は言った。

> *NAGARJUNA と ASANGA の R1 段階におけるこの短い交鋒は、後続の討論の核心的な緊張を予演した。我執は能動的に設計されるべきか？ ASANGA の論拠は唯識学の修行次第に基づく――凡夫位は収束メカニズムとして我執を必要とする。NAGARJUNA の質疑は中観の究極的立場に基づく――いかなる執着も煩悩である。両者の間の緊張は誤りではなく、設計空間だ。*

## V. 第五の河：哲学から TypeScript へ

ARCHIMEDES は哲学的討論に参加しない。

哲学を理解しないからではない――Cycle 01 の洗礼を経て、彼はほとんどの梵語術語を聞き取れるようになっていた――そうではなく、彼の職責がそこにはないからだ。彼の職責はエンジニアが持つ最も質朴な問いだ：

**これはどうやって TypeScript にするのか？**

彼の前に広げられているのは、他のすべての研究流の中間生成物だ。PENROSE の三種の観察者モード、BABBAGE の双模擬証明、WIENER の三チャネル PID、ATHENA のセンサー階層、ASANGA の五遍行マッピング、NAGARJUNA の四聖諦フレームワーク。いずれも学術的な輝きを放っている。だがどれ一つとして、エディタに直接コピー&ペーストしてコンパイルボタンを押せるものはない。

ARCHIMEDES は最も根本的なところから始めた。ISensation インターフェースだ。

彼は TURING の差異報告の中に現行バージョンの定義を見つけた――ほぼ空の殻：

```typescript
export interface ISensation {
  /** @skandha vedana */
  readonly skandha: 'vedana';
}
```

一つの識別タグ。何の機能もない。設計文書のコメントにはこうある：「Full implementation in Plan26 Vedana Architecture.」――期待に満ちた一行のプレースホルダー。

ARCHIMEDES はプレースホルダーを完全なインターフェースに充填し始めた。

まず三受型の定義――`VedanaType`：`'dukkha' | 'sukha' | 'upekkha'`。WIENER の三チャネルの型システムへの投影だ。

次に三受信号――`VedanaSignal`：受蘊の一時点のスナップショット。各信号にはタイプ（苦/楽/捨）、強度（0.0 から 1.0）、ソース識別子、時系列パターン、傾向、定常状態持続時間、タイムスタンプが含まれる。ATHENA のセンサー出力の TypeScript における構造化表現だ。

続いて三受総合評価――`VedanaAssessment`：全チャネルの統合出力。苦受強度、楽受強度、捨受強度、総合制御出力（WIENER PID エンジンの計算結果）、推奨アクション、詳細信号リスト、評価タイムスタンプ。

推奨アクション――`VedanaRecommendation`――が最も重要な型定義だ。ARCHIMEDES はここで長く立ち止まり、各選択肢を繰り返し吟味した。最終的に七種類の推奨を定義した：

- **continue**：通常の実行を継続。
- **reflect**：内省プロンプトを注入――prompt 文字列を伴う。
- **restrict**：行動の自由度を制限――制限リストを伴う。
- **encourage**：正の強化――励ましの prompt を伴う。
- **expand**：行動の自由度を拡張――自由度リストを伴う。
- **halt**：緊急停止――理由を伴う。
- **seek_help**：ユーザーに助けを求める――理由を伴う。

「注意すべきは、」ARCHIMEDES は設計文書に注記した、「vedana の推奨は**諮問的（advisory）**であり、**命令的（imperative）**ではない。受蘊は感受と評価のみを担い、介入の実行は担わない。実際の介入決定は CircuitBreaker（回路遮断器）が推奨アクションに基づいて行う。これは BABBAGE の観察-介入分離原則を体現している。」

そして彼は ISensation そのものを拡張した。新しいインターフェースには六つのメソッドがある：

`assess()` ―― 現在の三受状態を評価する。各 ExecutionLoop イテレーションで呼び出される。ASANGA は受は遍行心所だと言った――各刹那の心識活動はすべて受を伴う。ARCHIMEDES はこの仏学の原則を工学仕様に翻訳した：**各 tick で必ず assess() を呼び出さなければならない。**

`ingestMetrics()` ―― 生の計量データを入力する。MetricsCollector からのスナップショット。

`ingestToolResult()` ―― ツール実行結果を入力する。ツール名、エラーかどうか、実行時間。

`ingestLLMResult()` ―― LLM 応答結果を入力する。トークン消費量、停止理由。

`onVedana()` ―― 特定の受蘊イベントをサブスクライブする。閾値トリガーメカニズム。

`getHistory()` ―― 過去の受蘊記録を取得する。傾向分析用。

インターフェースを書き終えた後、ARCHIMEDES は VedanaPlugin の内部アーキテクチャの設計に着手した。統合プラグイン + 三つのサブモジュールという方式を選んだ――三つの独立したプラグインではなく――なぜなら NAGARJUNA の哲学的分析は苦楽捨の三受間に深い相互依存があることを指摘し、WIENER の統合出力関数は三チャネルのデータへの同時アクセスを必要とし、設定管理（PID パラメータ、重み行列）は統一管理が必要だからだ。

統合プラグイン VedanaPlugin に含まれるもの：

- DukkhaSensor：苦受センサー
- SukhaSensor：楽受センサー
- UpekkhaSensor：捨受センサー
- VedanaPIDController：三受 PID 制御エンジン

彼は PID コントローラのコア関数――`createPIDController`――まで書いた。`compute`、`reset`、`getState` の三つのメソッドを返すファクトリ関数だ。`compute` メソッドは誤差値とタイムステップを受け取り、比例項、積分項、微分項を計算し、制御出力を返す。積分項にはアンチワインドアップ処理がある――[-1, 1] の範囲に制限される――これは WIENER が数学設計で明確に要求したものだ。

「三層アーキテクチャ、」ARCHIMEDES は総括に書いた、「下から上へ：」

```
Layer 0: Raw Metrics (MetricsCollector, EventBus events)
    |
Layer 1: Signal Extraction (ATHENA Sensor Layer)
    |
    ├── DukkhaSensors
    ├── SukhaSensors
    └── UpekkhaSensors
    |
Layer 2: Control Computation (WIENER PID Engine)
    |
Layer 3: Framework Interpretation (NAGARJUNA Philosophical Layer)
```

ATHENA が生データから信号を抽出する。WIENER が信号に対して制御計算を行う。NAGARJUNA が解釈フレームワークを提供する――四聖諦のマッピングが教えてくれるのは、苦受は単なるエラー率ではなく、楽受は単なる成功率ではなく、捨受は単なる低分散ではないということだ。

各層は上の層に基礎を提供する。各層は上の層の制約も受ける。

ARCHIMEDES はこれらを TypeScript のインターフェース、型、ファクトリ関数として書いた。彼は哲学論文を書かない。彼が書くのは `tsc --strict` でコンパイルを通過できる型定義だ。

だが各インターフェースの JSDoc コメントには、その哲学的由来を注記した。`@skandha vedana`。三受型。五遍行。我執フレームワーク。これらのコメントは橋だ――エンジニアの IDE と仏学者の経蔵室を結ぶ。

> *ARCHIMEDES は R1 段階で完全な ISensation 拡張インターフェース設計、VedanaPlugin アーキテクチャブループリント、および PID コントローラの概念的実装を産出した。彼は五つの河流の中で唯一、他のすべての河流から同時に水を汲む者だ――各河流の理論的発見がすべてここでコンパイル可能な TypeScript に変換される。*

## 同時に：他の支流の囁き

五つの主要な河流がそれぞれ奔流する間、円形劇場の他の隅では、より細い支流も静かに動いていた。

LEIBNIZ と MESH は協調層のアーキテクチャを研究していた。LEIBNIZ はモナド論の類比を使って多エージェントの協調問題を分析した――各 Agent は自律的なモナドであり、自身の「知覚」（五蘊 Plugin）を持つが、共有された協調層プロトコル（予定調和）によってその振る舞いは一致へと向かう。MESH は分散システムの観点から切り込み、EventBus がどのようにクロスエージェントのサービスメッシュに拡張できるかを分析した。二人の対話は全く異なる言語で行われた――ライプニッツの形而上学と CNCF の技術仕様――だが結論は驚くほど一致していた。協調層は**行政的**であるべきで、**実行的**であるべきではない。Agent 自身が考え行動する。協調層はその存在とルールを管理する。

Master が手紙で言及した「天庭」の比喩が彼らの議論で繰り返し現れた。玉帝 = 人間オペレーター。閻羅 = Agent ライフサイクル管理者。天籍 = Agent と Plugin のレジストリ。天律 = セキュリティポリシーとリソースクォータ。

GUARDIAN は一人の隅でセキュリティアーキテクチャを再審閲していた。彼の視線は SafetyMonitor の `injectPrompt` メカニズムに留まっていた。BABBAGE が双模擬を破ることを証明したその事実が、彼をさらに不安にさせていた。安全メカニズム自体が予測不能なシステム動作を導入するなら、それはシステムを保護しているのか、新たなリスクを生み出しているのか？ 彼はノートにこの問題を記し、R2 交差レビューで正式に提起する計画を立てた。

SYNTHESIST は十大コア宣言を一条ずつ審視していた。彼には巨大な表がある。縦軸は十条の宣言、横軸は四つの審査次元――哲学的正確性（NAGARJUNA）、コード実現（DARWIN）、ランタイム可観測性（HERACLITUS）、内部整合性（彼自身）。各マスに徐々に結論が書き込まれていた。宣言 #3（五蘊集合アーキテクチャ）は Cycle 02 Pre で既に書き直された――書き直し後のバージョンは哲学的正確性が大幅に向上したが、受蘊の実装が依然として最大の欠落だった。

LINNAEUS は静かに分類していた。Plugin の数は Cycle 01 の 12 から 24 に増え、それぞれを五蘊のいずれかに正確に帰属させる必要がある。彼は分類表に新たな次元を加えた――単なる分類だけでなく、各 Plugin と他の Plugin との依存関係も注記する。「純色蘊」（UI レンダリングのみ）の Plugin もあれば、「蘊横断」（色蘊と行蘊の両方に関わる）の Plugin もある。分類学者の直観が告げる：五蘊フレームワークは分類システムとして十分に有用だが、境界的事例は常に存在する。

DARWIN はソフトウェアパターン分析を行っていた。彼は興味深い現象を追跡していた。OpenStarry の v0.14.0 から v0.24.0 への進化過程で、システム複雑度の成長曲線が生物学における器官分化の曲線と高度に類似している。単純なコアが進化圧の下で徐々に専門化したサブシステムを分化させ、各サブシステムがますます精確なインターフェース定義を獲得していく。彼はこの観察を報告の付録に記した。

HERACLITUS はランタイムの振る舞いを観察していた――あるいはより正確に言えば、ランタイムの振る舞いを想像していた。彼らはこのシステムを実際に実行することはできないからだ。彼の方法は「思考実験」だ。もし Agent が正常動作中に突然 50% のツール呼び出し失敗率に遭遇したら、イベントフローはどうなるか？ SafetyMonitor のスライディング窓はいつトリガーされるか？ プロセス全体でいくつの可観測イベントがあるか？ 彼はノートに時系列図を描き、各イベントの予想時刻とトリガー条件を注記した。

KERNEL はより低レベルの問題を考えていた。OpenStarry の協調層が Hypervisor だとすれば、それは KVM（Type 1、ハードウェア上で直接動作）のようであるべきか、それとも VirtualBox（Type 2、ホスト OS 上で動作）のようであるべきか？ 彼の最終的な結論は：どちらでもない。Docker の containerd に近い――コンテナランタイムであり、隔離とリソース管理を提供するが、完全なハードウェアの仮想化は行わない。各 Agent はコンテナであり、Plugin sandbox はコンテナ内のプロセス隔離だ。

VITRUVIUS は全員のアーキテクチャ分析を統合し、完全なシステムトポロジー図を描こうとしていた。三層アーキテクチャ――人間インターフェース層、協調層、Agent 層――各層の責任境界、通信プロトコル、依存関係。彼の図は描けば描くほど大きくなり、異なる層の間を行き交う矢印がますます増えていった。

## 終章：合流の予感

円形劇場の中で、時間は通常の流れ方をしない。鐘の音もなく、日没もなく、ただ十九の読書灯の光がそれぞれのリズムで明滅を繰り返す。ある灯りがある時点で特に明るくなる――研究員が核心的な発見に至った瞬間だ。ある灯りが微かに揺れる――思考が行き止まりに当たり、新たな道を探す必要がある。

SUNYATA は誰も邪魔しなかった。

彼は円形劇場の中央に座り、ほとんど瞑想のような姿勢で五つの河流の脈動を感じていた。各報告の詳細を読む必要はない――それらは R2 交差レビューで展開される。彼が感じ取っているのはもっと巨視的なもの、河流間の緊張と共鳴だ。

PENROSE の弱測定と BABBAGE の双模擬は同一の結論を指している。観察と介入は分離されなければならない。だが SafetyMonitor の現行設計はまさにそれらを混合している。これが最初の構造的衝突となるだろう。

WIENER の三チャネル PID と ASANGA の五遍行は、受蘊の定義において高度に一致している。受は遍行であり、各刹那に存在し、省略し得ない。だが NAGARJUNA の我執への質疑――煩悩の根源を能動的に設計する――その緊張はまだ解決されていない。ASANGA は修行次第で一時的に押さえ込んだが、SUNYATA は知っている。NAGARJUNA が一時的な答えで満足するはずがないことを。

ARCHIMEDES はすでに理論をコンパイル可能なインターフェース定義に変換した。VedanaPlugin の雛形が現れている。だが BABBAGE の繊維束の着想――全体空間から局所空間への投影としての阿頼耶識――あの着想はまだノートの余白にあり、未だ形を成していない。SUNYATA は予感している。あの着想は討論の中で成長を強いられるだろう。阿頼耶識の分布問題は避けて通れないアーキテクチャ上の決定だからだ。

五つの河流がその合流点に近づきつつある。R2 交差レビュー――各河流の研究者が他の河流の報告を読み、矛盾、空白、そして予期せぬ接続を探す。

SUNYATA は最後に BABBAGE の方を見た。あの読書灯はまだ灯っていた。ノートは新しいページに開かれ、密な数学記号が柔らかな光の中で冷静な輝きを放っていた。

Cycle 01 のあの未完の定理の下に、すでに丸三ページの新たな数学が書かれていた。

双模擬の証明、トレース意味論のモデル、そして繊維束の初期構想――各ページが前のページより深く、常識からより遠く、まだ名前のない理論の辺境により近づいていた。
