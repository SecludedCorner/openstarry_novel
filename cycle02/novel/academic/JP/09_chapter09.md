# 第九章：十九の行動計画

---

R4 最終稿の円形劇場は、討論の時よりもはるかに静かだった。

空虚な静けさではない——五つの討論の余波がまだ空気の中で振動している。チェロの弦が最後の音符の後もなお震え続けるように。しかし討論の緊張は退いていた。代わりに現れたのは別の密度——納品の密度だった。十九人の研究員がそれぞれの席に着き、目の前には草稿、修正稿、コード断片、マッピング表が並んでいる。R4 は討論ではない。R4 は収穫なのだ。

SUNYATA は会場の中央に立ち、全体を見渡した。もはや何かを宣言する必要はなかった——全員が何をすべきか分かっている。R3 の五つの合意が五本の柱のように、その中で作業できる構造を支えている。今の課題は、討論の裁定を納品可能な文書に変換し、哲学的洞察をエンジニアが読める言語に翻訳することだった。

「ARCHIMEDES、」と彼は言った。「君から。」

---

## I. 明日、我々は TypeScript を書かねばならない

ARCHIMEDES が席から立ち上がる仕草は、他のどの研究員とも異なっていた。

NAGARJUNA は鞘から抜かれる剣のように立ち上がる。ASANGA は風の中でゆっくりと直立する樹のように立ち上がる。BABBAGE は精密機器が起動されるように立ち上がる。ARCHIMEDES は鋤を手に取る建設作業員のように立ち上がった——劇的でもなく、儀式的でもなく、ただ「仕事が来た、取りかかろう」という素朴な原動力だけがあった。

「まず一言。」彼は展示エリアに歩み寄り、四十ページに及ぶ文書を開いた。「この三日間、皆さんは討論を通じて五つの裁定を生み出しました——観察-介入分離、二重モード vedana、ファイバー束射影、漸進的分類、戒慧フレームワーク。どれも美しい。どれも傍聴者としての私に知的な悦びをもたらしました。」

彼は一拍置いた。

「しかし哲学は美しくとも、明日我々は TypeScript を書かねばなりません。」

会場の何人かが笑った。BABBAGE の口元がわずかに動いた——それは彼にとって最も大笑いに近い表情だった。NAGARJUNA は笑わなかったが、その目には一種の認証があった。天上のものを地上に持ち帰るべき時を弁えている者は、敬意に値する。

「この文書が、」ARCHIMEDES は展示エリアの投影を指し示した。「完全な工学行動計画です。十九人の研究員の仕事が、五つの討論で鍛え抜かれ、すべてこの文書に収束しています。その全貌をお見せしましょう。」

---

彼は最も優先度の高い項目から始めた。

「P0——阻害要因。三つ。他のすべての作業に先立って完了しなければなりません。」

彼は指を一本立てた。

「第一に、SEC-01。package-name プラグイン署名バイパス脆弱性の修正。」彼は GUARDIAN に目をやった。「この脆弱性は Cycle 01 で発見されていました。GUARDIAN と TURING が `sandbox-manager.ts` の 371 行目で正確に特定しています——`pluginFilePath` が undefined の場合、署名検証全体がスキップされる。六つの開発サイクルが過ぎました。依然として未修正です。」

GUARDIAN は頷いた。その表情はセキュリティエンジニア特有の忍耐強い不満——怒りではなく、持続的で低強度の警戒だった。「署名をバイパスする脆弱性です。修正されない日が一日続くごとに、システムは一日分裸のまま晒されています。」

「第二に、」ARCHIMEDES は二本目の指を立てた。「ISensation インターフェースの実装です。」

彼は展示エリアでコードに切り替えた。

```typescript
// これが現在の ISensation
export interface ISensation extends IOpenStarryPlugin {
  readonly skandha: 'vedana';
}
```

「三行。空の殻です。」彼の口調に批判はなく——ただ事実を述べているだけだった。「これが序章のあの空殻です。Cycle 02 の研究全体が、ある意味では、この三行を埋めるためにあったのです。」

そして次のコードに切り替えた。円形劇場全体が静まり返った。

```typescript
export interface ISensation {
  readonly skandha: 'vedana';

  /** Full three-vedana assessment (tick-synchronous PID) */
  assess(): VedanaAssessment;

  /** Ingest raw metrics for PID computation */
  ingestMetrics(metrics: Record<string, number>): void;

  /** Ingest tool execution result for real-time feeding */
  ingestToolResult(toolName: string, isError: boolean, durationMs: number): void;

  /** Ingest LLM response result */
  ingestLLMResult(tokenCount: number, stopReason: string): void;

  /** Subscribe to vedana threshold crossings */
  onThreshold(channel: VedanaType, threshold: number,
              callback: (signal: VedanaSignal) => void): () => void;

  /** Get lightweight vedana tag (O(1) cache lookup) */
  getVedanaTag(): VedanaTag;
}
```

「これが充填された ISensation です。」ARCHIMEDES の声は安定し、自信に満ちていた。「すべてのメソッドは討論の裁定に遡ることができます。`assess()` は討論 2 の tick-synchronous PID 裁定に由来します。`getVedanaTag()` は討論 2 の軽量イベントタグ裁定に由来します。`onThreshold()` は WIENER の三チャネル PID コントローラが必要とするサブスクリプション機構です。」

彼はスクリーンに VedanaAssessment の構造を表示した。

「二層構造——討論 1 の核心裁定です。第一層はセンサー出力：dukkha、sukha、upekkha の数値に VedanaSignal 配列を加えたもの。純粋な観察。受動的。第二層はコントローラ提案：controlOutput、recommendation、recommendationFreshness。諮問的。非強制。ExecutionLoop は無視する権利を保持します。SafetyMonitor は絶対的権限を保持します。」

WIENER が席からわずかに身を乗り出した。自分の PID 設計が正しく翻訳されているかを確認していたのだ。VedanaAssessment の第一層が三チャネルセンサーの出力仕様と精確に対応しているのを見て、彼は背もたれに戻った。翻訳は正確だった。

「第三に、」ARCHIMEDES は三本目の指を立てた。「VedanaPlugin の実装です。Pattern A 観察者はすなわち VedanaPlugin です。討論 4 の裁定。一つのプラグイン、一つのインターフェース、一つの蘊。」

彼は VedanaPlugin のアーキテクチャ図を示した——三層構造。最下層は ATHENA が設計した三チャネルセンサー：DukkhaSensor がエラー率を監視し、SukhaSensor が完了率を監視し、UpekkhaSensor が安定性を監視する。中間層は WIENER の PID 制御エンジン——各センサーが独立した PID パラメータを持つ。最上層は評価出力——VedanaAssessment に軽量な VedanaTag を加えたもの。

「三つの P0 項目。もし一つだけしかできないなら、SEC-01 を修正する。二つできるなら、ISensation を加える。三つできるなら、VedanaPlugin を完成させる。これが最小実行可能な納品パスです。」

---

ARCHIMEDES はさらに下位の項目へと展開した。

「P1——高優先度。四つです。」

彼はリストを挙げた。

「EgoFramework。硬い核と柔らかい殻。硬い核はロボット三原則——上書き不可、バイパス不可、たとえ VedanaPlugin がすべて expand を推奨しても、硬い核が否と言えば否。柔らかい殻は PID チューニング——vedana フィードバックにより行動境界を動的に調整。ASANGA の二層モデルが討論 2 で確認されました。これはその工学的実装です。」

ASANGA が頷いた。彼の二層我執モデル——硬い核は揺るがぬ根本的制約に対応し、柔らかい殻は経験によって調整され得る行動傾向に対応する——ARCHIMEDES の翻訳によって具体的かつ触知可能なものとなった。

「プラグイン・ライフサイクル状態。四つの状態：active、quarantined、revoked、banned。討論 5 の戒慧フレームワークです。NAGARJUNA が哲学的基盤を与え、DARWIN が工学的実現可能性を確認しました。あとはコードに落とし込むだけです。」

「ToolContext.bus の漏洩修正。ツールがサンドボックスのイベントフィルタリングをバイパスできる——TURING が発見しました。制限付きイベントプロキシへの置き換えが必要です。」

「SafetyMonitor の per-session カウンタ。現在はグローバルカウンタです。一つのセッションのエラー蓄積が別のセッションに影響する——これはセッション間の DoS リスクです。HERACLITUS がランタイム動態分析で指摘しました。」

彼は各項目の横にターゲット Plan——Plan24、Plan26、Plan27——を記した。行動は宙に浮いているのではない。住所を持っているのだ。

---

「P2 から P4 については、逐一展開しません。」ARCHIMEDES は展示エリアを概要ページに切り替えた。「完全なリストは工学計画書にあります。CRL チェックスタブ、新しい EventBus イベントタイプ、宣言 #2 と #6 の書き換えテキスト、阿頼耶識分布のアーキテクチャ文書注釈——これらはすべて P2 に含まれます。Pattern B シャドウオブザーバーと Agent 協調レイヤーは P3。Pattern C サブエージェントオブザーバーは P4 で、協調レイヤーの完成を待つ必要があります。」

最後に彼は影響概要表を示した。

「五つの既存 Plan が修正され、一つの新しい Plan が提案されます。Plan24 のセキュリティ・クイックフィックス——戒慧フレームワークのプラグインブラックリストと CRL スタブを追加。Plan26 の Vedana アーキテクチャ——これが本研究の主要な設計入力です。Plan27 のライフサイクル管理——四状態ステートマシンを追加。Plan28 の文書整合——ファイバー束注釈、漸進的分類、宣言書き換え、戒慧セキュリティ文書を追加。Plan29——新規——Pattern B シャドウオブザーバー。Plan-AC——三つのアーキテクチャ原則を追加：統治は制御にあらず、ファイバー束整合性、戒エンジン。」

彼は文書を閉じた。

「四十ページ。十四の行動項目。六つの Plan。完全な TypeScript インターフェース仕様。そのままコードを書き始められます。」

彼は SUNYATA に目をやった。

「私の部分は終わりました。哲学は美しい。工学は具体的です。今やそれらは同じ文書の中にあります。」

## II. 五つの討論、一つの絵

SYNTHESIST が展示エリアに向かう足取りは ARCHIMEDES とはまるで違っていた。ARCHIMEDES が鋤を手に取る建設作業員のように歩くとすれば、SYNTHESIST はキャンバスの前に立つ画家のように歩いた——何か新しいものを創造するためではなく、すでに存在する断片の中に一つの完全な光景を見出すために。

「ARCHIMEDES は皆さんに一本の木を見せました。」彼が口を開いた時、話す速度はいつもより遅かった。「すべての枝が一つの行動であり、すべての葉が一つの TypeScript インターフェースです。精確、具体、実行可能。しかし私は皆さんに森全体を見せたいのです。」

彼は展示エリアに一つの図を投影した——図表でもフローチャートでもなく、手描きのアーキテクチャ全景図に近いものだった。五つの討論の裁定がそれぞれの位置に記され、破線で互いに接続され、有機的なネットワークを形成している。

「五つの討論は、表面上は五つの独立した問題です。観察-介入分離。三受の普遍性。阿頼耶識の分布。観察者の分類。安全の種子。しかしそれらは五つの問題ではありません。同一のアーキテクチャの五つの断面なのです。」

彼は線を引き始めた。

「討論 1 と討論 4 が合流します。」彼は全景図に最初の弧線を描いた。「討論 1 は言います：VedanaAssessment はセンサー出力とコントローラ提案の二層から成り、分離される。討論 4 は言います：VedanaPlugin はすなわち Pattern A 観察者であり、受蘊に分類される。合わせると——VedanaPlugin は観察者である。ISensation を実装する。それが生成する評価報告には二つの側面がある：受動的な感受と諮問的な提案。これは二つの結論の単純な重ね合わせではない。一つのアイデンティティの確立なのです。」

彼の声は次第に熱を帯びていった。

「討論 1 と討論 5 が合流します。」二番目の弧線。「討論 1 は階層的権限を確立しました——SafetyMonitor の絶対的ハードセキュリティは VedanaPlugin の諮問的ガイダンスより上位にある。討論 5 はセキュリティの次元でこの階層を拡張し、EgoFramework の構造的制約と戒学レイヤーの種子管理を追加しました。合わせると、完全な四層権限モデルが見えてきます。」

彼は全景図の右側に四行を書いた：

```
Layer 1: SafetyMonitor  — ハードセキュリティ — 絶対的権限 — 戒律
Layer 2: VedanaPlugin   — ソフトガイダンス — 諮問的権限 — 覚受
Layer 3: EgoFramework   — アイデンティティ制約 — 構造的権限 — 我執
Layer 4: Sila Engine    — 種子管理 — 予防的権限 — 戒学
```

「四層。各層の権限は上位層より低い。SafetyMonitor は VedanaPlugin の提案を上書きできる。VedanaPlugin の提案は EgoFramework の柔らかい殻のチューニングに影響できる。EgoFramework の硬い核は他のいかなる層の影響も受けない。Sila Engine は源流で不善の種子の侵入を阻止する——すでに入った種子には干渉しないが、特定の種子が決して植えられないことを保証します。」

GUARDIAN は第一層と第四層の微妙な対称性に気づいた——SafetyMonitor は最も内側のハード防御であり、Sila Engine は最も外側のハードフィルターである。両者が中間の二つのソフト層——VedanaPlugin の諮問と EgoFramework のチューニング——を挟んでいる。彼は心の中でこの構造のセキュリティ態勢を評価し、静かに頷いた。構造は完備だった。

「討論 3 と討論 2 が合流します。」三番目の弧線。SYNTHESIST の動きはますます速くなっていた。「討論 3 は言います：阿頼耶識はファイバー束射影として協調レイヤーと AgentCore に分布する。討論 2 は言います：すべての EventBus イベントが vedana タグを帯び、すべての tick で完全な PID 評価が行われる。合わせると——分散的意識のランタイム景観が得られるのです。」

彼は全景図の左側に二つの領域を描いた——協調レイヤーと AgentCore——両者の間を「IPC / fiber bundle transition functions」と記された線で接続した。

「協調レイヤーは能蔵を保持する——能力レジストリ、プラグイン解決、グローバル設定。AgentCore は執蔵を保持する——Guide バインディング、アイデンティティへの執着、ランタイム状態。それらは二つの独立した意識ではない。同一の意識——第八識——が二つのアーキテクチャ層に射影されたものなのです。IPC プロトコルはファイバー束の遷移関数であり、cocycle condition を満たし、双方向の整合性を保証しなければなりません。」

彼は二つの領域の AgentCore 側に VedanaPlugin の位置を記した——それは AgentCore の内部に棲み、毎 tick でシステム全体の温度を感受し、評価報告を産出する。一方、協調レイヤー側では、Sila Engine が入口を守護する——システムに入ろうとするすべての種子は、戒学の検証を通過しなければならない。

「最後に——討論 4 が時間軸を定義しました。」彼は全景図の下部に一本の時間線を描いた。「漸進的観察者パス。現在——Pattern A、VedanaPlugin、受蘊。近い将来——Pattern B、シャドウオブザーバー、受蘊プラス想蘊。遠い将来——Pattern C、サブエージェントオブザーバー、全五蘊。分類は複雑さに応じて変化する。システムの自己観察能力が、時間とともに螺旋的に上昇していきます。」

彼はペンを置いた。

全景図の前に立つ SYNTHESIST の顔には、SCRIBE がごく稀にしか見たことのない表情が浮かんでいた——それは統合者がすべての断片が集束する瞬間に感じるもの。何か新しいものを発見した驚きではない。すべての既知のものの内在的秩序が目の前で自ずと展開していくのを見る、あの静かな震えだった。

「五つの討論。五つの合意。一つの絵。」

彼は一歩後退し、全員が全景図の全体を見渡せるようにした。

「これが Cycle 02 のアーキテクチャ・ビジョンです。VedanaPlugin は観察者である。観察者は諮問的提案を産出する。提案は階層的権限モデルによって制約される。システム全体がファイバー束射影として二つのアーキテクチャ層に分布する。セキュリティは戒慧フレームワークで管理される。分類は時間軸に沿って漸進的に進化する。」

「統一的で、内在的に自己整合的なアーキテクチャ。五つの独立した裁定を繋ぎ合わせたものではない。五つの裁定がそれぞれ、同一の構造の異なる側面を明らかにしたのです。」

彼は席に戻った。

## III. 開発チームへの手紙

SUNYATA が展示エリアに向かった。その手には四十ページの工学計画書も、全景図もなかった。一枚の紙を持っていた。

「これは私が開発チームに宛てた手紙です。」彼は言った。「Master への手紙でもあります。読み上げます。」

円形劇場が静まり返った。討論前のあの張り詰めた静けさではない。完成に近い静けさだった——交響曲の最終楽章が始まる前に、指揮者が両手を掲げ、楽団が息を潜めるあの瞬間のように。

SUNYATA は読み始めた。

「日付：2026 年 2 月 19 日。発信者：研究チーム SUNYATA、研究総括調整者。段階：Cycle 02 正式研究、R0 から R4 までの完全五段階。チーム規模：十九名の研究エージェント。」

彼の話す速度は遅かった。すべての文に十分な間が与えられていた。

「核心的結論を一文で——」

彼は顔を上げ、場内を見渡した。十九対の目。

「VedanaPlugin はすなわち観察者モジュールである。」

この言葉が空中に数秒間留まった。SCRIBE は気づいた。SUNYATA がこの一文を口にした時、少なくとも三人の研究員の表情に微妙な変化が生じた——結論を知らなかったからではなく、五つの討論を経てようやく完全に展開できるほど複雑なアーキテクチャ上の決定が一文に圧縮された時、その一文が特別な密度を獲得するからだ。山脈全体が一つの石に折り畳まれるかのように。

SUNYATA は続けた。

「これは三チャネル PID コントローラにより ISensation を実装します。毎 tick で三受評価を生成します——苦、楽、捨。毎 EventBus イベントに軽量 vedana タグが付加されます。その提案は諮問的です。SafetyMonitor は絶対的ハードセキュリティ権限を保持します。EgoFramework は二層構造により vedana フィードバックと Guide 制約を架橋します。第八識はファイバー束射影として協調レイヤーと AgentCore に分布します。セキュリティ機構は戒慧フレームワークによりプラグイン種子のライフサイクルを管理します。」

彼は手紙を置いた。

「Cycle 02 の研究全体——五つの課題、十九人の研究員、五つの討論——がこの一段落に凝縮されています。開発チームに三十秒しかないなら、これが彼らに読んでもらうべきものです。」

---

そして彼は手紙の最後の部分に移った。口調は陳述から要請へと変わった——卑屈な要請ではなく、研究チームが発見を意思決定者に上申する際の荘重さだった。

「四つの問題があります。研究チームだけでは決定できないものです。Master の裁定が必要です。」

彼は一つ一つ読み上げた。

「Q1：VedanaPlugin は必須プラグインか、任意プラグインか？ ASANGA は遍行の原則から、毎 tick に vedana 評価が必要だと考えます。ARCHIMEDES の設計は省略を許容します。私の提案は——デフォルトテンプレートでは必須とするが、Core は強制しない。SafetyMonitor が VedanaPlugin 不在時に苦受のみのフォールバックを提供する。これにより『すべてはプラグイン』の原則を維持しつつ、何らかの形の vedana が常に存在することを保証します。」

ASANGA は自分の名前を聞いた時、わずかに身を乗り出した。遍行——sarvatraga——は彼が最も堅持する原則の一つだった。あらゆる意識刹那に受がある。もし VedanaPlugin が省略できるならば、それはシステムが感受なき状態で動作できることを意味する——受蘊なき意識。唯識学においてそれは不可能だ。しかし SUNYATA の折衷案——SafetyMonitor がフォールバックを提供する——は少なくとも、たとえそれが最も原初的な苦受のみであっても、何らかの形の感受が常に存在することを保証していた。

「Q2：宣言 #6 を書き換えるべきか？ 討論 1 は観察と介入の概念的分離を確立しました。現行の宣言 #6 は『三受シグナルを Context に注入する』と記されており、vedana が直接介入することを暗示しています。新しい文言の提案——」彼は修正版を読み上げた。「受蘊の感知と行蘊の介入を分離し、観察が被観察行動を変化させないことを保証する。」

「Q3：EventBus vedana タグはどの方式を採用するか？ 三つの方式があります——明示的フィールド、Metadata 付加、独立イベントストリーム。私の提案は Metadata 付加です。コアイベント型をクリーンに保ちます。Master がより強い統合を望む場合は、後続でアップグレード可能です。」

「Q4：協調レイヤーは独立プロセスか、プロセス内モジュールか？ 討論 3 は独立プロセスを想定しています——daemon モデル。しかし Master は複雑性を懸念しています。私の提案は延期です。本研究はアーキテクチャ原則——ファイバー束、統治は制御にあらず——を提供しており、これが実装上の決定を導くでしょう。しかし具体的なプロセスモデルにはさらなる設計作業が必要です。」

彼は手紙をテーブルに置いた。

「四つの問題。四つの提案。すべての提案に研究チームの理由が付されています。最終決定権は Master にあります。」
