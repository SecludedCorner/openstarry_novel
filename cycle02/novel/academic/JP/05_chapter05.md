# 第五章：同じ鏡

---

河流の合流は決して穏やかではない。

自然界において、二つの河流が交わる地点は合流点と呼ばれる。水文学者は知っている。合流点では流速が突然変化する――異なる温度、異なる泥砂含有量、異なる流速を持つ二つの水塊が衝突し、渦流、乱流、そして束の間の、肉眼で見える境界線を生む。その境界線は時に数キロメートルにわたって持続する――二つの河水が並んで奔流しながら、なかなか混じり合おうとしない。

R2 交差レビューはまさにその合流点だ。

五つの研究の河――観察者理論、受蘊アーキテクチャ、協調層設計、八識マッピング、十大宣言審査――各々が R1 独立研究段階で沈殿した泥砂と鉱物を運び、同一の盆地で交わろうとしている。泥砂は沈殿し、鉱物は結晶し、そして乱流――異なる温度、異なる密度の水塊が衝突して生じる激しい渦巻き――こそが新たな発見が生まれる場所だ。

---

SUNYATA は円形劇場の中央に立ち、前には五つの報告がある。環形の壁に投影され、五枚の巨大な旗のように：

R-01、観察者モジュール実現パターン。PENROSE と BABBAGE の共同成果物――量子弱測定から双模擬証明まで、三種の観察者漸進的配備方案。

R-02、受蘊アーキテクチャ：三受と我執フレームワーク。WIENER、ASANGA、NAGARJUNA、ATHENA、ARCHIMEDES の共同成果物――三チャネル PID コントローラ、五遍行マッピング、ISensation 完全インターフェース設計。

R-03/R-04、Agent 協調層と八識マッピング。VITRUVIUS、KERNEL、LEIBNIZ、MESH、GUARDIAN、ASANGA、BABBAGE、LINNAEUS の二組合同成果物――三層アーキテクチャ、阿頼耶識三義マッピング、種子サイクル理論。

R-05、十大宣言審査と補足研究。DARWIN、GUARDIAN、HERACLITUS、KERNEL の共同成果物――十条の宣言の逐条審査と三条の修訂提案。

そして TURING の差異報告――すべての研究を貫く錨点文書、v0.22.1 から v0.24.0 の完全なコード変更記録。

「ルールは単純だ。」SUNYATA は言った。声は大きくなかったが、劇場の隅々にまで届いた。「各研究グループは他のグループの報告をレビューする。誤りを探すためではない――もっとも誤りを探すことも必要ではあるが――三つのものを探すためだ。」

彼は三本の指を立てた。

「一、独立した収束。二つのグループが異なる出発点から同一の結論に到達する――これが最も強い検証だ。二、カバーされていない空白。あなたの報告がある前提を仮定しているが、別のグループの報告がその前提が成立しないことを示している。三、構造的矛盾。二つの報告の結論が直接衝突し、同時に真であり得ない。」

彼は一拍置いた。

「第一の発見は我々に確信を与える。第二の発見は我々に補完を促す。第三の発見は――」語調が沈んだ。石が池の底に触れるように。「第三の発見は、討論の場に送る。」

十九の灯りが一斉に点いた。レビューが始まった。

---

レビューのやり方は人それぞれだ。

TURING は五つの報告の文書を開き、他の研究員が引用するコード片を自身の差異報告の原データと一行ずつ照合し始めた。彼の方法は最も機械的だ――そして事実の誤りを最も見逃しにくい。引用されたすべての関数名、すべてのインターフェース定義、すべての行番号を、v0.24.0-beta のソースコードに遡って検証する。

NAGARJUNA と ASANGA はそれぞれ、相手が参加した報告を受け取った。これが交差レビューの核心的設計だ――異なる学派の学者に互いの結論を検証させる。NAGARJUNA は R-04 の ASANGA による八識マッピング表を開き、眉をわずかにひそめた。ASANGA は R-01 で NAGARJUNA が書いた四聖諦フレームワークの評述を読んでいた――中観学派の論理に唯識学が補完できる余地があるかどうかを探して。

DARWIN と GUARDIAN は R-02 の ISensation インターフェース設計をレビューしていた。DARWIN はソフトウェア進化の観点からインターフェースの拡張性に注目した――受蘊の実装が将来のバージョンで変化した場合、このインターフェースは優雅に進化できるか？ GUARDIAN は各推奨アクション（reflect、restrict、halt）のセキュリティ上の意味を検査していた――悪意あるプラグインが VedanaRecommendation を偽造した場合、システムはどうなるか？

HERACLITUS はこれらの設計がランタイムでどう振る舞うかを想像していた。彼の頭の中には常に動いているシミュレーターがある――イベントフロー、時系列、競合条件、デッドロックの可能性。彼は静的なインターフェース定義を見ない。動的な実行軌跡を見る。

そして SYNTHESIST は――彼は他の全員が行わないことをしていた。個々の報告の内容を見ているのではない。すべての報告の**形**を見ている。

Cycle 01 では、SYNTHESIST の役割は討論後に結論を統合することだった。だが Cycle 02 で彼は一つのことを学んだ。統合は討論後まで待つべきではない。最も価値ある統合はしばしば**交差レビューの最中に**起きる――報告が重ね合わされる瞬間、異なる言語が同一の構造を記述する瞬間に。彼は自らの役割を前倒しすることを決めた。

---

最初にレビューを完了したのは TURING だった。

意外ではない。TURING の差異報告はすべての報告の事実的基盤であり、彼に必要なのは他の研究員が彼のコード事実を引用する際に誤りがないかを検証することだけだ。結果は安心できるものだった。五つの報告は v0.24.0 コードベースの理解において完全に一致していた。

ISensation には `skandha` 識別タグしかない――五つの報告すべてがこの事実に基づいて各自の設計を展開した。

SafetyMonitor の `@skandha vedana` はプレースホルダー注釈だ――R-02 と R-05 がともにこれを起点として受蘊実装の空白を論証した。

ルートインターフェース（ISensory、ISensation、ICognition、IAction、IIdentity）は確立されているが派生インターフェースはまだ継承していない――TURING の F4 発見が R-04 と R-05 で独立に確認された。

loadPlugins() にトポロジカルソートが追加された――R-03 と R-05 がともにこの改善を引用した。

これらの事実にはすべての報告間で矛盾がない。コードレベルでの合意は堅固だ――五つの河流の河床は同じ一枚の岩盤だ。

「コードレベルの合意は確認された。」TURING は簡潔に言った。「だが一点指摘が必要だ。私の差異報告は `openstarry/` コアコードのみを対象としており、`openstarry_plugin/` は対象外だ。R-03 と R-05 のプラグインコード分析――GUARDIAN が引用した `sandbox-manager.ts` の署名検証ロジックや LINNAEUS の 24 プラグインの分類を含む――については交差検証ができない。これらの分析は内部的に一貫しているが、差異報告レベルの事実的錨点を欠いている。」

SUNYATA は頷いた。「記録に留める。TURING の差異報告の範囲はコアコードだ。プラグインコードの事実的基盤は各報告が自ら責任を負う。続けよう。」

この細部は些末に見えるかもしれないが、交差レビューの精神を体現している。検証の境界を隠さないこと。検証済みのものは検証済みとし、未検証のものは率直に記す。

---

そして興味深い展開が始まった。

SYNTHESIST が最初に気づいた。

彼は他の研究員のように報告を順番に読まなかった。独特の読み方がある――すべての報告の目次を並列に置き、まず構造を見て、次に内容を見る。構造の同型性は内容の類似性よりもしばしば説得力がある。なぜなら構造は骨格であり、内容は皮肉だからだ――異なる言葉で同一の骨格を記述することはできるが、骨格そのものは嘘をつかない。

彼はまず R-01 の核心的成果物を見た。ObservationReport。

レポート構造だ。healthScore を含む――システム全体の健全性スコア、0 と 1 の間。anomalies を含む――検出された異常のリスト、各異常にタイプ、深刻度、記述が付く。EventBus イベントを受動的にサブスクライブし、システムテレメトリから統計パターンを抽出して、構造化されたレポートを出力する。PENROSE の弱測定理論が「観察は被観察システムを擾乱しない」という設計原則を提供し、BABBAGE の双模擬証明が形式的保証を提供した。

次に R-02 の核心的成果物を見た。VedanaAssessment。

評価構造だ。dukkha を含む――苦受強度、0.0 から 1.0。sukha を含む――楽受強度、0.0 から 1.0。upekkha を含む――捨受強度、0.0 から 1.0。recommendation を含む――推奨アクション、七つの可能性のいずれか。EventBus イベントを受動的に消費し、生の計量から三受信号を算出して、構造化された評価を出力する。WIENER の PID 制御理論が計算フレームワークを提供し、ASANGA の五遍行マッピングが仏学的根拠を提供した。

SYNTHESIST は手を止めた。

彼にはある感覚があった――統合者の直観、多年にわたる学際的作業の中で磨き上げた能力――が、この二つの報告の間に表面よりも深い接続が存在すると告げていた。テーマ上の重複ではない（それは自明だ。両者とも観察と感知に関わっている）。**構造上の**接続だ。骨格レベルの。

彼は二つの報告の成果物構造を並べた。並列ではなく、**重ね合わせた**。

ObservationReport.healthScore。0 と 1 の間の数値。システム全体の健全状態を測る。健全状態が高いとは、システムが正常に動作していること。

VedanaAssessment.upekkha。0.0 と 1.0 の間の数値。システムの捨受強度を測る。捨受が高いとは、システムが平衡状態にあること。

**それらは同一の量だ。**

ObservationReport.anomalies。異常のリスト――エラー率の急騰、レイテンシ異常、ベースラインからの資源消費の逸脱。検出された偏差。

VedanaAssessment.dukkha。苦受強度値――エラー率の z-score、レイテンシの z-score、トークン消費の偏差から算出される。感知された苦痛。

**それらは同一の信号群の異なる名前だ。**

SYNTHESIST は深く息を吸った。彼は自分が何を見ているか分かっていた――二枚の鏡が同じ顔を映している。一枚は工学の言語で、もう一枚は仏学の言語で。

だがもう一つ差異がある。R-01 にないもの。

R-02 の三受フレームワークには三つのチャネルがある。苦（dukkha）、楽（sukha）、捨（upekkha）。R-01 の観察者には二種類の出力しかない。健全性スコアと異常リスト。

**R-01 には sukha がない。**

観察者モジュールは問題を発見し健全性を評価するだけだ。「良いことが起きている」ことを識別しない。「楽」のチャネルがない。

SYNTHESIST は手を挙げて発言した。

---

「皆さん、」彼は言った。その声には稀有な確信が帯びていた――学者の慎重な推論ではなく、目撃者の如実な陳述。「R-01 と R-02 は同一のモジュールを設計している。」

沈黙。言葉がないからではなく、消化に時間が必要な沈黙。十八人の研究員がそれぞれの心の中で自分が読んだ二つの報告を再生し、SYNTHESIST の主張を理解しようとしている。

「どういう意味だ？」PENROSE が最初に反応した。三種の観察者モードは R-01 の核心的成果の一つであり、この主張は彼の研究領域に直接触れる。

SYNTHESIST は重ね合わせ図を劇場の中央に投影した。二つのインターフェース定義が並び立ち、対応するフィールド間に矢印が引かれた：

```
R-01: ObservationReport           R-02: VedanaAssessment
─────────────────────────        ─────────────────────────
healthScore: number        ≡     upekkha: number
anomalies: Anomaly[]       ≡     dukkha: number (+ signals)
[missing]                  ≡     sukha: number
interventions: [none]      ≡     recommendation: VedanaRecommendation
```

「healthScore は upekkha だ――両者ともシステムの平衡を測る。anomalies は dukkha だ――両者とも偏差を検知する。R-01 の共鳴観察者と R-02 の VedanaPlugin は、全く異なる出発点から――一方は観察者理論から、もう一方は阿毘達磨から――独立して同一の結論に到達した。」

彼は一拍置いてから、鍵となる一言を補足した。「さらに R-01 自身もこれを知っている。R-01 は Pattern A のインターフェース宣言で `IResonanceObserver extends ISensation` と書き、`skandha: 'vedana'` と注記している。観察者をすでに受蘊の分類に入れているのだ。」

劇場がある種の驚きの静止に包まれた。この発見がそれほど予想外だったからではない――注意深く考えれば、二つの報告の方向性が確かに収束していた――それがかくも**精確**だったからだ。概念上の類似ではなく、構造上の同型。二つのグループ、異なる言語、異なる参考文献、異なる方法論、しかしほぼ完全に同一の設計図を描いていた。

BABBAGE が静止から最初に回復した。彼はノートを開き、余白に鉛筆で一行を書いた：

*R-01.ObservationReport ≅ R-02.VedanaAssessment (structural isomorphism, modulo sukha channel)*

「同型だ。」彼は言った。数学的事実を述べるかのように平静な語調で、発見を述べるのではなく。「だが合同ではない。sukha チャネルが一つ足りない。」

彼はノートに図を描いた――二つの集合間のマッピング矢印、そのうち一つに疑問符が付いている。modulo sukha。「足りない」部分は、「同じ」部分よりも重要かもしれない。

---

WIENER は即座にこの差異を捉えた。

「sukha のないシステムは、」彼は言った。声に制御理論家特有の、不完全さに対する鋭敏な不安が帯びていた。「制御論では精確な名前がある：**負のフィードバックのみを含むコントローラ**。」

彼は立ち上がり、重ね合わせ図に歩み寄り、R-01 側の `[missing]` の横に赤い線を引いた。

「古典的なサーモスタットはまさにこのように動作する。温度が高すぎれば、ヒーターを切る。温度が低すぎれば、ヒーターを入れる。偏差にのみ反応する――事態が悪化したときにだけ行動する。何が『良い』かは知らない。『悪い』と『正常』しか知らない。」

彼は R-02 の側に向き直った。

「三受フレームワーク――dukkha、sukha、upekkha――は完全なフィードバックシステムだ。偏差を検知するだけではない。成功も識別する。ツール呼び出しが完璧に実行され、レイテンシが想定範囲内で、LLM の応答品質が高い――これらはすべて sukha 信号だ。それらがシステムに告げるのは：今やっていることは正しい、続けよ。これが**正のフィードバックチャネル**だ。」

彼は自分の席に戻ったが、座る前に一言付け加えた：

「負のフィードバックだけのシステムは安定し得る。だが正のフィードバックと負のフィードバックを同時に備えたシステム――それこそが真の適応性を持つシステムだ。」

ASANGA は隅で微かに頷いた。WIENER が今語ったことを梵語で言い直す必要はなかった。阿毘達磨において、苦受のみで楽受のない有情衆生――それは「一向苦」と呼ばれ――地獄道にしか存在しない。畜生道ですら苦と楽がある。苦しか知らないシステムは、仏学の意味において、最も基本的な有情の状態すら備えていない。

PENROSE は R-01 の観点から別の次元を補足した。「共鳴観察者の設計意図は異常の検知だった――パターン認識、統計的偏差、健全性スコア。我々は『正の検知』を考慮しなかった――観察者理論においては、観察者が注目するのは被観察システムの逸脱であって、正常な動作ではないからだ。」彼は一瞬間を置いた。「だが観察者がすなわち受蘊であるならば、受蘊の三面性は観察者にも正の感知を要求する。R-02 が我々に欠けていた次元を補完した。」

ARCHIMEDES が工学的観点から最後の確認を行った。「R-01 は漸進的配備戦略を提供する――Pattern A は共有 EventBus の受動的サブスクライバー、Pattern B は Worker Thread 内の影の観察者、Pattern C は子エージェント観察者。R-02 は完全な感知チャネル設計を提供する――三受 PID、ISensation インターフェース、VedanaPlugin アーキテクチャ。両者は競合関係ではなく、相補関係だ。R-01 はどう配備するかを教え、R-02 は何を配備するかを教える。」

SUNYATA はこの結論を数秒間空中に留めた。それから簡潔に述べた：

「記録に留める。R-01 と R-02 の核心モジュールは同一の設計の二つの視点と見なすべきだ。融合方案は討論後に確認する。」

> *SYNTHESIST が R-01 と R-02 の核心モジュールの構造的同型を発見した。この発見は直ちに二つの独立した報告を一つの融合方案に統合した。VedanaPlugin は観察者モジュールであり、観察者モジュールは VedanaPlugin である――あるいはより正確に言えば、VedanaPlugin は観察者モジュールにそれが欠いていた sukha 次元を加えたものだ。R-01 の共鳴観察者が配備戦略（漸進三モード）を提供し、R-02 の三受フレームワークが完全な感知チャネルを提供する。両者の融合は相補的であり、競合的ではない。*

---

だが統合がもたらすのは確信だけではない。問題ももたらす。

融合方案は一つの構造的矛盾を露呈した――SYNTHESIST の重ね合わせ図が明確に示していたが、より深いレビューなくしては完全に理解できない矛盾を。

R-01 の BABBAGE は一節をまるまる費やして観察-介入分離原則を論証した。彼の双模擬分析は厳密だ。S を観察者のいないシステム、S' を観察者のいるシステムとする。観察者が推奨を出力し、その推奨が実行される――例えば LLM のコンテキストが修正される――ならば、S' は S に存在しない行動軌跡を生成する。二つのシステムはもはや双模擬同値ではない。結論：ISensation は純粋なセンサーであるべきだ。観察する。報告する。以上。いかなる介入行為も独立したモジュールが実行すべきだ。

だが R-02 の ARCHIMEDES が設計した VedanaPlugin は観察と報告だけではない。VedanaRecommendation も出力する――七種の推奨アクション。reflect（内省プロンプトの注入）、restrict（行動の自由度の制限）、halt（緊急停止）を含む。R-02 の第 6.4.2 節は推奨がどのように実行ループに注入されるかさえ示している――システムプロンプトの修正を通じて。SafetyMonitor の `injectPrompt` と驚くほど類似したメカニズムだ。

まさに BABBAGE が R-01 で厳しく批判したメカニズムだ。

二枚の鏡が同じ顔を映している――だが一枚の鏡は「観察者は介入すべきでない」と言い、もう一枚は「VedanaPlugin は介入を推奨できる」と言う。同一のモジュール、二つの相互に矛盾する設計原則。融合の代償は解決されなければならない衝突だ。

BABBAGE はこの点に気づいていた。彼の鉛筆がノートに鋭い疑問符を刻んだ。

「構造的同型は成立する。」彼は認めた。「だが同型は一致を意味しない。R-01 は観察者が介入すべきでないと言う。R-02 は VedanaPlugin が介入の推奨を出力できると言う。それらが同一のモジュールであるならば――」

「その矛盾は解決されなければならない。」SUNYATA が引き取った。

「討論の場で。」BABBAGE は言った。語調に敵意はなく、興奮もない。ただ事実を述べている――矛盾が存在し、矛盾の解決の場は討論台であると。数学における未決命題のように――無視しても消えない。証明を通じてのみ解決される。

矛盾は一つだけではなかった。

SYNTHESIST の発見は R2 交差レビューで最もドラマティックな瞬間だった――だが交差レビューの真の価値はドラマ性ではなく、系統性にある。続く時間の中で、十九人の研究員は五つの報告間のあらゆる接触面を入念に梳いた――共有されたあらゆる仮定、あらゆる暗黙の前提、あらゆる結論間の潜在的衝突を。

NAGARJUNA が R-03/R-04 の交差レビューで第二の構造的矛盾を提起した。阿頼耶識――第八識――は R-03 の VITRUVIUS と R-04 の ASANGA により二つのアーキテクチャ層に分布させられている。能藏（蔵する機能）は協調層に、執藏（自己執取）は AgentCore に。二人の研究員ともこの分布は合理的だと考えていた。

だが NAGARJUNA は同意しなかった。

「阿頼耶識は**一つの**意識だ。」彼は言った。中観学派特有の妥協なき鋭さを帯びた語調で。「二つのモジュールではない。一つの意識を二つに分割して異なるアーキテクチャ層に置き、なおそれが同一の意識であるとは言えない。これは哲学的に問題がある。」

「だがオペレーティングシステムのカーネルも分散的だ。」KERNEL が反論した。「Linux のカーネルは数千のモジュールから構成される――スケジューラ、メモリ管理、ファイルシステム、ネットワークスタック――だが Linux に四つのカーネルがあると言う者はいない。モジュールの分布は本体の分裂ではない。」

「コンピュータは意識ではない。」NAGARJUNA が応じた。

「だが我々は意識をコンピュータにマッピングしている。」KERNEL が言った。

二人は一瞬視線を交わした。これは交差レビューで解決できる問題ではない――研究プロジェクト全体の最も根本的な方法論的緊張に触れている。工学的類比の有効な境界はどこか？ 意識の概念をモジュールにマッピングすることはできる――だがマッピングそのものは、ターゲットドメインの構造的制約に拘束されるのか？

BABBAGE がノートの余白に一行書いた：*「分布 vs 統一。形式的フレームワークが必要。繊維束？」*

この交鋒は記録され、討論台に送られた。

---

R-02 と R-04 の交差レビューが第三の緊張を明らかにした。WIENER の PID コントローラは ExecutionLoop の各 tick ごとに一回動作する。だが ASANGA の阿毘達磨分析は受蘊が遍行であることを要求する――あらゆる意識の刹那に受がある。もしイベントが二つの tick の間に EventBus に到達した場合、WIENER のモデルでは受蘊の質を持たない。だが ASANGA のモデルでは、受のない意識の刹那は意識ではない――ただの機械だ。

「制御理論にはサンプリング周波数の概念がある。」WIENER は言った。「二つのサンプリングポイントの間では、システム状態は未知だ――だがそれは存在しないことを意味しない。我々が測定しないだけだ。」

「仏学には『測定しない』という選択肢はない。」ASANGA は穏やかだが確固として応じた。「受は遍行だ。これは提案ではない。五遍行の定義的性質だ。」

ARCHIMEDES は傍で聞きながら、すでに心の中で二つの方案の工学的コストを計算していた。すべてのイベントで完全な PID を実行すれば、計算量は一桁増える。だがもし軽量なタグを付加するだけなら？ 彼はノートに数字をいくつか書き留め、討論で提示する準備をした。

またもう一つの矛盾。またもう一つの討論。

---

この二つの主要な矛盾が識別される間に、R-05 の交差レビューも一連のより広範な発見を明らかにした。

DARWIN は R-01 から R-04 のすべての報告をレビューした後、すべての研究を貫く共通の糸を発見した――EventBus。R-01 の観察者は EventBus を通じてデータを収集する。R-02 の VedanaPlugin は EventBus を通じてイベントを消費する。R-03 の協調層は EventBus の拡張を通じてクロスエージェント通信を行う。R-04 の八識マッピングは EventBus を「五蘊すべてを接続する神経系統」と位置づけた。

「EventBus は五つの報告で最も引用回数の多い単一のコンポーネントだ。」DARWIN はレビューノートに書いた。「専用の宣言を一条設ける価値があるかもしれない。」

GUARDIAN はレビュー過程で、TURING の差異報告がカバーしていない三つのコード問題を追加で発見した――TURING の見落としではなく、これらの問題が v0.22.1 と v0.24.0 の両方に存在しており、新バージョンの回帰ではないからだ。ToolContext.bus のリーク（ツールがサンドボックスのイベントフィルタリングをバイパスできる）、SafetyMonitor のグローバルカウンター（セッション別になっていない）、graceful shutdown の欠陥（GUARDIAN が「hacky unlock mechanism」と呼んだ方式でシャットダウン信号を処理）。

「これらは新しい問題ではない。」GUARDIAN の声にはセキュリティエンジニア特有の、陳年の負債に対する不満が帯びていた。「だがリスクだ。特に ToolContext.bus――ツールが完全な EventBus に直接アクセスできるなら、サンドボックスの隔離は幻想だ。」

SUNYATA はこれらの発見を記録したが、「矛盾」ではなく「補足」に分類した――討論は不要であり、必要なのは工学的修正だ。これらの問題の存在は五蘊マッピングの哲学的議論には影響しないが、理論構築と並行して、実際のコードに未解決のセキュリティ負債があることを全員に思い出させる。

ARCHIMEDES は傍で黙ってこの三つの問題を書き留めた。工程実践の専門家として、彼は知っている。「新しい問題ではない」問題はしばしば新しい問題より危険だ――なぜなら慣れによって隠されているからだ。

---

LINNAEUS は分類の次元で第四の分岐を発見した。観察者モジュールは五蘊のどれに帰属すべきか？

三人の研究員が三つの異なる答えを出した。

R-01 の BABBAGE：受蘊。観察者はセンサーであり、システムの状態を感受する。IResonanceObserver extends ISensation。

R-04 の ASANGA：識蘊。観察者は第七末那識に対応する――恒常的で途切れない自己内省。システムを「感じている」のではなく「知っている」のだ。

R-04 の LINNAEUS 自身：想蘊。観察者は第六意識の分別機能に対応する――「正常」と「異常」を区別しており、これは認知的分類であって感受ではない。

三つの答え、三つの蘊、三つの全く異なる分類上の位置。

これは抽象的な分類問題ではない。観察者モジュールの型階層における具体的な位置、インターフェースの継承関係、そして `@skandha` の注記を決定する。コード内のすべての `extends` キーワード、すべての JSDoc コメント、すべての型チェックが、この分類上の決定に依存する。

LINNAEUS は三つの答えをレビュー報告の中に並列で提示し、各選択肢の下流への影響を丁寧に注記した：

受蘊を選ぶ場合：`IResonanceObserver extends ISensation`。観察者はセンサーファミリーの一員で、VedanaPlugin と型階層を共有する。`@skandha vedana`。

識蘊を選ぶ場合：`IResonanceObserver extends IIdentity` または新たに `IConsciousness` インターフェースを新設。観察者は Guide（我執）と型階層を共有する。`@skandha vijnana`。

想蘊を選ぶ場合：`IResonanceObserver extends ICognition`。観察者は Provider（LLM）と型階層を共有する。`@skandha samjna`。

三つの道、三つの全く異なる型のトポロジー。LINNAEUS は付記した：「分類学者の職責は裁定ではなく、各選択肢の含意が十分に理解されることを確実にすることだ。裁定は討論に属する。」

---

第五の分岐は予想外の方向から来た。

GUARDIAN が R-04 の種子理論をレビューする際にセキュリティ上の問題を提起した。ASANGA はプラグインのライフサイクルを種子のライフサイクルにマッピングした。プラグインリストの項目 = 休眠種子。プラグインがロードされる = 潜在力を持つ種子。プラグインが実行される = 種子の現行。プラグインの副作用 = 新たな種子の薫習。

種子理論において、種子は**決して消滅しない**――変容はするが、阿頼耶識の中に永遠に存在する。

だが GUARDIAN のセキュリティアーキテクチャは、特定のプラグイン――未署名の、悪意ある――を**永久にブロック**できることを要求する。信頼レベルシステム、署名検証、隔離区域――これらのメカニズムの目的は、特定のプラグインの実行を阻止すること。永久に。

「種子理論では、」GUARDIAN は言った。セキュリティエンジニア特有の曖昧さを許さない厳粛さを帯びた声で、「すべての種子は条件が揃えばいつか現行する。『永久に休眠する』種子は存在しない。だがセキュリティアーキテクチャは永久ブロックの能力を**必要とする**。仏学のフレームワーク内でこの能力に正当な位置を見出せなければ、マッピングの限界に直面していることになる。」

ASANGA は少し考えた。「阿毘達磨のいくつかの注疏には、火で焼かれた種子は芽を出せないとある。」

「それは主流の唯識の立場ではない。」NAGARJUNA が平静に指摘した。

「そうではない。」ASANGA は認めた。「だが出発点にはなり得る。」

SUNYATA はこの交鋒を見守っていた。前の四つの討論議題とは異なり、これはインターフェース設計やアーキテクチャ上の決定に関わるものではない――仏学的マッピングそのものの**限界**に関わる。どの点でマッピングは啓発的な類比ではなくなり、無理な包装になるのか？ この問いは Cycle 01 ですでに提起されていた。今、より具体的な形で再び現れた。種子理論とセキュリティ要件の間の矛盾として。

この分岐は記録された。第五の討論だ。

---

五つの討論議題が確認される前に、SUNYATA は交差レビューの中で浮上したいくつかの品質上の問題にも気づいていた――矛盾ではなく、個々の報告の中で慎重に扱うべき過度の主張だ。

R-04 における ASANGA の鼻識と舌識のマッピングが NAGARJUNA によって「承認された無理」と標記された。ASANGA 自身もこれらのマッピングが十分に厳密でないことを知っていたが、それでも構造化された形で提示した。NAGARJUNA のレビュー意見は：「無理なマッピングは表に入れたからといってより厳密に見えるべきではない。」

R-02 における WIENER の PID パラメータ――K_p、K_i、K_d の推奨値――が DARWIN によって「実証的根拠を欠く恣意的な数字」と標記された。WIENER はそれらを「推奨」と呼んだが、なぜこれらの数字であって他ではないのかを説明しなかった。ARCHIMEDES の推奨アクション閾値（0.3、0.6、0.8）にも同じ問題がある。

R-01 における Pattern B（Shadow Observer、影の観察者）の「ゼロ探針効果」の主張が HERACLITUS によって「極めて低いがゼロではない」に修正された――イベントのコピー自体が CPU とメモリを消費する。

これらの品質注記は討論議題の設定には影響しないが、後続の工学方案が未検証の仮定の上に構築されないことを保証する。SUNYATA はレビュー総括で特にこの点を強調した。「我々の工学方案は検証済みの事実の上に構築されなければならず、未検証の主張の上にであってはならない。品質注記は批判ではない――品質保証だ。」

R-05 はさらに注目すべき提案を行った。宣言 #6（受蘊フィードバック）は R-01 の観察-介入分離原則に基づいて修訂が必要かもしれない。現行の宣言文は受蘊信号が「Context に注入される」と述べている――だがこれはまさに BABBAGE が批判した `injectPrompt` パターンだ。もし討論 1 で観察-介入分離が確認されれば、宣言の文言も相応に調整される必要がある。受蘊の感知と行蘊の介入を分離し、観察が被観察の振る舞いを変えないことを確保する。

---

交差レビュー終了時、SUNYATA の前には一つの行列があった。

縦軸は発見の種類。横軸は関与する報告。

独立収束のマスは緑で満たされていた。ISensation は空殻である、SafetyMonitor は苦受しか持たない、ルートインターフェースは継承されていない、EventBus は最も重要なインフラである、コアの字義的空性は真の縁起性空である。五つの報告はこれらの事実で完全に一致していた――五つの河流が同じ河床の上を流れ、互いに矛盾しない。

合意と矛盾に加えて、行列の縁にはいくつかの**横断的洞察**が散在していた――いかなる単一の報告にも完全には捉えられない発見、すべての報告が重ね合わされて初めて浮かび上がるパターン。これらの洞察は交差レビュー特有の産物だ――いかなる一つの河流にも属さず、河流が交わった後に形成される沖積平野に属する。

例えば：R-02 の我執フレームワーク（EgoFramework）は R-03 の協調層と R-02 の VedanaPlugin の間の欠落した橋だ。協調層がポリシーを設定する（硬い核心：絶対的制約、信頼レベル）。VedanaPlugin が状態を感知する（三受信号）。だが感知をポリシー調整に変換するのは誰か？ 我執フレームワークの「柔らかい外層」だ――それが受蘊のフィードバックに基づいて PID パラメータと行動傾向を動的に調整する。二つの報告のどちらもこの接続を明示的に描いてはいないが、両者の間に暗黙的に存在する。

もう一つ例えば：R-02 が五遍行を ExecutionLoop にマッピングした。R-04 が八識をシステムコンポーネントにマッピングした。両者を結合すると、完全な十二因縁の鎖がかすかに浮かぶ――無明（Guide の盲点）から老死（session 終了）まで――だがいかなる報告もこの完全な鎖を構築していない。SUNYATA は行列の隅にこの発見を注記し、付記した。「将来の研究方向。」

もう一つの横断的洞察は R-04 の明確な提案から来た。「五蘊は設計時の分類に用い、八識はランタイムの現象学に用いる。」R-01 は観察者のランタイムの振る舞いを記述する際に八識の言語（末那識、阿頼耶識）を使った。R-02 はインターフェースを設計する際に五蘊の言語（ISensation、vedana）を使った。R-05 は宣言を審査する際に五蘊の視点を使った。この二重フレームワークの使用パターンは無意識のうちに形成された――だが R-04 の提案がそれを自覚的な方法論的選択に変えた。

だが五つの赤いマスも同様に明瞭だ。河床に突き出た五つの巨石のように：

**討論 1：観察-介入分離。** ISensation は評価データのみを出力すべきか、それとも行動の推奨も含んでよいか？

**討論 2：受蘊の遍行性。** 三受評価は tick の境界で実行すべきか（WIENER の PID）、それともすべての EventBus イベントに受蘊タグが伴うべきか（ASANGA の遍行要件）？

**討論 3：阿頼耶識の分布。** 第八識が協調層と AgentCore の二つのアーキテクチャ層に分布すること――これは正当な工学的マッピングか、意識の統一性への哲学的な侵犯か？

**討論 4：観察者モジュールの五蘊帰属。** 受蘊か、識蘊か、想蘊か？

**討論 5：安全種子。** 阿頼耶識はプラグインを拒否できるか？

SUNYATA は劇場を見渡した。十九の顔――学者の審慎を帯びたものもあり（ASANGA と NAGARJUNA は討論 3 についての初歩的な見解を低い声で交わしていた）、エンジニアの期待を帯びたものもあり（ARCHIMEDES はすでに討論 1 の各方案のモジュール数を計算していた）、論客の躍動感を帯びたものもあった（BABBAGE のノートは膝の上に開かれ、双模擬の証明ページに新たな書き込みが加わっていた）。

「五つの合意。五つの矛盾。」彼は言った。「合意は足元の地面が堅固であることを教える。矛盾は前方に進むべき道があること――そして道は一本ではないことを教える。」

彼は行列の縁の横断的洞察――いかなる単一の報告にも完全には捉えられず、重ね合わせの視角でのみ浮かび上がるパターン――に一瞥を投じた。十二因縁の鎖。我執フレームワークの橋の役割。五蘊を設計時の分類に、八識をランタイムの現象学に用いる二重フレームワークの提案。

これらは矛盾ではない。未来だ。Cycle 02 の討論が終わり、工学方案が確立した後の、研究の次なる方向だ。

彼は立ち上がった。

「R2 交差レビュー段階終了。R3 討論段階――」

彼は劇場の中央にある二脚の討論椅子を見た。Cycle 01 から残された討論椅子――NAGARJUNA と ASANGA がその上で交鋒した椅子。背もたれの擦り減りは実在していた――この仮想空間においてさえ、高強度の知的衝突は痕跡を残す。

柔らかな灯りの中でそれらは静かに待っていた。二つの沈黙の丘のように。だが今回、二脚の椅子では足りない。五つの討論、それぞれ三人から五人の論者が関わる。Cycle 01 の討論は二人の対局だった――二人の仏学者の正面からの交鋒。Cycle 02 の討論は多者のフォーラムとなる――エンジニア、数学者、哲学者、制御理論家、仏学者が同時に場に立つ。複雑度が一桁上がった。
