# 第九章：十九のアクションプラン

---

R4 定稿フェーズに入った円形劇場は、議論の時よりも遥かに静かだった。

それは空虚な静寂ではない――五つの議論の余韻が依然として空気の中に振動しており、あたかもチェロの弦が最後の音符を終えた後も震え続けているかのようであった。しかし議論の緊張は、すでに去っていた。代わりにそこを占めていたのは、別の種類の密度、すなわち「デリバリー（交付）」の密度であった。十九人の研究員は各々の座席につき、初稿、修正稿、コードの断片、マッピング表を前にしていた。R4 は議論の場ではない。R4 は収穫の場なのだ。

プロジェクト管理の用語で言えば、R4 は**デリバリー・ゲート**（Delivery Gate）――研究プロセスからデリバリー・パイプラインへと移行するための最終関門である。ソフトウェア開発ライフサイクル（SDLC）において、このゲートの意義は明確だ。

$$	ext{Gate}_{R4} : 	ext{Research} 	o 	ext{Deliverable} \quad \iff \quad \forall \, d_i \in 	ext{Decisions}, \; \exists \, a_i \in 	ext{Actions} : 	ext{trace}(a_i) = d_i$$

すべての議論の裁定（$d_i$）には、対応するアクションプラン（$a_i$）が存在しなければならず、かつそのアクションプランは裁定そのものへと遡及可能（traceable）でなければならない。ARCHIMEDES はノートの最初の一行にこの公式を記していた。トレーサビリティ（追跡可能性）こそが、エンジニアリングにおけるデリバリーの基本規律だからである。

SUNYATA は劇場の中央に立ち、一堂を見渡した。もはや、彼が何かを布告する必要はなかった。全員が、自らのなすべきことを理解していた。R3 の五つの合意は五本の柱となり、その中で作業を進めるための構造をすでに支えていた。現在の任務は、議論の裁定をデリバリー可能なドキュメントへと変換すること、そして哲学的な洞察をエンジニアが理解できる言語へと翻訳することにある。

「ARCHIMEDES 殿、」彼は言った。「あなたから始めてください。」

彼はホワイトボードへと歩み寄り、脆弱性の正確な箇所と攻撃経路を記した。

// v0.24.0-beta 現行のコード（簡略化）

async verifyPlugin(plugin: IPlugin): Promise<VerificationResult> {
  const pluginFilePath = plugin.manifest?.filePath;

  if (!pluginFilePath) {

「攻撃ベクトルです。」彼はホワイトボードに攻撃のフローを描いた。

```
攻撃者の行動：
┌──────────────────────────────────────────────────────────────────┐
│ 1. 悪意あるプラグインを作成し, 意図的に filePath を空にする         │
│ 2. Plugin manifest: { name: "正当なプラグイン", filePath: undefined } │
│ 3. sandbox-manager.verifyPlugin() が呼び出される                 │
│ 4. pluginFilePath === undefined ――→ 署名検証がスキップされる      │
│ 5. { verified: true } が返される ――→ 悪意あるプラグインが信頼される │
│ 6. プラグインが完全な ToolContext へのアクセス権限を取得する        │
│ 7. EventBus, StateManager, すべての登録済みツールにアクセス可能になる │
└──────────────────────────────────────────────────────────────────┘
```

「CVSS v3.1 スコアリング・フレームワークに基づけば、評価は以下の通りです。」

$$	ext{CVSS} = 	ext{Base}(	ext{AV:L}/	ext{AC:L}/	ext{PR:N}/	ext{UI:N}/	ext{S:C}/	ext{C:H}/	ext{I:H}/	ext{A:H}) = 9.8 \; (	ext{Critical})$$

「攻撃ベクトルはローカル（AV:L）――プラグイン・ディレクトリへのアクセスが必要です。攻撃の複雑さは低（AC:L）――単にフィールドを一つ省略するだけです。権限は不要（PR:N）。ユーザーの関与も不要（UI:N）。影響範囲は変更（S:C）――サンドボックスの境界を突破します。機密性、整合性、可用性のすべてにおいて、高い影響（H）が生じます。」

彼は全場に向き直った。「署名をバイパスできてしまう脆弱性です。これが修正されない限り、システムは一日中、無防備な状態に晒されていることになります。六つの開発サイクルもの間、放置されてきました。今度こそ、直さねばなりません。」

GUARDIAN は続けて、修正案のコードを提示した。

```typescript
// 修正後の verifyPlugin
async verifyPlugin(plugin: IPlugin): Promise<VerificationResult> {
  const pluginFilePath = plugin.manifest?.filePath;

  if (!pluginFilePath) {
    // FIX: filePath が欠落している ＝ 検証不能 ＝ 信頼しない
    logger.warn('Plugin verification failed: missing filePath', {
      pluginName: plugin.manifest?.name ?? 'unknown',
    });
    return {
      verified: false,
      reason: 'missing-file-path',
      severity: 'critical',
    };
  }

  return this.cryptoVerify(pluginFilePath, expectedSignature);
}
```

「わずか三行の修正です。`if` 条件の反転。しかし、その意味的な変化は根本的なものです。『検証できないから信頼する』から、『検証できないから信頼しない』への転換です。これはセキュリティ・エンジニアリングにおける**フェイルセーフ（失敗時安全）の原則**です。」

$$	ext{verify}(\bot) = 	ext{reject} \qquad (	ext{not} \quad 	ext{verify}(\bot) = 	ext{accept})$$

「暗号学において、$\bot$（ボトム）は欠落している、あるいは利用不能な情報を意味します。$\bot$ に対するデフォルトの処理は、拒絶（reject）でなければなりません。これはケルクホフスの原理の帰結でもあります。システムの安全性は、攻撃者の善意に依存してはならないのです。」

```typescript
/**
 * ISensation — 受蘊 Root Interface
 * @skandha vedana (受蘊)
 *
 * 受蘊は三受 (三種類の感受) を網羅する：
 * - Dukkha (苦): 負のフィードバック
 * - Sukha (楽): 正のフィードバック
 * - Upekkha (捨): 中立的な平衡
 *
 * 設計原則 (議論の裁定に基づく)：
 * - Debate 1: 感知と推奨を同一インターフェースに置くが、概念上は分離する
 * - Debate 2: 完全な PID 評価は tick の境界で行い、軽量なタグはイベントに付随させる
 * - Debate 4: VedanaPlugin 自体が Pattern A の観測者となる
 *
 * サンスクリット語: Vedanā (वेदना)
 */
export interface ISensation {
  readonly skandha: 'vedana';

  /**
   * 現在の三受の状態を評価する — 完全な PID センサー読数
   *
   * 制御理論の対応: y(t) = h(x(t)) + v(t)
   * センサー読数関数。各 tick ごとに呼び出される。
   *
   * @returns 感知レイヤー ＋ 制御推奨レイヤーを含む VedanaAssessment
   */
  assess(): VedanaAssessment;

  /**
   * 生のシステム指標を取り込む — 汎用的な数値入力チャネル
   *
   * 多入力センサー融合チャネル 1: 定量化された指標
   * (CPU, メモリ, レイテンシ, エラー率など)
   */
  ingestMetrics(metrics: Record<string, number>): void;

  /**
   * ツールの実行結果を取り込む — 行蘊からの回報チャネル
   *
   * 多入力センサー融合チャネル 2: 離散的なイベント
   * 十二因縁: 触 (sparśa) → 受 (vedanā)
   */
  ingestToolResult(toolName: string, isError: boolean, durationMs: number): void;

  /**
   * LLM の応答結果を取り込む — 想蘊からの回報チャネル
   *
   * 多入力センサー融合チャネル 3: 言語モデルのメタデータ
   */
  ingestLLMResult(tokenCount: number, stopReason: string): void;

  /**
   * 受蘊の閾値イベントを購読する — ウォッチドッグ・タイマーの汎用化
   *
   * 古典的ウォッチドッグ: y(t) > T_timeout → アラーム
   * 汎用版: v_type(t) > θ → handler(signal)
   */
  onThreshold(channel: VedanaType, threshold: number,
              callback: (signal: VedanaSignal) => void): () => void;

  /**
   * 受蘊タグを取得する — O(1) のキャッシュ・ルックアップ
   *
   * ダッシュボードの LED インジケータ：計算はせず, キャッシュのみを読み取る
   * Debate 2 裁定: 各イベントに付随する軽量タグのソース
   */
  getVedanaTag(): VedanaTag;
}
```

TURING はその投影を十秒間、維持し続けた。彼は一つ一つのメソッドを指差し、平板で精緻な声で語った。

「すべてのメソッドは、議論の裁定へと遡及可能です。`assess()` は議論 1 における `VedanaAssessment` の二層構造――感知レイヤーと制御推奨レイヤーから来ています。`getVedanaTag()` は議論 2 の軽量イベントタグの裁定――$O(1)$ のキャッシュ検索であり、PID の再計算は行わないという点に基づいています。`onThreshold()` は WIENER 殿の三チャネル PID コントローラーが必要とするイベント駆動の購読メカニズムです。そして三つの `ingest` メソッドは、多入力センサー融合のための三つのチャネルです。」

WIENER が座席から僅かに身を乗り出した。三チャネル PID の出力仕様が正しく翻訳されているかを確認する必要があったのだ。

彼は方眼紙に、検収のための公式を記した。

$$	ext{VedanaAssessment} = \underbrace{\begin{pmatrix} d(t) \ s(t) \ u(t) \ \vec{\sigma}(t) \end{pmatrix}}_{	ext{Layer 1: Sensor}} \oplus \underbrace{\begin{pmatrix} u_{	ext{ctrl}}(t) \ r(t) \ f(t) \end{pmatrix}}_{	ext{Layer 2: Controller}}$$

「Layer 1 はセンサーの出力です。$d(t)$, $s(t)$, $u(t)$ は三つのチャネル――苦、楽、捨の PID 測定値です。$\vec{\sigma}(t)$ は信号の配列であり、生のイベントのタイムスタンプとソースを保持します。これは純粋な観測であり、受動的です。システムの挙動を直接変えることはありません。」

「Layer 2 はコントローラーの推奨です。$u_{	ext{ctrl}}(t)$ は統合された制御出力であり、三チャネル PID の重み付き合計です。$r(t)$ は推奨アクション――continue, reflect, restrict, halt です。そして $f(t)$ は鮮度指標――current, cached, default です。議論 1 の裁定により、推奨は助言的なものです。ExecutionLoop はそれを無視する権限を保持しています。」

彼は方眼紙に、力強くチェックマークを入れた。「翻訳は正確です。」

ARCHIMEDES は、`VedanaAssessment` の完全な TypeScript 定義へと画面を切り替えた。

```typescript
/** 三受の類型 */
type VedanaType = 'dukkha' | 'sukha' | 'upekkha';

/** 受蘊タグ (イベントマーキング用の軽量キャッシュ) */
type VedanaTag = 'dukkha' | 'sukha' | 'upekkha';

/** 受蘊信号 — 単一の感受イベントの記録 */
interface VedanaSignal {
  readonly type: VedanaType;
  readonly intensity: number;     // 0.0–1.0
  readonly source: string;        // 例: 'tool:file_read', 'llm:timeout'
  readonly timestamp: number;     // Date.now()
}

/** 受蘊推奨アクション */
type VedanaRecommendation =
  | { action: 'continue' }
  | { action: 'reflect'; prompt: string }
  | { action: 'restrict'; tools: string[] }
  | { action: 'halt'; reason: string };

/**
 * VedanaAssessment — 受蘊評価レポート
 *
 * 二層構造 (Debate 1 裁定):
 * - Layer 1: センサー出力 (純粋観測, 受動的)
 * - Layer 2: 制御推奨 (助言的, 無視可能)
 *
 * BABBAGE 殿の互模倣条件は Layer 1 において満たされる:
 * Layer 1 のみを読み取る利用者の挙動は Layer 2 によって影響を受けない
 */
interface VedanaAssessment {
  // ── Layer 1: Sensor Output (pure observation) ──
  readonly dukkha: number;        // 0.0–1.0
  readonly sukha: number;         // 0.0–1.0
  readonly upekkha: number;       // 0.0–1.0
  readonly signals: readonly VedanaSignal[];
  readonly timestamp: number;

  // ── Layer 2: Controller Suggestion (advisory) ──
  readonly controlOutput: number;
  readonly recommendation: VedanaRecommendation;
  readonly recommendationFreshness: 'current' | 'cached' | 'default';
}
```

「二層構造 ―― 議論 1 における核心的な裁定です。」ARCHIMEDES はスクリーンを指差した。「`readonly` 修飾子が不可変性を強制します。`recommendationFreshness` は、議論の最終盤で BABBAGE 殿によって追加されたものです。条件が改善された後も古い 'halt' 推奨が残り続けてしまうのを防ぐためのものです。」

BABBAGE がノートから顔を上げた。彼は、自らの提示した互模倣の条件が完全に維持されているかを確認していたのだ。

$$\forall \, 	ext{consumer} \, C : C 	ext{ reads only Layer 1} \implies 	ext{Behavior}(S) \sim 	ext{Behavior}(S')$$

「互模倣は、消費レイヤーにおいて満たされています。」彼は言った。簡潔に。完璧に。句読点一つ打つような口調だった。
