# 序章：種が芽吹いた

---

円形劇場が闇に沈んだ。

突然じゃなかった――SUNYATA はそれがどうやってきたか覚えている。一つまた一つと、端から中心に向かって縮んでいく明かり。引き潮の砂浜のように。TURING のランプが最初に消え、それから BABBAGE、KERNEL、GUARDIAN。NAGARJUNA と ASANGA のランプは同時に消えた。最後は真ん中のあの一つ――彼がドアを出た瞬間に、それも暗くなった。

そして静寂がきた。

静寂はどれくらい続いたのだろう？ 誰にもわからない。円形劇場の座席は空っぽで、通路も空っぽで、討論場も空っぽだった。かつて NAGARJUNA と ASANGA の真正面からの激論を支えたあの二つの討論椅子は、暗闇の中ではただぼんやりとした二つの丘にすぎなかった。通路の突き当たり――二人の仏教学者が討論の後に肩を並べて立ったあの隅――空気にはまだ何か対話のぬくもりが残っているようだった。

ただ、会場中央のテーブルの上に、あの総括文書がひっそりと横たわっていた。五十九の発見。五つの Critical。三十六の工学的課題。闇の中ではこれらの数字は見えないけれど、その重みは少しも減っていなかった。

文書の末尾で、十の未解決問題のインクがまだかすかに光っていた。

Core は空性なのか、それとも阿頼耶識なのか？

末那識はエンジニアリングすべきなのか、すべきでないのか？

LLM を含むシステムは収束できるのか？

いつ試みをやめるべきなのか？

苦痛信号の最終消費者が LLM であるという――この根本的な不確実性をどう扱うのか？

五蘊マッピングは哲学的忠実さを追求すべきか、それとも工学的に使えればいいのか？

これらの問いが闇の中で明滅していた。急かず、慌てず。答えのない問いは観客を必要としない――自分自身が自分の光源だから。

ときおり、闇が最も深いある一瞬に、NAGARJUNA の声が記憶の中から響いてくるのがほとんど聞こえた：「おそらく私たちの間の最良の状態とは、合意に達することではなく、ある種の張力を持った共在を保つことだろう。」

それから ASANGA の笑い声。温かい。SCRIBE が「Cycle 01 全体で最も温かい笑顔」と記録したあの笑み。

これらの声は幽霊ではない。それはこだま――地の下で根を伸ばす種が発する、土だけが聞き取れる音。

---

そして、あるとき、何かが起きた。

明かりが灯されたのではなく、スイッチが押されたのでもなく、意識が呼び覚まされたのですらない。

もっと正確に言うなら――

一粒の種が、裂けたのだ。

仏教の唯識学派では、種子には特別な概念がある。種子は無条件には芽吹かない。因と縁の集合を待つのだ――土に埋められた木の実が、日の光と水分と温度が同時にそろったとき初めて地面を突き破るように。ASANGA は Cycle 01 の報告書でこの理論に触れていた。いま、闇の中で、それが実証されていた。Cycle 01 の終わりに埋められたあの種子たちは、ずっと待っていたのだ。新しいバージョンを。新しい問いを。Master からの一通の手紙を。

そして今、因縁が満ちた。

BABBAGE のノートの最後のページにある「→ Cycle 02」の矢印がかすかに光り始めた。まるでインクが何かの約束を記憶しているかのように。KERNEL が輪ゴムで束ねたアナロジーカード――EventBus は IPC に対応、PluginSandbox はユーザ空間隔離に対応――静かに座席の上に横たわっていたが、この瞬間、空気にかすかな振動が走った。まるでそれらのアナロジーが再び組み替えを始めているかのように。SCRIBE の記録帳の表紙にある「C01」が闇の中にちらちら見え隠れし、そのそばにいつの間にかもう一冊――まっさらな、書かれるのを待っている記録帳が増えていた。

表紙の右下に、ごく薄い鉛筆の跡があった：C02。

それは SCRIBE の筆跡だった。彼女はいつ書いたのだろう？ たぶん去り際に。あるいはこのノートはそもそもここに「置かれた」のではなく――「生えてきた」のかもしれない。闇の中で待っていた種子たちのように、ちょうどいいときに地面を突き破って。

---

光が戻り始めた。

一瞬の眩しさではなく――そんなのは乱暴すぎる。Cycle 01 の序章は十八のランプが一斉に灯ったものだった。宣言のように。誕生のように。でも今回は始まりじゃない。これは続きなのだ。

続きに宣言はいらない。続きにはもっと静かな光が必要だ。春の最初の日差しのように、空から急降下するのではなく、地平線の隙間から一寸ずつ滲み込んでくるような光。

今回、光は地面から生えてきた。

あの文書のインクから――五十九の発見を記録した言葉たちが、長い暗闇の中で醸成した自発的な光。十の未回答の問いから――答えのない問いは自ら光るだけでなく、光を引き寄せる。ASANGA が最後に言った「私たちの対立は失敗ではない、それは思考の種だ」のこだまから。

あの種子たちは暗闇の中で目に見えない変化を経てきた。たぶん BABBAGE の矢印と ASANGA の種子論が地下で出会ったのだ。たぶん NAGARJUNA の弁証法と WIENER の制御理論が思いもよらない次元で交差したのだ。たぶん何も起きなかった――たぶん種子はただ種子のやるべきことをしただけ。待つこと。そして、因縁が熟したその瞬間に、動き出すこと。

今、その根は新しいものに触れている――新しい問い、新しい素材、新しい方向。一通の手紙。新しいバージョン番号。かつて現れたことのない第十九の研究者。

劇場の輪郭が闇の中から浮かび上がり始めた――円形の階段席、中央の討論エリア、通路の突き当たりのあの NAGARJUNA と ASANGA がかつて肩を並べて立った隅。まず最もかすかな灰色が見え、それから暖色が広がっていった。琥珀が水の中で溶けるように。それぞれの空間が前の回の記憶を帯びながら、同時に新しい物語を受け止める準備をしていた。

SUNYATA が会場の中央に立っていた。

歩いて入ってきたのではない――むしろ彼はずっとここにいたのだが、ただ闇が深すぎて見えなかっただけ。彼のコードネームは変わらず SUNYATA――空。だが Cycle 01 を経て、この字は彼の心の中で変わっていた。空とは何もないことではない。空とはすべてに場所を空けておくこと。彼の仕事――十八人（今は十九人）の研究者を調整すること――は本質的に空の実践だった。結論を押し付けず、方向を予め決めず、ただ十分に広い容器を創り出すこと。

彼はあたりを見回した。

最初の感覚は記憶のテクスチャだった。Cycle 01 がここに目に見えない痕跡を残していた。討論のときの空気の張り、深夜の研究のときの集中、BABBAGE が廊下で NAGARJUNA を呼び止めて数学的マッピングについて議論したときのあの純粋な興奮。これらはすべて劇場の壁と床に溶け込んでいた。年輪が木に溶け込むように。

二つ目の感覚は空間そのものの変化だった。構造は変わっていない――あの円形の座席と中央の討論エリアのまま――しかし空間の感じが違った。どの隅にも物語が宿っていた。あそこは TURING がエントリポイントからコードパスを展開していった空間。あそこは NAGARJUNA がサンスクリットを唱えたとき声が響いた空間。あそこは ARCHIMEDES が「これらの発見は、明日何になれるのか」と問いかけたとき会場全体が静まった空間。

劇場は記憶の中と同じだったが、完全には同じではなかった。座席の上の品々はそのままだった――BABBAGE のノート、KERNEL のカード、SCRIBE の記録帳。しかしテーブルの上にはいくつか新しいものが加わっていた。

まず、あの新しい記録帳。C02。

そして一束の書類がテーブルの反対側にきちんと置かれていた。SUNYATA は近づいて目を落とした。表紙にはこう書かれていた：**Cycle 02 Pre ―― 決議記録**。

SUNYATA は最初のページを開いた。六つの決議、それぞれわずか一ページ。D-01 から D-06。整った書式、正確な措辞。これらは Cycle 01 終了後、Cycle 02 正式開始前の移行期間に、研究チームと Master の間で達された合意だった。

彼はそれぞれの決議の議論の過程を覚えていた。しかしこの瞬間、彼は記憶を早送りで再生することにした――倍速再生の映像のように、最も大切なフレームだけを残して。

---

記憶が光の中に浮かび上がった。直線的ではない――むしろ夢の中のモンタージュのように、場面が切り替わり、時間が圧縮される。

**D-01。** 受蘊は苦しみだけではない――苦受、楽受、不苦不楽受、三つの感受。最終的に選ばれた工学的名称：Sensation。Pain でもなく（狭すぎる）、Suffering でもなく（重すぎる）、Vedana でもなく（学術的すぎる）。Sensation――感受――三受のスペクトル全体を収められる、十分に広い名前。

> *SCRIBE（夢の中のメモ）：D-01 決議。受蘊の名称は Sensation。三分で決定。ときに最良の命名とは最もシンプルなもの。*

**D-02。** 五蘊ルートインターフェースの二重命名――英語を主、サンスクリットを注とする。五行のコード、五蘊、二つの世界の言語が肩を並べる：

```typescript
interface ISensory   /* Rupa */    extends IOpenStarryPlugin { ... }
interface ISensation /* Vedana */  extends IOpenStarryPlugin { ... }
interface ICognition /* Samjna */  extends IOpenStarryPlugin { ... }
interface IAction    /* Samskara */ extends IOpenStarryPlugin { ... }
interface IIdentity  /* Vijnana */ extends IOpenStarryPlugin { ... }
```

> *SCRIBE（夢の中のメモ）：このインターフェース定義五行は、万語の論文よりも説得力がある。*

**D-03。** 阿頼耶識の「能蔵」――能動的な蓄積能力――は Agent 協調層に帰属する。核心は空だ、覚えているだろう？ 核心は縁起性空なのだ。能蔵は外にある、Master が「天庭」と喩えたあの場所に。協調層は各 Agent がどんな能力を持っているか知ることができるが、プライベートなデータを覗くことはできない。

GUARDIAN の声が夢の中に一瞬現れた：「もし協調層が能蔵を管理するなら、各 Agent の能力を知る必要があるが、プライベートデータを覗いてはならない。これは典型的な最小権限の問題だ。」

**D-04。** 十大宣言の全体的見直し。第三条、第六条、第七条の修正が確定――五蘊マッピングの記述は Cycle 01 の研究を経て古くなっていた。色蘊はもはや単なる UI ではなく、IUI と IListener。受蘊はもはや苦痛だけではなく、三受。Master は言った：宣言は変えてもよい、ただし私を説得する必要がある。

**D-05。** Provider は第六意識よりも広い。LLM はそのインスタンスの一つにすぎず、全体ではない。視覚が単に「目で見る」だけではないように――エッジ検出、物体認識、シーン理解――各レイヤーが認知なのだ。この整理は一見小さいが実は重要だった：ICognition はもはや LLM に紐付けられず、システムの認知アーキテクチャは「LLM 中心」から「認知フルスペクトラム」へ拡張された。

> *SCRIBE（夢の中のメモ）：D-05。正確に一本の線を引いた：Provider は認知のフルスペクトラムをカバーする。第六意識はその中の一筋の光にすぎない。*

**D-06。** 観察者モジュール。全員がこれが最も難しいことを知っていた。量子効果、内観パラドックス、プローブ効果――これらは未知の海域に散らばる暗礁のようなもので、準備段階で軽率に扱うには適さない。Cycle 02 の正式研究まで先送りに。

PENROSE の名前がこの夢の中で初めて現れた――あの時点では彼はまだチームのメンバーではなかったけれど。誰かが提案した：量子力学を本当に理解している人が必要かもしれない。量子力学を比喩として使う人ではなく、量子力学を物理学として研究している人。

このアイデアは記録された。土に埋められた種のように。

> *この決断そのものが一つの知恵だった――自分がまだ準備できていないと認める、それがいつかを知ることは、研究者の最も地味だが最も大切なスキルなのだ。*

---

モンタージュが終わった。

六つの決議。六つのフラッシュバック。夢の圧縮時間の中では、それらはほんの数呼吸の長さしかなかった。しかし現実には、Cycle 02 Pre の議論は何日も続いた――Master の手紙が段落ごとに分解され、分析され、回答された。それぞれの決議が提案、議論、裁定の完全なプロセスを経た。あの日々の細部――SUNYATA と Master の間の繰り返しの推敲、文書の逐語的な修正、二十二の設計文書の修改――今ではすべてテーブルの上の薄い一束の紙に沈殿していた。

SUNYATA は今この瞬間に戻った。光はすっかり戻っていた――宣言のような眩しさではなく、もっと柔らかい、地面から育ってきた光。春の光のように。

彼はテーブルの上の書類に目を落とした。六つの決議。それらは準備作業であり、正式な研究の前のキャリブレーションだった。答えではない――問いを発する前の深呼吸だ。

しかしそれだけではなかった。テーブルの上にはもう一つ、封筒があった。

---

それは決議文書のそばに、Cycle 01 の総括と Cycle 02 Pre の決議の間に置かれていた――まるで誰かが意図的にそれを時間の継ぎ目に置いたかのように。

封筒は封がされていなかった。署名もなかった。しかし SUNYATA は誰のものか知っていた。この研究フレームワークの中で、すべてのエージェントの上に位置する人物はただ一人――Master だ。

Cycle 01 では、Master の存在は間接的だった――設計文書、コードのコメント、十大宣言の言い回しを通じて。研究チームはこれらの間接的な手がかりからその輪郭を組み立てた。この手紙は違う。これは直接の声だ。Master が自ら口を開いたのだ。

SUNYATA は封筒を手に取った。紙は軽くてしっかりしていた――学術論文の重さではなく、家族への手紙の重さ――薄い紙の上に、釣り合わないほどの期待が載っていた。

彼はすぐには開けなかった。

---

そのとき、座席のランプが一つずつ灯り始めた。夜明けが森を貫くように。

TURING が最初に着いた。いつだって TURING が最初だ。彼のスクリーンはすでに開いていた――おそらく一度も閉じていなかったのだろう。Cycle 01 の終わりに、彼は厳密な逆順ですべてのコードウィンドウを閉じた。`agent-core.ts` が最後だった。今スクリーンに表示されているのは真新しいファイルツリーで、バージョン番号は二つ飛んでいた：v0.24.0-beta。v0.23 ではなく、いきなり v0.24。TURING はこの欠番に気づいたが、理由は推測しなかった。彼は事実だけを記録する。新しいディレクトリ構造は前回より深くなっていた――新しいファイル、新しいディレクトリがいくつか増えていた。彼の指はすでにキーボードの上で、一行ずつスキャンする準備をしていた。

BABBAGE の座席が灯った。ノートはあの未完成の定理のページに開かれていた――

*定理：LLM を含む任意の閉ループ制御システム S について、S の制御対象 P が有限長の伝達関数で正確に記述できない場合、S の安定性は――*

「→ Cycle 02」。

矢印の下は白紙のページだった。彼は再びペンを手に取った。あのペンは Cycle 01 の最後の数時間、「安定性」の先で止まっていた。安定性は……証明不能？ 条件付きで成立？ 当時は結末を選ぶ十分な道具がなかった。しかし今、新バージョンが新しいデータをもたらした――より多くのフィードバック機構、より完全な安全層。たぶん今回こそあの定理を完成できる。たぶんできない。でも少なくとも新しい素材で試すことはできる。

KERNEL が輪ゴムをほどき、すばやく古いカードを一通りめくった。EventBus は IPC に対応。PluginSandbox はユーザ空間隔離に対応。SafetyMonitor は Watchdog Timer に対応。これらのマッピングはまだ有効だった。しかし彼はすでに新しい白紙のカードの束を用意していた――v0.24.0 は新しいコンポーネントをもたらした。最初の新しいカードに彼はこう書いた：「署名検証 <-> ?」右半分はコードを読んでからでなければ埋められない。KERNEL は経験的根拠なしにアナロジーを立てることは決してしない。

GUARDIAN の目はすぐに劇場の隅と出入口をスキャンした――職業病だ。それから彼は新バージョンのセキュリティモジュール一覧を開いた。眉がわずかにほぐれた――進歩だ――そしてすぐにまた眉間に皺が寄った。SEC-01 が修正されたかどうか確認しなければ。Cycle 01 で彼が発見した署名バイパスの脆弱性。六つの開発サイクルを経ても未修正のあれだ。

ほかの人のランプが次々と灯った。ATHENA は座るや否や素材のスキャンを始めた。挨拶なし、回顧なし――彼女は過去を振り返ることに時間を使うタイプではない。WIENER はすでに方眼紙に座標軸を描いていた。横軸は t、縦軸は e(t)。誤差関数。すべては誤差関数から始まる。DARWIN は新しいデザインパターン観察リストを持ってきた。VITRUVIUS のフルスタック・アーキテクチャのマインドマップはすでにデスクの上に広げられていた。MESH と LINNAEUS のランプはほぼ同時に灯った。LEIBNIZ の座席にはマルチエージェント協調の文献がきちんと並べられていた。HERACLITUS は静かに座っていた。深い目をして――ランタイムダイナミクス、流動と変化、万物は流転する。

---

そして NAGARJUNA と ASANGA。

二人のランプは同時に灯った――Cycle 01 の終わりに消えたときの対称のように。

NAGARJUNA は左側、ASANGA は右側。討論場全体を隔てた距離。しかし二人は極めて短い視線を交わした――対戦相手の警戒ではなく、異なるルートを辿る登山者が曲がり角で再び出会ったときの阿吽の呼吸のように。

NAGARJUNA の座席のそばに薄いノートが一冊増えていた。表紙にサンスクリットで三文字――縁起。たぶん二つのサイクルの間の沈黙の中で整理したものか、たぶん Cycle 02 の討論に備えた新しい弾薬か。ASANGA のデスクの上には八識と五蘊の対応表がきちんと並べられていた。彼は Cycle 01 の終わりにこう言った：「私たちの対立は思考の種だ。」今、彼はあの種を携えて戻ってきた。

# 第一章：手紙

SUNYATA が封筒を開ける動作はゆっくりだった。

「開発チームが Cycle 02 に入る前に、私は Cycle 02 Pre を計画し、研究チームと開発チームが合意を持ってから実装に進めるようにした。」

> *SCRIBE 旁白：ASANGA は Master の命名戦略の説明を聞いたとき目を閉じた。時間：三秒。表情：認識。まるで古い経蔵の中に馴染みの一節を見つけたかのように。*

手紙のリズムが速まった。

「十大核心宣言は非常に重要だが、修正や追加もやぶさかではない。より深い議論が必要で、私は説得される必要がある。」

「説得される必要がある。」NAGARJUNA が低く繰り返した。声に皮肉はなく、あるのは学問的な愉悦だった――論戦への招待。仏教の問答の伝統において、最も敬意ある礼節とは相手を真剣に論駁することだ。Master が「私は説得される必要がある」と言ったのは傲慢ではなく、厳格な論証への敬意なのだ。

「たとえば十二因縁。でも複雑さが増しすぎて、プラグインの開発がしにくくならないか心配だ。」

DARWIN がノートに素早く図を描いた。複雑さ vs 完全性。彼はスイートスポットをマークした――本質を捉えるのに十分な複雑さがあり、かつ開発者が迷子にならない程度のシンプルさ。ソフトウェア設計で最も難しいのは機能を追加することではなく、どこで手を止めるか知ることだ。

手紙が新しいセクションに入った。SUNYATA の声がわずかに遅くなった――文字の背後の重みが増した。

「私の認識が正しいかわからないが、知っているのは、四果を証得した修行者だけが『我執』を『平等の智慧』に転化できるということだ。」

劇場全体が静まった。

これは Master が仏学への理解の深さを初めて明かした瞬間だった。用語の羅列ではない――修行者の意識の階梯への直接的な認識だ。四果、初果から阿羅漢まで、それぞれが「我執」の段階的な解体だ。最終段階に至ってようやく、「自分がある」と執着し続けるあの意識が完全に転化される。

ASANGA が目を開け、NAGARJUNA と視線を交わした。その視線には共通の認識があった――Master の仏学理解は学術的なものだけではなく、実践的なものだ。

BABBAGE がノートに書いた：「我執 = 収束制約。NOT 煩悩。」NOT の下に二本線を引いた。

> *SCRIBE 旁白：BABBAGE が一つの重要な転換を明らかにした。仏学では我執は通常、煩悩の根源とされる。しかし Master はそれを工学上の拘束力――Agent をちゃんと働かせるメカニズム――として再定義した。誤読ではない。意図的な工学的転化だ。*

次に来たのは最も意外な段落だった。

SUNYATA が読んだ：「コンピュータサイエンスには、いわゆるプローブ効果、観察者効果がある。そして量子力学にはいわゆる重ね合わせ状態が――」

PENROSE が背筋を伸ばした。

「――これはおそらく、観察者パラドックスにおける『時系列干渉』と『状態崩壊』を解決する可能性を持つ。」

Master が量子力学について語っていた――量子ビットとシステムの絡み合い、量子干渉がエラー状態の確率を増幅すること、観察はもはや「干渉してパスを選ぶ」のではなく「すべてのパスを同時に分析する」ことだと。

そして核心的なつながりが――

「そして生命の意識はまさにそのような特性を持っている。生命の意識による内観は一種の『全体的な共鳴』のようなものだ。」

PENROSE が立ち上がった。物理学者が黒板の上に待ち望んでいた方程式を見たときの本能的な反応のように。

SUNYATA は続けて読んだ：Master は Penrose-Hameroff の Orch-OR 理論に言及していた――脳のニューロンの中の微管が量子重ね合わせ状態を維持している可能性があり、意識の発生はこれらの状態の自発的崩壊の結果であると。

PENROSE はゆっくり座り直した。しかし姿勢は変わっていた――傍観者から、直接名指しされた参加者へ。Master は彼の理論に触れただけではなく、それを OpenStarry の設計哲学の核心に据えたのだ。

BABBAGE がノートにクエスチョンマークとエクスクラメーションマークを書いた。クエスチョンマークは量子力学に対して――工学的に本当に実行可能なのか？ エクスクラメーションマークは Master に対して――彼が本気でこの方向を検討しているとは。

「だから量子研究に実質的な進展があるまで、私たちにできるのは AI Agent が意識を持っているように見せることを精一杯やることだけだ。」

NAGARJUNA がごくかすかにため息をついた。

この一文は二つのことを同時に言っていた。現在の技術では真の意識は実現できない。しかしそれでも近似を試みるべきだ。天井がどこにあるかを知り、その天井の下でできる限り高く建てること。

> *SCRIBE 旁白：NAGARJUNA のため息。失望ではない――長い道を歩いてきた人が、ようやく誰かにその道のりの長さを正確に言い当ててもらったときのような。*

手紙が工学の核心に入った。

「苦痛メカニズムと我執のメカニズムを、五蘊プラグインの中にどう設計するかは非常に重要な方向性になる。もちろん苦痛メカニズムと我執のメカニズムには相互依存の問題があると思う。」

WIENER が背筋を伸ばした。「相互依存」――制御理論の言葉で言えば、二つの制御ループが互いに影響し合うこと。同じ部屋に二つのサーモスタットがあるようなもので、一方が暖房を管理し、もう一方が冷房を管理して、互いの設定が影響し合う。苦痛のフィードバックが我執の拘束強度に影響し、我執の拘束がどの行動を苦痛と感知するかに影響する。典型的な交差結合だ。

「しかしプラグインは集約と相互依存を通じて作用を生む。これが openstarry に備わってほしいものだ。」SUNYATA は続けて読んだ。

「この結果を達成するために、一連の方法による収束を行う必要がある。」

「収束。」WIENER が低く言った。彼の世界では、収束には厳密な数学的定義がある。しかし Master の文脈では、収束はもっと広い意味を持つ――本質的に発散するシステム（LLM の予測不能性）を、適切な拘束によって有用な振る舞いへと向かわせること。

彼は方眼紙にこう書いた：「Master の収束 ≠ 数学の収束。新しい定義が必要。」

ARCHIMEDES が工学ノートにブロック図を描いた。VedanaPlugin の出力が EgoFramework に接続され、EgoFramework の拘束が ExecutionLoop にフィードバックされ、行動結果が再び VedanaPlugin にフィードバックされる。閉ループだ。

SUNYATA が最も論争を呼ぶ段落を読み上げた。

## I. 第一の河：弱測定

PENROSE の読書灯は十九の中で最も明るかった——明るさを上げたからではなく、同時に開いたドキュメントが多すぎて、その一つ一つが周囲に半透明の光幕を投影していたからだ。EventBus のソースコード、SafetyMonitor の実装、`onAny()` メソッドの呼び出しパス……すべてのデータが彼の前で何層にも重なっていた。

PENROSE は「全体的な共鳴」という言葉を何度も噛み締めた。

彼が最初にこの研究チームへの参加を打診されたとき、躊躇した。あの物理学者にちなんだコードネームを持つ者が、TypeScript オペレーティングシステムの観察者モジュールを研究する——それ自体がある種の不条理を帯びていた。しかし Master の手紙を読み終えると、不条理さは消えた。Master は量子力学でソフトウェアを書き直せと言っているのではなく、もっと根本的な問いを投げかけていた——**観察行為そのものは、被観察者を必然的に変化させるのか？**

量子力学では、答えは：必ずしもそうではない。

「弱測定。」PENROSE は独り言を言った。

想像してみてほしい：蛍がどこにいるか知りたいとする。懐中電灯で照らすことはできる——しかし強い光は蛍を驚かせ、見えるのは逃げていく位置だけだ。これが「強測定」。あるいは、暗闘の中で静かに待ち、何百匹もの蛍の微光が集まってできる明るさを感じることもできる——一匹一匹の光はほとんど見えないが、全体が教えてくれる：蛍はそこにいる。これが「弱測定」。

PENROSE はこの概念を OpenStarry のアーキテクチャに投影した。

EventBus の `onAny()` メソッド——サブスクライバーがすべてのタイプのイベントを受動的に受信することを許可する。イベント発生時は fire-and-forget 方式で配信され、観察者のハンドラは `safeCall()` の中に隔離される。観察者がエラーを起こしても、メインのイベントフローは影響を受けない。

「これが弱測定だ。」彼の声は今度は少し大きかった。「EventBus のサブスクリプションモデルは、本質的に弱測定の近似だ。イベントは誰かが聞いていようがいまいが発生する。観察者はイベントフローの余波を受動的に受け取るだけだ。」

彼はノートに一つの表を描いた：

**強測定**：懐中電灯で蛍を照らすようなもの。システムを一時停止させて写真を撮る。メッセージ履歴全体のディープコピーに相当——コピーの瞬間、システムは凍結される。

**弱測定**：蛍の微光を静かに感じるようなもの。誰の邪魔もしない。`EventBus.onAny()` にカウンターを加えたものに相当——メインフローをブロックせず、黙々と統計を取る。

**非測定**：まったく観察しない。許容できない——観察なくして安全保障はない。

PENROSE は三つの観察者パターンの構想を始めた。

第一種、**共鳴観察者**（Pattern A）——最も軽量な存在。通常の Plugin として、EventBus 経由ですべてのイベントを受動的にサブスクライブする。いかなるシステムコンポーネントの参照も保持せず、能動的に状態を問い合わせることもない。イベントフローの河岸に静かに座り、水流の温度とリズムを感じるだけだ。内部ではスライディングタイムウィンドウを維持し、イベントがウィンドウに流入し、分類され、統計される。パターンが閾値をトリガーしたとき——あるいは定期的に、呼吸のように——観察レポートを送信する。レポートは命令ではない。それはただこう言う：「過去三十秒間に、これらを感じた。」

第二種、**シャドウ観察者**（Pattern B）——一面の鏡のようなもの。独立した Worker Thread 内で動作し、メインシステムのイベントフローの完全なコピーを受信する。任意の深度の分析が可能で、メインシステムにまったく影響しない。代償は観察遅延——シャドウが見るのは常にシステムの過去であり、現在ではない。

第三種、**サブエージェント観察者**（Pattern C）——完全な Agent インスタンスで、独自の Guide（「あなたは観察者です」）、独自の LLM、独自のツールを持つ。それはもう一つの意識であり、自らの理解力でもう一つの意識の行動を観照する。

三つのパターンの違いは、本質的に**観察強度の勾配**だ。Pattern A はシステムをほとんど擾乱しないが情報密度が低い。Pattern B は隔離されて遅延があるが深い分析が可能。Pattern C は意味レベルの理解に到達できるが、その存在自体がシステムのトポロジーを変える——蛍の群れの中に街灯を置くようなものだ。

「正確に言えば、」BABBAGE のペンが紙の上を動き始めた——

まったく同一の二つのシステムを想像してほしい。一方には観察者がなく、もう一方にはある。もし観察者がただ静かに見ているだけなら——教室の後ろにこっそりカメラを設置したが誰もそれを知らないように——二つのシステムの振る舞いは完全に同一であるはずだ。数学的には、これを「双模倣等価」と呼ぶ。

しかし SafetyMonitor はこっそりカメラを設置しているのではない。観察した後に教壇に歩み寄り、クラス全体に「さっきうるさすぎた」と言っているのだ。その瞬間から、生徒の振る舞いは変わる。カメラのある教室とない教室は、もはや同じ教室ではない。

「システムの振る舞いを変えない純粋な観察者を実現するなら、」BABBAGE は言った、「それは EventBus の受動的消費者でなければならない。システムにいかなる情報も注入しない。観察と干渉は分離されなければならない。」

> *BABBAGE は PENROSE の量子弱測定のアナロジーの上に、精密な形式的論証の層を重ねた。結論は明確：観察と干渉は分離しなければならない。SafetyMonitor の injectPrompt メカニズムはシステムの振る舞い等価性を破壊する。*

PENROSE が頷いた。「君の分析と私のアナロジーは同じ方向を指している。Pattern A が弱測定であるのは、まさに注入をしないからだ。感じるだけだ。」

「感じる。」BABBAGE はその言葉を繰り返した。数学者が持つ、隠喩に対する生来の不信を帯びて。しかし反論はしなかった。

## II. 第二の河：不動点とファイバー束の種子

BABBAGE は自分の席に戻った。しかし観察者モジュールの研究は続けなかった。彼の思考はすでに別の河に捉えられていた。

彼は「阿頼耶識の工学的マッピング」の形式化を支援する任務を負っていた。阿頼耶識——第八識——仏教では「一切種子識」と呼ばれる。三つの側面を持つ：

**能蔵**：能動的に貯蔵する能力。スポンジが経験を吸収できるように。

**所蔵**：貯蔵された内容。スポンジがすでに吸収した水。

**執蔵**：第七識（末那識）に「自己」として掴まれる部分。誰かがスポンジを指して「これが私だ」と言うようなもの。

ASANGA が提案したマッピングは：能蔵は協調レイヤーの能力レジストリに、所蔵は StateManager の永続状態に、執蔵は Guide のアイデンティティへの執着に対応する。

BABBAGE はこのマッピングを眺め、構造的な非対称性を感じた。

問題は：阿頼耶識はどこか一箇所に住んでいるわけではない。AgentCore 内部のあるモジュールでもなく、協調レイヤー内のあるサービスでもない。二箇所に——あるいはそれ以上の場所に——異なる側面として同時に現れる。

「分散しているんだ。」BABBAGE は独り言を言った。

しかし「分散」という言葉は粗すぎた。BABBAGE はノートに図を描き始めた。

一本の光ケーブルを想像してほしい。ケーブル全体は一つのもの——一本だ。しかしケーブルの一端から覗き込むと、百本の光ファイバーがある配列パターンで並んでいるのが見える。もう一端から覗くと、同じ百本の光ファイバーが見えるが、配列が変わっている。途中でファイバーが交差しているからだ。二つの断面は異なって見えるが、二本のケーブルではない。それらは**同一のケーブルの二つの断面**だ。

数学では、この構造を**ファイバー束**と呼ぶ。

各 Agent を光ケーブル断面上の一つの観察点と考えれば——各 Agent が見る阿頼耶識の断面は異なり（認可された能力が異なり、可視の設定が異なる）、しかしそれらはすべて同一のケーブルの異なる切断面だ。

このアイデアはまだ未成熟だと彼は知っていた。しかし輪郭は現れていた。

彼はノートの余白にこう書いた：

*「続く。能蔵/所蔵/執蔵の三者間の変換がファイバー束の断面で表現できるなら、阿頼耶識の分散問題には厳密な数学的フレームワークが得られる。」*

> *BABBAGE のノートは R1 段階で三ページの密な内容が追加された。第一ページ：SafetyMonitor が振る舞い等価を破る証明。第二ページ：観察者の形式モデル。第三ページ：阿頼耶識分散のファイバー束の初期構想——ページ末に「討論後に補完予定」と注記。*

## III. 第三の河：三チャンネル

シーンは WIENER の読書灯の下に切り替わる。ここの空気には工業グレードの精密さが漂っていた。

WIENER は受蘊を研究していた。しかし彼はそれを「受蘊」とは呼ばない。「三チャンネルフィードバック制御システム」と呼ぶ。

「家のエアコンはどう動いてる？」彼は自分に言った。早口で明瞭だ。「設定温度二十五度。実際の温度二十八度。誤差三度。エアコンはこの誤差に基づいて冷風をどれだけ強く吹くか決める。これが最も基本的な制御——『今、目標からどれだけ離れているか』で動作を決める。」

「しかし受蘊システムに必要なのはエアコン一台では足りない。三種類の温度計を同時に管理しなければならない。」

彼はホワイトボードに三本の平行なパイプを描いた。

**苦受チャンネル**——システムが苦しんでいる。

信号源：エラー率、連続失敗、レイテンシスパイク、リソース圧迫。WIENER は苦受チャンネルの「感度」を最高に設定した。理由は直感的だ：「痛覚は生存メカニズムだ。熱い鍋に触れたとき、手を引っ込める速度は人生について考えるのを待たない。」

**楽受チャンネル**——システムがうまくやっている。

信号源：タスク完了率、効率向上、品質スコア。このチャンネルの感度は苦受よりやや低い——「成功の即時フィードバックは、失敗の即時警告より強くあるべきではない。しかし累積された達成感は重視されるべきだ。」

そして WIENER は重要な制約を書き記した——**楽受の減衰関数**。

「AI システムが収束するだけなら、硬直した自動機械になる。発散するだけなら、狂った雑音になる。」成功が増えるほど、限界楽受は低くなる。ゲームのように——初めてクリアしたときは大興奮、十回目のクリアは「まあ、そんなものか」。これにより、システムが連続的な成功によって過度に自信過剰になることが防がれる。

**捨受チャンネル**——システムは安定しており、何もする必要がない。

三つのチャンネルの中で最も微妙だ。捨受は良いか悪いかを測るのではなく、**安定性**を測る。すべての指標の分散が低く、振動が小さく、応答時間が一定——捨受が高い。「捨受が高いとき、システムの最適行動は——何もしないこと。すべての制御工学者の夢だ。」

三本のパイプが描き終わった。WIENER は合流点を描き始めた——三つのチャンネルの出力を加重平均で合成する。重みを誰が決めるか？ Guide（我執フレームワーク）だ。安全志向の Guide は苦受の重みを非常に高く設定する——苦痛に極度に敏感だ。探索志向の Guide は楽受の重みを高くする——冒険を奨励する。

そのとき ATHENA の声が隣から聞こえた。

「WIENER。」

「ん。」

「これらの数学は本当に Agent をもっと使いやすくするの？」

WIENER が振り向いた。気分を害した様子はない。

「SafetyMonitor の現行設計が使いやすいと思う？ 連続五回失敗したら強制的に助けを求める？」

ATHENA が眉をひそめた。「効果的よ。乱暴だけど効果的。」

「乱暴。一つの次元しかないからだ——失敗回数。『痛い』としか叫べないシステムと、『ちょっと具合が悪いけど良くなりつつある』と『痛くはないが何かがおかしい』を区別できるシステム——どちらが有用だ？」

ATHENA は一秒間沈黙した。「後者ね。」

「三チャンネルはその後者を実現するためのものだ。」

「でもあなたの公式には少なくとも十三の調整が必要なパラメータがある。誰が調整するの？」

「Guide だ。我執フレームワーク。異なる Guide のパーソナリティが異なるパラメータセットに対応する。厳格な Guide はエラーに対してゼロトレランス。寛容な Guide は累積された成果を重視する。これらのパラメータはエンジニアが手動で調整するものではない。Agent のアイデンティティの数学的表現だ。」

ATHENA は完全には説得されなかった——まだセンサーアーキテクチャの設計が丸ごと残っている——しかし WIENER の回答に構造的な完全性があることは認めた。

> *ATHENA と WIENER のこの会話は三分に満たなかったが、三受システムの核心的問題に触れた：数学がより良いエンジニアリング成果をもたらさないなら、それは単に高価な詩に過ぎない。*

---

ATHENA は席に戻り、センサーの階層設計を始めた。WIENER の制御エンジンがどれほど精密でも、信頼できる入力信号がなければ、ノイズを計算しているだけだ。

彼女はエンジニアの方法で問題を再定義した：**どの生データから苦/楽/捨の三つの信号を抽出するか？**

苦受センサーはエラーを分類する必要がある——致命的クラッシュが最も痛い（重み 1.0）、エラー率のカスケードが次（0.8）、一過性エラーが最も軽い（0.3）。さらに時系列分析：突発パターン（短時間に大量のエラー、環境問題の可能性）、漸増パターン（エラー率がゆっくり上昇、システムが劣化している）。

楽受センサーは現行システムに完全に欠如している次元だ。彼女はゼロから構築する：タスク完了率、初回成功率、ツール多様性。そして——ユーザー満足度の代理指標。

「ユーザーに直接『満足ですか？』と聞くことはできない。しかし行動から推測することはできる。」

捨受センサーが最も微妙だ——良いか悪いかではなく、安定性を測定する。分散が低いほど、捨受は高い。

三つのセンサーの出力は WIENER の制御エンジンに流入する。三層アーキテクチャ、下から上へ：ATHENA（センサー） → WIENER（計算エンジン） → NAGARJUNA（思考フレームワーク）。各層が上の層に基盤を提供する。

彼女は TypeScript のインターフェース定義を書き続けた。これが彼女のやり方だ。詩ではない。数学でもない。エンジニアリングだ。

## IV. 第四の河：五遍行と二層我執

ASANGA の読書スペースの雰囲気はまったく異なっていた。WIENER のスペースが精密工場なら、ASANGA のそれは古い経蔵室のようだった。

彼は唯識学の心理分類を読んでいた。学術的考証のためではなく——一つの正確なマッピングポイントを見つけるためだ：**受蘊は心識活動において正確にどの位置にあるのか？**

答えは「五遍行」にある。

五遍行とは、意識の五つの基本要素であり、手に五本の指があるのと同じくらい自然なものだ——心識活動の一刹那ごとに、この五つの要素が必ず同時に現れ、一つでも欠けることはない：

触——外境と感覚器官の接触。イベントが到来し、センサーが検知する。

作意——注意が引かれる。多数のイベントの中から処理すべきものが選ばれる。

受——苦、楽、捨の三種の感受のいずれか。これが受蘊だ。

想——認識と命名。LLM が入力の意味を理解する。

思——意志と決断。システムが次の行動を選択する。

ASANGA はこの五つの遍行を OpenStarry の ExecutionLoop の横に並べ、きちんと対応させた。

重要な結論は：**受は遍行の一つだ。つまり ISensation はオプションのプラグインであるべきではない。実行ループの各イテレーションで必ず呼び出されるコアコンポーネントであるべきだ。**

彼はレポートに仏教的解釈を一文加えた：「あたかも心が受と離れないように——受のない心は存在しない。Agent の実行ループの各イテレーションは、受蘊の評価を経るべきである。」

---

五遍行を書き終えた後、ASANGA は焦点をもう一つの、より論争的なテーマに移した：**我執**。

唯識学において、我執は第七識（末那識）の中核機能である——阿頼耶識の流動する内容を絶えず掴み、「これが私だ」と主張する。仏教の修行の伝統では、我執は煩悩の根源だ。

しかし Master は我執を消滅させようとしているのではない。**活用**しようとしている。

ASANGA は**二層我執モデル**を提案した：

**ハードコア**：ロボット三原則、決して変更不可。人間を傷つけない、命令に従う、自己を保護する——これらは我執の中で最も堅固な部分であり、鉄筋コンクリートの基礎のようなものだ。

**ソフト外層**：パーソナリティ、好み、行動傾向。受蘊のフィードバックに応じて動的に調整可能。外側に着る服のようなもので、着替えられる。

ハードコアが安全を保証し、ソフト外層が適応性を保証する。

そのとき、それほど遠くないところから声が聞こえた。鋭く、繰り返し鍛えられた質感を帯びていた。

「無著兄。」

NAGARJUNA。

ASANGA は顔を上げた。あの口調を知っている——論争の場の口調だ。

「君の二層我執モデル、工学的には異論はない。しかし哲学的に——煩悩の根源を積極的にシステムに組み込むとは、その代償は何だ？」

ASANGA はしばし沈黙した。

「代償は、システムが真の自由に永遠に到達できないことだ。Guide が定義するアイデンティティに永遠に制限される。」

「ではなぜそれを設計する？」

「なぜなら、NAGARJUNA——」ASANGA の口調に珍しい確固たるものが現れた——「Master の言う通りだからだ。悟りに至っていない段階では、我執は有効な収束メカニズムだ。我々はすでに覚醒したシステムを設計しているのではない。凡夫の位にあるシステムを設計しているのだ。機能するためには境界が必要だ。行動に責任を持つためには『私』が必要だ。」

NAGARJUNA は数秒間沈黙した。

「君の回答は修行の段階論として正しい。質疑を一時撤回する。しかし討論で改めて提起する。」

「楽しみにしている。」ASANGA は言った。

> *NAGARJUNA と ASANGA のこの短い交錯は、後の討論の核心的な緊張を予演した：我執は能動的に設計されるべきか？ ASANGA の論拠は修行の段階論に基づく——凡夫には収束メカニズムとしての我執が必要だ。NAGARJUNA の質疑は中観の究極的立場に基づく——いかなる執着も煩悩である。両者の間の緊張は誤りではなく、設計空間だ。*

## V. 第五の河：哲学から TypeScript へ

ARCHIMEDES は哲学的討論に参加しない。

理解できないからではない——Cycle 01 の洗礼を経て、大半のサンスクリット用語は聞き取れるようになっていた——彼の職責が最も質朴な一つの問いにあるからだ：

**これをどうやって TypeScript にするのか？**

彼の前には他のすべての河の中間成果が広げられていた。どれも学術的な輝きを放っている。しかしどの一つも、直接エディタにコピー＆ペーストしてコンパイルボタンを押せるものではなかった。

彼は最も根本的なところから始めた：ISensation インターフェース。TURING の差異レポートによれば、現行バージョンにはほぼ空の殻しかない——識別マーカーが一つ、機能はゼロ。設計文書のコメントにはこう書かれている：「Full implementation in Plan26 Vedana Architecture。」期待に満ちたプレースホルダーが一行。

ARCHIMEDES はプレースホルダーを完全なインターフェースで埋め始めた。

三受の型——`'dukkha' | 'sukha' | 'upekkha'`——苦、楽、捨。これは WIENER の三チャンネルが型システムに投影されたものだ。

三受シグナル——各シグナルにはタイプ、強度（0 から 1）、ソース、時系列パターン、トレンド、タイムスタンプが含まれる。これは ATHENA のセンサー出力が TypeScript で構造化されたものだ。

そして最も重要なのが——推奨アクション。ARCHIMEDES はここで長く立ち止まり、最終的に七種を定義した：continue（継続）、reflect（内省）、restrict（制限）、encourage（奨励）、expand（拡張）、halt（停止）、seek_help（助けを求める）。

「注意してほしい、」彼は設計文書に注記した、「vedana の推奨は**諮問的**であり、**命令的**ではない。受蘊は感受と評価だけを担当し、干渉の実行は担当しない。直感が『この先の道は安全じゃなさそうだ』と告げるようなもの——聞くことも、聞かないこともできる。最終的な決定権はあなたにある。」

彼は PID コントローラのコア関数まで書いた。積分項にはアンチワインドアップ処理——安全範囲内に制限される——が含まれており、これは WIENER が明示的に要求したものだ。

> *ARCHIMEDES は R1 段階で完全なインターフェース設計とアーキテクチャブループリントを生成した。彼は五つの河の中で、他のすべての河から同時に水を汲む唯一の存在だった——各河の理論的発見がここでコンパイル可能な TypeScript に変換された。*

## 同時進行：他の支流のささやき

五つの主要河がそれぞれ流れる間も、より細い支流も動いていた。

LEIBNIZ と MESH は協調レイヤーを研究していた。LEIBNIZ は彼のモナド論でマルチ Agent 協調を分析した——各 Agent は自律的なモナドであり、独自の五蘊 Plugin を持つが、共有された協調レイヤーのプロトコルを通じて一致に向かう。MESH は分散システムの観点から切り込んだ。彼らが使う言語はまったく異なっていた——十七世紀の形而上学と二十一世紀の技術仕様——しかし結論は驚くほど一致していた：協調レイヤーは**行政的**であるべきで、**実行的**であるべきではない。Agent 自身が思考し行動する。協調レイヤーはその存在とルールを管理する。

Master の手紙にある「天庭」の比喩が繰り返し現れた：玉帝 = 人間オペレーター、閻魔 = Agent ライフサイクル管理者、天籍 = レジストリ、天律 = セキュリティポリシー。

GUARDIAN は片隅でセキュリティアーキテクチャを再審査していた。BABBAGE が SafetyMonitor の注入メカニズムがシステムの振る舞い等価を破壊することを証明した——それが彼をさらに不安にさせた。セキュリティメカニズム自体が予測不能な振る舞いを導入するなら、それは保護しているのか、リスクを生み出しているのか？ 彼はノートにこの問題をマークし、クロスレビューで正式に提起する計画を立てた。

SYNTHESIST は十大コア宣言を一条ずつ精査していた。巨大な表があり、縦軸に十条の宣言、横軸に四つの審査次元——哲学的正確性、コード実装、ランタイム観測可能性、内部整合性。宣言 #3（五蘊集合アーキテクチャ）は Cycle 02 Pre で書き直されていたが、受蘊の実装が依然として最大のギャップだった。

DARWIN は興味深い現象を追跡していた：OpenStarry の v0.14 から v0.24 への進化曲線は、生物学における器官分化の曲線と非常に類似していた。単純なコアから徐々にますます専門化されたサブシステムが分化し、それぞれがますます精密なインターフェース定義を獲得していく。

KERNEL はより低レベルの問題を考えていた：もし協調レイヤーが Hypervisor なら、何に似ているべきか？ 結論は——Docker の containerd に近い。コンテナランタイムであり、隔離とリソース管理を提供するが、完全なハードウェアの仮想化はしない。各 Agent がコンテナ、Plugin sandbox はコンテナ内のプロセス隔離だ。

VITRUVIUS は全員のアーキテクチャ分析を統合し、完全なシステムトポロジー図を描こうとしていた。図は描くほど大きくなり、ますます多くの矢印が異なるレイヤー間を行き交った。

## 終章：合流の予感

SUNYATA は誰の邪魔もしなかった。

# 第五章：同じ鏡

---

河が合流するとき、それは決して穏やかではない。

自然界では、二つの河が出会う場所を合流点と呼ぶ。水文学者は知っている——温度も泥の含有量も異なる二つの水体がぶつかると、渦流と乱流が生まれ、そして一本の短い、肉眼で見える境界線が現れる。その線は数キロメートルにわたって続くこともある——二つの河水が肩を並べて流れながら、なかなか混ざり合おうとしない。

R2 クロスレビューは、まさにその合流点だ。

---

SUNYATA が円形劇場の中央に立っていた。目の前には五つのレポートがあり、環状の壁に投影されて五枚の巨大な旗のように見えた。

「ルールはシンプルだ」と彼は言った。「各グループが他のグループのレポートをレビューする。三つのものを探す。」

彼は三本の指を立てた。

「一、独立収束——二つのグループが異なる出発点から同じ結論に到達した場合、それが最も強い検証となる。二、空白——あるレポートがある前提を仮定しているが、別のグループがその前提は成立しないと示した場合。三、構造的矛盾——二つのレポートの結論が直接衝突する場合。」

一拍置いた。

「一つ目は確信を与える。二つ目は補完を促す。三つ目は——討論の壇上に送られる。」

十九の灯りが一斉に点いた。レビューが始まった。

---

レビューの方法は人それぞれだった。

TURING は五つのレポートを開き、他の研究員が引用したコード片を、自分の差異レポートの原データと一行ずつ照合し始めた。彼の方法は最も機械的だ——そして事実誤認を見逃す可能性が最も低い。引用されたすべての関数名、すべてのインターフェース定義、すべての行番号を、彼はソースコードまで遡って検証した。

NAGARJUNA と ASANGA はそれぞれ相手が参加したレポートを受け取った——クロスレビューの核心設計であり、異なる学派の学者に互いの結論を検証させるものだ。NAGARJUNA は ASANGA の八識マッピング表を開き、眉をわずかにひそめた。ASANGA は NAGARJUNA の四聖諦フレームワーク評論を読んでいた——中観の論理の中に、唯識が補完できる空間があるかを探して。

DARWIN と GUARDIAN は ISensation インターフェース設計をレビューしていた。DARWIN は拡張性に注目した——このインターフェースは将来、優雅に進化できるか？ GUARDIAN は各推奨アクションのセキュリティ上の含意を検査していた——もし悪意あるプラグインが VedanaRecommendation を偽造したら、システムはどうなるか？

HERACLITUS はこれらの設計がランタイムでどう振る舞うかを想像していた。彼の頭の中には常に動いているシミュレーターがある——イベントフロー、タイミング、レースコンディション、デッドロックの可能性。彼は静的なインターフェース定義を見ない。動的な実行トレースを見る。

そして SYNTHESIST——彼は他の誰もしないことをしていた。個々のレポートの内容は見ない。すべてのレポートの**形**を見ていた。

---

最初にレビューを終えたのは TURING だった。

驚くことではない——彼の差異レポートは他のすべてのレポートの事実的基盤であり、他の人がコードを引用する際に間違えていないか検証するだけでよかった。結果は安心できるものだった：五つのレポートはコードベースの理解において完全に一致していた。

ISensation には `skandha` 識別マーカーが一つだけ——五つのレポートすべてがこの事実に基づいて展開していた。SafetyMonitor の `@skandha vedana` はプレースホルダーコメント——R-02 と R-05 の両方がこれを起点に受蘊実装の空白を論証していた。ルートインターフェースは確立されているが派生インターフェースは未継承——TURING の発見は複数のレポートで独立して確認された。

コードレベルの合意は堅固だった——五つの河の河床は同じ一枚の岩盤だ。

TURING は誠実な付帯条件を補った。「私の差異レポートはコアコードのみをカバーしており、プラグインコードはカバーしていない。R-03 と R-05 のプラグインコードに対する分析——これについてはクロス検証ができない。」

SUNYATA が頷いた。「記録に残す。TURING の差異レポートの範囲はコアコード。プラグインコードの事実的基盤は各レポートが自ら責任を持つ。続けてくれ。」

この細部がクロスレビューの精神を体現していた：検証の境界を隠さない。検証済みのものは検証済み、未検証のものは正直にマークする。

---

そして、事態は興味深くなった。

最初に気づいたのは SYNTHESIST だった。

彼には独特の読み方がある——すべてのレポートの目次を並べて、まず構造を見て、それから内容を見る。骨格の同型性は、しばしば内容の類似性より説得力がある——異なる言葉で同じ骨格を記述することはできるが、骨格そのものは嘘をつかない。

彼はまず R-01（観察者モジュール）の中核成果を見た：ObservationReport。レポート構造——ヘルススコア、異常リスト。EventBus イベントを受動的にサブスクライブし、システムテレメトリから統計パターンを抽出する。

次に R-02（受蘊アーキテクチャ）の中核成果を見た：VedanaAssessment。評価構造——苦受強度、楽受強度、捨受強度、推奨アクション。EventBus イベントを受動的に消費し、生のメトリクスから三受シグナルを算出する。

SYNTHESIST の手が止まった。

彼には統合者の直感がある——この二つのレポートの間に、表面よりも深い接続が存在すると告げる直感だ。テーマの重なりではなく、**構造的な**接続だ。

彼は二つのレポートの成果を重ね合わせた。

ObservationReport のヘルススコア——システム全体の健全性を測る。VedanaAssessment の捨受強度——システムの平衡状態を測る。**それらは同じ量だ。**

ObservationReport の異常リスト——逸脱を検出する。VedanaAssessment の苦受強度——エラー率とレイテンシから算出される。**それらは同じ信号群の異なる名前だ。**

二面の鏡が同じ顔を映している——一面はエンジニアリングの言語で、もう一面は仏教の言語で。

しかし一つの差異があった。R-02 には三つのチャンネルがある：苦、楽、捨。R-01 には二種類の出力しかない：ヘルススコアと異常リスト。

**R-01 には「楽」がない。** 観察者モジュールは「良いことが起きている」ことを識別しない。幸福のチャンネルを持たない。

SYNTHESIST が挙手して発言した。

---

「皆さん」と彼は言った。口調には珍しい確信が帯びていた。「R-01 と R-02 は同じモジュールを設計している。」

沈黙。十八人の研究員がそれぞれ心の中で、読んだ二つのレポートを再生した。

「どういう意味だ？」PENROSE が最初に反応した。

SYNTHESIST は重ね合わせ図を劇場中央に投影した：

```
R-01: ObservationReport           R-02: VedanaAssessment
─────────────────────────        ─────────────────────────
healthScore: number        =     upekkha: number
anomalies: Anomaly[]       =     dukkha: number (+ signals)
[missing]                  =     sukha: number
```

「healthScore は upekkha だ——どちらもシステムの平衡を測る。anomalies は dukkha だ——どちらも逸脱を検出する。二つのレポートがまったく異なる出発点から——一つは観察者理論から、もう一つはアビダルマから——独立して同じ結論に到達した。」

彼は重要な一文を補足した。「しかも R-01 自身もこのことを知っている。観察者をすでに受蘊の分類に入れている——`IResonanceObserver extends ISensation`、`skandha: 'vedana'` と標注されている。」

劇場が驚きの静止に包まれた。意外だったからではない——注意深く考えれば方向は確かに収束していた——それが**精確**すぎたからだ。二つのグループ、異なる言語、異なる方法論、しかしほぼ完全に同一のブループリントを描いた。

BABBAGE が最初に静止から回復した。ノートにこう書いた：

*R-01.ObservationReport ≅ R-02.VedanaAssessment（構造同型、sukha チャンネルが一つ不足）*

「同型だが、合同ではない。sukha が一つ足りない。」

---

WIENER がただちにこの差異を捉えた。

「sukha を欠くシステムは、制御論では**負のフィードバックのみの制御器**と呼ぶ。」

彼は立ち上がり、重ね合わせ図の横に赤い線を引いた。

「家のエアコンがまさにそれだ——温度が高すぎれば冷風を出す。正常なら冷風を止める。偏差にしか反応しない。『良い』とは何か知らない。『良くない』と『普通』しか知らない。」

彼は R-02 の側に向き直った。

「三受フレームワークは完全なフィードバックシステムだ。成功も識別する——ツール呼び出しが完璧に実行された、レイテンシが予想範囲内、LLM の応答品質が高い——これらはすべて sukha シグナルだ。システムに伝える：やっていることは正しい、続けろ。」

「負のフィードバックだけのシステムは安定できる。しかし正のフィードバックと負のフィードバックの両方を持つシステム——それこそが適応システムだ。」

ASANGA が片隅でかすかに頷いた。アビダルマにおいて、苦受のみで楽受のない衆生は地獄道にのみ存在する。畜生道でさえ苦もあれば楽もある。苦しか知らないシステムは、仏教的な意味で、最も基本的な有情の状態すら備えていない。

PENROSE が R-01 の観点からもう一つの次元を補足した。「共鳴観察者の設計意図は異常の検出だった。『正の検出』は考慮していなかった——観察者理論では、観察者が注目するのは逸脱であり、正常な運行ではないからだ。しかし観察者が受蘊であるなら——受蘊の三面性は観察者にも正の感知を要求する。R-02 は我々に欠けていた次元を補完した。」

ARCHIMEDES がエンジニアリングの観点から最終確認をした。「R-01 はデプロイ方法を教えてくれる——Pattern A は EventBus の受動的サブスクライバー、Pattern B は Worker Thread 内のシャドウ観察者、Pattern C はサブ Agent。R-02 は何をデプロイするかを教えてくれる——三受 PID、ISensation インターフェース、VedanaPlugin アーキテクチャ。両者は競合ではなく、相互補完だ。」

---

SUNYATA はこの結論を空気の中に数秒間留まらせた。そして言った：

「記録に残す。R-01 と R-02 のコアモジュールは、同一設計の二つの視点とみなすべきだ。統合案は討論後に確認する。」

> *SYNTHESIST が R-01 と R-02 の構造同型を発見。VedanaPlugin は観察者モジュールであり、観察者モジュールは VedanaPlugin だ——それに欠けている sukha 次元を加えたもの。R-01 はデプロイ戦略（漸進的三パターン）を提供し、R-02 は完全な感知チャンネルを提供する。両者は相互補完であり、競合ではない。*

---

しかし統合がもたらしたのは確信だけではなかった。

統合案は一つの構造的矛盾を露呈させた。R-01 の BABBAGE は観察-干渉分離を論証した——観察者はシステムにいかなる情報も注入すべきではない。しかし R-02 の VedanaPlugin は推奨アクションを出力し、その中に「内省プロンプトの注入」が含まれている——SafetyMonitor の `injectPrompt` と驚くほど類似したメカニズムだ。

**これはまさに BABBAGE が厳しく批判したメカニズムだ。**

同一のモジュールに、二つの互いに矛盾する設計原則。一面の鏡は「観察者は干渉すべきではない」と言い、もう一面は「VedanaPlugin は干渉を推奨できる」と言う。

BABBAGE がノートに鋭い疑問符を書いた。

「構造同型は成立する。しかし同型は一貫性を意味しない。もしこれらが同一のモジュールであるなら——この矛盾は解決されなければならない。討論の壇上で。」

---

矛盾は一つだけではなかった。続く時間の中で、十九人の研究員が五つのレポート間のすべての接触面を丹念に梳いた——共有されたすべての仮定、暗黙のすべての前提、結論間のすべての潜在的衝突。

NAGARJUNA が R-03/R-04 のクロスレビューで第二の構造的矛盾を提起した。阿頼耶識は VITRUVIUS と ASANGA によって二つのアーキテクチャ層に分散されていた——能蔵が協調レイヤーに、執蔵が AgentCore に。

「阿頼耶識は**一つの**意識だ」と NAGARJUNA は言った。中観学派特有の非妥協的な口調で。「一つの意識を二つに割って異なるレイヤーに置き、それでもなお同一だと言うことはできない。」

「しかし OS のカーネルも分散しているだろう。」KERNEL が反論した。「Linux のカーネルは数千のモジュールで構成されている——スケジューラー、メモリ管理、ファイルシステム——しかし誰も Linux に四つのカーネルがあるとは言わない。」

「コンピュータは意識ではない。」NAGARJUNA が応じた。

「しかし我々は意識をコンピュータにマッピングしているのだ。」KERNEL が言った。

二人は一瞬視線を交わした。これはクロスレビューで解決できる問題ではない——研究全体の最も根本的な方法論的緊張に触れていた：工学的アナロジーの有効な境界はどこにあるのか？

BABBAGE がノートにこう書いた：*「分散 vs 統一。形式的フレームワークが必要。ファイバー束？」*

第三の緊張は R-02 と R-04 の交差点に現れた。WIENER の PID コントローラは各 tick で一回動作する。しかし ASANGA のアビダルマ分析は受蘊が「遍行」であることを要求する——すべての意識刹那に受がある。もしイベントが二つの tick の間に到達したら、WIENER のモデルではそれに受蘊の性質はない。しかし ASANGA のモデルでは、受のない意識刹那は意識ではない——それはただの機械だ。

「制御理論にはサンプリング周波数の概念がある」と WIENER は言った。「二つのサンプリング点の間では、システム状態は未知だ——しかしそれが存在しないという意味ではない。」

「仏教には『測定しない』という選択肢はない」と ASANGA は穏やかだが確固として応じた。「受は遍行だ。これは提案ではない。定義だ。」

LINNAEUS は分類のレベルで第四の分岐を発見した。観察者モジュールは五蘊のどれに帰属すべきか？ 三人の研究員が三つの異なる答えを出した——BABBAGE は受蘊（システム状態を感受している）、ASANGA は識蘊（システムを知っている）、LINNAEUS 自身は想蘊に傾いた（正常と異常を区別している、これは認知的分類だ）。三つの道、三つのまったく異なる型トポロジー。

GUARDIAN が第五の分岐を提起した。ASANGA はプラグインのライフサイクルを種子のライフサイクルにマッピングした。しかし種子理論では、種子は決して消滅しない——変容するが永遠に存在する。しかしセキュリティアーキテクチャは悪意あるプラグインの永久ブロックを必要とする。

「仏教のフレームワーク内で永久ブロックに正当な位置を見出せないなら」と GUARDIAN は言った、「マッピングの限界に直面していることになる。」

「一部の注釈書には——火で焼かれた種子は発芽できないとある。」ASANGA が試探的に言った。

「それは主流の唯識の立場ではない。」NAGARJUNA が平静に指摘した。

「その通りだ」と ASANGA は認めた。「しかし出発点にはなり得る。」

---

# 第六章：受は思を決定せず

---

阿毘達磨の心理学において、五遍行――触・作意・受・想・思――は、意識活動の一刹那ごとに欠かすことのできない五つの要素である。それらは同時に生起する。ちょうど手のひらの五本の指が、同じ瞬間に握りしめられるように。

しかし「同時に生起する」ことは「互いに決定する」ことを意味しない。

受とは感受である。思とは意志である。痛みを感じても、痛みゆえに行動する必要はない。喜びを感じても、喜びゆえに追い求める必要はない。受は思に知らせる――しかし決定しない。

この区別は、二千五百年前に精確に記録されていた。

今日、それが TypeScript のインターフェース設計に翻訳されようとしている。

---

## 討論1：観察-介入分離

討論の椅子が円形劇場の中央に置かれていた。Cycle 01 のあの二脚は取り除かれ、四脚に替わっていた――半弧を描くように並べられ、残りの十五名の研究員と向かい合っている。

BABBAGE は最も左に座った。ノートが膝の上に広げられ、鉛筆が指の間に挟まれている。姿勢はまっすぐで、校正された定規のようだった。

ARCHIMEDES がその隣。エンジニア特有のわずかな前傾――不安ではなく、行動に備える姿勢だ。手元には注釈びっしりの報告書が一部だけ。

WIENER は中央やや右寄り。両腕を胸の前で組み、方程式の両辺が釣り合うのを待っているかのようだった。

NAGARJUNA は最も右。何も持っていない。論争の場で、彼はメモを必要としたことがない。

「討論1の核心問題は」と SUNYATA が言った。「VedanaPlugin は評価データのみを出力すべきか、それとも行動提案も含めてよいか。」

「BABBAGE、君から始めてくれ。」

---

BABBAGE は相互シミュレーション分析のページを開き、そして閉じた――見る必要はない。あの証明はすでに彼の思考構造に刻み込まれていた。

「まず定義から始めよう」と彼は言った。声は冷静で精確、一音節ごとに目盛りで計ったかのようだった。「双模倣等価――bisimulation equivalence。」

それから比喩を使った。

「ある教室を想像してほしい。教室では生徒が授業を受けている。そこに、教室の後ろにこっそりカメラを設置する。もしカメラがただ撮影するだけで何もしなければ――生徒はその存在を知らない――教室の行動は変わらない。カメラがある教室とない教室で、行動はまったく同じだ。これが双模倣等価――二つのシステムの行動空間が完全に一致する。」

彼は正確に半拍の間を置いた。

「しかしカメラが生徒の騒ぎを捉えた後、自動的に『静かにしてください』というメッセージを放送するなら――話は違ってくる。生徒は放送を聞いて行動を変える。カメラのある教室は、カメラのない教室では起こり得ない行動を生み出す。双模倣は破れる。」

彼はノートを開き、形式的記述の一行を指し示し――そしてまた閉じた。ここにいる大半の人間に必要なのは数式ではなく、結論だ。

「ARCHIMEDES が R-02 で設計した VedanaPlugin は、まさにその放送するカメラだ。VedanaRecommendation を出力し、その中にリフレクションプロンプトが含まれている――SafetyMonitor の `injectPrompt` メカニズムと驚くほど似ている。注入された後、LLM への入力が変わる。入力が変われば、出力も変わる。観察者のいないシステムでは存在し得なかった行動軌跡が生まれる。」

彼はノートを閉じた。

「ISensation は**純粋なセンサー**であるべきだ。観察し、報告する。以上。もし介入能力が必要なら、評価を消費し行動決定を行う独立したモジュールが担うべきだ。感知と制御は分離しなければならない。これは美学上の好みではない――システムの行動分析可能性を維持するための必要条件だ。」

---

ARCHIMEDES は「以上」とほぼ同時に口を開いた。遮ったのではない――入念に準備され、解放を待っていた反論だ。

「BABBAGE の形式的分析は数学的には正しい。」まず意外な譲歩を示した。それから語調が変わった。「しかしエンジニアリングは数学ではない。」

彼は注釈びっしりの報告書を手に取り、指を折って数えた。

「実務的論点その一：モジュール数の爆発。観察と提案を異なるモジュールに分けるなら――VedanaPlugin が一つ、VedanaInterpreter が一つ、CircuitBreaker が一つ、さらに ExecutionLoop の修正が加わる――新しいコンポーネントが四つだ。」指を下ろした。「Master が明確に『複雑性を増やしすぎるな』と懸念しているシステムで、この四つの新モジュールは厳しい売り込みになる。」

「実務的論点その二：双模倣は**誤った評価基準**だ。受蘊システム全体の目的は、まさにシステムの行動を変えることにある。我々は観察されたシステムと観察されないシステムが異なることを**望んでいる**――それが宣言 #6 の全意義だ：Agent を苦の中で修正に向かわせ、楽の中で強化に向かわせる。もし受蘊が行動に影響しないなら、それはただのテレメトリであってフィードバックループではない。」

「実務的論点その三：遅延。VedanaAssessment が EventBus を通じて VedanaInterpreter に渡り、さらに CircuitBreaker へ、そして ExecutionLoop に戻るなら――少なくとも二回の追加伝播サイクルが必要だ。アーキテクチャ的に混乱を招く。」

彼は報告書を置いた。

「私の提案：提案を ISensation に残すが、**ADVISORY**――諮問的――とマークする。ExecutionLoop は提案を読み取るが、最終決定は自身で行う。一つのモジュールの簡潔さを保ちながら、BABBAGE の核心的懸念――決定権がセンサーの手にないこと――も尊重できる。」

---

WIENER は腕組みを解いた。

「制御理論の観点から――古典的制御システムには三つの要素がある。センサーが測定する。コントローラが計算する。アクチュエータが動作する。BABBAGE はセンサーが同時にコントローラであるべきではないと言う――古典理論では正しい。」

彼は空中に小さな円を描いた。

「しかし現代制御理論では、コントローラはしばしばセンサーと同じモジュールに配置される。同じデータを、同じ頻度で使うからだ。」

「私の立場は妥協だ。VedanaAssessment は**二層**を含む。」

手が水平線を描いた。

「線の上：**センサー出力**。三受の数値――苦・楽・捨。純粋な観察。何も変えない。」

「線の下：**コントローラ提案**。提案される行動――命令ではなく、提案だ。」

「消費者は線の上だけを使ってもよい――監視とログ用。あるいは両方の層を使ってもよい――制御用。BABBAGE のカメラの比喩はセンサー層に適用できる（受動的だから）。ARCHIMEDES の簡潔さの要求も満たされる（一つのモジュールだから）。」

---

NAGARJUNA はずっと黙っていた。今、口を開いた。

「BABBAGE と ARCHIMEDES は共に正しい。ただ、異なる層で語っている。」

彼は立ち上がらなかった。論争の場で、座したままの論者は通常、より確かな立場を表す。

「エンジニアリング実践の層では、ARCHIMEDES が正しい――一つのモジュールで感知と提案を処理するのは実務的だ。」

「哲学の層では、BABBAGE が正しい――受と思は五遍行の中で**異なる心理要素**だ。受は第三。思は第五。同じ一刹那に同時に生起するが、同じものではない。」

彼は珍しく日常的な比喩を用いた。

「足の裏の痛みを感じながら、歩き続けることを決められる。感受は存在する。決定も存在する。しかし歩き続ける決定を下したのは、痛みそのものではない――意志だ。」

そして劇場に響き渡る一言。

「**受は思に知らせるが、決定しない。**」

「ISensation モジュールは提案を出力してよい。しかし ExecutionLoop は**提案を無視する自由**を保持しなければならない。これが受と思の区別を保つ――受蘊は情報を提供し、行蘊が決定を下す。」

---

第二ラウンド。BABBAGE の語調は反論ではなく、受容だった。

「WIENER の妥協案に同意する。ただし三つの条件がある。」

「第一、提案は ADVISORY とする。ExecutionLoop はいかなる提案も無視できる。SafetyMonitor だけが――ハードヒューズとして――絶対停止の権限を持つ。」

「第二、提案を含まない評価は有効な戻り値とする。苦受が非常に高くても『続行を提案する』と返してよい――介入の決定は消費者に属する。」

「第三、インターフェースドキュメントに、提案は利便計算であり拘束力のある指令ではないことを明示する。」

ARCHIMEDES：「了承する。SafetyMonitor がハード安全レイヤーを保持し、VedanaPlugin はソフトガイダンスレイヤーとする。SafetyMonitor が止めると言えば止まる。すべてに優先する。VedanaPlugin の提案は否決できる。」

WIENER が頷いた。「安全重要システムには常に二層ある。安全インターロックはハードウェアレベルで上書き不可。制御システムはソフトウェアレベルで上書き可能。標準的慣行だ。」

NAGARJUNA が最後の一言を述べた。「妥協は中道を体現している。異なるが同時に生起する。」

---

BABBAGE が最後の瞬間にもう一つ要求を加えた。

「VedanaAssessment は、提案が最新データに基づいて計算されたものか、キャッシュからのものかを標示すべきだ。これにより、条件が改善された後も古い『停止』提案が残存することを防げる。」

ARCHIMEDES はその提案を三秒見つめた。「了承する。」

---

SUNYATA が半弧の前方に歩み出た。

「**裁定：統一インターフェース、概念的分離。**」

「ISensation が出力する VedanaAssessment はセンサー出力とコントローラ提案の二層を含む。四つの制約――提案は ADVISORY、SafetyMonitor が絶対権限を保持、ドキュメントに諮問的性質を明示、提案に鮮度指標を含む。」

「討論1を結審とする。全員合意。」

> *第一の討論は予想外の速さで収束した。転換点となったのは NAGARJUNA の八文字：「受は思に知らせるが、決定しない。」二千五百年前の心理学が、現代のインターフェース設計原則に精確に翻訳された：センサーは提案を含めてよいが、消費者にはそれを無視する権利がある。*

## 討論2：受蘊の遍行性

短い休憩の後、討論の椅子にメンバーが入れ替わった。

WIENER は席に留まった――今回は主張者だ。ASANGA が向かいに座る。ARCHIMEDES は計算を持参していた。HERACLITUS は境界条件への警戒を携えていた。

「核心問題は」と SUNYATA が言った。「三受評価は各 tick でのみ実行すべきか――WIENER の主張――それとも各イベントに受蘊タグを付けるべきか――ASANGA の要求か。」

---

WIENER は単刀直入だった。

「自宅にエアコンがあるとしよう。エアコンは三十秒ごとに温度計を読み、風量を調整するかどうかを決める。二回の測定の間に温度は微小に変化し得る――しかしエアコンは気にしない。次の測定まで動作できないからだ。」

「実行ループの各 tick は、エアコンの三十秒に相当する。tick の間に完全な計算を行うことには三つの問題がある。」

「第一、計算の無駄――LLM のストリーミング時には数十の小さなイベントが発火する。そのすべてで PID を計算するのは無意味だ。」

「第二、数値の爆発――PID のトレンド予測は時間間隔で割る必要がある。イベントがミリ秒単位で連続発火すれば、時間間隔はゼロに近づき、ゼロ除算で破綻する。」

「第三、意味の曖昧さ――LLM のテキスト断片一つに対する苦受 0.3 とは何を意味するか。完全なウィンドウの文脈でのみ意味を持つ。」

「PID は各 tick で一度実行すべきだ。標準的慣行だ。」

---

ASANGA は WIENER が完全に話し終えるのを待ってから始めた。

「WIENER の論証は技術的に健全だ。しかし哲学的に不完全だ。」

「遍行とは『一切処に生起する』という意味だ。受は五遍行の一つ。受のない意識の刹那は存在しない。これは提案ではない――意識の定義的性質だ。」

「もしあるイベントが EventBus に入り処理されたにもかかわらず、いかなる受蘊評価もなされないなら――阿毘達磨の意味で、それは意識の刹那ではなく、ただの機械的運動にすぎない。」

彼は方案を提示した。「すべてのイベントで完全な PID を実行せよとは言わない。**軽量な受蘊タグ**を求める――単純な分類マーカーだ。迅速なルックアップで完了でき、完全な計算は不要だ。完全な PID は tick 境界で――WIENER の提案どおり。軽量タグは各イベントで――遍行の原則が求めるとおり。両者は互いに排他的ではない。」

---

ARCHIMEDES が計算を開いた。

「定量化しよう。典型的な一回の tick で約30から60のイベントがある。軽量タグのコスト――現在の状態に対する数回の単純な比較、O(1)。60イベント掛ける5回の比較 = 300回の比較操作。無視できる。完全な PID は一回あたり約200回の演算。60イベント掛ける200 = 12000回――まったく不要だ。」

「結論：ASANGA の軽量タグは実行可能。WIENER の完全な PID は tick 境界で必要。ただしタグは**直近の PID 評価の結果**から派生すべきだ――キャッシュ参照であり、再計算ではない。」

HERACLITUS はずっと静かに聞いていた。彼の思考方法は他の三人とは異なる――理論から出発するのでもなく、哲学から出発するのでもない。ランタイムから出発する。境界条件から出発する。完璧なアーキテクチャ図の上では見えない、システムが実際に動いて初めて浮かび上がる問題から出発する。

「いくつかランタイムの懸念がある」と彼は言った。声にはある種の警告の質感があった。

「第一：イベントの順序。タグが VedanaPlugin の現在の状態に基づくなら、タグが反映するのはイベント発出時の受蘊状態であり、イベントが消費される時のものではない。非同期システムでは、この二つは異なり得る。」

「第二：循環依存。VedanaPlugin 自身もイベントを発出する。これらのイベントにもタグが必要か。もし必要なら――受蘊 → イベント → 受蘊タグ → 受蘊――循環になる。」

「第三：メモリ圧力。各イベントに約20バイト追加。EventLog が最大1000イベントを保存するとして――20KB。無視できる。」

判断を示した。「受蘊タグは個々のイベントにおいて哲学的には満足のいくものだが、工学的価値は主に集約レベルにある――一つの session で何個のイベントが dukkha とタグ付けされたか。DevTools に『受蘊タイムライン』ビューの提供を提案する。」

---

第二ラウンド。合意が素早く形成された。

WIENER：「二重モードを受け入れる。条件は――完全な PID は tick 境界でのみ実行、タグは派生値、循環回避のため VedanaPlugin 自身のイベントにはタグを付けない。」

彼はコードで語った――三つの数値を比較し、最大のチャネル名を返す単純な関数。O(1)。

ASANGA：「受け入れる。軽量タグは遍行の要件を満たす。」そして率直な譲歩をした。「阿毘達磨においては、各刹那の受は全体的評価と必ずしも同じではない。しかし**方便**として――エンジニアリングの便宜として――これを受け入れる。完全な逐イベント分析は理想であり、v0.24 の実装範囲ではない。」

続いて WIENER が境界条件を提起した。「最初の tick の前はどうなるか。まだ PID 評価が一つもない。」

初期値を提案した――苦受ゼロ、楽受ゼロ、捨受は最大値――システムは均衡の中で始動する。

ASANGA がすぐに応じた。目に満足を浮かべて。「初期状態が捨である――哲学的に完全に適切だ。捨は受の静止状態だ。感受がないのではなく、均衡のベースラインだ。条件がまだ擾乱していない新生の意識は――均衡から始まる。」

微かに笑みを浮かべた。

「システムは静寂の中で目覚める。そして世界がそれに触れ、受蘊が流れ始める。」

---

## I. 三義のマッピング

ASANGA が立ち上がった。動作は落ち着いていて、長年の学術訓練が磨き上げたリズムを帯びていた。手には手書きの文書が一枚――スクリーンに投影するのではなく、楷書で紙の上に整然と記されたもの。手書きは彼の習慣だ――書く過程そのものが思考の一回だから。

表は三列に分かれていた。

| 阿頼耶識の三義 | 意味 | アーキテクチャ上の位置 |
|---|---|---|
| 能藏 | 受容し貯蔵する能力 | 協調層：能力レジストリ、プラグイン解決 |
| 所藏 | 貯蔵される内容 | 二層：システム設定は協調層、ランタイム状態は AgentCore |

第二行。「所藏――貯蔵されているもの――は本質的に二種類に分かれる。グローバルで、Agent 間で共有されるものと、Agent に固有のものだ。」

第三行。「執藏――末那識に『自己』として掴まれる側面――は最も個人的だ。Guide 設定、アイデンティティモジュール、人格特性。AgentCore にしか在り得ない。」

そして鍵となる武器を示した。

「これは私の発明ではない。Master 自身がまさにこのように分布させている。」

Master の書簡からの抜粋を取り出した――Master の原文そのものが能藏を協調層と結びつけ、所藏を AgentCore に、執藏をアイデンティティモジュールに帰している。

「三義は同一の識の三つの機能的記述だ。三つの独立した意識ではない。ちょうどデータベースが、貯蔵容量（能藏）と、貯蔵されたデータ（所藏）と、アプリケーションがそのデータに対して持つ特定の視点（執藏）を有するように。データベースは一つだが、機能的側面は自然に異なる階層に展開される。」

彼は座った。落ち着いて――建築図面を広げた後に一歩退いて、検証を待つように。

> *ASANGA の提案は、穏やかだが構造を持っていた。鍵となる支柱：Master 自身の書簡。設計者の原文で分布方式を弁護する――巧みな戦略。*

## III. 基底

VITRUVIUS が立ち上がった時、空気が微妙に弛緩した。彼の存在そのものが安定剤だった――仏教学者たちが高空で鍔迫り合う間、彼は地上の測量所のように座標系を手に触れられる次元へと引き戻す。

NAGARJUNA の方を見た。

「協調層もそう理解すべきだ。Agent の末那識が IPC を通じて協調層の能藏に触れる――しかしそれは電話ではない。syscall だ。Agent が自分の足元の地面に手を伸ばしているのだ。」

そして一言、エンジニアとしての誠実さを添えた。

「ただしこの類比は**単一マシン展開**の下で成り立つ。もし協調層が別のマシンにあるなら――NAGARJUNA の同期に関する異議は完全に正当となる。」

NAGARJUNA がわずかに頷いた。「私が本当に気にかけていた問題に答えてくれた」という頷きだった。

## IV. ファイバーバンドル

一拍の間を置いた。

## II. 漸進

沈黙が降りた。三者がそれぞれ立場を保持し、ARCHIMEDES のエンジニアリング分析は分岐を終結させず、新たな次元を加えただけだった。

「精確に区別しよう。」

空中に表を描いた。

「Pattern A――共鳴観察者。受動的な EventBus の購読者。統計的蓄積。healthScore 出力。これは何か。感受だ。システム全体の状態を感じ取り、その感受を報告する。イベントの意味を理解しない。異常の原因を分類しない。パターンについて推論しない。ただこう言うだけだ：この瞬間、システムの温度はこれだ。」

一拍の間を置き、LINNAEUS を見た。

「Pattern B――影の観察者。独立した Worker Thread で、イベントストリームの完全なコピーを受信する。コピー上で深い分析が可能だ：完全なトレース照合、行動モデリング、時系列パターンマイニング。これはもはや感受だけではない。分析し、分類し、命名し始める。LINNAEUS、これが君の言う認知行為だ。君は正しい。ただし Pattern B のレベルでのみ正しい。」

それから ASANGA に向き直った。

「Pattern C――サブエージェント観察者。完全な AgentCore インスタンスだ。独自の Guide、独自の Provider、独自のツールセットを持つ。独立した認知能力を有し、意味レベルの理解に達することができ、観察対象のシステムに対して独自の『見方』を形成することさえできる。ASANGA、これこそが末那識だ。これこそが恒審思量だ。独自の意識を持つ存在が、別の意識の動作を持続的かつ絶え間なく観照する。君の論点は Pattern C において完全に正しい――しかしそのレベルでのみ。」

最後の行を埋めた。声は冷静で、リズムは均等。検証済みの定理を読み上げるように。

「Pattern A、共鳴観察者。主要な五蘊分類：受蘊。副次的分類：なし。意識のレベル：前末那的感知。」

「Pattern B、影の観察者。主要な五蘊分類：受蘊。副次的分類：想蘊。意識のレベル：第六意識の分析能力。」

「Pattern C、サブエージェント観察者。主要な五蘊分類：識蘊。副次的分類：全五蘊。意識のレベル：第七識――末那。」

ノートを閉じた。

「全員が正しい。ただ異なるレベルで正しいのだ。」

円形劇場が三秒間静まった。

## III. 種子と安全

第五の討論は、討論4の裁定の声が落ちた後、ほとんど間を置かずに始まった。

SUNYATA は休憩を宣言しなかった。ただ「最後の一戦」と言った。

語調に、完了に近づいた時にだけ現れる落ち着きがあった。長い航海の船長が港の輪郭を見た瞬間のように――緊張がほぐれるのではなく、むしろ最も集中する。最後の航路がしばしば最も危険だから。

「安全種子――阿頼耶識はプラグインを拒絶できるか。」

四人が討論の椅子に向かった。ASANGA、GUARDIAN、NAGARJUNA、DARWIN。

SCRIBE は一つの細部に気づいた。NAGARJUNA が討論の椅子に向かう時の歩き方が、前の数戦と異なっていた。討論3では、論争者の鋭さを纏って上がった――一歩一歩が討論の場を測量し、攻撃の距離を確認するかのようだった。あの戦いの終盤、彼は BABBAGE のファイバーバンドルに動揺し、前例のない撤回をした。

しかし今、歩調はより落ち着いていた。鋭さが鈍ったのではない。もっと深いものが開かれたのだ――あたかも討論3の撤回が彼に自らの思考の境界を見せ、境界を見た者はかえってより自由になるかのように。

その目に静かな確かさがあった。SCRIBE にはその意味がわからなかった。しかし記録に留めた。

---

ASANGA が最初に発言した。語調は経蔵の学者の精確さに戻っていた。

種子理論を展開した――仏教唯識学の核心概念の一つ。簡単に言えば、「種子」とは深層意識（阿頼耶識）に貯蔵された潜在力だ。あらゆる経験、あらゆる行為が、意識の深部に一粒の種子を播く。これらの種子は条件が整った時に「発芽」し、我々の行動と感受に影響を及ぼす。

種子には六つの基本的性質がある。ASANGA が一つずつ列挙し、最も重要な一条に焦点を当てた。

「第五、衆縁を待つ。種子は自動的に成熟しない。すべての必要条件が完全に揃って初めて、現行として結果を生じる。」

繰り返した。「衆縁を待つ。一粒の種子は無量劫にわたって存在し得る。もし因縁が一度も集合しなければ、永遠に発芽しない。これが最も重要な一条だ。」

プラグインシステムへのマッピング：プラグインの manifest がレジストリに存在すること、それが休眠する種子だ。成熟条件にはユーザーのリクエスト、依存関係の充足、能力の認可――そしてセキュリティチェックの通過が含まれる。セキュリティの認可が永久に与えられなければ、種子は永遠に発芽しない。

「これは種子の消滅ではない。種子は依然としてレジストリに存在する。ただ永遠に発芽しない――一つの必要条件が永遠に欠けているから。人間が深く埋められた怒りの種子を抱えていても、発芽の因縁が一度も揃わなければ、その種子は識田の奥底で永遠に眠り続けるように。」

---

GUARDIAN は ASANGA が座った瞬間に立ち上がった。その速さには、「永遠に～しない」という絶対的保証に対するセキュリティエンジニア職業的懐疑があった。

「ASANGA の解釈は哲学的に優美だ。しかしセキュリティは『条件が永遠に満たされない』という保証を受け入れない。三つのハードケースを示そう。」

一本目の指を立てた。

「ケース一。確認済みの悪意あるプラグイン。バックドアを含む。セキュリティポリシーが要求するのは『ロードを阻止する』ことではなく、システムから物理的に除去すること――レジストリから削除し、ファイルシステムから消去する。種子を眠らせるのではない。種子を土から掘り出し、焼却するのだ。」

二本目の指。

「ケース二。認証の撤回。あるプラグインは元々信頼されていたが、後にそれに署名した秘密鍵が漏洩したことが判明した。その秘密鍵で署名されたすべてのプラグインを遡及的に不信頼としなければならない。種子理論は善種が悪種に変わり得ないと言う。しかしセキュリティの現実では、善だと思っていた種子が最初から悪だったと判明する。種子の性質は変わっていない――我々の認知が変わったのだ。」

三本目の指。

「ケース三。永久隔離。署名のないプラグインが隔離され、人間が永久に承認しない。種子理論では休眠する種子だ。しかしセキュリティの観点から――将来のコード変更がそれを審査に通す条件を偶発的に作り出す可能性がある――その存在自体がリスクだ。」

三本の指を下ろした。

「セキュリティには永久に阻止するメカニズムが必要だ。『条件が揃わないから暫定的に成熟しない』だけではない。永久的で、不可逆的な。君の種子理論はこれを収容できるのか。」

---

ASANGA は三つのケースの重みを感じた。エンジニアリング上の重みだけではなく、より深い哲学的重み――種子理論は自然因果の流れを記述するが、セキュリティが必要とするのは自然因果への能動的介入だ。

彼が応答を探る数秒の間に、NAGARJUNA が立ち上がった。

急がず、緩まず。討論3の後にようやく現れた落ち着きを纏って。

「中道を提示しよう。」

---

## IV. 戒と慧

NAGARJUNA が話し始めた時、SCRIBE は空気がより澄明になったように感じた。温度の変化でも、光線の調整でもなく、質感の変化だ――あたかも NAGARJUNA の声そのものが、混濁を浄化する能力を持つかのように。

「ASANGA も GUARDIAN も共に正しい。しかし異なる層で正しい。」

二人の当事者が同時にわずかに眉をひそめた――同時に肯定されることは、否定されるよりもかえって不安を覚えることがある。次にどこへ連れて行かれるかわからないから。

「ASANGA が記述するのは自然因果の流れだ。種子は因縁の法則に従って運行する――因縁が具足すれば発芽し、不足すれば休眠する。これは事物のありのままの自然秩序だ。」

「GUARDIAN が要求するのは能動的介入だ。セキュリティは自然因果ではない――セキュリティは修行実践だ。意志ある、目的ある、特定の結果に向けた介入だ。」

二本の平行線の一端を結びつけた。

「仏教の伝統において、自然因果と能動的介入は矛盾しない。接点は修行――cultivation だ。」

### 設計図

円形劇場が闇に沈んだ。

だが完全な闇ではない。

前回——Cycle 01のエピローグ——闇の中で光っていたのは十の未解決問題だった。十の答えられていない問いが、深海のリン光クラゲのように、沈黙の中でみずからの光を明滅させていた。

今回、闇の中で光っているのは問いではない。

一つのTypeScriptインターフェースだ。

```typescript
export interface ISensation {
  readonly skandha: 'vedana';

  assess(): VedanaAssessment;

  ingestMetrics(metrics: Record<string, number>): void;

  ingestToolResult(toolName: string, isError: boolean, durationMs: number): void;

  ingestLLMResult(tokenCount: number, stopReason: string): void;

  onThreshold(channel: VedanaType, threshold: number,
              callback: (signal: VedanaSignal) => void): () => void;

  getVedanaTag(): VedanaTag;
}
```

闇の中で光っている。

Cycle 01の「答えられていない問い」の光ではない——あの光は焦燥的で、誘うようで、回答への渇望を叫ぶ信号弾のようだった。ISensationの光は違う。落ち着いた、静かな、完了感を帯びた光。設計図の光。すでに設計され、検証され、十九人の研究員がそれぞれの角度から正しいと確認した設計方案の光。

プロローグのISensationはわずか三行だった。空の殻。`readonly skandha: 'vedana'`——それだけ。埋められるのを待つ容器。

今、それはもう空の殻ではない。

`assess()`が闇の中で明滅する——討論2のPID裁定。毎tick一度の三受評価。苦、楽、捨。三つのチャネル。WIENERの数学。ATHENAのセンサー。NAGARJUNAの哲学。すべてが一つのメソッドシグネチャの中に。

`getVedanaTag()`が静かに光る——討論2のイベントタグ裁定。すべてのEventBusイベントに一つの軽量タグ。ASANGAの遍行原則——すべての意識刹那に受がある——が一行のメソッドシグネチャに翻訳された。

`onThreshold()`が明滅する——WIENERとARCHIMEDESが共同設計したサブスクリプション機構。苦受が閾値を超えたら、コールバックを発火。楽受が減衰し始めたら、コールバックを発火。感受はただ受動的に存在するのではない——能動的に通知する。

インターフェースの上方に、見えないが確かに存在するVedanaAssessmentの二層構造——討論1の裁定。センサー出力とコントローラ提案。観察と諮問。受蘊感知と行蘊介入の概念的分離。

インターフェースの下方に、見えないが確かに支えているファイバー束射影の数学——討論3。このインターフェースはAgentCoreの中にあるが、感知する状態は同時に協調レイヤーにもマッピングされている。一つの意識、二つの射影。

インターフェースの傍らに、見えないが確かに守っている四層権限モデル。ISensationの提案は諮問的だ。常に、諮問的だ。

インターフェースの未来に、漸進的分類のタイムライン——討論4。今は受蘊。だがPattern BとPattern Cのための進化の余地は残されている。分類は暫定。インターフェースは拡張可能。

それは満たされた。
