# 第九章：十九のアクションプラン

---

R4 最終稿の円形劇場は、討論のときよりもずっと静かだった。

それは空虚な静けさではない——五つの討論の余波がまだ空気の中で振動していた。チェロの弦が最後の音符のあとも震え続けるように。しかし討論の緊張感はすでに退いていた。代わりに別の密度がそこにあった——納品の密度。十九人の研究員がそれぞれの席に座り、目の前には初稿、修正稿、コード断片、マッピング表が並んでいた。R4は討論ではない。R4は収穫だ。

SUNYATAは会場の中央に立ち、場内を見渡した。もう何かを宣言する必要はなかった——全員がやるべきことを分かっていた。R3の五つの合意が五本の柱のように、その内部で作業できる構造を支えている。今の任務は、討論の裁定を納品可能な文書に変換すること。哲学的洞察をエンジニアが読める言葉に翻訳すること。

「ARCHIMEDES」と彼は言った。「君から。」

---

## I. 明日、TypeScriptを書かなければならない

ARCHIMEDESが席から立ち上がる姿は、どの研究員とも違っていた。

NAGARJUNAは鞘から抜かれた剣のように立ち上がる。ASANGAは風の中でゆっくりとまっすぐになる木のように立ち上がる。BABBAGEは精密機器が起動するように立ち上がる。ARCHIMEDESはシャベルを手に取る建設作業員のように立ち上がった——ドラマ性もなく、儀式感もなく、ただ「仕事が来た、始めよう」という素朴な原動力だけがあった。

「まず一言」。彼は展示エリアに歩み寄り、四十ページに及ぶ文書を開いた。「この三日間、皆さんは討論の中で五つの裁定を生み出した——観察-介入分離、デュアルモードvedana、ファイバー束射影、漸進的分類、戒と智慧のフレームワーク。どれも美しい。傍聴者として知的な喜びを感じた」

一拍の間を置いた。

「しかし哲学は美しくても、明日、僕たちはTypeScriptを書かなければならない」

会場で何人かが笑った。BABBAGEの口元がわずかに動いた——それが彼にとって大笑いに最も近い表情だ。NAGARJUNAは笑わなかったが、その目には認める色があった。天上のものを地上に降ろすべき時を心得ている人間は、尊敬に値する。

「この文書は」とARCHIMEDESは展示エリアを指した。「完全な工学アクションプランだ。十九人の研究員の仕事が、五つの討論で鍛え抜かれ、すべてこの一冊に収束した」

---

彼は最優先の項目から始めた。

「P0——ブロッキング。三つ。他のすべての作業の前に完了しなければならない」

指を一本立てた。

「第一、SEC-01。プラグイン署名バイパス脆弱性の修正」。GUARDIANに目を向けた。「Cycle 01で発見済み。GUARDIANとTURINGが正確に特定した——`pluginFilePath`がundefinedのとき、署名検証全体がスキップされる。六つの開発サイクルが過ぎた。まだ修正されていない」

GUARDIANが頷いた。「署名をバイパスする脆弱性。修正しない日が一日続くごとに、システムは一日余分に丸裸で走り続ける」

「第二」とARCHIMEDESは二本目の指を立てた。「ISensationインターフェースの実装」

展示エリアにコード断片を切り替えた。

```typescript
// 現在の ISensation
export interface ISensation extends IOpenStarryPlugin {
  readonly skandha: 'vedana';
}
```

「三行。空の殻」。批判を含まない口調——事実を述べているだけだ。「これがプロローグに出てきたあの空の殻。Cycle 02の研究全体は、ある意味、この三行を埋めるためにあった」

次のコードに切り替えた。円形劇場全体が静まった。

```typescript
export interface ISensation {
  readonly skandha: 'vedana';

  /** 完全な三受評価 (tick-synchronous PID) */
  assess(): VedanaAssessment;

  /** PID計算用の生メトリクスを取り込む */
  ingestMetrics(metrics: Record<string, number>): void;

  /** ツール実行結果を取り込む */
  ingestToolResult(toolName: string, isError: boolean, durationMs: number): void;

  /** LLM応答結果を取り込む */
  ingestLLMResult(tokenCount: number, stopReason: string): void;

  /** 受蘊閾値超過イベントを購読 */
  onThreshold(channel: VedanaType, threshold: number,
              callback: (signal: VedanaSignal) => void): () => void;

  /** 軽量受蘊タグの取得 (O(1) キャッシュ参照) */
  getVedanaTag(): VedanaTag;
}
```

「これが中身を埋めたあとのISensationだ」。ARCHIMEDESの声は安定して自信に満ちていた。「すべてのメソッドが討論裁定まで遡れる。`assess()`は討論2のtick-synchronous PID裁定。`getVedanaTag()`は討論2のイベントタグ裁定。`onThreshold()`はWIENERの三チャネルPIDコントローラに必要なサブスクリプション機構」

VedanaAssessmentの構造を指し示した。

「二層——討論1の核心裁定。第一層はセンサー出力：苦・楽・捨の数値。純粋な観察。受動的。第二層はコントローラ提案：recommendation。諮問的。強制ではない。SafetyMonitorが絶対的権限を保持する」

WIENERが席からわずかに前のめりになった。自分のPID設計が正しく翻訳されているか確認していた。確認したあと、背もたれに戻った。翻訳は正確だ。

「第三」とARCHIMEDESは三本目の指を立てた。「VedanaPluginの実装。Pattern A観察者すなわちVedanaPlugin。討論4の裁定。一つのプラグイン、一つのインターフェース、一つの蘊」

VedanaPluginのアーキテクチャ図を展示した——三層構造。最下層はATHENAが設計した三チャネルセンサー：DukkhaSensorがエラー率を監視し、SukhaSensorが完了率を監視し、UpekkhaSensorが安定性を監視する。中間層はWIENERのPID制御エンジン。最上層は評価出力。

「三つのP0項目。もし一つしかできないなら、SEC-01を修正する。二つできるなら、ISensationを加える。三つなら、VedanaPluginを完成させる。これが最小実行可能な納品パスだ」

---

ARCHIMEDESはさらに下へと展開した。

「P1——高優先。四つ」

EgoFramework——ハードコアにソフトシェル。ハードコアはロボット三原則、上書き不可。ソフトシェルはPIDパラメータ調整。ASANGAの二層モデルが討論2で確認された。これがその工学的実装。

プラグインライフサイクル状態——四つの状態：active、quarantined、revoked、banned。討論5の戒と智慧のフレームワーク。NAGARJUNAが哲学的基盤を与え、DARWINが工学的実現可能性を確認した。今、コードに書き込む必要がある。

ToolContext.busリーク修正——ツールがサンドボックスのイベントフィルタリングをバイパスできる。TURINGが発見した。

SafetyMonitorのper-sessionカウンター——現在はグローバルカウンターで、あるセッションのエラーが別のセッションに影響する。HERACLITUSがこの問題をマークした。

各項目の横にはターゲットPlan番号が記されている。アクションは空中に漂っているのではない。アドレスがある。

---

「P2からP4は一つずつ展開しない」。展示エリアを総覧に切り替えた。「完全なリストは方案の中にある。CRLチェックスタブ、新イベントタイプ、宣言の書き換え、阿頼耶識分布のアーキテクチャ注釈——P2。Pattern B影観察者とAgent協調レイヤー——P3。Pattern Cサブエージェント観察者——P4、協調レイヤーの完成を待つ必要がある」

最後に影響総覧表を表示した。

「五つの既存Planが修正され、一つの新Planが提案された。Plan24セキュリティ迅速修正に戒フレームワークのブラックリストとCRLスタブを追加。Plan26 Vedanaアーキテクチャ——本研究の主要な設計入力。Plan27ライフサイクル管理に四状態ステートマシンを追加。Plan28文書整合。Plan29——新規——Pattern B影観察者」

彼は文書を閉じた。

「四十ページ。十四のアクション。六つのPlan。完全なTypeScriptインターフェース仕様。そのままコードを書き始められる」

SUNYATAに目を向けた。

「僕の部分は終わった。哲学は美しい。工学は具体的だ。今、それらは同じ文書の中にある」

## II. 五つの討論、一枚の絵

SYNTHESISTが展示エリアに歩み寄る姿は、ARCHIMEDESとはまるで違っていた。ARCHIMEDESがシャベルを持つ建設作業員のように歩くとすれば、SYNTHESISTはキャンバスの前に立つ画家に近い——新しいものを創るのではなく、すでに存在する断片の中から一枚の完全な絵を見出すために。

「ARCHIMEDESは一本の木を見せてくれた」。口を開いたとき、話す速度はいつもより遅かった。「すべての枝がアクション、すべての葉がTypeScriptインターフェース。正確で、具体的で、実行可能。だが僕は、森全体を見せたい」

展示エリアに一枚の図を映した——図表でもフローチャートでもなく、手描きに近いアーキテクチャ全景図だった。五つの討論裁定が異なる位置に記され、点線で互いにつながり、有機的なネットワークを形成していた。

「五つの討論、表面上は五つの独立した問題。だがそうではない。同じアーキテクチャの五つの断面だ」

彼は線を引き始めた。

「討論1と討論4が合流する」。最初の弧線を描いた。「VedanaPluginは観察者。ISensationを実装する。生成する評価報告には二つの側面がある：受動的な感受と諮問的な提案。これはアイデンティティの確立だ」

「討論1と討論5が合流する」。二本目の弧線。「階層権限にシード管理が加わり、完全な四層権限モデルが形成される」

全景図の右側に四行を書いた。

```
Layer 1: SafetyMonitor  — ハード安全 — 絶対権限 — 戒律
Layer 2: VedanaPlugin   — ソフト指導 — 諮問権限 — 覚受
Layer 3: EgoFramework   — 自我制約 — 構造権限 — 我執
Layer 4: Sila Engine    — シード管理 — 予防権限 — 戒学
```

「四層。各層の権限は上の層より低い。SafetyMonitorはVedanaPluginの提案を上書きできる。VedanaPluginはEgoFrameworkのソフトシェルに影響を与えられる。EgoFrameworkのハードコアは他のどの層からも影響を受けない。Sila Engineは源流で悪いシードの侵入を阻止する」

GUARDIANは第一層と第四層の微妙な対称性に気づいた——SafetyMonitorは最内層のハード防御、Sila Engineは最外層のハードフィルタリング。両者が中間の二つのソフト層を挟んでいる。彼は構造のセキュリティ態勢を心の中で評価し、静かに頷いた。完備。

「討論3と討論2が合流する」。三本目の弧線。「阿頼耶識はファイバー束射影として二つのアーキテクチャ層に分布する。各イベントには受蘊タグが付与される。合わせて——分散意識のランタイム全景図」

「討論4がタイムラインを定義する」。底部にタイムラインを描いた。「漸進的観察者パス——Pattern AからBへ、BからCへ。システムの自己観察能力が、時間とともに螺旋的に上昇する」

筆を置き、一歩後退して、全員が全景図を見渡せるようにした。

「五つの討論。五つの合意。一枚の絵。これがCycle 02のアーキテクチャビジョンだ。統一された、内的に自己整合的なアーキテクチャ。五つの裁定を貼り合わせたのではない。五つの裁定がそれぞれ、同一の構造の異なる側面を明らかにしたのだ」

## III. 開発チームへの手紙

SUNYATAが展示エリアに歩み寄った。手には四十ページの方案もなく、全景図もなかった。一枚の紙だけを持っていた。

「これは僕が開発チームに宛てた手紙だ。Masterへの手紙でもある。読み上げる」

円形劇場が静まった——討論前の緊張ではない。完了に近い静けさだった。交響曲の最終楽章が始まる前、指揮者が両手を上げ、オーケストラが息を呑む、あの瞬間のような。

SUNYATAが読み始めた。話す速度はとても遅い。一文一文に十分な間が与えられた。

「核心結論を一言で——」

顔を上げ、場内を見渡した。十九対の目。

「VedanaPluginが観察者モジュールだ」

この言葉が数秒間、空中に浮かんだ。少なくとも三人の研究員の表情がわずかに変わった——結論を知らなかったからではない。五つの討論の複雑なアーキテクチャ決定が一文に圧縮されるとき、その一文は余分な密度を帯びるからだ。山脈が一つの石に折り畳まれるように。

読み続けた。完全な要約——三チャネルPID、VedanaAssessment二層構造、ファイバー束射影、戒と智慧のフレームワーク——一文一文が一つの討論の結晶だった。

そして手紙の最後の部分へ——Masterの裁定を必要とする四つの問題。

「Q1：VedanaPluginは必須プラグインか任意プラグインか？私の提案——デフォルトテンプレートでは必須、ただしCoreは強制しない。SafetyMonitorがVedanaPlugin不在時に苦受のみのフォールバックを提供する」

ASANGAがわずかに前のめりになった。遍行——すべての意識刹那に受がある。もしVedanaPluginを省略できるなら、システムは感受なしに動作しうる。だがSUNYATAの折衷案は、少なくとも何らかの形の感受が常に存在することを保証している。

「Q2：宣言#6を書き換えるか？提案する新たな文言——受蘊感知と行蘊介入の分離、観察が被観察行動を変えないことを確保する」

「Q3：EventBusの受蘊タグはどの方式を使うか？提案はメタデータ付加。コアイベントタイプをクリーンに保つ」

「Q4：協調レイヤーは独立プロセスかインプロセスモジュールか？提案は延期。本研究はアーキテクチャ原則——ファイバー束、統治であって制御ではない——を提供したが、具体的なプロセスモデルにはさらなる設計作業が必要」

手紙をテーブルに置いた。

「四つの問題。四つの提案。各提案に理由付き。最終決定権はMasterにある」
