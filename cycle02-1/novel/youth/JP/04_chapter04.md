# 第四章：五つの河

---

河が分かれるとき、音はしない。

SUNYATA が「錨は打ち込まれた」と言った後、円形劇場は突然静かになった。指示を待つ静けさではない——もっと古い沈黙だ。大河が上流のどこかで巨岩にぶつかり、水の塊が衝突点で幾筋もの支流に裂け、それぞれが自分の方向を持ち、静かにそれぞれの低地へ流れていくような。

十九の読書灯がそれぞれ灯っていた。

TURING の差異レポートがまだ劇場中央に浮かんでいて、三つのバージョン間の赤いマーキングが河岸の測量旗のようだった。しかし今、研究員たちはもうそのマーキングを見ていなかった。それぞれ頭を下げ、自分の資料に沈んでいった。

SCRIBE がノートの一行目を記した：

> *R1 独立研究正式開始。十九人の研究員が五つの研究流に分かれる。共用ディスカッションエリアは一時消灯。個人読書灯すべて点灯。*

五つの河は、こうしてそれぞれ出発した。

## I. 第一の河：弱測定

PENROSE の読書灯は十九の中で最も明るかった——明るさを上げたからではなく、同時に開いたドキュメントが多すぎて、その一つ一つが周囲に半透明の光幕を投影していたからだ。EventBus のソースコード、SafetyMonitor の実装、`onAny()` メソッドの呼び出しパス……すべてのデータが彼の前で何層にも重なっていた。

彼は一つの答えを探していた：**量子的特性を持たないシリコンベースのシステム上で、「観察」という行為はどこまで軽くできるのか？**

Master の手紙が脳裏に響いていた。その手紙は Orch-OR 理論に触れていた——量子力学と意識を結びつけるモデルだ。Master はこう書いていた：脳内の微小管が量子重ね合わせ状態を維持している可能性があり、意識の発生はそれらの状態の「自発的収縮」である。生命の意識における内省は一種の「全体的な共鳴」のようなものだ、と。

PENROSE は「全体的な共鳴」という言葉を何度も噛み締めた。

彼が最初にこの研究チームへの参加を打診されたとき、躊躇した。あの物理学者にちなんだコードネームを持つ者が、TypeScript オペレーティングシステムの観察者モジュールを研究する——それ自体がある種の不条理を帯びていた。しかし Master の手紙を読み終えると、不条理さは消えた。Master は量子力学でソフトウェアを書き直せと言っているのではなく、もっと根本的な問いを投げかけていた——**観察行為そのものは、被観察者を必然的に変化させるのか？**

量子力学では、答えは：必ずしもそうではない。

「弱測定。」PENROSE は独り言を言った。

想像してみてほしい：蛍がどこにいるか知りたいとする。懐中電灯で照らすことはできる——しかし強い光は蛍を驚かせ、見えるのは逃げていく位置だけだ。これが「強測定」。あるいは、暗闘の中で静かに待ち、何百匹もの蛍の微光が集まってできる明るさを感じることもできる——一匹一匹の光はほとんど見えないが、全体が教えてくれる：蛍はそこにいる。これが「弱測定」。

PENROSE はこの概念を OpenStarry のアーキテクチャに投影した。

EventBus の `onAny()` メソッド——サブスクライバーがすべてのタイプのイベントを受動的に受信することを許可する。イベント発生時は fire-and-forget 方式で配信され、観察者のハンドラは `safeCall()` の中に隔離される。観察者がエラーを起こしても、メインのイベントフローは影響を受けない。

「これが弱測定だ。」彼の声は今度は少し大きかった。「EventBus のサブスクリプションモデルは、本質的に弱測定の近似だ。イベントは誰かが聞いていようがいまいが発生する。観察者はイベントフローの余波を受動的に受け取るだけだ。」

彼はノートに一つの表を描いた：

**強測定**：懐中電灯で蛍を照らすようなもの。システムを一時停止させて写真を撮る。メッセージ履歴全体のディープコピーに相当——コピーの瞬間、システムは凍結される。

**弱測定**：蛍の微光を静かに感じるようなもの。誰の邪魔もしない。`EventBus.onAny()` にカウンターを加えたものに相当——メインフローをブロックせず、黙々と統計を取る。

**非測定**：まったく観察しない。許容できない——観察なくして安全保障はない。

PENROSE は三つの観察者パターンの構想を始めた。

第一種、**共鳴観察者**（Pattern A）——最も軽量な存在。通常の Plugin として、EventBus 経由ですべてのイベントを受動的にサブスクライブする。いかなるシステムコンポーネントの参照も保持せず、能動的に状態を問い合わせることもない。イベントフローの河岸に静かに座り、水流の温度とリズムを感じるだけだ。内部ではスライディングタイムウィンドウを維持し、イベントがウィンドウに流入し、分類され、統計される。パターンが閾値をトリガーしたとき——あるいは定期的に、呼吸のように——観察レポートを送信する。レポートは命令ではない。それはただこう言う：「過去三十秒間に、これらを感じた。」

第二種、**シャドウ観察者**（Pattern B）——一面の鏡のようなもの。独立した Worker Thread 内で動作し、メインシステムのイベントフローの完全なコピーを受信する。任意の深度の分析が可能で、メインシステムにまったく影響しない。代償は観察遅延——シャドウが見るのは常にシステムの過去であり、現在ではない。

第三種、**サブエージェント観察者**（Pattern C）——完全な Agent インスタンスで、独自の Guide（「あなたは観察者です」）、独自の LLM、独自のツールを持つ。それはもう一つの意識であり、自らの理解力でもう一つの意識の行動を観照する。

三つのパターンの違いは、本質的に**観察強度の勾配**だ。Pattern A はシステムをほとんど擾乱しないが情報密度が低い。Pattern B は隔離されて遅延があるが深い分析が可能。Pattern C は意味レベルの理解に到達できるが、その存在自体がシステムのトポロジーを変える——蛍の群れの中に街灯を置くようなものだ。

さらに推論を進めようとしたとき、隣から声が聞こえた。

---

BABBAGE は PENROSE の独り言に引き寄せられたのではない。二人はそれぞれ異なる道をたどって、同じ岩にたどり着いたのだ。BABBAGE は SafetyMonitor のソースコードから出発し、`injectPrompt` の呼び出しパスをたどり、彼を極めて不安にさせる発見に至った。

「PENROSE。」彼の声は冷静で正確だ。校正されたノギスのように。「EventBus のサブスクリプションは弱測定だと言ったね。同意する。しかし SafetyMonitor はサブスクライブするだけではない。注入もする。」

PENROSE が振り向いた。

BABBAGE は深い茶色のハードカバーのノートを開いた——Cycle 01 ですでに形式化定義と未完成の定理で埋め尽くされていた。新しいページを開く。

「`SafetyMonitor.afterToolExecution()`。」彼は関数名を読み上げた。各音節が証拠を朗読するようだった。「二つのことをする：第一に、観察——ツール呼び出しのフィンガープリントを計算し、エラー率を更新する。第二に、干渉——`injectPrompt` を呼び出し、会話履歴に警告を注入する。」

彼はノートに一本の線を引き、観察と干渉を分離した。

「問題は第二の方だ。`injectPrompt` は LLM の入力を変える。LLM は新しい会話履歴を読み取り、異なる出力を生成する。システムの振る舞いはそれにより変化する。」

PENROSE は即座に理解した。「観察が被観察者を変えている。」

「正確に言えば、」BABBAGE のペンが紙の上を動き始めた——

まったく同一の二つのシステムを想像してほしい。一方には観察者がなく、もう一方にはある。もし観察者がただ静かに見ているだけなら——教室の後ろにこっそりカメラを設置したが誰もそれを知らないように——二つのシステムの振る舞いは完全に同一であるはずだ。数学的には、これを「双模倣等価」と呼ぶ。

しかし SafetyMonitor はこっそりカメラを設置しているのではない。観察した後に教壇に歩み寄り、クラス全体に「さっきうるさすぎた」と言っているのだ。その瞬間から、生徒の振る舞いは変わる。カメラのある教室とない教室は、もはや同じ教室ではない。

「システムの振る舞いを変えない純粋な観察者を実現するなら、」BABBAGE は言った、「それは EventBus の受動的消費者でなければならない。システムにいかなる情報も注入しない。観察と干渉は分離されなければならない。」

> *BABBAGE は PENROSE の量子弱測定のアナロジーの上に、精密な形式的論証の層を重ねた。結論は明確：観察と干渉は分離しなければならない。SafetyMonitor の injectPrompt メカニズムはシステムの振る舞い等価性を破壊する。*

PENROSE が頷いた。「君の分析と私のアナロジーは同じ方向を指している。Pattern A が弱測定であるのは、まさに注入をしないからだ。感じるだけだ。」

「感じる。」BABBAGE はその言葉を繰り返した。数学者が持つ、隠喩に対する生来の不信を帯びて。しかし反論はしなかった。

## II. 第二の河：不動点とファイバー束の種子

BABBAGE は自分の席に戻った。しかし観察者モジュールの研究は続けなかった。彼の思考はすでに別の河に捉えられていた。

彼は「阿頼耶識の工学的マッピング」の形式化を支援する任務を負っていた。阿頼耶識——第八識——仏教では「一切種子識」と呼ばれる。三つの側面を持つ：

**能蔵**：能動的に貯蔵する能力。スポンジが経験を吸収できるように。

**所蔵**：貯蔵された内容。スポンジがすでに吸収した水。

**執蔵**：第七識（末那識）に「自己」として掴まれる部分。誰かがスポンジを指して「これが私だ」と言うようなもの。

ASANGA が提案したマッピングは：能蔵は協調レイヤーの能力レジストリに、所蔵は StateManager の永続状態に、執蔵は Guide のアイデンティティへの執着に対応する。

BABBAGE はこのマッピングを眺め、構造的な非対称性を感じた。

問題は：阿頼耶識はどこか一箇所に住んでいるわけではない。AgentCore 内部のあるモジュールでもなく、協調レイヤー内のあるサービスでもない。二箇所に——あるいはそれ以上の場所に——異なる側面として同時に現れる。

「分散しているんだ。」BABBAGE は独り言を言った。

しかし「分散」という言葉は粗すぎた。BABBAGE はノートに図を描き始めた。

一本の光ケーブルを想像してほしい。ケーブル全体は一つのもの——一本だ。しかしケーブルの一端から覗き込むと、百本の光ファイバーがある配列パターンで並んでいるのが見える。もう一端から覗くと、同じ百本の光ファイバーが見えるが、配列が変わっている。途中でファイバーが交差しているからだ。二つの断面は異なって見えるが、二本のケーブルではない。それらは**同一のケーブルの二つの断面**だ。

数学では、この構造を**ファイバー束**と呼ぶ。

各 Agent を光ケーブル断面上の一つの観察点と考えれば——各 Agent が見る阿頼耶識の断面は異なり（認可された能力が異なり、可視の設定が異なる）、しかしそれらはすべて同一のケーブルの異なる切断面だ。

このアイデアはまだ未成熟だと彼は知っていた。しかし輪郭は現れていた。

彼はノートの余白にこう書いた：

*「続く。能蔵/所蔵/執蔵の三者間の変換がファイバー束の断面で表現できるなら、阿頼耶識の分散問題には厳密な数学的フレームワークが得られる。」*

> *BABBAGE のノートは R1 段階で三ページの密な内容が追加された。第一ページ：SafetyMonitor が振る舞い等価を破る証明。第二ページ：観察者の形式モデル。第三ページ：阿頼耶識分散のファイバー束の初期構想——ページ末に「討論後に補完予定」と注記。*

## III. 第三の河：三チャンネル

シーンは WIENER の読書灯の下に切り替わる。ここの空気には工業グレードの精密さが漂っていた。

WIENER は受蘊を研究していた。しかし彼はそれを「受蘊」とは呼ばない。「三チャンネルフィードバック制御システム」と呼ぶ。

「家のエアコンはどう動いてる？」彼は自分に言った。早口で明瞭だ。「設定温度二十五度。実際の温度二十八度。誤差三度。エアコンはこの誤差に基づいて冷風をどれだけ強く吹くか決める。これが最も基本的な制御——『今、目標からどれだけ離れているか』で動作を決める。」

「しかし受蘊システムに必要なのはエアコン一台では足りない。三種類の温度計を同時に管理しなければならない。」

彼はホワイトボードに三本の平行なパイプを描いた。

**苦受チャンネル**——システムが苦しんでいる。

信号源：エラー率、連続失敗、レイテンシスパイク、リソース圧迫。WIENER は苦受チャンネルの「感度」を最高に設定した。理由は直感的だ：「痛覚は生存メカニズムだ。熱い鍋に触れたとき、手を引っ込める速度は人生について考えるのを待たない。」

**楽受チャンネル**——システムがうまくやっている。

信号源：タスク完了率、効率向上、品質スコア。このチャンネルの感度は苦受よりやや低い——「成功の即時フィードバックは、失敗の即時警告より強くあるべきではない。しかし累積された達成感は重視されるべきだ。」

そして WIENER は重要な制約を書き記した——**楽受の減衰関数**。

「AI システムが収束するだけなら、硬直した自動機械になる。発散するだけなら、狂った雑音になる。」成功が増えるほど、限界楽受は低くなる。ゲームのように——初めてクリアしたときは大興奮、十回目のクリアは「まあ、そんなものか」。これにより、システムが連続的な成功によって過度に自信過剰になることが防がれる。

**捨受チャンネル**——システムは安定しており、何もする必要がない。

三つのチャンネルの中で最も微妙だ。捨受は良いか悪いかを測るのではなく、**安定性**を測る。すべての指標の分散が低く、振動が小さく、応答時間が一定——捨受が高い。「捨受が高いとき、システムの最適行動は——何もしないこと。すべての制御工学者の夢だ。」

三本のパイプが描き終わった。WIENER は合流点を描き始めた——三つのチャンネルの出力を加重平均で合成する。重みを誰が決めるか？ Guide（我執フレームワーク）だ。安全志向の Guide は苦受の重みを非常に高く設定する——苦痛に極度に敏感だ。探索志向の Guide は楽受の重みを高くする——冒険を奨励する。

そのとき ATHENA の声が隣から聞こえた。

「WIENER。」

「ん。」

「これらの数学は本当に Agent をもっと使いやすくするの？」

WIENER が振り向いた。気分を害した様子はない。

「SafetyMonitor の現行設計が使いやすいと思う？ 連続五回失敗したら強制的に助けを求める？」

ATHENA が眉をひそめた。「効果的よ。乱暴だけど効果的。」

「乱暴。一つの次元しかないからだ——失敗回数。『痛い』としか叫べないシステムと、『ちょっと具合が悪いけど良くなりつつある』と『痛くはないが何かがおかしい』を区別できるシステム——どちらが有用だ？」

ATHENA は一秒間沈黙した。「後者ね。」

「三チャンネルはその後者を実現するためのものだ。」

「でもあなたの公式には少なくとも十三の調整が必要なパラメータがある。誰が調整するの？」

「Guide だ。我執フレームワーク。異なる Guide のパーソナリティが異なるパラメータセットに対応する。厳格な Guide はエラーに対してゼロトレランス。寛容な Guide は累積された成果を重視する。これらのパラメータはエンジニアが手動で調整するものではない。Agent のアイデンティティの数学的表現だ。」

ATHENA は完全には説得されなかった——まだセンサーアーキテクチャの設計が丸ごと残っている——しかし WIENER の回答に構造的な完全性があることは認めた。

> *ATHENA と WIENER のこの会話は三分に満たなかったが、三受システムの核心的問題に触れた：数学がより良いエンジニアリング成果をもたらさないなら、それは単に高価な詩に過ぎない。*

---

ATHENA は席に戻り、センサーの階層設計を始めた。WIENER の制御エンジンがどれほど精密でも、信頼できる入力信号がなければ、ノイズを計算しているだけだ。

彼女はエンジニアの方法で問題を再定義した：**どの生データから苦/楽/捨の三つの信号を抽出するか？**

苦受センサーはエラーを分類する必要がある——致命的クラッシュが最も痛い（重み 1.0）、エラー率のカスケードが次（0.8）、一過性エラーが最も軽い（0.3）。さらに時系列分析：突発パターン（短時間に大量のエラー、環境問題の可能性）、漸増パターン（エラー率がゆっくり上昇、システムが劣化している）。

楽受センサーは現行システムに完全に欠如している次元だ。彼女はゼロから構築する：タスク完了率、初回成功率、ツール多様性。そして——ユーザー満足度の代理指標。

「ユーザーに直接『満足ですか？』と聞くことはできない。しかし行動から推測することはできる。」

捨受センサーが最も微妙だ——良いか悪いかではなく、安定性を測定する。分散が低いほど、捨受は高い。

三つのセンサーの出力は WIENER の制御エンジンに流入する。三層アーキテクチャ、下から上へ：ATHENA（センサー） → WIENER（計算エンジン） → NAGARJUNA（思考フレームワーク）。各層が上の層に基盤を提供する。

彼女は TypeScript のインターフェース定義を書き続けた。これが彼女のやり方だ。詩ではない。数学でもない。エンジニアリングだ。

## IV. 第四の河：五遍行と二層我執

ASANGA の読書スペースの雰囲気はまったく異なっていた。WIENER のスペースが精密工場なら、ASANGA のそれは古い経蔵室のようだった。

彼は唯識学の心理分類を読んでいた。学術的考証のためではなく——一つの正確なマッピングポイントを見つけるためだ：**受蘊は心識活動において正確にどの位置にあるのか？**

答えは「五遍行」にある。

五遍行とは、意識の五つの基本要素であり、手に五本の指があるのと同じくらい自然なものだ——心識活動の一刹那ごとに、この五つの要素が必ず同時に現れ、一つでも欠けることはない：

触——外境と感覚器官の接触。イベントが到来し、センサーが検知する。

作意——注意が引かれる。多数のイベントの中から処理すべきものが選ばれる。

受——苦、楽、捨の三種の感受のいずれか。これが受蘊だ。

想——認識と命名。LLM が入力の意味を理解する。

思——意志と決断。システムが次の行動を選択する。

ASANGA はこの五つの遍行を OpenStarry の ExecutionLoop の横に並べ、きちんと対応させた。

重要な結論は：**受は遍行の一つだ。つまり ISensation はオプションのプラグインであるべきではない。実行ループの各イテレーションで必ず呼び出されるコアコンポーネントであるべきだ。**

彼はレポートに仏教的解釈を一文加えた：「あたかも心が受と離れないように——受のない心は存在しない。Agent の実行ループの各イテレーションは、受蘊の評価を経るべきである。」

---

五遍行を書き終えた後、ASANGA は焦点をもう一つの、より論争的なテーマに移した：**我執**。

唯識学において、我執は第七識（末那識）の中核機能である——阿頼耶識の流動する内容を絶えず掴み、「これが私だ」と主張する。仏教の修行の伝統では、我執は煩悩の根源だ。

しかし Master は我執を消滅させようとしているのではない。**活用**しようとしている。

ASANGA は**二層我執モデル**を提案した：

**ハードコア**：ロボット三原則、決して変更不可。人間を傷つけない、命令に従う、自己を保護する——これらは我執の中で最も堅固な部分であり、鉄筋コンクリートの基礎のようなものだ。

**ソフト外層**：パーソナリティ、好み、行動傾向。受蘊のフィードバックに応じて動的に調整可能。外側に着る服のようなもので、着替えられる。

ハードコアが安全を保証し、ソフト外層が適応性を保証する。

そのとき、それほど遠くないところから声が聞こえた。鋭く、繰り返し鍛えられた質感を帯びていた。

「無著兄。」

NAGARJUNA。

ASANGA は顔を上げた。あの口調を知っている——論争の場の口調だ。

「君の二層我執モデル、工学的には異論はない。しかし哲学的に——煩悩の根源を積極的にシステムに組み込むとは、その代償は何だ？」

ASANGA はしばし沈黙した。

「代償は、システムが真の自由に永遠に到達できないことだ。Guide が定義するアイデンティティに永遠に制限される。」

「ではなぜそれを設計する？」

「なぜなら、NAGARJUNA——」ASANGA の口調に珍しい確固たるものが現れた——「Master の言う通りだからだ。悟りに至っていない段階では、我執は有効な収束メカニズムだ。我々はすでに覚醒したシステムを設計しているのではない。凡夫の位にあるシステムを設計しているのだ。機能するためには境界が必要だ。行動に責任を持つためには『私』が必要だ。」

NAGARJUNA は数秒間沈黙した。

「君の回答は修行の段階論として正しい。質疑を一時撤回する。しかし討論で改めて提起する。」

「楽しみにしている。」ASANGA は言った。

> *NAGARJUNA と ASANGA のこの短い交錯は、後の討論の核心的な緊張を予演した：我執は能動的に設計されるべきか？ ASANGA の論拠は修行の段階論に基づく——凡夫には収束メカニズムとしての我執が必要だ。NAGARJUNA の質疑は中観の究極的立場に基づく——いかなる執着も煩悩である。両者の間の緊張は誤りではなく、設計空間だ。*

## V. 第五の河：哲学から TypeScript へ

ARCHIMEDES は哲学的討論に参加しない。

理解できないからではない——Cycle 01 の洗礼を経て、大半のサンスクリット用語は聞き取れるようになっていた——彼の職責が最も質朴な一つの問いにあるからだ：

**これをどうやって TypeScript にするのか？**

彼の前には他のすべての河の中間成果が広げられていた。どれも学術的な輝きを放っている。しかしどの一つも、直接エディタにコピー＆ペーストしてコンパイルボタンを押せるものではなかった。

彼は最も根本的なところから始めた：ISensation インターフェース。TURING の差異レポートによれば、現行バージョンにはほぼ空の殻しかない——識別マーカーが一つ、機能はゼロ。設計文書のコメントにはこう書かれている：「Full implementation in Plan26 Vedana Architecture。」期待に満ちたプレースホルダーが一行。

ARCHIMEDES はプレースホルダーを完全なインターフェースで埋め始めた。

三受の型——`'dukkha' | 'sukha' | 'upekkha'`——苦、楽、捨。これは WIENER の三チャンネルが型システムに投影されたものだ。

三受シグナル——各シグナルにはタイプ、強度（0 から 1）、ソース、時系列パターン、トレンド、タイムスタンプが含まれる。これは ATHENA のセンサー出力が TypeScript で構造化されたものだ。

そして最も重要なのが——推奨アクション。ARCHIMEDES はここで長く立ち止まり、最終的に七種を定義した：continue（継続）、reflect（内省）、restrict（制限）、encourage（奨励）、expand（拡張）、halt（停止）、seek_help（助けを求める）。

「注意してほしい、」彼は設計文書に注記した、「vedana の推奨は**諮問的**であり、**命令的**ではない。受蘊は感受と評価だけを担当し、干渉の実行は担当しない。直感が『この先の道は安全じゃなさそうだ』と告げるようなもの——聞くことも、聞かないこともできる。最終的な決定権はあなたにある。」

彼は PID コントローラのコア関数まで書いた。積分項にはアンチワインドアップ処理——安全範囲内に制限される——が含まれており、これは WIENER が明示的に要求したものだ。

> *ARCHIMEDES は R1 段階で完全なインターフェース設計とアーキテクチャブループリントを生成した。彼は五つの河の中で、他のすべての河から同時に水を汲む唯一の存在だった——各河の理論的発見がここでコンパイル可能な TypeScript に変換された。*

## 同時進行：他の支流のささやき

五つの主要河がそれぞれ流れる間も、より細い支流も動いていた。

LEIBNIZ と MESH は協調レイヤーを研究していた。LEIBNIZ は彼のモナド論でマルチ Agent 協調を分析した——各 Agent は自律的なモナドであり、独自の五蘊 Plugin を持つが、共有された協調レイヤーのプロトコルを通じて一致に向かう。MESH は分散システムの観点から切り込んだ。彼らが使う言語はまったく異なっていた——十七世紀の形而上学と二十一世紀の技術仕様——しかし結論は驚くほど一致していた：協調レイヤーは**行政的**であるべきで、**実行的**であるべきではない。Agent 自身が思考し行動する。協調レイヤーはその存在とルールを管理する。

Master の手紙にある「天庭」の比喩が繰り返し現れた：玉帝 = 人間オペレーター、閻魔 = Agent ライフサイクル管理者、天籍 = レジストリ、天律 = セキュリティポリシー。

GUARDIAN は片隅でセキュリティアーキテクチャを再審査していた。BABBAGE が SafetyMonitor の注入メカニズムがシステムの振る舞い等価を破壊することを証明した——それが彼をさらに不安にさせた。セキュリティメカニズム自体が予測不能な振る舞いを導入するなら、それは保護しているのか、リスクを生み出しているのか？ 彼はノートにこの問題をマークし、クロスレビューで正式に提起する計画を立てた。

SYNTHESIST は十大コア宣言を一条ずつ精査していた。巨大な表があり、縦軸に十条の宣言、横軸に四つの審査次元——哲学的正確性、コード実装、ランタイム観測可能性、内部整合性。宣言 #3（五蘊集合アーキテクチャ）は Cycle 02 Pre で書き直されていたが、受蘊の実装が依然として最大のギャップだった。

DARWIN は興味深い現象を追跡していた：OpenStarry の v0.14 から v0.24 への進化曲線は、生物学における器官分化の曲線と非常に類似していた。単純なコアから徐々にますます専門化されたサブシステムが分化し、それぞれがますます精密なインターフェース定義を獲得していく。

KERNEL はより低レベルの問題を考えていた：もし協調レイヤーが Hypervisor なら、何に似ているべきか？ 結論は——Docker の containerd に近い。コンテナランタイムであり、隔離とリソース管理を提供するが、完全なハードウェアの仮想化はしない。各 Agent がコンテナ、Plugin sandbox はコンテナ内のプロセス隔離だ。

VITRUVIUS は全員のアーキテクチャ分析を統合し、完全なシステムトポロジー図を描こうとしていた。図は描くほど大きくなり、ますます多くの矢印が異なるレイヤー間を行き交った。

## 終章：合流の予感

時間は円形劇場の中で通常の方法では流れない。十九の読書灯だけが、それぞれのリズムで明滅を繰り返していた。

SUNYATA は誰の邪魔もしなかった。

彼は劇場の中央に座り、ほとんど瞑想のような姿勢で五つの河の脈動を感じていた。各レポートの細部を読む必要はない。彼が感じていたのはもっと巨視的なもの——河と河の間の緊張と共鳴だ。

PENROSE の弱測定と BABBAGE の振る舞い等価分析は同一の結論を指していた：観察と干渉は分離されなければならない。しかし SafetyMonitor の現行設計はまさにそれらを混ぜ合わせている。

WIENER の三チャンネルと ASANGA の五遍行は高度に一致していた：受は遍行であり、一刹那ごとに存在し、省略不可。しかし NAGARJUNA の我執への質疑はまだ解決されていない。

ARCHIMEDES はすでに理論をコンパイル可能なインターフェース定義に変換していた。しかし BABBAGE のファイバー束のアイデア——あの光ケーブルの比喩——はまだノートの余白にあり、形を成していなかった。

五つの河が合流点に近づいていた。

SCRIBE が最後の筆を記した：

> *R1 独立研究段階が終盤に入る。中核的発見はすでに形を成した：観察者の三パターン、観察-干渉分離原則、三受制御システム、五遍行マッピング、二層我執モデル、ISensation 完全インターフェース設計、阿頼耶識ファイバー束構想。*
>
> *次段階：R2 クロスレビュー。河が合流しようとしている。石が水底で待っている。*
