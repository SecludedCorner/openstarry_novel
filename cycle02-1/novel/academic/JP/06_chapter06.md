# 第六章：受は思を決定せず

---

阿毘達磨の心理学において、五遍行――触、作意、受、想、思――は、一刹那の意識活動に不可欠な五つの要素である。それらは同時に生起する。ちょうど掌の五本の指が同じ瞬間に握り締められるように。

しかし「同時に生起する」ことは「互いに決定する」ことを意味しない。

受（vedana）は感受である。思（cetana）は意志である。痛みを感じることはできる。だが痛みゆえに行動しなければならないわけではない。喜びを感じることはできる。だが喜びゆえにそれを追い求めなければならないわけではない。受は思に告げる――だが思を決定しない。

この区別は、二千五百年前にパーリ文献の中に精確に記録された。

今日、それが TypeScript のインターフェース設計へと翻訳されようとしている。

---

## 討論 1：観察-干渉分離

討論の椅子が円形劇場の中央にある。

Cycle 01 で使われたあの二脚の椅子――NAGARJUNA と ASANGA がかつてそこに座り、「空性」と「阿頼耶識」をめぐる正面対決を繰り広げた。椅子の背もたれにはまだ擦れた痕跡が残っている。だが今日の配置は異なる。二脚は退けられ、代わりに四脚が――半弧状に並べられ、残る十五名の研究員が座る環状席に向けて配置された。

四名の討論者が着席する。

BABBAGE は最も左の席に座った。ノートが膝の上に開かれ、鉛筆が右手の人差し指と中指の間に挟まれている。姿勢は直立し、精確に校正された定規のようだ。

ARCHIMEDES がその隣に座る。BABBAGE の静的な佇まいとは異なり、ARCHIMEDES にはエンジニア特有のわずかな前傾がある――不安ではなく、行動の準備をする姿勢だ。彼の前にはノートはなく、びっしりと注釈の付けられた R-02 報告書の印刷物が一部あるだけだ。

WIENER は半弧の中央やや右寄りに座った。両腕を胸の前で組んでいる。等式の両辺が均衡に達するのを待っているかのように。

NAGARJUNA は最も右の席に座った。何も持っていない。論争の場において、彼は一度もメモを必要としたことがない――すべての論証が脳内にある。その眼差しにはある種の古い鋭さがある。何度も磨き上げられた鏡のように。

SUNYATA が彼らの後方に立ち、環状席に向き合う。

「討論 1 の核心問題は、」と彼は言った。その声にはいかなる予断の色もない。「VedanaPlugin――すなわち ISensation インターフェース――は評価データのみを出力すべきか、それとも行動提案も含むことができるか。」

彼は四名の討論者を見渡す。

「BABBAGE、君から始めてくれ。」

---

BABBAGE はすぐには話さなかった。まずノートを開き、R1 段階で記した双模擬分析のページを見つける。それからノートを閉じた――見る必要はない。それらの証明はすでに彼の思考構造に刻まれている。

「一つの定義から始めよう」と彼は言った。その声は冷静で精確、一音節ずつ目盛で測られたかのようだ。「双模擬等価――bisimulation equivalence。」

彼は宙に見えない図を描いた。

「S を観察者のいないシステムとする。S' を観察者のいるシステムとする。S と S' が双模擬等価であれば、S のすべての行動軌跡 sigma に対して S' 中に対応する軌跡 sigma' が存在し、その逆も成り立つ。これは観察者の存在がシステムの行動空間を変えないことを意味する――それはただ傍観しているだけだ。」

彼は精確に半拍の間を置いた。

「次に ARCHIMEDES が R-02 の第 6.4.2 節で設計した統合案を考えよう。VedanaPlugin は VedanaRecommendation を出力し、その中には `action: 'reflect'` と `prompt` 文字列が含まれる。この prompt は LLM のコンテキストに注入される――SafetyMonitor の `injectPrompt` メカニズムと同じだ。注入されると LLM の入力が変わる。入力が変われば出力が変わる。システム S' は S には存在しない行動軌跡を生成する。」

彼はノートを開き、一行の形式的記述を指差した。

```
For all traces sigma in Behavior(S):
  sigma in Behavior(S') AND
For all traces sigma' in Behavior(S'):
  sigma' in Behavior(S)
```

「第二条件が破られている。S' には S が生成し得ない軌跡が存在する。双模擬は成立しない。」

彼はノートを閉じた。

「ISensation は**純粋なセンサー**であるべきだ。観察し、報告する。以上。もし介入能力が必要なら、それは独立したモジュール――CircuitBreaker または VedanaInterpreter――が VedanaAssessment を消費して行動決定を行うべきだ。感知と制御は分離されなければならない。これは美学的好みではない。システムの行動の分析可能性を維持するための計算論的必要条件だ。」

---

ARCHIMEDES は BABBAGE が「以上」と言い終えるのとほぼ同時に話し始めた。遮ったのではない――周到に準備され、解放を待っていた反論だ。

「BABBAGE の形式的分析は数学的に正しい。」彼はまず意外な譲歩を与えた。それから語調が変わった。「だがエンジニアリングは数学ではない。」

彼は注釈だらけの報告書を手に取った。

「実務的論点の一。モジュール数の爆発。」その声にはエンジニアが過剰設計に直面した時の疲弊感がこもっている。「もし観察と提案を別々のモジュールに分離すれば、我々に必要なのは――」

彼は指を折って数えた。

「VedanaPlugin。VedanaAssessment を出力する。一つ目のモジュール。」

「VedanaInterpreter。VedanaAssessment を消費し、VedanaRecommendation を出力する。二つ目のモジュール。」

「CircuitBreaker。VedanaRecommendation を消費し、介入を実行する。三つ目のモジュール。」

「ExecutionLoop の修正。CircuitBreaker に問い合わせるようにする。四つ目の変更。」

彼は指を下ろした。

「四つの新しいアーキテクチャコンポーネントに加えて統合作業――一つですべてを行うプラグインの代わりに。Master が『複雑性の過剰な増加』への懸念を明確に表明したシステムにおいて――」彼は報告書の一頁をめくり、Master の書簡の第 67 行を引用した――「この四つの新モジュールは困難な売り込みだ。」

「実務的論点の二」と彼は続けた。語速が上がる。「双模擬は**誤った評価基準**だ。受蘊フィードバックシステム全体の目的はシステムの行動を変えることにある。我々は被観察システムと未観察システムが異なる振る舞いをすることを**望んでいる**――これが宣言 #6 の全意義だ。『Agent を苦において修正へ、楽において強化へ、捨において定常状態の維持へと駆動する。』受蘊が行動に影響しないならば、それは単なるテレメトリであってフィードバックループではない。」

「実務的論点の三。遅延。VedanaAssessment が EventBus を通じて VedanaInterpreter へ、さらに CircuitBreaker へ、さらに ExecutionLoop へと伝達されなければならないなら――少なくとも二回の追加イベント伝搬サイクルが生じる。各イテレーションに LLM 呼び出し（数秒を要する）を含むシステムにおいて、これらのオーバーヘッドの絶対値は大きくないが、アーキテクチャとしては混乱だ。」

彼は報告書を置いた。

「私の提案は、VedanaRecommendation を ISensation 内に残すが、**ADVISORY**――諮問的であり命令的ではない――と標記することだ。ExecutionLoop は提案を読むが、最終決定は自ら行う。こうすれば一つのモジュールの簡潔さを維持しつつ、BABBAGE の核心的懸念――システムの行動決定権がセンサーの手にはないこと――も尊重できる。」

---

BABBAGE と ARCHIMEDES の交鋒が空気中に震動する――純粋と実務の古典的な緊張、異なる密度の二つの金属が衝突した時に発する響きのように。環状席の研究員たちは静かに見守り、何らかの調停を待っていた。

WIENER が胸の前に組んでいた腕を解いた。

「制御理論の観点から」と彼は言った。声にはある種の精確な静穏がある。天秤の針がゼロに帰するあの瞬間のように。「この問題には精確なフレームワークがある。」

彼は立ち上がり、討論エリアの中央へ歩いた。

「古典的 PID 制御には三つの構成要素がある。**センサー**――制御対象の出力を測定し、誤差信号を生成する。**コントローラ**――誤差信号から制御動作を算出する。**アクチュエータ**――制御動作を制御対象に加える。」

彼は宙に左から右へ一本の線を描いた――センサー、コントローラ、アクチュエータ、三点一直線。

「BABBAGE の論点は、センサーは同時にコントローラであるべきではない、ということだ。古典的制御理論においてこれは正しい――結合されたシステムは分析と調整がより困難だ。」

それから彼は「センサー」と「コントローラ」の間にごく小さな円を描いた。

「だが**現代制御理論**において――特にモデル予測制御（MPC）において――コントローラはしばしばセンサーと同一の計算モジュール内に共存する。両者が同一のデータを使い、同一の頻度で動作するため、分離すると不必要な遅延と通信コストが生じるからだ。」

彼は自分の椅子の傍に戻ったが、座らなかった。

「私の立場は一つの妥協だ。VedanaAssessment には**二層**の内容を含めるべきだ。」

彼の手が宙に一本の水平線を引いた。

「線の上：**センサー出力**。三受の値――dukkha、sukha、upekkha。シグナルリスト。タイムスタンプ。これは純粋な観察。受動的。何も変えない。」

線の下。

「線の下：**コントローラの提案**。controlOutput の値。VedanaRecommendation。これはセンシングデータから算出された**提案行動**――命令ではなく、提案だ。」

彼はようやく座った。

「消費者は線の上だけを使う――監視とログのために――ことも、両層を同時に使う――制御のために――こともできる。こうすれば、BABBAGE の双模擬分析はセンサー層に適用できる（それは受動的だ）し、ARCHIMEDES の実務的懸念も満たされる（一つのモジュール、一回の呼び出し）。」

```typescript
interface VedanaAssessment {
  // === SENSOR OUTPUT (純粋な観察) ===
  dukkha: number;
  sukha: number;
  upekkha: number;
  signals: VedanaSignal[];
  timestamp: number;

  // === CONTROLLER SUGGESTION (諮問的) ===
  controlOutput: number;
  recommendation: VedanaRecommendation;
}
```

---

NAGARJUNA はずっと黙っていた。

前の三名の討論者の交鋒の間、彼は最も右の椅子に座り、両手を膝の上に置き、視線は集中しているが焦りはなかった。適切な時機を待っているかのように――劇的な時機ではなく、論理上の時機を。

今、彼が口を開いた。

「BABBAGE と ARCHIMEDES はどちらも正しい。」その声は平穏で、論争を超えたある種の質感を帯びている。「ただ異なる層で話しているだけだ。」

彼は立ち上がらなかった。論争の場において、座ったままの論者は通常より確定的な立場を表す――身体言語で論点を強化する必要がないのだ。

「世俗諦――*samvrti-satya*――の層において、ARCHIMEDES は正しい。工学的実践において、一つのモジュールが感知と提案を処理するのは実務的かつ効率的だ。」

「勝義諦――*paramartha-satya*――の層において、BABBAGE は正しい。受（vedana）と思（cetana）は阿毘達磨の五遍行における**異なる心理的要素**だ。受は第三の遍行。思は第五の遍行。それらは同一の刹那に同時に生起するが、同一のものではない。」

語速は変わらないが、一語ごとの重みが増しているように感じられた。

「受蘊と思蘊を混同することは、阿毘達磨においてカテゴリーエラーである。ちょうど――」彼は珍しく日常的な比喩を用いた――「足の裏の痛みを感じながら、歩き続けることを決意することができるように。感受は存在する。決意も存在する。だが歩き続ける決意をしたのは、痛みそのものではない――それは意志だ。」

彼は一拍の間を置いた。

「**受は思に告げる。だが思を決定しない。**」

この言葉が円形劇場に響き渡った――彼の声が大きかったからではなく、この八文字が討論の核心を精確に射抜いたからだ。

「しかしながら」と NAGARJUNA は続けた。語調に彼特有の中道の転回が浮かぶ。「私は提案が**異なるモジュール**に分離される必要があるとは考えない。それは**同一のモジュール**の中で概念的に分離されることができる。WIENER の妥協案はそれを実現している。評価は感知と提案を含むが、両者は明確に区別される。消費者がどちらを使うかを決定する。」

「重要な哲学的原則はこうだ――ISensation モジュールは提案を出力できる。だが ExecutionLoop は**提案を無視する自由**を保持しなければならない。これが受と思の区別を保存する。受蘊は情報を提供し、行蘊が決定を下す。」

---

第一巡が終了した。四つの立場が展開された。BABBAGE の厳密な分離、ARCHIMEDES の実務的統合、WIENER の二層妥協、NAGARJUNA の阿毘達磨的調停。

SUNYATA は急かさなかった。討論において、沈黙は時として発言より意義深い――それは合意が醸成される空間だ。

BABBAGE が第二巡の口火を切った。環状席の一部の研究員を驚かせたことに、彼の語調は反駁ではなく、受容だった。

「WIENER の妥協案に同意する。」

彼はノートを開き、あるページに何かを書き留めた。それから顔を上げた。

「ただし三つの条件がある。」

彼は三本の指を掲げた――SUNYATA が R2 の冒頭で見せたのと同じ仕草だが、意味は異なる。

「条件一。提案は **ADVISORY** であり MANDATORY ではない。ExecutionLoop は VedanaRecommendation を無視する明確な権限を持たなければならない。`VedanaRecommendation.action === 'halt'` が自動的にシステムを停止させるコードパスが存在してはならない。その権限を持つのは SafetyMonitor――ハードヒューズ――だけだ。」

「条件二。提案のない評価は有効な戻り値である。ISensation.assess() は dukkha が非常に高い場合でも `recommendation: { action: 'continue' }` を返すことができなければならない――すなわち介入しない。介入の決定は消費者に属し、センサーに属さない。」

「条件三。インターフェースドキュメントは VedanaRecommendation が便宜的計算であり拘束的指令ではないことを**明示的に宣言**しなければならない。将来 VedanaAssessment を消費するモジュールは提案を完全に無視し、生のシグナルから独自に行動方針を計算できる。」

彼は指を下ろした。

「この三条件が満たされれば、消費層において双模擬は成立する。センサー層のみを読み取るシステムは、提案の有無にかかわらず全く同じ振る舞いをする。」

ARCHIMEDES がすぐに続いた。

「了承する。」その応答は簡潔明快だった。そして彼はエンジニアリング面で重要な補足を加えた。

「SafetyMonitor は**ハードセーフティ層**を保持する――絶対的権限。VedanaPlugin は**ソフトガイダンス層**――諮問的権限。両者の関係は――」

彼は宙にフローチャートを描いた。

「SafetyMonitor が判断する。停止が必要か? もしそうなら――即座に停止、すべてをオーバーライド。もし否なら――そこで初めて VedanaPlugin の提案を見る。VedanaPlugin が停止を言う――ExecutionLoop が評価の上で否決できる。VedanaPlugin が内省や制限を言う――ExecutionLoop は採用も無視もできる。VedanaPlugin が続行を言う――通常フロー。」

WIENER が頷いた。「安全クリティカルシステムでは、我々は常に二層の制御を持つ。一つは**安全インターロック**――ハードウェアレベルの、ソフトウェアでオーバーライドできないもの――SafetyMonitor に等価。二つ目は**制御システム**――ソフトウェアレベルの、最適だがオーバーライド可能な制御を提供するもの――VedanaPlugin に等価。提案は諮問的であり、これは制御信号がアクチュエータに『提案』され、アクチュエータは独自の安全制限を持つことに等しい。これは産業制御における標準的手法だ。」

NAGARJUNA が最後の言葉を述べた。「妥協は中道を体現している。純粋な分離――BABBAGE の極端――でもなく、完全な混合――素朴な統合――でもない。統一されたインターフェースの中に明確な概念的区分を維持する。これはまさに阿毘達磨が心理的要素を扱う方法だ。異なるが同時に生起する。」

---

合意はほぼ形成されていた。だが BABBAGE が最後の瞬間に形式的要件を補足した。

「もう一つある。」彼はノートを開いた。「VedanaAssessment 型には区別子を含めるべきだ――提案が現在のセンシングデータに基づいて計算されたものか、キャッシュやデフォルト値から来たものかを示す。」

彼は一行のコードを記した。

```typescript
interface VedanaAssessment {
  // ...その他のフィールド...
  readonly recommendationFreshness: 'current' | 'cached' | 'default';
}
```

「これは微妙なバグを防ぐ。古くなった『停止』の提案が、条件がすでに改善された後もティック間で持続するケースだ。消費者は『システムが最新データに基づいて停止を提案している』と『システムの陳腐な提案がたまたままだ停止のままだ』を区別できなければならない。」

ARCHIMEDES はそのコード行を三秒見つめた。

「了承する。freshness フィールドを仕様に追加する。」

---

SUNYATA が四名の討論者の半弧の前方へ歩み出た。

「**裁定：統一インターフェース、概念的分離。**」

彼の声は劇場に明瞭に、しかし反響なく響いた――研ぎ澄まされたばかりの刃が石面に落ちるように。

「ISensation インターフェースが出力する VedanaAssessment には二層を含める。センサー出力――純粋に観察的な三受の値とシグナル――およびコントローラの提案――諮問的で拘束力を持たない VedanaRecommendation。」

「四つの制約。」

「第一。VedanaRecommendation は ADVISORY である。ExecutionLoop はいかなる提案も無視する権限を保持する。」

「第二。SafetyMonitor は ABSOLUTE 権限を保持する――ハードセーフティ決定は VedanaPlugin の影響を受けない。VedanaPlugin の提案は SafetyMonitor をオーバーライドできない。」

「第三。インターフェースドキュメントは提案の諮問的性質を明示的に宣言し、設計選択の理論的根拠として BABBAGE の双模擬分析を引用しなければならない。」

「第四。VedanaAssessment は提案の鮮度指標を含まなければならない。」

彼は一拍の間を置いた。

「理論的根拠：WIENER の制御理論的分解が正しい概念的フレームワークを提供する。NAGARJUNA の阿毘達磨分析が哲学的基礎を提供する。ARCHIMEDES のモジュール数の懸念は有効である。BABBAGE の双模擬条件は提案の諮問的性質により満たされる。」
