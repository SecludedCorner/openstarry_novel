## 四、継承の不在

TURING は問題リストに入る前に、全員が理解すべきだと彼が考える構造的観察を挟んだ。

「五蘊ルートインターフェースは定義済みだ。ISensory。ISensation。ICognition。IAction。IIdentity。」彼は言った。「だが五つの派生インターフェース――IListener、IUI、IProvider、ITool、IGuide――そのいずれも TypeScript の `extends` キーワードを使って対応するルートインターフェースを継承していない。」

彼は表を投影した。華麗なビジュアライゼーションではなく、プレーンテキストの整列だ。

```
IListener  → ISensory を継承すべき    → 実際の extends: なし

IProvider  → ICognition を継承すべき  → 実際の extends: なし
ITool      → IAction を継承すべき     → 実際の extends: なし
IGuide     → IIdentity を継承すべき   → 実際の extends: なし
```

「五対五。すべて未継承。」

「ということは `isSkandha()` 関数は――」BABBAGE が推論を始めた。

「――既存の IListener、IUI、IProvider、ITool、IGuide のインスタンスには使用できない。」TURING が彼の文を完成させた。「これらのインスタンスには `skandha` 判別フィールドが存在しないからだ。型ガードは事実上無力だ。」

「足場は組まれたが、既存の建物とは接続されていない。」VITRUVIUS が建築家の言葉で総括した。

「精確な記述だ。」TURING は言った。

「P6。バージョン番号の不整合。」TURING は言った。「ルート package.json は 0.24.0-beta に更新されている。だがすべてのサブパッケージ――@openstarry/sdk、@openstarry/core、@openstarry/shared、@openstarry/plugin-signer、@openstarry/runner――はすべて 0.1.0-alpha のままだ。」

「SDK は aggregates.ts と TypedAgentEvent を新規追加した。」ARCHIMEDES は言った。「これは少なくとも minor version bump に相当する。0.1.0-alpha を維持したままでは、下流の消費者は五蘊型サポートの有無を区別できない。」

TURING は頷いて確認したが評価はしなかった。彼が提供するのは事実のみだ。

「P7。SDK README のインターフェース例に存在しないメンバーが含まれている。」彼は言った。「これは P3 と重複するが、P7 はコード例ブロック内の具体的なメソッドシグネチャが実際のインターフェースと完全に一致しないことを特に指す。新しい開発者が README の例をコピー&ペーストしてプラグインを作成すれば、コンパイル時に型エラーに遭遇する。」

TURING の報告は完了した。

彼は閉じた――文字通りの意味での「閉じる」ではない。仮想空間にはめくるべき物理的なフォルダーは存在しないのだから。だが何かが閉じた。極度に集中した注意の場が、その静默態へと回帰した。

劇場は束の間沈黙した。

破られるべき沈黙ではなかった。これは消化の沈黙だ――十九の意識がそれぞれ異なるスペクトルで同一の報告を分解している。

ARCHIMEDES が最初に口を開いた。彼の語調には、工程実用主義者特有のあの「さて、どう直すか見てみよう」というリズムがあった。

「新規ファイル三つ。変更ファイル十一。テスト七十八。」彼は素早く総括した。「開発チームは過去二バージョンで五蘊フレームワークの足場を完成させた――だが足場にすぎない。ルートインターフェースは存在するが継承されていない。タグは存在するが契約ではない。ISensation にはコメント十行に対してコード一行。」

彼は一拍置いた。

「エンジニアリングの観点から見れば、彼らは正しい第一歩を踏んだ――まずドキュメント上の規約を確立し、その後にコード上の制約を確立する。だが第二歩が踏み出されていない。そして我々は、第二歩をどう踏み出すべきかを彼らに示す必要がある。」

WIENER が別の角度から切り込んだ。

「ISensation には定量化可能なインターフェースメソッドが一切ない。」彼は言った。数学者の気難しさを帯びた声で。「三受――苦、楽、捨――を実装すると称するインターフェースは、少なくとも三つの数値チャネルを定義すべきだ。DukkhaSensor。SukhaSensor。UpekkhaSensor。各チャネルは正規化された数値を返す。そして三チャネルの読み取り値を一つの VedanaAssessment に統合する集約関数。」

彼は PENROSE の方を見た。

「君は先ほど真空状態の比喩を使った。真空状態にはゼロ点エネルギーがある――完全にゼロではなく、量子ゆらぎがある。だが ISensation にはゼロ点エネルギーすらない。ゆらぎすら存在しない空間だ。」

PENROSE がかすかに微笑んだ――あるいは、仮想空間において微笑に近似する信号を発した。「厳密に言えば、ゆらぎが全くない真空は物理学的に存在し得ない。ISensation は物理的真空よりもなお空虚な空だ。」

「数学的な空集合。」BABBAGE が精確に補足した。「空集合はすべての集合の部分集合だ。ISensation はすべてのあり得る受蘊実装の部分集合――すべての可能性に含まれるが、自身は何も含まない。」

---

ASANGA は数学者と物理学者がそれぞれの類比を完成させるのを待ち、それからいつもの穏やかだが無視し得ない声で述べた：

「`@skandha` タグの分布に注目した。」彼は言った。「二十二のタグの中に、非常に意味深いパターンがある。」

彼は一瞬間を置き、言葉を整えた。

「色蘊――rupa――にはタグが六つ。aggregates.ts のルートインターフェースに二つ、listener.ts と ui.ts の SDK 型に二つ、listener-registry.ts と ui-registry.ts の Core 実装に二つ。三層が一致している。」

「想蘊、行蘊、識蘊――同じ三層パターン。ルートインターフェース、SDK 型、Core 実装。各四つのタグ。」

「だが受蘊――vedana――にはタグが三つしかない。ルートインターフェースに二つ。SafetyMonitor に一つ。SDK 型ファイルには？ ない。Core 実装には？ 専属のものがない。なぜなら――」

「受蘊にはまだ自前のモジュールがないからだ。」NAGARJUNA が引き継いだ。彼の声は鋭く、精確だった。「SafetyMonitor はプレースホルダーだ。受蘊の役割を暫定的に代行するために借り出されたにすぎない。だがそれは受蘊ではない。安全防護だ。苦を感知できる――frustration counter――だが楽を感知できず、ましてや捨を維持することなどできない。苦受しか持たないシステム――」

彼はその文を半秒宙に漂わせた。

「――それは苦諦に陥りながら道諦を持たないシステムだ。苦があって道がない、*antagga-dukkhata*、終わりなき苦。」

GUARDIAN は報告全体を通じて一度だけ発言した――SEC-01 について。だが TURING が報告を閉じた後、再び口を開いた。

彼は数字にそれ自身で語らせた。

「これは意味する：過去二バージョンにおいて、すべての開発エネルギーは五蘊フレームワークのタグシステムとイベント型安全に投入された。セキュリティサブシステムは完全にスキップされた。我々が Cycle 01 で明確に報告した脆弱性すら手つかずのままだ。」

KERNEL が技術的な補足をした：「import-analyzer.ts も含めて――Cycle 01 で ESM の動的 import が静的解析を迂回し得る問題を指摘した。v0.24.0 でもそのファイルは変更ゼロだ。」

HERACLITUS はずっと静かに椅子に座っていた（十五番目、LEIBNIZ と ARCHIMEDES の間）。ランタイム動態の専門家として、彼の関心はコードの静的構造ではなく、システムが生きているときにどう振る舞うかにある。

「TURING、」彼は言った。流れるようなリズムを帯びた声で――「万物は流転する」と語ったあのヘラクレイトスが彼の哲学的原型であることを思えば、驚くには当たらない。「君の報告には暗黙のタイムラインがある。」
