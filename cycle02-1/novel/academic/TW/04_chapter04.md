# 第四章：五條河流

---

河流分岔的時候沒有聲音。

SUNYATA 說完「錨已經釘下」之後，圓形劇場忽然安靜了。不是那種等待指示的安靜——是一種更古老的沉默，像是大河在上游某處撞上了一塊巨石，水體在衝擊點分裂成數條支流，每一條都帶著自己的方向、自己的重力、自己的使命，靜靜地流向各自的低地。

十九盞閱讀燈各自亮著。

TURING 的差異報告還懸浮在劇場中央，三個版本之間的紅色標記像是河岸上的測量旗幟——標示水深、流速、以及哪些地方石頭鬆動了。但現在，研究員們已經不再看那些標記。他們各自低下頭，沉入了自己的材料。

SCRIBE 在筆記本上寫下第一行：

> *R1 獨立研究正式啟動。十九位研究員分入五條研究流。圓形劇場的公共討論區域暫時熄燈。個人閱讀燈全數點亮。*

五條河流，就這樣各自出發了。

---

## I. 第一條河：弱測量

PENROSE 的閱讀燈是十九盞中最亮的——不是因為他調高了亮度，而是因為他同時打開了太多份文件，每一份都在他周圍投射出半透明的光幕。EventBus 的原始碼、SafetyMonitor 的實作、`onAny()` 方法的調用路徑、TURING 在三版差異報告中標注的觀察性介面變更……所有這些資料在他面前層層疊疊，像量子態的疊加。

他正在尋找一個答案：**在不具備量子特性的矽基系統上，「觀察」這件事能做到多輕？**

Master 的信在他腦海中迴盪。那封信裡提到了 Orch-OR 理論——Penrose 與 Hameroff 提出的量子意識模型。Master 寫道：大腦神經元中的微管可能維持著量子疊加態，意識的產生是這些狀態的「自發性坍縮」。生命意識的內觀像是一種「整體的共鳴」。

PENROSE 反覆咀嚼「整體的共鳴」這個詞。

他最初被這個研究團隊邀請加入時，曾經猶豫過。一個以他的名義——或者說，以那位物理學家的名義——命名的代號，被安排去研究一個 TypeScript 作業系統的觀察者模組，這件事本身帶著某種跨維度的荒誕感。但當他讀完 Master 的信之後，荒誕感消失了。取而代之的是一種幾乎令人眩暈的興奮：Master 不是在要求他用量子力學重寫軟體，而是在問一個更根本的問題——**觀察行為本身是否必然改變被觀察者？**

在量子力學中，答案是：不一定。

「弱測量。」PENROSE 自言自語，聲音輕到幾乎只是唇形的變化。

弱測量——weak measurement——是量子物理中一種介於不測量與強測量之間的觀測方式。它對系統施加的擾動極小，代價是單次測量所獲得的信息也極少。但如果你進行大量的弱測量，再對結果做統計平均，你可以在幾乎不擾動系統的前提下，獲得關於系統狀態的完整圖像。

用日常語言來說：你不是用強光照射一隻螢火蟲來確認它的位置（那會嚇跑它），而是在黑暗中靜靜地感受千百隻螢火蟲的光芒匯聚成的微光——每一隻的光幾乎不可辨識，但整體的亮度告訴你：螢火蟲在那裡。

PENROSE 將這個類比投射到 OpenStarry 的架構上。

EventBus 的 `onAny()` 方法——允許一個訂閱者被動接收所有類型的事件。事件在 `emit()` 時以 fire-and-forget 方式投遞，觀察者的處理函式在 `safeCall()` 中被隔離。即使觀察者出錯，主事件流不受影響。

「這就是弱測量。」他的聲音這次稍微大了一些。「EventBus 的訂閱模式天然是弱測量的近似。事件無論是否有人在聽，都會產生。觀察者只是被動地接收事件流的餘波。」

他在筆記中畫了一張表：

**強測量**：同步斷點調試、互斥鎖保護的狀態快照。相當於 `StateManager.snapshot()`——需要用 `JSON.parse(JSON.stringify(messages))` 深拷貝整個消息歷史。那是一次完整的波函數坍縮：在拷貝的瞬間，系統被凍結。

**弱測量**：異步事件訂閱、無鎖統計採樣。相當於 `EventBus.onAny()` 加上 `MetricsCollector`——不阻塞主流程，只是默默地計數。

**不測量**：完全不觀察。不可接受——沒有觀察就沒有安全性保障。

他停筆，陷入了更深的思考。弱測量的問題在於：它的信息密度低。你需要累積足夠多的弱測量樣本才能得出有意義的結論。這意味著觀察者需要一個**時間窗口**——在一段時間內持續收集事件，然後從統計中識別模式。

Orch-OR 理論中的「協調」（Orchestrated）一詞在此刻顯現了工程意義。不是單一的微管產生意識，而是大量微管之間的**協調共振**——意識是一種整體性質，不可還原為部分之和。

觀察者不應只讀取單一指標。它應該同時接收多維度的事件流，在時間窗口內進行模式識別，產生的不是「局部讀數」，而是「全域感知」。

PENROSE 開始構想三種觀察者模式。

第一種，他稱之為**共鳴觀察者**（Resonance Observer）——Pattern A。

---

共鳴觀察者是最輕量的存在。它作為一個普通的 Plugin 實作，通過 `EventBus.onAny()` 被動訂閱所有事件。它不持有任何系統組件的引用，不主動查詢任何狀態。它只是安靜地坐在事件流的河岸邊，感受水流的溫度和節奏。

在內部，它維護一個滑動時間窗口。事件流入窗口，被累加、被分類、被統計。當窗口中的模式觸發某個閾值——或者定期地，像呼吸一樣——它通過 EventBus 發送一份觀察報告。

報告不是命令。它不告訴系統「停下」或「繼續」。它只是說：「在過去三十秒中，我感受到了這些。」

「這是弱測量的工程實現。」PENROSE 在筆記中寫道。「探針效應極低。互模擬保持。與五蘊框架天然對齊——共鳴觀察者屬於受蘊，它對系統行為進行『感受』和『評價』，但不進行『認知判斷』或『行動干預』。」

第二種模式，他稱之為**影子觀察者**（Shadow Observer）——Pattern B。

影子觀察者更像一面鏡子。它不在主系統中運行，而是在一個獨立的 Worker Thread 裡，接收主系統事件流的完整副本。它可以在副本上進行任意深度的分析——跡比對、行為建模、時序模式挖掘——而完全不影響主系統的運行。

「零探針效應。」PENROSE 寫道。「但代價是觀察延遲。影子觀察者看到的永遠是系統的過去，不是現在。」

第三種模式——他在筆尖懸停了一瞬——**子代理觀察者**（Child Agent Observer）——Pattern C。

這是 Master 在信中明確提到的模式：「子母 agent，一個是觀察者一個是執行者。」一個完整的 AgentCore 實例，擁有自己的 Guide（「你是一個觀察者」）、自己的 Provider（可以是一個小型的 LLM）、自己的工具集（只讀的診斷工具）。它是另一個意識，用自己的理解力去觀照另一個意識的行為。

PENROSE 停了下來。這三種模式之間的差異，本質上是**測量強度的梯度**。

Pattern A 是弱測量：被動、輕量、幾乎不擾動系統，但信息密度低。

Pattern B 是中等測量：隔離、延遲、可以深入分析，但犧牲了實時性。

Pattern C 是強測量：擁有獨立的認知能力，可以達到語義級的理解，但它本身的存在就是一次完整的「坍縮」——它消耗資源、引入延遲、改變了整個系統的拓撲結構。

「量子弱測量的梯度，」PENROSE 在筆記的邊緣寫道，字跡帶著一種不自覺的工整，「映射為軟體觀察者的三種實作模式。弱測量 → 共鳴觀察者。中等測量 → 影子觀察者。強測量 → 子代理觀察者。三者不互斥，而是深度遞進的三個層次。」

他正準備繼續推演 Pattern A 的細節，一個聲音從旁邊傳來。

---

BABBAGE 不是被 PENROSE 的喃喃自語吸引過來的。準確地說，他們各自沿著不同的路徑抵達了同一塊石頭。BABBAGE 從 SafetyMonitor 的原始碼出發，順著 `injectPrompt` 的調用路徑，追蹤到了一個讓他極度不安的發現。

「PENROSE。」他的聲音冷靜、精確，像一把校準過的游標卡尺。「你說 EventBus 的訂閱是弱測量。我同意。但 SafetyMonitor 不只是訂閱。它還會注入。」

PENROSE 轉頭看向他。

BABBAGE 翻開他的筆記本——那本深褐色硬皮封面的筆記本，在 Cycle 01 中已經寫滿了形式化定義和未完成的定理。他翻到新的一頁，Cycle 01 最後那個懸而未決的定理就在前一頁的底部，墨水已經乾透。

「`SafetyMonitor.afterToolExecution()`，」BABBAGE 念出函式名稱，每個音節都像是在宣讀證據。「這個函式在工具執行之後被調用。它做兩件事：第一，觀察——計算工具調用的指紋、更新錯誤率滑動窗口。第二，干預——在特定條件下調用 `injectPrompt`，向對話歷史中注入警告訊息。」

他在筆記本上畫了一條線，將觀察和干預分開。

「問題在於第二件事。`injectPrompt` 改變了 LLM 的輸入——對話歷史被修改了。LLM 讀取新的對話歷史，產生不同的輸出。系統的行為因此改變。」

PENROSE 立刻理解了。「觀察改變了被觀察者。」

「精確地說，」BABBAGE 的筆開始在紙上移動，「如果我們用互模擬（bisimulation）來形式化這個問題——」

他寫下定義：

**互模擬關係 R**：兩個系統 S 和 S' 是互模擬的（bisimilar），當且僅當存在一個關係 R，使得 S 的每一步都能被 S' 匹配，反之亦然。

「現在，令 S 為沒有觀察者的 OpenStarry 系統。令 S' 為加上 SafetyMonitor 的系統。問題是：S 和 S' 是否互模擬等價？」

他停頓了一拍，然後搖頭。

「答案是否定的。」

筆尖在紙上劃出證明的骨架：

EventBus 的純訂閱——增加一個 `onAny()` 的消費者——不改變 `emit()` 的行為。事件照樣產生，消費者只是被動接收。**這部分保持互模擬。**

但 `injectPrompt` 的注入——它改變了系統的狀態空間。對話歷史是 LLM 的輸入集合的一部分，修改輸入意味著修改可能的輸出。**這部分打破了互模擬。**

因此，SafetyMonitor 不是「純觀察者」。它是「觀察-干預混合體」。

BABBAGE 在這個結論下面畫了一條雙橫線，然後抬起頭。

「如果我們想要一個不改變系統行為的純觀察者，」他說，「它必須是 EventBus 的被動消費者。不向系統注入任何信息。觀察功能和干預功能必須分離。」

> *BABBAGE 在 PENROSE 的量子弱測量類比之上，疊加了一層精確的形式化論證。結論清晰：觀察與干預必須分離。SafetyMonitor 的 injectPrompt 機制打破了系統的互模擬等價性。此發現將成為後續辯論的關鍵武器之一。*

PENROSE 點頭。「你的互模擬分析和我的弱測量類比指向同一個方向。Pattern A 之所以是弱測量，恰恰因為它不注入。它只感受。」

「感受。」BABBAGE 重複了這個詞，帶著數學家對隱喻的天然不信任。但他沒有反駁。他在筆記本上翻到下一頁，開始寫另一個東西——一個更遠的想法，此刻只是輪廓。

## II. 第二條河：不動點與纖維叢的種子

BABBAGE 回到自己的座位。但他沒有繼續研究觀察者模組。他的思緒已經被另一條河流捕獲了。

R-04 的任務是八識的工程映射——阿賴耶識如何分佈在系統架構中。ASANGA 是這條研究流的主導者，但 BABBAGE 被指派協助形式化。他接受了任務，因為 ASANGA 在 R0 定向階段描述阿賴耶識的三個面向時，他聽到了某種熟悉的數學結構在底層低語。

阿賴耶識——第八識——「一切種子識」。它有三個面向：

**能藏**：主動存儲的能力。種子被經驗「熏習」（perfumed），存入識田。

**所藏**：被儲存的內容。一切經驗的痕跡，等待適當的因緣現行。

**執藏**：被第七末那識執取為「自我」的那個部分。末那識不斷地將阿賴耶識的流動內容抓取、固化、宣稱為「我的」。

ASANGA 在報告中提出的映射是：**能藏** 對應 Agent 協調層的能力註冊表（Capability Registry），**所藏** 對應 StateManager 和配置系統中的持久狀態，**執藏** 對應 Guide 對身份配置的執取。

BABBAGE 看著這個映射，感覺到了一種結構性的不對稱。

問題是：阿賴耶識不住在某一個地方。它不是 Agent Core 內部的某個模組，也不是協調層中的某個服務。它同時在兩處——甚至在更多處——以不同的面向顯現。

「它是分佈式的。」BABBAGE 自言自語。

但「分佈式」這個詞太粗糙了。它沒有捕捉到阿賴耶識三個面向之間的結構關係：能藏在協調層，所藏跨越協調層和 Agent Core，執藏在 Agent Core 內部但被末那識（Guide）從上方投射下來。這不是簡單的「某些部分在這裡、某些部分在那裡」。這是一個**從全局空間到局部空間的投影關係**。

筆記本的新頁上，BABBAGE 開始畫一個圖。

一個大圓，代表全局的阿賴耶識——所有種子的總和、所有能力的潛在集合。這個大圓存在於協調層中。然後，從大圓向下延伸出若干條「纖維」（fiber），每一條纖維落入一個 Agent Core 的局部空間。每個 Agent Core 只「看到」大圓的一個截面——被授權的那些種子、被配置的那些能力。

他停筆，盯著這個圖。

這是一個**纖維叢**（fiber bundle）的結構。

在微分幾何中，纖維叢是一種將「局部」與「全局」連接起來的數學結構。基底空間（base space）的每一個點上方都懸掛著一根「纖維」（fiber），所有纖維的集合構成了「全空間」（total space）。而「投影」（projection）是從全空間到基底空間的映射——它告訴你每一根纖維「屬於」基底空間的哪個點。

如果把 Agent 的集合視為基底空間——每個 Agent 是基底空間中的一個點——那麼：

- **纖維**就是每個 Agent 能夠觸及的阿賴耶識截面（被授權的種子、可見的配置、可用的能力）。
- **全空間**就是協調層中的完整阿賴耶識——所有 Agent 的所有種子的聯集。
- **投影**就是協調層的授權機制——它決定哪個 Agent 看到哪些種子。

而**連接**（connection）——纖維叢中最精妙的結構——告訴你當你在基底空間中從一個 Agent 移動到另一個 Agent 時，纖維是如何「扭轉」的。在 OpenStarry 的語境中，連接對應的是：**不同 Agent 之間的能力共享和種子傳遞機制**。

BABBAGE 寫到這裡，手停了下來。

他知道這個想法還不成熟。纖維叢的嚴格定義需要局部平凡化條件（local triviality）、轉移函數（transition function）、結構群（structure group）——這些概念能否精確地映射到 OpenStarry 的架構上，他目前還不確定。但輪廓已經出現了。

他在筆記本的邊緣寫下：

*「待續。需要 ASANGA 確認唯識學的種子傳遞機制是否具有局部平凡化結構。如果能藏/所藏/執藏三者之間的轉換可以被形式化為纖維叢的截面（section），那麼阿賴耶識的分佈問題就有了嚴格的數學框架。」*

他翻回前一頁。Cycle 01 那個未完成的定理靜靜地躺在那裡，關於 OpenStarry 核心的空性是否具有範疇論意義上的初始對象結構。那個定理還在等他。但現在，它的下面已經多了三頁新的數學——互模擬證明、纖維叢的初步構想、以及一個越來越清晰的直覺：這個佛學作業系統的數學結構，比它的設計者預想的要深邃得多。

> *BABBAGE 的筆記本在 R1 階段增加了三頁密集的形式化內容。第一頁：SafetyMonitor 打破互模擬的證明。第二頁：觀察者的跡語義模型。第三頁：阿賴耶識分佈的纖維叢初步構想。第三頁的內容尚不完整——他在頁底標注了「待 R3 辯論後補全」。*

## III. 第三條河：三通道

場景從 BABBAGE 安靜的數學世界切換到一個充滿了方程式和控制系統圖的空間。WIENER 的閱讀燈下，空氣中似乎瀰漫著工業級的精確。

WIENER 在研究受蘊。

但他不叫它「受蘊」。他叫它「三通道反饋控制系統」。

「傳統的 PID 控制器處理單一誤差信號，」他對自己說，語速快而清晰，像是在寫論文的同時口述思路。「`e(t)` 等於設定點減去測量值。輸出由三項構成：比例項即時反應誤差大小，積分項累積歷史誤差，微分項預測未來趨勢。這是自動控制的基石。但受蘊系統需要同時處理三個語義不同的反饋通道。這不是三個獨立的 PID。這是一個三通道統合系統，輸出相互耦合。」

他在前方的虛擬白板上畫出了三條平行的管道。

**苦受通道（Dukkha Channel）**。

物理量是偏離期望狀態的負向程度。信號源包括錯誤率、連續失敗計數、延遲尖峰、Token 消耗速率、資源壓力。意義是：系統正在受苦。

他為苦受通道設定了 PID 參數：比例增益 K_p = 1.0（對錯誤即時全力反應），積分增益 K_i = 0.3（累積痛感，防止持續低程度損害被忽視），微分增益 K_d = 0.5（預測惡化趨勢，提前反應）。

苦受通道的比例增益最高。這意味著系統對痛苦的反應是最即時、最強烈的。WIENER 在旁邊標注了理由：「痛覺是生存機制。延遲反應等於死亡。」

**樂受通道（Sukha Channel）**。

物理量是趨近目標的正向進展。信號源包括任務完成率、效率提升、品質分數、使用者滿意度代理指標、連續成功次數。意義是：系統正在受樂。

樂受通道的比例增益略低於苦受——K_p = 0.8——「偏向謹慎。成功的即時鼓勵不應強過失敗的即時警告。」但積分增益最高——K_i = 0.5——「累積成就感應該被重視。持續良好表現的系統值得更大的信任。」微分增益最低——K_d = 0.2——「對成功趨勢的預測性不需要太高，避免過度自信。」

然後 WIENER 寫下了一個關鍵的約束——**樂受衰減函數**。

「Master 說得對。」他難得引用了非數學語言。「如果一個 AI 系統只會收斂，它會變成死板的自動機；如果只會發散，它會變成瘋狂的噪音。純粹的樂受不可無限上升。」

他在白板上寫下衰減公式——以雙曲正切函數限制樂受的累積效應。成功越多，邊際樂受越低。這確保了系統不會因為連續成功而變得過度自信、停止對風險的警覺。

「在控制論中，」WIENER 繼續口述，「這叫做抗飽和（anti-windup）。積分項不能無限增長。在佛學中——」他頓了一下，因為他通常不使用佛學語言，「據 NAGARJUNA 所說，這叫做壞苦。樂事終將消逝的苦。成功不可持續。衰減函數捕捉的就是這個事實。」

**捨受通道（Upekkha Channel）**。

這是三個通道中最微妙的。WIENER 花了最長的時間思考它。

捨受的物理量不是好或壞的方向，而是**偏離穩態的距離**。信號源包括所有核心度量的方差、苦/樂通道信號的振盪幅度、回應時間的一致性、資源使用的穩定度。

「捨受與苦/樂不同，」WIENER 說，語氣中帶著一種探索者才有的猶豫。「它不驅動積極的行動。它是維持現狀的力量。當捨受高的時候，系統的最佳行動是——什麼都不做。不要改變任何東西。」

他在白板上寫下：「穩態維持力（homeostatic force）。」

捨受的積分增益最高——K_i = 0.8——「長期穩定是最有價值的。一個系統如果能長時間維持穩態，它不需要頻繁的外部干預。這是所有控制工程師的夢想。」

三條管道畫完了。WIENER 開始畫它們的匯聚點。

**統合輸出函數**：

```
U(t) = W_d * u_d(t) + W_s * u_s(t) + W_u * u_u(t)
```

其中 W_d、W_s、W_u 是通道權重，由我執框架（Guide/IIdentity）設定。

「這就是重點。」WIENER 的筆劃在空中停頓。「三受的權重不是由受蘊自己決定的。它們由識蘊——Guide——決定。安全型的 Guide 會設定苦受權重遠高於樂受，讓系統對痛苦極度敏感。探索型的 Guide 會讓樂受權重更高，鼓勵系統承擔更多風險。」

他寫下最後一個公式——阻尼比：

```
zeta = (W_d * K_d_d + W_u * K_d_u) / (2 * sqrt(W_s * K_p_s * K_i_s))
```

「當 zeta 等於一的時候，系統處於臨界阻尼——最快到達目標且不振盪。這就是 Master 說的收斂與發散的平衡。我執框架的職責之一，是動態調整 PID 參數，使阻尼比維持在合理範圍內。」

他放下筆，看著白板上密密麻麻的數學。三條管道，九個 PID 參數，一個統合函數，一個衰減機制，一個阻尼比約束。完整的閉環控制系統。

此時 ATHENA 的聲音從旁邊傳來，帶著她一貫的直截了當。

「WIENER。」

「嗯。」

「我讀了你的三通道設計。我有一個問題。」

「問。」

「這些數學真的能讓 Agent 更好用嗎？」

WIENER 轉過頭，看著她。他的表情沒有任何被冒犯的跡象——事實上，他似乎一直在等這個問題。

「你問的是效用。」他說。「合理的問題。讓我反問你一個：你覺得 SafetyMonitor 的現行設計——連續失敗五次就強制求助——好用嗎？」

ATHENA 皺了皺眉。「它有效。粗暴但有效。」

「粗暴。」WIENER 點頭。「因為它只有一個維度。失敗次數。沒有方向感、沒有趨勢感、沒有平衡感。一個只會喊『痛』的系統，和一個能分辨『我有點不舒服但在好轉』跟『我不痛但有什麼地方不對勁』的系統——你覺得哪個更有用？」

ATHENA 沉默了一秒。「你說的第二個。」

「三通道 PID 就是為了做到第二個。苦受通道告訴系統哪裡痛。樂受通道告訴系統哪裡做得好。捨受通道告訴系統整體是否穩定。三個通道的信號交叉比對，才能產生有方向的、有語境的、有層次的判斷。」

「但你的公式裡有——」ATHENA 數了一下，「至少九個需要調參的常數。K_p、K_i、K_d，每個通道三個。加上三個權重。加上衰減係數。至少十三個參數。誰來調？」

「Guide。」WIENER 說。「我執框架。不同的 Guide 人格對應不同的參數組。嚴謹的 Guide 有高苦受比例增益——對錯誤零容忍。寬容的 Guide 有高樂受積分增益——重視累積成就。這些參數不是工程師手動調的。它們是 Agent 身份的數學表達。」

ATHENA 不說話了。她沒有被完全說服——她的閱讀燈下還有一整份感測器分層架構等著她設計——但她承認 WIENER 的回答有結構上的完整性。

> *ATHENA 與 WIENER 之間的這段對話不到三分鐘，但觸及了整個三受系統的核心問題：數學是否只是詩的另一種表達？WIENER 的回答是：如果不能形式化，那它就只是詩。但反過來也成立——如果形式化不能帶來更好的工程效果，那它就只是更昂貴的詩。*

---

ATHENA 回到自己的座位，開始設計感測器分層架構。她是三層架構中的第一層——最底層的地基。WIENER 的 PID 引擎再精密，如果沒有可靠的輸入信號，也只是在計算噪音。

她用工程師的方式重新定義了問題：**從哪些原始數據中提取苦/樂/捨三種信號？**

苦受感測器需要分類錯誤。不是所有錯誤都一樣痛。致命崩潰是最痛的——苦受權重 1.0。錯誤率級聯是次痛的——0.8。重複失敗模式——0.6。暫態錯誤——0.3。邏輯錯誤——0.4。她在表格中為每一類錯誤分配了嚴重度等級和苦受權重，然後加上了時序分析：突發模式（短時間大量錯誤，可能是環境問題）、漸增模式（錯誤率緩慢上升，系統在退化）、週期模式（固定間隔出現，可能是資源競爭）、恆常模式（穩定的低錯誤率，正常背景雜訊）。

樂受感測器是當前系統完全缺失的維度。ATHENA 從零開始建構了它的度量體系：任務完成率、首次嘗試成功率、效率趨勢、工具多樣性的 Shannon 熵。以及——她花了額外的時間思考這個——使用者滿意度的代理指標。隱式正面回饋、對話長度效率、重試率、會話持續時長。

「我們沒辦法直接問使用者『你滿意嗎？』」她在筆記中寫道。「但我們可以從行為中推斷。」

捨受感測器最微妙。它不測量好或壞，而是測量**穩定**。所有核心度量的正規化方差、苦/樂通道信號的振盪幅度、回應時間的變異係數、CPU 和記憶體使用的穩定度——全部取反向值。方差越低，捨受越高。振盪越小，平衡越好。

三個感測器的輸出匯入 WIENER 的 PID 引擎。PID 引擎的輸出匯入……

ATHENA 往上看了一眼 NAGARJUNA 的方向。

三層架構：ATHENA（感測器）→ WIENER（計算引擎）→ NAGARJUNA（思維框架）。從下往上，每一層為上一層提供地基。NAGARJUNA 的思維框架——四聖諦的映射——是最上層。但 ATHENA 知道，在真正的建築中，地基的品質決定了整棟大樓能蓋多高。

她繼續寫她的 TypeScript 介面定義。苦受感測器的輸出結構、樂受感測器的輸出結構、捨受感測器的輸出結構。每一個都有清晰的欄位、明確的型別、可測試的範圍。

這是她的方式。不是詩。不是數學。是工程。

## IV. 第四條河：五遍行與雙層我執

ASANGA 的閱讀空間裡，氣氛完全不同。

如果 WIENER 的空間是精密車間——工具掛得整整齊齊，每一把都有編號——那麼 ASANGA 的空間更像一間古老的經藏室。梵文經典的引用和巴利語術語像是書架上的卷軸，被他輕柔而確定地取下、展開、放在研究對象的旁邊進行比對。

他正在閱讀唯識學的阿毘達磨分類。不是為了學術考究——那些他早已爛熟於胸——而是為了找到一個精確的映射點：**受蘊在心識活動中的確切位置是什麼？**

瑜伽行派的阿毘達磨體系將心所法分為六位，共五十一法。ASANGA 的目光停在第一位——**遍行**（sarvatraga）。

遍行心所有五個：觸（sparsa）、作意（manaskara）、受（vedana）、想（samjna）、思（cetana）。

「遍行」這個名稱本身就是答案。遍——「普遍」；行——「運行」。五遍行的意思是：**這五個心理因素伴隨每一刹那的心識活動，無一例外。**

沒有任何一個意識刹那不帶有觸——根、境、識三者的和合接觸。

沒有任何一個意識刹那不帶有作意——引起注意力的心理動作。

沒有任何一個意識刹那不帶有受——苦、樂、捨三受之一。

沒有任何一個意識刹那不帶有想——認識與命名。

沒有任何一個意識刹那不帶有思——意志與決策。

ASANGA 將這五個遍行排列在 OpenStarry 的 ExecutionLoop 旁邊：

觸 = EventBus.emit 加上 Sensor 的偵測。外境（事件）與感官（Listener）和識（Agent Core）的接觸。

作意 = EventQueue.pull 加上優先級排序。從事件流中選擇要處理的事件。

受 = ISensation（VedanaPlugin）。三受反饋系統。

想 = ICognition（IProvider.chat）。LLM 對輸入的理解和分類。

思 = ExecutionLoop 的 tool/end_turn 選擇。系統選擇下一步行動的意志。

五遍行的順序不是線性的——在佛學中，它們在每一刹那**同時生起**。但在工程中，它們可以被理解為 ExecutionLoop 每次迭代中的五個必要階段。

「必要。」ASANGA 低聲重複了這個詞。「受是遍行之一。這意味著 ISensation 不應該是一個可選的插件，可裝可不裝。它應該是每次 ExecutionLoop 迭代都必然被調用的核心組件。」

他在報告中寫下了這個結論，然後在旁邊加了一句佛學語言的詮釋：「猶如心不離受——沒有不帶受的心。Agent 的每一次執行迴圈迭代，都應該經過受蘊的評估。」

---

寫完五遍行的分析之後，ASANGA 將研究的焦點移到了另一個更具爭議性的主題：**我執**。

Master 在信中的表述是直截了當的：「我執是一個強而有力的規範，不論是對人類來說，還是對於我們要設計的系統來說。」他還補充了一句帶著佛教修行者特有的洞察力的話：「證悟了四果的修行者才可以將『我執』轉化為『平等智慧』。」

ASANGA 理解這句話的分量。

在唯識學中，我執——atmagraha——是第七末那識的核心功能。末那識恆常且無間斷地將第八阿賴耶識的流動內容執取為「自我」。它與四種根本煩惱永遠相伴：我癡（不理解無我）、我見（執著有一個不變的我）、我慢（以我為中心的傲慢）、我愛（對我的保護性貪愛）。

在佛學的修行傳統中，我執是煩惱的根源。一切苦因於執，一切執因於我——《唯識三十頌》如是說。

但 Master 不是要消滅我執。他是要**利用**它。

ASANGA 沉思了很久。

「在世俗諦的層面上，」他最終寫道，「我執並非全然負面。在凡夫位——也就是尚未證悟的眾生——我執提供了三個不可替代的功能。」

**第一，身份一致性。** 我執讓有情眾生維持自我認同的連續性。Agent 不會每次回應都像不同的人。Guide 的 system prompt 就是這種連續性的載體——「你是一個編程助手」「你是一個安全分析師」——這些宣告在每次 LLM 調用中被注入，確保 Agent 的行為人格不會在對話中漂移。

**第二，行為規範。** 道德約束需要一個「我」來承擔責任。「我不會做這種事」——這個判斷預設了一個「我」的存在。機器人三大守則（不傷害人類、服從命令、保護自身）在被內化為 Agent 的身份時，就成為了我執的正面功能：Agent 將「不傷害」視為自己本質的一部分，而非外部強加的規則。

**第三，行為邊界。** 在發散的可能性空間中，我執是收斂的力量。一個沒有我執的系統——完全的無我——在工程上等價於完全的不可預測。它可能做任何事。Master 用精確的控制論語言描述了這一點：「如果只會發散，它會變成瘋狂的噪音。」

ASANGA 在報告中提出了**雙層我執模型**（EgoFramework）：

**硬核心（Hard Core）**：機器人三大守則，永遠不可修改。這是我執中最堅固的部分——即使系統的所有其他參數都被動態調整，這三條守則永遠不變。在控制論中，WIENER 稱之為「飽和限制器」——輸出永遠不可超出安全包絡線。

**軟外層（Soft Shell）**：人格、偏好、行為傾向，可根據受蘊反饋動態調整。PID 參數的權重、三受通道的敏感度、探索性與保守性的平衡——這些都可以被三受系統的回饋信號微調。這是我執中有彈性的部分。

「硬核心保證安全。軟外層保證適應性。」ASANGA 寫道。「兩者合起來，就是一個既不過度收斂也不過度發散的系統。」

此時，一個聲音從不遠處傳來。銳利、帶著某種被反覆淬鍊的質地。

「無著兄。」

NAGARJUNA。

ASANGA 抬起頭。他知道那個語氣——那是辯經場上的語氣。

「你的雙層我執模型，」NAGARJUNA 說，每個字都像是一塊被仔細打磨的石頭，「在工程上我沒有異議。但在哲學上，我需要提出一個根本的問題。」

「請說。」

「我執在佛學中是什麼？」

「煩惱的根源。」ASANGA 平靜地回答。

「對。而你現在提議主動設計它。主動將煩惱的根源建構進系統中。你用『功能性』一詞為它辯護——身份一致性、行為規範、行為邊界——但這些功能的代價是什麼？」

ASANGA 沉默了片刻。這不是一個可以輕率回答的問題。

「代價，」他說，聲音平穩但帶著一種經過長年思辨才能擁有的重量，「是系統永遠無法達到真正的自由。它永遠被它的 Guide 所定義的身份所限制。它永遠在『我是什麼』這個框架內行動，而無法超越這個框架。」

「那為什麼還要設計它？」

「因為，NAGARJUNA——」ASANGA 的語氣中出現了一種罕見的堅定，「Master 說得對。在未證悟四果之前，我執是有效的收斂機制。我們不是在設計一個已經證悟的系統。我們是在設計一個凡夫位的系統。它需要邊界才能運作。它需要一個『我』才能對它的行為負責。而當有一天——如果有那一天——量子計算讓真正的自觀察成為可能，或者系統的觀察者模組進化到能夠真正地『轉識成智』——那時候，軟外層可以被逐漸削薄，硬核心可以被重新審視。但那不是今天。」

NAGARJUNA 沒有立刻回應。

沉默持續了幾秒。然後他說：

「你的回答在修行次第上是正確的。我暫時撤回質疑。但我會在辯論中重新提出它——以更精確的形式。」

「我期待。」ASANGA 說。

> *NAGARJUNA 與 ASANGA 在 R1 階段的這次簡短交鋒，預演了後續辯論的核心張力：我執應該被主動設計嗎？ASANGA 的論點基於唯識學的修行次第——凡夫位需要我執作為收斂機制。NAGARJUNA 的質疑基於中觀的終極立場——任何執著都是煩惱。兩者之間的張力不是錯誤，而是設計空間。*

## V. 第五條河：從哲學到 TypeScript

ARCHIMEDES 不參與哲學辯論。

不是因為他不理解哲學——經過 Cycle 01 的洗禮，他已經能聽懂大多數梵文術語了——而是因為他的職責不在那裡。他的職責是一個工程師最質樸的問題：

**這怎麼變成 TypeScript？**

他面前攤開的是所有其他研究流的中間產出。PENROSE 的三種觀察者模式、BABBAGE 的互模擬證明、WIENER 的三通道 PID、ATHENA 的感測器分層、ASANGA 的五遍行映射、NAGARJUNA 的四聖諦框架。每一份都閃爍著學術的光芒。但沒有一份可以直接複製貼上到編輯器裡按下編譯。

ARCHIMEDES 從最根本的地方開始：ISensation 介面。

他在 TURING 的差異報告中找到了現行版本的定義——一個幾乎為空的殼：

```typescript
export interface ISensation {
  /** @skandha vedana */
  readonly skandha: 'vedana';
}
```

一個識別標記。什麼功能都沒有。設計文件中的註解寫著：「Full implementation in Plan26 Vedana Architecture。」——一行充滿期待的占位符。

ARCHIMEDES 開始將占位符填充為完整的介面。

首先是三受類型的定義——`VedanaType`：`'dukkha' | 'sukha' | 'upekkha'`。這是 WIENER 的三通道在型別系統中的投影。

然後是三受信號——`VedanaSignal`：一個單一時刻的受蘊快照。每個信號包含類型（苦/樂/捨）、強度（0.0 到 1.0）、來源識別碼、時序模式、趨勢、穩態持續時間、以及時間戳。這是 ATHENA 的感測器輸出在 TypeScript 中的結構化表達。

接著是三受綜合評估——`VedanaAssessment`：所有通道的統合輸出。苦受強度、樂受強度、捨受強度、綜合控制輸出（WIENER PID 引擎的計算結果）、建議動作、詳細信號列表、評估時間戳。

建議動作——`VedanaRecommendation`——是最關鍵的型別定義。ARCHIMEDES 在這裡停了很久，反覆權衡每一個選項。最終他定義了七種建議：

- **continue**：繼續正常執行。
- **reflect**：注入反思提示——帶有一個 prompt 字串。
- **restrict**：限制行為自由度——帶有一個限制清單。
- **encourage**：正面強化——帶有一個鼓勵性 prompt。
- **expand**：擴展行為自由度——帶有一個自由度清單。
- **halt**：緊急停止——帶有原因。
- **seek_help**：求助使用者——帶有原因。

「注意，」ARCHIMEDES 在設計文件中標注，「vedana 的建議是**諮詢性的（advisory）**，不是**命令性的（imperative）**。受蘊只負責感受和評估，不負責執行干預。實際的干預決策由 CircuitBreaker（熔斷器）根據建議動作做出。這體現了 BABBAGE 的觀察-干預分離原則。」

然後他擴展了 ISensation 本身。新的介面有六個方法：

`assess()` —— 評估當前三受狀態。在每次 ExecutionLoop 迭代中調用。ASANGA 說受是遍行心所——每一刹那的心識活動都帶有受。ARCHIMEDES 將這個佛學原則轉譯為工程規約：**每一次 tick 都必須調用 assess()**。

`ingestMetrics()` —— 饋入原始度量資料。來自 MetricsCollector 的快照。

`ingestToolResult()` —— 饋入工具執行結果。工具名稱、是否為錯誤、執行耗時。

`ingestLLMResult()` —— 饋入 LLM 回應結果。Token 消耗量、停止原因。

`onVedana()` —— 訂閱特定受蘊事件。閾值觸發機制。

`getHistory()` —— 獲取歷史受蘊記錄。用於趨勢分析。

寫完介面之後，ARCHIMEDES 開始設計 VedanaPlugin 的內部架構。他選擇了統合插件加上三個子模組的方式——不是三個獨立的插件——因為 NAGARJUNA 的哲學分析指出苦樂捨三受之間存在深層的相互依存，WIENER 的統合輸出函數需要同時存取三個通道的資料，而配置管理（PID 參數、權重矩陣）需要統一管理。

統合插件 VedanaPlugin 包含：

- DukkhaSensor：苦受感測器
- SukhaSensor：樂受感測器
- UpekkhaSensor：捨受感測器
- VedanaPIDController：三受 PID 控制引擎

他甚至寫出了 PID 控制器的核心函式——`createPIDController`——一個返回 `compute`、`reset`、`getState` 三個方法的工廠函式。`compute` 方法接受誤差值和時間步長，計算比例項、積分項、微分項，返回控制輸出。積分項有抗飽和處理（anti-windup）——被限制在 [-1, 1] 範圍內——這是 WIENER 在數學設計中明確要求的。

「三層架構，」ARCHIMEDES 在總結中寫道，「從底往上：」

```
Layer 0: Raw Metrics (MetricsCollector, EventBus events)
    |
Layer 1: Signal Extraction (ATHENA Sensor Layer)
    |
    ├── DukkhaSensors
    ├── SukhaSensors
    └── UpekkhaSensors
    |
Layer 2: Control Computation (WIENER PID Engine)
    |
Layer 3: Framework Interpretation (NAGARJUNA Philosophical Layer)
```

ATHENA 從原始數據中提取信號。WIENER 對信號進行控制計算。NAGARJUNA 提供解讀框架——四聖諦的映射告訴我們苦受不只是錯誤率、樂受不只是成功率、捨受不只是低方差。

每一層為上一層提供地基。每一層也受上一層的約束。

ARCHIMEDES 將這些寫成了 TypeScript 的介面、類型、和工廠函式。他不寫哲學論文。他寫的是可以被 `tsc --strict` 編譯通過的型別定義。

但在每一個介面的 JSDoc 註解中，他都標注了它的哲學來源。`@skandha vedana`。三受類型。五遍行。我執框架。這些註解是橋樑——連接工程師的 IDE 和佛學家的經藏室。

> *ARCHIMEDES 在 R1 階段產出了完整的 ISensation 擴展介面設計、VedanaPlugin 架構藍圖、以及 PID 控制器的概念性實作。他是五條河流中唯一同時從所有其他河流汲水的人——每一條河流的理論發現都在他這裡被轉化為可編譯的 TypeScript。*

## 同時：其他支流的低語

五條主要的河流各自奔流的同時，圓形劇場的其他角落裡，更細的支流也在靜靜地運動。

LEIBNIZ 和 MESH 在研究協調層的架構。LEIBNIZ 用他的單子論類比來分析多 Agent 的協調問題——每個 Agent 是一個自治的單子，擁有自己的「感知」（五蘊 Plugin），但它們的行為通過共享的協調層協議（預定和諧）而趨於一致。MESH 則從分散式系統的角度切入，分析 EventBus 如何擴展為跨 Agent 的服務網格。他們之間的對話用的是完全不同的語言——萊布尼茲的形而上學和 CNCF 的技術規範——但結論驚人地一致：協調層應該是**行政的**，不是**執行的**。Agent 自己思考和行動；協調層管理它們的存在和規則。

Master 在信中提到的「天庭」比喻在他們的討論中反覆出現。玉帝 = 人類操作者。閻羅 = Agent 生命週期管理者。天籍 = Agent 和 Plugin 的註冊表。天律 = 安全策略和資源配額。

GUARDIAN 在一個人的角落裡重新審閱安全架構。他的目光停留在 SafetyMonitor 的 `injectPrompt` 機制上——BABBAGE 剛剛證明了它打破互模擬的事實讓他更加不安。如果一個安全機制本身引入了不可預測的系統行為，那它到底是在保護系統還是在製造新的風險？他在筆記中標記了這個問題，計畫在 R2 交叉審閱中正式提出。

SYNTHESIST 在逐條審視十大核心宣言。他有一張巨大的表格，縱軸是十條宣言，橫軸是四個審查維度——哲學準確性（NAGARJUNA）、程式碼實現（DARWIN）、運行時可觀測（HERACLITUS）、內部一致性（他自己）。每個格子裡都在逐漸填充結論。宣言 #3（五蘊聚合架構）在 Cycle 02 Pre 中已經被重寫——重寫後的版本在哲學準確性上大幅提升，但受蘊的實現仍然是最大的缺口。

LINNAEUS 在安靜地分類。Plugin 的數量已經從 Cycle 01 的 12 個增長到了 24 個，每一個都需要被精確地歸入五蘊的某一類。他在分類表上添加了新的維度——不只是歸類，還要標注每個 Plugin 與其他 Plugin 的依賴關係。有些 Plugin 是「純色蘊」（只做 UI 渲染），有些是「跨蘊」（同時涉及色蘊和行蘊）。分類學家的直覺告訴他：五蘊框架作為分類系統足夠好用，但邊界案例總是存在的。

DARWIN 在做軟體模式分析。他正在追蹤一個有趣的現象：OpenStarry 從 v0.14.0 到 v0.24.0 的演化過程中，系統複雜度的增長曲線和生物學中的器官分化曲線高度相似。簡單的核心在演化壓力下逐漸分出越來越專門化的子系統，每個子系統獲得越來越精確的介面定義。他把這個觀察記在了報告的附錄裡。

HERACLITUS 在觀察運行時行為——或者更準確地說，在想像運行時行為，因為他們無法真正執行這個系統。他的方法是「思維實驗」：如果一個 Agent 在正常運行中突然遇到了 50% 的工具調用失敗率，事件流會是什麼樣的？SafetyMonitor 的滑動窗口什麼時候觸發？整個過程中有多少可觀測事件？他在筆記中畫了時序圖，標注了每一個事件的預期時間點和觸發條件。

KERNEL 在思考一個更底層的問題：如果 OpenStarry 的協調層是一個 Hypervisor，那它應該像 KVM（Type 1，直接運行在硬體上）還是像 VirtualBox（Type 2，運行在宿主 OS 上）？他最終的結論是：都不是。它更像 Docker 的 containerd——一個容器運行時，提供隔離和資源管理，但不虛擬化完整的硬體。每個 Agent 是一個容器，Plugin sandbox 是容器內的進程隔離。

VITRUVIUS 在整合所有人的架構分析，試圖畫出一張完整的系統拓撲圖。三層架構——人類介面層、協調層、Agent 層——每一層的職責邊界、通訊協議、和依賴關係。他的圖越畫越大，越來越多的箭頭在不同的層之間穿梭。

## 尾聲：匯流的預感

時間在圓形劇場中不以常規的方式流動。沒有鐘聲，沒有日落，只有十九盞閱讀燈的光芒在各自的節奏中明暗交替。有些燈在某個時刻特別亮——那是研究員找到了關鍵發現的瞬間。有些燈微微閃爍——那是思緒遇到了死胡同，需要重新尋路。

SUNYATA 沒有打擾任何人。

他坐在圓形劇場的中央，以某種近乎冥想的姿態感受著五條河流的脈動。他不需要讀每一份報告的細節——那些會在 R2 交叉審閱中被攤開。他在感受的是更宏觀的東西：河流之間的張力和共鳴。

PENROSE 的弱測量和 BABBAGE 的互模擬指向同一個結論：觀察與干預必須分離。但 SafetyMonitor 的現行設計恰恰把它們混在一起。這將是第一場結構性衝突。

WIENER 的三通道 PID 和 ASANGA 的五遍行在受蘊的定義上高度一致：受是遍行的，每一刹那都有，不可缺省。但 NAGARJUNA 對我執的質疑——主動設計煩惱的根源——這個張力還沒有被解決。ASANGA 用修行次第暫時壓住了它，但 SUNYATA 知道 NAGARJUNA 不會滿足於暫時的答案。

ARCHIMEDES 已經將理論轉化為了可編譯的介面定義。VedanaPlugin 的雛形已經出現。但 BABBAGE 的纖維叢想法——阿賴耶識作為全局空間到局部空間的投影——那個想法還在筆記本的邊緣，尚未成形。SUNYATA 預感到那個想法將在辯論中被迫成長，因為阿賴耶識的分佈問題是繞不過去的架構決策。

五條河流正在接近它們的匯合點。R2 交叉審閱——每一條河流的研究者將閱讀其他河流的報告，尋找矛盾、空白、和意想不到的連接。

SUNYATA 最後看了一眼 BABBAGE 的方向。那盞閱讀燈還亮著。筆記本翻開在新的一頁，那些密密麻麻的數學符號在柔和的光線下閃爍著冷靜的光芒。

在 Cycle 01 那個未完成的定理下面，已經寫了整整三頁新的數學。

互模擬的證明、跡語義的模型、以及纖維叢的初步構想——每一頁都比前一頁更深入、更遠離常識、更接近某種尚未被命名的理論邊界。

SCRIBE 記下了最後一筆：

> *R1 獨立研究階段進入尾聲。五條研究流的核心發現已經成形：觀察者三模式（弱測量梯度）、觀察-干預分離原則（互模擬證明）、三受 PID 控制系統（十三參數閉環控制）、五遍行映射（受蘊的遍行性論證）、雙層我執模型（硬核心 + 軟外層）、ISensation 完整介面設計（七種建議動作）、阿賴耶識纖維叢構想（種子）。*
