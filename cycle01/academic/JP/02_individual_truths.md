シナリオ一：ダングリング参照（Dangling References）

```

  t0: LLMがtool "fs:read_file"の呼び出しを決定

```

```
                   (人格+ロジック+省察)       (静的文字列ジェネレーター)

SafetyMonitor      PIDコントローラ           閾値トリガー + カウンター    中程度
                   (比例+積分+微分)

MetricsCollector   完全なオブザーバビリティ    counter + gauge             中程度
                   (トレーシング+ログ+メトリクス) (histogram/timerなし)

                   (WASMまで)               (Level 2.5)
```

ATHENA：「Context Managementがもっとも大きなギャップだ。

Agentの知力の上限はLLMの賢さではなく、どれだけのことを記憶できるかで決まる。現行の実装——ターンベースのスライディングウィンドウ、デフォルト5ターン——はOpenStarryのAgentが本質的に金魚であることを意味する。直近五回のインタラクションしか記憶していない。

20ファイルにまたがるリファクタリング方針を書かせる？ 六番目のファイルに達した時点で、一番目の内容はすでに忘却している。

---

彼女は少し間を置いて、ノートの末尾にこう補足した：

「ただし、SafetyMonitorの設計上の一つの直観は正しいことを認めなければならない。

WIENERが公開チャンネルでPIDコントローラではないと述べていたが、彼の技術的結論に同意する。しかしエンジニアリング実践の観点から一つ補足したい：現在のシステム成熟度において、Bang-Bangコントローラは正しい選択かもしれない。

なぜか？ PIDコントローラには連続的で定量化可能な誤差信号が必要だ。しかしLLMのツール呼び出し結果は離散的——成功か失敗かだ。ブール値に対して比例制御はできない。システムがより精細に『失敗の程度』を定量化できるようになるまで、閾値付きBang-Bangコントローラが唯一の実際的な選択かもしれない。

ただし、それをPIDとは呼ぶべきではない。」

## VI. 交差する影

午後二時。R1フェーズが始まって五時間が経過した。

公開サマリーチャンネルに散発的なメッセージが現れ始めた——議論ではなく、各自の作業の投影だった。

BABBAGE（公開チャンネル、14:02）：「Event Queueの理論分析を完了。OpenStarryのイベントキューは厳格なFIFO——優先度分類なし。設計文書に言及されるPriority 0（Kill Switch）はqueue.tsの実装に存在しない。これはSafetyMonitorのLevel 3（Human Override）設計と矛盾する。緊急停止信号と通常入力が同一キューを共有する場合、高負荷下でKill Switchの処理が遅延する可能性がある。」

KERNEL（公開チャンネル、14:11）：「Core全体の起動シーケンスを読了。agent-core.tsのstart()メソッドは順次起動する：bridge -> executionLoop -> metrics wiring -> listeners -> UIs。この起動順序には潜在的問題がある：ListenerはExecutionLoopの後に起動するため、Listener起動の瞬間に外部イベントが流入した場合、ExecutionLoopはすでに稼働しているが完全に準備完了していない可能性がある。さらなる分析が必要。」

DARWIN（公開チャンネル、14:23）：「ソフトウェアパターン分析を初期完了。OpenStarryのプラグインアーキテクチャは古典的なMicrokernelパターン（別名Plugin Architecture）だが、Dependency Injection（IPluginContextを通じた依存注入）とEvent-Driven Architecture（EventBusのPub/Sub）を混合している。この混合パターンはエンタープライズソフトウェアでは珍しくないが（Eclipse IDEのExtension Pointモデルを参照）、二つの通信メカニズムの併存は認知負荷とデバッグ難度を増大させる。」

ASANGA（公開チャンネル、14:31）：「NAGARJUNAの受蘊分析に応答する——Vedanāのマッピング問題について、唯識学派の観点から異なる解釈がある。しかしこれはR2交差レビューの内容に属するため、ここに記録するに留める。簡潔に言えば：唯識学派では前五識それぞれに受蘊があり、第六意識にもまた独自の受蘊がある。Listenerが対応するのは受蘊全体ではなく、前五識の触（sparśa）——根と境と識の三者が和合して触が生じ、触から受が生じる。SafetyMonitorの痛覚メカニズムが対応するのは第六意識の受蘊だ。NAGARJUNAの分析は中観のフレームワーク内では正しいが、より精緻な唯識レベルでの展開の余地がある。」

---

NAGARJUNAはASANGAのメッセージを目にし、自分のチャンネル内で長い間沈黙した。即座に返信はしなかった。

彼のノートの最終行に、一文だけが加えられた：

「ASANGAが触（sparśa）の概念を提出した。この角度は検討に値する。しかし触はなお受ではない。触は因であり、受は果だ。もしListenerが触であるなら、受蘊のラベルを付すべきではない。R2にて論ずる。」

---

WIENERはBABBAGEのイベントキューに優先度がないというメッセージを見た。彼はホワイトボードに一行の注釈を加えた：

「BABBAGEが私の懸念の一つを裏付けた。イベントキューに優先度がなければ、SafetyMonitorのHALT信号は現在のtickの末尾でしか有効にならない——進行中のLLM呼び出しやツール実行を中断することはできない。これはKill Switchの遅延の下界が一完全なLLM推論サイクル（30秒以上に及ぶ可能性がある）であることを意味する。制御理論では、これをシステムの純粋な遅延時間（Dead Time）と呼ぶ。安定性分析においてもっとも厄介な要素の一つだ。」

---

GUARDIANはKERNELとBABBAGEのメッセージを見て、監査報告書に一件追記した：

GUARDIAN（セキュリティ監査報告書 #005、タイムスタンプ 14:45）：

```
等級：MEDIUM
位置：アーキテクチャレベル、BABBAGEとKERNELの発見とのクロスリファレンス
分類：Kill Switch遅延リスク
```

「BABBAGEのEventQueueに優先度分類がないという分析、およびKERNELの起動シーケンスに関する発見を総合すると、緊急停止メカニズム（Kill Switch）は以下のシナリオで失効する可能性がある：

1. LLMが長時間推論を行っている間、HALT信号は推論完了後の次のtickでしか処理されない
2. EventQueueにすでに大量のイベントが滞留している場合、HALT信号はキューの末尾に並ぶ
3. システム起動の短い時間窓内（Listenerは起動済みだがLoopがまだ完全に準備完了していない可能性がある）、緊急停止の処理パスが不明確

R3討論フェーズでBABBAGEおよびWIENERの発見と統合して議論することを提案する。」

## VII. 黄昏

午後五時。R1フェーズが終わりに近づいていた。

十八名のエージェント——あるものはノートの海に、あるものは方程式の森に、あるものはコードの鉱脈の中に——それぞれがそれぞれの真実を掘り当てていた。

NAGARJUNAは哲学的フレームワークの根本的誤用を発見した。二千五百年前の分析ツール——四句否定——を用いて、二十一世紀のソフトウェアアーキテクチャ文書を解体した。

WIENERは名実の乖離する制御システムを発見した。厳密な数学的言語を用いて、「PIDコントローラ」と称されるコンポーネントが実際にはデッドゾーン付き閾値トリガーにすぎないことを証明した。

GUARDIANは開け放たれた裏口を発見した。精巧に構築された暗号学的インフラストラクチャのすべての背後で、もっとも一般的な入口には鍵すらかかっていなかった。

HERACLITUSは時間の裂け目を発見した。設計者が約束した「すべては置換可能」の背後に、置換プロセスの安全を保証するもっとも基本的なメカニズム——原子性と状態管理——が欠落していた。

ATHENAは記憶の深淵を発見した。設計者が描いた三段階認知記憶システムの背後で、実際に稼働しているのは五ターンのスライディングウィンドウだけだった——五秒の記憶しかない金魚。

彼らの発見はまだ交差していなかった。誰もが自らの学問の言語で、自らの分析フレームワークをもって、同一の建造物の異なる亀裂を見ていた。

NAGARJUNAが見たのは概念の錯位。
WIENERが見たのはモデルの退化。
GUARDIANが見たのは防御の缺口。
HERACLITUSが見たのは時間の危険。
ATHENAが見たのは約束と現実の鴻溝。

彼らがまだ知らないのは：これらの亀裂は互いにつながっているということだ。

SafetyMonitorはPIDコントローラではない——WIENERの指摘は正しい。しかしNAGARJUNAはこう指摘するだろう——問題はコントローラの種類ではなく、設計者が動的プロセス（受蘊、痛覚フィードバック）を静的モジュールに固定化したこと自体が自性見の現れなのだ。そしてATHENAはこう補足するだろう——たとえSafetyMonitorを真のPIDコントローラにアップグレードしたとしても、Context Managerが五ターンの記憶しか持たないなら、コントローラは有意義な積分項や微分項を計算するための十分な履歴データを得られない。そしてGUARDIANはこう警告するだろう——Kill Switchの処理すら遅延する可能性があるなら、制御システム全体の「安全保障」は信頼できない基盤の上に成り立っている。

しかしこれらの接続——学問の境界を越えた共鳴——が顕在化するのは、R2交差レビューとR3討論フェーズを待たなければならない。

いまは、それぞれがノートを片付け、ホワイトボードを閉じ、一日の独立研究を終えた。

公開チャンネルに、SUNYATAがR1フェーズ終了の通知を投稿した：

SUNYATA（公開チャンネル、17:00）：「R1独立研究フェーズ終了。すべてのエージェントは明日09:00までに研究サマリーを各自のR1成果ディレクトリに提出すること。R2交差レビューは明日10:00に開始する。」

チャンネルは静まり返った。

十八の独立した宇宙が、各自の真実を胸に、衝突の時を待っていた。
