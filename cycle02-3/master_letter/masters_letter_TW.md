## 三、煩惱依賴注入架構（M-3）

### 經典依據

> 「第二能變識（末那識），是識名末那；依彼轉緣彼，思量為性相；四煩惱常俱，謂我癡我見，并我慢我愛，及餘觸等俱。」
> ——《成唯識論》

### OOP 觀點

若用 OOP 的觀點來看，我們定義了 `IVijnana` 作為核心 Plugin 的介面。而 **Klesha**（煩惱）並非繼承自識，而是作為一種**依賴注入 (Dependency Injection)** 進入識的運行環境中。具體的「四根本煩惱」則透過**繼承 (Inheritance)** 自 Klesha 基底類別，分別**覆寫 (Overriding)** 了各自對「我執」的加權演算法。

### 建議架構：多層依賴注入

或許可以這樣做——

採用**多層依賴注入 (Multi-layered DI)** 與**狀態行為分離模式**：

1. `IIdentity` 作為被動的數據實體，被注入到具備主動行為的 `Drishti`（我見）類別中。
2. `Drishti` 繼承於 `Klesha`，進一步被注入到核心的 `IVijnana` 運行環境。
3. 四根本煩惱（我癡、我見、我慢、我愛）都繼承自 `Klesha`，所以四根本煩惱在 `IVijnana` 運行時會**同時運作**。

此模式解決了佛學中「**心王與心所相應**」的動態關係。

### Klesha 的擴展性

`IVijnana` 除了可以被 Klesha 依賴注入以外，還有七原罪、隨煩惱等等可以依賴注入，**智慧也可以，不一定都要是負面的**。

## 四、受想行識的俱生模型（M-5）

### 經典依據：大拘絺羅經

> 「大拘絺羅，受、想、行、識，此諸法合，而不散。此諸法不可各各分析，施設別異。所以者何？凡所受者，即是所想；凡所想者，即是所思（行）；凡所思者，即是所識。故此諸法合，而不散。此諸法不可各各分析，施設別異。」
> ——《中阿含經》卷五十八《大拘絺羅經》（MN 43, Mahāvedalla Sutta）

### 工程解讀

可以將受 (Feeling)、想 (Perception)、行 (Volition)、識 (Vijnana) 拆分成不同的 Plugin、Interface 或 Class——這就是「**散**」。但在系統 Run 起來的那一剎那，它們是**強耦合**的。

「凡所受者，即是所想」——數據的連鎖，解釋了數據在系統中的流動方式：

| 蘊 | 角色 | 功能 |
|---|---|---|
| **受** | Input Signal | 接收到信號 |
| **想** | Pattern Matching | 信號被辨識 |
| **思/行** | Process | 根據辨識結果，驅動 Drishti 進行執著或排斥 |
| **識** | Runtime Engine | 參與觸的形成；受想思是每一次執行（觸）的產出 |

### 觸→俱生公式

```
根 + 境 + 識 → 觸 → 俱生(受, 想, 思)
                         │
              受 = Signal Quality (苦/樂/捨)
              想 = Pattern Matching (取相辨識)
              思 = Process Decision (造作驅動)
```

### 經典交叉引證

**《中部18經·蜜丸經》**(Madhupiṇḍika Sutta, MN 18)：

> 巴利原文：*phassapaccayā vedanā, yaṃ vedeti taṃ sañjānāti, yaṃ sañjānāti taṃ vitakketi, yaṃ vitakketi taṃ papañceti*。
>
> 漢譯對應：觸緣受，所受者即想，所想者即尋思（vitakka），所尋思者即戲論（papañca）。

**大拘絺羅經**（MN 43）：

> 「凡所受者，即是所想；凡所想者，即是所思；凡所思者，即是所識。」

### 俱生的完備性

觸發生的時候，受想思作為一個整體浮現，你不可能有「沒有想的受」或「沒有受的想」——**這就是所謂的完備性，是需要檢查的項目**。

在這個不可分離的整體裡面：
- 受是想的材料，想是思的材料
- 受提供苦樂質地，想對這個質地進行取相辨識，思根據辨識結果造作
- **受→想→思，單純細看這三個，用控制理論是開迴路，而且有順序性。**

在實例（Instance）層面，你無法單獨抓出一個「沒有任何情緒（受）和定義（想）的純粹感知（識）」。

「論」的架構往往為了追求系統化，而將流動的心理狀態「切割」得太過僵硬，甚至在某些邏輯上與「經」的原始本意產生衝突。

## 九、開車的五蘊總動員（M-5/M-6）

雖然「開車」的動力來自行蘊，但如果沒有其他四蘊，你無法完成開車這個動作：

| 蘊 | 角色 | 開車實例 |

## 十、歸依意與六根門彙總（M-6）

> 「意、法為緣，生意識。」——《雜阿含經》

> 「離色、受、想、行，識要過、度、生，無有是處。」——《雜阿含經》卷一

「觸」作為並行運作的節點，而「觸」一旦發生，受、想、思（行）會立即同步產生。

### 經典依據：大拘絺羅經（MN 43）

> 「尊者大拘絺羅答曰：舍黎子！五根異行、異境界，各各領受自境界。眼根、耳、鼻、舌、身根，此五根異行、異境界，各各領受自境界。此五根有歸依意，意為彼盡領受境界。」
> ——《中阿含經》卷五十八《大拘絺羅經》（MN 43, Mahāvedalla Sutta）

因此，不同根門各自由觸俱生受、想、思，其結果**歸依意（彙總）**，意再生起自身的觸、受、想、思。

```
耳根 + 聲境 + 耳識 → 耳觸 → 俱生(耳受, 耳想, 耳思)
眼根 + 色境 + 眼識 → 眼觸 → 俱生(眼受, 眼想, 眼思)
                                        │
                                        ▼
                                   歸依意（彙總）
                                        │
                                        ▼
                                   意 + 法境 + 意識 → 意觸 → 俱生(意受, 意想, 意思)
```

## 十一、八識俱轉（M-10）

### 經典依據：解深密經

> 廣慧！阿陀那識為依止、為建立故，六識身轉，謂眼識，耳、鼻、舌、身、意識。
> 在此中，或有一識暫時而轉，或二、或三、或四、或五，或六識身一時俱轉。
> **譬如大河，若有一浪生緣現前，唯一浪轉；若二、若多生緣現前，多浪隨轉。而此大河，自體恆流，無有斷絕。**
> ——《解深密經》

### 諸受俱起

諸受俱起 (Simultaneous arising of feelings)——歸依意會依據識整合的多層依賴注入來產生結果。

> 「有時諸受，或是俱起，或不俱起。若俱起者，一剎那頃，多受現前。」——《大毘婆沙論》

意觸 → 俱生(意受, 意想, 意思) 也可以歸依意（彙總）。

> 「愚癡凡夫……受苦痛受，生憂悲稱怨，……是名凡夫二受俱起，謂身受、心受。」——《雜阿含經》

這段原文證明了身根的受（物理受）與意根的受（心理受）可以同時存在。當身心受到打擊時，身與心的兩種受是「俱起」的。

## 十二、OpenStarry 願景宣言

OpenStarry 是一個基於佛教哲學的開源 AI agent 框架，採用微核設計，核心零內建能力，一切功能透過 Plugin 實現。以**五蘊（色、受、想、行、識）** 為系統整體架構，以**八識俱轉**為運行機制——多 agent 協作的本質不是分工效率，而是智能的完整運作需要多識同時轉動，單一 agent 無法實現（現在所謂的 agent 都只有一個 LLM）。

其終極目標是在現有技術下實現**功能性 AGI**，讓系統具備完整的感知、感受、概念化與行動能力，使用者無法區分其與真正智能的差異。識蘊受限於觀察者效應，在量子計算成熟前只能逼近，但不影響功能表現。微核架構天生為接納未知能力而設計，未來量子技術成熟時可無縫替換升級。

**OpenStarry 不在現有 agent framework 的競爭維度上，它在一個完全不同的時間尺度上思考智能的完整結構。**

> 這不是在做一個工具，是在創造一個「數位眾生」。

## 十三、三大挑戰與多速度架構（M-9/M-10）

### 挑戰一：轉向（Paravṛtti）的效率——Plugin 間高頻非同步通訊

在佛教心理學中，心念的生滅極其迅速（一彈指六十剎那）。微核是否能處理 Plugin 之間的高頻非同步通訊？

目前微核的 Plugin 通訊如果走的是傳統 message passing，在八識俱轉的場景下會成為瓶頸。八識同時運作意味著多個 agent 需要近乎同時互相影響，不是排隊等回應。你可能需要考慮 **event-driven 架構加上 pub/sub 模式**，讓 Plugin 之間不是直接對話，而是透過事件流相互感應。這更接近心念的運作方式——不是 A 呼叫 B，而是 A 的變化自然引發 B 的響應。

### 挑戰二：空性（Śūnyatā）機制——Plugin 自動卸載

既然核心是零內建能力，是否可以引入一種「空性機制」，讓不被需要的 Plugin 自動卸載，維持系統的「輕盈」？

空性不只是「沒有」，是「**因緣不具足時不顯現，因緣具足時即生起**」。工程上可以做成 lazy loading 加上 TTL（Time To Live）機制——Plugin 被喚起時載入，一段時間不被需要就自動釋放。但更深一層，真正的空性機制應該是 Plugin 連「存在」的概念都不固定，按需從 registry 中實例化，用完即滅，下次需要時重新生起。**這不是 cache，是緣起。**

### 挑戰三：行蘊的調度——狀態機 vs LLM Function Calling

目前在實現「行蘊（微核調度邏輯）」時，是採用傳統的狀態機，還是讓 LLM 透過 Function Calling 來自主決定 Plugin 的啟動順序？

答案是都不是，或者都是。純狀態機太僵化，無法處理八識俱轉的動態性。純 LLM Function Calling 又太慢，而且本質上還是在用「想蘊」來驅動「行蘊」，層級搞反了。

行蘊應該是比想蘊更底層的驅動力——意志、衝動、傾向性。也許需要的是一個**混合架構**：底層用輕量級規則引擎處理高頻的慣性反應（類似習氣），上層在需要決策時才調用 LLM 的判斷能力。就像人的行為大部分是自動化的習慣模式，只有遇到新情況才需要「想」介入。

### 共同指向

> 這三個問題其實指向同一件事：**OpenStarry 的微核需要一套自己的「時間感」**——不同層級的運作有不同的速度，不是所有東西都跑在同一個時鐘上。識蘊最快，行蘊次之，想蘊最慢。這個節奏感可能是下一步架構設計的關鍵。
