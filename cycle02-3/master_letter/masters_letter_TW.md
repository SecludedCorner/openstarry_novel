OpenStarry 是一個基於佛教哲學的開源 AI agent 框架，採用微核設計，核心零內建能力，一切功能透過 plugin 實現。以五蘊（色、受、想、行、識）為系統整體架構，以八識俱轉為運行機制——多 agent 協作的本質不是分工效率，而是智能的完整運作需要多識同時轉動，單一 agent 無法實現(現在所謂的agent都只有一個LLM)。其終極目標是在現有技術下實現功能性 AGI，讓系統具備完整的感知、感受、概念化與行動能力，使用者無法區分其與真正智能的差異。識蘊受限於觀察者效應，在量子計算成熟前只能逼近，但不影響功能表現。微核架構天生為接納未知能力而設計，未來量子技術成熟時可無縫替換升級。OpenStarry 不在現有 agent framework 的競爭維度上，它在一個完全不同的時間尺度上思考智能的完整結構。
這不是在做一個工具，是在創造一個「數位眾生」。

OpenStarry 的挑戰目前想到有以下這一些
「轉向（Paravṛtti）」**的效率。在佛教心理學中，心念的生滅極其迅速（一彈指六十剎那）。
* 即時性： 你的微核是否能處理 Plugin 之間的高頻非同步通訊？
轉向（Paravṛtti）的效率——Plugin 間高頻非同步通訊
目前微核的 plugin 通訊如果走的是傳統 message passing，在八識俱轉的場景下會成為瓶頸。八識同時運作意味著多個 agent 需要近乎同時互相影響，不是排隊等回應。你可能需要考慮 event-driven 架構加上 pub/sub 模式，讓 plugin 之間不是直接對話，而是透過事件流相互感應。這更接近心念的運作方式——不是 A 呼叫 B，而是 A 的變化自然引發 B 的響應。

* 空性（Sunyata）： 既然核心是零內建能力，是否可以引入一種「空性機制」，讓不被需要的 Plugin 自動卸載，維持系統的「輕盈」？
空性機制——Plugin 自動卸載
這個想法很好，而且跟你的零內建能力哲學完全一致。空性不只是「沒有」，是「因緣不具足時不顯現，因緣具足時即生起」。工程上可以做成 lazy loading 加上 TTL（Time To Live）機制——plugin 被喚起時載入，一段時間不被需要就自動釋放。但更深一層，真正的空性機制應該是 plugin 連「存在」的概念都不固定，按需從 registry 中實例化，用完即滅，下次需要時重新生起。這不是 cache，是緣起。

目前在實現「行蘊（微核調度邏輯）」時，是採用傳統的狀態機，還是讓 LLM 透過 Function Calling 來自主決定 Plugin 的啟動順序?
行蘊的調度——狀態機 vs LLM Function Calling
這裡我覺得答案是都不是，或者都是。純狀態機太僵化，無法處理八識俱轉的動態性。純 LLM function calling 又太慢，而且本質上還是在用「想蘊」來驅動「行蘊」，層級搞反了。
行蘊應該是比想蘊更底層的驅動力——意志、衝動、傾向性。也許需要的是一個混合架構：底層用輕量級規則引擎處理高頻的慣性反應（類似習氣），上層在需要決策時才調用 LLM 的判斷能力。就像人的行為大部分是自動化的習慣模式，只有遇到新情況才需要「想」介入。

這三個問題其實指向同一件事： OpenStarry 的微核需要一套自己的「時間感」——不同層級的運作有不同的速度，不是所有東西都跑在同一個時鐘上。識蘊最快，行蘊次之，想蘊最慢。這個節奏感可能是下一步架構設計的關鍵。
