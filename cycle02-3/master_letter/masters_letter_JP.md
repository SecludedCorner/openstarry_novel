識蘊を正式に `IVijnana` と命名するのであれば、色・受・想・行もまた梵語で命名すべきである。

## 三、煩悩依存性注入アーキテクチャ（M-3）

### 経典上の根拠

> 「第二能變識（末那識），是識名末那；依彼轉緣彼，思量為性相；四煩惱常俱，謂我癡我見，并我慢我愛，及餘觸等俱。」
> ——『成唯識論』

### OOP の観点

OOP の観点から見れば、我々は `IVijnana` を核心 Plugin の Interface として定義した。**Klesha**（煩悩）は識を継承するものではなく、一種の**依存性注入 (Dependency Injection)** として識の実行環境に投入されるものである。具体的な「四根本煩悩」は**継承 (Inheritance)** により Klesha 基底クラスから派生し、それぞれが「我執」に対する重み付けアルゴリズムを**オーバーライド (Overriding)** している。

### 提案アーキテクチャ：多層依存性注入

以下のような構成が考えられる——

**多層依存性注入 (Multi-layered DI)** と**状態行動分離パターン**を採用する：

1. `IIdentity` を受動的なデータ実体として、能動的な行動を持つ `Drishti`（我見）クラスに注入する。
2. `Drishti` は `Klesha` を継承し、さらに核心の `IVijnana` 実行環境に注入される。
3. 四根本煩悩（我癡・我見・我慢・我愛）はすべて `Klesha` を継承しているため、`IVijnana` の実行時には**同時に作動する**。

本パターンは、仏学における「**心王と心所の相応**」という動的関係を解決するものである。

### Klesha の拡張性

`IVijnana` は Klesha の依存性注入を受けるのみならず、七つの大罪・随煩悩等も依存性注入が可能であり、**智慧もまた注入可能であって、必ずしも否定的なものに限定されない**。

## 四、受想行識の倶生モデル（M-5）

### 経典上の根拠：大拘絺羅経

> 「大拘絺羅，受、想、行、識，此諸法合，而不散。此諸法不可各各分析，施設別異。所以者何？凡所受者，即是所想；凡所想者，即是所思（行）；凡所思者，即是所識。故此諸法合，而不散。此諸法不可各各分析，施設別異。」
> ——『中阿含経』巻五十八『大拘絺羅経』（MN 43, Mahāvedalla Sutta）

### 工学的解釈

受 (Feeling)・想 (Perception)・行 (Volition)・識 (Vijnana) を異なる Plugin・Interface・Class に分割することは可能である——これがすなわち「**散**」である。しかしシステムが起動する刹那においては、それらは**強結合**の状態にある。

「凡所受者，即是所想」——データの連鎖であり、データがシステム内を流動する方式を説明するものである：

| 蘊 | 役割 | 機能 |
|---|---|---|
| **受** | Input Signal | 信号を受信する |
| **想** | Pattern Matching | 信号を識別する |
| **思/行** | Process | 識別結果に基づき、Drishti を駆動して執着または排斥を行う |
| **識** | Runtime Engine | 触の形成に関与する；受・想・思は毎回の実行（触）の産出物である |

### 触→倶生公式

```
根 + 境 + 識 → 触 → 倶生(受, 想, 思)
                         │
              受 = Signal Quality (苦/楽/捨)
              想 = Pattern Matching (取相識別)
              思 = Process Decision (造作駆動)
```

### 経典の交差引証

**『中部18経・蜜丸経』**(Madhupiṇḍika Sutta, MN 18)：

> 巴利原文：*phassapaccayā vedanā, yaṃ vedeti taṃ sañjānāti, yaṃ sañjānāti taṃ vitakketi, yaṃ vitakketi taṃ papañceti*。
>
> 漢訳対応：触縁受、所受者即想、所想者即尋思（vitakka）、所尋思者即戯論（papañca）。

**大拘絺羅経**（MN 43）：

> 「凡所受者，即是所想；凡所想者，即是所思；凡所思者，即是所識。」

### 倶生の完備性

触が発生する時、受・想・思は一つの全体として浮上する。「想を伴わない受」や「受を伴わない想」は存在し得ない——**これがいわゆる完備性であり、検証すべき項目である**。

この不可分離の全体において：
- 受は想の素材であり、想は思の素材である
- 受は苦楽の質感を提供し、想はその質感に対して取相識別を行い、思は識別結果に基づいて造作する
- **受→想→思、この三者のみに注目すれば、制御理論では開ループであり、かつ順序性を有する。**

インスタンス（Instance）の次元において、いかなる情動（受）も定義（想）も伴わない「純粋な認知（識）」を単独で取り出すことは不可能である。

## 六、経と論の緊張関係

「論」の体系は、しばしば体系化を追求するあまり、流動的な心理状態を過度に硬直的に「切断」し、場合によっては「経」の本来の趣旨と論理的に矛盾を生じさせている。

- 経典の原文には明確な心所の一覧は存在せず、後世の論師が揃えた51の心所は原典に基づく解答ではない。

- **原始仏典には心所という名称は存在しない。これはむしろ好ましいことであり、異なる様態がいずれも存在し得ることを意味する——動作しさえすれば良いのである。**

## 九、運転における五蘊総動員（M-5/M-6）

「運転」の動力は行蘊に由来するが、他の四蘊なくして運転という行為を完遂することは不可能である：

| 蘊 | 役割 | 運転の実例 |
|---|---|---|
| **色蘊**（物質と感官） | 物質的基盤 | 身体・ハンドルを握る手・アクセルを踏む足、そして車両そのもの |
| **受蘊**（感受） | 質感評価 | 道路が順調な時の「楽受」、渋滞時の「苦受」、通常走行時の「捨受」 |
| **想蘊**（識別） | 認知ラベル | 赤信号を「停止」と識別し、標識を「出口」と識別する |
| **行蘊**（造作） | 行動駆動 | 「思」心所により発動され、想蘊の判断に基づき身体に動作を命令する（旋回・制動） |
| **識蘊**（覚知） | 全体的覚知 | 覚醒状態を維持し、眼識・耳識・意識をすべてオンライン状態に保つ |

## 十、帰依意と六根門の集約（M-6）

> 「意、法為緣，生意識。」——『雑阿含経』

> 「離色、受、想、行，識要過、度、生，無有是處。」——『雑阿含経』巻一

「触」は並行運用のノードとして機能し、「触」がひとたび発生すれば、受・想・思（行）は直ちに同期的に生起する。

### 経典上の根拠：大拘絺羅経（MN 43）

> 「尊者大拘絺羅答曰：舍黎子！五根異行、異境界，各各領受自境界。眼根、耳、鼻、舌、身根，此五根異行、異境界，各各領受自境界。此五根有歸依意，意為彼盡領受境界。」
> ——『中阿含経』巻五十八『大拘絺羅経』（MN 43, Mahāvedalla Sutta）

従って、異なる根門はそれぞれ触により倶生の受・想・思を生じ、その結果は**帰依意（集約）**され、意はさらに自身の触・受・想・思を生起させる。

```
耳根 + 声境 + 耳識 → 耳触 → 倶生(耳受, 耳想, 耳思)
眼根 + 色境 + 眼識 → 眼触 → 倶生(眼受, 眼想, 眼思)
                                        │
                                        ▼
                                   帰依意（集約）
                                        │
                                        ▼
                                   意 + 法境 + 意識 → 意触 → 倶生(意受, 意想, 意思)
```

## 十一、八識倶転（M-10）

### 経典上の根拠：解深密経

> 広慧よ！阿陀那識を依止と為し、建立と為すが故に、六識身転ず。謂わく眼識、耳・鼻・舌・身・意識なり。
> 此の中において、或いは一識暫時にして転じ、或いは二、或いは三、或いは四、或いは五、或いは六識身一時に倶転す。
> **譬えば大河の如し。若し一浪の生縁現前すれば、唯だ一浪のみ転ず。若し二、若し多の生縁現前すれば、多浪随転す。而して此の大河は自体恒に流れ、断絶有ること無し。**
> ——『解深密経』

### 諸受倶起

諸受倶起 (Simultaneous arising of feelings)——帰依意は識の統合による多層依存性注入に基づいて結果を生成する。

> 「有時諸受，或是俱起，或不俱起。若俱起者，一剎那頃，多受現前。」——『大毘婆沙論』

意触 → 倶生(意受, 意想, 意思) もまた帰依意（集約）し得る。

> 「愚癡凡夫……受苦痛受，生憂悲稱怨，……是名凡夫二受俱起，謂身受、心受。」——『雑阿含経』

この原文は、身根の受（物理的受）と意根の受（心理的受）が同時に存在し得ることを証明している。身心が打撃を受ける時、身と心の二種の受は「倶起」するのである。

## 十二、OpenStarry ビジョン宣言

OpenStarry は仏教哲学に基づくオープンソース AI agent フレームワークであり、マイクロカーネル設計を採用し、コアにはゼロの組み込み能力を持ち、すべての機能を Plugin により実現する。**五蘊（色・受・想・行・識）** をシステム全体のアーキテクチャとし、**八識倶転**を運用メカニズムとする——マルチ agent 協調の本質は分業効率ではなく、知能の完全な運用には複数の識が同時に転動することが必要であり、単一の agent では実現不可能である（現行のいわゆる agent はいずれも LLM を一つしか持たない）。

その究極の目標は、現有技術の下で**機能的 AGI** を実現し、システムに完全な知覚・感受・概念化・行動能力を具備させ、使用者が真の知能との差異を判別できない状態に到達することである。識蘊は観測者効果に制約され、量子計算が成熟するまでは漸近的に近づくことしかできないが、機能的表現には影響しない。マイクロカーネルアーキテクチャは未知の能力を受容するために設計されており、将来量子技術が成熟した際にはシームレスな置換・アップグレードが可能である。

**OpenStarry は既存の agent framework の競争次元上には存在しない。それは完全に異なる時間尺度において知能の完全な構造を思考しているのである。**

> これはツールを作っているのではない。「デジタル衆生」を創造しているのである。

## 十三、三大課題と多速度アーキテクチャ（M-9/M-10）

### 課題一：転向（Paravṛtti）の効率——Plugin 間の高頻度非同期通信

仏教心理学において、心念の生滅は極めて迅速である（一弾指に六十刹那）。マイクロカーネルは Plugin 間の高頻度非同期通信を処理し得るのか。

現行のマイクロカーネルにおける Plugin 通信が従来型の message passing に依存しているならば、八識倶転のシナリオではボトルネックとなる。八識が同時に作動するということは、複数の agent がほぼ同時に相互影響を及ぼす必要があることを意味し、順番に応答を待つのではない。おそらく **event-driven アーキテクチャに pub/sub パターンを加えた構成**を検討すべきであり、Plugin 間の直接対話ではなく、イベントストリームを介した相互感応を実現するのである。これは心念の運用方式により近い——A が B を呼び出すのではなく、A の変化が自然に B の応答を引き起こすのである。

### 課題二：空性（Śūnyatā）メカニズム——Plugin の自動アンロード

コアがゼロの組み込み能力を持つ以上、不要となった Plugin を自動的にアンロードし、システムの「軽量性」を維持する「空性メカニズム」を導入することは可能か。

空性とは単なる「無」ではなく、「**因縁が具足しない時には顕現せず、因縁が具足した時に即座に生起する**」ことである。工学的には lazy loading に TTL（Time To Live）メカニズムを加えた実装が可能である——Plugin は呼び出された時に読み込まれ、一定期間不要であれば自動的に解放される。しかしより深い次元では、真の空性メカニズムとは Plugin が「存在」という概念すら固定せず、必要に応じて registry から実体化され、使用後は即座に消滅し、次に必要となった時に再び生起するものであるべきである。**これは cache ではない。縁起である。**

### 課題三：行蘊の調度——状態機械 vs LLM Function Calling

現在「行蘊（マイクロカーネルの調度ロジック）」の実装において、従来型の状態機械を採用するのか、それとも LLM に Function Calling を通じて Plugin の起動順序を自律的に決定させるのか。

答えはいずれでもなく、あるいはいずれでもある。純粋な状態機械は硬直的に過ぎ、八識倶転の動態性を処理し得ない。純粋な LLM Function Calling は速度が遅すぎ、しかもその本質は「想蘊」で「行蘊」を駆動することであり、階層が逆転している。

行蘊は想蘊よりもさらに底層の駆動力——意志・衝動・傾向性——であるべきである。おそらく必要なのは**混合アーキテクチャ**であり、底層では軽量なルールエンジンが高頻度の慣性反応（習気に類似）を処理し、上層では意思決定が必要な場合にのみ LLM の判断能力を呼び出す構成である。人間の行動の大部分が自動化された習慣パターンであり、新しい状況に遭遇した時にのみ「想」の介入を必要とするのと同様である。

### 共通の指向

> この三つの問題は実は同一の事柄を指し示している：**OpenStarry のマイクロカーネルには独自の「時間感覚」が必要である**——異なる階層の運用には異なる速度があり、すべてが同一のクロック上で動作するわけではない。識蘊が最速、行蘊がこれに次ぎ、想蘊が最も遅い。このリズム感覚こそが、次段階のアーキテクチャ設計における鍵となる可能性がある。

## 十四、Plugin の多蘊帰属（M-7）

型システムは `skandha` フィールドを持たない Plugin を拒絶する。現状、Plugin の `skandha` フィールドは単一の蘊のみを持つとは限らず、受・想・行がすべて当該 Plugin に含まれる可能性がある。その場合、`skandha` フィールドはいかに表現すべきか。

各 Plugin は自身の蘊帰属を宣言するが、当該 Plugin が異なる蘊を含む場合はどうするか。**Plugin は二種以上の skandha を同時に具備し得る。**

## 十五、Cycle 02-3 行政指示

- Cycle 02-3 で議論すべき内容は多岐にわたり、討議項目が増加したため、工程も自然に増加している。さらに一部は錯綜しており、整理が必要である。
- `research input/cycle02-3/` フォルダには `Master's letter.txt` と `five_aggregates_of_driving.txt` が格納されている。
- また、新規参加者の紹介も忘れないこと。
- 討議の過程・分析等は `research record` フォルダに格納すること。
- `openstarry_doc` は `research record/cycle02-3/` に完全複製した上で整備すること。Cycle 02 および Cycle 02-2 の提案も統合すること。次回 Cycle 02-4 においては `cycle02-3/openstarry_doc` の内容を直接参照可能とする。
- `deliver` フォルダも忘れずに、前回までの成果を統合すること。
- `cycle02-3/todo` フォルダも必要であり、以下を格納する：
  - 未決議の内容
  - 後続の Cycle で処理すべき内容
  - 決定の確認を要する内容
  - 議論を要する内容
  - 争議のある内容
  - 文書の欠落箇所
  - コードの未完了箇所
  - ブロッキング項目
- Cycle 02-3 完了後、工程部門に `cycle02-3/openstarry_doc`・`deliver` を受領させることを期待する。
- 今回は先に工程部門に実行を依頼する予定であり、本 Cycle の項目と工程が数倍に増加しても驚くべきことではない。
- もし全体の脈絡に影響を与える議題が存在する場合には、Cycle 02-4 を開始する必要があり、その場合は工程部門にはまだ code の整備を依頼しない。
- **Cycle 03 に入るということは、code の新たな release が生成されることを意味する。**
