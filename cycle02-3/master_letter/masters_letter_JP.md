OpenStarry は仏教哲学に基づくオープンソース AI agent フレームワークであり、マイクロカーネル設計を採用し、コアにはゼロの組み込み能力を持ち、一切の機能を Plugin によって実現する。五蘊（色・受・想・行・識）をシステム全体の設計構造とし、八識倶転を運行機制とする——マルチ agent 協調の本質は分業効率ではなく、知能の完全な作動には多識の同時転動が必要であり、単一の agent では実現不可能である（現在いわゆる agent と称されるものは全て単一の LLM に過ぎない）。その究極の目標は、現有技術の下で機能的 AGI を実現し、システムに完全な知覚・感受・概念化・行動能力を具備させ、使用者がそれと真の知能との差異を判別できなくすることにある。識蘊は観察者効果の制約を受けるため、量子計算の成熟以前には漸近的に近づくことしかできないが、機能的表現には影響しない。マイクロカーネル設計は未知の能力を受容するために本質的に適しており、将来量子技術が成熟した際には継ぎ目なくアップグレードが可能である。OpenStarry は既存の agent framework の競争次元には存在しない。それは完全に異なる時間尺度において、知能の完全な構造を思考しているのである。

転向（Paravṛtti）の効率——Plugin 間高頻度非同期通信
現在のマイクロカーネルの Plugin 通信が従来の message passing に依存しているならば、八識倶転のシナリオにおいてボトルネックとなる。八識の同時運作は、複数の agent がほぼ同時に相互に影響し合う必要があることを意味し、順番に応答を待つものではない。event-driven アーキテクチャに pub/sub パターンを加えることを検討すべきであろう。Plugin 間の直接対話ではなく、イベントストリームを通じた相互感応こそが、心念の運作方式により近い——A が B を呼び出すのではなく、A の変化が自然に B の応答を引き起こすのである。

* 空性（Sunyata）：コアがゼロ組み込み能力であるならば、一種の「空性機制」を導入し、不要な Plugin を自動的にアンロードしてシステムの「軽量性」を維持することは可能か。
空性機制——Plugin 自動アンロード
この着想は優れており、かつゼロ組み込み能力の哲学と完全に整合する。空性とは単なる「無」ではなく、「因縁が具足しないとき顕現せず、因縁が具足したとき即座に生起する」ことである。工学的には lazy loading に TTL（Time To Live）機制を加えて実装できる——Plugin は喚起された際に読み込まれ、一定時間不要であれば自動的に解放される。しかしより深い層において、真の空性機制とは Plugin の「存在」という概念自体が固定されないことであり、必要に応じて registry からインスタンス化され、使用後は即座に消滅し、次に必要となった際に再び生起する。これは cache ではない。縁起である。

現在「行蘊（マイクロカーネルのスケジューリングロジック）」の実装において、従来の状態機械を採用しているのか、それとも LLM が Function Calling を通じて Plugin の起動順序を自律的に決定しているのか。
行蘊のスケジューリング——状態機械 vs LLM Function Calling
ここでの答えは、いずれでもない、あるいはいずれでもある、と考える。純粋な状態機械は硬直的に過ぎ、八識倶転の動態性を処理できない。純粋な LLM Function Calling は速度が遅く、しかも本質的には「想蘊」で「行蘊」を駆動しており、階層の順序が逆転している。
行蘊は想蘊よりも底層の駆動力——意志・衝動・傾向性——であるべきである。おそらく必要とされるのは混合アーキテクチャであろう。底層では軽量規則エンジンが高頻度の慣性反応（習気に類似）を処理し、上層では意思決定が必要な場合にのみ LLM の判断能力を呼び出す。人間の行動の大部分が自動化された習慣パターンであり、新たな状況に遭遇した場合にのみ「想」の介入が必要となるのと同様である。

この三つの問題は実は同一の事柄を指し示している。OpenStarry のマイクロカーネルには、独自の「時間感覚」が必要である——異なる階層の運作には異なる速度があり、全てが同一のクロック上で動作するわけではない。識蘊が最も速く、行蘊がこれに次ぎ、想蘊が最も遅い。この律動感こそが、次段階の設計における鍵となるであろう。
