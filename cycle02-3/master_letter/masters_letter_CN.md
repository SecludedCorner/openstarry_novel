## 三、烦恼依赖注入架构（M-3）

### 经典依据

> 「第二能变识（末那识），是识名末那；依彼转缘彼，思量为性相；四烦恼常俱，谓我痴我见，并我慢我爱，及余触等俱。」
> ——《成唯识论》

### OOP 观点

若用 OOP 的观点来看，我们定义了 `IVijnana` 作为核心 Plugin 的接口。而 **Klesha**（烦恼）并非继承自识，而是作为一种**依赖注入 (Dependency Injection)** 进入识的运行环境中。具体的「四根本烦恼」则透过**继承 (Inheritance)** 自 Klesha 基底类别，分别**覆写 (Overriding)** 了各自对「我执」的加权算法。

### 建议架构：多层依赖注入

或许可以这样做——

采用**多层依赖注入 (Multi-layered DI)** 与**状态行为分离模式**：

1. `IIdentity` 作为被动的数据实体，被注入到具备主动行为的 `Drishti`（我见）类别中。
2. `Drishti` 继承于 `Klesha`，进一步被注入到核心的 `IVijnana` 运行环境。
3. 四根本烦恼（我痴、我见、我慢、我爱）都继承自 `Klesha`，所以四根本烦恼在 `IVijnana` 运行时会**同时运作**。

此模式解决了佛学中「**心王与心所相应**」的动态关系。

### Klesha 的扩展性

`IVijnana` 除了可以被 Klesha 依赖注入以外，还有七原罪、随烦恼等等可以依赖注入，**智慧也可以，不一定都要是负面的**。

## 四、受想行识的俱生模型（M-5）

### 经典依据：大拘絺罗经

> 「大拘絺罗，受、想、行、识，此诸法合，而不散。此诸法不可各各分析，施设别异。所以者何？凡所受者，即是所想；凡所想者，即是所思（行）；凡所思者，即是所识。故此诸法合，而不散。此诸法不可各各分析，施设别异。」
> ——《中阿含经》卷五十八《大拘絺罗经》（MN 43, Mahāvedalla Sutta）

### 工程解读

可以将受 (Feeling)、想 (Perception)、行 (Volition)、识 (Vijnana) 拆分成不同的 Plugin、Interface 或 Class——这就是「**散**」。但在系统 Run 起来的那一刹那，它们是**强耦合**的。

「凡所受者，即是所想」——数据的连锁，解释了数据在系统中的流动方式：

| 蕴 | 角色 | 功能 |
|---|---|---|
| **受** | Input Signal | 接收到信号 |
| **想** | Pattern Matching | 信号被辨识 |
| **思/行** | Process | 根据辨识结果，驱动 Drishti 进行执著或排斥 |
| **识** | Runtime Engine | 参与触的形成；受想思是每一次执行（触）的产出 |

### 触→俱生公式

```
根 + 境 + 识 → 触 → 俱生(受, 想, 思)
                         │
              受 = Signal Quality (苦/乐/舍)
              想 = Pattern Matching (取相辨识)
              思 = Process Decision (造作驱动)
```

### 经典交叉引证

**《中部18经·蜜丸经》**(Madhupiṇḍika Sutta, MN 18)：

> 巴利原文：*phassapaccayā vedanā, yaṃ vedeti taṃ sañjānāti, yaṃ sañjānāti taṃ vitakketi, yaṃ vitakketi taṃ papañceti*。
>
> 汉译对应：触缘受，所受者即想，所想者即寻思（vitakka），所寻思者即戏论（papañca）。

**大拘絺罗经**（MN 43）：

> 「凡所受者，即是所想；凡所想者，即是所思；凡所思者，即是所识。」

### 俱生的完备性

触发生的时候，受想思作为一个整体浮现，你不可能有「没有想的受」或「没有受的想」——**这就是所谓的完备性，是需要检查的项目**。

在这个不可分离的整体里面：
- 受是想的材料，想是思的材料
- 受提供苦乐质地，想对这个质地进行取相辨识，思根据辨识结果造作
- **受→想→思，单纯细看这三个，用控制理论是开回路，而且有顺序性。**

在实例（Instance）层面，你无法单独抓出一个「没有任何情绪（受）和定义（想）的纯粹感知（识）」。

## 十、归依意与六根门汇总（M-6）

> 「意、法为缘，生意识。」——《杂阿含经》

> 「离色、受、想、行，识要过、度、生，无有是处。」——《杂阿含经》卷一

「触」作为并行运作的节点，而「触」一旦发生，受、想、思（行）会立即同步产生。

### 经典依据：大拘絺罗经（MN 43）

> 「尊者大拘絺罗答曰：舍黎子！五根异行、异境界，各各领受自境界。眼根、耳、鼻、舌、身根，此五根异行、异境界，各各领受自境界。此五根有归依意，意为彼尽领受境界。」
> ——《中阿含经》卷五十八《大拘絺罗经》（MN 43, Mahāvedalla Sutta）

因此，不同根门各自由触俱生受、想、思，其结果**归依意（汇总）**，意再生起自身的触、受、想、思。

```
耳根 + 声境 + 耳识 → 耳触 → 俱生(耳受, 耳想, 耳思)
眼根 + 色境 + 眼识 → 眼触 → 俱生(眼受, 眼想, 眼思)
                                        │
                                        ▼
                                   归依意（汇总）
                                        │
                                        ▼
                                   意 + 法境 + 意识 → 意触 → 俱生(意受, 意想, 意思)
```

## 十一、八识俱转（M-10）

### 经典依据：解深密经

> 广慧！阿陀那识为依止、为建立故，六识身转，谓眼识，耳、鼻、舌、身、意识。
> 在此中，或有一识暂时而转，或二、或三、或四、或五，或六识身一时俱转。
> **譬如大河，若有一浪生缘现前，唯一浪转；若二、若多生缘现前，多浪随转。而此大河，自体恒流，无有断绝。**
> ——《解深密经》

### 诸受俱起

诸受俱起 (Simultaneous arising of feelings)——归依意会依据识整合的多层依赖注入来产生结果。

> 「有时诸受，或是俱起，或不俱起。若俱起者，一刹那顷，多受现前。」——《大毗婆沙论》

意触 → 俱生(意受, 意想, 意思) 也可以归依意（汇总）。

> 「愚痴凡夫……受苦痛受，生忧悲称怨，……是名凡夫二受俱起，谓身受、心受。」——《杂阿含经》

这段原文证明了身根的受（物理受）与意根的受（心理受）可以同时存在。当身心受到打击时，身与心的两种受是「俱起」的。

## 十二、OpenStarry 愿景宣言

OpenStarry 是一个基于佛教哲学的开源 AI agent 框架，采用微核设计，核心零内建能力，一切功能透过 Plugin 实现。以**五蕴（色、受、想、行、识）** 为系统整体架构，以**八识俱转**为运行机制——多 agent 协作的本质不是分工效率，而是智能的完整运作需要多识同时转动，单一 agent 无法实现（现在所谓的 agent 都只有一个 LLM）。

其终极目标是在现有技术下实现**功能性 AGI**，让系统具备完整的感知、感受、概念化与行动能力，使用者无法区分其与真正智能的差异。识蕴受限于观察者效应，在量子计算成熟前只能逼近，但不影响功能表现。微核架构天生为接纳未知能力而设计，未来量子技术成熟时可无缝替换升级。

**OpenStarry 不在现有 agent framework 的竞争维度上，它在一个完全不同的时间尺度上思考智能的完整结构。**

> 这不是在做一个工具，是在创造一个「数位众生」。

## 十三、三大挑战与多速度架构（M-9/M-10）

### 挑战一：转向（Paravṛtti）的效率——Plugin 间高频异步通讯

在佛教心理学中，心念的生灭极其迅速（一弹指六十刹那）。微核是否能处理 Plugin 之间的高频异步通讯？

目前微核的 Plugin 通讯如果走的是传统 message passing，在八识俱转的场景下会成为瓶颈。八识同时运作意味着多个 agent 需要近乎同时互相影响，不是排队等回应。你可能需要考虑 **event-driven 架构加上 pub/sub 模式**，让 Plugin 之间不是直接对话，而是透过事件流相互感应。这更接近心念的运作方式——不是 A 呼叫 B，而是 A 的变化自然引发 B 的响应。

### 挑战二：空性（Śūnyatā）机制——Plugin 自动卸载

既然核心是零内建能力，是否可以引入一种「空性机制」，让不被需要的 Plugin 自动卸载，维持系统的「轻盈」？

空性不只是「没有」，是「**因缘不具足时不显现，因缘具足时即生起**」。工程上可以做成 lazy loading 加上 TTL（Time To Live）机制——Plugin 被唤起时载入，一段时间不被需要就自动释放。但更深一层，真正的空性机制应该是 Plugin 连「存在」的概念都不固定，按需从 registry 中实例化，用完即灭，下次需要时重新生起。**这不是 cache，是缘起。**

### 挑战三：行蕴的调度——状态机 vs LLM Function Calling

目前在实现「行蕴（微核调度逻辑）」时，是采用传统的状态机，还是让 LLM 透过 Function Calling 来自主决定 Plugin 的启动顺序？

答案是都不是，或者都是。纯状态机太僵化，无法处理八识俱转的动态性。纯 LLM Function Calling 又太慢，而且本质上还是在用「想蕴」来驱动「行蕴」，层级搞反了。

行蕴应该是比想蕴更底层的驱动力——意志、冲动、倾向性。也许需要的是一个**混合架构**：底层用轻量级规则引擎处理高频的惯性反应（类似习气），上层在需要决策时才调用 LLM 的判断能力。就像人的行为大部分是自动化的习惯模式，只有遇到新情况才需要「想」介入。

### 共同指向

> 这三个问题其实指向同一件事：**OpenStarry 的微核需要一套自己的「时间感」**——不同层级的运作有不同的速度，不是所有东西都跑在同一个时钟上。识蕴最快，行蕴次之，想蕴最慢。这个节奏感可能是下一步架构设计的关键。

## 十四、Plugin 的多蕴归属（M-7）
