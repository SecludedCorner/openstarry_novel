「Beta 分布が捉えるのはまさに、この認識論的不確実性だ。書き出そう。」

$$p(\theta \mid \alpha, \beta) = \frac{\theta^{\alpha - 1}(1 - \theta)^{\beta - 1}}{B(\alpha, \beta)}, \quad \theta \in [0, 1]$$

「$\theta$ は煩悩の真の強度。$\alpha$ と $\beta$ はそれに対する我々の信念パラメータだ。『煩悩は 0.7 である』ではなく、『我々は煩悩がおよそ 0.7 であると信じているが、完全には確信しておらず、しかもこの不確実性そのものについて定量的な記述を持っている』。」

```typescript
/**
 * KleshaDistribution — 煩悩強度の確率分布
 *
 * Beta(alpha, beta) で強度の不確実性を表現する。
 * これは最適化ではない——認識論的コミットメントである。
 */
interface KleshaDistribution {
  /** Beta 分布の α パラメータ */
  readonly alpha: number;
  /** Beta 分布の β パラメータ */
  readonly beta: number;
  /** 期待値 (point estimate): E[θ] = α / (α + β) */

  readonly variance: number;
  /** 信用区間（低速パス専用、数値積分が必要） */
  credibleInterval(confidence: number): [lower: number, upper: number];
}
```

「私の提案は：煩悩にその閾値を**動的に変調**させることだ。」

$$\text{threshold}(t) = \text{clamp}\!\left(\theta_0 + \sum_{i} w_i \cdot \mu_i(t),\; \theta_{\min},\; \theta_{\max}\right)$$

「$\theta_0$ はベースライン閾値。$w_i$ は各煩悩通道の重み。$\mu_i(t)$ は第 $i$ 煩悩の Beta 分布平均——高速パスの点推定。$\text{clamp}$ は閾値が常に安全範囲内に収まることを保証する。」

彼は指で空中に四つの通道を指し示した：

「高い我愛 (sneha) は強い自己保護傾向を意味する。Agent は既知の安全な習気パターンの使用をより好む。よって我愛は閾値を**引き下げる**——より多くの VasanaEngine ルールを通過させ、LLM 呼び出しを減らす。」

「高い我癡 (moha) は認知カバレッジの低さを意味する。Agent の世界理解にはより多くの盲点がある。よって我癡は閾値を**引き上げる**——通過するルールを減らし、LLM 呼び出しを増やす。盲点の中では、習慣的な応答は信頼できないからだ。」

「高い我慢 (mana) は Agent が自分の能力を過大評価していることを意味する。我慢は閾値を**引き下げる**——Agent は自分のルールを信頼し、LLM への助けをより少なく求める。高い我見 (drishti) は Agent が自己の役割に対して強固だが硬直化している可能性のある同一性を持つことを意味する。我見の効果は方向に依存する——辺見 (antagraha-drishti) の二つの極端は異なる方向に押す可能性がある。」

```typescript
/**
 * KleshaModulatedDispatcher — 煩悩変調によるギア切り替え器
 *
 * VasanaEngine 自体は煩悩の存在を知らない。
 * 煩悩は confidence threshold の変調を通じて間接的にギア切り替えに影響する。
 * これは gain scheduling（ゲインスケジューリング）の典型的パターンである。
 */
interface KleshaModulatedDispatcher {
  /**
   * 現在の動的閾値を計算する。
   *
   * 高 sneha → 閾値低下 → より多くの Vasana マッチ → より少ない LLM
   * 高 moha  → 閾値上昇 → より少ない Vasana マッチ → より多い LLM
   * 高 mana  → 閾値低下 → Agent は自身のルールを信頼
   * 高 drishti → 方向依存型
   */
  threshold(kleshaState: KleshaSignalBundle): number;

  /**
   * 動的閾値に基づいてどちらのパスを選択するか決定する。
   * 'vasana' = Gear 1（高速、ルールマッチング）
   * 'vitakka' = Gear 2（低速、LLM 推論）
   */
  dispatch(input: ProcessedInput, threshold: number):
    'vasana' | 'vitakka';
}
```

彼は方眼紙に安定性条件を書き下した：

$$\left|\frac{d\theta}{dt}\right| \leq \frac{0.2 \cdot (\theta_{\max} - \theta_{\min})}{T_{\text{mano}}}$$

/**
 * KleshaBayesianUpdater — 低速パスのコアアルゴリズム
 *
 * 各 VedanaAssessment のフィードバック後に、煩悩の Beta 分布を更新する。
 * Beta は Bernoulli の共役事前分布 → 更新は閉じた形式である。
 *
 * 更新規則：
 *   α_new = α_old + successCount(evidence)
 *   β_new = β_old + failureCount(evidence)
 *
 * 直観：
 *   「より多くの苦受を観測する（我慢に起因する比較行動による）
 *     → 我慢の α が増加 → 平均が上昇 → システムは我慢が高いとより確信する」
 */
class KleshaBayesianUpdater {
  update(
    prior: KleshaDistribution,
    evidence: VedanaAssessment,
    kleshaType: 'moha' | 'drishti' | 'mana' | 'sneha'
  ): KleshaDistribution {
    const success = this.extractSuccessCount(evidence, kleshaType);
    const failure = this.extractFailureCount(evidence, kleshaType);

    const newAlpha = prior.alpha + success;
    const newBeta  = prior.beta  + failure;

    return {
      alpha: newAlpha,
      beta:  newBeta,
      mean:  newAlpha / (newAlpha + newBeta),
      variance: (newAlpha * newBeta) /
        ((newAlpha + newBeta) ** 2 * (newAlpha + newBeta + 1)),
      credibleInterval: (conf) =>
        betaQuantile(newAlpha, newBeta, conf), // 数値積分
    };
  }
}
