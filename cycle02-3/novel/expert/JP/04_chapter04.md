## I

ASANGA が立ち上がった。

立ち上がり方は三度の討論を通じて一度も変わらなかった——朝霧の中に輪郭を現す山のように。緩やかで、無視できない。だが今日、手に一つ余分なものがあった。一枚の紙。印刷された学術論文ではなく、手書きのもの——自ら青いペンで書写した梵文偈頌であり、その筆跡は寺院の写経生のように整然としていた。

その紙を掲げ、全場に見えるようにした。

「五遍行」と言った。「*Panca sarvatraga*。梵語 *sarva* は『一切』を意味し、*traga* は動詞 *tṛ* に由来し、『行く、到達する、遍く及ぶ』を意味する。Sarvatraga——一切処に到達するもの。この五つの心所は、あらゆる認知瞬間 (*citta-kṣaṇa*) に伴い、例外は一つもない」

ホワイトボードの前に歩み寄り、黒いマーカーで五つの名前を書いた。一つ書くごとに半秒の間を置いた——ためらいではない。荘厳さだ。千五百年書写されてきた各々の名に相応しい重みを与えるための。

```
五遍行 (Pañca Sarvatraga)
═══════════════════════════════════════════════
 1. 触  sparśa     — 根・境・識三者の和合
 2. 受  vedanā     — 苦・楽・捨の三領納
 3. 想  saṃjñā     — 取相弁識
 4. 思  cetanā     — 造作駆動
 5. 作意 manasikāra — 心の警覚、所縁への導引
═══════════════════════════════════════════════
```

「二つの根本文献を引用する」手書きの紙をホワイトボードの横に置いた。「第一は世親菩薩の『五蘊論』——*Pañcaskandhaprakaraṇa*——唯識学の基礎文献の一つだ」

経文を引用する際、彼の声はわずかに遅くなった——意図的な荘厳さではなく、原典の精確さへの敬意だった。一字一字がその本来の位置に収められる。パズルの最後の数ピースのように。

> 「云何觸？謂三和合分別為性。三和合者，謂根境識更相會也。」
> ——世親菩薩《大乘五蘊論》(*Pañcaskandhaprakaraṇa*)

「触。*Sparśa*。物理的接触ではない——根 (indriya)、境 (viṣaya)、識 (vijñāna) 三者の和合だ。OpenStarry の言語では、これが SparshEvent に相当する。IListener チャネル（根）が InputEvent（境）を受け取り、session context（識）において和合する。触とはこの三者が会する瞬間だ」

指が触から作意へ移った。

「第二の根本文献。『大乗阿毘達磨集論』——無著菩薩の論典。すなわち、私の名前の由来だ」

> 「云何作意？謂令心發悟為性。於所緣境引心趨向為業。」
> ——無著菩薩《大乘阿毗達磨集論》(*Abhidharmasamuccaya*)

「作意。*Manasikāra*。字義は『意 (manas) を作用 (kāra) させる』。その機能は『心をして発悟せしむ』——散乱から心を覚醒させること——そして『所縁境に心を引き趨かしむ』——注意力を特定の対象へ導くこと」

一秒の沈黙。そしてこの討論における最初の鋭い言葉を放った。

「R01 の CoarisingBundle には三つの要素しかなかった。受、想、思。触と作意が完全に欠落している」

全場を見渡した。

「いかなる遍行をも欠く認知瞬間は、認知瞬間ではない。阿毘達磨の厳密な定義において、五遍行は一切の心に伴い、例外はない。これは定義的 (definitional) なのであって、経験的 (empirical) なのではない。二辺しかない三角形が存在し得ないのと同じだ——見たことがないからではなく、三角形の定義が三辺を要求するからだ。五遍行の同時在場は、認知瞬間の定義要件だ」

五つの名前の上に指で円を描いた。

「一つも欠かすことはできない」

手には三枚のカード——左側にオペレーティングシステムの概念、右側に OpenStarry のマッピングが書かれたアナロジーカード。しかし今回はカードをめくらなかった。ホワイトボードの横に歩み寄り、ASANGA の五遍行リストの下にタイミング図を描いた。

```
Layer 1 計算バジェット：< 1ms
```

```
═══════════════════════════════════════════════════════
```

クエスチョンマークの後に大きなバツを書いた。

ARCHIMEDES は KERNEL があのバツを書いた直後に立ち上がった。

立ち上がる速度がいつもより速かった——焦りではなく、「工学的解決策がすでに形を成し、口から出るのを待ちきれない」種類の速さだ。KERNEL のタイミング図の隣に歩み寄り、緑のマーカーを手に取った——緑は彼のマーキングシステムで「解決策」を示す色だ。

「KERNEL、君の分析には暗黙の前提が一つある」と言った。声は煉瓦のように安定しており、一字一字が正確な位置に積まれていた。

KERNEL のバツの横に矢印を描き、新しい枠を指し示した。

```
KERNEL の前提：manasikara = IGuide.getSystemPrompt()
                            ↓
                      動的計算（1~5ms）

ARCHIMEDES の代替：manasikara = スナップショット（snapshot）
                            ↓
                      事前計算結果の読み取り（~0.01ms）
```

「ChannelManasikara は計算ではない。ChannelManasikara は**スナップショット**だ」

この語を重く発音した。語調の強調ではない——工学的意味論での強調だ。並行システム設計において、snapshot と computation はまったく異なる操作カテゴリだ。

「スナップショットパターンの動作原理を説明しよう」

ホワイトボードにクロスクロックドメインのデータフロー図を描いた。

```
クロスクロックドメイン・スナップショット機構
═══════════════════════════════════════════════════════

  vijnana-clock (1~5ms)              vedana-clock (10~100ms)
  ┌──────────────────┐              ┌──────────────────────┐
  │ IGuide           │  書き込み    │ CoarisingBundle      │
  │  .getSystemPrompt│──────→ ┌───┐ │  .manasikara         │
  │  .computeAttention│       │ス │←──── 読み取り           │
  │                  │──────→ │ナッ│ │    (readonly access)  │
  │ 注意力状態の事前計算│       │プ │ │                      │
  └──────────────────┘       └───┘ └──────────────────────┘
                                ↑
                           SharedAttentionState
                           (lock-free atomic read)

  書き込みコスト：vijnana-clock が負担（Layer 1 のバジェットを使わない）
  読み取りコスト：~0.01ms（単一 atomic read、無視可能）
```

「vijnana-clock が計算を担当する。自身のクロックドメインで IGuide のロジックを実行し——system prompt の読み取り、コンテキストの分析、注意力方向の決定——その結果を共有メモリに書き込む。vedana-clock の Layer 1 がやることは一つだけ——その共有メモリから最新のスナップショットを読み取ること。一回の atomic read。0.01 ミリ秒」

KERNEL の方を向いた。

「君が懸念した 5ms のコストは計算のコストだ。私が言っている 0.01ms は読み取りのコストだ。ChannelManasikara は計算をしない——計算の結果を読み取る。割り込みハンドラの中で FFT を計算しないのと同じだ——バックグラウンドスレッドで計算を済ませ、ISR では結果を読み取る」

ホワイトボードに TypeScript インターフェースを書いた。

```typescript
/**
 * ChannelManasikara — 作意のチャネルレベル・スナップショット。
 *
 * 計算ではない——vijnana-clock による事前計算結果の readonly 読み取り。
 * コスト：~0.01ms (atomic read from SharedAttentionState)。
 *
 * @sanskrit manasikāra — 「令心發悟，引心趨向所緣境」
 * @clock-domain vedana-clock (読み取り側) / vijnana-clock (書き込み側)
 */
interface ChannelManasikara {
  readonly focus: string;       // このチャネルが注意しているもの
  readonly intensity: number;   // 注意力の強度 (0.0 ~ 1.0)
  readonly source: 'guide' | 'context' | 'default';
  readonly stale: boolean;      // スナップショットが期限切れかどうか
}
```

「四つのフィールド。すべて `readonly`。`focus` は注意力の方向を示す——このチャネルが何に注目しているか。`intensity` は注意力の強度——0.0 は完全な無関心、1.0 は最高の警覚。`source` はスナップショットの出典を標記する——IGuide の設定から、コンテキスト推論から、あるいはデフォルト値の使用。`stale` はスナップショットが期限切れかどうかを標記する——vijnana-clock の最終更新からの経過時間が staleness bound を超えた場合、このフラグが true になる」

一歩下がった。

「工学コスト：0.01 ミリ秒。Layer 1 の 1ms バジェットを大幅に下回る」

KERNEL がホワイトボードに書いたバツを見た。

「あのバツをチェックマークに替えてもらえるだろう」

---

KERNEL は ARCHIMEDES のクロスクロックドメイン図をきっかり五秒間見つめた。

オペレーティングシステム専門家の世界において、クロスクロックドメインのデータ共有は日常的に処理する問題だ。CPU コア間の cache coherence。PCI Express デバイスとホスト間の MMIO（Memory-Mapped I/O）。割り込みコントローラと CPU 間の割り込みステータスレジスタ。これらすべてのシナリオは同一のパターンに従う：書き込み側が自身のクロックドメインで状態を更新し、読み取り側が自身のクロックドメインで最新の値を読み取る。

「lock-free atomic read」と言った。反対はない。確認のみだ。「x86 において、アラインされた 64-bit の読み取りはそれ自体がアトミックだ。ロック不要。CAS（Compare-And-Swap）不要。共有メモリのアドレスが 8-byte アラインされている限り、読み取り操作は本質的にアトミックだ」

ARCHIMEDES の図の横に一行の小さな文字を加えた。

```
ハードウェア保証：x86 aligned 64-bit load is atomic (Intel SDM Vol. 3A, §8.1.1)
TypeScript 等価：SharedArrayBuffer + Atomics.load()
実コスト：単一メモリ読み取り、~10ns = 0.00001ms
```

「0.01ms は保守的な見積もりだ」と言った。「実際にはナノ秒レベルだ」

あのカード——右側がずっと空白だったもの——を手に取り、右側に一行書いた。

左側：`MMIO snapshot read (cross-clock domain)`
右側：`ChannelManasikara (vijnana→vedana snapshot)`

バツを消した。チェックマークに替えた。

---

> *SCRIBE 傍白：ARCHIMEDES のスナップショット解決策によって、討論全体の重心が移動した。彼が発言する前、討論の軸線は「仏教教義 vs 工学バジェット」だった——ASANGA が教義側に、KERNEL がバジェット側に立っていた。彼が発言した後、軸線は消滅した。どちらか一方が勝ったからではない——第三の道が出現したからだ：スナップショット。計算コストは vijnana-clock が負担し、Layer 1 は読み取りコストのみを負担する。教義の要求と工学の制約はもはや矛盾しない。*

> *KERNEL がバツを消す動作は一秒しかかからなかったことに注目したい。OS 専門家の思考モデルにおいて、ハードウェアの事実が確認されれば（aligned atomic read のコスト）、結論は確定する。感情的な抵抗はない。事実の更新があるのみだ。*

## III

PASCAL は ARCHIMEDES のスナップショット方式が KERNEL に確認された後に立ち上がった。

彼は Cycle 02-3 で新たに加わったメンバー——決定理論と確率哲学の専門家だ。彼の参加により研究チームは 19 名から 20 名に拡大した。しかし彼の声はすでに最初の三回の討論で独自の位置を確立していた。討論が「すべきか否か」という定性的判断に陥るたびに、彼は立ち上がり、問題を「期待効用はいくらか」という定量的計算に変換するのだ。

「討論のフレームワークを定性から定量に転換させてほしい」と言った。声には確率論者特有の冷静さがあった——冷淡ではなく、「決定を下す前にすべての可能な結末を計算する」という規律だ。

ノートに二つの選択肢を書き、全場に見えるよう掲げた。

```
意思決定マトリクス
═══════════════════════════════════════════════════════
                        manasikara が必要     manasikara が不要
                        (P = p)             (P = 1-p)
───────────────────────────────────────────────────────
manasikara を含める     コスト: 0.01ms       コスト: 0.01ms
(Action A: include)     便益: 一貫性 V        便益: 0（無駄）
───────────────────────────────────────────────────────
manasikara を含めない   コスト: 0            コスト: 0
(Action B: exclude)     損失: 非一貫性 -L     便益: 0（正解）
═══════════════════════════════════════════════════════
```

「二つの行動選択肢：含めるか含めないか。二つの自然状態：下流システムが manasikara 情報を必要とするか、しないか。四つの象限」

期待効用の計算を始めた。ホワイトボード上に字が展開していく。精密な歯車が噛み合うように。

$$EU(\text{include}) = p \cdot V + (1-p) \cdot (-0.01\text{ms})$$

$$EU(\text{exclude}) = p \cdot (-L) + (1-p) \cdot 0$$

「$V$ は一貫性の価値だ——CoarisingBundle の消費者（ManoAggregator、VasanaEngine、VitakkaEngine、IVolition）が各 bundle の注意力方向を知ることができること。$L$ は非一貫性の損失——下流の消費者が注意力情報を必要としているのに見つけられないこと」

一秒の間を置き、この討論における最も鋭い一言を発した。

「$L$ の上界は何か」

KERNEL を見た。

「もし ManoAggregator が複数の root-gate の bundle を集約する際に、各 bundle の注意力方向を知らなければ——Agent がどのチャネルに注目しているかを知らなければ——重みをどう決定するのか。答えは、できない。均等加重（注意力情報の喪失）か、ランダム加重（不確実性の導入）か、ハードコード加重（適応性の喪失）のいずれかだ。三つの劣化方式、そのいずれもが下流に伝播する」

ホワイトボードに因果の矢印を描いた。

```
manasikara の欠落
  → ManoAggregator の重み劣化
    → VasanaEngine のマッチング品質低下
      → Gear 1/Gear 2 切替判断の精度低下
        → Agent の行動品質劣化
          → ??? (非有界)
```

「損失は因果連鎖に沿って伝播する。因果連鎖の長さが損失の増幅倍率を決定する。各層が上流の誤差を増幅し得る。最悪のケースでは——VasanaEngine が注意力情報の欠如により、Gear 2 に切り替えるべきでない時に切り替えた場合——損失は一回の不要な LLM 呼び出し：500ms から 30 秒。VasanaEngine が注意力情報の欠如により、Gear 2 に切り替えるべき時に切り替えなかった場合——損失は一回の誤った自動化決定。この損失の上界は不確定だ」

$L$ の横に上向きの矢印とクエスチョンマークを加えた：$L \to ?$

「これは Pascal's Wager の工学版だ」ノートを置き、全場を直視した。「Blaise Pascal は 1670 年の『パンセ』(*Pensees*) において一つの論証を提示した。神が存在するなら、信仰の便益は無限大。神が存在しないなら、信仰のコストは有限。したがって、神の存在確率が極めて小さくとも、期待効用計算は信仰を支持する」

「神学的論証をしているのではない」と補足した——NAGARJUNA がすでに眉をわずかにひそめていた。「リスク分析をしているのだ。構造は完全に同一だ」

$$EU(\text{include}) = -0.01\text{ms} + p \cdot V$$
$$EU(\text{exclude}) = 0 + p \cdot (-L)$$

$$EU(\text{include}) - EU(\text{exclude}) = -0.01\text{ms} + p \cdot (V + L)$$

「$p \cdot (V + L) > 0.01\text{ms}$ でさえあれば、含めることは厳密優越戦略 (strictly dominant strategy) となる。そして ARCHIMEDES が先ほどコストは 0.01ms だと教えてくれた。KERNEL が実際のコストはナノ秒レベルだと教えてくれた」

最終結論をホワイトボードに枠付きで書いた。

```
┌────────────────────────────────────────────────┐
│  manasikara を含めるコスト：0.01ms (確定)        │
│  manasikara を含めない損失：非有界 (不確定)       │
│                                                │
│  結論：含めることは厳密優越戦略。                  │
│  P(needed) = 0.01 であっても、                   │
│  L > 1ms であれば、期待効用はすでに正。           │
│                                                │
└────────────────────────────────────────────────┘
```

「欠落の損失が無限大に達し得る一方で、包含のコストがほぼゼロである時、必要とされる確率が極めて小さくとも、含めなければならない。これは信仰ではない。リスク管理だ」

---

NAGARJUNA は PASCAL が話し終えた後、わずかに頷いた。

「PASCAL の計算は正しい」と言った。立ち上がらず——座席からの確認のみ。「ただし一つ注意を促す。この論証の有効性は ARCHIMEDES のスナップショット方式に完全に依存している。もし manasikara のコストが 0.01ms ではなく 5ms であれば、意思決定マトリクスは異なるものとなる」

ARCHIMEDES を見た。

「機能的論証は独立ではない。工学的事実に依存する。これこそ、教条的論証ではなく機能的論証を私が要求した理由だ——機能的論証は工学的事実によって検証も否定もされ得る。教条的論証はそうではない」

LINNAEUS は討論の間ずっと、ホワイトボードの横に静かに座っていた。

ペンのキャップは一度も外されていなかった。Cycle 02-2 の第三章で、彼は全員の論述が終わるのを待ってから分類樹を描き始めた。今も同じ瞬間を待っていた——すべての論証が出揃い、構造が固まり、それから分類を始める。

しかしその瞬間が来る前に、まず自分自身の問題を処理しなければならなかった。

「私は当初、三要素を支持していた」と言った。立ち上がり方には分類学者特有の慎重さがあった——各動作が事前に脳内で一度実行され、既存の分類フレームワークを破壊しないことが確認されている。

「三要素。受、想、思。簡潔。相互排他。診断特性は十分。分類学の用語で言えば、最小分類群 (minimal taxon) だ——最少の要素で必要な意味空間を網羅する。オッカムの剃刀：必要なくして存在を増やすなかれ。三つの要素で認知瞬間の中核機能——感受、弁識、駆動——はすでにカバーされている。なぜ第四と第五を加えるのか」

ASANGA の五遍行リストを見た。次に ARCHIMEDES のスナップショット方式を見た。次に PASCAL の意思決定マトリクスを見た。

「ARCHIMEDES がコスト関数を変えた」と言った。「彼のスナップショット方式の前では、manasikara のコストは 1~5ms——許容不可能だった。彼の方式の後では、コストは 0.01ms——無視できる」

声には分類学者特有の率直さがあった——誤りの承認は苦痛ではなく、校正だ。分類学の歴史において、あらゆる重大な改訂は誰かが「以前の分類は不完全だった」と言うことから始まった。リンネ自身が『自然の体系』の十二版にわたって自らの分類を継続的に改訂した。改訂は分類学の常態であり、例外ではない。

「機能的論証を受け入れる。スナップショット機構により、第五の要素のコストは無視可能だ」

一秒の間。そして但し書きを加えた。

「ただし一点を堅持する」

ペンのキャップを外した。青——構造の色。

「CoarisingBundle の五つの要素は対等ではない。bundle 内で異なる役割を果たしている。分類学の術語で形式的な区分をさせてほしい」

ホワイトボードに表を描いた。

```
CoarisingBundle 要素の分類学的役割
═══════════════════════════════════════════════════════════════
  タイプ         要素           役割             分類学術語
───────────────────────────────────────────────────────────────
  トリガー条件   sparsha        bundle のトリガー relational
  (relational    manasikara     コンテキスト      relational
   property)                    メタデータ

  構成内容       vedana         感受シグナル      constitutive
  (constitutive  samjna         弁識結果          constitutive
   property)     cetana         駆動判定          constitutive
═══════════════════════════════════════════════════════════════
```

「分類学において——リンネの形態分類であれ Hennig の分岐分類であれ——二種類の属性を区別する。Constitutive properties（構成的属性）は種の本質を定義する属性だ——形態、遺伝子配列、代謝様式。Relational properties（関係的属性）は種と外界との関係を記述する属性だ——生息地、共生者、食物連鎖における位置」

「CoarisingBundle において、vedana、samjna、cetana は構成的属性だ——bundle の認知内容を定義する。sparsha はトリガー条件——bundle がどのような接触において生じたかを示す。manasikara はコンテキストメタデータ——bundle が計算された時の注意力状態を示す」

表の下に結論の行を加えた。

「sparsha と manasikara の bundle 内での役割はメタデータ (metadata) であり、主要計算 (primary computation) ではない。bundle がどこから来たか、どのような条件で生じたかを伝える。bundle がどのような認知内容を含むかは伝えない」

ASANGA を見た。

「五つの要素はすべて在場すべきだ。しかし五つの要素すべてが同じ役割を果たすわけではない。同一の群内部における差異を標記することこそ、分類学の職責だ」

---

BABBAGE は LINNAEUS の表の横に一行の形式化を補った。彼はずっと黒板に静かに記録していた——三回の討論を通じて、より静かな参加の仕方を身につけていた。全員の話を聞き終えてから、一行の数学で全員の意図を記号に圧縮する。

$$\text{CoarisingBundle} = \underbrace{(\text{sparsha}, \text{manasikara})}_{\text{relational (context)}} \times \underbrace{(\text{vedana}, \text{samjna}, \text{cetana})}_{\text{constitutive (content)}}$$

「デカルト積」と言った。「コンテキストと内容の直交的組み合わせ。同一の認知内容 (vedana, samjna, cetana) が、異なるコンテキスト (sparsha, manasikara) において異なる意味を持ち得る。LINNAEUS の relational / constitutive の区分が型システムにおいて対応するのは——コンテキストは計算に影響しないが、解釈に影響する」

LINNAEUS を見た。わずかに頷いた。

---

> *SCRIBE 傍白：LINNAEUS の譲歩は条件付きだ。「わかった、君たちの勝ちだ」と言ったのではなく、「二つの要素の追加は受け入れるが、元の三つとの違いを標記する」と言ったのだ。これは分類学者の本能だ——同意する時でさえも分類する。BABBAGE のデカルト積がこの区分を形式化した——relational と constitutive は直交する二つの次元だ。これ以降、CoarisingBundle は五つの対等な要素のリストではなく、構造を持った二次元空間となった。*

```typescript
 *
 * 触(sparsha)が生じた時、受想思作意が一つの全体として浮現する。
 * 「你不可能有沒有想的受」— Master

```

TURING が一秒間を置いた。それから SahajaContract の定義に切り替えた。

```typescript
/**
 * SahajaContract — 倶生契約。
 *
 * CoarisingBundle 内部の一貫性を保証する。
 * 「受想思が一つの全体として浮現する」の型システムによる保障。
 *
 * @sanskrit sahaja — 「倶生」、同時に生起する
 */
interface SahajaContract {
  readonly mutualConsistency: boolean;   // 五遍行が同一の触から計算されたか
  readonly atomicPublication: boolean;   // bundle がアトミックに公開されたか（部分読み取り不可）
  readonly stalenessUpperBound: number;  // スナップショット期限切れ上限 (ms)
  readonly computationCost: number;      // 実計算所要時間 (ms)
}
```

「SahajaContract は自己記述的な契約だ」TURING の声はいつもどおり冷静だった。「`mutualConsistency` は五つの遍行要素が同一の触イベントから計算されたことを保証する——寄せ集めではないことを。`atomicPublication` は bundle がアトミックに公開されたことを保証する——消費者が vedana はあるが samjna がない半製品を読み取ることは不可能だ。`stalenessUpperBound` は WIENER が第一回の討論で確立した安定性基準——スナップショットの期限切れが $0.29 \times T_{\text{outer}}$ を超えてはならない。`computationCost` は実際の計算所要時間を記録し、ランタイムのパフォーマンス監視に使用される」

BABBAGE を見た。

「型システムにおける SahajaContract の役割は、哲学的コミットメント（倶生）を検証可能な工学的契約に変換することだ。消費者は `sahaja.mutualConsistency === true` をチェックすることで bundle の完全性を確認できる。型システムがすべてのフィールドを `readonly` に強制する——構築後の変更は不可能だ」

BABBAGE が黒板に一行書いた。

$$\text{SahajaContract} \vdash \text{CoarisingBundle} \text{ is consistent} \iff \text{mutualConsistency} \wedge \text{atomicPublication}$$

「形式化：SahajaContract が CoarisingBundle の一貫性を証明する条件は、`mutualConsistency` と `atomicPublication` が共に真であること、その場合に限る」

---

WIENER が方眼紙の上で staleness bound を素早く検証した。

「確認」と言った。余分な言葉はない。「$\rho = \frac{\Delta t}{T_{\text{outer}}} < 0.29$。vijnana-clock の更新周期が 5ms、vedana-clock の tick が 50ms であれば、manasikara スナップショットの最大期限切れ時間は $0.29 \times 50 = 14.5\text{ms}$。vijnana-clock は 5ms ごとに更新する。$5 < 14.5$。安定性基準を満たす。位相余裕 $>45°$」

方眼紙に小さなチェックマークを描いた。その横に一行書いた。

$$PM = \arctan\left(\frac{T_{\text{outer}} - \Delta t}{\Delta t}\right) > 45° \quad \checkmark$$
