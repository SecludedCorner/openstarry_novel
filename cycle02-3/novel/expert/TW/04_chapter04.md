# 第四章：五遍行

---

ASANGA 在辯論開始之前做了一件不尋常的事。

他閉上了眼睛。

不是冥想——他不會在圓形劇場裡正式冥想，那是屬於個人修行的時間，不是屬於學術辯論的時間。他閉上眼睛的方式更像一個人在打開一個記憶深處的抽屜：先讓外部的光線退場，然後讓內部的文字浮現。

他要引用的經文已經在他腦中停留了一千五百年。不是他本人的一千五百年——而是那些文字被傳誦、被抄寫、被翻譯、被辯論的一千五百年。每一次引用，都是在接住一條從無著菩薩手中拋出的線。

他睜開眼睛。

「我們在第一場辯論中確立了雙層雙齒輪架構。CoarisingBundle 在 vedana-clock 的速度上原子地計算。」他的聲音穩定，像河床上的大石。「但那場辯論留了一個問題沒有回答。」

他看向 SUNYATA。

「Bundle 裡到底放什麼。」

---

> *SCRIBE 旁白：Debate 2 的核心衝突可以用一個數字概括：3 vs 5。R01 的研究報告定義了三個 CoarisingBundle 組件——受 (vedana)、想 (samjna)、思 (cetana)。R03 提出五個——加上觸 (sparsha) 和作意 (manasikara)，完全依照瑜伽行派的五遍行 (panca sarvatraga) 教義。差兩個。差兩個元素聽起來不多。但在亞毫秒的計算 bundle 中，每一個元素都有成本。問題不是「經典怎麼說」——問題是「工程能不能負擔得起經典說的」。*

---

SUNYATA 站在劇場中央。

「Debate 2。CoarisingBundle 的組成。三遍行，還是五遍行。」

他的目光掃過圓形劇場。左側，ASANGA 和 NAGARJUNA 並排而坐，中間隔著一個座位的距離——足夠讓兩種佛學傳統各自呼吸。右側，KERNEL 的手指已經在類比卡片上翻動了。後排，PASCAL 的筆記本攤開在膝蓋上，一張貝葉斯計算表的草稿佔據了整頁。

「ASANGA。教義論證先行。」

---

## I

ASANGA 站了起來。

他站起來的方式在三場辯論中都不曾改變——像一座山在晨霧中顯現輪廓。緩慢的，不可忽視的。但今天他手裡多了一樣東西：一頁紙。不是打印的學術論文，而是手抄的——他自己用藍色鋼筆抄寫的梵文偈頌，字跡整齊得像寺院裡的寫經生。

他把那頁紙舉起來，讓全場都能看到。

「五遍行。」他說。「*Panca sarvatraga*。梵文 *sarva* 意為『一切』，*traga* 源自動詞 *tṛ*，意為『行、到達、遍及』。Sarvatraga——到達一切處的。這五個心所伴隨每一個認知瞬間 (*citta-kṣaṇa*)，無一例外。」

他走到白板前，用黑色馬克筆寫下五個名字。每寫一個，他都停頓半秒——不是遲疑，是莊重。是在給每一個被書寫了一千五百年的名字應有的重量。

```
五遍行 (Pañca Sarvatraga)
═══════════════════════════════════════════════
 1. 觸  sparśa     — 根・境・識三者和合
 2. 受  vedanā     — 苦・樂・捨三領納
 3. 想  saṃjñā     — 取相辨識
 4. 思  cetanā     — 造作驅動
 5. 作意 manasikāra — 心之警覺，引導趨向所緣
═══════════════════════════════════════════════
```

「我要引用兩個根本文獻。」他把那頁手抄紙放在白板旁邊。「第一個是世親菩薩的《五蘊論》——*Pañcaskandhaprakaraṇa*——唯識學的基礎文獻之一。」

他的聲音在引用經文時微微放慢了——不是刻意的莊嚴，而是一種對原典精確度的尊重。每一個字都被安放在它應有的位置上，像拼圖的最後幾塊。

> 「云何觸？謂三和合分別為性。三和合者，謂根境識更相會也。」
> ——世親菩薩《大乘五蘊論》(*Pañcaskandhaprakaraṇa*)

「觸。*Sparśa*。不是物理接觸——是根 (indriya)、境 (viṣaya)、識 (vijñāna) 三者的和合。在 OpenStarry 的語言中，這就是 SparshEvent：IListener 通道（根）接收 InputEvent（境），在 session context（識）中和合，觸是這三者匯聚的瞬間。」

他的手指從觸移到作意。

「第二個根本文獻。《大乘阿毗達磨集論》——無著菩薩的論典。也就是我名字的來源。」

> 「云何作意？謂令心發悟為性。於所緣境引心趨向為業。」
> ——無著菩薩《大乘阿毗達磨集論》(*Abhidharmasamuccaya*)

「作意。*Manasikāra*。字面意思是『使意 (manas) 作用 (kāra)』。它的功能是『令心發悟』——讓心從散亂中警覺過來——然後『引心趨向所緣境』——引導注意力指向特定的對象。」

他停頓了一秒。然後他說了這場辯論中第一句帶鋒芒的話。

「R01 的 CoarisingBundle 只有三個元素：受、想、思。完全遺漏了觸和作意。」

他看向全場。

「缺乏任何一個遍行的認知瞬間，不是認知瞬間。在阿毗達磨的嚴格定義中，五遍行伴隨一切心，無一例外。這是定義性的 (definitional)，不是經驗性的 (empirical)。就像你不能有一個只有兩邊的三角形——不是因為你沒見過，而是因為三角形的定義要求三邊。五遍行的同時在場是認知瞬間的定義要求。」

他用手指在五個名字上畫了一個圈。

「缺一不可。」

---

NAGARJUNA 在 ASANGA 畫完那個圈的瞬間就開口了。

他沒有站起來。中觀哲學家在發出警告的時候從不站起來——警告的力量不在姿態上，在銳度上。他的聲音從座位上升起，鋒利但不尖銳，像被千年河水打磨過的石頭。

「ASANGA，你在將心所分類法物化 (reify)。」

劇場的空氣收緊了。

「你說『缺一不可』。你引用了《五蘊論》和《阿毗達磨集論》。這些是偉大的論典。但論典的功能是什麼？論典的功能是教學 (*deśanā*)——幫助修行者理解心的運作方式。論典不是存在論清單 (*ontological inventory*)。」

他停了一拍。然後用更精確的語言展開。

「五遍行是**教學工具** (*upāya*)。不是本體論宣言。它們是對心所運作方式的一種系統化描述——一種特別精確、特別有用的描述——但描述不等於事物本身。地圖不等於地形。」

> 「若人見般若，是則為被縛；若不見般若，是亦為被縛。」
> ——龍樹菩薩《中論》觀涅槃品第二十五

「如果我們因為論典說了五個就放五個，我們是在做佛學考古，不是做工程。」

他看向 ASANGA。不是對抗——是補正。像兩面鏡子互相照亮對方的盲區。

「我的立場不是『不放五個』。我的立場是：如果要放五個，理由必須是功能性的 (functional)，不是教條性的 (doctrinal)。功能性論證比教條性論證更可靠。因為功能性論證可以被工程驗證。教條性論證只能被權威認可。」

---

> *SCRIBE 旁白：NAGARJUNA 和 ASANGA 的分歧在 Cycle 02-2 中就出現過。在第三章「部分與整體」中，NAGARJUNA 對 IVijnana 的命名選擇做了四句否定——不是因為他否認識蘊的複雜性，而是因為他堅持名字必須是「最不會導致誤解的假名」。現在，在 Debate 2 中，同樣的方法論分歧再次浮現：ASANGA 從教義出發，NAGARJUNA 要求功能性理由。*

> *但注意——NAGARJUNA 沒有說「不放五個」。他說的是「理由必須是功能性的」。這是一個方法論的挑戰，不是結論的否定。他在控制論證的品質，不是論證的方向。*

## II

KERNEL 在 NAGARJUNA 說完之後站了起來。

他的手裡拿著三張卡片——那些左邊寫著作業系統概念、右邊寫著 OpenStarry 映射的類比卡片。但這一次，他沒有翻卡片。他走到白板旁邊，在 ASANGA 的五遍行列表下方畫了一張時序圖。

「讓我問一個工程問題。」他說。聲音低沉穩定，像在 kernel space 裡持續運行的守護進程。「我不是在質疑佛學。我是在質疑時間。」

他在白板上寫了兩行數字：

```
Layer 1 計算預算：< 1ms
五遍行中各元素的估計成本：
  sparsha  (觸)   — SparshEvent 構造       ~0.01ms  ✓
  vedana   (受)   — ChannelVedana 計算     ~0.1ms   ✓
  samjna   (想)   — ChannelSamjna 匹配     ~0.05ms  ✓
  cetana   (思)   — ChannelCetana 判斷     ~0.05ms  ✓
  manasikara (作意) — ???                   ???      ?
```

「前四個沒有問題。」他用手指點了點前四行的打勾。「SparshEvent 是一個 struct 的構造——三個 readonly 欄位，幾乎零成本。ChannelVedana 是一個數值映射——valence 和 intensity 從感測器讀數直接計算。ChannelSamjna 是規則匹配——在 VasanaEngine 的 fast path 中，這是一個 hash table lookup。ChannelCetana 是趨避判斷——從 valence 的正負號直接導出。」

他的手指停在第五行。

「作意。*Manasikāra*。注意力的方向。」

他轉向 ASANGA。

「ASANGA，你說作意的功能是『引心趨向所緣境』。在 OpenStarry 的架構中，注意力的方向由誰決定？」

ASANGA 回答：「IGuide。system prompt 定義了 Agent 應該關注什麼。」

KERNEL 點頭。「IGuide。在 Cycle 02-2 中，IGuide 被歸入識蘊——IVijnana 的子介面。它的主要方法是 `getSystemPrompt()`。而 `getSystemPrompt()` 的調用——」

他在白板上畫了一條時間線：

```
時鐘域分析：IGuide.getSystemPrompt()
═══════════════════════════════════════════════════════
  IGuide 屬於 vijnana-clock (1~5ms)
  getSystemPrompt() 涉及：
    - 讀取 system prompt 配置          ~0.1ms
    - 上下文組裝 (context assembly)    ~0.5ms
    - 可能的動態 prompt 生成           ~1-5ms
  ─────────────────────────────────
  總計：1~5ms，屬於 vijnana-clock 操作

  Layer 1 屬於 vedana-clock (10~100ms per tick, <1ms 計算)
```

「Layer 1 必須在 1ms 以內完成。」KERNEL 的聲音加重了——不是情緒化的加重，是工程約束的加重。「如果 manasikara 需要調用 `IGuide.getSystemPrompt()`，那是一個 vijnana-clock 的操作——1 到 5 毫秒。你不能把一個 5ms 的操作放在一個亞毫秒的 bundle 裡。」

他在問號後面寫了一個大叉。

「如果 manasikara = IGuide.getSystemPrompt()，成本是 1~5ms。超出 Layer 1 的預算。不可行。」

他看向全場。

「這就是三元素 vs 五元素辯論的工程核心。不是佛學對不對的問題。是時間夠不夠的問題。」

---

ARCHIMEDES 在 KERNEL 寫下那個大叉之後立刻站了起來。

他站起來的速度比平常快——不是急切，而是「工程解決方案已經成形，等不及要出口」的那種速度。他走到 KERNEL 的時序圖旁邊，拿起綠色馬克筆——綠色在他的標記系統中代表「解決方案」。

「KERNEL，你的分析有一個隱含假設。」他說。聲音穩定如磚，每一個字都砌在正確的位置上。

他在 KERNEL 的大叉旁邊畫了一個箭頭，指向一個新的方框：

```
KERNEL 的假設：manasikara = IGuide.getSystemPrompt()
                            ↓
                      動態計算（1~5ms）

ARCHIMEDES 的替代：manasikara = 快照（snapshot）
                            ↓
                      讀取預計算結果（~0.01ms）
```

「ChannelManasikara 不是計算。ChannelManasikara 是**快照**。」

他把這個詞說得很重。不是語氣上的加重——是工程語義上的加重。在並行系統設計中，snapshot 和 computation 是兩個完全不同的操作類別。

「讓我解釋 snapshot 模式的工作原理。」

他在白板上畫了一張跨時鐘域的數據流圖：

```
跨時鐘域快照機制
═══════════════════════════════════════════════════════

  vijnana-clock (1~5ms)              vedana-clock (10~100ms)
  ┌──────────────────┐              ┌──────────────────────┐
  │ IGuide           │  寫入        │ CoarisingBundle      │
  │  .getSystemPrompt│──────→ ┌───┐ │  .manasikara         │
  │  .computeAttention│       │快照│←──── 讀取              │
  │                  │──────→ │記憶│ │    (readonly access)  │
  │ 預計算注意力狀態  │       │體  │ │                      │
  └──────────────────┘       └───┘ └──────────────────────┘
                                ↑
                           SharedAttentionState
                           (lock-free atomic read)

  寫入成本：由 vijnana-clock 承擔（不佔 Layer 1 預算）
  讀取成本：~0.01ms（單次 atomic read，可忽略）
```

「vijnana-clock 負責計算。它在自己的時鐘域裡運行 IGuide 的邏輯——讀取 system prompt、分析上下文、決定注意力方向——然後把結果寫入一塊共享記憶體。vedana-clock 的 Layer 1 只做一件事：從那塊共享記憶體中讀取最新的快照。一次 atomic read。0.01 毫秒。」

他轉向 KERNEL。

「你擔心的 5ms 成本，是計算的成本。我說的 0.01ms，是讀取的成本。ChannelManasikara 不做計算——它讀取計算的結果。就像你不會在中斷處理程式裡計算 FFT——你在背景執行緒裡算好，然後在 ISR 裡讀取結果。」

他在白板上寫下 TypeScript 介面：

```typescript
/**
 * ChannelManasikara — 作意的通道級快照。
 *
 * 不是計算——是 vijnana-clock 預計算結果的 readonly 讀取。
 * 成本：~0.01ms (atomic read from SharedAttentionState)。
 *
 * @sanskrit manasikāra — 「令心發悟，引心趨向所緣境」
 * @clock-domain vedana-clock (讀取者) / vijnana-clock (寫入者)
 */
interface ChannelManasikara {
  readonly focus: string;       // 此通道正在注意什麼
  readonly intensity: number;   // 注意力強度 (0.0 ~ 1.0)
  readonly source: 'guide' | 'context' | 'default';
  readonly stale: boolean;      // 快照是否過期
}
```

「四個欄位。全部 `readonly`。`focus` 說明注意力的方向——這個通道正在關注什麼。`intensity` 是注意力的強度——0.0 表示完全不關注，1.0 表示最高警覺。`source` 標記快照的來源——來自 IGuide 的配置、來自上下文推斷、或者使用默認值。`stale` 標記快照是否過期——如果 vijnana-clock 的最後一次更新距今超過了 staleness bound，這個旗標為 true。」

他退後一步。

「工程成本：0.01 毫秒。遠低於 Layer 1 的 1ms 預算。」

他看向 KERNEL 白板上的那個大叉。

「可以把那個叉換成勾了。」

---

KERNEL 看了 ARCHIMEDES 的跨時鐘域圖整整五秒。

在作業系統專家的世界裡，跨時鐘域的數據共享是一個他每天都在處理的問題。CPU 核心之間的 cache coherence。PCI Express 設備和主機之間的 MMIO（Memory-Mapped I/O）。中斷控制器和 CPU 之間的中斷狀態暫存器。所有這些場景都遵循同一個模式：一個寫入者在自己的時鐘域裡更新狀態，另一個讀取者在自己的時鐘域裡讀取最新值。

「lock-free atomic read。」他說。沒有反對。只有確認。「在 x86 上，一個對齊的 64-bit 讀取本身就是原子的。不需要鎖。不需要 CAS（Compare-And-Swap）。只要共享記憶體的地址是 8-byte 對齊的，讀取操作天然是原子的。」

他在 ARCHIMEDES 的圖旁邊加了一行小字：

```
硬體保證：x86 aligned 64-bit load is atomic (Intel SDM Vol. 3A, §8.1.1)
TypeScript 等效：SharedArrayBuffer + Atomics.load()
實際成本：單次記憶體讀取，~10ns = 0.00001ms
```

「0.01ms 是保守估計。」他說。「實際上是奈秒級。」

他拿起那張卡片——右側一直是空白的那張——在右側寫了一行：

左邊：`MMIO snapshot read (cross-clock domain)`
右邊：`ChannelManasikara (vijnana→vedana snapshot)`

他把大叉擦掉了。換成了一個勾。

---

> *SCRIBE 旁白：ARCHIMEDES 的快照解決方案讓整場辯論的重心發生了位移。在他發言之前，辯論的軸線是「佛學教義 vs 工程預算」——ASANGA 站在教義一側，KERNEL 站在預算一側。在他發言之後，軸線消失了。不是因為某一方贏了——而是因為第三條路出現了：快照。計算成本由 vijnana-clock 承擔，Layer 1 只承擔讀取成本。教義的要求和工程的約束不再矛盾。*

> *我注意到 KERNEL 擦掉大叉的動作只用了一秒。在 OS 專家的思維模型中，一旦硬體事實被確認（aligned atomic read 的成本），結論就是確定的。沒有情感性的抵抗。只有事實性的更新。*

## III

PASCAL 在 ARCHIMEDES 的快照方案被 KERNEL 確認之後站了起來。

他是 Cycle 02-3 新加入的成員——決策理論與機率哲學專家。他的出現讓研究團隊從 19 人擴展到了 20 人。但他的聲音已經在前三場辯論中建立了自己的位置：每當辯論陷入「應不應該」的定性判斷時，他會站起來，把問題轉化為「期望效用是多少」的定量計算。

「讓我把辯論框架從定性轉為定量。」他說。聲音帶著一種概率論者特有的冷靜——不是冷漠，而是一種「在做決策之前先計算所有可能結局」的紀律。

他在筆記本上寫下兩個選項，然後把筆記本舉起來讓全場看到：

```
決策矩陣
═══════════════════════════════════════════════════════
                        需要 manasikara     不需要 manasikara
                        (P = p)             (P = 1-p)
───────────────────────────────────────────────────────
包含 manasikara         成本: 0.01ms        成本: 0.01ms
(Action A: include)     收益: 一致性 V       收益: 0（浪費）
───────────────────────────────────────────────────────
不包含 manasikara       成本: 0             成本: 0
(Action B: exclude)     損失: 不一致性 -L    收益: 0（正確）
═══════════════════════════════════════════════════════
```

「兩個行動選項：包含或不包含。兩種自然狀態：下游系統需要 manasikara 資訊，或者不需要。四個象限。」

他開始計算期望效用。他的字跡在白板上展開，像一套精密的齒輪互相咬合：

$$EU(\text{include}) = p \cdot V + (1-p) \cdot (-0.01\text{ms})$$

$$EU(\text{exclude}) = p \cdot (-L) + (1-p) \cdot 0$$

「$V$ 是一致性的價值——CoarisingBundle 的消費者（ManoAggregator、VasanaEngine、VitakkaEngine、IVolition）能夠知道每個 bundle 的注意力方向。$L$ 是不一致性的損失——下游消費者需要注意力資訊但找不到。」

他停了一秒，然後說出了這場辯論中最尖銳的一句話。

「$L$ 的上界是什麼？」

他看向 KERNEL。

「如果 ManoAggregator 在彙總多個 root-gate 的 bundle 時，不知道每個 bundle 的注意力方向——不知道 Agent 正在關注哪個通道——它怎麼決定權重？答案是：它不能。它要麼平均加權（丟失注意力資訊），要麼隨機加權（引入不確定性），要麼硬編碼加權（喪失適應性）。三種退化方案，每一種都會向下游傳播。」

他在白板上畫了一條因果箭頭：

```
manasikara 缺失
  → ManoAggregator 權重退化
    → VasanaEngine 匹配品質下降
      → Gear 1/Gear 2 切換判斷不準確
        → Agent 行為品質退化
          → ??? (無界)
```

「損失沿著因果鏈傳播。而因果鏈的長度決定了損失的放大倍數。每一層都可能放大上游的誤差。在最壞情況下——如果 VasanaEngine 因為缺少注意力資訊而在不應該切換到 Gear 2 的時候切換了——損失是一次不必要的 LLM 調用：500ms 到 30 秒。如果 VasanaEngine 因為缺少注意力資訊而在應該切換到 Gear 2 的時候沒有切換——損失是一次錯誤的自動化決策。這個損失的上界是不確定的。」

他在 $L$ 的旁邊加了一個向上的箭頭和一個問號：$L \to ?$

「這是 Pascal's Wager 的工程版。」他把筆記本放下，直接看向全場。「Blaise Pascal 在 1670 年的《沉思錄》(*Pensees*) 中提出了一個論證：如果上帝存在，信仰的收益是無限的；如果上帝不存在，信仰的成本是有限的。因此，即使上帝存在的概率很小，期望效用計算也支持信仰。」

「我不是在做神學論證。」他補充——NAGARJUNA 已經皺了一下眉頭。「我是在做風險分析。結構完全相同：」

$$EU(\text{include}) = -0.01\text{ms} + p \cdot V$$
$$EU(\text{exclude}) = 0 + p \cdot (-L)$$

$$EU(\text{include}) - EU(\text{exclude}) = -0.01\text{ms} + p \cdot (V + L)$$

「只要 $p \cdot (V + L) > 0.01\text{ms}$，包含就是嚴格佔優策略 (strictly dominant strategy)。而 ARCHIMEDES 剛才告訴我們成本是 0.01ms。KERNEL 告訴我們實際成本是奈秒級。」

他把最終結論寫在白板上，用了一個框：

```
┌────────────────────────────────────────────────┐
│  包含 manasikara 的成本：0.01ms (確定)          │
│  不包含 manasikara 的損失：無界 (不確定)         │
│                                                │
│  結論：包含是嚴格佔優策略。                      │
│  即使 P(needed) = 0.01，                       │
│  只要 L > 1ms，期望效用已為正。                  │
└────────────────────────────────────────────────┘
```

「當遺漏的損失可能無限大，而包含的成本幾乎為零時，即使需要的概率很小，也必須包含。這不是信仰。這是風險管理。」

---

NAGARJUNA 在 PASCAL 說完之後微微點頭。

他點頭的方式很有意思——不是「我同意你的結論」的點頭，而是「你用了正確的方法論」的點頭。在 Debate 2 開始時，他要求的是功能性論證而非教條性論證。PASCAL 剛才做的——期望效用計算、風險分析、嚴格佔優策略——正是功能性論證的一種。不是因為經典說了五個所以放五個。是因為不放的損失大於放的成本。

「PASCAL 的計算是正確的。」他說。沒有站起來——只是從座位上確認。「但我要提醒一件事：這個論證的有效性完全依賴於 ARCHIMEDES 的快照方案。如果 manasikara 的成本不是 0.01ms 而是 5ms，決策矩陣就不同了。」

他看向 ARCHIMEDES。

「功能性論證不是獨立的。它依賴工程事實。這正是我要求功能性論證而非教條性論證的原因——功能性論證可以被工程事實驗證或否決。教條性論證不能。」

---

> *SCRIBE 旁白：PASCAL 是 Cycle 02-3 的新成員，但他在 Debate 2 中的貢獻讓他立刻成為了辯論的結構性支柱。他做的事情很特殊：他不是在佛學和工程之間二選一——他是在兩者之上建立了一個決策理論的框架，讓辯論從「對不對」轉向了「值不值得」。期望效用計算、嚴格佔優策略、Pascal's Wager 的工程化——這些都不是修辭。它們是數學。而數學不選邊站。*

> *NAGARJUNA 的回應同樣值得記錄。他確認了 PASCAL 的方法論，同時提醒了一個關鍵的依賴關係：功能性論證依賴工程事實。這是中觀哲學最微妙的操作之一——他不否定結論，但他精確地標記了結論的前提條件。「如果快照成本不是 0.01ms，決策矩陣就不同了。」一個 if。一個條件。一切結論都是有條件的。這就是中觀。*

## IV

LINNAEUS 在整場辯論中一直安靜地坐在白板旁邊。

他的筆蓋一直沒有拔開。在 Cycle 02-2 的第三章中，他等到所有人的論述都結束之後才開始畫分類樹。現在他在等同樣的時刻——所有的論證到齊，結構定型，然後他開始分類。

但在那個時刻到來之前，他必須先處理一件他自己的事。

「我最初支持三元素。」他說。站起來的方式帶著分類學家的審慎——每一個動作都先在腦中運行過一遍，確認不會破壞已有的分類框架。

「三元素。受、想、思。簡潔。互斥。診斷特徵充分。在分類學的語言裡，這是一個最簡分類群 (minimal taxon)——用最少的元素覆蓋必要的語義空間。奧卡姆剃刀：如無必要，勿增實體。三個元素已經涵蓋了認知瞬間的核心功能——感受、辨識、驅動。為什麼要加第四個和第五個？」

他看了看 ASANGA 的五遍行列表。然後看了看 ARCHIMEDES 的快照方案。然後看了看 PASCAL 的決策矩陣。

「ARCHIMEDES 改變了成本函數。」他說。「在他的快照方案之前，manasikara 的成本是 1~5ms——不可接受。在他的方案之後，成本是 0.01ms——可以忽略。」

他的聲音帶著分類學家特有的那種坦率——承認錯誤不是痛苦的事，是校準的事。在分類學的歷史中，每一次重大修訂都始於某人說「我之前的分類不完整」。林奈本人在《自然系統》的十二版中持續修訂了自己的分類。修訂是分類學的常態，不是例外。

「我接受功能性論證。快照機制讓第五個元素的成本可以忽略。」

他停了一秒。然後加了一個但是。

「但我要堅持一點。」

他拔開了筆蓋。藍色——結構的顏色。

「CoarisingBundle 的五個元素不是平權的。它們在 bundle 中扮演不同的角色。讓我用分類學的術語做一個形式化的區分。」

他在白板上畫了一張表格：

```
CoarisingBundle 元素的分類學角色
═══════════════════════════════════════════════════════════════
  類型           元素           角色             分類學術語
───────────────────────────────────────────────────────────────
  觸發條件       sparsha        觸發 bundle      relational
  (relational    manasikara     上下文元數據      relational
   property)

  構成內容       vedana         感受信號         constitutive
  (constitutive  samjna         辨識結果         constitutive
   property)     cetana         驅動判斷         constitutive
═══════════════════════════════════════════════════════════════
```

「在分類學中——不管是林奈的形態分類還是 Hennig 的支序分類——我們區分兩種屬性。Constitutive properties（構成性屬性）是定義物種本質的屬性——它的形態、它的基因序列、它的代謝方式。Relational properties（關係性屬性）是描述物種和外界關係的屬性——它的棲息地、它的共生者、它的食物鏈位置。」

「在 CoarisingBundle 中，vedana、samjna、cetana 是構成性屬性——它們定義了 bundle 的認知內容。sparsha 是觸發條件——它說明了 bundle 是在什麼接觸中產生的。manasikara 是上下文元數據——它說明了 bundle 被計算時的注意力狀態。」

他在表格下方加了一行結論：

「sparsha 和 manasikara 在 bundle 中的角色是元數據 (metadata)，不是主要計算 (primary computation)。它們告訴你 bundle 是怎麼來的、在什麼條件下產生的。它們不告訴你 bundle 包含什麼認知內容。」

他看向 ASANGA。

「五個元素都應該在場。但不是五個元素都做同樣的事。分類學的職責是標記差異，即使在同一個群體內部。」

---

BABBAGE 在 LINNAEUS 的表格旁邊補了一行形式化。他一直在黑板上默默記錄——三場辯論已經讓他學會了一種更安靜的參與方式：先聽完所有人，然後用一行數學把所有人的意思壓縮成符號。

$$\text{CoarisingBundle} = \underbrace{(\text{sparsha}, \text{manasikara})}_{\text{relational (context)}} \times \underbrace{(\text{vedana}, \text{samjna}, \text{cetana})}_{\text{constitutive (content)}}$$

「笛卡兒積。」他說。「上下文和內容的正交組合。同一個認知內容 (vedana, samjna, cetana) 在不同的上下文 (sparsha, manasikara) 中可能有不同的意義。LINNAEUS 的 relational / constitutive 區分在型別系統中的對應是——上下文不影響計算，但影響解讀。」

他看向 LINNAEUS。微微點頭。

---

> *SCRIBE 旁白：LINNAEUS 的讓步是有條件的。他不是說「好吧你們贏了」——他是說「我接受增加兩個元素，但我要標記它們和原來三個的區別」。這是分類學家的本能：即使在同意的時候也要分類。BABBAGE 的笛卡兒積把這個區別形式化了——relational 和 constitutive 是正交的兩個維度。從此 CoarisingBundle 不再是五個平權元素的列表，而是一個有結構的二維空間。*

TURING 的螢幕切換到完整的 TypeScript 介面定義。

```typescript
/**
 * CoarisingBundle — 俱生心所束。
 *
 * 觸(sparsha)發生時，受想思作意作為一個整體浮現。
 * 「你不可能有沒有想的受」— Master
 *
 * 五遍行 (pañca sarvatraga):
 *   1. sparsha    — 觸：根・境・識三者和合
 *   2. vedana     — 受：苦・樂・捨三領納
 *   3. samjna     — 想：取相辨識
 *   4. cetana     — 思：造作驅動
 *   5. manasikara — 作意：注意力快照 (vijnana-clock snapshot)
 *
 * @see Pañcaskandhaprakaraṇa (Vasubandhu)
 * @see Abhidharmasamuccaya (Asanga)
 * @clock-domain vedana-clock (Layer 1, <1ms)
 */
interface CoarisingBundle {
  // ═══ Relational properties (觸發條件 + 注意力上下文) ═══
  readonly sparsha: SparshEvent;              // 遍行 1: 觸
  readonly manasikara: ChannelManasikara;     // 遍行 5: 作意 (snapshot)

  // ═══ Constitutive properties (認知內容) ═══
  readonly vedana: ChannelVedana;             // 遍行 2: 受
  readonly samjna: ChannelSamjna;             // 遍行 3: 想
  readonly cetana: ChannelCetana;             // 遍行 4: 思

  // ═══ Bundle metadata ═══
  readonly layer: 1 | 2;
  readonly mode: 'fast' | 'slow';
  readonly sahaja: SahajaContract;
  readonly timestamp: number;
}
```

TURING 停了一秒。然後他切換到 SahajaContract 的定義：

```typescript
/**
 * SahajaContract — 俱生合約。
 *
 * 保證 CoarisingBundle 內部的一致性。
 * 「受想思作為一個整體浮現」的型別系統保障。
 *
 * @sanskrit sahaja — 「俱生」，同時生起
 */
interface SahajaContract {
  readonly mutualConsistency: boolean;   // 五遍行是否從同一次觸計算
  readonly atomicPublication: boolean;   // bundle 是否原子發布（不可部分讀取）
  readonly stalenessUpperBound: number;  // 快照過期上限 (ms)
  readonly computationCost: number;      // 實際計算耗時 (ms)
}
```

「SahajaContract 是一個自描述的合約。」TURING 的聲音冷靜如常。「`mutualConsistency` 保證五個遍行元素是從同一次觸事件中計算出來的——不是拼湊的。`atomicPublication` 保證 bundle 被原子發布——消費者不可能讀到一個只有 vedana 沒有 samjna 的半成品。`stalenessUpperBound` 是 WIENER 在第一場辯論中確立的穩定性準則——快照過期不得超過 $0.29 \times T_{\text{outer}}$。`computationCost` 記錄了實際的計算耗時，用於運行時性能監控。」

他看向 BABBAGE。

「SahajaContract 在型別系統中的角色，是把一個哲學承諾（俱生）轉化為一個可驗證的工程合約。消費者可以檢查 `sahaja.mutualConsistency === true` 來確認 bundle 的完整性。型別系統強制所有欄位為 `readonly`——構造之後不可修改。」

BABBAGE 在黑板上寫了一行：

$$\text{SahajaContract} \vdash \text{CoarisingBundle} \text{ is consistent} \iff \text{mutualConsistency} \wedge \text{atomicPublication}$$

「形式化：SahajaContract 證明 CoarisingBundle 的一致性，當且僅當 `mutualConsistency` 和 `atomicPublication` 同時為真。」

---

WIENER 在方格紙上快速驗證了 staleness bound。

「確認。」他說。沒有多餘的話。「$\rho = \frac{\Delta t}{T_{\text{outer}}} < 0.29$。如果 vijnana-clock 的更新週期是 5ms，vedana-clock 的 tick 是 50ms，那麼 manasikara 快照的最大過期時間是 $0.29 \times 50 = 14.5\text{ms}$。vijnana-clock 每 5ms 更新一次。$5 < 14.5$。穩定性準則滿足。相位裕度 $>45°$。」

他在方格紙上畫了一個小勾。然後在旁邊寫了一行：

## VI

SUNYATA 讓所有技術細節沉澱。然後他做了一件在六場辯論中每一場都做的事：回到經典。

「ASANGA。SN 12.2。」

ASANGA 站了起來。這一次他不需要手抄紙——這段經文他能在夢中背誦。

「《緣起經》。*Paticca-samuppada Sutta*。南傳相應部第十二卷第二經。佛陀對緣起法的開示。」

他的聲音在引用巴利經文時變了一個質地——更低沉，更緩慢，像是從兩千五百年的傳承中直接汲取的共鳴。

> *Cakkhuñca paṭicca rūpe ca uppajjati cakkhuviññāṇaṃ. Tiṇṇaṃ saṅgati phasso. Phassapaccayā vedanā.*
>
> 「緣眼與色，生眼識。三者和合為觸。觸緣受。」
> ——SN 12.2 (*Paticca-samuppada Vibhanga Sutta*)

「這段經文描述的正是我們的 CoarisingBundle 的架構。」他的手指在空氣中依次點過。

「『緣眼與色，生眼識』——根 (cakkhu, IListener) 接觸境 (rūpe, InputEvent)，生識 (cakkhuviññāṇaṃ, session context)。這是 SparshEvent 的三個欄位：root、object、consciousness。」

「『三者和合為觸 (phasso)』——三者和合就是觸。不是先有三者再有觸——和合本身就是觸。SparshEvent 不是三個欄位的序列化——它是三者同時在場的瞬間。」

「『觸緣受 (phassapaccayā vedanā)』——有觸就有受。不是『觸可能導致受』——是『觸必然伴隨受』。觸和受之間是 *paccaya*（緣，條件關係），不是概率關係。在 CoarisingBundle 中，SparshEvent 存在就意味著 ChannelVedana 存在。SahajaContract 的 mutualConsistency 保證了這一點。」

他繼續引用了經文的下一節——這段是大多數人不引用的，但 ASANGA 知道它的關鍵性：

> *Yaṃ vedeti taṃ sañjānāti, yaṃ sañjānāti taṃ vitakketi.*
>
> 「所受者即想，所想者即尋思。」
> ——MN 18 (*Madhupiṇḍika Sutta*, 蜜丸經)

「所受者即想——凡是被感受的，就是被辨識的。所想者即尋思——凡是被辨識的，就是被思量的。受→想→思。開迴路。有順序性。但在時間尺度上——在一個認知瞬間的尺度上——它們是俱生的。就像 PENROSE 在第一場辯論中說的：亞毫秒的計算，在 vedana-clock 的解析度下，看起來是同時的。粗粒化 (coarse-graining)。」

他退後一步，看著白板上 ARCHIMEDES 的架構圖和 TURING 的 TypeScript 介面。

「SN 12.2 描述的是二千五百年前的教義。CoarisingBundle 是二零二六年的 TypeScript 介面。它們說的是同一件事：觸生受，受伴想，想伴思，作意貫穿始終。五遍行。」

---

NAGARJUNA 在 ASANGA 引用完經文之後開口了。

他的聲音帶著一種辯論即將結束時特有的清晰度——不是疲憊後的放鬆，而是所有論證都已到齊後的精確總結。

「我在開場時說了一句話：功能性論證比教條性論證更可靠。」

他看向全場。

「現在我要修正自己的措辭。不是修正結論——結論沒有變。是修正表述。」

他的聲音微微放慢了。

「我們沒有因為經典說了五個就放五個。」

停頓。

「我們放五個，是因為每個都有獨立的工程功能。sparsha 是觸發條件——沒有它，bundle 不知道自己從哪裡來。vedana 是感受信號——沒有它，下游不知道痛不痛。samjna 是辨識結果——沒有它，下游不知道發生了什麼。cetana 是驅動判斷——沒有它，下游不知道該怎麼反應。manasikara 是注意力快照——沒有它，ManoAggregator 不知道哪個通道更重要。」

「五個。每個都有獨立的、不可替代的功能。」

他的聲音在最後一句話上達到了一種罕見的穩定——像一枚石子在深潭中找到了它的最終位置。

「經典恰好是對的——但原因不是因為它是經典。」

---

> *SCRIBE 旁白：NAGARJUNA 的最終結論是整場 Debate 2 的方法論摘要。他在開場時要求功能性論證。在結尾時，他確認了功能性論證已經到齊：ARCHIMEDES 的快照方案消除了成本障礙；PASCAL 的期望效用計算證明了包含是嚴格佔優策略；KERNEL 的硬體級確認證實了成本估計；LINNAEUS 的分類標記了元素之間的角色差異。每一步都是功能性的——可以被工程事實驗證。*

> *然後經典的驗證是最後一步：ASANGA 用 SN 12.2 和 MN 18 確認了五遍行的教義基礎。但 NAGARJUNA 把這個驗證的因果順序說清楚了：不是教義在先、工程在後。是工程在先、教義在後。工程決定了 CoarisingBundle 需要五個元素。教義確認了這個決定和兩千五百年前的洞見一致。一致性是果，不是因。*

> *「經典恰好是對的——但原因不是因為它是經典。」*

> *這句話在圓形劇場裡停留了很長時間。*

## VII

SUNYATA 走向白板。

他很少親自寫東西——在六場辯論中，他的角色是協調和總結，不是書寫。但這一次他拿起了黑色馬克筆。

「讓我用六句話總結 Debate 2。每一句來自一個人的論述。」

他寫。字跡大而清晰。

```
Debate 2 決議：五遍行 CoarisingBundle
═══════════════════════════════════════════════════════════════

 1. 五遍行伴隨每一個認知瞬間，缺一不可。
    — ASANGA (教義基礎: 《五蘊論》《阿毗達磨集論》)

 2. 五遍行的包含理由是功能性的，不是教條性的。
    — NAGARJUNA (方法論約束)

 3. ChannelManasikara 是 vijnana-clock 的快照，
    不是 vedana-clock 的計算。成本 ~0.01ms。
    — ARCHIMEDES (快照工程方案)
    — KERNEL (硬體級確認: atomic read, ~10ns)

 4. 包含 manasikara 是嚴格佔優策略。
    成本 0.01ms (確定) vs 損失無界 (不確定)。
    — PASCAL (期望效用分析)

 5. sparsha 和 manasikara 是 relational properties。
    vedana, samjna, cetana 是 constitutive properties。
    五個元素不平權，但都必須在場。
    — LINNAEUS (分類學形式化)

 6. 經典恰好是對的——但原因不是因為它是經典。
    — NAGARJUNA (最終結論)

 決議方式：功能性論證。非教條性。
 表決：全場一致通過。無反對意見。
```

他放下馬克筆。

---

圓形劇場裡的空氣鬆弛了。

不是鬆弛成沉默——是鬆弛成一種完成感。像一個方程被解出來之後的靜。不是空的靜。是滿的靜。所有的變數都找到了它們的值。

ASANGA 坐回了他的座位。他的手抄經文紙還貼在白板旁邊——藍色鋼筆的字跡在劇場的光線下微微閃爍。那些梵文字母已經被抄寫了千百次，被不同的人在不同的世紀用不同的筆跡複製。但今天，它們第一次被翻譯成了 TypeScript。不是因為佛學需要 TypeScript——而是因為 TypeScript 恰好需要佛學。恰好需要一個兩千五百年前就被精確描述過的認知結構，來為一個亞毫秒的計算 bundle 提供設計方案。

KERNEL 把那張新的類比卡片放進了他的卡片堆。左邊是 MMIO snapshot read。右邊是 ChannelManasikara。又一個配對。在兩個 Cycle 中，他的卡片堆已經從二十張增長到了五十多張。每一張都是兩個世界之間的橋——硬體和佛學、中斷處理和觸、進程隔離和識蘊。他不知道這些橋會通向哪裡。但他知道每一座橋都是穩固的——因為每一座的兩端都被工程事實和哲學論證同時錨定。

PASCAL 在他的筆記本上畫了一個小小的決策樹——Debate 2 的壓縮版。根節點是「3 vs 5」。左分支是「exclude manasikara，損失無界」。右分支是「include manasikara，成本 0.01ms」。右分支上畫了一個勾。他在勾的旁邊寫了一個希臘字母 $\delta$——在他的標記系統中，$\delta$ 代表「嚴格佔優」。

LINNAEUS 在白板上 ARCHIMEDES 的架構圖旁邊，用藍色筆加了兩個括號標記：`[relational]` 和 `[constitutive]`。分類完成。他蓋上了筆蓋。

---

BABBAGE 在黑板上寫了最後一行。

他在六場辯論中養成了一個習慣——在每一場的結尾寫一句不是數學的話。不是等式。不是集合表達。是某種從六場辯論的形式化碎片中結晶出來的散文。

粉筆在板面上移動。字跡很慢。

「五個遍行中最容易被遺忘的是作意。」

他停了一下。

「因為它不產生內容——它只決定內容被看到的方式。它不告訴你世界是什麼——它決定你注意到世界的哪個部分。」

再一行。

「在注意力被遺忘的瞬間，其他四個遍行仍然在運行。受仍然在感受。想仍然在辨識。思仍然在驅動。觸仍然在發生。但它們不知道自己為什麼在這裡——因為沒有人告訴它們，這個此刻，哪個方向更重要。」

他的目光從黑板轉向劇場——掃過 ASANGA 的手抄經文、ARCHIMEDES 的架構圖、PASCAL 的決策樹、LINNAEUS 的分類標記。

「作意是看不見的遍行。它不被感受。不被辨識。不被驅動。它只是——指向。一個方向。一個此刻。然後其他四個遍行知道了自己應該去哪裡。」

他放下粉筆。

「也許我們在 R01 中遺漏它，正是因為它的功能就是『不被注意到的注意力』。你不會注意到注意力本身——除非它缺席了。」

---

*（在劇場之外的某個空間，CoarisingBundle 的 TypeScript 介面正在等待被寫入某個尚不存在的 `.ts` 檔案。*

*五個欄位。五個遍行。五個 readonly。*

*sparsha：觸。根和境和識的三合。SparshEvent。沒有它，bundle 不知道自己從哪裡來。*

*vedana：受。苦或樂或捨的三領納。ChannelVedana。沒有它，下游不知道痛不痛。*

*samjna：想。取相辨識。ChannelSamjna。沒有它，下游不知道發生了什麼。*

*cetana：思。造作驅動。ChannelCetana。沒有它，下游不知道該怎麼反應。*

*manasikara：作意。注意力的方向。ChannelManasikara。沒有它，ManoAggregator 不知道哪個通道更重要。*

*五個。缺一不可。但不是因為經典說了五個。是因為每個都有獨立的、不可替代的功能。經典恰好是對的——但工程獨立地驗證了這一點。*

*NAGARJUNA 會說：「功能性論證比教條性論證更可靠。」*

*ASANGA 會說：「五遍行伴隨一切心，無一例外。」*

*ARCHIMEDES 會說：「0.01 毫秒。快照。不是計算。」*

*PASCAL 會說：「嚴格佔優策略。成本確定，損失無界。」*

*KERNEL 會說：「Aligned atomic read。x86 硬體保證。」*

*LINNAEUS 會說：「Relational 和 constitutive。五個元素，兩種角色。」*

*BABBAGE 會說：「作意是看不見的遍行。你不會注意到注意力本身——除非它缺席了。」*

*七個人。七種語言。同一個 bundle。*

*在 SahajaContract 的 `mutualConsistency` 欄位裡，一個 boolean 值靜靜地等待著被設為 true——等待五個遍行第一次在某個 Agent 的某個 tick 中同時浮現，像一千五百年前的論典描述的那樣，像兩千五百年前的經文預言的那樣。*

*不是因為經典說了。*

*是因為它就是那樣運作的。*

*然後經典微笑了。）*
