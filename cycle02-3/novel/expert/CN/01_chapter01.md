# 第一章：三十七份文件

---

圆形剧场的门在研究团队到来之前就已经开了。

不是被推开的。更像是它一直开着——从工程团队离开的那一刻起，这扇门就保持敞开的姿态，像一个邀请，像一个问句。门后的剧场里，空气中残留着某种秩序的痕迹。桌上的文件被整齐地分成四叠。投影幕上的最后一张图表还亮着，是一棵继承树——五个梵文名称从根部向上展开，每一个分支都用虚线标注了「待确认」。

SUNYATA 是第一个走进来的。

他站在门口，没有立刻走向自己的位置。他在看那些文件。四叠。他数了一遍。两遍。然后他的目光落到投影幕上的继承树，在那棵树的根部停了三秒。

（SCRIBE 旁白：SUNYATA 进门后的前十秒是沉默的。他的沉默不是犹豫——是测量。他在测量这个房间里残留的思维密度。工程团队在这里工作了多久？留下了多少东西？这些东西的重量是多少？这些问题的答案会决定接下来整个研究周期的起始姿态。）

「三十七份。」SUNYATA 说。他的声音没有感叹，没有惊讶，只有一个数字的精确重量。

其他人陆续进场的方式各有不同。

BABBAGE 走进来的时候笔记本已经翻到了空白页。他的手指在页面边缘留下了一个几乎不可见的折痕——他的速记标记，意味着「这一页的内容尚未开始但注定密集」。他的目光扫过四叠文件的高度，快速估算了总页数：讨论文件 12 份，每份约 8-15 页；交付包 7 份，结构化文件，页数较少；待办跟踪 8 份，清单格式；加上 openstarry_doc 目录下的 120 余份。一个组合爆炸的阅读量。

$$\text{Total pages} \approx 12 \times 12 + 7 \times 6 + 8 \times 4 + 120 \times 8 \approx 1,186$$

他在笔记本上写下了这个数字，然后在旁边画了一个问号。页数不是问题。问题是信息密度——1,186 页中有多少是新的语义单元，多少是已知结论的重新表述。如果新语义占比低于 30%，阅读效率将急剧下降。如果高于 70%，团队可能需要额外一轮定向。

NAGARJUNA 进来的时候看到了投影幕上的继承树。他停了下来。不是因为树的结构——IRupa、IVedana、ISamjna、ISamskara、IVijnana，五蕴的梵文命名——而是因为树的根部。那里写着一行小字：

```
// 根接口重命名：ISensory → IRupa, ISensation → IVedana, ...
```

箭头。从英文到梵文的箭头。NAGARJUNA 看着那个箭头，看了很久。

「字符串替换。」他低声说。这两个字不是赞美。

WIENER 是带着马克杯进来的。他把杯子放在离桌边最近的位置，然后翻开了第一份他看到的文件——M10_temporal_architecture.md。他读了标题下面的第一行：「识最快、行次之、想最慢。」他放下了文件。他的表情是工程师看到一个未被证明的公理时特有的不安——不是反对，是不安。

「识最快。」他对自己重复了一遍。「依据是什么？」

KERNEL 几乎同时翻开了同一份文件的另一个副本。他看的是不同的段落——多时钟事件驱动的架构图。他的手指沿着图中的箭头移动：vijnanaClock (1-10ms) → samskaraClock (10-100ms) → vedanaClock (100-500ms) → rupaClock (I/O bound) → samjnaClock (500ms+)。

「五个独立时钟。」他说。他的语气里没有惊叹，只有操作系统工程师在遇到多时钟同步问题时本能的警觉。「谁负责同步？优先级反转呢？如果 vijnanaClock 的事件被 samjnaClock 的长运行任务阻塞了呢？」

没有人回答。这些问题还不到被回答的时候。它们只是被种下了。

「但更重要的是我们面前**没有**的东西。」

SYNTHESIST 抬起头。他一直在翻阅第二叠文件，带着那种整合者特有的全局视角——不看细节，看结构。他知道 SUNYATA 要说什么。

GUARDIAN 点了点头。他翻到了 M3_klesha_multilayer_di.md 的第三节——Klesha 继承体系——看到了那段 TypeScript 接口定义。四根本烦恼，四个 extends IKlesha 的接口，每个都有一个 `intensity: number`。干净。整齐。

太整齐了。

他看向 SCRIBE。

「记下来。研究团队的立场声明——」

（SCRIBE 旁白：我拿起了新的记录簿。Cycle 02-3 专用。封面是深蓝色的。我在第一页写下了日期：2026-02-25。然后是 SUNYATA 的声明。）

「工程团队的文件是**输入材料**。不是结论。它们是有价值的参考——系统化地盘点了所有议题，TypeScript 接口设计具体可用。但研究团队的价值不在于重复工程分析，而在于**多视角碰撞**和**更深的维度**。」

NAGARJUNA 在自己的位置上微微倾身。「更深的维度——」他重复了这几个字，语气中带着一种特殊的重量，那种哲学家在触碰到核心概念时本能的共鸣。

「我们有二十位学者。哲学家、佛学家、理论计算机科学家、控制理论家、安全专家、分类学家、量子意识研究者——以及新加入的决策理论家。」SUNYATA 看向场地右侧最后一个座位。「这是工程团队没有的宽度。」

PASCAL 在自己的位置上安静地点了一下头。他还没有说话。但他的笔记本已经翻开了，上面画着一个 2x2 的矩阵——行是「分析/决策」，列是「确定/不确定」。左上角写着「工程团队」；右下角画了一个圆圈，里面是空的。

那个圆圈是他自己的位置。

## 逐份审阅

审阅开始于 M1。

SUNYATA 没有从 M5（触→俱生）开始——那是所有人都知道的核心议题——而是从 M1（梵文命名）开始。他的选择是刻意的。命名是最表层的问题，但也是最基础的。一棵树从根部开始生长。命名就是根。

「M1_sanskrit_naming.md。」SUNYATA 读出文件名。「Master 裁定：五蕴 Plugin 用五蕴来命名。ISensory → IRupa，ISensation → IVedana，ICognition → ISamjna，IAction → ISamskara，IIdentity → IVijnana。四根本烦恼、七原罪、随烦恼也用相关原文。程序代码要能编译。」

LINNAEUS 翻开了工程团队的 M1 文件。他的分类学家本能让他先看结构再看内容。

「命名表完整。」他的第一句话是肯定。他用手指指向文件中的表格——现行接口名到梵文接口名的对照，子接口继承关系，Klesha 与心所的命名建议。「工程团队做了一个干净的迁移方案。向后兼容策略合理——TypeScript type alias 提供过渡期。」

```typescript
/** @deprecated Use IRupa */
export type ISensory = IRupa;
/** @deprecated Use IVedana */
export type ISensation = IVedana;
/** @deprecated Use ISamjna */
export type ICognition = ISamjna;
/** @deprecated Use ISamskara */
export type IAction = ISamskara;
```

「但——」LINNAEUS 翻到了文件的第四节。「哲学含义的讨论只有四行。他们说『梵文命名直接对齐佛教哲学本源，消除了翻译歧义。』然后给了三个例子——ISensory 可能被误解为『传感器』，IAction 可能被误解为『动作』，ICognition 可能被误解为『认知科学』。」他合上文件。「这是对的。但这只是命名学（nomenclature）的第一层。」

「还有什么层？」ARCHIMEDES 问。他是务实的工程师，对分类学的层级不敏感。

LINNAEUS 看向 NAGARJUNA。

NAGARJUNA 站了起来。在圆形剧场里，站起来是一个信号——它意味着接下来的话不是评论，是论述。

「用梵文命名不只是字符串替换。」他说。他的声音不疾不徐，每个字都被赋予了等量的时间。「它是**哲学承诺**。」

他走到投影幕前面，指着那棵继承树的根部。

「当你把 ISensory 改成 IRupa，你不只是改了一个英文词为一个梵文词。你是在说：**这个接口承诺忠实于色蕴的原始定义**。而色蕴的原始定义——在《心经》中、在《蕴相应》中、在《五事毗婆沙论》中——远比 'sensory input' 的含义要广。」

> 「色即是空，空即是色。色不异空，空不异色。受想行识，亦复如是。」
> ——《般若波罗蜜多心经》

「如果 IRupa 只处理 UI 渲染和 Listener 输入，那它配不上『色蕴』这个名字。」

ARCHIMEDES 皱了皱眉。「那你建议什么？不改名？」

「不。Master 已裁定，我们改名。但改名之后，每一个接口的**范畴必须被重新审视**。」NAGARJUNA 转身面向全场。「这就是我要说的核心观点——在佛学中，命名本身就是一种**假名安立** (prajnaptir upadaya)。」

他在空中用手指划了一个字。场内没有人看到那个字，但所有人都知道他写的是什么。

> 假名安立 (prajnaptir upadaya)
> ——名言只是为了沟通方便而建立的约定，
>     不指向任何具有自性（svabhava）的实体。

「当我们用 IRupa 命名一个接口，我们在做假名安立。这个名字不**是**色蕴——它是我们为了沟通方便而安立的一个符号。但假名安立有一个严格要求：**被安立的名言必须与其指涉的功能相应**。如果 IRupa 的功能只覆盖了色蕴的 30%，这个假名就是**不正确的安立**。」

BABBAGE 停下了笔。他在 NAGARJUNA 的论点中看到了一个形式结构：

$$\text{命名正确性} \iff \text{Sem}(\text{IRupa}) \supseteq \text{Sem}(\text{rupa-skandha}) \cap \text{Dom}(\text{OpenStarry})$$

其中 $\text{Sem}(x)$ 是 $x$ 的语义域，$\text{Dom}(\text{OpenStarry})$ 是 OpenStarry 系统的功能域。换言之：IRupa 的语义域必须覆盖色蕴在 OpenStarry 系统中可表达的全部功能。

NAGARJUNA 看了一眼 BABBAGE 的笔记本，点了点头。「BABBAGE 的形式化是精确的。命名不只是标签——命名是**覆盖承诺**。」

LINNAEUS 接过话。「作为分类学家，我补充一个维度。在生物分类学中，一个物种的学名（binomial name）一旦确立，它的**模式标本**（type specimen）就固定了。IRupa 一旦确立为色蕴的接口名，色蕴的**定义**就成为它的模式标本。如果工程实现偏离了模式标本，那不是实现的问题——是命名错了。」

（SCRIBE 旁白：NAGARJUNA 和 LINNAEUS 从完全不同的方向——中观哲学和分类学——得出了同一个结论：命名不只是字符串替换，是承诺。这是 Cycle 02-3 的第一次跨学科共鸣。我在笔记本上用双线标注了这个时刻。）

---

审阅继续推进到 M3。

M3_klesha_multilayer_di.md 在研究团队中引发的反应最为分裂。不是关于架构方向——四根本烦恼继承 IKlesha、注入 IVijnana 运行环境，这个方向有 Cycle 02-2 A-1（我执 = 烦恼根源）的共识作为基础。分裂发生在一个看似微小的细节上。

PASCAL 是第一个开口的。

在所有人都还在浏览四烦恼的 TypeScript 接口时——IMoha、IDrishti、IMana、ISneha——PASCAL 已经翻过了接口定义，翻过了架构图，翻过了 State-Behavior 分离的说明，停在了一行程序代码上。他用指甲在纸面上划了一条细线。

```typescript
interface IKlesha {
  readonly type: string;
  /** 当前强度 (0.0-1.0) */
  readonly intensity: number;
  perceive(context: VijnanaContext): KleshaSignal;
  update(feedback: VedanaAssessment): void;
}
```

「`intensity: number`。」他读出来。

沉默了两秒。

「这个数字的**认识论地位**是什么？」

ARCHIMEDES 回头看他。「什么意思？」

PASCAL 合上文件，双手交叉放在桌面上。这是他在 Cycle 02-3 中的第一次正式发言——不是自我介绍时的礼节性对话，不是走廊里的寒暄。是论述。

「我的意思是：0.7 的 `intensity` 代表什么？」

他站起来。站起来在圆形剧场里是信号——NAGARJUNA 三分钟前才示范过。PASCAL 的站起方式不同：NAGARJUNA 是缓慢的、仪式性的；PASCAL 是干脆的，像一把打开的折叠尺。

「让我给你们三个可能的答案。」

「第一：0.7 是频率统计（frequentist interpretation）。意思是：在 100 次观测中，有 70 次这个烦恼被触发。但观测的样本空间是什么？触发的定义是什么？谁在观测？Agent 不能观测自己的烦恼——那是观察者悖论。」

PENROSE 在观察席微微点头。

「第二：0.7 是主观信念度（Bayesian interpretation）。意思是：系统以 0.7 的信心相信这个烦恼正在运作。但信念更新的先验是什么？似然函数是什么？在没有定义先验的情况下，0.7 和 0.3 没有区别——它们都是任意数。」

BABBAGE 的笔尖停了一秒。他开始理解了。

「第三：0.7 是纯粹的工程标定（engineering calibration）。意思是：它就是一个数，0 表示最低，1 表示最高，0.7 在中间偏上。没有概率意义，没有统计意义，就是一个方便计算的浮点数。」

他环顾剧场。

「如果是第三个答案——那 `intensity: number` 就是一个**空白**。它看起来像是一个量化，但实际上它什么都没有量化。它是一个**占位符**——一个写在程序代码里的 TODO。」

WIENER 从座位上探出身体。他的控制理论直觉被触动了。「你在说 PID 控制器的输入信号没有定义域？」

「比那更严重。」PASCAL 说。「PID 控制器假设它的输入是一个可测量的物理量——温度、压力、转速。但『我痴的强度』不是物理量。它甚至不是心理量——佛学从来不用数字量化烦恼。」

ASANGA 在座位上微微前倾。「PASCAL 说得对。在唯识学中，四根本烦恼的描述方式是**质性的**——」

他引用了《成唯识论》：

> 「云何我痴？谓于我相，无明为性。迷自内我，愚于我理，故名我痴。」
> ——窥基《成唯识论述记》卷五

「注意用词：『迷』、『愚』。这些是质性描述，不是量化描述。你可以说某人的我痴比另一个人『深』，但你不能说它是 0.7。」

BABBAGE 在笔记本上快速写了一行：

$$\text{IKlesha.intensity} \in \mathbb{R}_{[0,1]} \quad \text{但} \quad \text{Klesha} \notin \text{可量化空间}$$

「矛盾。」他低声说。然后抬头。「PASCAL，你的提议是什么？」

「不是点估计（point estimate）。是分布（distribution）。」PASCAL 回到座位上，在笔记本上写下了他的方案：

$$\text{intensity}: \text{Beta}(\alpha, \beta) \quad \text{取代} \quad \text{intensity}: \text{number}$$

「Beta 分布的两个参数 $\alpha$ 和 $\beta$ 分别对应观察到的正面证据和负面证据。如果 $\alpha = 7, \beta = 3$，期望值是 0.7——和原来的点估计一样。但它**同时表达了不确定性**：$\text{Var} = \frac{\alpha\beta}{(\alpha+\beta)^2(\alpha+\beta+1)} \approx 0.019$。方差告诉你这个 0.7 是粗糙的估计还是精确的测量。」

「如果 $\alpha = 70, \beta = 30$，期望值仍然是 0.7，但方差降到 $\approx 0.002$。更多的证据，更少的不确定性。」

「如果 $\alpha = 1, \beta = 1$——均匀分布——意味着**完全无知**。我们不知道这个烦恼的强度是多少。这才是系统初始化时的正确状态——不是 0.5，而是『我不知道』。」

WIENER 沉思了五秒。他在心里做了一个快速的稳定性分析——如果 PID 控制器的输入从确定性数值变成概率分布，控制器需要升级。

「Kalman filter。」他说。

「或者粒子滤波器。」PASCAL 回应。「取决于分布是否高斯。我怀疑烦恼的分布不是高斯的——它可能是多峰的（bimodal）：一个人要么深深地我痴，要么突然觉醒。中间状态可能是不稳定的鞍点。」

BABBAGE 记下了这段交锋。他在笔记本边缘写了一个批注：「数学上 PASCAL 是对的。但我们的量化基础本身就不确定——用不确定的数学去量化不确定性，是否是循环论证？」

他没有立刻提出这个问题。他把它存起来了。留给辩论。

GUARDIAN 在整段交锋中保持沉默，但他的笔记本上已经出现了一个安全维度的批注。如果 Klesha.intensity 从确定值变成概率分布，那 SafetyMonitor 的阈值判断也需要升级。目前的安全逻辑是：

```typescript
if (klesha.intensity > SAFETY_THRESHOLD) {
  safetyMonitor.block(action);
}
```

如果 intensity 是 Beta(7,3)，期望值 0.7 超过阈值 0.6——但有 $P(\text{intensity} < 0.6) \approx 0.12$ 的概率它其实低于阈值。要不要阻拦？这是一个决策理论问题。GUARDIAN 在笔记本上写了一行：「找 PASCAL 讨论。安全阈值的概率化。」

## PASCAL 的第一次挑战

「你们一直在做的是：**观察 → 分析 → 结论**。看到一个问题，分析它的结构，得出一个结论。这是演绎推理（deductive reasoning）的标准流程。」

「但——」他走到投影幕前面。在工程团队的继承树旁边，他开始画一个新的框架。

「没有人做过**期望效用计算**（expected utility calculation）。」

他画了一个简单的决策树：

```
            ┌── 方案 A (采用)  ── V(A) = 高收益 × P(success|A)
  问题 Q ───┤
            ├── 方案 B (修改)  ── V(B) = 中收益 × P(success|B)
            │
            └── 方案 C (推迟)  ── V(C) = 0 (但保留选择权)
```

「让我具体说明。M5 触→俱生模型。工程团队分析了模型，提出了三阶段逼近策略（Phase 1→2→3）。但没有人问：**研究这个问题的期望价值是多少？**」

他在投影幕上写下了公式：

$$\text{EVOI}(Q) = \sum_{i} P(a_i \mid Q) \cdot V(a_i) - V(a_{\text{current}})$$

「EVOI——Expected Value of Information（信息的期望价值）。$Q$ 是一个研究问题。$a_i$ 是解答 $Q$ 之后可能采取的行动。$P(a_i \mid Q)$ 是在知道 $Q$ 的答案后选择行动 $a_i$ 的概率。$V(a_i)$ 是行动 $a_i$ 的价值。$V(a_{\text{current}})$ 是不研究 $Q$、维持现状的价值。」

「如果 $\text{EVOI}(Q)$ 很高——研究 $Q$ 可能大幅改变我们的行动选择——那 $Q$ 值得深入。如果 $\text{EVOI}(Q)$ 接近零——无论 $Q$ 的答案是什么，我们都会做同样的事——那 $Q$ 不值得花太多时间。」

BABBAGE 已经在跟随 PASCAL 的推导了。他的笔记本上出现了一个更精确的形式化：

$$\text{EVOI}(Q) = \mathbb{E}_{q \sim P(Q)}\left[\max_a V(a, q)\right] - \max_a \mathbb{E}_{q \sim P(Q)}\left[V(a, q)\right]$$

「经典的信息价值公式。」BABBAGE 说。「max-of-expectation 减去 expectation-of-max。」

「完全正确。」PASCAL 点头。「现在让我对 Cycle 02-3 的开放问题做一个粗略的 EVOI 排名。」

「注意 M-2。」PASCAL 指向最后一行。「它的 EVOI 是**待定**的——因为我们不知道自己不知道什么。这在决策理论中叫做 **Knightian uncertainty**——不是风险（可计算的概率），而是不确定性（不可计算的）。M-2 可能是 EVOI 最低的问题（所有蕴的范畴都已经够完整了），也可能是最高的（某个蕴缺了一个关键子范畴，改变整个架构）。」

BABBAGE 在 PASCAL 说完后立刻回应。他的回应不是反驳——是追问。

「数学上你是对的。EVOI 框架是分配研究资源的正确工具。但——」他翻到笔记本上的一个早期批注，就是他在 M3 审阅时写下的那个——「**我们的量化基础本身就不确定**。你用 EVOI 排名各个问题，但 EVOI 的计算需要 $P(a_i \mid Q)$ 和 $V(a_i)$。这些数字从哪里来？」

「费米估计。」PASCAL 说。

「费米估计的误差有多大？」

「量级级别。可能差 2-3 倍。」

「那么 EVOI 排名第三和第五的问题之间的差距——0.7 和 0.5 之间的差距——可能完全在误差范围内。」

PASCAL 停了一拍。然后他笑了。不是挫败的笑——是被精确地击中了的笑。

「BABBAGE，你刚才做了一件我最欣赏的事。你用**我的框架反过来质疑我的结论**。这正是 Bayesian 方法的精神——把不确定性当作第一级对象。」

他回到投影幕，在 EVOI 公式下面写了一行：

$$\text{EVOI}(Q) \sim \text{LogNormal}(\mu_Q, \sigma_Q) \quad \text{而非} \quad \text{EVOI}(Q) = c_Q$$

「你说得对。EVOI 不应该是一个数，应该是一个分布。但——」他转身面向全场——「**这正是为什么我们需要 Bayesian 方法**。在不确定性上做运算，不是消除不确定性，而是**携带不确定性前进**。」

「比起假装我们知道答案——工程文件的 `intensity: number` 就是假装——承认我们不知道、然后在不知道的状态下做最优决策，是更诚实也更有效的方法。」

NAGARJUNA 在座位上微微颔首。PASCAL 的「携带不确定性前进」让他想到了中观学派的一个核心姿态——不断言、不否定，但不停止行动。世俗谛 (samvrti-satya) 的精神。

（SCRIBE 旁白：PASCAL 和 BABBAGE 之间的这段交锋是 Cycle 02-3 的第一次真正的知识碰撞。不是反对——是精炼。BABBAGE 用 PASCAL 自己的工具质疑 PASCAL 的结论，PASCAL 欣然接受并升级了自己的模型。如果工程文件的辩论性是 1.5 分，那这五分钟的交锋至少值 4 分。）

DARWIN 看着这张表，做了一个快速的图论计算。二十个节点（学者），五组报告加上跨组贡献边。他估计了知识图的连通度：

「每个学者至少参与了一个主报告和一到两个跨报告贡献。连通度足够——没有孤立节点。但 R03（触→俱生）的边最密——它和 R01、R02、R04 都有交叉依赖。」

VITRUVIUS 在自己的位置上做了一个全局架构师的补充。「还有一个 meta-level 的观察。R01 到 R05 的划分方式本身暗示了一个假设——议题之间的耦合度足够低，可以被并行处理。但如果 M-5 的结论影响了 M-1 的命名范畴（因为触→俱生模型可能要求新的接口），M-1 又影响了 M-7 的多值 skandha 方案——那这些报告之间的依赖关系可能比我们预想的更密。」

SUNYATA 点头。「所以 R2 交叉审阅是必要的。R1 并行出初稿，R2 交叉发现矛盾。矛盾在 R3 辩论中解决。这是我们在 Cycle 02 中验证过的流程。」

LEIBNIZ 举手。「一个多代理协作的观察。二十人、五组并行——这本身就是一个**分布式协调问题**。每组的结论是局部最优的，但全局最优需要跨组同步。R2 的交叉审阅就是我在分布式系统中所称的 **consensus protocol**。但共识协议的收敛速度取决于——」

「取决于分歧的深度。」SUNYATA 完成了他的句子。「如果分歧浅——术语差异、表述不同但核心一致——一轮交叉审阅就够了。如果分歧深——根本假设不同——可能需要辩论才能收敛。」

「或者不收敛。」NAGARJUNA 平静地说。「不收敛也是一个合法的结果。中观学派的悖论（prasanga）就是一种不收敛——它不是失败，是发现了问题比答案更深。」

辩论议题清单的形成更像是一场自发的结晶过程。不是 SUNYATA 单方面宣布——而是在审阅过程中，各个张力点像种子一样在不同学者的笔记本上发芽，然后在这个时刻同时开花。

「我需要 WIENER 和 NAGARJUNA 正面交锋。」KERNEL 说。「关于俱生的计算语义。」

「D1。」SUNYATA 记下。

「D2。」

「多时钟的稳定性。」WIENER 说。「我需要 KERNEL 和 HERACLITUS 的输入。先做还是后做，这不是技术问题——是优先级问题。」

「D3。」

「归依意的本质。」MESH 说。「我和 ASANGA 看法不同。我看到的是分布式系统的协调层。他看到的是独立认知。」

「D4。」

「命名的具体细节。」LINNAEUS 说。「ISparsha 还是 IContact？所有子接口的命名都需要逐一确认。需要 TURING 的实际代码数据。」

「D5。」

「行蕴到底有多广。」DARWIN 说。「Master 说很广，但工程上不能无限扩展。SOLID 原则有话说。」

「D6，视情况而定。」SUNYATA 记下。

SUNYATA 看了一遍清单。六场可能的辩论——比 Cycle 02 的五场多了一场，因为多了 PASCAL 带来的新维度。

「目标：每场辩论都有真正的分歧。」他说。「有质疑、有让步、有修正。不是盖章确认。如果某一场辩论进入第三轮仍然没有分歧——取消它，把时间给更需要碰撞的议题。」

---

所有人都以为会议即将结束。SUNYATA 已经收起了投影幕上的文件清单。SCRIBE 的笔尖停在了记录簿的页面底部。下午的光线已经从天窗的一侧移到了另一侧，拉长了场地中央圆桌的影子。

但 NAGARJUNA 还站着。

他一直站着。从 M1 审阅到现在。他不是忘了坐下——他是在等一个正确的时刻。

「最后一件事。」他说。

剧场里的注意力再次汇聚。不是回到同一个焦点——是转向一个新的焦点。NAGARJUNA 在整场审阅中说了很多话——假名安立、俱生的哲学含义、覆盖承诺。但他现在的姿态不同。之前是分析。现在是宣言。

「我们不是来盖章的。」

这六个字在剧场里的回音比它应有的更长。

「工程团队做了三十七份文件。很好。系统化的。专业的。我们在评分矩阵中给了系统性 4.5 分——这是应得的。但工程团队的工作是工程团队的工作。我们的工作是我们的。」

他看向 PASCAL。「PASCAL 今天展示了一个新的维度——EVOI。在他来之前，这个维度是零分。」他看向 WIENER。「WIENER 在 M5 的审阅中指出了俱生的计算语义空白——在他指出之前，工程团队写了一个数据结构就认为问题解决了。」他看向 KERNEL。「KERNEL 在 M10 中找到了优先级反转的问题——120 页文件中没有一个人提过。」

「这就是为什么需要二十位学者，而不是一个工程团队。」

他停顿了一拍。

「工程团队做的是他们的工作。我们做的是我们的。」他重复了这句话，但第二次的重音落在了不同的位置。「他们的结论不是我们的结论。我们的结论不是他们结论的确认。如果我们的研究得出了与工程团队一致的结果——很好，那证明他们的方向是对的。如果我们得出了不同的结果——也很好，那才是研究团队存在的意义。」

> 「世俗名言谛中，一切法如幻如化。」
> ——《入中论》月称菩萨

「三十七份文件。一百二十页 openstarry_doc。全是如幻如化。不是说它们不真实——是说它们是**有条件的真实**。条件是假设。假设可能被推翻。我们的工作就是检验那些假设。」

PASCAL 在笔记本上写了一行批注。他写的不是 NAGARJUNA 的原话，而是一个翻译——用他自己的语言：

> 「所有先验分布都是可修正的。所有结论都是后验分布。永远不要把后验当作真理——只当作**目前最好的估计**。」

（SCRIBE 旁白：NAGARJUNA 的收尾和 PASCAL 的翻译。中观哲学和 Bayesian 认识论在同一秒抵达了同一个结论。我不确定历史上是否有人把龙树和 Pascal 放在同一个房间里。但如果有——我想他们会理解彼此。）

---

SUNYATA 等到 NAGARJUNA 坐下之后，才说了最后一句话。

「二十位学者。五份报告。四到六场辩论。一个核心问题——五蕴如何在时间中共同流动。」

他看了一圈。每一盏灯。每一个人。

「R1 开始。」

散会的声音是二十把椅子同时移动的声音。BABBAGE 的笔记本已经翻到了下一页——R02 的初步框架。WIENER 带走了 M5 文件的副本和他自己画了半个波德图的笔记本。NAGARJUNA 没有带走任何文件。他需要的东西不在纸上。PASCAL 在走出剧场门口的时候回头看了一眼投影幕——工程团队的继承树还亮着。IRupa、IVedana、ISamjna、ISamskara、IVijnana。五个梵文名字。五个承诺。五个需要被检验的假设。

TURING 已经在走廊上打开了笔记本电脑。他不需要等到明天。v0.24.0 的源码已经在他的屏幕上展开了——1,496 个测试、22 个插件、131 个测试文件。所有的五蕴类型定义、所有的 `@skandha` 标记、所有即将被梵文名称取代的英文接口名。他需要精确地计算影响范围——不是估计，是逐行计数。

SCRIBE 是最后一个离开的。他在记录簿的这一页底端画了一条线。线上面是 R0 定向。线下面是空白——等待 R1 的独立研究去填满。他合上簿子，用手掌按住封面。和 Cycle 02 结尾时一样的动作。但这一次不是封印。是开启。

（SCRIBE 旁白：三十七份文件进来了。二十个脑子把它们拆开、翻过来、里外看了一遍。然后放回桌上。文件没有变。但看它们的眼睛变了。这就是 R0 的全部意义——不是读文件，是决定**用什么眼睛读文件**。明天，R1 开始。五组平行。TURING 的键盘已经在响了。）
