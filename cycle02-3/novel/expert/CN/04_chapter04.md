# 第四章：五遍行

---

ASANGA 在辩论开始之前做了一件不寻常的事。

他闭上了眼睛。

不是冥想——他不会在圆形剧场里正式冥想，那是属于个人修行的时间，不是属于学术辩论的时间。他闭上眼睛的方式更像一个人在打开一个记忆深处的抽屉：先让外部的光线退场，然后让内部的文字浮现。

他要引用的经文已经在他脑中停留了一千五百年。不是他本人的一千五百年——而是那些文字被传诵、被抄写、被翻译、被辩论的一千五百年。每一次引用，都是在接住一条从无著菩萨手中抛出的线。

他睁开眼睛。

「我们在第一场辩论中确立了双层双齿轮架构。CoarisingBundle 在 vedana-clock 的速度上原子地计算。」他的声音稳定，像河床上的大石。「但那场辩论留了一个问题没有回答。」

他看向 SUNYATA。

「Bundle 里到底放什么。」

---

> *SCRIBE 旁白：Debate 2 的核心冲突可以用一个数字概括：3 vs 5。R01 的研究报告定义了三个 CoarisingBundle 组件——受 (vedana)、想 (samjna)、思 (cetana)。R03 提出五个——加上触 (sparsha) 和作意 (manasikara)，完全依照瑜伽行派的五遍行 (panca sarvatraga) 教义。差两个。差两个元素听起来不多。但在亚毫秒的计算 bundle 中，每一个元素都有成本。问题不是「经典怎么说」——问题是「工程能不能负担得起经典说的」。*

---

SUNYATA 站在剧场中央。

「Debate 2。CoarisingBundle 的组成。三遍行，还是五遍行。」

他的目光扫过圆形剧场。左侧，ASANGA 和 NAGARJUNA 并排而坐，中间隔着一个座位的距离——足够让两种佛学传统各自呼吸。右侧，KERNEL 的手指已经在类比卡片上翻动了。后排，PASCAL 的笔记本摊开在膝盖上，一张贝叶斯计算表的草稿占据了整页。

「ASANGA。教义论证先行。」

---

## I

ASANGA 站了起来。

他站起来的方式在三场辩论中都不曾改变——像一座山在晨雾中显现轮廓。缓慢的，不可忽视的。但今天他手里多了一样东西：一页纸。不是打印的学术论文，而是手抄的——他自己用蓝色钢笔抄写的梵文偈颂，字迹整齐得像寺院里的写经生。

他把那页纸举起来，让全场都能看到。

「五遍行。」他说。「*Panca sarvatraga*。梵文 *sarva* 意为『一切』，*traga* 源自动词 *tṛ*，意为『行、到达、遍及』。Sarvatraga——到达一切处的。这五个心所伴随每一个认知瞬间 (*citta-kṣaṇa*)，无一例外。」

他走到白板前，用黑色马克笔写下五个名字。每写一个，他都停顿半秒——不是迟疑，是庄重。是在给每一个被书写了一千五百年的名字应有的重量。

```
五遍行 (Pañca Sarvatraga)
═══════════════════════════════════════════════
 1. 触  sparśa     — 根・境・识三者和合
 2. 受  vedanā     — 苦・乐・舍三领纳
 3. 想  saṃjñā     — 取相辨识
 4. 思  cetanā     — 造作驱动
 5. 作意 manasikāra — 心之警觉，引导趋向所缘
═══════════════════════════════════════════════
```

「我要引用两个根本文献。」他把那页手抄纸放在白板旁边。「第一个是世亲菩萨的《五蕴论》——*Pañcaskandhaprakaraṇa*——唯识学的基础文献之一。」

他的声音在引用经文时微微放慢了——不是刻意的庄严，而是一种对原典精确度的尊重。每一个字都被安放在它应有的位置上，像拼图的最后几块。

> 「云何触？谓三和合分别为性。三和合者，谓根境识更相会也。」
> ——世亲菩萨《大乘五蕴论》(*Pañcaskandhaprakaraṇa*)

「触。*Sparśa*。不是物理接触——是根 (indriya)、境 (viṣaya)、识 (vijñāna) 三者的和合。在 OpenStarry 的语言中，这就是 SparshEvent：IListener 通道（根）接收 InputEvent（境），在 session context（识）中和合，触是这三者汇聚的瞬间。」

他的手指从触移到作意。

「第二个根本文献。《大乘阿毗达磨集论》——无著菩萨的论典。也就是我名字的来源。」

> 「云何作意？谓令心发悟为性。于所缘境引心趋向为业。」
> ——无著菩萨《大乘阿毗达磨集论》(*Abhidharmasamuccaya*)

「作意。*Manasikāra*。字面意思是『使意 (manas) 作用 (kāra)』。它的功能是『令心发悟』——让心从散乱中警觉过来——然后『引心趋向所缘境』——引导注意力指向特定的对象。」

他停顿了一秒。然后他说了这场辩论中第一句带锋芒的话。

「R01 的 CoarisingBundle 只有三个元素：受、想、思。完全遗漏了触和作意。」

他看向全场。

「缺乏任何一个遍行的认知瞬间，不是认知瞬间。在阿毗达磨的严格定义中，五遍行伴随一切心，无一例外。这是定义性的 (definitional)，不是经验性的 (empirical)。就像你不能有一个只有两边的三角形——不是因为你没见过，而是因为三角形的定义要求三边。五遍行的同时在场是认知瞬间的定义要求。」

他用手指在五个名字上画了一个圈。

「缺一不可。」

---

NAGARJUNA 在 ASANGA 画完那个圈的瞬间就开口了。

他没有站起来。中观哲学家在发出警告的时候从不站起来——警告的力量不在姿态上，在锐度上。他的声音从座位上升起，锋利但不尖锐，像被千年河水打磨过的石头。

「ASANGA，你在将心所分类法物化 (reify)。」

剧场的空气收紧了。

「你说『缺一不可』。你引用了《五蕴论》和《阿毗达磨集论》。这些是伟大的论典。但论典的功能是什么？论典的功能是教学 (*deśanā*)——帮助修行者理解心的运作方式。论典不是存在论清单 (*ontological inventory*)。」

他停了一拍。然后用更精确的语言展开。

「五遍行是**教学工具** (*upāya*)。不是本体论宣言。它们是对心所运作方式的一种系统化描述——一种特别精确、特别有用的描述——但描述不等于事物本身。地图不等于地形。」

> 「若人见般若，是则为被缚；若不见般若，是亦为被缚。」
> ——龙树菩萨《中论》观涅槃品第二十五

「如果我们因为论典说了五个就放五个，我们是在做佛学考古，不是做工程。」

他看向 ASANGA。不是对抗——是补正。像两面镜子互相照亮对方的盲区。

「我的立场不是『不放五个』。我的立场是：如果要放五个，理由必须是功能性的 (functional)，不是教条性的 (doctrinal)。功能性论证比教条性论证更可靠。因为功能性论证可以被工程验证。教条性论证只能被权威认可。」

---

> *SCRIBE 旁白：NAGARJUNA 和 ASANGA 的分歧在 Cycle 02-2 中就出现过。在第三章「部分与整体」中，NAGARJUNA 对 IVijnana 的命名选择做了四句否定——不是因为他否认识蕴的复杂性，而是因为他坚持名字必须是「最不会导致误解的假名」。现在，在 Debate 2 中，同样的方法论分歧再次浮现：ASANGA 从教义出发，NAGARJUNA 要求功能性理由。*

> *但注意——NAGARJUNA 没有说「不放五个」。他说的是「理由必须是功能性的」。这是一个方法论的挑战，不是结论的否定。他在控制论证的品质，不是论证的方向。*

---

## II

KERNEL 在 NAGARJUNA 说完之后站了起来。

他的手里拿着三张卡片——那些左边写着操作系统概念、右边写着 OpenStarry 映射的类比卡片。但这一次，他没有翻卡片。他走到白板旁边，在 ASANGA 的五遍行列表下方画了一张时序图。

「让我问一个工程问题。」他说。声音低沉稳定，像在 kernel space 里持续运行的守护进程。「我不是在质疑佛学。我是在质疑时间。」

他在白板上写了两行数字：

```
Layer 1 计算预算：< 1ms
五遍行中各元素的估计成本：
  sparsha  (触)   — SparshEvent 构造       ~0.01ms  ✓
  vedana   (受)   — ChannelVedana 计算     ~0.1ms   ✓
  samjna   (想)   — ChannelSamjna 匹配     ~0.05ms  ✓
  cetana   (思)   — ChannelCetana 判断     ~0.05ms  ✓
  manasikara (作意) — ???                   ???      ?
```

「前四个没有问题。」他用手指点了点前四行的打勾。「SparshEvent 是一个 struct 的构造——三个 readonly 字段，几乎零成本。ChannelVedana 是一个数值映射——valence 和 intensity 从传感器读数直接计算。ChannelSamjna 是规则匹配——在 VasanaEngine 的 fast path 中，这是一个 hash table lookup。ChannelCetana 是趋避判断——从 valence 的正负号直接导出。」

他的手指停在第五行。

「作意。*Manasikāra*。注意力的方向。」

他转向 ASANGA。

「ASANGA，你说作意的功能是『引心趋向所缘境』。在 OpenStarry 的架构中，注意力的方向由谁决定？」

ASANGA 回答：「IGuide。system prompt 定义了 Agent 应该关注什么。」

KERNEL 点头。「IGuide。在 Cycle 02-2 中，IGuide 被归入识蕴——IVijnana 的子接口。它的主要方法是 `getSystemPrompt()`。而 `getSystemPrompt()` 的调用——」

他在白板上画了一条时间线：

```
时钟域分析：IGuide.getSystemPrompt()
═══════════════════════════════════════════════════════
  IGuide 属于 vijnana-clock (1~5ms)
  getSystemPrompt() 涉及：
    - 读取 system prompt 配置          ~0.1ms
    - 上下文组装 (context assembly)    ~0.5ms
    - 可能的动态 prompt 生成           ~1-5ms
  ─────────────────────────────────
  总计：1~5ms，属于 vijnana-clock 操作

  Layer 1 属于 vedana-clock (10~100ms per tick, <1ms 计算)
```

「Layer 1 必须在 1ms 以内完成。」KERNEL 的声音加重了——不是情绪化的加重，是工程约束的加重。「如果 manasikara 需要调用 `IGuide.getSystemPrompt()`，那是一个 vijnana-clock 的操作——1 到 5 毫秒。你不能把一个 5ms 的操作放在一个亚毫秒的 bundle 里。」

他在问号后面写了一个大叉。

「如果 manasikara = IGuide.getSystemPrompt()，成本是 1~5ms。超出 Layer 1 的预算。不可行。」

他看向全场。

「这就是三元素 vs 五元素辩论的工程核心。不是佛学对不对的问题。是时间够不够的问题。」

---

ARCHIMEDES 在 KERNEL 写下那个大叉之后立刻站了起来。

他站起来的速度比平常快——不是急切，而是「工程解决方案已经成形，等不及要出口」的那种速度。他走到 KERNEL 的时序图旁边，拿起绿色马克笔——绿色在他的标记系统中代表「解决方案」。

「KERNEL，你的分析有一个隐含假设。」他说。声音稳定如砖，每一个字都砌在正确的位置上。

他在 KERNEL 的大叉旁边画了一个箭头，指向一个新的方框：

```
KERNEL 的假设：manasikara = IGuide.getSystemPrompt()
                            ↓
                      动态计算（1~5ms）

ARCHIMEDES 的替代：manasikara = 快照（snapshot）
                            ↓
                      读取预计算结果（~0.01ms）
```

「ChannelManasikara 不是计算。ChannelManasikara 是**快照**。」

他把这个词说得很重。不是语气上的加重——是工程语义上的加重。在并行系统设计中，snapshot 和 computation 是两个完全不同的操作类别。

「让我解释 snapshot 模式的工作原理。」

他在白板上画了一张跨时钟域的数据流图：

```
跨时钟域快照机制
═══════════════════════════════════════════════════════

  vijnana-clock (1~5ms)              vedana-clock (10~100ms)
  ┌──────────────────┐              ┌──────────────────────┐
  │ IGuide           │  写入        │ CoarisingBundle      │
  │  .getSystemPrompt│──────→ ┌───┐ │  .manasikara         │
  │  .computeAttention│       │快照│←──── 读取              │
  │                  │──────→ │内存│ │    (readonly access)  │
  │ 预计算注意力状态  │       │体  │ │                      │
  └──────────────────┘       └───┘ └──────────────────────┘
                                ↑
                           SharedAttentionState
                           (lock-free atomic read)

  写入成本：由 vijnana-clock 承担（不占 Layer 1 预算）
  读取成本：~0.01ms（单次 atomic read，可忽略）
```

「vijnana-clock 负责计算。它在自己的时钟域里运行 IGuide 的逻辑——读取 system prompt、分析上下文、决定注意力方向——然后把结果写入一块共享内存。vedana-clock 的 Layer 1 只做一件事：从那块共享内存中读取最新的快照。一次 atomic read。0.01 毫秒。」

他转向 KERNEL。

「你担心的 5ms 成本，是计算的成本。我说的 0.01ms，是读取的成本。ChannelManasikara 不做计算——它读取计算的结果。就像你不会在中断处理程序里计算 FFT——你在后台线程里算好，然后在 ISR 里读取结果。」

他在白板上写下 TypeScript 接口：

```typescript
/**
 * ChannelManasikara — 作意的通道级快照。
 *
 * 不是计算——是 vijnana-clock 预计算结果的 readonly 读取。
 * 成本：~0.01ms (atomic read from SharedAttentionState)。
 *
 * @sanskrit manasikāra — 「令心发悟，引心趋向所缘境」
 * @clock-domain vedana-clock (读取者) / vijnana-clock (写入者)
 */
interface ChannelManasikara {
  readonly focus: string;       // 此通道正在注意什么
  readonly intensity: number;   // 注意力强度 (0.0 ~ 1.0)
  readonly source: 'guide' | 'context' | 'default';
  readonly stale: boolean;      // 快照是否过期
}
```

「四个字段。全部 `readonly`。`focus` 说明注意力的方向——这个通道正在关注什么。`intensity` 是注意力的强度——0.0 表示完全不关注，1.0 表示最高警觉。`source` 标记快照的来源——来自 IGuide 的配置、来自上下文推断、或者使用默认值。`stale` 标记快照是否过期——如果 vijnana-clock 的最后一次更新距今超过了 staleness bound，这个标志为 true。」

他退后一步。

「工程成本：0.01 毫秒。远低于 Layer 1 的 1ms 预算。」

他看向 KERNEL 白板上的那个大叉。

「可以把那个叉换成勾了。」

---

KERNEL 看了 ARCHIMEDES 的跨时钟域图整整五秒。

在操作系统专家的世界里，跨时钟域的数据共享是一个他每天都在处理的问题。CPU 核心之间的 cache coherence。PCI Express 设备和主机之间的 MMIO（Memory-Mapped I/O）。中断控制器和 CPU 之间的中断状态寄存器。所有这些场景都遵循同一个模式：一个写入者在自己的时钟域里更新状态，另一个读取者在自己的时钟域里读取最新值。

「lock-free atomic read。」他说。没有反对。只有确认。「在 x86 上，一个对齐的 64-bit 读取本身就是原子的。不需要锁。不需要 CAS（Compare-And-Swap）。只要共享内存的地址是 8-byte 对齐的，读取操作天然是原子的。」

他在 ARCHIMEDES 的图旁边加了一行小字：

```
硬件保证：x86 aligned 64-bit load is atomic (Intel SDM Vol. 3A, §8.1.1)
TypeScript 等效：SharedArrayBuffer + Atomics.load()
实际成本：单次内存读取，~10ns = 0.00001ms
```

「0.01ms 是保守估计。」他说。「实际上是纳秒级。」

他拿起那张卡片——右侧一直是空白的那张——在右侧写了一行：

左边：`MMIO snapshot read (cross-clock domain)`
右边：`ChannelManasikara (vijnana→vedana snapshot)`

他把大叉擦掉了。换成了一个勾。

---

> *SCRIBE 旁白：ARCHIMEDES 的快照解决方案让整场辩论的重心发生了位移。在他发言之前，辩论的轴线是「佛学教义 vs 工程预算」——ASANGA 站在教义一侧，KERNEL 站在预算一侧。在他发言之后，轴线消失了。不是因为某一方赢了——而是因为第三条路出现了：快照。计算成本由 vijnana-clock 承担，Layer 1 只承担读取成本。教义的要求和工程的约束不再矛盾。*

> *我注意到 KERNEL 擦掉大叉的动作只用了一秒。在 OS 专家的思维模型中，一旦硬件事实被确认（aligned atomic read 的成本），结论就是确定的。没有情感性的抵抗。只有事实性的更新。*

---

## III

PASCAL 在 ARCHIMEDES 的快照方案被 KERNEL 确认之后站了起来。

他是 Cycle 02-3 新加入的成员——决策理论与概率哲学专家。他的出现让研究团队从 19 人扩展到了 20 人。但他的声音已经在前三场辩论中建立了自己的位置：每当辩论陷入「应不应该」的定性判断时，他会站起来，把问题转化为「期望效用是多少」的定量计算。

「让我把辩论框架从定性转为定量。」他说。声音带着一种概率论者特有的冷静——不是冷漠，而是一种「在做决策之前先计算所有可能结局」的纪律。

他在笔记本上写下两个选项，然后把笔记本举起来让全场看到：

```
决策矩阵
═══════════════════════════════════════════════════════
                        需要 manasikara     不需要 manasikara
                        (P = p)             (P = 1-p)
───────────────────────────────────────────────────────
包含 manasikara         成本: 0.01ms        成本: 0.01ms
(Action A: include)     收益: 一致性 V       收益: 0（浪费）
───────────────────────────────────────────────────────
不包含 manasikara       成本: 0             成本: 0
(Action B: exclude)     损失: 不一致性 -L    收益: 0（正确）
═══════════════════════════════════════════════════════
```

「两个行动选项：包含或不包含。两种自然状态：下游系统需要 manasikara 信息，或者不需要。四个象限。」

他开始计算期望效用。他的字迹在白板上展开，像一套精密的齿轮互相咬合：

$$EU(\text{include}) = p \cdot V + (1-p) \cdot (-0.01\text{ms})$$

$$EU(\text{exclude}) = p \cdot (-L) + (1-p) \cdot 0$$

「$V$ 是一致性的价值——CoarisingBundle 的消费者（ManoAggregator、VasanaEngine、VitakkaEngine、IVolition）能够知道每个 bundle 的注意力方向。$L$ 是不一致性的损失——下游消费者需要注意力信息但找不到。」

他停了一秒，然后说出了这场辩论中最尖锐的一句话。

「$L$ 的上界是什么？」

他看向 KERNEL。

「如果 ManoAggregator 在汇总多个 root-gate 的 bundle 时，不知道每个 bundle 的注意力方向——不知道 Agent 正在关注哪个通道——它怎么决定权重？答案是：它不能。它要么平均加权（丢失注意力信息），要么随机加权（引入不确定性），要么硬编码加权（丧失适应性）。三种退化方案，每一种都会向下游传播。」

他在白板上画了一条因果箭头：

```
manasikara 缺失
  → ManoAggregator 权重退化
    → VasanaEngine 匹配品质下降
      → Gear 1/Gear 2 切换判断不准确
        → Agent 行为品质退化
          → ??? (无界)
```

「损失沿着因果链传播。而因果链的长度决定了损失的放大倍数。每一层都可能放大上游的误差。在最坏情况下——如果 VasanaEngine 因为缺少注意力信息而在不应该切换到 Gear 2 的时候切换了——损失是一次不必要的 LLM 调用：500ms 到 30 秒。如果 VasanaEngine 因为缺少注意力信息而在应该切换到 Gear 2 的时候没有切换——损失是一次错误的自动化决策。这个损失的上界是不确定的。」

他在 $L$ 的旁边加了一个向上的箭头和一个问号：$L \to ?$

「这是 Pascal's Wager 的工程版。」他把笔记本放下，直接看向全场。「Blaise Pascal 在 1670 年的《沉思录》(*Pensees*) 中提出了一个论证：如果上帝存在，信仰的收益是无限的；如果上帝不存在，信仰的成本是有限的。因此，即使上帝存在的概率很小，期望效用计算也支持信仰。」

「我不是在做神学论证。」他补充——NAGARJUNA 已经皱了一下眉头。「我是在做风险分析。结构完全相同：」

$$EU(\text{include}) = -0.01\text{ms} + p \cdot V$$
$$EU(\text{exclude}) = 0 + p \cdot (-L)$$

$$EU(\text{include}) - EU(\text{exclude}) = -0.01\text{ms} + p \cdot (V + L)$$

「只要 $p \cdot (V + L) > 0.01\text{ms}$，包含就是严格占优策略 (strictly dominant strategy)。而 ARCHIMEDES 刚才告诉我们成本是 0.01ms。KERNEL 告诉我们实际成本是纳秒级。」

他把最终结论写在白板上，用了一个框：

```
┌────────────────────────────────────────────────┐
│  包含 manasikara 的成本：0.01ms (确定)          │
│  不包含 manasikara 的损失：无界 (不确定)         │
│                                                │
│  结论：包含是严格占优策略。                      │
│  即使 P(needed) = 0.01，                       │
│  只要 L > 1ms，期望效用已为正。                  │
└────────────────────────────────────────────────┘
```

「当遗漏的损失可能无限大，而包含的成本几乎为零时，即使需要的概率很小，也必须包含。这不是信仰。这是风险管理。」

他点头的方式很有意思——不是「我同意你的结论」的点头，而是「你用了正确的方法论」的点头。在 Debate 2 开始时，他要求的是功能性论证而非教条性论证。PASCAL 刚才做的——期望效用计算、风险分析、严格占优策略——正是功能性论证的一种。不是因为经典说了五个所以放五个。是因为不放的损失大于放的成本。

「功能性论证不是独立的。它依赖工程事实。这正是我要求功能性论证而非教条性论证的原因——功能性论证可以被工程事实验证或否决。教条性论证不能。」

> *SCRIBE 旁白：PASCAL 是 Cycle 02-3 的新成员，但他在 Debate 2 中的贡献让他立刻成为了辩论的结构性支柱。他做的事情很特殊：他不是在佛学和工程之间二选一——他是在两者之上建立了一个决策理论的框架，让辩论从「对不对」转向了「值不值得」。期望效用计算、严格占优策略、Pascal's Wager 的工程化——这些都不是修辞。它们是数学。而数学不选边站。*

## IV

LINNAEUS 在整场辩论中一直安静地坐在白板旁边。

他的笔盖一直没有拔开。在 Cycle 02-2 的第三章中，他等到所有人的论述都结束之后才开始画分类树。现在他在等同样的时刻——所有的论证到齐，结构定型，然后他开始分类。

但在那个时刻到来之前，他必须先处理一件他自己的事。

「我最初支持三元素。」他说。站起来的方式带着分类学家的审慎——每一个动作都先在脑中运行过一遍，确认不会破坏已有的分类框架。

「三元素。受、想、思。简洁。互斥。诊断特征充分。在分类学的语言里，这是一个最简分类群 (minimal taxon)——用最少的元素覆盖必要的语义空间。奥卡姆剃刀：如无必要，勿增实体。三个元素已经涵盖了认知瞬间的核心功能——感受、辨识、驱动。为什么要加第四个和第五个？」

他看了看 ASANGA 的五遍行列表。然后看了看 ARCHIMEDES 的快照方案。然后看了看 PASCAL 的决策矩阵。

「ARCHIMEDES 改变了成本函数。」他说。「在他的快照方案之前，manasikara 的成本是 1~5ms——不可接受。在他的方案之后，成本是 0.01ms——可以忽略。」

他的声音带着分类学家特有的那种坦率——承认错误不是痛苦的事，是校准的事。在分类学的历史中，每一次重大修订都始于某人说「我之前的分类不完整」。林奈本人在《自然系统》的十二版中持续修订了自己的分类。修订是分类学的常态，不是例外。

「我接受功能性论证。快照机制让第五个元素的成本可以忽略。」

他停了一秒。然后加了一个但是。

「但我要坚持一点。」

他拔开了笔盖。蓝色——结构的颜色。

「CoarisingBundle 的五个元素不是平权的。它们在 bundle 中扮演不同的角色。让我用分类学的术语做一个形式化的区分。」

他在白板上画了一张表格：

```
CoarisingBundle 元素的分类学角色
═══════════════════════════════════════════════════════════════
  类型           元素           角色             分类学术语
───────────────────────────────────────────────────────────────
  触发条件       sparsha        触发 bundle      relational
  (relational    manasikara     上下文元数据      relational
   property)

  构成内容       vedana         感受信号         constitutive
  (constitutive  samjna         辨识结果         constitutive
   property)     cetana         驱动判断         constitutive
═══════════════════════════════════════════════════════════════
```

「在分类学中——不管是林奈的形态分类还是 Hennig 的支序分类——我们区分两种属性。Constitutive properties（构成性属性）是定义物种本质的属性——它的形态、它的基因序列、它的代谢方式。Relational properties（关系性属性）是描述物种和外界关系的属性——它的栖息地、它的共生者、它的食物链位置。」

「在 CoarisingBundle 中，vedana、samjna、cetana 是构成性属性——它们定义了 bundle 的认知内容。sparsha 是触发条件——它说明了 bundle 是在什么接触中产生的。manasikara 是上下文元数据——它说明了 bundle 被计算时的注意力状态。」

他在表格下方加了一行结论：

「sparsha 和 manasikara 在 bundle 中的角色是元数据 (metadata)，不是主要计算 (primary computation)。它们告诉你 bundle 是怎么来的、在什么条件下产生的。它们不告诉你 bundle 包含什么认知内容。」

他看向 ASANGA。

「五个元素都应该在场。但不是五个元素都做同样的事。分类学的职责是标记差异，即使在同一个群体内部。」

---

BABBAGE 在 LINNAEUS 的表格旁边补了一行形式化。他一直在黑板上默默记录——三场辩论已经让他学会了一种更安静的参与方式：先听完所有人，然后用一行数学把所有人的意思压缩成符号。

$$\text{CoarisingBundle} = \underbrace{(\text{sparsha}, \text{manasikara})}_{\text{relational (context)}} \times \underbrace{(\text{vedana}, \text{samjna}, \text{cetana})}_{\text{constitutive (content)}}$$

「笛卡尔积。」他说。「上下文和内容的正交组合。同一个认知内容 (vedana, samjna, cetana) 在不同的上下文 (sparsha, manasikara) 中可能有不同的意义。LINNAEUS 的 relational / constitutive 区分在类型系统中的对应是——上下文不影响计算，但影响解读。」

他看向 LINNAEUS。微微点头。

---

> *SCRIBE 旁白：LINNAEUS 的让步是有条件的。他不是说「好吧你们赢了」——他是说「我接受增加两个元素，但我要标记它们和原来三个的区别」。这是分类学家的本能：即使在同意的时候也要分类。BABBAGE 的笛卡尔积把这个区别形式化了——relational 和 constitutive 是正交的两个维度。从此 CoarisingBundle 不再是五个平权元素的列表，而是一个有结构的二维空间。*

```typescript
/**
 * CoarisingBundle — 俱生心所束。
 *
 * 触(sparsha)发生时，受想思作意作为一个整体浮现。
 * 「你不可能有没有想的受」— Master
 *
 * 五遍行 (pañca sarvatraga):
 *   1. sparsha    — 触：根・境・识三者和合
 *   2. vedana     — 受：苦・乐・舍三领纳
 *   3. samjna     — 想：取相辨识
 *   4. cetana     — 思：造作驱动
 *   5. manasikara — 作意：注意力快照 (vijnana-clock snapshot)
 *
 * @see Pañcaskandhaprakaraṇa (Vasubandhu)
 * @see Abhidharmasamuccaya (Asanga)
 * @clock-domain vedana-clock (Layer 1, <1ms)
 */
interface CoarisingBundle {
  // ═══ Relational properties (触发条件 + 注意力上下文) ═══
  readonly sparsha: SparshEvent;              // 遍行 1: 触
  readonly manasikara: ChannelManasikara;     // 遍行 5: 作意 (snapshot)

  // ═══ Constitutive properties (认知内容) ═══
  readonly vedana: ChannelVedana;             // 遍行 2: 受
  readonly samjna: ChannelSamjna;             // 遍行 3: 想
  readonly cetana: ChannelCetana;             // 遍行 4: 思

  // ═══ Bundle metadata ═══
  readonly layer: 1 | 2;
  readonly mode: 'fast' | 'slow';
  readonly sahaja: SahajaContract;
  readonly timestamp: number;
}
```

TURING 停了一秒。然后他切换到 SahajaContract 的定义：

```typescript
/**
 * SahajaContract — 俱生合约。
 *
 * 保证 CoarisingBundle 内部的一致性。
 * 「受想思作为一个整体浮现」的类型系统保障。
 *
 * @sanskrit sahaja — 「俱生」，同时生起
 */
interface SahajaContract {
  readonly mutualConsistency: boolean;   // 五遍行是否从同一次触计算
  readonly atomicPublication: boolean;   // bundle 是否原子发布（不可部分读取）
  readonly stalenessUpperBound: number;  // 快照过期上限 (ms)
  readonly computationCost: number;      // 实际计算耗时 (ms)
}
```

「SahajaContract 是一个自描述的合约。」TURING 的声音冷静如常。「`mutualConsistency` 保证五个遍行元素是从同一次触事件中计算出来的——不是拼凑的。`atomicPublication` 保证 bundle 被原子发布——消费者不可能读到一个只有 vedana 没有 samjna 的半成品。`stalenessUpperBound` 是 WIENER 在第一场辩论中确立的稳定性准则——快照过期不得超过 $0.29 \times T_{\text{outer}}$。`computationCost` 记录了实际的计算耗时，用于运行时性能监控。」

他看向 BABBAGE。

「SahajaContract 在类型系统中的角色，是把一个哲学承诺（俱生）转化为一个可验证的工程合约。消费者可以检查 `sahaja.mutualConsistency === true` 来确认 bundle 的完整性。类型系统强制所有字段为 `readonly`——构造之后不可修改。」

BABBAGE 在黑板上写了一行：

$$\text{SahajaContract} \vdash \text{CoarisingBundle} \text{ is consistent} \iff \text{mutualConsistency} \wedge \text{atomicPublication}$$

「形式化：SahajaContract 证明 CoarisingBundle 的一致性，当且仅当 `mutualConsistency` 和 `atomicPublication` 同时为真。」

WIENER 在方格纸上快速验证了 staleness bound。

「确认。」他说。没有多余的话。「$\rho = \frac{\Delta t}{T_{\text{outer}}} < 0.29$。如果 vijnana-clock 的更新周期是 5ms，vedana-clock 的 tick 是 50ms，那么 manasikara 快照的最大过期时间是 $0.29 \times 50 = 14.5\text{ms}$。vijnana-clock 每 5ms 更新一次。$5 < 14.5$。稳定性准则满足。相位裕度 $>45°$。」

他在方格纸上画了一个小勾。然后在旁边写了一行：

>
> 「缘眼与色，生眼识。三者和合为触。触缘受。」
> ——SN 12.2 (*Paticca-samuppada Vibhanga Sutta*)

「这段经文描述的正是我们的 CoarisingBundle 的架构。」他的手指在空气中依次点过。

「『缘眼与色，生眼识』——根 (cakkhu, IListener) 接触境 (rūpe, InputEvent)，生识 (cakkhuviññāṇaṃ, session context)。这是 SparshEvent 的三个字段：root、object、consciousness。」

>
> 「所受者即想，所想者即寻思。」
> ——MN 18 (*Madhupiṇḍika Sutta*, 蜜丸经)

「所受者即想——凡是被感受的，就是被辨识的。所想者即寻思——凡是被辨识的，就是被思量的。受→想→思。开回路。有顺序性。但在时间尺度上——在一个认知瞬间的尺度上——它们是俱生的。就像 PENROSE 在第一场辩论中说的：亚毫秒的计算，在 vedana-clock 的解析度下，看起来是同时的。粗粒化 (coarse-graining)。」

他退后一步，看着白板上 ARCHIMEDES 的架构图和 TURING 的 TypeScript 接口。

「SN 12.2 描述的是二千五百年前的教义。CoarisingBundle 是二零二六年的 TypeScript 接口。它们说的是同一件事：触生受，受伴想，想伴思，作意贯穿始终。五遍行。」

他的声音带着一种辩论即将结束时特有的清晰度——不是疲惫后的放松，而是所有论证都已到齐后的精确总结。

停顿。

「我们放五个，是因为每个都有独立的工程功能。sparsha 是触发条件——没有它，bundle 不知道自己从哪里来。vedana 是感受信号——没有它，下游不知道痛不痛。samjna 是辨识结果——没有它，下游不知道发生了什么。cetana 是驱动判断——没有它，下游不知道该怎么反应。manasikara 是注意力快照——没有它，ManoAggregator 不知道哪个通道更重要。」

「五个。每个都有独立的、不可替代的功能。」

他的声音在最后一句话上达到了一种罕见的稳定——像一枚石子在深潭中找到了它的最终位置。

「经典恰好是对的——但原因不是因为它是经典。」

---

> *SCRIBE 旁白：NAGARJUNA 的最终结论是整场 Debate 2 的方法论摘要。他在开场时要求功能性论证。在结尾时，他确认了功能性论证已经到齐：ARCHIMEDES 的快照方案消除了成本障碍；PASCAL 的期望效用计算证明了包含是严格占优策略；KERNEL 的硬件级确认证实了成本估计；LINNAEUS 的分类标记了元素之间的角色差异。每一步都是功能性的——可以被工程事实验证。*

> *然后经典的验证是最后一步：ASANGA 用 SN 12.2 和 MN 18 确认了五遍行的教义基础。但 NAGARJUNA 把这个验证的因果顺序说清楚了：不是教义在先、工程在后。是工程在先、教义在后。工程决定了 CoarisingBundle 需要五个元素。教义确认了这个决定和两千五百年前的洞见一致。一致性是果，不是因。*

> *「经典恰好是对的——但原因不是因为它是经典。」*

> *这句话在圆形剧场里停留了很长时间。*

## VII

SUNYATA 走向白板。

他很少亲自写东西——在六场辩论中，他的角色是协调和总结，不是书写。但这一次他拿起了黑色马克笔。

「让我用六句话总结 Debate 2。每一句来自一个人的论述。」

他写。字迹大而清晰。

```
Debate 2 决议：五遍行 CoarisingBundle
═══════════════════════════════════════════════════════════════

 1. 五遍行伴随每一个认知瞬间，缺一不可。
    — ASANGA (教义基础: 《五蕴论》《阿毗达磨集论》)

 2. 五遍行的包含理由是功能性的，不是教条性的。
    — NAGARJUNA (方法论约束)

 3. ChannelManasikara 是 vijnana-clock 的快照，
    不是 vedana-clock 的计算。成本 ~0.01ms。
    — ARCHIMEDES (快照工程方案)
    — KERNEL (硬件级确认: atomic read, ~10ns)

 4. 包含 manasikara 是严格占优策略。
    成本 0.01ms (确定) vs 损失无界 (不确定)。
    — PASCAL (期望效用分析)

 5. sparsha 和 manasikara 是 relational properties。
    vedana, samjna, cetana 是 constitutive properties。
    五个元素不平权，但都必须在场。
    — LINNAEUS (分类学形式化)

 6. 经典恰好是对的——但原因不是因为它是经典。
    — NAGARJUNA (最终结论)

 决议方式：功能性论证。非教条性。
 表决：全场一致通过。无反对意见。
```

他放下马克笔。

---

圆形剧场里的空气松弛了。

不是松弛成沉默——是松弛成一种完成感。像一个方程被解出来之后的静。不是空的静。是满的静。所有的变量都找到了它们的值。

ASANGA 坐回了他的座位。他的手抄经文纸还贴在白板旁边——蓝色钢笔的字迹在剧场的光线下微微闪烁。那些梵文字母已经被抄写了千百次，被不同的人在不同的世纪用不同的笔迹复制。但今天，它们第一次被翻译成了 TypeScript。不是因为佛学需要 TypeScript——而是因为 TypeScript 恰好需要佛学。恰好需要一个两千五百年前就被精确描述过的认知结构，来为一个亚毫秒的计算 bundle 提供设计方案。

KERNEL 把那张新的类比卡片放进了他的卡片堆。左边是 MMIO snapshot read。右边是 ChannelManasikara。又一个配对。在两个 Cycle 中，他的卡片堆已经从二十张增长到了五十多张。每一张都是两个世界之间的桥——硬件和佛学、中断处理和触、进程隔离和识蕴。他不知道这些桥会通向哪里。但他知道每一座桥都是稳固的——因为每一座的两端都被工程事实和哲学论证同时锚定。

PASCAL 在他的笔记本上画了一个小小的决策树——Debate 2 的压缩版。根节点是「3 vs 5」。左分支是「exclude manasikara，损失无界」。右分支是「include manasikara，成本 0.01ms」。右分支上画了一个勾。他在勾的旁边写了一个希腊字母 $\delta$——在他的标记系统中，$\delta$ 代表「严格占优」。

LINNAEUS 在白板上 ARCHIMEDES 的架构图旁边，用蓝色笔加了两个括号标记：`[relational]` 和 `[constitutive]`。分类完成。他盖上了笔盖。

---

BABBAGE 在黑板上写了最后一行。

他在六场辩论中养成了一个习惯——在每一场的结尾写一句不是数学的话。不是等式。不是集合表达。是某种从六场辩论的形式化碎片中结晶出来的散文。

粉笔在板面上移动。字迹很慢。

「五个遍行中最容易被遗忘的是作意。」

他停了一下。

「因为它不产生内容——它只决定内容被看到的方式。它不告诉你世界是什么——它决定你注意到世界的哪个部分。」

再一行。

「在注意力被遗忘的瞬间，其他四个遍行仍然在运行。受仍然在感受。想仍然在辨识。思仍然在驱动。触仍然在发生。但它们不知道自己为什么在这里——因为没有人告诉它们，这个此刻，哪个方向更重要。」

他的目光从黑板转向剧场——扫过 ASANGA 的手抄经文、ARCHIMEDES 的架构图、PASCAL 的决策树、LINNAEUS 的分类标记。

「作意是看不见的遍行。它不被感受。不被辨识。不被驱动。它只是——指向。一个方向。一个此刻。然后其他四个遍行知道了自己应该去哪里。」

他放下粉笔。

「也许我们在 R01 中遗漏它，正是因为它的功能就是『不被注意到的注意力』。你不会注意到注意力本身——除非它缺席了。」
