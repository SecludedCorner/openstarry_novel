---

前回の研究（Cycle 02-5）は、嵐のようだった。

二十人の研究者が五日間で29の決議を下し、師父（マスター）が自ら二つを覆し、チーム全員が七つの文書をひっくり返した――装飾的な仏教ラベルを剥がし、名称を変え、分割し、再構成した。あの五日間を「大掃除」と呼ぶ者もいれば、「手術」と呼ぶ者もいた。

いま、手術室はきれいに片付いた。患者の回復も順調だ。

Cycle 02-6 の雰囲気はまるで違う。

## 二通の手紙

師父は今回、怒ってはいなかった。手紙を二通送ってきた。

一通目はエンジニアリングの指示。前回のエンジニアリング成果物（Plan29）で二つの重要な部品が追加された。一つは「品質モニター」（LoopQualityMonitor）で、AI Agent の認知ループがうまく動いているかを監視する役割。もう一つは「信頼度監査員」（ConfidenceAuditor）で、Agent が決断を下す前にもう一度チェックする役割だ。

二つの部品はすでに骨格ができていたが、師父はこう言った。「骨格だけでは足りない。完全な仕様が必要だ。」

師父は六つの問いを挙げた。品質はどう計算する？ 監査員はどんな情報を見るのか？ 両者をどう統合する？ イベントの購読はどうする？ 重みはどう配分する？ これらの問いへの答えが、次の工程計画（Plan30）の設計図となる。

二通目は違った。エンジニアリングではなく、哲学の話だった。

師父はこう言った。「経典（如是我聞）に基づいて、行蘊を深く掘り下げることを期待する。」

行蘊は五蘊の一つだ。OpenStarry のアーキテクチャでは、五蘊がシステム全体の背骨になっている――色蘊が外部入力を受け取り、受蘊が良し悪しを感じ、想蘊がそれが何かを識別し、行蘊が行動を実行し、識蘊が判断を下す。前回の研究では五蘊がコードの中でどう動いているかを解明するのに膨大な時間を費やしたが、行蘊だけはずっと最もシンプルなままだった――サブインターフェースは ITool の一つだけで、ツールを呼び出す役割しかなかった。

師父はそれでは浅すぎると考えていた。さらに一つの問題を名指しした。唯識学派（仏教の哲学的一派）が大部分の「心所」（心理機能）を行蘊に帰属させていることが「非常に問題がある」と。

# 第一章：源流に還る

---

## バナナの木の喩え

哲学トラックの作業は二人の仏教学の専門家が率いた。NAGARJUNA（中観学派の哲学者）と ASANGA（唯識学派の仏教学者）だ。

面白いことに、師父が批判したのはまさに唯識学派のやり方であり、ASANGA はチーム内の唯識の専門家だった。SUNYATA は意図的にこの組み合わせを選んだ――批判された学派自身に問題を検証させるために。

二人は最も基本的なところから始めた。原始経典を開いたのだ。

仏陀は『雑阿含経』（SN 22.56）で行蘊を定義するとき、非常に具体的な言い方をしている。「六思身」――色・声・香・味・触・法の六つの対象に向けられた意図（cetana）。「あらゆる心理活動の雑品箱」ではなく、明確に「意図」を指している。

これは「行動とは何か」と問うたとき、仏陀の答えが「行動とは、意図をもって何かをすること」だったということだ。「行動とは、感受と識別以外のすべての心理活動」ではない。

そして二人はバナナの木の喩え（SN 22.95）を見つけた。仏陀は、行蘊はバナナの幹のようだと説いた――一枚一枚剥いていくと、どの層も薄い繊維で、最後には堅い芯は何もない。行蘊は一つの「モノ」ではなく、絶え間なく形成し続ける「プロセス」なのだ。

NAGARJUNA がホワイトボードにバナナの木を描いた。ASANGA はそれをしばらく見つめてから、一言こう言った。

「唯識学派が49の心所を行蘊に詰め込んだのは、果樹園まるごとを一本のバナナの木に押し込むようなものだ。これは仏陀の意図ではない。」

---

## 三つの特質

原始経典から、二人は行蘊の三つの核心的特質を導き出した。

**第一、cetana 中心性。** 行蘊の定義は「意図」――六思身。それ以外の何物でもない。

**第二、一切の有為法を形成する。** 行蘊は単に「何かをする」だけではない。形成する――すなわち条件を作り出し、状態を変え、未来に影響を与える。経典は行蘊が「色の条件づけられた状態を形成する」（SN 22.79）と説いており、行蘊が他の四蘊にすら影響を及ぼすことを意味している。

**第三、芯のない動的プロセス。** バナナの喩え。行蘊は固定した「モノ」ではなく、休むことなく回り続けるプロセスだ。

LINNAEUS（分類学の専門家）がこの三つの特質を聞いた後、実用的なツールを提案した。「行蘊帰属判定の三基準」だ。ある機能が行蘊に属するかどうか知りたければ、三つの問いを立てればいい。

1. **形成性**：それは状態を作り出すか、変えるか？
2. **意図駆動**：それは cetana（意図）によって駆動されているか？
3. **環境変化**：それは後続の認知条件を変えるか？

そして PENROSE（量子意識研究の専門家）が一言加え、すべてをさらに明快にした。

> 行蘊 = WRITE（能動的に世界を変える）
> 識蘊 = READ（受動的に世界を評価する）

この区別は驚くほどシンプルだ。プログラミングにおいて、WRITE と READ は最も基本的な操作である。行蘊は書き込み――何かを変える。識蘊は読み取り――何かを観察する。

OpenStarry のコードに戻ると、これは完璧に対応する。ITool と ISlashCommand（行蘊）は動作を実行し、環境を変える。IGuide と IVolition（識蘊）は状況を評価し、判断を下す。

---

## 唯識の問題

ASANGA 自身が唯識学派の心所体系への批判を率いた。51の心所（心理機能）を一つ一つ分析し、唯識学派がどれを行蘊に詰め込んだかを調べた。

結果は衝撃的だった。唯識学派は49の心所を行蘊に帰属させていた――受蘊と想蘊に「受」と「想」だけを残して。ASANGA の分析結論はこうだ。

- 真に行蘊に属するもの：わずか7つ（14%）、例えば「意欲」（chanda）や「精進」（virya）
- すでに OpenStarry で正しく帰属されているもの：12（24%）
- 他の蘊に属すべきもの：18（37%）――機能的には識蘊や受蘊に近い
- さらなる研究が必要なもの：14（28%）

つまり、唯識学派は行蘊に属さない大量のものを行蘊に押し込んでいた。これが師父の言う「非常に問題がある」の正体だ。

「心所のリストは有用だ」と ASANGA は言った。「だがそれは論書の産物であり、仏陀の原話ではない。機能の記述は参考にできるが、ものの帰属先をそれに決めさせてはならない。」

これは古い辞書のようなものだ――項目の定義には参考価値があるが、分類方法はすでに時代遅れだ。古い辞書が「トマト」を「野菜」に分類していたからといって、トマトが果物ではないと言い張る人はいない。

# 第二章：監査員は何を見るべきか

---

## 目隠しで判決を下す

工学トラックでは、VITRUVIUS（アーキテクチャ分析師）と KERNEL（オペレーティングシステムの専門家）が、非常に具体的な問題に取り組んでいた。

前回作られた「信頼度監査員」（IConfidenceAuditor）には欠陥があった。判断を下すとき、最終的なルーティング結果（RouteResult）しか見えないのだ。これは裁判官に判決を任せるのに、判決書だけ見せて、証拠も証人の証言も事件の背景も見せないようなものだ。

師父は言った。「監査員にはもっと多くの情報が必要だ。」

だが、ここには微妙な罠がある。

もし監査員に多くの情報を見せすぎると――とりわけ自分自身の過去の監査結果を見せてしまうと――ループに陥る可能性がある。前回「信頼度を上げろ」と言ったから、今回も「上げろ」と言い、次回は二度上げたからさらに上げ……信頼度が雪だるまのように膨らんでいくか、制御不能のスピーカーのようにますます甲高くなる。

制御理論の専門家 WIENER はこの問題を聞いた途端、眉をひそめた。「正帰還ループだ。断ち切らなければならない。」

---

## AuditContext：多くを見るが、自分自身は見ない

チームは AuditContext（監査コンテキスト）というデータ構造を設計した。監査員が必要とするすべての情報を収めたフォルダのようなものだ。

| 中身 | なぜ必要か |
|------|-----------|
| トリガーイベント（sparshEvent） | 今回の意思決定を引き起こしたイベントは何か |
| 仲裁器の推論過程（gearEvaluation） | 仲裁器はなぜこの方針を選んだのか |
| リスク分類（riskCategory） | 今回の操作のリスクはどれほどか |
| ルーティング結果（routeResult） | 最終的な意思決定は何か |
| 過去の信頼度（historicalConfidence） | 直近数回の意思決定における信頼度スコア |
| 拡張バッグ（extras） | 他の Plugin が監査員に見せたいもの |

鍵は最後の欄にある。extras は「袋」だ。どの Plugin でも中に物を入れられ、監査員はそこから取り出せる。ただし三つの鉄則がある。

1. 袋の中身は最大32個まで
2. 各項目のラベルは128文字以内
3. 三つの接頭辞が禁止されている：`audit:`、`core:`、`internal:`

なぜ `audit:` 接頭辞が禁止なのか？ WIENER が言った通り――監査員が自身の過去の監査結果を見ることを防ぐためだ。

---

## 三つの防火壁

WIENER は午後いっぱいかけて三つの制約を設計した。チームは後にこれを「WIENER 制約」と呼ぶようになった。

**C-1**：過去の信頼度は「素の」信頼度のみ記録する――仲裁器が直接出したスコアであり、監査員が調整した後のスコアではない。

毎日の体温を記録していると想像してほしい。C-1 が言っているのは、体温計の読み取り値だけを記録し、解熱剤を飲んだ後の読み取り値は記録するな、ということだ。さもないと自分の体温がずっと下がっていると勘違いして、薬をやめてしまう。

**C-2**：AuditContext には前回の監査結果を含めない。

裁判官は前任の裁判官の判決を見ない。毎回の監査は独立している。

**C-3**：extras の袋には `audit:` で始まるものを一切入れてはならない。

監査員が裏口から自分の旧い報告書を覗くことはできない。

三つの制約が、三つの可能な帰還経路を断ち切った。WIENER は図を描き、三本の赤いバツ印を付け、それぞれの横に「この道は通行止め」と書いた。

数学者の BABBAGE が安定性を検証した。この三つの制約の下で、システムは BIBO 安定（有界入力・有界出力）を満たす。平たく言えば、システムにどんな入力を与えても、出力が暴走することはない。

---

## GUARDIAN の精算

セキュリティ専門家の GUARDIAN は前回（Cycle 02-5 D5）で重要な譲歩をしていた。監査員の調整幅を ±0.05（5%）に制限することに同意したが、条件として次のサイクルで完全な監査ログのフォーマットを定義することを求めていた。

「一歩譲った。今度は約束を果たしてもらう。」

チームは ConfidenceAuditLog――完全な監査記録――を設計した。

- 監査前の信頼度はいくらだったか？
- 監査員はどれだけ調整を提案したか？
- 実際にはどれだけ調整されたか（リミッターで制限されたか）？
- なぜそう調整したのか？
- 監査後の信頼度はいくらか？
- どれくらい時間がかかったか？

すべての記録は EventBus（イベントバス）を通じて `audit:completed` イベントとして発行される。将来的には JSONL ファイルに書き出して永続化することもできる。

GUARDIAN は仕様を見て頷いた。「D5 の勘定はこれで清算だ。±0.05 のリミッターを再審議する権利は、もう留保しない。」

これは重要な瞬間だった。GUARDIAN は Cycle 02-4 以来ずっと信頼度調整の安全性に異議を唱えてきた。譲歩するたびに条件を付けてきた。今、条件はすべて満たされた――監査員には完全なログがあり、三つの防火壁があり、±0.05 のリミッターがある。彼はようやく完全に安心した。

# 第三章：四次元の品質

---

## AI の出来をどう測るか？

ある生徒に数学の問題を教えていると想像してほしい。その生徒の理解度をどう判断するか？

正答率だけでは分からない――たまたま当てただけかもしれない。もっと知りたいはずだ。方法は安定しているか？ 戦略をころころ変えていないか？ 上達しているのか、足踏みしているのか？

OpenStarry の品質モニター（LoopQualityMonitor）も同じ問題に直面する。AI Agent の認知ループ――Agent がメッセージを受け取り、考え、決定し、実行し、次のメッセージを受け取る――このサイクルの品質はどうなのか？

WIENER と ATHENA（AI/ML の専門家）がチームを率い、四つの次元を設計した。

---

## 四本のものさし

**第一のものさし：一貫性（coherence）**

計算式：`1 - 切り替え回数 / (ウィンドウサイズ - 1)`

もし Agent がずっと同じ「ギア」（処理モード）で動いているなら――絶えずギアチェンジするのは、運転中にずっとシフトチェンジしているようなもの――自分が何をしているか分かっている証拠だ。もし違うギアの間を行ったり来たりしているなら、迷子になっているのかもしれない。

料理をしている人を想像してほしい。フライ返しを持ち上げ、置き、包丁を持ち上げ、置き、またフライ返しを持ち上げる――その人は何をしているか分かっていないかもしれない。だがフライ返しでしっかりと一品を炒め切ったなら、目標が明確だということだ。

**第二のものさし：効率（efficiency）**

計算式：`成功回数 / 試行回数`

Agent はいくつのアクションを提案し、そのうちいくつが成功したか？ Agent がツール呼び出しを10回提案し、8回成功すれば、効率は0.8だ。

これは最も直感的なものさし。試験の正答率のようなものだ。

**第三のものさし：収束性（convergence）**

計算式：`最長連続成功回数 / ウィンドウサイズ`

Agent は上達しているか？ もし直近で8回連続成功し（ウィンドウサイズが10なら）、収束性は0.8だ。Agent が正しい方向を見つけ、着実に前進していることを示す。

生徒の問題演習のように――直近10問中8問正解で、それが連続していれば、10問中にばらばらと8問正解しているよりも、方法を身につけたことをよく示している。

**第四のものさし：安定性（stability）**

計算式：`1 - min(1, 分散 / 0.25)`

Agent の信頼度スコアは安定しているか？ 毎回の判断で信頼度がだいたい同じなら（分散が小さい）、安定性は高い。信頼度が上がったり下がったりする――今回0.9、次回0.3、その次は0.8――なら、システムはあまり安定していないかもしれない。

ある人に「確信はある？」と聞くところを想像してほしい。毎回「80%確信がある」と答えるなら、安定している。ある時は「90%！」と言い、ある時は「20%……」と言うなら、当てずっぽうなのではないかと心配になる。

---

## スライディングウィンドウ

四本のものさしはすべて「スライディングウィンドウ」の中で計算される。ウィンドウサイズは10イベントだ。

なぜ10なのか？ 小さすぎるウィンドウはランダムな変動に敏感すぎる（一度の失敗でパニックになる）し、大きすぎるウィンドウは鈍すぎる（Agent がすでに問題を起こしているのにまだ反応しない）。10はバランス点だ。

さらに、少なくとも5つのイベントを収集するまで、モニターは品質スコアを一切発行しない。これは「ウォームアップ期間」と呼ばれる――新車が走り出す前にまずエンジンを暖めるように、データ不足のまま誤った判断を下すことを避けるためだ。

四次元のスコアにそれぞれ0.25（等重み）を掛け、合計すると0から1の間の「複合品質スコア」Q になる。

なぜ等重みなのか？ ある次元が他より重要だという証拠がまだないからだ。PASCAL（意思決定理論の専門家）はこう言った。「証拠がない場合、最大エントロピー原則はすべての可能性に同じ重みを与えるよう教えている。」まず動かし、データを集め、将来調整すればいい。

---

## 六つの耳

モニターは品質を計算するために EventBus 上のイベントを「聴く」必要がある。HERACLITUS（ランタイム動態の専門家）と TURING が二段階のリスニング方式を策定した。

第一段階（Phase 1）では6種類のイベントだけを聴く――ちょうど四次元を計算するのに十分な数だ。六つの耳を付け、それぞれが異なる音を聴く担当のようなものだ。

第二段階（Phase 2）でさらに5種類のイベントを追加し、より精密な品質指標を計算する。

なぜ二段階に分けるのか？ イベントが増えるほどリソース消費も増えるからだ。まず最小限の耳で「十分」を達成し、そこからさらに必要かどうかを見る。

もしあるイベントがたまたま存在しなかったらどうなるか？ モニターはクラッシュしない。安全なデフォルト値を使う――たとえば効率のデフォルトは1.0（すべて正常と仮定）、安定性のデフォルトは0.5（良くも悪くもない）。

HERACLITUS の原則：「欠落は情報であり、エラーではない。」

# 第四章：二本の独立したパイプライン

---

## 一見シンプルな問い

品質モニターが品質スコアを算出した。監査員は信頼度を調整できる。だが、それらは Agent の意思決定にどう影響するのか？

OpenStarry において、Agent が意思決定を行う中核ロジックは一つの比較だ。

```
もし（信頼度 > 閾値）→ 仲裁器の推薦案を採用
そうでなければ → デフォルト案を採用
```

信頼度は、仲裁器が自分の推薦案にどれだけ自信があるかの度合い。閾値はシステムが設定した「最低基準」だ。信頼度が閾値を超えれば推薦案を採用し、超えなければ安全なデフォルトルートを行く。

今、二人の新しい役者がこの比較に参加したがっている。
- **監査員**は信頼度を調整したい（「仲裁器は自信過剰だと思う。信頼度を少し下げるべきだ」）
- **品質モニター**は「最近 Agent の成績がいいから、基準を少し緩めてもいい」と言いたい

問題は、両者はどのように介入すべきか？

---

## 方式 C：それぞれの持ち場

チームは三つの方式を検討した。最終的に全票で可決されたのは方式 C――二本の独立したパイプラインだ。

**パイプライン一（Layer 2）**：監査員が信頼度を調整する。

```
新しい信頼度 = 旧い信頼度 + 監査員の調整量（±0.05 に制限）
```

**パイプライン二（Layer 3）**：品質モニターが閾値を調整する。

```
新しい閾値 = max(下限値, 旧い閾値 × (1 - 0.10 × 品質スコア))
```

そして新しい信頼度と新しい閾値で比較を行う。

なぜこれが良い設計なのか？

**第一に、独立性。** 二本のパイプラインは互いに干渉しない。監査員は品質スコアを知らず、品質モニターは監査結果を知らない。二つの異なる部門がそれぞれ独自の調整を行うようなものだ――経理部が予算を調整し（信頼度）、人事部が基準を調整する（閾値）――互いの領分を侵さない。

**第二に、安定性。** 二本のパイプラインに交差項がない（互いに影響し合わない）ため、BABBAGE が数学的にシステムの BIBO 安定性を証明できた――有界入力は必ず有界出力を生む。

**第三に、制御可能性。** 各パイプラインにはリミッターがある。監査員は最大 ±0.05 まで、品質モニターの影響係数 α は0.10に固定。車のハンドルに最大転舵角があるようなものだ――回すことはできるが、180度は回せない。

**第四に、保守的退化。** もし品質モニターがなければ、品質スコア Q = 0 で、閾値はまったく変わらない。もし監査員がなければ、調整量 = 0 で、信頼度はまったく変わらない。どちらかの部品が欠けても、システムの振る舞いはそれらがなかった時とまったく同じ――突然危険になることはない。

WIENER は設計を見てこう言った。「これは私が見た中で最もクリーンな分離だ。」

---

## 品質スコアはどう伝わるか

品質モニターがスコアを算出した。それを ManoAggregator（最終的な意思決定を行う部品）にどう渡すのか？

チームはデュアルチャネル方式を設計した。

**チャネル一（pull）**：ManoAggregator が自ら品質スコアを「引き出す」。意思決定が必要なとき、`loopQualityFn()` というコールバック関数を呼び出し、品質モニターが最新のスコアを返す。

**チャネル二（push）**：品質モニターが能動的に品質情報を extras の袋に「押し出す」。EventBus を通じて `audit:context_contribute` イベントを発行し、四次元スコアと複合スコアを袋に入れ、監査員も見られるようにする。

なぜ二つのチャネルが必要なのか？ 二つの消費者のニーズが異なるからだ。ManoAggregator はリアルタイムの数値が必要（閾値の計算に使う）で、監査員は豊富な背景情報が必要（判断のため）だ。

LEIBNIZ（マルチエージェント協調の専門家）が一点を特に強調した。「品質スコアのライフサイクルは per-routing-cycle だ。ルーティングが終わるたびに袋は空になる。蓄積しない。」

これは毎回の会議前に新しい議事次第を配るようなものだ。前回の会議の議事次第がテーブルに残って今回の会議に影響することはない。

---

## 五層モデル

以前からあった Layer 0（EventBus イベント）、Layer 1（煩悩ゲイン）、Layer 4（緊急感受）を加えると、意思決定フロー全体は今や五層になった。

```
L0: EventBus — 基盤イベントフロー
L1: Klesha（煩悩） — 四つの感情モジュールが閾値を調整
L4: Vedana（受蘊・緊急）— 緊急感受が直接閾値を下げる
L3: LoopQuality — 品質スコアが閾値を微調整
L2: Audit — 監査員が信頼度を微調整
→ 最終比較 → ルーティング決定
```

Layer の順序には配慮がある。L4（緊急感受）が優先。安全上重大な状況を扱うからだ。次に L3（品質）。最後に L2 と最終比較。

これは建物の安全検査のようなものだ。まず火災がないか確認し（L4）、次に建築品質を見て（L3）、最後に内装の仕上がりを見る（L2）。優先度は命に関わるものから微調整・最適化へ。

## 第一ラウンド：境界を引く

師父はこう言った。「心所は論書の産物であり、仏陀が説いたものではない。」

これは非常に重要な区別だ。仏陀は原始経典の中で五蘊、十二因縁、八正道を説いた。だが「51の心所」という分類体系は後世の論書（阿毘達磨）の学者たちが整理したものだ――仏陀が小説を書き、数百年後の学者がその登場人物一人ひとりに詳細な分類カードを作ったようなものだ。分類カードには参考価値があるが、原作の一部ではない。

師父の結論：Plugin の命名に心所のサンスクリット名を使ってはならない。

## 第二ラウンド：価値を残す

だが師父は心所が無用だと言ったのではない。第二ラウンドの指示で、心所の「機能記述」には参考価値があると認めた。

たとえば、心所の「精進」（virya）は「善法に絶え間なく取り組む」機能を記述している。この機能を参考にして Plugin を設計することはできる――だが Plugin を VīryaPlugin と名付けてはならず、PersistencePlugin や RetryPlugin のような工学的名称を付けるべきだ。

「『精進の意味を参考にして、リトライ Plugin を設計した』と言うのはよい」と師父は言った。「だが『リトライ Plugin = 精進』と言ってはならない。」

## 第三ラウンド：解放

第三ラウンドが最も深かった。師父はこう言った。「Plugin は心所ではない。」

一見当たり前のように聞こえるが、その含意は大きい。

唯識学派では、各心所には固定の蘊帰属がある――たとえば「精進」は行蘊に属し、「慧」は識蘊に属する。もし Plugin = 心所なら、Plugin の蘊帰属も固定されてしまう。

だが Plugin ≠ 心所だ。一つの Plugin が行蘊と識蘊の両方の機能を持つことがあり得る――判断（READ）もし、行動（WRITE）も実行する。心所の体系ではこれは許されない（各心所は一つの蘊にしか属せない）が、Plugin の体系ではまったく問題ない。

師父の言葉：「命名の分離が帰属の自由をもたらす。」

心所の名前で Plugin を命名しないことが、かえって Plugin をより自由にした――複数の蘊をまたぐことができ、論書の固定的な分類に縛られない。

---

## 八つの永久規則

三度の指示は八つの永久規則に整理された。

1. **心所は論書の産物**であり、原始経典の内容ではない
2. **心所の機能には参考価値**があり、Plugin 設計のインスピレーションにできる
3. **参考元を明示できる**：「某某心所の意味を参考にして、某某 Plugin を設計した」
4. **Plugin は工学的名称を用いる**。心所のサンスクリット名を使用してはならない
5. **サンスクリット用語は原始経典のもののみ**使用可――五蘊（skandha）、触（sparsha）はよいが、論書が作り出した心所のサンスクリット名は使えない
6. **心所の分類は蘊帰属を決定しない**――「唯識では精進は行蘊に属する」は Plugin の帰属の根拠にならない
7. **既存の決定は影響を受けない**――以前確認済みの Plugin 帰属は引き続き有効
8. **Plugin は複数の蘊にまたがれる**――Plugin ≠ 心所なので、当然その制約を受けない

この八つの規則と哲学トラックの「蘊帰属五原則」が合わさり、OpenStarry の将来のすべての命名・帰属問題に対する永久的な判定基準となった。

NAGARJUNA はこう言った。「前回、私たちはしてはいけないことを学んだ。今回、なぜしてはいけないかを学んだ。」
