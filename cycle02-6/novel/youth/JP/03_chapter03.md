# 第三章：四次元の品質

---

## AI の出来をどう測るか？

ある生徒に数学の問題を教えていると想像してほしい。その生徒の理解度をどう判断するか？

正答率だけでは分からない――たまたま当てただけかもしれない。もっと知りたいはずだ。方法は安定しているか？ 戦略をころころ変えていないか？ 上達しているのか、足踏みしているのか？

OpenStarry の品質モニター（LoopQualityMonitor）も同じ問題に直面する。AI Agent の認知ループ――Agent がメッセージを受け取り、考え、決定し、実行し、次のメッセージを受け取る――このサイクルの品質はどうなのか？

WIENER と ATHENA（AI/ML の専門家）がチームを率い、四つの次元を設計した。

---

## 四本のものさし

**第一のものさし：一貫性（coherence）**

計算式：`1 - 切り替え回数 / (ウィンドウサイズ - 1)`

もし Agent がずっと同じ「ギア」（処理モード）で動いているなら――絶えずギアチェンジするのは、運転中にずっとシフトチェンジしているようなもの――自分が何をしているか分かっている証拠だ。もし違うギアの間を行ったり来たりしているなら、迷子になっているのかもしれない。

料理をしている人を想像してほしい。フライ返しを持ち上げ、置き、包丁を持ち上げ、置き、またフライ返しを持ち上げる――その人は何をしているか分かっていないかもしれない。だがフライ返しでしっかりと一品を炒め切ったなら、目標が明確だということだ。

**第二のものさし：効率（efficiency）**

計算式：`成功回数 / 試行回数`

Agent はいくつのアクションを提案し、そのうちいくつが成功したか？ Agent がツール呼び出しを10回提案し、8回成功すれば、効率は0.8だ。

これは最も直感的なものさし。試験の正答率のようなものだ。

**第三のものさし：収束性（convergence）**

計算式：`最長連続成功回数 / ウィンドウサイズ`

Agent は上達しているか？ もし直近で8回連続成功し（ウィンドウサイズが10なら）、収束性は0.8だ。Agent が正しい方向を見つけ、着実に前進していることを示す。

生徒の問題演習のように――直近10問中8問正解で、それが連続していれば、10問中にばらばらと8問正解しているよりも、方法を身につけたことをよく示している。

**第四のものさし：安定性（stability）**

計算式：`1 - min(1, 分散 / 0.25)`

Agent の信頼度スコアは安定しているか？ 毎回の判断で信頼度がだいたい同じなら（分散が小さい）、安定性は高い。信頼度が上がったり下がったりする――今回0.9、次回0.3、その次は0.8――なら、システムはあまり安定していないかもしれない。

ある人に「確信はある？」と聞くところを想像してほしい。毎回「80%確信がある」と答えるなら、安定している。ある時は「90%！」と言い、ある時は「20%……」と言うなら、当てずっぽうなのではないかと心配になる。

## スライディングウィンドウ

四本のものさしはすべて「スライディングウィンドウ」の中で計算される。ウィンドウサイズは10イベントだ。

なぜ10なのか？ 小さすぎるウィンドウはランダムな変動に敏感すぎる（一度の失敗でパニックになる）し、大きすぎるウィンドウは鈍すぎる（Agent がすでに問題を起こしているのにまだ反応しない）。10はバランス点だ。

さらに、少なくとも5つのイベントを収集するまで、モニターは品質スコアを一切発行しない。これは「ウォームアップ期間」と呼ばれる――新車が走り出す前にまずエンジンを暖めるように、データ不足のまま誤った判断を下すことを避けるためだ。

四次元のスコアにそれぞれ0.25（等重み）を掛け、合計すると0から1の間の「複合品質スコア」Q になる。

なぜ等重みなのか？ ある次元が他より重要だという証拠がまだないからだ。PASCAL（意思決定理論の専門家）はこう言った。「証拠がない場合、最大エントロピー原則はすべての可能性に同じ重みを与えるよう教えている。」まず動かし、データを集め、将来調整すればいい。

## 六つの耳

モニターは品質を計算するために EventBus 上のイベントを「聴く」必要がある。HERACLITUS（ランタイム動態の専門家）と TURING が二段階のリスニング方式を策定した。

第一段階（Phase 1）では6種類のイベントだけを聴く――ちょうど四次元を計算するのに十分な数だ。六つの耳を付け、それぞれが異なる音を聴く担当のようなものだ。

第二段階（Phase 2）でさらに5種類のイベントを追加し、より精密な品質指標を計算する。

なぜ二段階に分けるのか？ イベントが増えるほどリソース消費も増えるからだ。まず最小限の耳で「十分」を達成し、そこからさらに必要かどうかを見る。

もしあるイベントがたまたま存在しなかったらどうなるか？ モニターはクラッシュしない。安全なデフォルト値を使う――たとえば効率のデフォルトは1.0（すべて正常と仮定）、安定性のデフォルトは0.5（良くも悪くもない）。

HERACLITUS の原則：「欠落は情報であり、エラーではない。」
