# 第六章：十七票

## 最も静かな討論

Cycle 02-5 の討論は戦争のようだった――五つの討論、29の決議、二つが師父に覆された。激しく反対する者がいた。途中で立場を変える者がいた。GUARDIAN はセキュリティの問題で一歩も引かなかった。

Cycle 02-6 の討論は将棋のようだった。静かで、精確で、一手一手に道理がある。

三つの討論、245分、17の決議。否決ゼロ。13が全会一致（20/20）。残る4つも18票から19票だった。

なぜこれほど順調だったのか？

SUNYATA の後の分析はこうだ。「R1 がよくできていた。」

独立研究（R1）は14本の報告を生み出した。クロスレビュー（R2）で24の合意点が見つかり、相違点はわずか3つだった。大部分の問題が討論の前にすでに合意に達していれば、討論は確認と精緻化の場となり、ゼロから始める論争にはならない。

## D1：行蘊の七票

第一の討論はおよそ75分を費やし、行蘊に焦点を当てた。

七つの決議が順次採決された。

**D1-R1（20/20）**：行蘊の核心的定義。cetana 中心、形成機能、芯のないプロセス。反対意見は一切なかった。哲学トラックの結論があまりにも堅固だった――原始経典を直接引用しており、曖昧な余地がなかった。

**D1-R3（20/20）**：ISamskara（行蘊インターフェース）にサブインターフェースを新設しない。四つの拡張方向を長期候補としてアーカイブ。三基準を永久ツールとして確認。

**D1-R4a（19/20）**：認知系列（citta-vīthi）付録を次のサイクルにスケジュール。一票が反対し、今回のサイクルでできると主張。

**D1-R4b（18/20）**：四智を正式に除外。四智は仏教の四つの智慧――大円鏡智、平等性智、妙観察智、成所作智。二票が参考として残すべきだと反対。しかし四つのテストすべてに失敗した――除去しても設計は何も変わらず、コードの中にも存在せず、いかなる工学的意思決定も駆動していなかった。

**D1-R4c（20/20）**：心智論述総合評価表を確認。どの仏教概念がシステムに統合済みか、どれが付録になれるか、どれが除外されるべきか――すべて整理完了。

**D1-R5（20/20）**：「活動と変換」原則を確認。行蘊 = WRITE、識蘊 = READ。

**D1-R6（20/20）**：蘊帰属五原則。全会一致で可決。これは一回限りのサイクルの決議ではなく、将来のすべての研究のベースラインとなる永久的規則だ。

## D2：監査の五票

第二の討論はおよそ85分を費やした。

**D2-R1（20/20）**：AuditContext の完全な型定義。

**D2-R2（19/20）**：extras の袋のプロトコル。一票が反対し、デュアルイベントモデルは複雑すぎる、直接購読でよいと主張。だが多数はデュアルイベントモデルの方が汎用的だと考えた――品質モニターだけでなく、どの Plugin でも使える。

**D2-R3（20/20）**：ConfidenceAuditLog。GUARDIAN がその場で確認した。「D5 の勘定はこれで済んだ。」

**D2-R4（20/20）**：Layer 統合方式 C。二本の独立したパイプライン。

**D2-R5（20/20）**：Auditor 戦略は三期に分ける。Phase 0 は「パススルー監査員」――何も調整せず、ログを記録するだけ。Phase 1 はルールベース監査員。Phase 2 は LLM 補助監査員。

Phase 0 は無意味に聞こえる――何もしない監査員？ だが ARCHIMEDES はこう説明した。「その価値は監査にあるのではなく、テストにある。パイプライン全体がエンドツーエンドで通じているかを検証するのだ。水道管を取り付けた後、まず蛇口を開いて水が出るか確かめるようなものだ――水が出れば、配管に問題はない。」

## D3：品質の五票

第三の討論もおよそ85分を費やした。

**D3-R1（20/20）**：四次元の計算式。第三章で述べた四本のものさしだ。

**D3-R2（20/20）**：EventBus イベント購読。6つの Phase 1 イベント + 5つの Phase 2 イベント。同時に、以前コード中に散在していたイベント文字列を統一的に定数に昇格させた。

**D3-R3（20/20）**：extras のライフサイクル。per-routing-cycle、1ティックの遅延は許容範囲内。

**D3-R4（20/20）**：重み配分は Phase 1 では等重みに固定。PASCAL の最大エントロピー論証は説得力があった。

**D3-R5（19/20 + 1条件付き）**：Plan30 の方向性。唯一「条件付き賛成」のある決議だった。

GUARDIAN が条件付き賛成を投じた。「同意する。ただし Plan30 の Wave 3 に ConfidenceAuditLog の SDK 型定義を含めなければならない。型定義を Plan31 に先送りしてはならない。」

条件は受け入れられた。GUARDIAN は賛成に転じた。

## なぜ否決ゼロだったのか

SUNYATA は討論終了後に統計をまとめた。

| Cycle | 決議数 | 否決 | 全会一致 | 師父の覆議 |
|-------|--------|------|----------|-----------|
| 02-5 | 29 | 0 | 約20 | 2 |
| 02-6 | 17 | 0 | 13 | 0 |

Cycle 02-6 の決議数は前回の6割にすぎないが、全会一致の比率はより高く（76% vs 約69%）、しかも師父の介入を必要としなかった。

DARWIN（ソフトウェアパターン分析師）が観察した。「02-5 は正しさの討論――何が正しく、何が間違っているか。02-6 は仕様の討論――正しいものはどんな形をしているか。後者の方が本質的に合意に達しやすい。価値判断を伴わないからだ。」

つまり、前回は「何をするか」の問題を解決し、今回は「どうやるか」の問題を解決した。「どうやるか」の意見の相違は、通常「何をするか」より小さい。
