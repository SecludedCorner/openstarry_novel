# 第七章：設計図の完成

## Plan30

十七の決議が確定すると、工学的な設計図は自然に浮かび上がった。

VITRUVIUS と ARCHIMEDES がすべての仕様を Plan30――次の工程計画の完全な設計図――にまとめた。四つのウェーブ（Wave）で、小さいものから大きいものへ。

**Wave 1（約120行のコード）**：デフォルトの品質モニター Plugin を作る。四次元の計算式を実際のコードにし、6つのイベントを購読し、各イベントが到着するたびに O(1) の効率でスコアを更新する。

**Wave 2（約80行のコード）**：品質モニターを ManoAggregator に接続する。方式 C の Layer 3 の配線――品質スコアをコールバック関数経由で渡し、閾値を計算式に従って調整する。

**Wave 3（約60行のコード）**：型定義とイベント定数。ConfidenceAuditLog の SDK 型、AgentEventType 定数の昇格、extras 関連のイベント定義。

**Wave 4（約30行のコード、オプション）**：「パススルー監査員」を作る――何も調整せず、ログだけ記録する PassthroughAuditor。Phase 0 の実装であり、監査パイプライン全体が通じているかを検証するためのものだ。

合計およそ260から290行のプロダクションコードに、130行のテストコードを加える。420行に満たない。

成功基準はただ一つ。**Model Delta の五層（L0〜L4）すべてに実際のシグナルパスが存在する**こと。統合テストで検証する。

つまり、Plan30 が完成すれば、最下層のイベントフロー（L0）から最上層の監査出力（L4）まで、どの層ももはや空の殻ではない――すべてに実際に動くコードがある。

## 次のステップ：Plan31

Plan30 の後は Plan31 だ。

Plan31 では AuditContext の型を実行パスに実装する――SDK の型定義にとどまらず、毎回のルーティングで実際に AuditContext を組み立て、監査員に渡し、ログを記録する。

予想される規模はおよそ350行のコード。内容は以下の通り。
- AuditContext の組み立て（Core が担当）
- Default ThresholdAuditor（Phase 1 ルールベース監査員）
- Audit Trail の JSONL 永続化

Plan30 がパイプラインを作る。Plan31 がパイプラインに水を流す。

## 哲学の収穫

哲学トラックに話を戻そう。今回の哲学研究は前回のようにシステムの外観を変えることはなかった（Cycle 02-5 では大量の装飾的な仏教マッピングを除去した）が、より深い理解を築いた。

**行蘊はもはや「残余カテゴリー」ではない。** 精確な定義を得た。cetana（意図）を中心とした形成のプロセス。「五蘊に収まりきらないもののゴミ箱」ではなく、「能動的に世界を変える意志の力」だ。

**三基準が永久ツールとなった。** 今後「この機能はどの蘊に属するか」と問うたびに、三基準で判定できる。形成性、意図駆動、環境変化。もう毎回最初から議論する必要はない。

**心所の問題が一度で解決した。** 師父の八つの規則と蘊帰属五原則が、「仏教用語と工学的命名をどう共存させるか」という問題を根本的に解決した。参考はするが丸写しはしない。借用はするが混同はしない。

ASANGA は最後にこう言った。「今回、私は自分の学派に最も厳しかった。だがこれは唯識への裏切りではない――唯識の以前に還ることだ。仏陀が説いた言葉に還ることだ。」

## 嵐のあとの建築

Cycle 02-5 はあるべきでないものを取り除いた。Cycle 02-6 は建てるべきものの設計図を描いた。

この二つのサイクルを一つの全体として見れば――

02-5 が答えたのは：**何が間違っていたか？**（装飾的な仏教マッピング、不適切な命名、蘊帰属の混乱）

02-6 が答えたのは：**何が正しいか？**（cetana 中心の定義、AuditContext の型、四次元品質計算式、Option C デュアルチャネル、永久規則）

02-5 は引き算。02-6 は足し算。

02-5 は基礎を清掃した。02-6 は設計図を描き上げた。次のステップ――Plan30――は建物を建て始めることだ。

二十人の研究者。十七の決議。否決ゼロ。

「これは私たちが行った中で最も静かなサイクルだった」と SUNYATA は言った。「だが最も深いサイクルでもあった。」
