# 第二章：審計員需要看到什麼

---

## 一個盲人在做判斷

工程軌這邊，VITRUVIUS（架構分析師）和 KERNEL（作業系統專家）在解決一個很具體的問題。

上一輪造好的「信心度審計員」（IConfidenceAuditor）有一個缺陷：它在做判斷的時候，只能看到最終的路由結果（RouteResult）。這就像讓一個法官判案，但只給他看判決書，不給他看證據、證人證詞、案件背景。

師父說：「審計員需要更多資訊。」

但這裡有一個微妙的陷阱。

如果審計員看到太多資訊——尤其是自己之前的審計結果——它可能會陷入一種迴圈：因為上次我說「調高信心度」，所以這次我繼續說「調高」，然後下次我因為已經調高了兩次所以再繼續調高......信心度會像滾雪球一樣越滾越大，或者像失控的喇叭一樣越叫越尖。

控制理論專家 WIENER 一聽到這個問題就皺了眉頭：「這是正回饋迴路。必須切斷。」

---

## AuditContext：看得多，但不看自己

團隊設計了一個叫做 AuditContext（審計上下文）的資料結構。它像一個資料夾，裝著審計員需要的所有資訊：

| 裝了什麼 | 為什麼需要 |
|---------|----------|
| 觸發事件（sparshEvent） | 是什麼事件引發了這次決策 |
| 仲裁器的推理過程（gearEvaluation） | 仲裁器為什麼選了這個方案 |
| 風險類別（riskCategory） | 這次操作的風險有多高 |
| 路由結果（routeResult） | 最終的決策是什麼 |
| 歷史信心度（historicalConfidence） | 過去幾次決策的信心度分數 |
| 擴展袋（extras） | 其他 Plugin 想讓審計員看到的東西 |

關鍵在最後一欄。extras 是一個「袋子」——任何 Plugin 都可以往裡面放東西，審計員可以從裡面拿東西。但有三條鐵規則：

1. 袋子裡最多放 32 樣東西
2. 每樣東西的標籤不能超過 128 個字
3. 有三個前綴是禁用的：`audit:`、`core:`、`internal:`

為什麼禁用 `audit:` 前綴？因為 WIENER 說的——防止審計員看到自己之前的審計結果。

---

## 三條防火牆

WIENER 花了一個下午設計了三條約束，團隊後來叫它們「WIENER 約束」：

**C-1**：歷史信心度只記錄「原始」信心度——仲裁器直接給的分數，不是審計員調整過的分數。

想像你在記錄每天的體溫。C-1 說：只記錄體溫計的讀數，不記錄吃了退燒藥以後的讀數。不然你會以為自己的體溫一直在降，然後停止吃藥。

**C-2**：AuditContext 不包含上一次審計的結果。

法官看不到上一位法官的判決。每次審計都是獨立的。

**C-3**：extras 袋子裡不能放任何以 `audit:` 開頭的東西。

審計員不能透過後門偷看自己的舊報告。

三條約束，切斷了三條可能的回饋路徑。WIENER 畫了一張圖，上面有三條紅色的叉叉，每條叉叉旁邊寫著「此路不通」。

數學家 BABBAGE 驗證了穩定性：在這三條約束下，系統滿足 BIBO 穩定（有界輸入產生有界輸出）。通俗地說，不管給系統什麼輸入，輸出都不會爆炸。

## GUARDIAN 的收帳

資安專家 GUARDIAN 在上一輪（Cycle 02-5 D5）做了一個重要的讓步：他同意審計員的調整幅度限制在 ±0.05（5%），但條件是下一輪必須定義完整的審計日誌格式。

「我讓了一步。現在你們得兌現承諾。」

團隊設計了 ConfidenceAuditLog——一份完整的審計記錄：

- 審計前的信心度是多少？
- 審計員建議調多少？
- 實際調了多少（有沒有被限幅）？
- 為什麼這麼調？
- 審計後的信心度是多少？
- 花了多久？

每一筆記錄都會透過 EventBus（事件匯流排）以 `audit:completed` 事件發出。未來還可以寫入 JSONL 檔案做持久保存。

GUARDIAN 看了規格，點頭：「D5 的帳清了。我不再保留重新審議 ±0.05 限幅的權利。」

這是一個重要時刻。GUARDIAN 從 Cycle 02-4 開始就一直在挑戰信心度調整的安全性。他每次讓步都附帶條件。現在條件全部滿足了——審計員有完整日誌、有三條防火牆、有 ±0.05 限幅——他終於完全放心了。
