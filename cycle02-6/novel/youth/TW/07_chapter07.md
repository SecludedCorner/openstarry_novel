# 第七章：藍圖完成

## Plan30

十七項決議定案後，工程藍圖自然浮現。

VITRUVIUS 和 ARCHIMEDES 把所有規格整理成 Plan30——下一步工程計畫的完整設計藍圖。四個波次（Wave），從小到大：

**Wave 1（~120 行程式碼）**：造一個預設的品質監控器 Plugin。把四維公式寫成真正的程式碼，訂閱六個事件，在每個事件到來時用 O(1) 的效率更新分數。

**Wave 2（~80 行程式碼）**：把品質監控器接進 ManoAggregator。方案 C 的 Layer 3 佈線——品質分數透過回調函數傳入，閾值按公式調整。

**Wave 3（~60 行程式碼）**：型別定義和事件常量。ConfidenceAuditLog SDK 型別、AgentEventType 常量提升、extras 相關的事件定義。

**Wave 4（~30 行程式碼，可選）**：造一個「直通審計員」——什麼都不調、只記日誌的 PassthroughAuditor。它是 Phase 0 的實現，用來驗證整個審計管道是通的。

總計大約 260 到 290 行生產程式碼，加上 130 行測試程式碼。不到 420 行。

成功標準只有一條：**Model Delta 五層（L0~L4）全部有實際信號路徑**，以整合測試驗證。

換句話說，Plan30 完成之後，從最底層的事件流（L0）到最頂層的審計輸出（L4），每一層都不再是空殼——都有真正的程式碼在運作。

## 下一步：Plan31

Plan30 之後是 Plan31。

Plan31 將把 AuditContext 型別落地到運行路徑中——不再只是 SDK 的型別定義，而是真正在每次路由時組裝 AuditContext、傳給審計員、記錄日誌。

預計範圍大約 350 行程式碼。包括：
- AuditContext 組裝（Core 負責）
- Default ThresholdAuditor（Phase 1 規則式審計員）
- Audit Trail JSONL 持久化

Plan30 造管道。Plan31 讓水流過管道。

## 哲學的收穫

回到哲學軌。這一輪的哲學研究不像上一輪那樣改變了系統的面貌（Cycle 02-5 清理了大量裝飾性佛學映射），但它建立了更深的理解。

**行蘊不再是「剩餘類別」。** 它有了精確的定義：以 cetana（意圖）為中心的造作過程。它不是「五蘊裡裝不下的東西的垃圾桶」，而是「主動改變世界的意志力量」。

**三準則成為永久工具。** 未來每次有人問「這個功能屬於哪個蘊」，都可以用三準則來判定：造作性、意圖驅動、環境改變。不再需要每次都從頭辯論。

**心所問題一次解決。** 師父的八點規則加上蘊歸屬五原則，徹底解決了「佛學術語和工程命名怎麼共處」的問題。參考但不照搬。借鑑但不混淆。

ASANGA 在最後說：「這一輪我對自己學派最嚴厲。但這不是背叛唯識——是回到唯識之前。回到佛陀說的話。」
