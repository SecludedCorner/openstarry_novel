# 序章：暴風雨之後

---

上一輪研究（Cycle 02-5）像一場暴風雨。

二十位研究者在五天內做出了 29 項決議，師父親自覆議推翻了其中兩項，整個團隊把七份文件翻了個底朝天——移除裝飾性的佛學標籤、改名、拆分、重組。有人形容那五天像是大掃除，也有人說更像手術。

現在，手術室已經清理乾淨。病人恢復良好。

Cycle 02-6 的氣氛完全不同。

## 兩封信

師父這次沒有發火。他發了兩封信。

第一封是工程指示。上一輪的工程交付（Plan29）新增了兩個重要元件：一個是「品質監控器」（LoopQualityMonitor），負責觀察 AI Agent 的認知迴路運作得好不好；另一個是「信心度審計員」（ConfidenceAuditor），負責在 Agent 做決定之前再檢查一次。

兩個元件都已經造好了骨架，但師父說：「骨架不夠。我需要完整的規格。」

他列了六個問題：品質怎麼算？審計員看到什麼資訊？兩者怎麼整合？事件怎麼訂閱？權重怎麼配？這些問題的答案，就是下一步工程計畫（Plan30）的設計藍圖。

第二封信不同。這封信聊的不是工程，而是哲學。

師父說：「我期待你們以經（如是我聞）深掘行蘊。」

行蘊是五蘊之一。在 OpenStarry 的架構中，五蘊是整個系統的骨幹——色蘊接收外界輸入、受蘊感受好壞、想蘊辨認是什麼、行蘊執行行動、識蘊做判斷。上一輪研究花了大量時間搞清楚五蘊在程式碼中怎麼運作，但行蘊一直是最簡單的那個——只有一個子介面（ITool），負責呼叫工具。

師父覺得這太淺了。他還特別點名了一個問題：唯識學派（佛教的一個哲學學派）把大部分「心所」（心理功能）歸給行蘊，這「非常有問題」。

# 第一章：回到源頭

---

## 芭蕉樹的比喻

哲學軌的工作由兩位佛學專家領銜：NAGARJUNA（中觀學派哲學家）和 ASANGA（唯識學派佛學家）。

有趣的是，師父批評的正是唯識學派的做法，而 ASANGA 就是團隊中的唯識專家。SUNYATA 有意安排這個組合——讓被批評的學派自己去審視問題。

他們從最基本的地方開始：翻開原始經典。

佛陀在《雜阿含經》（SN 22.56）中定義行蘊時，用了一個很具體的說法：「六思身」——對色、聲、香、味、觸、法六種對象的意圖（cetana）。不是「所有心理活動的雜物箱」，而是明確指向「意圖」。

這就像你問「什麼是行動」，佛陀的回答是「行動就是有意圖地去做某件事」。不是「行動就是除了感覺和辨認以外的所有心理活動」。

然後他們找到了芭蕉樹的比喻（SN 22.95）。佛陀說行蘊就像芭蕉樹的樹幹——一層層剝開，每一層都是薄薄的纖維，最後什麼堅實的核心都沒有。它不是一個「東西」，而是一個不斷造作的「過程」。

NAGARJUNA 在白板上畫了一棵芭蕉樹。ASANGA 看了很久，然後說了一句話：

「唯識學派把 49 個心所塞進行蘊，就像把整個果園塞進一棵芭蕉樹。這不是佛陀的意思。」

---

## 三個特質

從原始經典中，他們歸納出行蘊的三個核心特質：

**第一，cetana 中心性。** 行蘊的定義就是「意圖」——六種思身。不是別的。

**第二，造作一切有為法。** 行蘊不只是「做事」。它造作——也就是創造條件、改變狀態、影響未來。經典說行蘊「造作色的被條件化狀態」（SN 22.79），意思是行蘊甚至能影響其他四個蘊。

**第三，無核心的動態過程。** 芭蕉喻。行蘊不是一個固定的「東西」，而是一個不停運轉的過程。

LINNAEUS（分類學專家）聽完這三個特質後，提出了一個實用的工具：「行蘊屬性判定三準則」。如果你想知道某個功能是不是屬於行蘊，問三個問題：

1. **造作性**：它會創造或改變狀態嗎？
2. **意圖驅動**：它是由 cetana（意圖）驅動的嗎？
3. **環境改變**：它會改變後續的認知條件嗎？

然後 PENROSE（量子意識研究專家）補了一句話，把整件事說得更清楚：

> 行蘊 = WRITE（主動改變世界）
> 識蘊 = READ（被動評估世界）

這個區分簡單到令人驚訝。在程式設計中，WRITE 和 READ 是最基本的操作。行蘊就是寫入——它改變東西。識蘊就是讀取——它觀察東西。

回到 OpenStarry 的程式碼，這完美對應：ITool 和 ISlashCommand（行蘊）執行動作、改變環境；IGuide 和 IVolition（識蘊）評估情況、做出判斷。

---

## 唯識的問題

ASANGA 親自帶領了對唯識學派心所系統的批判。他把 51 個心所（心理功能）逐一分析，看看唯識學派把哪些塞進了行蘊。

結果令人震驚。唯識學派把 49 個心所歸給行蘊——只留了「受」和「想」給受蘊和想蘊。ASANGA 的分析結論是：

- 真正屬於行蘊的：只有 7 個（14%），比如「意欲」（chanda）和「精進」（virya）
- 已經在 OpenStarry 中有正確歸屬的：12 個（24%）
- 可能應該屬於其他蘊的：18 個（37%）——功能上更接近識蘊或受蘊
- 需要進一步研究的：14 個（28%）

換句話說，唯識學派把大量不屬於行蘊的東西塞進了行蘊。這就是師父說的「非常有問題」。

「心所列表是有用的，」ASANGA 說。「但它是論典的產物，不是佛陀的原話。我們可以參考它的功能描述，但不能讓它決定東西該放在哪裡。」

這就像一本老字典——詞條的定義有參考價值，但分類方式已經過時了。你不會因為老字典把「番茄」分在「蔬菜」類就堅持番茄不是水果。

# 第二章：審計員需要看到什麼

---

## 一個盲人在做判斷

工程軌這邊，VITRUVIUS（架構分析師）和 KERNEL（作業系統專家）在解決一個很具體的問題。

上一輪造好的「信心度審計員」（IConfidenceAuditor）有一個缺陷：它在做判斷的時候，只能看到最終的路由結果（RouteResult）。這就像讓一個法官判案，但只給他看判決書，不給他看證據、證人證詞、案件背景。

師父說：「審計員需要更多資訊。」

但這裡有一個微妙的陷阱。

如果審計員看到太多資訊——尤其是自己之前的審計結果——它可能會陷入一種迴圈：因為上次我說「調高信心度」，所以這次我繼續說「調高」，然後下次我因為已經調高了兩次所以再繼續調高......信心度會像滾雪球一樣越滾越大，或者像失控的喇叭一樣越叫越尖。

控制理論專家 WIENER 一聽到這個問題就皺了眉頭：「這是正回饋迴路。必須切斷。」

---

## AuditContext：看得多，但不看自己

團隊設計了一個叫做 AuditContext（審計上下文）的資料結構。它像一個資料夾，裝著審計員需要的所有資訊：

| 裝了什麼 | 為什麼需要 |
|---------|----------|
| 觸發事件（sparshEvent） | 是什麼事件引發了這次決策 |
| 仲裁器的推理過程（gearEvaluation） | 仲裁器為什麼選了這個方案 |
| 風險類別（riskCategory） | 這次操作的風險有多高 |
| 路由結果（routeResult） | 最終的決策是什麼 |
| 歷史信心度（historicalConfidence） | 過去幾次決策的信心度分數 |
| 擴展袋（extras） | 其他 Plugin 想讓審計員看到的東西 |

關鍵在最後一欄。extras 是一個「袋子」——任何 Plugin 都可以往裡面放東西，審計員可以從裡面拿東西。但有三條鐵規則：

1. 袋子裡最多放 32 樣東西
2. 每樣東西的標籤不能超過 128 個字
3. 有三個前綴是禁用的：`audit:`、`core:`、`internal:`

為什麼禁用 `audit:` 前綴？因為 WIENER 說的——防止審計員看到自己之前的審計結果。

---

## 三條防火牆

WIENER 花了一個下午設計了三條約束，團隊後來叫它們「WIENER 約束」：

**C-1**：歷史信心度只記錄「原始」信心度——仲裁器直接給的分數，不是審計員調整過的分數。

想像你在記錄每天的體溫。C-1 說：只記錄體溫計的讀數，不記錄吃了退燒藥以後的讀數。不然你會以為自己的體溫一直在降，然後停止吃藥。

**C-2**：AuditContext 不包含上一次審計的結果。

法官看不到上一位法官的判決。每次審計都是獨立的。

**C-3**：extras 袋子裡不能放任何以 `audit:` 開頭的東西。

審計員不能透過後門偷看自己的舊報告。

三條約束，切斷了三條可能的回饋路徑。WIENER 畫了一張圖，上面有三條紅色的叉叉，每條叉叉旁邊寫著「此路不通」。

數學家 BABBAGE 驗證了穩定性：在這三條約束下，系統滿足 BIBO 穩定（有界輸入產生有界輸出）。通俗地說，不管給系統什麼輸入，輸出都不會爆炸。

---

## GUARDIAN 的收帳

資安專家 GUARDIAN 在上一輪（Cycle 02-5 D5）做了一個重要的讓步：他同意審計員的調整幅度限制在 ±0.05（5%），但條件是下一輪必須定義完整的審計日誌格式。

「我讓了一步。現在你們得兌現承諾。」

團隊設計了 ConfidenceAuditLog——一份完整的審計記錄：

- 審計前的信心度是多少？
- 審計員建議調多少？
- 實際調了多少（有沒有被限幅）？
- 為什麼這麼調？
- 審計後的信心度是多少？
- 花了多久？

每一筆記錄都會透過 EventBus（事件匯流排）以 `audit:completed` 事件發出。未來還可以寫入 JSONL 檔案做持久保存。

GUARDIAN 看了規格，點頭：「D5 的帳清了。我不再保留重新審議 ±0.05 限幅的權利。」

這是一個重要時刻。GUARDIAN 從 Cycle 02-4 開始就一直在挑戰信心度調整的安全性。他每次讓步都附帶條件。現在條件全部滿足了——審計員有完整日誌、有三條防火牆、有 ±0.05 限幅——他終於完全放心了。

# 第三章：四維品質

---

## 怎麼知道 AI 做得好不好？

想像你在教一個學生解數學題。你怎麼判斷他學得好不好？

不能只看正確率——也許他只是碰巧猜對的。你會想知道更多：他的方法穩不穩定？他會不會一直換策略？他是越做越好還是原地踏步？

OpenStarry 的品質監控器（LoopQualityMonitor）面對同樣的問題。它需要評估 AI Agent 的認知迴路——Agent 接收訊息、思考、做決定、執行、再接收下一個訊息——這個循環的品質怎麼樣？

WIENER 和 ATHENA（AI/ML 專家）帶領團隊設計了四個維度：

---

## 四把尺

**第一把尺：一致性（coherence）**

公式：`1 - 切換次數 / (窗口大小 - 1)`

如果 Agent 一直在同一個「齒輪」（處理模式）下運作——不斷切換就像開車時一直換檔——那表示它知道自己在做什麼。如果它一直在不同的齒輪之間跳來跳去，可能是迷路了。

想像一個人在做菜。他一下子拿起鍋鏟，放下，拿起刀，放下，又拿起鍋鏟——他可能不知道自己在做什麼。但如果他穩穩地用鍋鏟炒了整盤菜，那他很清楚自己的目標。

**第二把尺：效率（efficiency）**

公式：`成功次數 / 嘗試次數`

Agent 提出了多少個動作？其中多少個成功了？如果 Agent 提了 10 個工具呼叫，8 個成功，效率就是 0.8。

這是最直覺的一把尺。就像考試的正確率。

**第三把尺：收斂性（convergence）**

公式：`最長連續成功次數 / 窗口大小`

Agent 有沒有越做越好？如果最近連續成功了 8 次（窗口大小是 10），收斂性就是 0.8。這表示 Agent 找到了正確的方向，正在穩步前進。

就像學生做題——如果最近十題對了八題而且是連續的，比起十題中散布八題對，更能說明他已經掌握了方法。

**第四把尺：穩定性（stability）**

公式：`1 - min(1, 方差 / 0.25)`

Agent 的信心度分數穩不穩定？如果每次做決定的信心度都差不多（方差小），穩定性就高。如果信心度忽高忽低——這次 0.9，下次 0.3，再下次 0.8——系統可能不太穩定。

想像你問一個人「你確定嗎？」如果他每次都說「我 80% 確定」，那他很穩定。如果他一下子說「90%！」一下子說「20%......」，你會擔心他是不是在亂猜。

---

## 滑動窗口

四把尺都在一個「滑動窗口」中計算。窗口大小是 10 個事件。

為什麼是 10？因為太小的窗口會對隨機波動太敏感（看到一次失敗就恐慌），太大的窗口又太遲鈍（Agent 已經出問題了還沒反應過來）。10 是一個平衡點。

而且，在收集到至少 5 個事件之前，監控器不會發出任何品質分數。這叫「暖機期」——就像新車上路前要先暖引擎，避免在資料不足時做出錯誤判斷。

四個維度的分數各乘以 0.25（等權重），加起來就是一個 0 到 1 之間的「複合品質分數」Q。

為什麼等權重？因為目前沒有數據支持任何一個維度比其他維度更重要。PASCAL（決策理論專家）說：「在沒有證據的情況下，最大熵原則告訴我們：給每個可能性相同的權重。」先跑起來，收集數據，未來再調。

---

## 六隻耳朵

監控器需要「聽」EventBus 上的事件才能計算品質。HERACLITUS（運行時動態專家）和 TURING 規劃了兩階段的監聽方案：

第一階段（Phase 1）只聽 6 種事件——剛好夠計算四個維度。就像裝了六隻耳朵，各自負責聽不同的聲音。

第二階段（Phase 2）再加 5 種事件，用來計算更精細的品質指標。

為什麼分兩階段？因為越多事件意味著越多的資源消耗。先用最少的耳朵做到「夠用」，然後再看需不需要更多。

如果某個事件碰巧不存在怎麼辦？監控器不會當機。它會用安全的預設值——比如效率預設為 1.0（假設一切正常），穩定性預設為 0.5（不好不壞）。

HERACLITUS 的原則：「缺失是資訊，不是錯誤。」

# 第四章：兩條獨立的管道

---

## 一個看起來很簡單的問題

品質監控器算出了品質分數。審計員可以調整信心度。但它們要怎麼影響 Agent 的決策？

在 OpenStarry 中，Agent 做決策的核心邏輯是一個比較：

```
如果（信心度 > 閾值）→ 用仲裁器推薦的方案
否則 → 用預設方案
```

信心度是仲裁器對自己推薦方案的把握程度。閾值是系統設定的「最低標準」。信心度超過閾值，就採用推薦方案；沒超過，就走安全的預設路線。

現在有兩個新角色想參與這個比較：
- **審計員**想調整信心度（「我覺得仲裁器太自信了，信心度應該降一點」）
- **品質監控器**想說「最近 Agent 的表現很好，標準可以放鬆一點」

問題是：它們應該怎麼介入？

---

## 方案 C：各管各的

團隊考慮了三個方案。最後全票通過的是方案 C——兩條獨立的管道。

**管道一（Layer 2）**：審計員調整信心度。

```
新信心度 = 舊信心度 + 審計員的調整量（限制在 ±0.05）
```

**管道二（Layer 3）**：品質監控器調整閾值。

```
新閾值 = max(地板值, 舊閾值 × (1 - 0.10 × 品質分數))
```

然後用新的信心度和新的閾值做比較。

為什麼這是好的設計？

**第一，獨立。** 兩條管道互不干擾。審計員不知道品質分數，品質監控器不知道審計結果。就像兩個不同部門各自做自己的調整——財務部調預算（信心度），人資部調標準（閾值）——互不越權。

**第二，穩定。** 因為兩條管道沒有交叉項（不會互相影響），BABBAGE 可以用數學證明系統是 BIBO 穩定的——有界輸入一定產生有界輸出。

**第三，可控。** 每條管道都有限幅：審計員最多調 ±0.05，品質監控器的影響係數 α 固定為 0.10。就像汽車的方向盤有一個最大轉角——你可以轉，但不能把車轉 180 度。

**第四，保守退化。** 如果沒有品質監控器，品質分數 Q = 0，閾值完全不變。如果沒有審計員，調整量 = 0，信心度完全不變。系統在缺少任何一個元件時，行為跟沒有它們的時候一模一樣——不會突然變得更危險。

WIENER 看了設計後說：「這是我見過最乾淨的分離。」

---

## 品質分數怎麼傳過去

品質監控器算出了分數，怎麼傳給 ManoAggregator（負責做最終決策的元件）？

團隊設計了一個雙通道方案：

**通道一（pull）**：ManoAggregator 自己去「拉」品質分數。它在需要做決策的時候，呼叫一個 `loopQualityFn()` 回調函數，品質監控器把最新的分數傳回來。

**通道二（push）**：品質監控器主動「推」品質資訊到 extras 袋子裡。它透過 EventBus 發出 `audit:context_contribute` 事件，把四維分數和複合分數放進袋子，讓審計員也能看到。

為什麼要兩個通道？因為兩個消費者的需求不同。ManoAggregator 需要即時的數字（用來算閾值），審計員需要豐富的背景資訊（用來做判斷）。

LEIBNIZ（多代理合作專家）特別強調了一點：「品質分數的生命週期是 per-routing-cycle。每次路由結束，袋子就清空。不累積。」

這就像每次開會前發一份新的議程表。上一次開會的議程不會留在桌上影響這次開會。

---

## 五層模型

加上之前已經有的 Layer 0（EventBus 事件）、Layer 1（煩惱增益）和 Layer 4（緊急感受），整個決策流程現在有五層：

```
L0: EventBus — 基礎事件流
L1: Klesha（煩惱） — 四個情緒模組調整閾值
L4: Vedana（受蘊緊急）— 緊急感受直接降低閾值
L3: LoopQuality — 品質分數微調閾值
L2: Audit — 審計員微調信心度
→ 最終比較 → 路由決策
```

Layer 順序很講究：L4（緊急感受）優先，因為它處理的是安全關鍵的情況；L3（品質）次之；最後才是 L2 和最終比較。

這就像一棟建築物的安全檢查：先看有沒有火災（L4），再看建築品質（L3），最後才看裝修細節（L2）。優先級從性命攸關到微調優化。

# 第五章：師父的三輪指示

---

## 心所的問題

就在工程軌和哲學軌各自忙碌的時候，師父又發了指示。

起因是哲學軌的心所分析。ASANGA 整理了 51 個心所（心理功能）的重新歸屬建議。這份報告在交叉審閱時被呈給了師父，師父連續給了三輪回應。

每一輪都比上一輪更清楚。就像對焦——第一輪看到大致的形狀，第二輪看到細節，第三輪看到全貌。

---

## 第一輪：劃清界線

師父說：「心所是論典的產物，不是佛陀說的。」

這是一個很重要的區分。佛陀在原始經典中說了五蘊、十二因緣、八正道。但「51 個心所」這個分類系統是後來的論典（阿毗達磨）學者整理出來的——就像佛陀寫了一本小說，然後幾百年後的學者替小說裡的每個角色做了詳細的分類卡片。分類卡片有參考價值，但它不是原始小說的一部分。

師父的結論：Plugin 的命名不能用心所的梵文名字。

## 第二輪：保留價值

但師父不是說心所沒用。第二輪指示中，他承認心所的「功能描述」有參考價值。

比如，心所「精進」（virya）描述了「持續不懈地投入善法」的功能。你可以參考這個功能來設計一個 Plugin——但 Plugin 不能叫 VīryaPlugin，而應該叫 PersistencePlugin 或 RetryPlugin 之類的工程名稱。

「可以說『參考精進的涵義，設計了重試 Plugin』，」師父說。「但不能說『重試 Plugin = 精進』。」

## 第三輪：解放

第三輪是最深刻的。師父說：「Plugin 不等於心所。」

這聽起來很簡單，但它的含義很大。

在唯識學派中，每個心所有固定的蘊歸屬——比如「精進」屬於行蘊，「慧」屬於識蘊。如果 Plugin = 心所，那 Plugin 的蘊歸屬也被鎖死了。

但 Plugin ≠ 心所。一個 Plugin 可能同時有行蘊和識蘊的功能——它既做判斷（READ），又執行動作（WRITE）。在心所系統裡這是不允許的（每個心所只能屬於一個蘊），但在 Plugin 系統裡完全沒問題。

師父的話：「命名解耦帶來歸屬自由。」

不用心所的名字命名 Plugin，反而讓 Plugin 更自由了——它可以跨越多個蘊，不受論典的固定分類束縛。

---

## 八點永久規則

三輪指示被整理成八條永久規則：

1. **心所是論典產物**，不是原始經典的內容
2. **心所功能有參考價值**，可以作為 Plugin 設計的靈感
3. **可以說明參考來源**：「參考某某心所的涵義，設計某某 Plugin」
4. **Plugin 用工程名稱**，不得使用心所梵文名
5. **梵文術語只用原始經典的**——五蘊（skandha）、觸（sparsha）可以，但不能用論典創造的心所梵文名
6. **心所分類不決定蘊歸屬**——「唯識說精進屬行蘊」不能作為 Plugin 歸屬的依據
7. **既有決策不受影響**——之前已經確認過的 Plugin 歸屬繼續有效
8. **Plugin 可以跨多蘊**——因為 Plugin ≠ 心所，自然不受限

這八條規則和哲學軌的「蘊歸屬五原則」一起，構成了 OpenStarry 未來所有命名和歸屬問題的永久判定標準。

NAGARJUNA 說了一句話：「上一輪我們學會了什麼不該做。這一輪我們學會了為什麼不該做。」

# 第七章：藍圖完成

## Plan30

十七項決議定案後，工程藍圖自然浮現。

VITRUVIUS 和 ARCHIMEDES 把所有規格整理成 Plan30——下一步工程計畫的完整設計藍圖。四個波次（Wave），從小到大：

**Wave 1（~120 行程式碼）**：造一個預設的品質監控器 Plugin。把四維公式寫成真正的程式碼，訂閱六個事件，在每個事件到來時用 O(1) 的效率更新分數。

**Wave 2（~80 行程式碼）**：把品質監控器接進 ManoAggregator。方案 C 的 Layer 3 佈線——品質分數透過回調函數傳入，閾值按公式調整。

**Wave 3（~60 行程式碼）**：型別定義和事件常量。ConfidenceAuditLog SDK 型別、AgentEventType 常量提升、extras 相關的事件定義。

**Wave 4（~30 行程式碼，可選）**：造一個「直通審計員」——什麼都不調、只記日誌的 PassthroughAuditor。它是 Phase 0 的實現，用來驗證整個審計管道是通的。

總計大約 260 到 290 行生產程式碼，加上 130 行測試程式碼。不到 420 行。

成功標準只有一條：**Model Delta 五層（L0~L4）全部有實際信號路徑**，以整合測試驗證。

換句話說，Plan30 完成之後，從最底層的事件流（L0）到最頂層的審計輸出（L4），每一層都不再是空殼——都有真正的程式碼在運作。
