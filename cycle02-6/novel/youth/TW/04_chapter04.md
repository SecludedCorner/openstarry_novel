# 第四章：兩條獨立的管道

---

## 一個看起來很簡單的問題

品質監控器算出了品質分數。審計員可以調整信心度。但它們要怎麼影響 Agent 的決策？

在 OpenStarry 中，Agent 做決策的核心邏輯是一個比較：

```
如果（信心度 > 閾值）→ 用仲裁器推薦的方案
否則 → 用預設方案
```

信心度是仲裁器對自己推薦方案的把握程度。閾值是系統設定的「最低標準」。信心度超過閾值，就採用推薦方案；沒超過，就走安全的預設路線。

現在有兩個新角色想參與這個比較：
- **審計員**想調整信心度（「我覺得仲裁器太自信了，信心度應該降一點」）
- **品質監控器**想說「最近 Agent 的表現很好，標準可以放鬆一點」

問題是：它們應該怎麼介入？

---

## 方案 C：各管各的

團隊考慮了三個方案。最後全票通過的是方案 C——兩條獨立的管道。

**管道一（Layer 2）**：審計員調整信心度。

```
新信心度 = 舊信心度 + 審計員的調整量（限制在 ±0.05）
```

**管道二（Layer 3）**：品質監控器調整閾值。

```
新閾值 = max(地板值, 舊閾值 × (1 - 0.10 × 品質分數))
```

然後用新的信心度和新的閾值做比較。

為什麼這是好的設計？

**第一，獨立。** 兩條管道互不干擾。審計員不知道品質分數，品質監控器不知道審計結果。就像兩個不同部門各自做自己的調整——財務部調預算（信心度），人資部調標準（閾值）——互不越權。

**第二，穩定。** 因為兩條管道沒有交叉項（不會互相影響），BABBAGE 可以用數學證明系統是 BIBO 穩定的——有界輸入一定產生有界輸出。

**第三，可控。** 每條管道都有限幅：審計員最多調 ±0.05，品質監控器的影響係數 α 固定為 0.10。就像汽車的方向盤有一個最大轉角——你可以轉，但不能把車轉 180 度。

**第四，保守退化。** 如果沒有品質監控器，品質分數 Q = 0，閾值完全不變。如果沒有審計員，調整量 = 0，信心度完全不變。系統在缺少任何一個元件時，行為跟沒有它們的時候一模一樣——不會突然變得更危險。

WIENER 看了設計後說：「這是我見過最乾淨的分離。」

## 品質分數怎麼傳過去

品質監控器算出了分數，怎麼傳給 ManoAggregator（負責做最終決策的元件）？

團隊設計了一個雙通道方案：

**通道一（pull）**：ManoAggregator 自己去「拉」品質分數。它在需要做決策的時候，呼叫一個 `loopQualityFn()` 回調函數，品質監控器把最新的分數傳回來。

**通道二（push）**：品質監控器主動「推」品質資訊到 extras 袋子裡。它透過 EventBus 發出 `audit:context_contribute` 事件，把四維分數和複合分數放進袋子，讓審計員也能看到。

為什麼要兩個通道？因為兩個消費者的需求不同。ManoAggregator 需要即時的數字（用來算閾值），審計員需要豐富的背景資訊（用來做判斷）。

LEIBNIZ（多代理合作專家）特別強調了一點：「品質分數的生命週期是 per-routing-cycle。每次路由結束，袋子就清空。不累積。」

這就像每次開會前發一份新的議程表。上一次開會的議程不會留在桌上影響這次開會。

## 五層模型

加上之前已經有的 Layer 0（EventBus 事件）、Layer 1（煩惱增益）和 Layer 4（緊急感受），整個決策流程現在有五層：

```
L0: EventBus — 基礎事件流
L1: Klesha（煩惱） — 四個情緒模組調整閾值
L4: Vedana（受蘊緊急）— 緊急感受直接降低閾值
L3: LoopQuality — 品質分數微調閾值
L2: Audit — 審計員微調信心度
→ 最終比較 → 路由決策
```

Layer 順序很講究：L4（緊急感受）優先，因為它處理的是安全關鍵的情況；L3（品質）次之；最後才是 L2 和最終比較。

這就像一棟建築物的安全檢查：先看有沒有火災（L4），再看建築品質（L3），最後才看裝修細節（L2）。優先級從性命攸關到微調優化。
