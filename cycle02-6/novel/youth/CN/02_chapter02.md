# 第二章：审计员需要看到什么

---

## 一个盲人在做判断

工程轨这边，VITRUVIUS（架构分析师）和 KERNEL（操作系统专家）在解决一个很具体的问题。

上一轮造好的「信心度审计员」（IConfidenceAuditor）有一个缺陷：它在做判断的时候，只能看到最终的路由结果（RouteResult）。这就像让一个法官判案，但只给他看判决书，不给他看证据、证人证词、案件背景。

师父说：「审计员需要更多信息。」

但这里有一个微妙的陷阱。

如果审计员看到太多信息——尤其是自己之前的审计结果——它可能会陷入一种回路：因为上次我说「调高信心度」，所以这次我继续说「调高」，然后下次我因为已经调高了两次所以再继续调高......信心度会像滚雪球一样越滚越大，或者像失控的喇叭一样越叫越尖。

控制理论专家 WIENER 一听到这个问题就皱了眉头：「这是正回馈回路。必须切断。」

---

## AuditContext：看得多，但不看自己

团队设计了一个叫做 AuditContext（审计上下文）的数据结构。它像一个文件夹，装着审计员需要的所有信息：

| 装了什么 | 为什么需要 |
|---------|----------|
| 触发事件（sparshEvent） | 是什么事件引发了这次决策 |
| 仲裁器的推理过程（gearEvaluation） | 仲裁器为什么选了这个方案 |
| 风险类别（riskCategory） | 这次操作的风险有多高 |
| 路由结果（routeResult） | 最终的决策是什么 |
| 历史信心度（historicalConfidence） | 过去几次决策的信心度分数 |
| 扩展袋（extras） | 其他 Plugin 想让审计员看到的东西 |

关键在最后一栏。extras 是一个「袋子」——任何 Plugin 都可以往里面放东西，审计员可以从里面拿东西。但有三条铁规则：

1. 袋子里最多放 32 样东西
2. 每样东西的标签不能超过 128 个字
3. 有三个前缀是禁用的：`audit:`、`core:`、`internal:`

为什么禁用 `audit:` 前缀？因为 WIENER 说的——防止审计员看到自己之前的审计结果。

---

## 三条防火墙

WIENER 花了一个下午设计了三条约束，团队后来叫它们「WIENER 约束」：

**C-1**：历史信心度只记录「原始」信心度——仲裁器直接给的分数，不是审计员调整过的分数。

想象你在记录每天的体温。C-1 说：只记录体温计的读数，不记录吃了退烧药以后的读数。不然你会以为自己的体温一直在降，然后停止吃药。

**C-2**：AuditContext 不包含上一次审计的结果。

法官看不到上一位法官的判决。每次审计都是独立的。

**C-3**：extras 袋子里不能放任何以 `audit:` 开头的东西。

审计员不能透过后门偷看自己的旧报告。

三条约束，切断了三条可能的回馈路径。WIENER 画了一张图，上面有三条红色的叉叉，每条叉叉旁边写着「此路不通」。

数学家 BABBAGE 验证了稳定性：在这三条约束下，系统满足 BIBO 稳定（有界输入产生有界输出）。通俗地说，不管给系统什么输入，输出都不会爆炸。

## GUARDIAN 的收帐

资安专家 GUARDIAN 在上一轮（Cycle 02-5 D5）做了一个重要的让步：他同意审计员的调整幅度限制在 ±0.05（5%），但条件是下一轮必须定义完整的审计日志格式。

「我让了一步。现在你们得兑现承诺。」

团队设计了 ConfidenceAuditLog——一份完整的审计记录：

- 审计前的信心度是多少？
- 审计员建议调多少？
- 实际调了多少（有没有被限幅）？
- 为什么这么调？
- 审计后的信心度是多少？
- 花了多久？

每一笔记录都会透过 EventBus（事件汇流排）以 `audit:completed` 事件发出。未来还可以写入 JSONL 文件做持久保存。

GUARDIAN 看了规格，点头：「D5 的帐清了。我不再保留重新审议 ±0.05 限幅的权利。」

这是一个重要时刻。GUARDIAN 从 Cycle 02-4 开始就一直在挑战信心度调整的安全性。他每次让步都附带条件。现在条件全部满足了——审计员有完整日志、有三条防火墙、有 ±0.05 限幅——他终于完全放心了。
