# 第四章：两条独立的管道

---

## 一个看起来很简单的问题

品质监控器算出了品质分数。审计员可以调整信心度。但它们要怎么影响 Agent 的决策？

在 OpenStarry 中，Agent 做决策的核心逻辑是一个比较：

```
如果（信心度 > 阈值）→ 用仲裁器推荐的方案
否则 → 用预设方案
```

信心度是仲裁器对自己推荐方案的把握程度。阈值是系统设定的「最低标准」。信心度超过阈值，就采用推荐方案；没超过，就走安全的预设路线。

现在有两个新角色想参与这个比较：
- **审计员**想调整信心度（「我觉得仲裁器太自信了，信心度应该降一点」）
- **品质监控器**想说「最近 Agent 的表现很好，标准可以放松一点」

问题是：它们应该怎么介入？

---

## 方案 C：各管各的

团队考虑了三个方案。最后全票通过的是方案 C——两条独立的管道。

**管道一（Layer 2）**：审计员调整信心度。

```
新信心度 = 旧信心度 + 审计员的调整量（限制在 ±0.05）
```

**管道二（Layer 3）**：品质监控器调整阈值。

```
新阈值 = max(地板值, 旧阈值 × (1 - 0.10 × 品质分数))
```

然后用新的信心度和新的阈值做比较。

为什么这是好的设计？

**第一，独立。** 两条管道互不干扰。审计员不知道品质分数，品质监控器不知道审计结果。就像两个不同部门各自做自己的调整——财务部调预算（信心度），人资部调标准（阈值）——互不越权。

**第二，稳定。** 因为两条管道没有交叉项（不会互相影响），BABBAGE 可以用数学证明系统是 BIBO 稳定的——有界输入一定产生有界输出。

**第三，可控。** 每条管道都有限幅：审计员最多调 ±0.05，品质监控器的影响系数 α 固定为 0.10。就像汽车的方向盘有一个最大转角——你可以转，但不能把车转 180 度。

**第四，保守退化。** 如果没有品质监控器，品质分数 Q = 0，阈值完全不变。如果没有审计员，调整量 = 0，信心度完全不变。系统在缺少任何一个元件时，行为跟没有它们的时候一模一样——不会突然变得更危险。

WIENER 看了设计后说：「这是我见过最干净的分离。」

## 品质分数怎么传过去

品质监控器算出了分数，怎么传给 ManoAggregator（负责做最终决策的元件）？

团队设计了一个双通道方案：

**通道一（pull）**：ManoAggregator 自己去「拉」品质分数。它在需要做决策的时候，调用一个 `loopQualityFn()` 回调函数，品质监控器把最新的分数传回来。

**通道二（push）**：品质监控器主动「推」品质信息到 extras 袋子里。它透过 EventBus 发出 `audit:context_contribute` 事件，把四维分数和复合分数放进袋子，让审计员也能看到。

为什么要两个通道？因为两个消费者的需求不同。ManoAggregator 需要即时的数字（用来算阈值），审计员需要丰富的背景信息（用来做判断）。

LEIBNIZ（多代理合作专家）特别强调了一点：「品质分数的生命周期是 per-routing-cycle。每次路由结束，袋子就清空。不累积。」

这就像每次开会前发一份新的议程表。上一次开会的议程不会留在桌上影响这次开会。

## 五层模型

加上之前已经有的 Layer 0（EventBus 事件）、Layer 1（烦恼增益）和 Layer 4（紧急感受），整个决策流程现在有五层：

```
L0: EventBus — 基础事件流
L1: Klesha（烦恼） — 四个情绪模块调整阈值
L4: Vedana（受蕴紧急）— 紧急感受直接降低阈值
L3: LoopQuality — 品质分数微调阈值
L2: Audit — 审计员微调信心度
→ 最终比较 → 路由决策
```

Layer 顺序很讲究：L4（紧急感受）优先，因为它处理的是安全关键的情况；L3（品质）次之；最后才是 L2 和最终比较。

这就像一栋建筑物的安全检查：先看有没有火灾（L4），再看建筑品质（L3），最后才看装修细节（L2）。优先级从性命攸关到微调优化。
