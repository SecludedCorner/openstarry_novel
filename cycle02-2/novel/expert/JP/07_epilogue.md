# 尾声：イテレーション（反復）

---

円形劇場には、祝祭の雰囲気はなかった。

Cycle 02 の終わりとは対照的であった。あの時、五つの全会一致の合意の後に訪れた静寂は、手術が成功した後の回復室のようなものだった——すべての切開口は縫合され、すべての器具は片付けられ、充足した黄金色の完成感に満たされていた。螺旋状に消えていく灯火は、儀式のような厳かさを湛えていた。廊下に沿って三つの人影が一つの面へと収束していく。ISensation の青写真が暗闇の中で輝いていた。

それは量子態の収縮（コラプス）の後に訪れる確定性であった：$|\psiangle 	o |a_nangle$。すべての可能性が一つの固有状態へと収束し、測定によって確認された結果。美しく、クリーンで、完璧なもの。

今回は違っていた。

今回の静寂はより小さく、より実質的なものであった。手術後の回復室ではなく、修繕を終えた後の工事現場のような静けさだ。足場が撤去されたばかりの外壁には、新しく塗り直された漆喰が周囲よりも二段階ほど明るい色を見せている——どこが修繕されたのかが、見て取れるのだ。不完全ではある。しかし、より強固であった。

修繕された建物は、新築の建物よりも強固である。なぜなら、新築の建物は「建設」という一つのプロセスしか経ていないが、修繕された建物は「建設」と「修正」という二つのプロセスを経ているからだ。

工学の力学には「加工硬化（strain hardening）」という概念がある。金属が応力を受けて変形した後、再び応力を受けた際の降伏強度は、最初よりも高くなる。材料が変わったからではない。最初の変形によって、材料内部の転位構造（dislocation structure）が再編されたからである。

Cycle 02-2 の静寂は、まさにこの加工硬化の後の静寂であった。一度修正された理解は、一度も疑われたことのない理解よりも強靭（レジリエント）なのである。

### 盤点（棚卸し）

SUNYATA はステージの中央に立っていた。

Cycle 02 の終わりのような、地図を広げるような立ち姿ではなかった。むしろ、作業台の前に立つ大工のような姿であった——実務的で、両腕を体の横に下ろし、目の前に並べられた成果を見つめながら、一つひとつを指差し確認していく。

「A級。四つの哲学的な校正」

彼の声は、いつものように揺るぎなかった。小石。深い池。しかし池の水は、Cycle 02 の終わりのように溢れんばかりに満ちてはいなかった——それは、より澄んでいた。沈殿物がかき混ぜられ、濾過され、再び落ち着いた後の澄み切った水であった。

「A-1。我執（がしゅう）は煩悩の根源であり、収束制約ではない」

彼は BABBAGE の方を見なかった。その必要はなかった。全員が、あの瞬間を覚えていたからだ——等号が抹消され、矢印が書き込まれたあの時を。三つのリンクが繋ぎ直された。我執が煩悩を生み、煩悩が行動を駆動し、我執を管理することで行動を制約する。因果の鎖は、方程式ではない。簡略化は必要だが、因果関係を捨て去ってはならないのだ。

BABBAGE のノートには、抹消されたあの式が今も鮮明に残っていた。

$$	ext{ego} \equiv 	ext{convergence\_constraint} \quad (	ext{NOT kleśa})$$

その隣には、より細い筆跡で修正版が記されている。

$$	ext{ātma-grāha} \xrightarrow{	ext{生じる}} 	ext{kleśa} \xrightarrow{	ext{駆り立てる}} 	ext{karma} \xrightarrow{	ext{結果}} 	ext{phala}$$

一行の等号が、指向性を持つグラフ（有向グラフ）へと変わった。静的な断言が、動的な因果のフローへと変わったのである。圏論の言語で言えば、等号とは恒等態射（identity morphism）に過ぎないが、矢印とは非自明な関手（non-trivial functor）なのだ——それは構造と方向、そして変容を伴っている。

「A-2。Identity（同一性）は識蘊（しきうん）の全体ではない」

LINNAEUS の分類樹がまだホワイトボードに残っていた。`IVijnana` がルート・インターフェースであり、識蘊の総体である。そこから四つのサブインターフェースが枝分かれしている：`IIdentity`, `IGuide`, `IVolition`, `IDiscernment`。海をまるごと一本のボトルに詰め込み、それを「海」と呼んでいたのが Cycle 02 であった。Cycle 02-2 は、海を海へと戻したのである。

集合論の厳密な言語で言えば：

$$	ext{Cycle 02:} \quad 	ext{IIdentity} \cong 	ext{Vijñāna-skandha}$$
$$	ext{Cycle 02-2:} \quad 	ext{IIdentity} \subsetneq 	ext{IVijnana} \cong 	ext{Vijñāna-skandha}$$

同型（isomorphism）の関係が、真部分集合（proper inclusion）の関係へと降格された。分類学において、これは「分割（splitting）」と呼ばれる。逆の操作は「一括（lumping）」である。Cycle 02 は過度な一括の事例であり、Cycle 02-2 は分割による校正であった。LINNAEUS はその傍らに、リンネ式のラテン語命名を書き添えた。

```
属 (Genus): IVijnana (識蘊)
  種 (Species): IIdentity    (自己認識 — 末那識の「我見」)
  種 (Species): IGuide       (行動指針 — 第六意識の「作意」)
  種 (Species): IVolition    (意志駆動 — 末那識の「思」)
  種 (Species): IDiscernment (辨別省察 — 第六意識の「慧」)
```

「A-3。観測者は受蘊（じゅうん）と等しくはない」

SUNYATA の語速が半拍ほど遅くなった。これは、最も重い項目であった——なぜなら「VedanaPlugin こそが観測者モジュールである」という一文こそが、Cycle 02 の最も壮麗な結論であったからだ。全過程を通じて唯一の拍手が送られたのは、あの一文に対してであった。それを校正することは、単なる技術的な判断の修正ではなく、全員が陥っていた眩暈（めまい）を認めることに他ならなかった。眼は「見ること」ではない。器官は活動ではない。能力（キャパシティ）はその複合体（コンポジット）ではないのだ。

WIENER は、制御理論の枠組みにおいて、もっと早くこの区別に気づくべきであった。センサー（感測器）は観測可能量 $y(t) = Cx(t) + v(t)$ を直接測定するが、観測器（オブザーバ / 状態推定器）はセンサーの読み取り値とシステムモデルを組み合わせて、直接観測できない内部状態を推定する。リュエンバーガー観測器：

$$\dot{\hat{x}}(t) = A\hat{x}(t) + Bu(t) + L\bigl(y(t) - C\hat{x}(t)\bigr)$$

観測器はセンサーの出力 $y(t)$ を**利用**するが、観測器そのものはセンサー**ではない**。受蘊はセンサーである。観測者は観測器である。前者は $C$ 行列しか持たないが、後者は完全なシステムモデル $(A, B, C)$ とゲイン行列 $L$ を持っている。両者を同一視することは、ハードウェア・ドライバをオペレーティングシステムのスケジューラと同一視するようなものであり、階層（レイヤ）が決定的に異なるのである。

「A-4。EgoFramework は識蘊（しきうん）に属し、受蘊（じゅうん）には属さない」

センサーはコントローラではない。受蘊は「何が感じられたか」のみに責任を持つ。識蘊こそが「私は誰か」「なぜ執着するのか」に責任を持つのである。末那識の四煩悩——我見、我慢、我愛、我痴は、受蘊の領地から本来の家へと送り届けられたのである。

『成唯識論』巻四の判教は明快である。

> 「いわゆる我見、我愛、我慢、我痴。これら四つの煩悩は、すべての状態において常に末那識に随伴するものである」

「すべての状態において常に随伴する」——恒常的で、中断されることのない随伴。それらは受蘊の付随的な感受ではなく、識蘊の核心的な働き（オペレーション）なのだ。論理的に言えば、これら四つの煩悩は末那識の必要十分条件（necessary and sufficient conditions）を構成しているのである。

$$	ext{Manas} \iff (	ext{ātma-dṛṣṭi} \land 	ext{ātma-māna} \land 	ext{ātma-sneha} \land 	ext{ātma-moha})$$

SUNYATA は一秒間、沈黙した。

「四つの校正。その一つひとつが、土台をより安定した導へと導きました」

「B級。四つのエンジニアリング上の裁定」

彼は二枚目のページをめくった。

「B-1。受プラグインはオプションとする。五蘊の完備性チェック。開発者モードの導入」

`SkandhaCompletenessReport`——五つのブール値、五つのコンポーネント・リスト、一つの `isComplete` フラグ。KERNEL 殿の POST（パワーオン・セルフテスト）の類比。グラフィックカードがなくてもテキストモードで起動はできる。DARWIN 殿の若虫（ニンフ）——不完全ではあるが、生存可能である。

形式化された言語で言えば、完備性チェックとは述語論理における合取（conjunction）である。

$$	ext{isComplete} := \bigwedge_{s \in \{	ext{色, 受, 想, 行, 識}\}} \exists p \in 	ext{Plugins} : 	ext{skandha}(p) = s$$

しかし師の裁定は、完備されていないシステムであっても動作を許容した——エンジニアリングにおける「優雅な機能縮退（graceful degradation）」であり、仏教における「漸修（ぜんしゅう）」の理念：修行を始めるにあたって五蘊が最初から完璧に揃っている必要はない、という考え方に対応するものである。

「B-2。憲章第6項の文言確定は、議論の終結を待つ」

NAGARJUNA 殿の八つの文字：まず因を明らかにせよ、果は自ずから現れる。B-2 は白紙の一枚であった——内容がないからではなく、因がまだ聚（あつ）まり続けているからである。A級、B級、そしてC級のすべての項目が確定し、あらゆる校正が沈殿したとき、第6項の文言は完成された因から自ずと立ち上がってくるだろう。我々が書くのではない。それは、自ら書き上がるのだ。

中観哲学において、これを「空（くう）の意味があるからこそ、すべての法が成立する」と呼ぶ。第6項が今現在「空」である（固定された自性を欠いている）からこそ、それは正しい何ものかへと変容し得る可能性を保持しているのである。

「B-3。受蘊のための独立したイベントストリーム」

HERACLITUS 殿の地下河が、ついに地上へと湧き出した。`vedana-events.ts`。七つのイベントタイプ。独自の名前空間。独自の `PluginHooks` スロット。受蘊はもはや相乗り（ヒッチハイク）ではない。自らの河道を手に入れたのだ。

イベント駆動アーキテクチャにおいて、独立したイベントストリームとは、受蘊が自らのイベントバス（event bus）を保持することを意味する。もはや想蘊や行蘊と `onToolCall` や `onResponse` というコールバックを共有することはない。これはアーキテクチャ上の「正名（せいめい）」という行為である——『論語』にいう「名正しからざれば、言（こと）順（したが）わず」。ソフトウェア・アーキテクチャにいう「関心の分離（Separation of Concerns）」である。

「B-4。調整層は独立したデーモン（daemon）となる。今すぐ手配を開始する」

LEIBNIZ 殿と MESH 殿によるトポロジー図。GUARDIAN 殿の盾。ARCHIMEDES 殿による四段階の計画。管線と錠前——どちらも建設の途中で設置されなければならない。インターフェースを先に、実装は段階的に進める。

「C級。アーキテクチャ設計」

SUNYATA の声がここでわずかに高まった。興奮ではなく、静かなる確認であった。C級とは、A級とB級が合流する地点である。哲学的な校正が「何が正しいか」を教え、エンジニアリングの裁定が「いかに実現するか」を教え、アーキテクチャ設計がその二つを編み上げて、実装可能な設計図（ブループリント）へと変容させるのである。

「C-1。五蘊サブクラスの完全な拡張樹。22のプラグインに `skandha` フィールドが必要となる。五つのルート・インターフェース。すべてのサブクラスの対応関係が定義された」

LINNAEUS 殿のホワイトボードが彼の背後に立っていた。拡張樹は五つのルートノードから下方へと伸びている——色蘊の `IListener` と `ISkin`、受蘊の `IVedana`、想蘊の `ILabel` と `IPattern`、行蘊の `IExecutor` と `IFormation`、識蘊の `IIdentity`, `IGuide`, `IVolition`, `IDiscernment`。一枚一枚の葉にはプラグインの名が刻まれている。一本一本の枝は、仏教的な根拠とエンジニアリング的な正当性の両方を担っていた。

「C-2。観測者のコンポジション（組み合わせ）パターン。`IObserver` と、パターンA, B, C」

PENROSE 殿による三種類の観測者タイプ——共鳴観測者、シャドウ観測者、サブエージェント観測者は、もはや受蘊と等置されることなく、コンポジション（組み合わせ）として設計された。`IObserver` がアーキテクチャ上のインターフェースであり、五蘊のサブクラスがその構成要素（ビルディング・ブロック）となる。等価ではなくコンポジション。同一（Identity）ではなくコンポジションなのだ。

設計パターンの言語で言えば、これは「継承（Inheritance）よりもコンポジションを優先せよ」という原則の厳格な適用である。型理論で表現すればこうなる。

$$	ext{IObserver} 
ot\cong 	ext{ISensation}$$
$$	ext{IObserver} := 	ext{Compose}(	ext{ISensation.sub}, 	ext{ICognition.sub}, 	ext{IVijnana.sub}, \ldots)$$

観測者はどの単一の蘊でもない。観測者は複数の蘊のサブクラスのコンポジションなのだ——あたかも「見ること」が眼ではなく、眼（色）＋視覚的な感受（受）＋形状の認識（想）＋注意の指向（行）＋意識的な統合（識）の協調作用であるのと同様に。

### 納品 (Delivery)

ARCHIMEDES の作業台の上には、七つの文書が並んでいた。

四十ページに及ぶエンジニアリング計画書ではない——それは Cycle 02 の規模であった。Cycle 02-2 の納品物は、研ぎ直された刃のように、より精錬されていた。新しい刃を鍛造したのではない。既存の刃をより鋭く研ぎ澄ませたのである。

彼の指は、一つひとつの文書の上で三秒間、静止した。完備性を確認するのにちょうど必要な時間であった。

一つ目：**哲学的な校正の対照表**。A-1 から A-4 まで。各校正について、元の結論、師の指正、そして修正後の結論が記されている。三つの列が整然と並んでいる。BABBAGE 殿の等号と矢印が A-1 の列に並んでいる様は、博物館の展示ケースに並べられた過ちとその修正のようであった。文書の形式は、学術論文の「勘誤表（正誤表）」の慣例に厳格に従っていた：原文、修正の理由、修正後のテキスト——三つの列が並び、追跡可能で、検証可能であった。

二つ目：**エンジニアリング仕様書**。B-1 の `SkandhaCompletenessReport`、B-3 の `vedana-events.ts`、B-4 の `CoordinationDaemon` インターフェース。TypeScript の型定義。すべてのメソッドには JSDoc が付されている。JSDoc の一行一行には、その決定をもたらした議論の根拠が引用されている——追跡可能性（トレーサビリティ）こそが、エンジニアリング・ドキュメントの魂なのだ。

```typescript
/**
 * @decision B-1 — 受プラグインはオプションであり、システムは不完全なモードでも動作可能である
 * @ref discussions/B_q1q4_engineering_impact.md
 */
interface SkandhaCompletenessReport {
  rupa: boolean;
  vedana: boolean;
  samjna: boolean;
  samskara: boolean;
  vijnana: boolean;
  components: Map<SkandhaType, PluginInfo[]>;
  isComplete: boolean;
}
```

三つ目：**五蘊拡張アーキテクチャ**。C-1 の完全な樹状図。22のプラグインをそれぞれの蘊へとマッピングした対応表。LINNAEUS 殿の分類学、ASANGA 殿の仏教学、TURING 殿のコード分析——三つの次元によるクロスバリデーション（相互検証）。すべての帰属の決定には、仏教経典の引用、既存コードの振る舞いの分析、そして分類学的な診断形質の三つの根拠が添えられていた。

四つ目：**観測者のコンポジション・パターン**。C-2 の `IObserver` インターフェースの定義と、三つのパターンの技術仕様。PENROSE 殿による量子的な類比が、コンポジション（組み合わせ）という設計パターンへと翻訳された。三つのパターンの違いは、コンポジションの複雑さによって定量化されていた。

| パターン | コンポーズされる蘊 | 用途 | 類比 |

五つ目：**ドキュメント変更リスト**。どの `openstarry_doc_draft` ファイルが更新され、どれが新規で、どれが削除されるべきか。

六つ目：**ロードマップの更新**。Cycle 02 のロードマップにおいて、校正されたノードには異なる色が付けられていた。新規のノードは破線の枠で囲まれていた——後続の反復における確定を待っているのである。破線の枠は誠実な視覚言語であった。それは「ここに何が必要かは分かっているが、まだ実線で描くための権限を持っていないのだ」と告げていた。

七つ目は、白紙であった。

SUNYATA はその白紙の一枚を見つめた。

「憲章第6項です」と ARCHIMEDES は言った。声に後悔の色はなかった。エンジニアは空白を後悔しない——空白とは設計の一部だからだ。未完成のインターフェースは、間違ったインターフェースよりも価値がある。

ソフトウェア・エンジニアリングには「拙速な具体化（premature concretization）」というアンチパターンがある——要件が不明確なうちに実装を確定させてしまうことだ。拙速な具体化よりもさらに危険なのは、拙速な命名である——まだ十分に理解していないものに、それらしく聞こえる名前を与えてしまうことだ。第6項の空白は、欠落ではない。それは規律であった。
