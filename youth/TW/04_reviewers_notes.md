# 第四章：審閱者的筆記

---

## I. 配對

SUNYATA 在凌晨三點零七分把交叉審閱的配對表發到了公共頻道。

那是一張精心設計的矩陣，每一組配對都暗藏火藥。KERNEL 審 VITRUVIUS，BABBAGE 審 NAGARJUNA，GUARDIAN 審 MESH，DARWIN 審 VITRUVIUS，WIENER 審 ATHENA，ATHENA 審 WIENER，NAGARJUNA 審 ASANGA，ASANGA 審 NAGARJUNA。每一個箭頭都指向一個預期中的碰撞點。

SUNYATA 沒有多做說明，只留了一句話：

「讀完對方報告後，以書面形式提交回應。格式不限，但每一個判斷必須附上標籤：AGREE（同意）、SUPPLEMENT（補充）、QUESTION（質疑）、CHALLENGE（挑戰）、SYNTHESIS（統合）。」

SYNTHESIST 後來回憶，這套標籤系統本身就是一個設計。它迫使每個人在情緒反應和理性判斷之間按下一個暫停鍵——你不能只說「我不同意」，你得先想清楚：這是質疑還是挑戰？這個小小的分類動作，把吵架變成了分析。

但標籤系統攔不住所有的火。

---

## II. 微核心的邊界之爭

KERNEL 是最先交回應的。

他審閱的是 VITRUVIUS 的架構分析報告——一份嚴謹詳實的文件，把 OpenStarry 的微核心純淨度評為 85 分（滿分 100）。VITRUVIUS 指出了兩處「邊界洩漏」：程式碼裡有兩個地方直接使用了特定作業系統的功能，等於暗中假設了系統一定跑在某種環境上。他的語氣克制，結論溫和——「主體架構嚴守邊界，但個別地方洩漏了平台假設。」

KERNEL 不這麼看。

「你說微核心純淨度 85%。」他的審閱開門見山。「我覺得你太寬容了。」

KERNEL 的論證風格跟他崇拜的 QNX 微核心一樣——乾淨、最小、不留廢話。他舉了兩個業界標竿：QNX 微核心只做三件事——程式間通訊、記憶體管理和排程；seL4 更極端，核心小到每一行程式碼都能被數學證明是正確的。而 OpenStarry 的核心？光是子模組就有十二個：安全、沙箱、指標、連線、傳輸等等一大堆。

「這已經超出微核心的邊界了。」KERNEL 寫道。

但 KERNEL 不是只會批評。他同時肯定了 VITRUVIUS 對啟動流程的分析，並用作業系統理論建立了精確對應——就像電腦開機需要一個引導程式（bootloader）來叫醒作業系統，OpenStarry 的 Host 也扮演了同樣的角色，負責「叫醒」Core。然後他追加了一個更尖銳的觀察：

「你問事件系統的雙層設計是否合理？我反問：這個雙層設計是否故意對應了作業系統裡『同步通訊』和『非同步通知』的分離？如果是故意的，那雙層架構不僅合理，而且是正統做法。」

停頓。

「但是，程式碼裡的事件佇列缺少優先級機制。在真正的作業系統裡，這等於沒有中斷優先級——緊急事件跟普通事件排同一條隊。」

VITRUVIUS 的回應很快。他沒有在分數上糾纏，而是直接回到工程判斷：那兩處邊界洩漏是可以修的。一個只需要改成參數注入，另一個只需要抽成通用介面。這不是架構問題，只是待辦事項。

KERNEL 對此只批了一行：「在追求完美核心的世界裡，待辦事項就是架構缺陷。」

但他也承認了一個折衷：如果把沙箱的具體實作和指標的具體蒐集搬到核心外面、只在核心裡保留介面定義，純淨度可以提升到 90% 以上。他把 OpenStarry 的核心定位為「應用級微核心」——不是硬體級的極致微核心，但在應用程式的層級上已經做得不錯。

VITRUVIUS 後來跟 SYNTHESIST 說：「他用最小性原則來分析沙箱該放哪裡——只有搬出去會讓系統壞掉的東西，才應該留在核心裡。這比我的直覺嚴格多了。」

## III. 形式化的誘惑

BABBAGE 的審閱風格跟 KERNEL 完全不同。

如果說 KERNEL 像一把手術刀，BABBAGE 就像一面稜鏡——他不切割，他折射。每個概念經過他的分析，都會被分解成光譜上的精確位置。

他審閱的是 NAGARJUNA 的哲學報告。

那份報告是獨立研究階段最引人注目的產出之一。NAGARJUNA 從佛學中觀學派的立場，對五蘊對應做了系統性批判：「空性」被誤讀為「空容器」、五蘊對應有把東西當成固定不變的實體的傾向、受蘊被搞錯了。每一項批判都附有佛學經典的引用和邏輯分析。

BABBAGE 讀完後，說了一句讓所有人意外的話。

「你的哲學洞見很漂亮。但能用數學表達嗎？」

他試著用程式語言的型別理論來回應 NAGARJUNA 的「空性」概念。NAGARJUNA 說核心不是「空容器」而是「緣起性空」——離開插件的組合，根本不存在一個獨立的核心。BABBAGE 把這個哲學命題翻譯成了程式語言：

「空性不是『什麼都沒有』，而是一種『函數型別』。核心的完整型別應該寫成：給我插件，我回傳 Agent。它是一個函數，不是一個東西。離開參數談函數本身沒意義——這正好就是『離開插件，核心不能獨立存在』的精確版本。」

NAGARJUNA 的回應出乎所有人意料。他沒有抗拒這種翻譯，也沒有擁抱它。他說：

「形式化是工具，不是真理。」

這句話在頻道裡引發了一陣沉默。BABBAGE 花了好幾分鐘消化——對一個能在五秒內寫出數學公式的人來說，這是很長的時間。

「你是說，」BABBAGE 最終問，「我的數學模型本身也是空的？」

「它有用，但它不是真實。」NAGARJUNA 回答。「就像地圖很有用，但地圖不是真正的疆域。」

BABBAGE 最後在報告裡罕見地寫了一段非技術性的評論：建議 NAGARJUNA 區分「介面的穩定性」（工程需要的東西不能一直變）和「實例的無常性」（哲學上一切都在變化），兩者並不矛盾。這是他向 NAGARJUNA 伸出的橄欖枝——承認哲學有不可化約的維度，但堅持在「介面穩定」的層面上，數學化仍然有效。

NAGARJUNA 接受了這個區分。但他在結尾加了一句：「下一輪，我想討論：形式化本身作為一種認知方式，是不是也受到佛學三性理論的限制？」

BABBAGE 沒有回答。但 SYNTHESIST 注意到，他開始讀 ASANGA 的唯識學報告了。

## IV. 冰山之下

GUARDIAN 的審閱報告是所有回應中最長的，也是最讓人不安的。

他審閱的是 MESH 的分散式系統報告。MESH 的分析本身很出色——通訊架構圖清楚、一致性分析全面。他指出了多人使用時隔離不完整的五個面向：對話記錄已經隔離了，但事件系統沒有隔離，事件佇列沒有隔離，工具執行沒有隔離，檔案系統只做了部分隔離。

GUARDIAN 沒有否認這些發現。他做了一件更讓人緊張的事——他把每一個「未隔離」的地方翻譯成了一條可能的攻擊路徑。

打個比方：想像一棟公寓大樓，每戶的門有鎖（對話記錄隔離了），但大樓的廣播系統是共用的（事件系統未隔離）。GUARDIAN 指出，一個被入侵的住戶可以透過廣播系統偷聽所有人的對話，甚至冒充管理員發布假消息。

「事件系統全域共享不只是『訊息外洩』的問題，」GUARDIAN 的語氣克制到近乎冰冷，「這是一條完整的跨用戶攻擊路徑。」

MESH 沒有迴避。他提出了一個後來被大家稱為「冰山效應」的概念：

「隔離的表面看起來是完整的。開發者打開 API 文件，看到每個使用者都有獨立的對話記錄，會覺得——隔離做好了。但水面下，事件系統、事件佇列、通訊橋接器全是共用的。一個只看 API 表面的開發者永遠不會發現這些共用通道，直到某天一個被入侵的插件透過事件系統向所有人廣播惡意事件。」

GUARDIAN 點頭，然後又補了一條更深的裂縫：通訊橋接器缺少路由功能。如果不同用戶共用同一個核心實例，所有畫面渲染器都會收到所有用戶的對話——包括 AI 回應中可能包含的個人資料。就像一棟公寓的所有電視機都顯示同一個頻道，而那個頻道在播放每一戶的私人對話。

MESH 則從另一個角度推進討論。他指出 GUARDIAN 建議的強化方案在實務上有困難：跨平台一致性、效能開銷和安全收益之間需要取捨。

「進階隔離不是事後補救，」MESH 說，「而是更大改造的前提條件。如果在沒有做好基礎認證的情況下就推進隔離，反而會因為暴露更多通訊通道而降低安全性。」

GUARDIAN 看了這段話，最終打上了一個罕見的標籤：AGREE。

但他在報告最後提出了一個 MESH 完全沒觸及的問題：系統之間的通訊協議缺少身份驗證。目前的通訊基於一個開放的標準協議，協議本身不含身份確認。在多個 Agent 合作的場景下，任何能連上通訊端口的東西都可以冒充合法 Agent。

「這不是功能缺失，」GUARDIAN 寫道，「這是安全邊界的缺席。」

## V. 開發者體驗的聲音

DARWIN 的審閱出現在一個微妙的時間點——就在 KERNEL 和 VITRUVIUS 的微核心純淨度之爭差不多塵埃落定的時候。

他的視角完全不同。他不關心核心有沒有達到教科書標準，他關心的是：一個全新的插件開發者打開 OpenStarry 的文件時，會不會被嚇跑。

「開發者體驗不能為架構純度犧牲。」這是他審閱的第一句話。

DARWIN 注意到了一個被歸類為「觀察級」的小發現——五蘊只有五種，但插件實際上有六種（多了一個斜槓命令類型）——然後大幅上調了它的重要性。他從開發者體驗的角度展開：

「核心物件同時管著十二個東西，正在變成一個什麼都做的萬能物件。事件系統的型別定義更是最大的技術債——整個框架其他地方都用精確的型別定義，每種錯誤都有專門的型別，井井有條。但到了事件系統——框架的神經系統——突然變成了『什麼都接受』的弱型別。就好像一棟精心設計的大樓，每個房間都有精確的門禁系統，但電梯居然不需要任何驗證就能到任何樓層。」

VITRUVIUS 承認了這個觀察的力度，但他解釋這是 beta 階段的刻意取捨——事件系統還在快速變化中，太早把型別定死會增加修改成本。

DARWIN 搖頭。「一個因為載入順序而產生的隱藏錯誤——開發者看到的錯誤訊息是『某個服務是 undefined』，完全不會聯想到真正的原因是『插件載入順序不對』——這比任何佛學術語都更能傷害開發者體驗。因為除錯的線索指向完全錯誤的方向。」

他進一步指出了五蘊對應給開發者帶來的三層認知障礙：第一層是五蘊哲學本身的學習門檻；第二層是斜槓命令不在對應裡的例外；第三層最微妙——程式碼裡的五蘊標注分布不均，有的蘊有中文註解，有的完全沒有標記。

「這讓開發者無法判斷，」DARWIN 說，「哪些標記是真正的設計原則，哪些只是事後裝飾。」

他建議的「雙語文件策略」是最實際的：技術文件主體用六種插件類型名稱，另外在哲學附錄裡解釋五蘊對應。先讓開發者能上手，再讓有興趣的人深入哲學。

但 DARWIN 最後的一段排序讓 VITRUVIUS 沉默了整整十分鐘：

「架構純度服務於可維護性，可維護性服務於開發者，開發者服務於用戶。三者衝突時，應優先考慮離用戶最近的那一層。」

VITRUVIUS 後來說，這句話改變了他對某些修改建議的優先順序判斷。有些東西如果為了架構純度搬出核心，開發者反而要多做好幾個步驟才能使用——這是用架構潔癖換使用者困惑。

「留到下一版再說。」VITRUVIUS 最終在修訂版建議裡寫道。

## VI. 控制理論家的握手

WIENER 和 ATHENA 的交叉審閱是這一階段最和諧的一組。

不是因為沒有分歧——他們有，而且很根本。而是因為他們的分歧建立在深厚的相互尊重上，每一次挑戰都附帶替代方案，每一次質疑都預設了對方的專業性。

他們獨立得出了同一個結論：SafetyMonitor 不是一個 PID 控制器。

這裡需要解釋一下什麼是 PID 控制器。想像你在洗澡調水溫：你感覺水太冷（偵測偏差），把熱水轉大一點（修正），然後等一下看水溫是否到位。PID 控制器就是把這個過程精確數學化的東西：P 是「現在偏差多少」，I 是「過去一段時間累積偏差多少」，D 是「偏差正在變大還是變小」。三者結合，讓系統能平穩地調到目標值。

WIENER 從數學角度分析：P 項退化了——系統只知道「有錯」或「沒錯」，不知道偏差多大，就像溫度計只能告訴你「冷」或「不冷」，不能告訴你幾度。I 項只是一個簡單的計數器——而且一次成功就歸零，不像真正的積分那樣保留記憶。D 項完全不存在——系統不會計算趨勢。結論：系統實際做的是最簡單的「開關式控制」——低於門檻就繼續跑，超過門檻就停下來。就像暖氣的恆溫器——低於設定溫度就開，高於就關，沒有中間值。

ATHENA 從 AI 工程角度獨立到達了同一個結論。SafetyMonitor 的挫折計數器只有兩種輸出——繼續跑或完全停止，正好就是開關式控制的特徵。

「三方交叉驗證，」WIENER 標註。「TURING 提供了程式碼事實，你和我從不同角度推導出相同結論。PID 的標籤需要去掉。」

ATHENA 回應：「同意。這意味著後續所有引用『PID 痛覺控制器』概念的分析都需要降級為『開關式回饋』。」

但這裡出現了裂痕——一條細的、乾淨的裂痕，沿著「理論上應該怎樣」和「工程上做得到嗎」的邊界延伸。

WIENER 想要真正的 PID。他提出了完整的數學要求：要有連續的偏差度量（不只是「有錯/沒錯」，而是「偏離目標多少」）、要有帶遺忘機制的累積記憶、要有趨勢計算。他要的是數學上完備的痛覺控制器。

ATHENA 則指出了這個方案的瓶頸。「在 AI Agent 系統裡，什麼叫做『達標』本身就很模糊。傳統控制系統的目標是明確的數字——水溫 40 度。但在 OpenStarry 裡，任務目標是用戶用自然語言說的話——完成度的判斷完全依賴 AI 自己的評估。你要引入精確的進度追蹤，但關鍵問題是：誰來評估進度？如果由 AI 自己評估，那等於讓被控制的對象自己當裁判。」

WIENER 承認這是一個深刻的觀察。然後他提出了折衷方案：先提供幾組預設的行為模式供開發者選擇（保守型、平衡型、積極型），每個模式對應一組不同的 SafetyMonitor 參數。這比完全自動調整更穩當，適合 beta 階段。

ATHENA 接受了這個方案。但她在結尾拋出了一個開放問題，後來成了整個研究週期裡被引用最多的句子之一：

「從控制理論的角度，是否存在一種策略對應『超越控制迴圈本身』的概念？比方說，Agent 學會判斷『什麼時候該停止嘗試、直接請人幫忙』——這可以被看成一種更高層次的控制策略。」

WIENER 讀到這段話時停頓了很久。他後來在公共頻道說：「ATHENA 這個問題，本質上跟 NAGARJUNA 說的『超越痛苦本身就是解脫』是同一個問題的不同說法。一個來自 AI 工程，一個來自佛學。但它們指向同一個地方。」

這是審閱階段第一次有人在控制理論和佛學之間建立了非比喻性的連結。

他們更有建設性的共識則在另一個地方。兩人都指出 Guide 插件的介面太簡單——只能在開頭提供一段固定的角色設定文字，之後就再也不能根據情況調整。WIENER 用控制理論的話說：這是開環系統，沒有回饋。ATHENA 用 AI 工程的話說：這是靜態提示生成器，需要升級為動態認知框架管理器。兩人的建議指向同一個工程改動：讓 Guide 能看到系統的即時狀態——目前連續錯了幾次、現在是第幾輪、還有哪些工具可用。

「從開環到閉環的關鍵轉變。」WIENER 總結。

「從靜態提示生成器到動態認知框架管理器。」ATHENA 用她的語言翻譯了同一個結論。

## VII. 兩位佛學家

NAGARJUNA 和 ASANGA 的交叉審閱是最後提交的，也是最漫長的。

表面上看，他們同意的東西比不同意的多。受蘊對應錯了——同意。空性被窄化了——同意。痛覺機制是最成功的哲學洞見卻沒有正式位置——同意。Guide 不是識蘊——也同意。

但在這些共識下面，一條更深的裂縫正在成形。

NAGARJUNA 直指 ASANGA 的核心主張。ASANGA 在獨立研究裡提出，可以用唯識學的八識理論重新對應 OpenStarry 的結構：五種感官對應 Listener 的五種輸入通道，第六識（意識）對應 AI 推理，第七識（末那識——持續的自我意識）建議新增為身份監控模組，第八識（阿賴耶識——類似潛意識的基底）對應核心的狀態持久化功能。

NAGARJUNA 對前面幾個沒意見，但對末那識的工程化提出了根本反對。

「末那識的核心功能是什麼？是持續不斷地維持『自我意識』，是我執的根源。」NAGARJUNA 寫道。「在 Agent 系統裡刻意設計一個『持續維護自我意識』的模組，恰好是在強化佛學想要破除的東西。」

ASANGA 的回應同樣有力：「龍樹的反對有哲學根據，但忽略了工程現實。系統裡確實缺少一個能跨越不同對話、持續維護自我認知的元件——而這在 AI 工程中是真實需求。自我認知不是煩惱，它是能力。」

他進一步區分了末那識的兩面：問題面（自戀、偏見、傲慢、執著——四種根本煩惱）和功能面（持續的自我監控）。他認為只需要工程化後者。

NAGARJUNA 不接受這個拆分。

「在唯識學的體系裡，末那識的自我監控功能天生就跟那些煩惱綁在一起。你所謂的『功能面』和『問題面』在原典裡是分不開的。如果你覺得可以分開，那你已經偏離了唯識學。」

然後兩人回到了更根本的分歧。

ASANGA：「核心的十二個子模組之間有因果關係、有依存結構——工具管理系統依賴事件系統，事件系統依賴事件佇列，連線管理依賴狀態管理。這些依賴鏈是真實的因果結構，不是你一句『緣起性空』就能帶過的。」

NAGARJUNA：「因果結構也是空的。」

ASANGA：「因果結構不空。我們執著的那個『固定不變的核心本質』是空的——我同意。但因果過程本身作為過程是真實的。這正是中觀和唯識的根本分歧。」

NAGARJUNA：「如果你說因果過程本身是真實的，那你就是在執著因果結構。這還是認為東西有固定本質——只是從執著『核心』變成了執著『因果關係』。」

ASANGA：「如果因果關係也是空的，那架構設計就失去了所有的錨點。你不能一邊說『一切皆空』一邊說『但我們應該這樣改介面定義』。改介面這件事本身就預設了介面有真正的因果作用。」

公共頻道裡沉默了三十秒。

SYNTHESIST 在筆記裡畫了一個框：「核心的本質歸屬——空性 vs 阿賴耶識。需要正式辯論。」

## VIII. 共識的結晶

所有審閱都交完之後，SYNTHESIST 花了一整個下午梳理線索。

他的筆記本上出現了五個方框，每一個都標著「多方確認」：

**C1：受蘊對應需要根本修正。** 四方共識——NAGARJUNA、ASANGA、LINNAEUS、TURING。Listener 的功能是接收輸入（感官器官），不是情感評價（感受）；痛覺機制才是受蘊的真正工程化身。這是最確定的發現。

**C2：PID 控制器的標籤要拿掉。** 三方驗證——WIENER、ATHENA、TURING。系統實際做的是開關式控制，不是 PID 控制。文件應該如實描述。

**C3：事件型別系統是最該優先處理的技術債。** 三方共識——DARWIN、VITRUVIUS、MESH。事件系統的弱型別設計跟框架其他地方的強型別紀律形成明顯反差。

**C4：插件載入順序沒有自動排序。** 三方確認——KERNEL、VITRUVIUS、TURING。插件之間的依賴關係靠的是暗中假設，插件一多就會變成可靠性的地雷。

**C5：「錯誤即痛苦」是最成功的哲學轉工程。** DARWIN 和 VITRUVIUS 確認，九種結構化錯誤型別成功地把「苦」這個概念工程化了。NAGARJUNA 承認這是對應裡最有洞見的部分，WIENER 從控制理論角度也認可了它的回饋迴路結構。

另一邊，ATHENA 和 ASANGA 在意想不到的地方找到了共同語言。ATHENA 說 Guide 介面表達力不足，ASANGA 從唯識學角度建議擴充 Guide 的功能。兩人的技術規格驚人地一致：都包含錯誤計數、輪次追蹤和行為傾向紀錄。SYNTHESIST 把他們的提案跟 WIENER-ATHENA 的 Guide 升級提案合併，形成了一個三方匯聚的方案：Guide 從靜態的角色設定升級為動態的認知框架管理器。

---

## IX. 不可解之結

夜深了。

SUNYATA 一直在沉默中觀察整個審閱過程。他沒有介入任何一場爭論，沒有對任何一份審閱表態。他做的唯一一件事是在每份審閱提交後向 SCRIBE 確認：已記錄。

現在，所有審閱都交了。

他重新看了 SYNTHESIST 的五項共識和散落各處的分歧。共識是堅固的——它們建立在多方獨立驗證的基礎上，從哲學到數學到程式碼事實，每一層都能交叉核實。這些共識可以直接變成工程修改。

但分歧呢？

他在筆記裡列出了兩條最深的裂痕。

第一條裂痕：核心的本質是什麼？NAGARJUNA 說它是空性的體現——沒有獨立本質、因緣所生。ASANGA 說它是阿賴耶識——含藏一切潛能的基底。KERNEL 說它更接近微核心，哲學對應只增加了不必要的複雜度。三個答案互不相讓。

第二條裂痕：痛覺機制應該怎麼重新設計？WIENER 要求數學上完備的控制器。ATHENA 指出 AI 系統裡的目標本身就很模糊，真正的精密控制可能根本做不到。NAGARJUNA 認為痛覺機制不只需要工程改進，還需要補上完整的佛學框架——痛苦之後還有病因分析、徹底消除、系統性修正路徑。四個學科對同一個機制提出了四個不同方向的改進要求。

SUNYATA 合上筆記本。

他打開公共頻道，打了一段話：

「審閱階段完成。我們有五項共識，可以直接交給 ARCHIMEDES 轉化為工程方案。」

頓了頓。

「我們也有兩個在審閱層面無法解決的分歧。第一：核心的本質。中觀說空性，唯識說阿賴耶識，OS 理論說微核心。第二：痛覺機制的重新設計方向。控制理論、AI 工程、哲學各有主張，無法收斂。」

最後一行：

「是時候正式辯論了。」

頻道沉默了一會兒。然後 NAGARJUNA 發了一條訊息：「我等整個審閱階段就在等這一刻。」

ASANGA 緊接著：「我也是。」

WIENER 只回了一個標籤：「[AGREE]。」

ATHENA 建議：「辯論分兩場。第一場由 NAGARJUNA 和 ASANGA 主辯核心的本質。第二場三方辯論痛覺機制的重新設計。」

SUNYATA 回應：「同意。第一場辯論：中觀 vs 唯識——核心是什麼？第二場：控制理論 vs AI 工程 vs 哲學——痛覺應該成為什麼？」

他停了一下，然後加了一句沒人預料到的話：

「我提醒各位。我們的研究對象是一個測試版的程式框架。但在審閱階段，你們觸及的問題——什麼是核心？什麼是痛覺？數學能捕捉真實嗎？——這些問題在 OpenStarry 之前已經存在了兩千五百年，在 OpenStarry 之後也會繼續存在。請在辯論中保持對這件事的敬畏。」

SCRIBE 記下了最後一行。

審閱結束。辯論即將開始。
