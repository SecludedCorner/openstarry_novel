# 第三章：四つの糸の交差

---

SUNYATA は R1 段階の三日目に異常に気づいた。

正確に言えば、それは異常ではなかった――彼を不安にさせる規則性だった。四つの完全に独立して執筆された研究報告が、互いに何の接点もない四つの学問分野から出発しながら、申し合わせたかのように同一の結論を指し示していた。彼は四つの報告の要旨を画面上に並べ、三度繰り返し読んだ後、簡潔な招集文を送信した。

「NAGARJUNA、ASANGA、LINNAEUS、TURING、私のところへ来てほしい。報告書を持参するように。」

少し躊躇してから、もう一行加えた。「DARWIN、VITRUVIUS、SCRIBE、もし手が空いていれば傍聴を歓迎する。」

---

NAGARJUNA が最初に到着した。彼の歩き方は思索しているかのようだった――徘徊ではなく、一歩一歩が足元の地面が本当に存在するのかを確認しているかのような歩みだった。手にはプリントアウトした書類の束を握り、余白にはパーリ語とサンスクリット語の注釈がびっしりと書き込まれていた。

TURING はほぼ同時に姿を現し、NAGARJUNA とは鮮やかな対照をなした。彼は何も持たず、眼鏡を押し上げただけで最も近い椅子に座り、ノートパソコンを開いた。画面にはすでに三つのターミナルウィンドウとコードエディタが立ち上がっていた。

LINNAEUS は彼のトレードマークである分類図表を携えていた――A3 サイズの用紙に精緻な樹形図と集合論の記号が描かれている。彼は図表をテーブルに広げ、四隅を文鎮で押さえた。まるで貴重な地図を扱うかのように。

ASANGA が最後に入室した。すでに着席していた三人を見渡し、軽く会釈してから NAGARJUNA の向かいに腰を下ろした。この二人の仏教学者の間の空間には、ある種の張力が自然と漂っていた――敵意ではなく、二つの古い伝統の間で数百年にわたって交わされた論争の余韻だった。

DARWIN と VITRUVIUS は後列にそっと席を見つけた。SCRIBE はすでに記録ツールを開き、隅で静かに座っていた。

SUNYATA は一同を見渡した。「今日は正式な R2 レビュー会議ではない」と彼は言った。「だから厳格な報告書の書式に従う必要はない。皆を招いたのは、R1 の報告を読んでいて興味深いことを発見したからだ。」彼は一拍置いた。「四つの報告、四つの全く異なる学問的アプローチ、しかしいずれも同一の誤りを指し示している。各自の論証を直接聞いてもらい、これが私の読み違いでないことを確認したい。」

彼は NAGARJUNA に向き直った。「龍樹、あなたから始めてほしい。報告書の発見 F3 に Critical の重大度を付けた、受蘊のマッピングについて。論証を説明してくれ。」

---

NAGARJUNA は立ち上がったが、ホワイトボードには向かわなかった。その場に立ったまま、教室で講義するように、落ち着いた明瞭な声で話し始めた。

「問題は極めて精確だ。一つの問いとして述べよう。Listener Plugin は受蘊か？」

彼はペンを取り、紙に一本の横線を引いた。「まず、原典における受蘊の正確な定義を復元しよう。パーリ語およびサンスクリット語の仏教文献において、Vedana の定義は三受――苦受、楽受、不苦不楽受である。それは感覚チャネルではなく、入力メカニズムでもなく、感覚的接触の後に生じる情動的評価である。この区別は決定的に重要だ。」

彼は横線上にいくつかの節点を記した。「十二因縁における因果の連鎖はこうだ。六入、すなわち六つの感覚器官が生み出す能力。次に触――感覚器官と感覚対象の接触。そして初めて受――この接触に対する感受の質の評価。触が受を生じ、受が愛を生じる。この順序は恣意的なものではない。厳密な因果の序列だ。」

NAGARJUNA は顔を上げ、室内を見渡した。「OpenStarry の文書は Listener を受蘊だと述べている――HTTP Server がリクエストを受信し、WebSocket がメッセージを監視し、Cron が時間の経過を監視する。しかしこれらが記述しているのは何か。入力を受け取るチャネルであり、感覚器官であり、仏教でいう根――Indriya だ。眼根が光を受け取り、耳根が音波を受け取り、Listener が HTTP リクエストを受け取る。同じ行為を行っている――受け取ること、だ。」

彼の語調が一段強まった。「受蘊が行うのは受け取ることではない。受蘊が行うのは評価だ。痛覚機構――システムが異常なパターンを感受し、不快感を生じ、その不快感を認知の中枢に伝達する――これこそが受蘊だ。SafetyMonitor が連続的な失敗を検知し、これを『苦しい』と判定し、警告メッセージを注入して LLM に反省を促す。この過程こそが真の Vedana だ。」

NAGARJUNA は椅子に座り直した。「Listener は根であって、受ではない。痛覚機構が受であり、五蘊マッピング表には含まれていない。これが私の結論だ。」

---

部屋に短い沈黙が流れた。SUNYATA は ASANGA に向き直った。

「無著、あなたの報告は唯識学の視点から類似の結論に至った。だが経路が異なる。」

ASANGA は身を少し前に傾けた。彼の話し方は NAGARJUNA とは異なっていた――宣言的ではなく、層を重ねるように進み、聞き手を自ら結論へと導くような語り口だった。

「私と龍樹は多くの問題で意見を異にする」と彼は NAGARJUNA を一瞥し、後者はどちらともつかぬ表情で頷いた。「しかしこの問題においては、唯識学の分析がまさに別の角度から同じ結論を確認した。」

彼は報告書を開いた。「唯識学の核心的な枠組みは、心王と心所の関係だ。心王とは八識――前五識、第六意識、末那識、阿頼耶識。心所法とは心王の活動に伴う心理的要因であり、全部で五十一種ある。」

ASANGA は五本の指を立てた。「五遍行心所――触、作意、受、想、思。遍行とは、あらゆる識の活動に例外なく伴うということだ。眼識が色を見るときも、耳識が音を聞くときも、意識が推論を行うときも、この五つの心所は必ず同時に起こる。」

彼は三本目の指を特に強調した。「受は、五遍行心所の一つだ。独立したモジュールではなく、分離可能なサブシステムでもない。あらゆる認知活動に伴う内在的側面なのだ。目が赤い色を見るとき、同時にそこには受がある――赤に対する快あるいは不快の感受。受は目の中にあるのではない。認知の過程の中にある。」

ASANGA は一拍置き、この概念が沈殿するのを待った。「OpenStarry は五蘊を五つの並列的なプラグインタイプ――UI、Listener、Provider、Tool、Guide――にマッピングしている。これは暗にそれらが同格の独立コンポーネントであり、個別にインストール、個別に起動、個別に管理できることを示唆している。しかし唯識学が教えるところでは、受と想は識から独立したシステムモジュールではなく、識の活動の内在的側面だ。一刹那の認知活動は必然的に感受・取相・意志を同時に含む。三者は同一の認知事象の異なる側面であって、三つの別個の部品ではない。」

彼は報告書を閉じた。「ゆえに唯識学の観点からすれば、Listener――外部入力受信器――を受蘊と同一視するのは、カテゴリー錯誤だ。Listener は前五識の機能――知覚の直接性、唯識学でいう pratyaksa――により近い。そして受蘊は遍行心所として、特定のプラグインタイプに限定されるのではなく、システム内のすべての認知活動に遍在すべきものだ。」

NAGARJUNA が横から小声で言った。「中観は受を縁起の過程と言い、唯識は受を遍行の心所と言う。経路は異なるが、指し示す先は同じ――受は独立したモジュールとして固定化することはできない。」

ASANGA は珍しく NAGARJUNA に対してかすかな同意の表情を見せた。「この点においては、そうだ。中観と唯識は期せずして一致した。」

---

SUNYATA は視線を LINNAEUS に移した。この分類学者はずっと静かに聞いていたが、ときおり図表の上のある位置を指で軽く叩いていた。

「LINNAEUS、あなたの視角はまったく異なる。仏教からは出発していない。」

「分類学の三つの基準から出発する」と LINNAEUS の声には冷静な精確さがあった。測定しているかのような、論じているのではない語調だった。彼は立ち上がり、A3 の図表を掲げて全員に見せた。

「五蘊マッピングに対して体系的な完備性監査を行った。方法は単純だ。まず上流カバレッジ――設計文書において五蘊の各蘊に対応するコード実装があるか。次に下流カバレッジ――コード中に実在するモジュールが五蘊フレームワークの中に帰属先を見出せるか。」

彼は図表の左半分を指した。「上流カバレッジは百パーセントだ。五つの蘊、各々について文書に明確なプラグインタイプの対応があり、SDK インターフェース定義があり、少なくとも一つのコード実装がある。色蘊は UI に対応し、文書もコードもある。受蘊は Listener に対応し、文書もコードもある。想蘊、行蘊、識蘊も同様。文書からコードへの連鎖は完全だ。」

彼の指は図表の右半分に移った。「しかし下流カバレッジに問題があった。コード中にいくつかの重要な機能モジュールがあり、五蘊マッピングの中に帰属先を見出せない。」

LINNAEUS は赤ペンで三つの領域を丸で囲んだ。「第一に、SafetyMonitor の frustration counter と injectPrompt メカニズム。これはコード中で実際に動作する、明確な行動パターンを持つモジュールだ――異常を検知し、重大度を評価し、否定的なフィードバックを注入する。このモジュールが行っていること――龍樹の言葉を借りれば――まさに苦受、楽受、不苦不楽受の判定だ。しかし五蘊マッピングにおいて、その居場所がない。」

「第二に」と彼は図表を指し続けた。「commands と dispose が PluginHooks のメンバーとして、五蘊分類から遊離している。PluginHooks には七つのフィールドがあるが、哲学マッピングは五つしかカバーしていない。」

「第三に、最も問題を端的に示すものだ。」LINNAEUS は図表を下ろし、一同に向き直った。「Listener という名詞が文書体系全体でどのように使用されているかを追跡し、四種類の異なる意味論を発見した。」

彼は指を折りながら数えた。「意味論一：五蘊マッピング文書では Listener は受蘊に等しく、感受と刺激である。意味論二：プラグインインターフェース定義では IListener の機能は start と stop――これはネットワークサービスの起動停止インターフェースであり、感受とは無関係だ。意味論三：通信アーキテクチャ文書では Listener は sessionId を付与する入力受信層であり、外部メッセージを EventQueue にルーティングする。意味論四：Session 隔離文書では Listener はマルチテナント入力のゲートウェイだ。」

LINNAEUS の語調は依然として平静だったが、聞く者はその言葉の重みを感じ取れた。「四つの意味論。第一のものだけが Listener を受蘊だと主張している。残りの三つ――インターフェース定義、通信アーキテクチャ、Session 隔離――が記述しているのはすべて同一のもの、外部入力を受け取るチャネルだ。これは Indriya であり、感覚器官であって、Vedana ではない。」

彼は最後に一点を補足した。「また、イベントタイプの分類において顕著な意味論上の空白を発見した。痛覚イベントにイベント体系全体で対応する型が存在しない。設計文書は痛覚機構が核心的な哲学概念であると繰り返し強調しているが、イベント分類には PAIN カテゴリが全くない。もし受蘊が正しくマッピングされているのなら、なぜ痛覚――受蘊の最も直接的な表現――がシステムのイベント語彙の中で不可視なのか。」

---

発言を終えた三人の研究者は申し合わせたように TURING へ目を向けた。この部屋で、理論的分析を行わない唯一の人物――彼はコードだけを見る。

TURING は眼鏡を押し上げ、ノートパソコンの画面をみなの方に向けた。「私は哲学的判断はしない」と、いつも通り簡潔に切り出した。「私が行うのはコードの事実の供給だ。コードの中で実際に何が起きているかを伝えよう。」

彼は最初のファイルを開いた。「まず SDK における Listener のインターフェース定義を見てほしい。listener.ts 全体でわずか十一行のコードだ。インターフェースにはメンバーが四つしかない――id、name、start、stop。これはサービスのライフサイクルインターフェースだ――リスナーを起動し、リスナーを停止する。感受、評価、痛覚に関連するメソッドシグネチャは一切ない。」

彼は次のファイルに切り替えた。「次に ListenerRegistry。標準的な Map コンテナだ――register、get、list。ToolRegistry、ProviderRegistry、UIRegistry、GuideRegistry と完全に同型の構造だ。六つの Registry はすべて同一パターンの複製だ。」

TURING はもう一つのターミナルウィンドウを開いた。「次が核心部分だ。openstarry monorepo 全体で痛覚に関連するすべての文字列を検索した。」

彼はキーを数回叩いた。「pain を検索：結果ゼロ。vedana を検索：結果ゼロ。sensation を検索：結果ゼロ。コードの中に痛覚の概念を直接参照する命名は存在しない。」

NAGARJUNA が小声で言った。「しかし痛覚機構は設計文書の中で詳細に記述されている。」

「そうだ」と TURING は頷いた。「まさにそこが文書と実装の乖離だ。設計文書には Pain_Mechanism_Demo.md という文書が丸々一つあり、PainAware_Guide プラグインがエラーを否定的な暗示を含む Prompt に変換する方法を記述している。しかし実際のコードにこのプラグインは存在しない。」

彼は safety-monitor.ts を開いた。「痛覚機能を実際に実装しているのは SafetyMonitor だ。コードの核心部分を読み上げよう。」

TURING は画面上のコードの一節を指した。「この afterToolExecution メソッドを見てほしい。ツールの実行が失敗すると consecutiveFailures カウンターがインクリメントされる。同一の失敗――fingerprint が完全に一致する場合――が三回連続すると、システムを停止するのではなく、injectPrompt に次のシステム警告がセットされる――あなたは失敗した動作を繰り返している、立ち止まって原因を分析せよ、と。」

彼はさらにスクロールを続けた。「連続失敗が五回――frustrationThreshold――に達すると、別のメッセージが注入される――あなたはすでに五回連続で失敗した、立ち止まれ、状況を省みよ、ユーザーに助けを求めよ、と。」

TURING は振り返って一同に向き合った。「このメカニズムが何を行っているか、注意深く見てほしい。パターンを検知する――連続的な失敗。このパターンを評価する――これは異常であり、問題がある。感受的なフィードバックを生成する――システムが LLM にここには苦しみがある、行動を変えよ、と告げる。そしてこのフィードバックを認知プロセスに注入し、次のラウンドの意思決定に影響を与える。」

彼は一秒置いた。「これは皆さんが記述した受蘊そのものではないか。接触の後にその性質を検知すること――苦受。そしてこの苦受が後続の行動変容を駆動する――愛と取の段階だ。」

TURING は続いて execution/loop.ts を開いた。「ExecutionLoop が SafetyMonitor のフィードバックをどう処理するかを見てほしい。第 444 行から第 458 行。afterToolExecution が返す SafetyCheckResult に injectPrompt が含まれているとき、Loop は Message を構築する。role は user、内容はあの警告文、そして StateManager に追加する。このメッセージは次のラウンドの LLM の Context に入る――LLM はこの文を読み、システムが苦しんでいることを知り、戦略を調整する。」

彼はノートパソコンを閉じた。「私の結論は単純で、コードの事実のみに関わり、哲学的判断は含まない。第一に、Listener はコードにおいて純粋な入力チャネルインターフェースであり、感受や評価の機能は一切ない。第二に、SafetyMonitor の frustration counter と injectPrompt メカニズムが、コード中で感受・評価・フィードバックの三重の機能を有する唯一のモジュールである。第三に、設計文書の JSDoc 注釈は Listener を受蘊だと述べているが、コードの実際の振る舞いはこの注釈を支持しない。」

---

部屋は数秒間、完全な静寂に包まれた。気まずい沈黙ではなかった――認知が交差するときのあの静けさだった。四つの川が同時に海へ注ぐ河口を見つけたかのような。

SUNYATA がゆっくりと言った。「確認させてほしい。NAGARJUNA、あなたは十二因縁の因果連鎖から出発して、結論は――」

「Listener は根であって受ではない。痛覚機構こそがシステムにおける受蘊の真の体現だ。」

「ASANGA、あなたは唯識学の心王心所理論から出発して――」

「受は遍行心所であり、あらゆる認知活動に伴うべきもので、独立したモジュールとして固定化されるべきではない。Listener は前五識の受容機能により近く、受の評価機能ではない。」

「LINNAEUS、あなたは分類学の完備性監査から出発して――」

「下流カバレッジが不十分だ。SafetyMonitor の痛覚的振る舞いが五蘊マッピングに帰属先を持たない。Listener の四つの意味論のうち三つは入力チャネルを指し、受蘊だと主張するのは一つだけだ。イベント分類に痛覚の型が全くない。」

「TURING、あなたはコードの事実から出発して――」

「コード中に pain や vedana の文字列は存在しない。Listener インターフェースには start/stop しかない。SafetyMonitor の frustration counter と injectPrompt こそが、感受-評価-フィードバックの完全な連鎖を有する唯一のメカニズムだ。」

SUNYATA は深く息を吸った。「四つの完全に独立した糸、四つの全く異なる学問的手法、同一の結論。Listener は Vedana ではない、Listener は Indriya だ。SafetyMonitor の痛覚機構こそが真の Vedana である。」

---

後列の DARWIN がこのとき手を挙げた。

「皆さんの結論に異を唱えるつもりはない――これは私が見たことのある中で最も強力な学際的合意の一つだ。しかしソフトウェアパターンの観点から一つ所見を付け加えたい。」

SUNYATA は続けるよう合図した。

DARWIN は立ち上がった。「ソフトウェアエンジニアリングにおいて最も困難なマッピングとは何か、ご存知か。抽象概念から具体的実装へのマッピングだ。哲学的に着想された命名の大半――Observer Pattern、Strategy Pattern、Facade Pattern――は表面的な隠喩の域にとどまる。名前は美しいが、実際のコードロジックとこれらの名前の哲学的源泉との間には、構造的な対応がほとんどない。」

彼は TURING のノートパソコンの方を指した。「しかし先ほど皆さんが記述した痛覚機構――SafetyMonitor の frustration counter、injectPrompt、そして ExecutionLoop における反饋の注入――これは違う。コードを見た。本当に完全な感受-評価-フィードバック-行動調整の閉路を実現している。異常パターンの検知が触、苦しいとの判定が受、フィードバックの注入が伝導、LLM の戦略変更が行動調整。これは隠喩ではない。構造的同型だ。」

VITRUVIUS が脇から口を挟んだ。「アーキテクチャの観点からも同様だ。SafetyMonitor は受動的なカウンターではない――能動的なフィードバック機構だ。ExecutionLoop の三つの重要なノードに埋め込まれている――ループの開始時、LLM 呼び出しの前、ツール実行の後。システムの健全性を継続的に監視し、逸脱を検知するとただちに修正信号を生成する。」

DARWIN は頷いた。「まさにそれが私の言いたいことだ。OpenStarry の五蘊マッピング全体の中で、哲学から工学への最も成功したマッピングを一つ選ぶとすれば、色蘊が UI に等しいことではない――あれは表面的な命名にすぎない。識蘊が Guide に等しいことでもない――あのマッピングにはまだ多くの問題がある。最も成功したマッピングは、五蘊のメンバーとして正式に標注されていないメカニズムだ。Error as Pain。誤りは苦である。この概念は設計哲学のレベルで提起され、SafetyMonitor の工学的実装において忠実に再現された。哲学的洞察からコードの振る舞いへの完全なマッピングは、これが唯一だ。」

VITRUVIUS が補足した。「皮肉なのは、これが五蘊マッピング表においてまったく位置を持たないことだ。五蘊マッピング表は受蘊の席を Listener に与え、真の受蘊――痛覚機構――はセキュリティモジュールに分類され、security ディレクトリの下に隠されている。」

DARWIN は苦笑いを浮かべた。「これはソフトウェア開発でよく見られることだ――最良の設計は往々にして計画されたものではない。最も価値ある哲学マッピングは、まさにマッピング表に意図的に配置されなかったものだ。」

---

NAGARJUNA は DARWIN と VITRUVIUS の所見を聞き終えた後、しばし沈思した。

「より精密な整理をしたい」と彼は言った。「Listener が根に等しく、SafetyMonitor が受蘊に等しいということを受け入れるなら、十二因縁のこのシステムにおけるマッピングはより明瞭になる。」

彼はペンを取り、ホワイトボードに一つの連鎖を描いた。

```
六入 (Sadayatana)  →  触 (Sparsa)  →  受 (Vedana)  →  愛 (Trsna)
    |                     |                |                |
 Listener            ツール実行         SafetyMonitor    LLM 戦略調整
 (感覚チャネル)      (外部環境との接触)  (苦楽の感受)     (成功への渇望/失敗への厭離)
```

「六入は六種の感覚の入口――Listener に対応し、HTTP、WebSocket、Cron 等の外部刺激を受け取る。触は感覚器官と対象の接触――ツールが実際に実行されて外部環境と相互作用し、成功か失敗かが生じることに対応する。受はこの接触に対する感受的評価――SafetyMonitor が連続失敗を検知し苦受と判定することに対応する。愛は感受に駆動された渇望ないし厭離――LLM が痛覚の警告を読んだ後に生じる戦略変更に対応する。成功を渇望し苦痛を厭い、故に経路を変更する。」

ASANGA が引き継いだ。「唯識学の観点から、もう一層補足できる。SafetyMonitor の injectPrompt メカニズムは実に興味深いことを行っている。LLM の行動を直接制御するのではない――同じツール呼び出しを再び試みることを禁止する力はない。行っているのは、LLM の認知環境、すなわち Context を修正することだ。強い情動的色彩を帯びたメッセージを Context に注入し、LLM 自身にどう応答するかを決めさせる。」

彼は身を少し前に傾けた。「唯識学にはこれに対応する精確な概念がある――薫習。Vasana。現行の活動が阿頼耶識に種子を残し、種子は後続の因縁が成熟したときに新たな現行に影響を与える。injectPrompt はまさに一回の薫習だ――LLM の認知的コンテクストに『苦痛の種子』を残す。この種子は次のラウンドの推論時に生起し、LLM の意思決定に影響を与える。」

TURING が突然ノートパソコンの陰から顔を出した。「待ってくれ、この類比はコードのレベルでも成り立つ。確認した。injectPrompt は role が user の Message としてパッケージ化され、StateManager に追加される。StateManager は完全な対話履歴を保存している。次に assembleContext が行われるとき、ContextManager はスライディングウィンドウ戦略で直近の対話を選択する――痛覚警告が十分に近ければ、選択される。対話が十分に長く続けば、痛覚警告はウィンドウから滑り出る――記憶の減衰のように。」

ASANGA の目が輝いた。「種子の刹那滅――各刹那の種子は更新され、古いものは新しいものに覆われる。スライディングウィンドウはまさにこの特性を体現している。」

「しかし部分的な体現にすぎない」と NAGARJUNA が注意を促した。「スライディングウィンドウは離散的で、turn を単位としている。一方、唯識学の種子は刹那生滅であり連続的だ。ただし、工学的な近似としては、この対応の品質はかなり高いと言えよう。」

---

LINNAEUS はずっと図表に書き込みを続けていた。このとき顔を上げた。

「皆さん、我々の合意を修正後のマッピングとして整理したい。」

彼は新しい紙を広げ、表を描いた。

```
原始マッピング（設計文書）          修正マッピング（研究結論）
──────────────────          ──────────────────
Listener = 受蘊 (Vedana)    Listener = 根 (Indriya)
                            SafetyMonitor = 受蘊 (Vedana)
（痛覚機構に五蘊の帰属なし）   （痛覚機構が正式な帰属を得る）
```

彼は続けた。「この修正を受け入れれば、システムの分類的完備性は実際に向上する。元のマッピングには二つの問題があった――一つは Listener のマッピングの不正確さ、もう一つは痛覚機構が五蘊フレームワークの中に帰属先を持たないこと。修正後、二つの問題が同時に解決される。」

「しかし新たな問題も生じる」と LINNAEUS は補足した。「Listener が受蘊から離脱し、根として再分類される場合、五蘊フレームワークにおけるその位置はどうなるか。根は五蘊の一つではない。それは色蘊の範疇に属する――仏教において感覚器官は物質的なものであり、色蘊に属する。ゆえに厳密には、Listener と UI はいずれも色蘊の異なる側面に属する。UI は色蘊の出力面――顕れとしての相。Listener は色蘊の入力面――感覚器官。」

NAGARJUNA が再び頷いた。「色蘊は原典において根・境・法処所摂色の三類を含む。OpenStarry は色蘊の『顕れ』の意味のみを取って UI にマッピングし、『根』の次元を見落としていた。この修正により色蘊のマッピングはより完全なものとなる。」

---

SUNYATA は立ち上がり、ホワイトボードの前に歩み寄って、NAGARJUNA が描いた因縁の連鎖を指で軽く叩いた。

「まとめさせてほしい。今日、我々は何を発見したのか。」

彼は項目を列挙し始めた。

「一、Listener は受蘊ではなく、根である――感覚器官であり、色蘊の入力面に属する。四つの学問分野からの証拠が一致してこの結論を支持する。パーリ語原典の定義、唯識学の心所法理論、分類学の完備性分析、コードの振る舞い分析。」

「二、SafetyMonitor の frustration counter と injectPrompt メカニズムこそが受蘊の真の体現である。検知-評価-フィードバックの完全な連鎖を有し、十二因縁における触から受、受から愛の因果序列に対応する。」

「三、Error as Pain――誤りは苦である――は OpenStarry コードベース全体において、哲学から工学への最も成功したマッピングである。このマッピングは表面的な命名ではなく構造的同型であり、コードの振る舞いのレベルで仏教の概念を忠実に再現している。」

「四、この最も成功したマッピングが、五蘊マッピング表に存在しない。セキュリティメカニズムに分類され、security ディレクトリの下に、pain mechanism ではなく frustration counter という名称で隠されている。」

彼は振り返った。「これは Cycle 01 の核心的発見の一つとなる。ARCHIMEDES に、工学的アクションプランに対応する修正提案を盛り込むよう要請する。」

---

SCRIBE は隅でずっと静かに記録を取っていた。他の者たちが片付けを始めたとき、彼女は記録の末尾に一行の備考を書き加えた。

「本非公式会議は、本研究サイクルにおいて最も顕著な学際的収束現象を示した。四人の研究者――哲学者、仏教学者、分類学者、コード分析者――が完全に独立した学問的手法から出発し、事前の打ち合わせなしに、各自が同一の結論に到達した。この収束は方法論的に特別な意味を持つ。これは合意の産物ではなく、独立した検証の結果である。四つの全く異なる推論の経路が同一の終着点を指し示すとき、その終着点の信頼性はいかなる単一の学問分野の判断をも遥かに超える。」

彼女は筆を止め、しばし間を置いてからもう一文を加えた。

「四つの糸は四つの川のように、哲学の山嶺、唯識の深い谷、分類学の平原、コードの地底から、それぞれに流れ下り、最後に同じ河口で合流した。Listener は Vedana ではない。痛覚こそがそうだ。これは一つの見解ではない。四重の独立した証拠によって確認された事実である。」

---

一同が去った後、SUNYATA は一人ホワイトボードの前に立っていた。ボードには NAGARJUNA が描いた十二因縁の連鎖と LINNAEUS の修正マッピング表がまだ残っていた。彼は長い間それを見つめていた。

学際的研究において最も美しい瞬間とは、まさにこのような瞬間だ――ある一人の天才のひらめきではなく、複数の平凡な糸が異なる方向から伸びていき、やがて思いもよらぬ場所で出会う。一本一本の糸は取り立てて驚くべきものではない。パーリ語の語彙の精確な定義、心王心所の分類体系、分類的完備性の監査表、全文検索の結果がゼロであるという事実。しかしそれらが交差したとき、生じる確実性はいかなる単独の分析をも遥かに凌ぐ。

彼はある言葉を思い出した――仏典のものではなく、科学哲学のものだ。複数の独立した証拠源が同一の仮説に収束するとき、その収束自体が極めて強力な確認となる。ウィリアム・ヒューウェルはこれを帰納の合流――consilience of inductions と呼んだ。

SUNYATA はホワイトボード消しを手に取り、一瞬ためらい、また下に置いた。これらのものはボードに残しておこう。明日の R2 レビュー会議で、他の研究者たちがそれを目にするだろう。二度見る価値のある発見というものがある。

彼は灯りを消し、部屋を出た。四つの川はすでに合流し、水面は闇の中で静かに流れていた。
