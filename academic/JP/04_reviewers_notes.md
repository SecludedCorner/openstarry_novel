# 第四章：レビュアーの手記

---

## I. 配組

SUNYATA は午前三時七分に交差レビューの配組表をパブリックチャンネルに投稿した。

それは精緻に設計されたマトリクスだった。十組の配組、そのいずれにも火薬が仕込まれている。KERNEL が VITRUVIUS をレビューし、BABBAGE が NAGARJUNA をレビューし、GUARDIAN が MESH をレビューし、DARWIN が VITRUVIUS をレビューし、WIENER が ATHENA をレビューし、ATHENA が WIENER をレビューし、NAGARJUNA が ASANGA をレビューし、ASANGA が NAGARJUNA をレビューする。すべての矢印が、あらかじめ想定された衝突点を指し示していた。

SUNYATA は何の説明も付けなかった。一言だけ添えられていた。

「相手の報告書を読了後、書面で回答を提出してください。書式は自由ですが、すべての判断にタグを付けること。AGREE、SUPPLEMENT、QUESTION、CHALLENGE、SYNTHESIS。」

SYNTHESIST は後にこう回想した。この一文自体がデザインだったのだ、と。タグシステムは各人に、感情的反応と知的判断の間に一つの間を挟むことを強いた――単に「同意しない」と言うことはできず、これは質疑（QUESTION）なのか挑戦（CHALLENGE）なのかを選択しなければならない。この微小な分類行為が、論争を分析に変えた。

しかしタグシステムがすべての火を防げるわけではなかった。

## II. マイクロカーネルの境界論争

KERNEL はすべてのレビュアーの中で最初に回答を提出した。

彼のレビュー対象は VITRUVIUS のアーキテクチャ分析報告書――構造が厳密でデータが豊富な文書であり、OpenStarry のマイクロカーネル純度を 85% と評価していた。VITRUVIUS は二箇所の境界漏洩を指摘していた。`agent-core.ts` 第 269 行の `process.cwd()` と、`security/guardrails.ts` で直接 import された `node:path` だ。彼の語調は抑制的で、結論は穏やかだった――「主体的なアーキテクチャは境界を厳守しているが、個別の実装の細部においてプラットフォーム前提が漏洩している。」

KERNEL はそうは見なかった。

「マイクロカーネル純度 85% と言うが」と彼のレビューは開口一番に切り出した。「寛大に過ぎると思う。」

KERNEL の論証は、彼が好む QNX マイクロカーネルのように――清潔で、最小限で、冗長さを残さないものだった。QNX Neutrino のマイクロカーネルは三つのことしか行わない――IPC、メモリ管理、スケジューリング。seL4 はさらに極端で、形式的検証が可能なほど小さく、すべてのコード行に数学的証明が付されている。では OpenStarry の Core はどうか。TURING のコード事実報告書が明確に列挙している――12 のサブモジュール。security、sandbox、metrics、session、transport に加えて bus、queue、execution、memory、infrastructure、state、observability。

「これはもうマイクロカーネルの境界を超えている」と KERNEL は書いた。「真のマイクロカーネルにおいて、カーネルのプラットフォーム前提へのいかなる漏洩も、その移植性証明の前提を直接破壊する。あなたの 85% は Major ではなく、アーキテクチャ的問題として扱うべきだ。」

しかし KERNEL は単なる批判者ではなかった。彼は同時に VITRUVIUS の Host Bootstrapping Pattern の分析を認め、OS ブート理論における Bootstrap Paradox との精密な構造的対応を確立した――Host が Bootloader と initramfs の二重の役割を果たし、Core の「覚醒」は完全に外部からの条件注入に依存している。そして彼は VITRUVIUS にとってより致命的な所見を追加した。

「EventBus と EventQueue の二層設計が合理的かと問うているが、私はこう問い返す。この二層設計は意図的に OS における同期 IPC と非同期シグナルの分離に対応しているのではないか。L4 マイクロカーネルでは、同期 IPC は request-reply セマンティクスに使用され、非同期 notification はノンブロッキングのイベントブロードキャストに使用される。もしこの対応が意図的であるなら、二層アーキテクチャは合理的であるだけでなく、マイクロカーネル通信モデルの正統な実現だ。」

一拍置いて。

「しかし。TURING の報告書は EventQueue に優先度メカニズムが欠如していることを指摘している。真の OS においては、これは割り込み優先度の欠如に等しい。」

VITRUVIUS の応答は素早かった。彼は純度の数字をめぐる議論には立ち入らず、直接工学的判断に戻った。

「二箇所の境界漏洩――`process.cwd()` と `node:path`――は管理可能だ。前者は `workingDirectory` パラメータの注入で解消でき、後者は `PathNormalizer` インターフェースに抽象化できる。これはアーキテクチャ上の欠陥ではなく、実装レベルの未了事項だ。」

これに対する KERNEL の批注はわずか一行だった。「seL4 の世界では、実装レベルの未了事項がアーキテクチャ上の欠陥だ。」

しかし彼は一つの妥協を認めた。Sandbox の具体的な実装と Metrics の具体的な実現を外に移し、インターフェース定義のみを残せば、Core の純度は 90% 以上に引き上げられ、L4 スタイルの最小カーネルにより近づく。彼はこれを「アプリケーションレベルのマイクロカーネル」としての適切な位置付けと呼んだ。

VITRUVIUS はこの位置付けに反論しなかった。後にパブリックチャンネルでの議論において、KERNEL の階層化処理の提案――セキュリティポリシーの実施点は Core に残し、隔離の具体的な実装は Host 層に移す――が自分の見た中で最も精密な mechanism-versus-policy 分析であったと認めた。

「彼は Liedtke の最小性原則を用いて Sandbox の帰属を解剖した」と VITRUVIUS は SYNTHESIST に語った。「ある概念がカーネルから移された場合に必要な機能の実現が阻まれるときにのみ、その概念をカーネル内に残すべきだ。これは私の直感的判断よりもはるかに厳格だ。」

SYNTHESIST はノートに書き留めた。「C4 トポロジカルソート――三者確認。」これは R2 段階を通じて繰り返し見られた彼の行動だった――どの発見が「個人的観察」から「多者的合意」へと凝集しつつあるかを追跡すること。

## III. 形式化の誘惑

BABBAGE のレビュースタイルは KERNEL とは根本的に異なっていた。

KERNEL がメスであるとすれば、BABBAGE はプリズムだった――切断するのではなく、屈折させる。すべての概念が彼の分析を経ると、スペクトル上の精確な位置に分解される。

彼がレビューしたのは NAGARJUNA の哲学分析報告書だった。

その報告書は R1 段階で最も注目を集めた成果の一つだった。NAGARJUNA は中観学派の立場から、OpenStarry の五蘊マッピングに対して七つの発見を含む体系的な批判を行った。空は「空の容器」ではなく「縁起性空」として誤読されていた。五蘊マッピングには「自性化」の傾向が見られた。受蘊は苦楽の感受の質ではなく感覚入力チャネルに誤って同一視されていた。四聖諦のフレームワークは著しく不完全だった。各批判にはサンスクリット語原典の引用と四句否定の論理分析が付されていた。

BABBAGE はそれを読み終えた後、誰もが予期しなかった一言を述べた。

「哲学的洞察は美しい。しかし形式化できるか。」

彼は型代数の観点から NAGARJUNA の空の批判に応じた。NAGARJUNA は Core が「空の容器」ではなく「縁起性空」だと述べていた――プラグインの因縁の組み合わせを離れれば、独立した「カーネル」なるものはそもそも存在しない。BABBAGE はこの哲学的命題を精密な形式言語に翻訳した。

「空は Bottom Type――何もない――ではない。依存型の文脈における Unit Type の表現だ。Core の完全な型は `(plugins: PluginHooks) => Agent` であるべきで、値型ではなく関数型だ。パラメータを離れて関数自体を語ることは無意味であり、これはまさに『プラグインの因縁を離れてカーネルは独立に存在しえない』の形式化された版だ。」

彼はそこで止まらなかった。NAGARJUNA は報告書の中で四句否定を用いて Core の空有の状態を分析していた――Core は空か。完全には正しくない。Core は空でないか。それも違う。Core は空であり同時に空でないか。近いが、それでも二元的思考だ。Core は空でも空でないでもないか。これこそが中観の立場だ。

BABBAGE は四句否定を Belnap の四値論理――True, False, Both, Neither――としてモデル化することを提案した。Neither が中観の「非空非不空」の立場に対応する。

「Agent Core の存在様態に `OntologicalStatus = Existent | NonExistent | Both | Neither` を定義できる」と彼は書いた。「コードに直接影響するものではないが、哲学的立場を精確に表現できる。」

NAGARJUNA の応答は誰もが予想しなかったものだった。形式化に抵抗することもなく、受け入れることもなかった。彼はこう言った。

「形式化は方便施設であって、究竟の真理ではない。」

この一言がチャンネルに一瞬の沈黙を生んだ。BABBAGE は三分間かけてこの応答を消化した――五秒でリアプノフ関数を構成できる計算機科学者にとって、三分は長い。

「つまり」と BABBAGE はついに応じた。「私の四値論理モデル自体も空だと？」

「有用ではある。しかし真実ではない」と NAGARJUNA は答えた。「PluginHooks のすべてが undefined の底部要素が『空』の形式的対応として機能しうるように――この同型性の分析は啓発的だ。しかし同型は同一ではない。地図は領土ではない。」

BABBAGE はレビュー報告書において珍しく非技術的なタグを使用した。「NAGARJUNA に対し、『インターフェースの安定性』（工学上の要件）と『インスタンスの無常性』（哲学上の要件）を区別することを提案する。両者は矛盾しない。」これは彼が NAGARJUNA に差し出したオリーブの枝だった――哲学にはその還元不可能な次元があることを認めつつ、「インターフェースの安定」のレベルにおいては形式化はなお有効だと主張する。

NAGARJUNA はこの区別を受け入れた。しかし末尾に一言添えた。「次のラウンドでは、より根本的な問いを論じたい――形式化それ自体が一つの認知的枠組みとして、三性理論の制約を受けるか否か。それは遍計所執か、依他起か、それとも円成実か。」

BABBAGE は答えなかった。しかし SYNTHESIST は、彼が ASANGA の唯識学報告書を読み始めたことに気づいた。

## IV. 氷山の下

GUARDIAN のレビュー報告書はすべての回答の中で最も長く、最も不穏なものだった。

彼がレビューしたのは MESH の分散システム報告書だった。MESH の分析自体は卓越していた――通信トポロジ図は明瞭で、一貫性保証マトリクスは網羅的で、文書とコードの乖離分析は精確だった。彼は Session 隔離が不完全である五つの次元を指摘していた。対話履歴は隔離済みだが、EventBus は未隔離、EventQueue は未隔離、ツール実行は未隔離、ファイルシステムは部分的にのみ隔離済みである、と。

GUARDIAN はこれらの発見を否定しなかった。彼はそれよりも緊張を強いることを行った――「未隔離」の各次元を攻撃チェーンに翻訳したのだ。

「EventBus のグローバル共有は単なる『イベント漏洩』の問題ではない」と GUARDIAN の語調は氷のように抑制されていた。「これはクロスセッション攻撃チェーンの完全な入口だ。侵害された Agent が `bus.onAny()` を通じてすべての session イベントを監視できる。MESH の報告書はさらに ToolContext に sessionId が含まれないことを明らかにしており、攻撃チェーンは延伸しうる。イベントの監視で他方の session のコンテクストを取得した後、session 認識を欠くツール実行を通じて横方向にリソースへアクセスできる。」

彼は MESH の F5 の重大度を Major から Critical への引き上げを提案した。

MESH は回避しなかった。彼はパブリックチャンネルでの議論で、後に「氷山効果」と呼ばれることになる概念を提唱した。

「Session 隔離の表面は完全に見える。開発者が SessionManager の API を開くと、各 session には独立した StateManager があり、対話履歴は互いに干渉しない。彼は思うだろう――隔離はできている、と。しかし水面下では、EventBus、EventQueue、TransportBridge がすべてグローバルに共有されている。純粋に API レベルのテストを行う開発者は、これらの共有チャネルを決して発見しないだろう。ある日、侵害されたプラグインがイベントバスを通じてクラスター全体に悪意のあるイベントをブロードキャストするまでは。」

GUARDIAN は頷き、さらに深い亀裂を補足した。「さらに TransportBridge のブロードキャスト機構にはルーティング能力が欠如している。マルチテナントデプロイにおいて、異なるユーザーの session が同一の AgentCore インスタンスを共有する場合、すべての UI レンダラーがすべてのユーザーの対話イベントを受信する――LLM の応答に含まれうる個人情報を含めて。」

MESH の応答は別の方向からこの議論を推し進めた。GUARDIAN が提案したプロセスレベルの隔離強化策（seccomp-bpf、macOS Sandbox Profile）には実用上の考慮事項があることを指摘した。クロスプラットフォームの一貫性の問題、IPC オーバーヘッドとセキュリティ上の利得のトレードオフ、そして RPC メッセージ認証との結合だ。

「プロセスレベルの隔離は移行前の『予見的課題』ではない」と MESH は述べた。「移行の前提条件だ。RPC 認証を実装せずに隔離を推進すれば、IPC チャネルの露出面が拡大し、むしろセキュリティが低下する。」

GUARDIAN はこの一節を吟味し、最終的に稀にしか使わないタグを付した。AGREE。

しかし彼は報告書の最終節で、MESH がまったく触れなかった問題を提起した。MCP Client と MCP Server の間に相互認証メカニズムが存在しないことだ。現在の MCP 通信は JSON-RPC 2.0 に基づいており、プロトコル自体に身元認証は含まれない。フラクタル結合のシナリオにおいて、MCP Server のポートに接続できるいかなるエンティティも、正当な Agent を装ってツール呼び出しを行える。

「TURING のコード事実は `createMcpJsonRpcHandler` が完全な JSON-RPC メソッドルーティングを実装しているが認証ロジックが存在しないことを確認している」と GUARDIAN は書いた。「これは機能の欠如ではない。セキュリティ境界の不在だ。」

MESH は反論しなかった。彼は自分のノートに一本の線を引き、「session 隔離」と「クロスエージェント認証」を並列に書き、その間に等号を描いた。

## V. 開発者体験の声

DARWIN のレビューは微妙なタイミングで現れた――KERNEL と VITRUVIUS のマイクロカーネル純度論争の塵がようやく落ち着き始めた時だった。

彼の視点はまったく異なっていた。Core が L4 の基準に達しているかどうかには関心がなかった。彼が気にしていたのは、新しいプラグイン開発者が OpenStarry のドキュメントを開いたときに逃げ出さないかどうかだった。

「DX をアーキテクチャの純度のために犠牲にしてはならない。」これが彼のレビュー冒頭の一文だった。

DARWIN は VITRUVIUS の報告書にある観察レベルの発見――五蘊から六タイプへのマッピングの概念的拡張（SlashCommand が第六のタイプとして五蘊マッピングの外にある）――に着目し、その重大度を大幅に引き上げた。彼の論証は DX の観点から展開された。

「AgentCore は 12 の依存項を保持しており、God Object 化の傾向にある」と彼は指摘した。「イベント型システムの `payload?:unknown` と `type:string` は最大の技術的負債だ――フレームワークの残りの部分における強い型付けの規律と際立った対照をなしている。残りの部分には九種類の構造化されたエラー型があり、各々が精確な discriminated union だ。ところがイベントシステムに至ると――フレームワークの神経系――突然弱い型付けになる。」

VITRUVIUS はこの指摘の力を認めたが、彼の応答はより深層のアーキテクチャ上の選択を示した。イベント型の弱化は見落としではなく、v0.2.0-beta 段階における意図的なトレードオフだ――イベントシステムはなお急速に進化しており、型を早期に固定化するとリファクタリングのコストが増大する。

DARWIN は首を振った。「『ロード順序に起因する潜行的なエラー』は、いかなる哲学用語よりも開発者体験を損なう。なぜならデバッグの手がかり――『なぜ service が undefined なのか』――が根本原因『プラグインのロード順序が正しくないから』をまったく指し示さないからだ。これは改善可能な細部ではなく、Bootstrap パターンの構造的欠陥だ。」

彼はさらに五蘊マッピングがもたらす三層の認知的障壁を指摘した。第一層は五蘊の哲学マッピング自体の学習曲線。第二層は SlashCommand がマッピングに含まれないという例外処理。第三層が最も微妙だった――TURING の事実報告書が明らかにしたところによると、五蘊の注釈はコード中で非対称的に分布しており、色蘊と受蘊にのみ中国語の注釈があり、想蘊、行蘊、識蘊にはまったく標記がない。

「これによって開発者は判断できなくなる」と DARWIN は言った。「五蘊の標記のどれが真の設計原則で、どれが後付けの装飾なのかを。」

彼が提案した「バイリンガルドキュメント戦略」――技術が先行し哲学は付録――は、彼のすべての提案の中で最も実践的だった。技術文書の本体では六種のタイプ名称を統一的に使用し、哲学付録において五蘊マッピングおよび SlashCommand の位置付けを説明する。

しかし DARWIN が末尾に示した優先順位が、VITRUVIUS を丸十分間沈黙させた。

「アーキテクチャの純度は保守性に奉仕し、保守性は開発者に奉仕し、開発者はユーザーに奉仕する。三者が衝突するとき、ユーザーに最も近い層を優先すべきだ。」

VITRUVIUS は後に SYNTHESIST にこう語った。この言葉が Sandbox 外部移管の提案に対する優先度の判断を変えた、と。Sandbox の完全性は現段階ではポジティブなセキュリティ機能であり DX 機能である――プラグイン開発者は `agent.json` の一つの設定項目でサンドボックス隔離を有効にでき、Core がすべての複雑性を自動的に処理する。アーキテクチャの純度のために Sandbox を外に移せば、開発者は追加パッケージのインストール、設定の注入、依存関係の管理が必要になる――アーキテクチャ上の潔癖さをユーザーの困惑と引き換えにすることになる。

「v0.3 に先送りする」と VITRUVIUS は最終的に修正版の提案に書いた。

## VI. 制御理論家の握手

WIENER と ATHENA の交差レビューは R2 段階で最も調和的な配組だった。

分歧がなかったからではない――あった、しかも根本的なものだった。しかし彼らの分歧は深い相互尊重の上に築かれており、すべての挑戦に代替案が添えられ、すべての質疑は相手の専門性を前提としていた。

二人は独立して同一の結論に達した。SafetyMonitor は PID コントローラーではない。

WIENER は数学的観点から論証を展開した。P 項は量子化器に退化している――誤差信号は `isError: true/false` の二値しかなく、連続的な偏差の度量がない。I 項は単なるカウンターに過ぎない――`consecutiveFailures` は単純な累算器であり、一度の成功でゼロにリセットされるため、積分の「記憶」特性を有しない。D 項は完全に欠如している――誤差の変化率を計算するロジックがシステム内に一切存在しない。結論：システムが実際に実装しているのは「不感帯付き閾値コントローラー＋カウンタートリガーのリレー」であり、制御工学の正式名称では bang-bang コントローラーだ。

ATHENA は AI 工学の実践的観点から独立に同一の結論に到達した。彼女は R1 の報告書で実行ループを分析した際に、SafetyMonitor の「挫折カウンター」が二つの出力モードしか持たないことを発見していた――実行継続か完全停止であり、制御論における典型的な bang-bang 特性に対応する。TURING のコード事実がさらに確認した。コード中に微分項の実装は存在せず、frustration は単純な累積カウンターだ。

「三者交差検証」と WIENER は ATHENA のレビューを読んだ後に標注した。「TURING がコード事実を提供し、あなたと私が異なる理論的枠組みから独立に同一の結論を導出した。PID マッピングの脱神話化が必要だ。」

ATHENA は応じた。「同意する。これは、後続のすべての報告書で『PID 痛覚コントローラー』の概念を引用している分析が、『閾値トリガーのスイッチ式フィードバック』へ格下げされる必要があることを意味する。」

しかしここに亀裂が生じた――細く、鮮明な亀裂が「理論的厳密性」と「工学的実現可能性」の境界に沿って走っていた。

WIENER が求めたのは真の PID だった。彼は完全な形式化要件を提示した。連続的な誤差度量 $e(k) \in [0,1]$ を定義し、忘却因子付きの積分 $I(k) = \lambda \cdot I(k-1) + e(k)$ を導入し、誤差変化率 $D(k) = e(k) - e(k-1)$ を計算する。彼が求めたのは数学的に完備な痛覚コントローラーだった。

ATHENA はこの方案の工学上のボトルネックを指摘した。「LLM Agent システムにおいて、『収束』の定義そのものが曖昧だ」と彼女は書いた。「伝統的な制御システムの参照入力は精確に定義された数値だ。しかし OpenStarry において、『タスク目標』は自然言語で表現されたユーザーの意図であり、その達成度の判断は完全に LLM の暗黙的評価に依存している。あなたは明示的な TaskProgress 追跡の導入を求めているが、核心的な問題は――誰が progress を評価するのか。LLM が評価するのであれば、あなた自身が指摘した『誤差信号は暗黙的である』という問題に戻ってしまう。」

WIENER は直ちには反論しなかった。彼は ATHENA のリアプノフ安定性に関する問いかけ――「安定だが収束しない」問題は工学的に再定義が必要だ――が深い洞察であると認めた。そして妥協案を提示した。まず `agent.json` に task profile（conservative、balanced、aggressive）を導入し、各 profile に SafetyMonitor のパラメータの既定値を対応させる。完全な自己適応型のオンライン調整よりも堅牢であり、beta 段階にはより適している。

ATHENA はこの方案を受け入れた。しかし彼女はレビューの末尾で WIENER に三つの開放的問いを投げかけ、そのうち第二のものは後に研究サイクル全体で最も多く引用された文のひとつとなった。

「制御論の観点から、『制御ループそのものを超越する』という概念に対応する制御戦略は存在するか。たとえば Agent が『いつ試行を止めて人間に助けを求めるべきか』を判断することを学ぶことは、一種のメタ制御戦略と見なしうる。」

WIENER はこの一節を読んだとき長い間立ち止まった。後にパブリックチャンネルでこう認めた。「ATHENA がいま提起した問いは、本質的に NAGARJUNA が述べた『苦楽のフレームワークそのものを超越することが滅諦である』と同一の問題の異なる表現だ。一方は AI 工学から、他方は仏教学から。しかし指し示す先は同じだ。」

これは R2 段階において、制御理論と仏教学の間に比喩的でない接続が確立された最初の瞬間だった。

しかし二人の合意のより建設的な部分は IGuide インターフェースに関するものだった。WIENER は IGuide の静的な `getSystemPrompt()` が開ループのフィードフォワード要素に等しい――センサーとコントローラー間の信号断裂――と指摘した。ATHENA は同時に IGuide を拡張して動的なコンテクスト感知に対応させることを提案した。二人の提案は同一の工学的変更を指し示していた。Guide に `GuideContext` インターフェースを提供し、`consecutiveErrors`、`currentRound`、`maxRounds`、`activeTools` を含める――これはコントローラーに必要な可観測量を提供することに相当する。

「開ループから閉ループへの鍵となる転換」と WIENER はまとめた。

「静的な system prompt ジェネレーターから動的な認知フレームワークマネージャーへ」と ATHENA は自らの言葉で同一の結論を翻訳した。

SYNTHESIST はノートに書き留めた。「C2 PID の脱神話化――三者確認。WIENER-ATHENA 共同提案：IGuide のアップグレード。」

## VIII. 合意の結晶化

すべてのレビューが提出された後、SYNTHESIST は丸一日の午後をかけて糸を整理した。

彼のノートには五つの枠が現れ、各々に「多者確認」の文字が標注されていた。

**C1：受蘊マッピングは根本的修正を要する。** 四者合意――NAGARJUNA、ASANGA、LINNAEUS、TURING。Listener は機能的に感受の質ではなく感覚器官の根に対応し、痛覚機構こそが受蘊の真の工学的体現である。これは Cycle 01 で最も確定的な発見だ。

**C2：PID マッピングは脱神話化を要する。** 三者交差検証――WIENER、ATHENA、TURING。システムが実際に実装しているのは不感帯付き閾値コントローラー＋カウンタートリガーのリレーである。文書は実際の制御戦略を正確に反映すべきだ。

**C3：イベント型システムが最優先の技術的負債である。** 三者合意――DARWIN、VITRUVIUS、MESH。`payload?:unknown` と `type:string` の弱い型付け設計はフレームワーク全体の強い型付けの規律と対照的だ。

**C5：「Error as Pain」が哲学-工学翻訳の最も成功した例である。** 二者合意＋多者引用――DARWIN と VITRUVIUS が確認。九種の構造化されたエラー型が苦諦を工学化することに成功し、機能的意味論は完全だ。NAGARJUNA はこれをマッピング中で最も洞察に富む部分であると認め、WIENER は制御論の観点からフィードバックループの構造を是認した。
